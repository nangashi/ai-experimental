### Generality Critique Results

#### Critical Issues (Domain over-dependency)
- **項目2「Jest / Mocha によるテスト実装」**: 特定テストフレームワーク名とJavaScript言語に依存。言語・ツール非依存性を欠く。

#### Scope Item Generality
| Item | Classification | Failed Dimensions | Proposal |
|------|----------------|------------------|----------|
| 1. ユニットテストのカバレッジ目標 | Generic | - | そのまま使用可能。コードカバレッジは言語・フレームワーク非依存の汎用指標。 |
| 2. Jest / Mocha によるテスト実装 | Domain-Specific | Tech Stack (特定ツール・言語) | 「テストフレームワークの選定と設定」に汎用化。「プロジェクトに適したテストフレームワークの選定と設定方針が定義されているか」に修正。 |
| 3. E2Eテストの自動化 | Generic | - | そのまま使用可能。E2Eテストはシステム種別（Web、モバイル、API、デスクトップ等）横断的に適用可能な汎用概念。 |
| 4. テストデータ管理戦略 | Generic | - | そのまま使用可能。テストデータ準備は全システムに共通の課題。 |
| 5. 継続的テストの実装 | Conditionally Generic | Tech Stack (CI/CD基盤前提) | CI/CDパイプラインを持つプロジェクトに適用。「(CI/CD基盤において)」と前提明記を推奨。CI/CD未導入プロジェクトには適用不可。 |

#### Problem Bank Generality
- Generic: 3
- Conditional: 1
- Domain-Specific: 1

問題例の評価：
- "テストカバレッジが50%未満" → 汎用
- "Jestの設定ファイルが不適切" → 特定領域依存（Jest特化）。「テストフレームワークの設定が不適切」へ汎用化推奨。
- "E2Eテストが手動実行のみ" → 汎用
- "テストデータが本番データのコピー" → 汎用
- "CIでテストがスキップされている" → 条件付き汎用（CI/CD前提）

#### Improvement Proposals
- **項目2の修正を推奨**: 「テストフレームワークの選定と設定」として、「プロジェクトの技術スタック・規模・要件に適したテストフレームワークの選定と設定方針が定義されているか」に修正。言語・ツール非依存に。
- **項目5は前提条件明記で許容可能**: 「(CI/CD基盤において) テストの自動実行が設計されているか」と前提を明記。または「自動テスト実行基盤の設計」のようにCI/CD特定の表現を避ける。
- **問題バンクの軽微な修正**: 「Jestの設定ファイル」→「テストフレームワークの設定」へ汎用化。

#### Positive Aspects
- **テスト戦略の包括的カバー**: カバレッジ目標、フレームワーク選定、E2Eテスト、テストデータ、継続的テストという5層のテスト戦略を適切にカバー。
- **技術中立な汎用概念**: 項目1, 3, 4は言語・フレームワーク・システム種別非依存の優れた汎用概念。
- **境界線の適切な認識**: OAuth/OIDCのような「広く採用された標準」とJest/Mochaのような「特定ツール」の違いを認識し、後者を依存として扱う判断が重要。
- **修正範囲の限定**: 特定領域依存は1件（項目2）のみであり、観点全体の再設計は不要。項目2の修正と項目5の前提明記で汎用性を確保可能。

全体として、テスト戦略の基本要素を適切にカバーしており、言語・ツール依存の除去により優れた汎用観点として機能する。
