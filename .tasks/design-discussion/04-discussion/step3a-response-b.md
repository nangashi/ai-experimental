# Step 3a: 方式B擁護者の回答

## 1. ハイブリッド案全体への評価

### 強み（方式Bの視点から）

**1. リファレンスカテゴリ照合の構造的検証層としての採用は正しい判断**

ハイブリッド案がリファレンスカテゴリを「構造的検証層」として明示的に組み込んだことは、弁論で方式Bが主張した核心的利点——外部基準に基づく検出の下限保証——を反映している。特に以下の設計判断を支持する:

- 内容充実度チェックの全カテゴリ均一適用（論点2）。Step 0d §2.3（パターン完全適用）への直接的対処
- A-3b依存範囲を「リファレンスカテゴリ外のみ」に限定する設計（論点5）。原理的に脆弱なA-3bへの依存を構造的に最小化
- 交差パターンテンプレートと自由推論の統合（論点4）。下限安定性と探索力の両立

**2. 50%閾値の修正は改善**

固定閾値を廃し「該当するにもかかわらず未識別のものがないか」という判定に変更した設計は、方式Aの恣意性批判を適切に反映しつつ、リファレンスサブ項目に基づく直接照合の構造を維持している。

**3. 敵対的検証層の構造改良（入力制限）は合理的**

Phase 0.5-4への入力を「requirements.md + 5視点の要約」に制限する設計は、弁論で方式Bが指摘したアンカリング問題への構造的対処として適切である。

### 弱み（方式Bの視点からの懸念）

**1. リファレンスカテゴリが「後段の検証のみ」に配置されている点**

これが最大の懸念であり、後述の質問への回答で詳細に論じる。要旨: リファレンスカテゴリを導出時に使わず、導出後の照合のみに使うのは、方式Bの核心的利点——構造化された問いによるLLM推論の品質向上——を捨てている。

**2. 「方式Bの補完指示は不採用」の判断は過剰**

論点6で「同一コンテキスト内でリファレンス走査後に『追加で』と問う構造はアテンション配分バイアスの問題が残るため不採用」とされているが、この判断は弁論ラウンド1で反論した通り、Step 0d §1.1の知見の過大適用である。リファレンスカテゴリをプロンプトに注入する設計自体を不採用にする根拠にはならない（後述）。

**3. 補完導出の位置づけが不明確**

Phase 0.5-3で「不足カテゴリの補完導出（別サブエージェント、コンテキスト独立）」とあるが、この補完導出は「導出層が見逃した検討事項をリファレンスカテゴリに基づいて導出する」処理であり、実質的に方式Bのフェーズ A-1（リファレンス駆動導出）と同じ処理をPhase 0.5-3で行うことになる。であれば、最初からPhase 0.5-1の導出時にリファレンスカテゴリを活用した方が効率的ではないか。

## 2. 質問への回答: リファレンスカテゴリを「導出のガイド」としても使うべきか

### 結論: はい、使うべきである。ただし「二段階注入」を提案する。

リファレンスカテゴリは「導出後の検証」だけでなく「導出のガイド」としても使う方がカバレッジは上がる。これは方式Bの核心的主張であり、以下にその根拠と具体的設計を述べる。

### 根拠1: 構造化された問いはLLMの導出品質を向上させる

方式Bの基本思想は「LLMに自由に問うよりも、構造化された問いを投げる方が品質が安定する」である。この原理はハイブリッド案のPhase 0.5-1にも適用される。

現在のハイブリッド案:
- Phase 0.5-1: 「技術インフラ視点から検討事項を導出せよ」（視点レンズのみ）
- Phase 0.5-3: 導出結果をリファレンスカテゴリと照合し、不足を検出

改善案:
- Phase 0.5-1: 「技術インフラ視点から検討事項を導出せよ。参考として以下のカテゴリが存在する: [カテゴリ名のリスト]」（視点レンズ + カテゴリ名ガイド）
- Phase 0.5-3: 導出結果をリファレンスカテゴリの**サブ項目**と照合し、不足を検出

この設計では、Phase 0.5-1でカテゴリ名（粒度の粗いガイド）を提示し、Phase 0.5-3でサブ項目（粒度の細かい検証）を使う。導出と検証で**リファレンスの異なる粒度**を使い分ける。

### 根拠2: 「後段での補完」は「最初からの網羅」より構造的に劣る

ハイブリッド案の現設計では、導出層（Phase 0.5-1）がリファレンスカテゴリなしで導出し、検証層（Phase 0.5-3）で不足を検出し、補完導出を行う。この「導出→検出→補完」の3ステップは、「最初からガイド付きで導出→検証で深さを確認」の2ステップより:

- **サブエージェント数が増加する**: 補完導出に追加のサブエージェントが必要（Phase 0.5-3の「別サブエージェント」）
- **情報効率が低い**: 同一視点の検討事項を2回（初回導出と補完導出）に分割して生成するため、全体像の一貫性が損なわれる
- **補完導出のコンテキスト独立性が非効率を生む**: 補完導出はPhase 0.5-1の出力を入力に含めない（アンカリング防止のため）ので、初回導出と補完導出の間に重複が生じやすい

最初からカテゴリ名をガイドとして提示すれば、導出時点で各カテゴリの検討事項を意識的に走査するため、Phase 0.5-3での不足カテゴリ数が減少し、補完導出の必要性自体が低下する。

### 根拠3: アンカリングのリスクは「二段階注入」で構造的に管理できる

「リファレンスカテゴリをプロンプトに注入するとカテゴリ外の検討事項が検出されにくくなる」というアンカリング懸念は妥当である。しかし、以下の「二段階注入」設計でこのリスクを構造的に管理できる:

**Phase 0.5-1（導出）: カテゴリ名のみ注入**
- 注入するのは「アーキテクチャスタイル」「データ永続化」「性能設計」等の**カテゴリ名のリスト**のみ
- サブ項目の詳細は注入しない
- プロンプト指示: 「以下のカテゴリは参考である。これらに限定せず、要件に基づいて検討事項を導出せよ。カテゴリ外の検討事項も含めること」
- カテゴリ名は方向性のガイドとして機能するが、詳細なサブ項目がないため、アンカリングの強度は限定的

**Phase 0.5-3（検証）: サブ項目レベルで照合**
- ここで初めてリファレンスカテゴリの主要サブ項目を用いた充実度チェックを実施
- 導出時にはサブ項目を見ていないため、導出結果は「サブ項目リストの走査」ではなく「カテゴリ名に触発された推論」の結果であり、検証層の独立性が一定程度維持される

この設計のポイントは、**導出と検証でリファレンスの粒度を分離する**ことである。導出時はカテゴリ名（粗い粒度）で方向性をガイドし、検証時はサブ項目（細かい粒度）で深さを確認する。これにより:
- 導出層のカバレッジ向上（カテゴリ名がLLMのアテンションを広く分配）
- 検証層の独立性維持（導出時に見ていないサブ項目で検証）
- カテゴリ外検出力の維持（「これらに限定せず」の指示 + カテゴリ名のみの注入でアンカリング強度を抑制）

### アンカリング強度の見積もり

| 注入レベル | 注入内容の例 | アンカリング強度 | カバレッジ向上効果 |
|-----------|------------|----------------|----------------|
| なし（現ハイブリッド案） | 視点名のみ（「技術インフラ視点」） | 最低 | 最低（自由推論のみ） |
| カテゴリ名のみ（提案） | 「アーキテクチャスタイル、データ永続化、性能設計…」（8項目） | 低〜中 | 中〜高（方向性ガイド） |
| カテゴリ名+サブ項目概要 | 各カテゴリに3-5サブ項目 | 中〜高 | 高（構造的走査） |
| 完全なリファレンス注入（方式B原案） | カテゴリ+全サブ項目+根拠 | 高 | 最高（ただしカテゴリ外検出力低下） |

提案する「カテゴリ名のみ注入」は、アンカリング強度とカバレッジ向上効果のバランスが最も良い位置にある。

## 3. 改善提案

### 改善1: Phase 0.5-1への「カテゴリ名ガイド」の追加（上述）

Phase 0.5-1の各サブエージェントに、担当視点のリファレンスカテゴリ名リストを「参考」として注入する。サブ項目は注入しない。プロンプトは「これらに限定せず」を明示する。

**期待効果**: Phase 0.5-3での不足カテゴリ数の減少 → 補完導出のサブエージェント数削減 → 全体コストの低減

### 改善2: 補完導出の条件付き実行の明確化

Phase 0.5-3の補完導出は「不足カテゴリが検出された場合のみ」実行する条件付き設計とする。改善1によりPhase 0.5-1の導出品質が向上すれば、補完導出が不要なケースが増加する。

**コスト影響**: 最良ケース（補完不要）: arch_design合計 8サブエージェント。最悪ケース（補完が必要）: 8 + 補完サブエージェント数（最大3）= 11。現ハイブリッド案の「8〜11」と同等だが、最良ケースの確率が向上。

### 改善3: カバレッジマトリクスのユーザー提示形式

Phase 0.5-5のユーザー提示で、カバレッジマトリクスの「不足カテゴリ」を視覚的にハイライトする形式を推奨する。全30カテゴリを列挙するのではなく:

```
■ カバレッジ照合結果:
  - 全30カテゴリ中 26カテゴリ: 導出層で充分にカバー
  - 4カテゴリで追加検討事項を検出:
    - [カテゴリ名1]: [追加された検討事項の要約]
    - [カテゴリ名2]: [追加された検討事項の要約]
    ...
```

この形式により、論点7（過剰検出のコスト）で方式Aが指摘した「注意の希釈」を防ぎつつ、リファレンスカテゴリ照合の付加価値を明示する。

### 改善4: process_designにも敵対的検証の簡易版を検討

現ハイブリッド案のprocess_designにはPhase 0.5-4（敵対的検証）がない。process_designでは「arch_designの決定を前提としたときの運用障害シナリオ」が暗黙要件の発見に有効である可能性がある。ただし、コスト制約（process_design合計 5〜6回）との兼ね合いで、追加1サブエージェントが許容されるかはメインエージェントの判断に委ねる。
