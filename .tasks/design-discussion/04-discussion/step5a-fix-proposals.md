### 修正M-01: 重複排除の判定基準の明示（F-01対応）

**重要度**: 中

**修正内容**: Phase 0.5-3aのプロンプト構造に重複排除の判定基準を追加する。

```
## 重複排除の基準
以下のいずれかに該当する場合、2つの検討事項を「同一」とみなし統合する:
1. 同一の要件IDに紐付き、かつ検討の対象（何を決定するか）が同一
2. 異なる視点から導出されたが、決定すべき事項が実質的に同一（例: 「認証方式の選定」が技術インフラ視点と規制視点の両方で導出）

統合時のルール:
- 最も詳細な検討概要を保持する
- 全ての導出元視点を記録する
- 全ての要件ID紐付きを集約する
- 異なる初期選択肢が挙がっている場合は全てを統合する
```

**パターン完全適用**: process_designのPhase 0.5-3(p)にも同じ基準を適用する。

### 修正M-03: 内容充実度チェックの判定ガイドライン（F-03対応）

**重要度**: 高

**修正内容**: Phase 0.5-3bのプロンプト構造に判定ガイドラインを追加する。

```
## サブ項目の判定ガイドライン

### 「該当・識別済み」の条件
対応する検討事項が以下を全て満たす:
- 検討の対象（何を決定するか）が具体的に記述されている
- 少なくとも2つの選択肢が示されている
- 要件との紐付き根拠が明記されている

### 「該当・未識別」の条件（以下のいずれか）
- 対応する検討事項が存在しない
- 対応する検討事項はあるが、検討の対象が曖昧（例:「セキュリティを考慮する」のみ）
- 対応する検討事項はあるが、選択肢が示されていない

### 「非該当」の条件
- このプロジェクトの特性上、サブ項目が検討対象にならない
  - 理由は以下のいずれかであること:
    a. 要件書に明示的に除外されている
    b. プロジェクトの技術スタックから論理的に不要（例: シングルページアプリでオフライン対応不要）
  - 「要件に記載がない」のみでは「非該当」と判定しない。暗黙要件の可能性があるため「該当・未識別」とする

### 要件曖昧さの区別
「該当・未識別」の原因を以下のラベルで区別する:
- [導出不足]: 要件から導出可能だが検討事項に含まれていない
- [要件曖昧]: 要件自体が曖昧で、具体的な検討事項の導出が困難。この場合、Phase 0.5-5のユーザー検証で要件の明確化を促す
```

**パターン完全適用**: process_designのPhase 0.5-3(p)にも同じ判定ガイドラインを適用する。
