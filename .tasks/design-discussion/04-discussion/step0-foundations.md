# Step 0: 前提・定義・制約の確立

## 1. 用語定義

### 1.1 設計検討事項（Design Concern）

**定義**: ソフトウェアシステムの設計フェーズにおいて、方針決定または設計判断が必要な事項。以下の3層を含む。

- **方針レベル（Architecture Concern）**: システム全体の構造・スタイル・品質特性に関わる決定事項。arch_designの責務。例: アーキテクチャスタイルの選定、データ永続化戦略、認証方式
- **手順レベル（Process Concern）**: 方針を具体的な開発・運用の手順に落とし込む際に必要な決定事項。process_designの責務。例: ブランチ戦略、CI/CDパイプライン設計、テスト戦略
- **横断的関心事（Cross-cutting Concern）**: 複数のコンポーネントや設計次元にまたがり、方針・手順の両レベルで検討が必要な事項。例: セキュリティ、ロギング、エラーハンドリング、国際化

**注意**: 「設計検討事項」は Decision Point（個別の意思決定ポイント）よりも広い概念であり、Quality Attribute Scenario（品質属性シナリオ）を含むが、それに限定されない。技術選定、構造分割方針、運用設計方針など、設計フェーズで意識的に検討すべきすべての事項を包含する。

### 1.2 漏れの種類

| 種類 | 定義 | 発生フェーズ | 検出困難度 |
|------|------|------------|-----------|
| **識別漏れ（Identification Gap）** | 検討すべき事項が設計プロセスの中で一度も認識されないまま後工程に進むこと | 導出フェーズ | 高：認識されていないものは検出できない |
| **検討深度の不足（Depth Gap）** | 検討事項として識別されたが、十分な深さで検討されないまま方針決定が行われること | 設計フェーズ | 中：識別済みのため事後レビューで検出可能 |
| **引き継ぎ漏れ（Handoff Gap）** | ある設計フェーズの検討結果が次のフェーズに正しく伝達されないこと。**双方向**で発生する | フェーズ遷移時 | 中：入出力の突合で検出可能 |

**引き継ぎ漏れの双方向性**:
- **arch → process 方向**: arch_designの設計決定がprocess_designに伝達されず、process_designがarchの前提と矛盾する手順を設計するケース
- **process → arch 方向（逆流）**: process_design中にarch_designの不足が発覚するケース。例: CI/CDパイプラインの設計中に「ロールバック方針がarch_designで未決定」と判明する。この場合、process_designからarch_designへの差し戻しパスが必要

**過剰検出（False Positive）の扱い**: 本議論のスコープには含めない。過剰検出はユーザーの確認コストを増やすが、漏れ（False Negative）とは異なり致命的リスクを生まない。ただし、方式の設計上、過剰検出が大量に生じる方式はコスト許容範囲を超える可能性があるため、コスト面の制約として間接的に考慮する。

### 1.3 網羅性の評価基準

「漏れなく検討できた」状態を以下の条件で定義する:

1. **カテゴリカバレッジ**: 事前定義された分類枠組み（Step 0bで確立）の全視点について、少なくとも1つの検討事項が識別されているか、「該当なし」の判断とその根拠が明記されている
2. **要件トレーサビリティ**: 入力（requirements.md）の全要件（SR/NFR/制約）に対して、対応する検討事項が1つ以上存在する。対応が不要な要件にはその理由が明記されている
3. **交差分析完了**: 要件間の相互作用（特にNFR間のトレードオフ）が識別され、検討事項として記録されている
4. **暗黙要件の明示化**: 入力に明示されていないが、技術コンテキスト・ドメイン特性から必要と判断される検討事項が識別されている

**重要な限界**: 条件4は原理的に完全性を保証できない（「未知の未知」問題）。条件4の実用的な検証はStep 0bの分類枠組みに依存するため、仮定A-4（分類枠組みの網羅性）が崩れた場合、条件4の検証も同時に不完全になる。本方式はこの限界を認識した上で、構造的に検出率を最大化することを目標とする。「漏れゼロ」の保証ではなく、「漏れの発生確率の構造的低減」を目指す。

## 2. スコープと前提条件

### 2.1 入力の品質に関する前提

- **入力（requirements.md）は不完全を許容する**: 実際のプロジェクトでは要件が不完全・曖昧・矛盾を含むことが常態。方式は入力の不完全性に対してロバストである必要がある
- **ただし最低限の構造は前提**: 機能要件（SR）、非機能要件（NFR）、制約が区別可能な形式で記述されている。完全に非構造化されたテキスト入力は対象外
- **暗黙要件の存在を前提**: 明示されていないが重要な検討事項（バックアップ戦略、監視設計等）は常に存在するものとして扱う

### 2.2 対象の設計フェーズの責務範囲

| フェーズ | 責務 | 入力 | 出力 |
|---------|------|------|------|
| arch_design | システム全体のアーキテクチャ方針決定 | requirements.md, 技術コンテキスト | アーキテクチャ設計文書 |
| process_design | 開発・運用プロセスの設計 | requirements.md, arch_design出力 | プロセス設計文書 |

**フェーズ間の関係**: arch_designが先行し、process_designがその出力を入力として受ける。

**フィードバックパスの扱い**: 逆方向引き継ぎ（process→arch）が判明した場合の「検出メカニズム」の有無は方式の評価対象に含める。ただし、検出後の対応手順（arch_designの再実行、architecture.mdの更新等）は本議論のスコープ外とする。

### 2.3 LLMの実行環境の制約

| 制約 | 内容 | 方式設計への影響 |
|------|------|----------------|
| コンテキストウィンドウ | 1回の呼び出しで処理可能な情報量に上限がある | 大規模プロジェクトでは入力分割が必要 |
| サブエージェントの独立性 | サブエージェント間でコンテキストは共有されない | ファイル経由の情報共有が必要 |
| ファイル経由の情報共有 | サブエージェント間の情報受渡しはファイルI/Oに依存 | 中間成果物のフォーマット設計が重要 |
| **知識源の非独立性** | 同一LLMの複数呼び出しは、コンテキスト分離はできるが知識源は同一の訓練データに依存 | 「別サブエージェント=独立した知識源」ではない。系統的盲点は複数サブエージェントで共有される |

### 2.4 LLMの系統的特性

LLMは以下の系統的バイアスを持つ。これは方式設計の根本的な制約であり、プロンプト指示のみでは解消できない。

1. **訓練データ分布バイアス**: 訓練データに多く含まれる技術スタック（Web開発、クラウドインフラ）の検討事項は想起されやすく、ニッチなドメイン（組み込み、産業制御）は想起されにくい
2. **アンカリング効果**: 先に提示された情報が後続の推論に影響する。同一コンテキスト内で「先に生成した結果に依存せず再分析せよ」と指示しても、先の生成結果がアテンションに影響する
3. **アテンション配分バイアス**: 入力文書の記述量・強調度が推論結果に影響する。記述が多い領域の検討事項は多く生成され、記述が少ない領域は薄くなる
4. **同一盲点の共有**: 同一LLMの複数サブエージェントは同じ訓練データに基づくため、特定のドメインや概念に対する盲点を系統的に共有する。コンテキスト分離で確保できるのは「推論過程の独立性」であり、「知識源の独立性」ではない

## 3. 方式が満たすべき条件

### 3.1 検証可能性
- 漏れの有無を事後的に検証できる仕組みを方式自体が含む
- 「このカテゴリは検討済み」「このカテゴリは該当なしと判断した」が追跡可能
- 検証基準が自己参照的でない（検出対象と検出基準が循環しない）

### 3.2 適応性
- プロジェクト固有の検討事項（特定ドメイン、特定技術スタック）に対応できる
- 事前定義されたカテゴリに含まれない検討事項も識別できるメカニズムがある
- 入力の品質・量に応じてスケールする（5要件のプロジェクトでも50要件でも機能する）

### 3.3 実装可能性
- LLMスキル（Claude Codeのスキル定義）として実装可能
- サブエージェント呼び出し回数はフェーズあたり10-20回以内（コスト制約）
- ユーザーインタラクション回数はフェーズあたり3-5回以内（UX制約）

### 3.4 コスト許容範囲
- 検出力の向上とコスト（サブエージェント呼び出し数、処理時間、ユーザー待ち時間）のトレードオフを意識する
- 「すべてを手動確認する」方式は検出力は高いがコスト不許容
- 方式の各ステップについて、そのステップが追加する検出力とコストの関係が説明可能であること

## 4. 方式が依存する仮定の明示

以下の仮定は、提案される方式の実効性に影響する。議論の中でこれらの仮定の妥当性を検証する。

| ID | 仮定 | 妥当性の検証基準 | 崩れた場合の影響 |
|----|------|----------------|----------------|
| A-1 | LLMは要件文書から検討事項を「概ね妥当に」導出できる | シナリオ走査で標準的なケースの検出率を確認 | 方式全体の基盤が崩れる。導出層の品質が低い場合、検証層のみで網羅性を保証する必要がある |
| A-2 | コンテキスト分離により推論過程の独立性は確保できる | 別サブエージェント呼び出しで異なる視点からの分析が得られる | 多層方式の効果が大幅に低下。同一視点での重複分析になる |
| A-3a | ユーザーは提示された検討事項の妥当性（関連性・優先度）を判断できる | ユーザーインタラクションで妥当性フィードバックが得られる | 不要な検討事項が残存する（過剰適用のリスク増加）が、致命的ではない |
| A-3b | ユーザーは提示されていない検討事項の欠落に気づくことができる | ユーザーが未提示の検討事項を追加する場面がある | 方式の自動処理で検出できなかった漏れが、ユーザー確認でも検出されない。この場合の安全網は外部専門家レビュー以外に存在しない |
| A-4 | 分類枠組み（カテゴリ分類）は検討事項の主要部分をカバーする | 外部基準（ISO 25010等）をベースにすることで網羅性を確保 | 分類外の検討事項が系統的に見落とされる |
| A-5 | LLMは交差分析（要件間の相互作用の識別）を一定の精度で実行できる | シナリオ走査で主要な2要件の組み合わせ（性能×セキュリティ、セキュリティ×コスト等）から検討事項が導出されることを確認。3要件以上の組み合わせは検出困難であることを前提とする | 要件間相互作用に起因する検討事項が見落とされる。3要件以上の組み合わせは特に検出困難 |
| A-6 | ファイル経由の情報共有は十分な情報保持精度を持つ | 中間ファイルのフォーマット設計により情報劣化を制御可能 | サブエージェント間の情報劣化により、後段の処理品質が低下する |
