# 方式A: 動的導出方式

## 1. 方式概要

### 1.1 基本原理

動的導出方式は、事前定義のチェックリストに依存せず、**入力（requirements.md）を起点にLLMの推論能力で設計検討事項を導出する**方式である。

核となるテーゼ: **プロジェクト固有の要件文書こそが、検討事項の最良の出発点である。** チェックリストは汎用性と引き換えに、過剰検出（プロジェクトに無関係な項目）と検出漏れ（リストにないプロジェクト固有の事項）の双方のリスクを内包する。構造的に制約されたLLM推論は、プロジェクト固有の検討事項をより的確に導出できる。

### 1.2 チェックリスト方式との本質的な差異

| 観点 | チェックリスト方式 | 動的導出方式 |
|------|-----------------|------------|
| 検討事項の源泉 | 事前定義リスト（外部基準由来） | 要件文書からのLLM推論 |
| 網羅性の根拠 | リストの完全性 | 推論の構造的多様性 + トレーサビリティ |
| 適応性 | リストに含まれる範囲に限定 | プロジェクト固有の事項を導出可能 |
| 過剰検出リスク | 高い（汎用項目の全量提示） | 低い（要件に紐づく項目のみ） |
| 系統的見落とし | リスト外は原理的に検出不能 | LLMの訓練データ分布バイアスに依存 |

### 1.3 三層構造

本方式は以下の3層で構成される:

1. **導出層（Derivation Layer）**: 複数の独立サブエージェントが、異なる視点レンズを通じて要件から検討事項を推論的に導出する
2. **交差検証層（Cross-Validation Layer）**: 導出結果を統合し、要件トレーサビリティと要件間相互作用の分析により構造的な漏れを検出する
3. **敵対的検証層（Adversarial Probing Layer）**: シナリオベースの障害分析により、導出層・交差検証層が見逃した検討事項を探索する

## 2. arch_design / process_design への適用方針

### 2.1 arch_designへの適用

arch_designは「システム全体のアーキテクチャ方針決定」を責務とする。入力はrequirements.mdと技術コンテキストである。

#### Phase A1: 視点駆動型独立導出（導出層）

Step 0bの5視点をそれぞれ独立したサブエージェントに割り当て、**構造的に独立な**導出を並列実行する。

| サブエージェント | 視点レンズ | 導出指示の概要 |
|---------------|----------|--------------|
| SA-A1a | 技術インフラ | 要件から、システム構造・品質特性・技術基盤に関する設計検討事項を導出せよ |
| SA-A1b | ドメイン・ビジネスロジック | 要件から、ドメインモデル・ビジネスルール・状態管理に関する設計検討事項を導出せよ |
| SA-A1c | ユーザー体験 | 要件から、UX品質・レスポンス設計・アクセシビリティに関する設計検討事項を導出せよ |
| SA-A1d | 運用 | 要件から、監視・デプロイ・バックアップ・キャパシティに関する設計検討事項を導出せよ |
| SA-A1e | 規制・ガバナンス | 要件から、セキュリティ・データ保護・監査・コンプライアンスに関する設計検討事項を導出せよ |

**各サブエージェントへの入力**: requirements.md + 視点の定義（Step 0bの該当セクション概要）のみ。他サブエージェントの出力は一切含めない。

**各サブエージェントへの出力要件**:
- 検討事項ごとに: (1) 事項名、(2) 対応する要件ID（SR/NFR/制約）、(3) なぜこの要件からこの事項が導出されるかの推論根拠、(4) arch_design / process_design / 横断的 の分類
- **最低件数の設定なし**: 該当なしの視点は「該当なしと判断した根拠」を明記する（強制的な件数要件はアテンション配分バイアスの逆方向バイアスを生むため）

**設計判断の根拠**: 5つの独立サブエージェントにより**コンテキスト独立性**（Step 0d §1.4）を確保する。同一コンテキスト内での「独立に再分析せよ」という指示的独立性ではなく、物理的に別のコンテキストで推論を実行する。

#### Phase A2: 要件間相互作用分析（交差検証層 - Part 1）

Phase A1と**並列実行**する独立サブエージェント。

- **入力**: requirements.md のみ（Phase A1の出力は含めない → アンカリング防止）
- **タスク**: 要件間の相互作用（特にNFR間のトレードオフ）を識別し、相互作用から生じる設計検討事項を導出する
- **出力**: 相互作用ペア/トリプルのリストと、各相互作用から導出される検討事項

**設計判断の根拠**: 要件間の交差分析（Step 0 §1.3 条件3）は、個別視点からの導出とは異なる認知モードを要する。Phase A1の視点駆動型導出は「1つの要件→検討事項」のマッピングに強いが、「要件A×要件B→新たな検討事項」の発見には別の構造が必要。

#### Phase A3: 統合・構造化（交差検証層 - Part 2）

- **入力**: Phase A1（5サブエージェント出力）+ Phase A2出力
- **タスク**:
  1. 全出力の統合・重複排除
  2. 各検討事項に対する分類タグ付け: arch_design / process_design / 横断的
  3. **要件トレーサビリティ検証**: requirements.mdの全要件（SR/NFR/制約）に対して、対応する検討事項が1件以上存在するかを確認。対応なしの要件を**未カバー要件リスト**として出力
  4. **横断的関心事の境界仕様**: 横断的と分類された事項について、arch_designが決定する範囲とprocess_designに委譲する範囲の境界を明記
- **出力**: 統合検討事項リスト + 未カバー要件リスト + 横断的関心事境界仕様

#### Phase A4: シナリオベース敵対的探索（敵対的検証層）

- **入力**: requirements.md + Phase A3の統合検討事項リスト
- **タスク**: 「現在の検討事項リストが存在する状態で、このシステムが本番運用で障害を起こす現実的なシナリオを列挙せよ。各シナリオについて、どの検討事項の欠落がその障害を引き起こすかを特定せよ」
- **出力**: 障害シナリオリストと、各シナリオに対応する追加検討事項の提案

**設計判断の根拠**: 視点駆動型導出（Phase A1）と要件トレーサビリティ（Phase A3）は「存在する情報からの導出」に強いが、「存在しない情報（暗黙要件）」の発見には弱い。シナリオベースの障害分析は異なる認知モード（帰結→原因の逆推論）を使うため、導出層が見逃した暗黙要件を発見する可能性がある。

#### Phase A5: ユーザー検証

- **提示内容**: (1) 統合検討事項リスト（視点別に整理）、(2) 未カバー要件リスト、(3) 敵対的探索で発見された追加事項、(4) 横断的関心事の境界仕様
- **ユーザーに求める判断**: (a) 検討事項の妥当性確認、(b) 未カバー要件の対応要否判断、(c) ユーザーのドメイン知識に基づく追加事項の指摘

**サブエージェント呼び出し数**: SA-A1a〜A1e(5) + SA-A2(1) + SA-A3(1) + SA-A4(1) = **8回**（制約の10-20回以内）
**ユーザーインタラクション回数**: Phase A5の1回 + 必要に応じた確認1-2回 = **2-3回**（制約の3-5回以内）

### 2.2 process_designへの適用

process_designは「開発・運用プロセスの設計」を責務とする。入力はrequirements.md + arch_design出力である。

#### Phase P1: プロセス視点導出（導出層）

process_designに関連する視点を独立サブエージェントに割り当てる。arch_designと異なり、全5視点ではなくプロセスに関連性の高い視点に集中する。

| サブエージェント | 視点レンズ | 導出指示の概要 |
|---------------|----------|--------------|
| SA-P1a | 開発プロセス | 要件とアーキテクチャ方針から、テスト戦略・CI/CD・ブランチ戦略等の設計検討事項を導出せよ |
| SA-P1b | 運用プロセス | 要件とアーキテクチャ方針から、デプロイ・監視・障害対応等の運用手順に関する検討事項を導出せよ |
| SA-P1c | ガバナンスプロセス | 要件とアーキテクチャ方針から、監査対応・コンプライアンス手順・ドキュメント管理の検討事項を導出せよ |

**各サブエージェントへの入力**: requirements.md + arch_design出力（アーキテクチャ設計文書）+ 視点定義。他サブエージェントの出力は含めない。

#### Phase P2: アーキテクチャ整合性検証（交差検証層 - Part 1）

Phase P1と**並列実行**する2つの独立サブエージェント。

| サブエージェント | タスク |
|---------------|-------|
| SA-P2a（順方向検証） | arch_design出力の全設計決定に対して、process_designで対応が必要な手順・プロセスが識別されているか確認。未対応の設計決定を**引き継ぎ漏れリスト**として出力 |
| SA-P2b（逆方向検出） | process_designの検討中に「arch_designで決定されているべきだが未決定の事項」を識別。**arch差し戻し候補リスト**として出力 |

**設計判断の根拠**: Step 0 §1.2で定義された引き継ぎ漏れ（Handoff Gap）の双方向性に対応する。順方向（arch→process）と逆方向（process→arch）の検出を独立サブエージェントで実行する。

#### Phase P3: 統合・構造化（交差検証層 - Part 2）

Phase A3と同様の統合処理。追加で:
- arch_design出力との整合性確認
- 横断的関心事の境界仕様（Phase A3出力）との突合
- 未カバー要件リストの生成

#### Phase P4: 敵対的探索 + 逆方向ギャップ探索（敵対的検証層）

- Phase A4と同様のシナリオベース障害分析に加え、「arch_designの決定が不十分なためにprocess_designで手順化できない領域」の特定を行う

#### Phase P5: ユーザー検証

- Phase A5と同様。追加で、arch差し戻し候補リストの対応方針をユーザーに確認する

**サブエージェント呼び出し数**: SA-P1a〜P1c(3) + SA-P2a〜P2b(2) + SA-P3(1) + SA-P4(1) = **7回**
**ユーザーインタラクション回数**: Phase P5の1回 + 確認1-2回 = **2-3回**

## 3. 横断的関心事のarch/process境界の扱い

### 3.1 横断的関心事の発生と分類

横断的関心事（セキュリティ、ロギング、エラーハンドリング、国際化等）は、Phase A1の各視点サブエージェントから独立に導出される。Phase A3の統合時に、複数の視点から導出された同一の横断的関心事をマージし、以下の**境界仕様**を生成する。

### 3.2 境界仕様のフォーマット

各横断的関心事について:

```
[横断的関心事名]: セキュリティ
  arch_designの責務: 認証・認可モデルの選定、暗号化方針、セキュリティアーキテクチャ
  process_designの責務: セキュリティテスト戦略、脆弱性スキャン手順、インシデント対応手順
  境界判定基準: 「何を（WHAT）」はarch_design、「どのように運用するか（HOW to operate）」はprocess_design
  相互依存: process_designのセキュリティテスト戦略は、arch_designの認証モデル決定に依存する
```

### 3.3 境界仕様の検証タイミング

1. **arch_design Phase A3**: 横断的関心事を識別し、境界仕様を生成
2. **arch_design Phase A5**: ユーザーに境界仕様の妥当性を確認
3. **process_design Phase P2a**: 境界仕様に基づき、process_design側の責務が全て検討事項として識別されているか検証
4. **process_design Phase P2b**: 境界仕様の逆方向チェック（process側の検討でarch側の不足が判明した場合）

### 3.4 境界が曖昧なケースの扱い

arch/processの境界が明確に引けない場合（例: フィーチャーフラグの設計 — アーキテクチャ的側面とリリースプロセス的側面が混在）は、**両フェーズで重複して検討事項として記録**し、各フェーズでの検討範囲を明記する。重複を排除するのではなく、重複を管理する方針とする。

## 4. 網羅性の保証メカニズム

### 4.1 三層による構造的検出率の最大化

Step 0 §1.3の4条件に対して、各層がどのように寄与するかを示す。

| 網羅性条件 | 導出層（Phase A1/P1） | 交差検証層（Phase A2-A3/P2-P3） | 敵対的検証層（Phase A4/P4） |
|-----------|---------------------|-------------------------------|--------------------------|
| 1. カテゴリカバレッジ | 5視点の独立サブエージェントにより構造的にカバー | 統合時に5視点の出力有無を確認 | — |
| 2. 要件トレーサビリティ | 各導出で要件IDとの紐付けを必須化 | Phase A3/P3で全要件の対応確認 | — |
| 3. 交差分析完了 | — | Phase A2/P2の専用サブエージェント | シナリオ分析が要件間相互作用を間接的に検出 |
| 4. 暗黙要件の明示化 | 視点レンズが暗黙要件の想起を促進 | — | シナリオベース障害分析が暗黙要件の主要な検出手段 |

### 4.2 各層の検出対象と限界

#### 導出層の検出力と限界

**検出力**: 要件文書に記述がある領域の検討事項を、視点ごとに網羅的に導出する。視点レンズによるフレーミングが、アテンション配分バイアス（記述量の多寡に引きずられる）を軽減する。

**限界**:
- LLMの訓練データ分布バイアスにより、低頻度ドメインの検討事項は導出されにくい。5つの独立サブエージェントを使っても、知識源は同一であるため（Step 0 §2.3「知識源の非独立性」）、系統的盲点は全サブエージェントで共有される
- 視点レンズは推論の方向を制約するが、知識の有無は制約できない

#### 交差検証層の検出力と限界

**検出力**: 要件トレーサビリティにより「入力に明示された要件の見落とし」を構造的に検出する。要件間相互作用分析により、個別視点では発見できないトレードオフを検出する。

**限界**:
- 要件トレーサビリティは「要件文書に書かれている要件」のみを対象とする。暗黙要件はこの層では検出できない
- 要件間相互作用分析の品質はA-5（交差分析の精度）に依存する。3要件以上の複合的相互作用の検出は困難

#### 敵対的検証層の検出力と限界

**検出力**: シナリオベースの帰結→原因の逆推論により、導出層・交差検証層とは異なる認知モードで漏れを探索する。暗黙要件（バックアップ、災害復旧、エッジケース処理等）の検出に特に有効。

**限界**:
- 同一LLMの知識に依存するため、系統的盲点は解消されない
- シナリオの網羅性自体が保証されない（どのシナリオを想定するかもLLMの推論に依存）
- 導出層の結果がコンテキストに含まれるため、アンカリング効果で「導出済みの事項に関連するシナリオ」に偏る可能性がある

### 4.3 識別漏れ（Identification Gap）への対処構造

```
識別漏れの発生確率 = P(導出層で未検出) × P(交差検証層で未検出) × P(敵対的検証層で未検出) × P(ユーザー検証で未検出)
```

**注意**: 上記の確率は独立事象ではない。特に導出層と敵対的検証層は同一LLMの知識に依存するため、正の相関がある。独立性の仮定に基づく「多層で指数的に漏れが減少する」という議論は不正確であり、本方式はそのような主張を行わない。ただし、構造的に異なる認知モード（視点駆動、トレーサビリティ、シナリオベース）を組み合わせることで、単一の導出パスに比べて検出率は向上する。

## 5. カバレッジマップとの対応

### 5.1 5視点と方式の各要素の対応

| Step 0bの視点 | Phase A1サブエージェント | Phase A2での扱い | Phase A4での扱い |
|-------------|----------------------|----------------|----------------|
| 技術インフラ | SA-A1a（技術インフラ視点レンズ） | 性能×セキュリティ等のNFR間トレードオフを検出 | インフラ障害シナリオで暗黙要件を探索 |
| ドメイン・ビジネスロジック | SA-A1b（ドメイン視点レンズ） | ビジネスルール間の矛盾・相互作用を検出 | ビジネスロジック障害シナリオで境界条件を探索 |
| ユーザー体験 | SA-A1c（UX視点レンズ） | UX要件と性能要件の相互作用を検出 | ユーザー操作シナリオで暗黙的UX要件を探索 |
| 運用 | SA-A1d（運用視点レンズ） | 運用要件とコスト制約の相互作用を検出 | 運用障害シナリオでDay-2要件を探索 |
| 規制・ガバナンス | SA-A1e（規制視点レンズ） | 規制要件とアーキテクチャ制約の相互作用を検出 | コンプライアンス違反シナリオで規制要件の漏れを探索 |

### 5.2 process_designでの視点カバレッジ

process_designでは5視点を3つのプロセス視点（開発、運用、ガバナンス）に再マッピングする。

| Step 0bの視点 | process_designでの対応 |
|-------------|---------------------|
| 技術インフラ | Phase P1a（開発プロセス）: ビルド・テスト環境、CI/CDパイプライン |
| ドメイン・ビジネスロジック | Phase P1a（開発プロセス）: テスト戦略（ドメインロジックのテスト方針） |
| ユーザー体験 | Phase P1a（開発プロセス）: UXテスト手順、アクセシビリティテスト |
| 運用 | Phase P1b（運用プロセス）: デプロイ手順、監視運用、障害対応 |
| 規制・ガバナンス | Phase P1c（ガバナンスプロセス）: 監査手順、コンプライアンス確認プロセス |

### 5.3 カバレッジギャップの明示的管理

Phase A3/P3の統合時に、5視点×導出結果のマトリクスを生成し、検討事項が0件の視点を**明示的にフラグ**する。ただし、これは「チェックリスト化」ではなく、「導出結果の偏り検出」として機能する。0件の視点について:
- 「該当なし」の根拠が明記されていれば: 許容
- 根拠がなければ: 導出の不足としてPhase A4/P4の重点探索対象に追加

## 6. この方式が依存する仮定

### 6.1 仮定への依存度

| 仮定ID | 依存度 | 根拠 |
|-------|-------|------|
| A-1（LLMの導出能力） | **極めて高い** | 本方式の全基盤。チェックリストによる補完がないため、A-1が崩れると検出率が直接低下する |
| A-2（コンテキスト分離の独立性） | **高い** | 導出層の5サブエージェント独立性、敵対的検証層の異なる認知モード、いずれもA-2に依存する |
| A-3a（ユーザーの妥当性判断） | 中 | Phase A5/P5のユーザー検証で利用 |
| A-3b（ユーザーの欠落検出） | **高い** | 系統的盲点の最終安全網としてユーザーのドメイン知識に依存する。LLM側の多層構造では系統的盲点を解消できない（知識源の非独立性）ため、ユーザー検証が唯一の知識源独立な検出手段 |
| A-4（分類枠組みの網羅性） | **低い** | チェックリスト方式と異なり、分類枠組みは導出の「ガイドレール」（視点レンズ）としてのみ使用。枠組み外の事項もLLM推論で導出可能 |
| A-5（交差分析精度） | 中 | Phase A2の要件間相互作用分析に依存 |
| A-6（ファイル経由の情報保持） | 中 | Phase A3の統合処理での情報保持に依存 |

### 6.2 A-4への依存が低い点の意味

動的導出方式はA-4（分類枠組みの網羅性）への依存が構造的に低い。これは本方式の主要な利点の1つである。

- チェックリスト方式: A-4が崩れると、分類外の検討事項が**構造的に検出不能**になる
- 動的導出方式: A-4が崩れても、LLM推論で分類外の事項を導出する可能性が残る。視点レンズは推論の方向を示すが、推論結果を制約しない

ただし、A-1（LLMの導出能力）への依存が高まるトレードオフがある。分類枠組みへの依存をLLMの推論能力への依存に置換している構造である。

## 8. 想定される課題とその対策

### 課題1: LLMの知識限界による系統的盲点（最大のリスク）

**問題**: 動的導出方式はチェックリストという外部的なアンカーを持たないため、LLMの訓練データに含まれない領域の検討事項は、多層構造を通じても検出できない。

**影響度**: 高。低頻度ドメイン（特定業界規制、ニッチな技術スタック）で系統的な漏れが発生する。

**対策**:
- **第一防衛線**: 5視点の独立導出（Phase A1）で、低頻度ドメインにも強制的にアテンションを配分
- **第二防衛線**: 敵対的検証（Phase A4）で、シナリオベースの障害分析を通じた暗黙要件の検出
- **最終防衛線**: ユーザー検証（Phase A5）で、知識源独立な検出を行う
- **残存リスクの明示**: 本方式は「LLMが知らない検討事項」の完全検出を保証しない。これは方式の原理的限界であり、ユーザーに明示する

### 課題2: 出力の再現性・安定性

**問題**: LLMの確率的生成に依存するため、同一入力でも実行ごとに導出される検討事項が異なる可能性がある。

**影響度**: 中。重要な検討事項が特定の実行では導出され、別の実行では導出されないリスク。

**対策**:
- 視点レンズと出力形式の構造化により、出力のバリエーションを一定範囲に制約する
- 要件トレーサビリティ（Phase A3）が「入力に明示された要件の対応漏れ」を構造的に検出するため、最低限のカバレッジは安定する
- 暗黙要件の導出は本質的に確率的であることを認め、敵対的検証層で補完する

### 課題3: サブエージェント間の重複と統合コスト

**問題**: 5視点の独立導出は、視点間の重複が発生する（例: セキュリティは技術インフラ視点と規制視点の双方から導出される）。

**影響度**: 低。重複は漏れよりもリスクが低い（Step 0 §1.2「過剰検出の扱い」）。

**対策**:
- Phase A3の統合・重複排除で処理
- 重複の存在は「複数視点が同一事項を重要と認識した」ことの証拠であり、その事項の重要度判定にも利用できる

### 課題4: Phase A4（敵対的検証）のアンカリング

**問題**: Phase A4は導出層の結果をコンテキストに含むため、アンカリング効果で「導出済みの事項に関連する障害シナリオ」に偏る可能性がある。

**影響度**: 中。敵対的検証層が導出層の補完として十分に機能しないリスク。

**対策**:
- Phase A4の指示で「導出済みリストに含まれない領域での障害」に焦点を当てるよう明示的にフレーミングする
- ただし、この指示的対策の実効性は限定的（Step 0d §1.1）であることを認識する。構造的対策としては、Phase A4の入力からPhase A1出力を除外する設計も考えられるが、その場合「現在の検討事項リストで不足する事項」の特定が不可能になるトレードオフがある
- 本方式ではこのトレードオフを「敵対的検証に導出結果の知識が必要」と判断し、アンカリングリスクを受容する設計判断とする

### 課題5: コスト効率

**問題**: arch_designで8回、process_designで7回のサブエージェント呼び出しは、単純な導出（1回）に比べて高コストである。

**影響度**: 低（制約の10-20回以内に収まっている）。

**対策**:
- Phase A1の5サブエージェントとPhase A2は並列実行可能であり、実時間コストは直列実行の1/6に近い
- Phase P1の3サブエージェントとPhase P2も並列実行可能
- 各層が追加する検出力: 導出層（基本的な検討事項の導出）→ 交差検証層（トレーサビリティと要件間相互作用）→ 敵対的検証層（暗黙要件とシナリオベースの漏れ検出）。各層の役割が明確に分離されている

## 9. まとめ: 動的導出方式の強みと適用条件

### 強み

1. **高い適応性**: プロジェクト固有の検討事項を、要件から直接導出する。チェックリストにない事項も識別可能
2. **低い過剰検出率**: 要件に紐づかない汎用的な事項が大量に提示されることがない
3. **A-4（分類枠組みの網羅性）への低依存**: 分類枠組みは「ガイドレール」であり、枠組み外の事項も導出可能
4. **構造的なバイアス対策**: 視点分割・サブエージェント分離・異なる認知モードの組み合わせにより、単一パスの導出に比べて系統的な偏りを軽減

### 適用条件（この方式が特に有効なケース）

- 標準的なドメインのプロジェクト（Web、クラウド等）で、LLMの訓練データが十分に存在する領域
- 要件文書の品質が一定以上あり、導出の起点として機能するケース
- プロジェクト固有の検討事項が多く、汎用チェックリストでは網羅しきれないケース

### 適用に注意が必要なケース

- ニッチなドメイン（特定業界規制、組み込み、産業制御等）で、LLMの知識が薄い場合 → ユーザー検証の比重が増す
- 要件文書が極めて簡素な場合 → 導出の起点が弱く、暗黙要件の割合が大きくなるため、敵対的検証層の負荷が増大
