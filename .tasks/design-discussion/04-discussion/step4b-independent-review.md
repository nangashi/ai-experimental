### 指摘1
- **該当箇所**: §1.2 データフロー概要（行59-89）と §2 Phase 0.5-3a 処理定義（行227-230）
- **指摘内容**: データフロー図と処理定義でPhase 0.5-2bの出力先が矛盾している。データフロー図ではPhase 0.5-2bの出力がPhase 0.5-3b（リファレンス照合）に直接接続されているが、Phase 0.5-3aの処理定義では入力にPhase 0.5-2bが含まれている（行230: 「4. Phase 0.5-2bの出力」）。一方、Phase 0.5-3bの入力定義（行281-282）にはPhase 0.5-2bは含まれていない。
- **重要度**: 高
- **根拠**: データフロー図の矢印構造（行63-70）を走査すると、Phase 0.5-1とPhase 0.5-2aの出力はPhase 0.5-3aに合流するが、Phase 0.5-2bの出力矢印はPhase 0.5-3aを経由せずPhase 0.5-3bに直接向かっている。処理定義ではPhase 0.5-3aの入力4番目にPhase 0.5-2bが明記されており、図と定義が一致しない。実装時にどちらに従うべきか不明となり、誤実装のリスクが生じる。
- **対応案**: 処理定義（Phase 0.5-2bをPhase 0.5-3aの入力とする）が設計意図として正しいと推定される（統合・重複排除をPhase 0.5-3aで一括実行する方が合理的）。データフロー図を修正し、Phase 0.5-2bからPhase 0.5-3aへの矢印に変更する。

### 指摘2
- **該当箇所**: Phase 0.5-3c（行325-358）、Phase 0.5-5 Part 1（行421-423）、§8 制約テーブル L-5（行940）
- **指摘内容**: Phase 0.5-3cの補完導出結果（`phase-0.5-3c-supplement-{n}.md`）を既存の統合結果（Phase 0.5-3aの出力）に統合し、重複排除する処理フェーズが未定義。Phase 0.5-5の Part 1 では「Phase 0.5-3a/3cで統合・補完された検討事項リスト」（行421）と記述しているが、この「統合」を実行する処理主体と手順がどのフェーズにも定義されていない。また、§8 の L-5 緩和策「Phase 0.5-5の最終提示で重複排除」（行940）とあるが、Phase 0.5-5はユーザーインタラクションフェーズであり、重複排除の処理ステップは含まれていない。
- **重要度**: 高
- **根拠**: Phase 0.5-3aはPhase 0.5-3cの前に実行が完了する（Phase 0.5-3c はPhase 0.5-3bの後に条件付き実行）。したがって、Phase 0.5-3aの出力にPhase 0.5-3cの結果を後から統合するには、明示的な統合フェーズが必要。これが未定義のため、(a) 補完された検討事項が最終出力に正しく反映されない可能性、(b) L-5の緩和策が実効性を持たない可能性がある。
- **対応案**: 以下のいずれかを採用する。(A) Phase 0.5-3c完了後に「Phase 0.5-3d: 補完統合」フェーズを追加し、Phase 0.5-3aの出力にPhase 0.5-3cの結果を統合・重複排除するサブエージェント（1回）を定義する。サブエージェント数は合計11-13に増加するが制約内。(B) Phase 0.5-5の処理定義に、ユーザー提示前の前処理として統合・重複排除ステップを明記する（親プロセスが実行）。いずれの場合もL-5の緩和策の実装箇所を更新する。

### 指摘4
- **該当箇所**: §3.1 arch_design用リファレンスカテゴリ全体（行550-743）、Step 0b §1.7 コスト・経済性（step0b行75-81）
- **指摘内容**: Step 0b §1.7で定義されたコスト・経済性の4領域（インフラコスト、運用コスト、開発コスト、ライセンス）に対して、リファレンスカテゴリの対応が不十分。Step 0b §2のカバレッジマップでは§1.7を「運用」視点にマッピングしているが、arch_design用リファレンスカテゴリで直接対応するのはOP-5「キャパシティプランニング」のサブ項目「コスト最適化戦略」のみ。「開発コスト」（技術的負債の許容範囲設計、ライブラリ選定のTCO）、「ライセンス」（OSS利用方針、商用ライセンス影響）は明示的なカテゴリ・サブ項目として定義されていない。
- **重要度**: 中
- **根拠**: Step 0b §2のカバレッジマップでは1.7全体が「運用」視点にマッピングされているが、OP-1〜OP-6のサブ項目を走査すると、「運用コスト」「開発コスト」「ライセンス」に直接対応するサブ項目が存在しない。Phase 0.5-3bのリファレンス照合時にこれらの領域がチェックされず、カバレッジギャップが検出されないリスクがある。ただし、Phase 0.5-1の動的導出で「コスト」関連の検討事項が導出される可能性はあり、完全な欠落ではなく検証層での検出漏れリスク。
- **対応案**: 以下のいずれかを採用する。(A) OP-5のサブ項目に「開発コスト（技術的負債、ライブラリTCO）」「ライセンスコンプライアンス」を追加する。(B) 新カテゴリ「OP-7: コスト設計」を追加し、Step 0b §1.7の4領域をサブ項目として定義する。(A)が最小変更で推奨。

### 指摘7
- **該当箇所**: §3 process_design Phase 0.5-3(p)（行504-529）、§8 L-6（行941）
- **指摘内容**: Phase 0.5-3(p)の軽量敵対的検証は、同一サブエージェント・同一コンテキスト内で統合・照合処理と敵対的検証を実行する。§8 L-6でアンカリングリスクの残存を認識しているが、緩和根拠が「3シナリオの軽量な検証に限定し影響を最小化」（行529）とされている。しかし、Step 0d §1.1の知見に基づけば、アンカリング効果はシナリオ数の多寡ではなく、先行コンテキストの存在自体に起因する。シナリオ数を制限しても、生成される3シナリオ自体が先行する統合結果にアンカリングされるため、検出の質への影響は件数では緩和されない。
- **重要度**: 中
- **根拠**: Step 0d §1.1: 「同一のコンテキストウィンドウ内で『先の結果に依存せず独立に分析せよ』と指示しても、アテンション機構がすでに先の結果をエンコードしているため、推論分布が先の結果に偏る」。Phase 0.5-3(p)では統合・照合の処理結果がコンテキストに蓄積された状態で敵対的検証を実行するため、3シナリオ全てが統合結果に引きずられる構造。コスト制約による設計判断であることは理解するが、緩和根拠の説明が不十分。
- **対応案**: L-6の緩和策の記述を修正し、「シナリオ数の制限によるアンカリング軽減」ではなく、「コスト制約によるトレードオフとして受容。アンカリングによる検出力低下はarch_designの独立Phase 0.5-4で部分的に補完される（arch_designの敵対的検証がprocess_designの検討事項もカバーしうる場合）」等、受容の正確な根拠を記載する。

### 指摘8
- **該当箇所**: Phase 0.5-4 入力定義（行369-373）、Phase 0.5-3c（行325-358）
- **指摘内容**: Phase 0.5-4（敵対的検証）に渡される「トレーサビリティギャップ情報」はPhase 0.5-3bの出力であり、Phase 0.5-3cの補完導出結果は反映されていない。Phase 0.5-3cで不足カテゴリが補完された場合でも、Phase 0.5-4は補完前のギャップ情報に基づいて「注力領域」を設定する。結果として、既に補完済みの領域に敵対的検証のアテンションが向けられ、真に不足している領域への注力が相対的に低下する。
- **重要度**: 低
- **根拠**: Phase 0.5-4の設計意図は「帰結→原因」の逆推論による「未知の未知」の探索であり、Phase 0.5-3cの補完（リファレンスカテゴリのサブ項目に基づく検討事項の導出）とは検出メカニズムが異なる。補完済み領域でもシナリオベースの逆推論で新たな検討事項が見つかる可能性があるため、重複は限定的。ただし、「注力領域」の指示が補完前の情報に基づくことで、Phase 0.5-3cで十分にカバーされた領域に不要な重点配分が生じる非効率のリスクは残存する。
- **対応案**: Phase 0.5-3cが実行された場合、Phase 0.5-4の「注力領域」からPhase 0.5-3cで補完されたカテゴリ名を除外する（または「補完済み」のラベルを付与する）。実装コストが低い修正のため、対応推奨。

### 指摘9
- **該当箇所**: Phase 0.5-3b 処理内容（行286-291）
- **指摘内容**: Phase 0.5-3bのサブ項目ごとの該当性判定（該当/非該当/該当・識別済み/該当・未識別）は、LLMの推論に依存する。Step 0d §1.2の訓練データ分布バイアスにより、低頻度ドメインのサブ項目（例: 組み込みシステム固有の要件、特定業界規制）がLLMにより「非該当」と誤判定されるリスクがある。非該当の理由明記は求められている（行290）が、理由の妥当性検証はPhase 0.5-5のユーザーに委ねられている。
- **重要度**: 低
- **根拠**: Phase 0.5-5でユーザーに「非該当と判定されたカテゴリの判断は妥当か」（行444）を確認する設計があるため、最終的な安全網は存在する。ただし、ユーザーがドメイン専門知識を持たない場合（仮定A-3b: 「ユーザーは提示されていない検討事項の欠落に気づくことができる」の崩壊ケース）、LLMの誤判定がそのまま通過する。この組み合わせリスクは§6.2の仮定依存度マトリクスで明示されていない。
- **対応案**: §6.2のA-3bの「崩壊時の安全網」欄に「ただし、Phase 0.5-3bの非該当判定に訓練データ分布バイアスが作用した場合、A-3bの崩壊と複合して低頻度ドメインの検討事項が系統的に欠落するリスクがある」旨を追記する。

### 観点別の補足（問題が検出されなかった領域）

- **Phase 0.5-1のカテゴリ名ガイド**: カテゴリ名のみの注入（サブ項目なし）+ 「限定せず」指示の組み合わせは、アンカリング抑制とカバレッジ向上のバランスとして妥当。Step 0d §1.1の教訓は同一コンテキスト内の先行生成結果への依存についてであり、入力として与えられたカテゴリ名による知識活性化とは性質が異なる。
- **Phase 0.5-3bの該当性判定**: 50%閾値からの改善として、サブ項目ごとの4値判定は明確な前進。理由明記+ユーザー検証の二重安全網あり。
- **Phase 0.5-4の入力制限**: 要約+ギャップ情報への制限は、アンカリング軽減と検出力維持の両立として合理的。
- **5視点のカバレッジ**: §7のカバレッジマップ対応テーブルで全5視点が導出層・検証層・交差分析・敵対的検証の全てでカバーされている。
- **コスト分析の数値整合性**: arch 10-12、process 6の計算は各フェーズの定義と一致。
