# Step 4c: 方式限界分析

## 分析1: 安全網マッピング

Step 0b §2のカバレッジマップの各領域について、本方式の安全網（導出層 + 検証層 + 敵対的検証 + ユーザー検証）の「内側」と「外側」をマッピングする。

### 内側の領域

| 領域 | 検出メカニズム | 確度 |
|------|-------------|------|
| **§1.1 ISO 25010 品質特性（性能効率, 信頼性, セキュリティ, 保守性）** | 導出層: TI-3/TI-4/RG-5/TI-7のカテゴリ名ガイドが想起を促進。検証層: 対応サブ項目で照合。高頻度ドメインのためLLMの訓練データ分布とも合致 | 高 |
| **§1.1 ISO 25010 品質特性（互換性, 移植性）** | 導出層: TI-8のカテゴリ名ガイド。検証層: TI-8サブ項目で照合 | 中〜高。ただしプロジェクトの技術スタックが高頻度ドメイン（Web/クラウド）の場合に限る |
| **§1.1 ISO 25010 品質特性（使用性）** | 導出層: UX-1〜UX-5のカテゴリ名ガイド。検証層: UXサブ項目で照合 | 中〜高 |
| **§1.1 ISO 25010 品質特性（機能適合性）** | 導出層: DM-1〜DM-6のカテゴリ名ガイド。要件トレーサビリティマトリクス（Phase 0.5-3a）が機能要件の未カバーを検出 | 高。機能適合性は要件書に明示されることが多く、トレーサビリティ照合の効果が大きい |
| **§1.2 アーキテクチャビューポイント（4+1 View: 論理/開発/プロセス/物理ビュー）** | 導出層: 5視点が4+1 Viewの主要ビューをカバー。検証層: TI/DM/OPカテゴリのサブ項目が各ビューの典型的検討事項を含む | 中〜高。ビューポイントは直接的なカテゴリではないが、サブ項目が実質的にカバー |
| **§1.2 アーキテクチャビューポイント（C4: コンテキスト/コンテナ）** | 導出層: 技術インフラ視点とドメイン視点で外部境界・実行単位の分割を導出。検証層: TI-1, DM-6のサブ項目 | 中〜高 |
| **§1.3 DDD戦略/戦術設計** | 導出層: DM-1〜DM-4のカテゴリ名ガイドがDDD概念の想起を促進。検証層: DM-1サブ項目（バウンデッドコンテキスト、集約境界等） | 中。高頻度ドメイン（Web開発）ではDDD概念はLLMに馴染みがあるが、DDDを適用しないプロジェクトでは過剰検出側のリスク |
| **§1.5 運用・進化（監視, バックアップ, デプロイ, マイグレーション）** | 導出層: OP-1〜OP-6のカテゴリ名ガイド。検証層: OPサブ項目で照合。敵対的検証: 障害シナリオから運用系の漏れが逆推論されやすい | 高。障害シナリオとの親和性が高い |
| **§1.6 規制・コンプライアンス（データ保護, 監査要件）** | 導出層: RG-1〜RG-5のカテゴリ名ガイド。検証層: RGサブ項目で照合 | 中。一般的な規制（GDPR, PCI DSS）はカバーされるが、業界固有規制は§1.2の訓練データ分布バイアスの影響を受ける |
| **§1.8 開発プロセス（テスト戦略, CI/CD, ブランチ戦略）** | 導出層: PD-1〜PD-4のカテゴリ名ガイド（process_design）。検証層: PDサブ項目で照合 | 高。開発プロセスはLLMの訓練データに豊富 |
| **§1.2 シナリオ（4+1 View）をメタ次元として** | 敵対的検証: Phase 0.5-4が実質的にシナリオベースの検証を実施。障害シナリオが品質属性のシナリオ検証に近い機能を果たす | 中。ただし敵対的検証は「障害」シナリオに偏り、正常系の品質属性シナリオ（例: 性能が目標を満たすシナリオ）は対象外 |
| **2要件間の標準的交差（性能×セキュリティ等）** | 交差分析: Phase 0.5-2aのテンプレート（10パターン）が構造的にカバー。Phase 0.5-2bの自由推論が非標準パターンを補完 | 高。テンプレートの10パターンは主要なNFR間トレードオフを網羅 |

### 外側の領域

| 領域 | 非検出の理由 | 影響評価 |
|------|------------|---------|
| **§1.7 コスト・経済性（インフラコスト, 運用コスト, 開発コスト, ライセンス）** | リファレンスカテゴリに「コスト・経済性」の独立カテゴリが存在しない。OP-5（キャパシティプランニング）にコスト最適化が含まれるが、TCO分析、ライセンスコンプライアンス、開発コストのトレードオフ等は構造的にカバーされない。導出層で暗黙要件として検出される可能性はあるが、カテゴリ名ガイドがないため想起が弱い | **中〜高**。コスト設計はアーキテクチャ決定に直結する（例: マネージドサービス vs 自前構築の判断）。カバレッジマップの§1.7がリファレンスカテゴリに十分にマッピングされていない |
| **3要件以上の複合的交差** | Step 0 A-5仮定で「3要件以上の組み合わせは検出困難」と明記。Phase 0.5-2bの自由推論が部分的にカバーするが構造的保証はない。例: 「性能要件 × セキュリティ要件 × コスト制約」の三者間トレードオフ | **中**。重要な複合交差が存在するプロジェクトでは致命的だが、多くのプロジェクトでは2要件交差で主要なトレードオフはカバーされる |
| **極めてドメイン固有な暗黙要件** | LLMの訓練データに含まれないドメイン固有の規制・慣行（例: 特定国の金融規制の技術的要件、医療機器の特定規格）。リファレンスカテゴリRG-4に「業界固有コンプライアンス」があるが、サブ項目は一般的な例（PCI DSS, HIPAA, SOX）のみで、ニッチな規制はカバー外 | **プロジェクト依存**。該当するプロジェクトでは高影響だが、一般的なWebシステムでは低影響 |
| **§1.2 コンポーネントビュー（C4の最深層）** | リファレンスカテゴリにコンポーネントレベル（内部構造設計、インターフェース設計の詳細）の独立カテゴリがない。TI-7（保守性設計）のサブ項目に部分的に含まれるが、設計検討事項の導出フェーズ（arch_design）のスコープが方針レベルであるため、コンポーネント詳細は意図的にスコープ外 | **低**。arch_designの責務範囲（方針レベル）を考慮すると、コンポーネント詳細がスコープ外であること自体は適切。ただし、方針レベルの決定がコンポーネント設計を制約するケースの識別は不完全 |
| **LLMの訓練データに存在しない新技術・新規制** | 訓練データのカットオフ以降に登場した技術（新しいクラウドサービス、新しいフレームワーク）や規制は、導出層・検証層のいずれでも検出不可能。カテゴリ名ガイドは既知のカテゴリに基づくため、新カテゴリの想起を促進できない | **低〜中**。requirements.mdにユーザーが明示的に記述している場合はトレーサビリティで検出可能。未記述の場合は安全網の完全外 |
| **process→arch逆流の根本原因** | Phase 0.5-2b(p)で逆流は「検出」されるが、arch_designで欠落した検討事項の「根本原因」（なぜ導出層で漏れたか）は分析されない。検出後の対応手順はスコープ外（Step 0 §2.2）と明記されている | **中**。逆流の検出自体は安全網内だが、同種の逆流が再発する構造的原因の解消は安全網外 |
| **正常系シナリオでの品質属性ギャップ** | 敵対的検証（Phase 0.5-4）は障害シナリオに特化。「システムが正常に動作しているが、非機能要件を満たしていない」シナリオ（例: 応答時間が目標をわずかに超過、同時接続数がピーク時に上限付近）は検出対象に含まれない | **低〜中**。これらは検討深度の不足（Depth Gap）に分類される問題であり、識別漏れ（Identification Gap）よりは影響が小さい |

## 分析2: 仮定崩壊のカスケード

### A-1崩壊時: LLM推論力が期待を下回る

**崩壊条件**: LLMがrequirements.mdから検討事項を妥当に導出できない（質が著しく低い、または重要カテゴリを系統的に見落とす）。

**カスケード**:
1. **Phase 0.5-1（導出層）が直接劣化**: 5視点の各サブエージェントが浅い・不完全な検討事項を生成。カテゴリ名ガイドがあっても、推論力が低い場合はガイドのカテゴリ名を列挙するだけの表面的出力になる
2. **Phase 0.5-2a/2b（交差分析）も劣化**: 交差分析はLLMの推論に依存するため同時に劣化。テンプレートパターンとのマッチングは構造的に機能するが、マッチング結果の質（どのような検討事項が導出されるか）が劣化
3. **Phase 0.5-3a（統合）は入力品質に依存**: 入力（Phase 0.5-1/2a/2b）が劣化しているため、統合結果も劣化。ただし統合処理自体（重複排除、構造化）は入力品質に対して比較的ロバスト
4. **Phase 0.5-3b（検証層）が安全網として部分機能**: リファレンスカテゴリのサブ項目との照合自体は構造的処理のため、LLM推論力への依存度は導出層より低い。「該当・未識別」の検出はサブ項目リストとの突合で可能。**ただし**、照合判定（「該当」か「非該当」か）自体にLLM推論が必要なため、推論力崩壊時は「非該当」と誤判定して不足を見逃すリスクがある
5. **Phase 0.5-3c（補完導出）は導出層と同じ限界**: A-1崩壊時は補完導出の品質も低い
6. **Phase 0.5-4（敵対的検証）は部分機能**: 障害シナリオの生成自体はLLM推論に依存するが、「帰結→原因」の逆推論モードは「原因→帰結」とは異なる認知経路であるため、導出層と完全に同じ失敗パターンにはならない。しかし推論力全体が低い場合、この差異は限定的

**安全網の機能段階**: 2段階まで（検証層の構造的照合 + ユーザー検証）。ただし検証層の判定品質も劣化するため、実効的にはユーザー検証（Phase 0.5-5）が唯一の信頼できる安全網。

**ユーザーに到達する問題**: 検討事項リストが浅く不完全。カバレッジ照合結果の「非該当」判定の信頼性が低い。ユーザーは提示された検討事項の質の低さからA-1崩壊に気づく可能性がある（間接的な検出）が、**提示されなかった**検討事項の欠落には気づけない（A-3bの限界）。

### A-2崩壊時: コンテキスト分離が推論過程の独立性を確保できない

**崩壊条件**: 別サブエージェント呼び出しでも、何らかの理由で推論過程が相関する（例: システムプロンプトの暗黙的影響、モデルの内部状態の持続）。

**カスケード**:
1. **Phase 0.5-1（5視点並列）の独立性喪失**: 5つのサブエージェントが類似した検討事項を生成。視点分離の効果が失われ、実質的に単一視点での1回の導出と同等になる
2. **Phase 0.5-2b（自由推論交差）がPhase 0.5-2a（テンプレート交差）と重複**: 入力分離の効果が失われるため、テンプレートパターン外の交差を検出する目的が達成されない
3. **Phase 0.5-4（敵対的検証）が導出層と同じ結果**: 逆推論モードの効果が失われ、「原因→帰結」と「帰結→原因」が同じ出力を生成

**安全網の機能段階**: 1段階（ユーザー検証のみ）。検証層はコンテキスト独立性に直接依存しない（リファレンスカテゴリとの構造的照合）ため、検証層は機能する。ただし導出層の入力品質が劣化するため、間接的に影響を受ける。

**ユーザーに到達する問題**: 5視点分離による多角的な導出の効果が消失し、特定視点に偏った検討事項リストが生成される。ユーザーは偏りに気づきにくい（提示されていない視点の不足は自覚しにくい）。

**補足**: 現在のClaude Codeのサブエージェント実装では、各サブエージェントは独立したAPI呼び出しであり、コンテキスト分離は構造的に確保される。この仮定の崩壊リスクは実装レベルでは低い。

---

### A-3a崩壊時: ユーザーが検討事項の妥当性を判断できない

**崩壊条件**: ユーザーが技術的背景を持たない、またはプロジェクトのドメインに不慣れ。

**カスケード**:
1. **Phase 0.5-5（ユーザー検証）のフィルタリング機能喪失**: 不要な検討事項（過剰検出）が除去されずに最終リストに残存
2. **過剰検出の蓄積**: 後続の設計フェーズで不要な検討事項の検討に時間が費やされる

**安全網の機能段階**: 他の全段階が正常に機能。A-3aの崩壊は過剰検出のフィルタリングのみに影響し、漏れの検出には影響しない。

**ユーザーに到達する問題**: 過剰な検討事項リスト。コスト面での非効率性は増すが、漏れのリスクは増加しない。Step 0で「致命的ではない」と評価されている通り。

---

### A-3b崩壊時: ユーザーが欠落に気づけない

**崩壊条件**: ユーザーが自身の専門外の欠落を検出できない（最も現実的な崩壊シナリオ）。

**カスケード**:
1. **Phase 0.5-5（ユーザー検証）の最終安全網が機能しない**: 導出層+検証層+敵対的検証で検出されなかった漏れが、ユーザー検証でも検出されずに最終リストに残存
2. **漏れがそのまま後続フェーズに伝播**: 設計検討事項として認識されないまま、実装・運用フェーズに到達

**安全網の機能段階**: 3段階（導出層 + 検証層 + 敵対的検証）まで。ユーザー検証は最終安全網として設計されているが、A-3b崩壊時はこの段階が無効化される。

**ユーザーに到達する問題**: 「提示されていない検討事項の欠落」がそのまま残存。Step 0 §4で「この場合の安全網は外部専門家レビュー以外に存在しない」と明記されている通り、方式の構造内では対処不可能。

**影響の範囲**: A-3bの崩壊が影響するのは「導出層+検証層+敵対的検証のすべてで検出されなかった漏れ」のみ。すなわち、リファレンスカテゴリ外かつ動的導出でも生成されず、障害シナリオからも逆推論されない領域のみ。この領域は分析1の「外側の領域」と一致する。

---

### A-4崩壊時: 分類枠組みの網羅性が不足

**崩壊条件**: リファレンスカテゴリ（§3）が重要な設計検討事項の領域をカバーしていない。

**カスケード**:
1. **Phase 0.5-1（導出層）は部分的に影響**: カテゴリ名ガイドが不完全なため、ガイドによる想起促進が機能しない領域が発生。ただし「カテゴリに限定せず」の指示があるため、動的導出はカテゴリ外もカバー可能
2. **Phase 0.5-3b（検証層）が直接劣化**: リファレンスカテゴリとの照合でカバレッジギャップを検出する設計のため、カテゴリ自体が不完全ならギャップ検出が不完全になる。**具体的には**: リファレンスカテゴリに存在しない領域の「未識別」は検出されない
3. **Phase 0.5-3c（補完導出）がトリガーされない**: Phase 0.5-3bが不足を検出しないため、補完導出が実行されない
4. **Phase 0.5-4（敵対的検証）は独立して機能**: 障害シナリオベースの逆推論はリファレンスカテゴリに依存しない。A-4崩壊時でもカテゴリ外の漏れを検出する可能性がある

**安全網の機能段階**: 2段階（敵対的検証 + ユーザー検証）。導出層のオープンエンド導出もカテゴリ外を部分的にカバー。

**ユーザーに到達する問題**: リファレンスカテゴリの欠落領域の検討事項が系統的に不足。カバレッジ照合結果が「全カテゴリ照合完了」と表示されるが、それが網羅性を意味しない。ユーザーはカバレッジ照合の「全カテゴリ照合完了」を過信するリスクがある。

**Step 0 §1.3との関係**: 「条件4の実用的な検証はStep 0bの分類枠組みに依存するため、仮定A-4が崩れた場合、条件4の検証も同時に不完全になる」と明記されている。この循環が本方式の根本的限界。

---

### A-5崩壊時: 交差分析能力が不足

**崩壊条件**: LLMが要件間の相互作用を識別できない（2要件間でも検出率が低い）。

**カスケード**:
1. **Phase 0.5-2a（テンプレート交差）は部分的に維持**: テンプレートパターンとの構造的マッチングは推論力に部分的に依存するが、パターンが明示的であるため完全な崩壊にはならない。ただし、マッチング結果から導出される設計検討事項の質は劣化する
2. **Phase 0.5-2b（自由推論交差）が劣化**: テンプレート外の交差検出は自由推論に完全依存するため、A-5崩壊で最も影響を受ける
3. **Phase 0.5-3a（統合）の交差分析由来セクションが薄くなる**: 交差分析の入力品質が低いため
4. **Phase 0.5-4（敵対的検証）は間接的に補完**: 障害シナリオの一部は要件間相互作用に起因するものを含むため、敵対的検証が交差分析の不足を部分的に補う

**安全網の機能段階**: 2段階（テンプレート交差の構造的保証 + 敵対的検証）。テンプレートパターンの10パターンが2要件間の主要交差をカバーするため、完全な崩壊には至らない。

**ユーザーに到達する問題**: 非標準的な要件間相互作用（テンプレート外）に起因する設計検討事項の不足。特に3要件以上の複合交差は検出される可能性がほぼない。

---

### A-6崩壊時: ファイル経由の情報共有精度が不足

**崩壊条件**: 中間ファイルのフォーマットでは表現しきれない情報（文脈、ニュアンス、条件付き判断）が失われる。

**カスケード**:
1. **Phase 0.5-3a（統合）の入力品質が部分的に劣化**: Phase 0.5-1の5ファイルの構造化フォーマットが情報を損失なく伝達できない場合、統合時に情報が失われる
2. **Phase 0.5-3b（検証層）の照合精度が低下**: Phase 0.5-3aの出力が不完全なため、照合結果の「該当・未識別」判定に誤りが増加
3. **Phase 0.5-4（敵対的検証）は影響限定的**: 入力が「5視点の要約+ギャップ情報」に意図的に制限されており、そもそも情報量を絞った設計。A-6崩壊の影響は小さい
4. **process_design（Phase 0.5-2a(p)）の順方向検証が劣化**: arch_design出力とprocess_design出力の突合の精度が低下

**安全網の機能段階**: 全段階が部分的に劣化するが、いずれも完全には失敗しない。フォーマット設計の改善で緩和可能。

**ユーザーに到達する問題**: 情報劣化に起因する「検討深度の不足」（Depth Gap）。識別漏れ（Identification Gap）よりは影響が小さい。Phase 0.5-5の提示内容が情報劣化により不完全になる可能性はあるが、検討事項の「存在」自体は伝達される。

## 分析3: 構造的不可能性

| # | 不可能な検出 | 根本原因 | 影響 |
|---|------------|---------|------|
| **S-1** | **LLMの訓練データに存在しない概念の導出** | 導出層・検証層・敵対的検証のすべてが同一LLMの知識源に依存する（Step 0 §2.4 項目4「同一盲点の共有」）。リファレンスカテゴリも最終的にはLLMが生成（または人間がLLMの知識範囲内で設計）した枠組みであるため、LLMの知識源の外にある概念は原理的に検出できない | 通常プロジェクトでは低影響（LLMの知識は十分広い）。最先端技術・ニッチドメインのプロジェクトでは高影響 |
| **S-2** | **要件書に記述されず、かつLLMの暗黙知にもない暗黙要件** | 暗黙要件の検出は「LLMが一般的な技術知識から推論する」メカニズムに依存。要件書に手がかりがなく、LLMの訓練データにも存在しない暗黙要件は検出の起点がない | 組織固有の暗黙的な業務ルール、口伝えで共有されている制約等。方式の構造的限界だが、**いかなる自動化方式でもこの限界は共有される** |
| **S-3** | **リファレンスカテゴリの「非該当」誤判定の検出** | Phase 0.5-3bで「このサブ項目はプロジェクトに非該当」と判定した場合、その判定の正しさを検証する独立メカニズムがない。判定自体がLLM推論に依存し、誤判定を検出する追加レイヤーが存在しない | 「非該当」と判定された検討事項が実際には必要だった場合、全安全網を通過して検出されない。Phase 0.5-5でユーザーに「非該当判定の妥当性」を確認する設計はあるが、A-3b崩壊時には機能しない |
| **S-4** | **検証層が導出層と同じ限界を持つケースでのカバレッジギャップ検出** | Phase 0.5-3b（検証層）の「該当/非該当」判定はLLM推論に依存する。導出層で見落とした検討事項が、検証層で「非該当」と誤判定される確率はゼロでない。特に、導出層で見落とした**理由**（LLMの知識不足）と同じ理由で検証層も見落とす場合、検証層は安全網として機能しない | リファレンスカテゴリのサブ項目レベルの照合は構造的に機能するが、「該当性判定」のステップにLLM推論が介在する点が弱点。例: 特定の業界規制（RG-4）のサブ項目が「このプロジェクトに該当するか」の判定は、LLMのドメイン知識に依存する |
| **S-5** | **Phase間の情報制限設計に起因する情報喪失の完全回避** | Phase 0.5-4への入力を「要約+ギャップ情報」に制限する設計は、アンカリング防止のためだが、同時に導出層のコンテキスト（具体的な検討事項の詳細、推論過程）を敵対的検証に伝達できない。これにより、導出層が「部分的に検討したが深度が不足」している検討事項を、敵対的検証が独立に再発見する可能性は低下する | 意図的な設計トレードオフ。アンカリング防止と情報伝達は原理的にトレードオフ関係にあり、両立は不可能。現設計は「アンカリング防止を優先し、情報喪失を許容する」判断 |
| **S-6** | **自己参照的な網羅性検証の完全性** | 「カテゴリカバレッジ」の検証基準（Step 0 §1.3条件1）はリファレンスカテゴリに依存し、リファレンスカテゴリの網羅性は外部基準（ISO 25010等）に依存するが、外部基準自体の網羅性は保証されない。最終的にはメタレベルの循環が残存する | 方式設計の根本的限界。「漏れゼロ」の保証は原理的に不可能であり、Step 0 §1.3の「重要な限界」で認識済み |

| **5視点×並列導出による同一検討事項の多重生成** | 5つの視点が互いに関連する検討事項を独立に生成する。例: セキュリティに関する検討事項が「技術インフラ視点」と「規制・ガバナンス視点」の両方から生成される | **中**。視点間の境界はリファレンスカテゴリの配置で制御されているが、横断的関心事（セキュリティ、ロギング等）は複数視点から言及される。Phase 0.5-3aの重複排除でマージされるが、完全一致でない類似検討事項の統合判定にはLLM推論が必要 | Phase 0.5-3aの統合・構造化フェーズが明示的に重複排除を担当。「最も充実した記述を保持し、複数の視点からの根拠を集約」する設計。このメカニズム自体は適切だが、処理負荷が高い（5ファイル+2ファイルの統合） |

### 過剰検出の総合評価

過剰検出の総量は、以下の要因で見積もられる:
- Phase 0.5-1: カテゴリ名ガイドによる不要カテゴリの検討事項生成（推定: 全検討事項の10-20%）
- Phase 0.5-3c: 既存導出との重複（推定: 補完導出の20-30%が重複）
- Phase 0.5-4: 既存カバレッジ済み項目の再検出（推定: 障害シナリオの30-50%が既存カバー可能性「中〜高」）

これらの過剰検出は、Phase 0.5-5のユーザー検証で除外される設計。ユーザーの確認コストとしては、検討事項リスト全体の15-25%程度の不要項目を判別する負担が発生すると推定される。

## 総合評価

### 方式の限界の全体像

本方式（リファレンス補強型動的導出）の限界は、以下の3層に構造化できる:

**第1層: 原理的限界（いかなる方式でも共有）**
- S-2: 要件書に手がかりがなく、LLM知識にもない暗黙要件は検出不可能
- S-6: 網羅性検証の完全性は原理的に達成不可能（メタレベルの循環）
- これらはStep 0 §1.3で「漏れゼロの保証ではなく、漏れの発生確率の構造的低減を目指す」と明記されており、方式の設計目標と一致

**第2層: 方式固有の構造的限界（設計トレードオフの帰結）**
- S-5: アンカリング防止と情報伝達のトレードオフ。Phase間の入力制限が情報喪失を伴う
- L-4/S-4: 3要件以上の複合交差と検証層の判定品質。コスト制約内での検出力の限界
- L-6: process_designの敵対的検証がプロンプト拡張。コスト制約との兼ね合い
- 分析1の「外側の領域」の一部（コスト・経済性のカバレッジ不足）

**第3層: 仮定崩壊時のリスク（条件付き限界）**
- A-1崩壊: 方式全体が劣化。検証層が下限保証するが、検証層の判定品質も同時に劣化
- A-4崩壊: リファレンスカテゴリの盲点が系統的な見落としを生む。敵対的検証のみが安全網
- A-3b崩壊: 最終安全網（ユーザー検証）が無効化。方式の構造内では対処不可能

### 安全網の有効性評価

| 安全網の層 | 検出対象 | 独立性 | 限界 |
|-----------|---------|--------|------|
| 導出層（Phase 0.5-1/2） | 要件書から直接導出可能な検討事項 | コンテキスト独立（5並列） | LLMの知識源に依存。知識源非独立 |
| 検証層（Phase 0.5-3b） | リファレンスカテゴリのサブ項目内の欠落 | 外部基準（ISO 25010等）に基づく | 「該当/非該当」判定にLLM推論が必要。カテゴリ外は検出不可 |
| 敵対的検証（Phase 0.5-4） | 障害シナリオから逆推論可能な欠落 | 認知経路の多様化（逆推論） | 入力制限による情報喪失。「正常系」シナリオは対象外 |
| ユーザー検証（Phase 0.5-5） | 上記3層で検出されなかった欠落 | 知識源独立（人間の判断） | A-3b仮定に依存。ユーザーの専門性に制約される |

安全網は4層構造だが、知識源の非独立性（S-1）により、導出層・検証層・敵対的検証の3層は同一LLMの知識限界を共有する。真に独立な安全網はユーザー検証のみ。

### 推奨事項（限界を踏まえた運用上の注意点）

1. **ユーザー検証（Phase 0.5-5）の質がシステム全体の検出力の上限を決定する**: ユーザーは提示された検討事項リストの確認だけでなく、「自分のドメイン知識に基づいて、リストにないが重要な検討事項」を積極的に追加する役割を認識すべき
2. **「カバレッジ照合完了」は「網羅的」を意味しない**: Phase 0.5-3bの照合結果はリファレンスカテゴリの範囲内での検証であり、カテゴリ外の欠落は検出されない。この限界をユーザーに明示的に伝える必要がある
3. **コスト・経済性の観点は現在のリファレンスカテゴリでカバレッジが薄い**: 分析1で特定されたカバレッジギャップ。OP-5への統合か独立カテゴリの追加が検討に値する
4. **ニッチドメインのプロジェクトではA-1/A-4崩壊リスクが高い**: 低頻度ドメインのプロジェクトでは、外部専門家レビューの追加を推奨
