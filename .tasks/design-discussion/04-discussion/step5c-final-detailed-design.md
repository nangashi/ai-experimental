# Step 5c: 最終統合設計 — 検討事項漏れ防止方式

本文書は、Step 3b（詳細設計）にStep 5a（修正提案M-01〜M-25）およびStep 5b（追加修正A〜G）を統合した最終設計である。自己完結した単一文書として、未解決事項を含まない。

## 0. 設計の前提と方針

### 0.1 採用方針: リファレンス補強型動的導出

Step 3aで確定した「リファレンス補強型動的導出」方式を具体化する。核心思想:

- **導出層**: 動的推論（方式A）を主構造とし、視点駆動型の独立サブエージェントで検討事項を導出する
- **構造的検証層**: リファレンスカテゴリ（方式B）を検証層として後段に配置し、カバレッジギャップを検出する
- **敵対的検証層**: シナリオベースの障害分析で「未知の未知」を能動的に探索する

### 0.2 Step 3aからの追加決定（A/Bの回答を反映）

| 論点 | Step 3aの方針 | A/Bの回答を踏まえた追加決定 | 根拠 |
|------|-------------|------------------------|------|
| Phase 0.5-1の入力 | 視点レンズのみ | **カテゴリ名ガイドを追加**（Bの二段階注入提案を採用） | カバレッジ向上とアンカリング抑制のバランスが最適。サブ項目は注入しない |
| Phase 0.5-3の構造 | 1サブエージェント | **2サブエージェントに分割**（Aの責務分割提案を採用） | 統合+構造化と照合+充実度チェックを分離し、各サブエージェントの処理負荷を適正化 |
| Phase 0.5-4の入力 | requirements + 5視点要約のみ | **要約 + トレーサビリティギャップ情報を追加**（Aの中間案を採用） | ギャップ指向の検出モードを維持しつつ、アンカリングは抑制 |
| 横断的関心事 | 明示的な位置づけなし | **Phase 0.5-3aで境界仕様を生成**（Aの提案を採用） | arch→processの引き継ぎに必要 |
| process_designの敵対的検証 | なし | **Phase 0.5-3のプロンプト拡張で軽量実施**（Aの提案を採用、追加サブエージェントなし） | コスト制約（5-6回）を維持しつつ暗黙要件検出の機会を確保 |
| 補完導出 | 常時実行 | **条件付き実行**（Bの提案を採用） | カテゴリ名ガイドにより不足カテゴリ数が減少する見込み。不足検出時のみ実行 |

## 2. arch_design: フェーズ別詳細設計

### Phase 0.5-0: ドメイン検出・入力サイズ判定

**目的**: requirements.mdのドメイン特性を検出し、リファレンスカテゴリの動的拡張と入力サイズに応じた処理分割戦略を決定する。

**サブエージェント数**: 1

**入力**:
1. `requirements.md`（全文）

**処理内容**:

#### A. ドメイン検出
1. 要件書のドメインを判定（Web/モバイル/組み込み/産業制御/医療/金融等）
2. ドメイン固有の設計カテゴリが存在する場合、リファレンスカテゴリへの追加カテゴリとサブ項目を生成する
3. 標準カテゴリ名ガイドの中で、このドメインに不適切なカテゴリを「低優先」とマークする

#### B. 入力サイズ判定
1. requirements.mdの要件数とファイルサイズを判定
2. 以下の基準で処理分割戦略を決定:
   - 要件数30件以下かつrequirements.md 50KB以下: **標準処理**
   - 要件数31-60件またはrequirements.md 50-100KB: **分割処理A**
   - 要件数61件以上またはrequirements.md 100KB超: **分割処理B**

**出力ファイル**: `domain-profile.md`

**出力構造**:
```
## 1. ドメイン判定結果
- 判定ドメイン: [ドメイン名]
- 判定根拠: [要件書のどの記述から判定したか]

## 2. arch_design用追加カテゴリ（0-5件）
### [カテゴリ名]
- [サブ項目1]
- [サブ項目2]
...

## 3. process_design用追加カテゴリ（0-3件）
### [カテゴリ名]
- [サブ項目1]
- [サブ項目2]
...

## 4. 標準カテゴリの優先度調整
- [カテゴリ名]: 低優先（理由: ...）

## 5. 入力サイズ判定
- 要件数: N件
- ファイルサイズ: XKB
- 処理分割戦略: 標準処理 / 分割処理A / 分割処理B
```

**設計意図**:
- 低頻度ドメインのリファレンスカテゴリカバレッジを構造的に強化（F-05対応）
- ドメイン検出と入力サイズ判定を1サブエージェントで実行し効率化（F-06対応）

### Phase 0.5-1: 視点駆動型独立導出

**目的**: requirements.mdから、5つの独立した視点で設計検討事項を導出する。

**サブエージェント数**: 5（並列実行）

**各サブエージェントの入力**:
1. `requirements.md`（全文。分割処理の場合は§11参照）
2. 担当視点の定義（Step 0b §2のカバレッジマップから該当視点の記述）
3. **担当視点のリファレンスカテゴリ名リスト**（§4で定義、カテゴリ名のみ。サブ項目は含めない）
4. **domain-profile.mdの追加カテゴリ名**（Phase 0.5-0の出力から担当視点に関連するもの）

**プロンプト構造**:
```
あなたは{視点名}の専門家です。以下の要件文書を読み、{視点名}の観点から
設計検討事項を導出してください。

## 入力
{requirements.md の内容}

## 視点の定義
{Step 0b §2 カバレッジマップの該当視点の記述}

## 参考カテゴリ（これらに限定せず、要件に基づいて導出すること）
以下は{視点名}に関連する一般的なカテゴリです。これらはガイドとして参照してください。
これらのカテゴリ外の検討事項も含めてください。
- {カテゴリ名1}
- {カテゴリ名2}
- ...
{domain-profile.mdの追加カテゴリ名がある場合}
- {ドメイン固有カテゴリ名1}（ドメイン固有）
- ...

## 暗黙要件の導出ガイドライン
暗黙要件として挙げる際は以下の基準に従うこと:
1. **必須暗黙要件**: 技術スタックやドメインの特性上、対処しなければシステムが
   正常に動作しない要件（例: バックアップ、ログ設計、エラーハンドリング方針）
2. **推奨暗黙要件**: 明示されていないが、一般的なベストプラクティスとして
   検討が推奨される要件（例: 国際化、アクセシビリティ）

暗黙要件には以下を明記すること:
- 分類: [必須] または [推奨]
- 暗黙と判断した根拠（なぜ要件書に記載されていないと判断したか）

## 出力形式
各検討事項について以下を記述:
- 検討事項名
- 対応する要件ID（SR-xxx / NFR-xxx / 制約-xxx、または「暗黙要件」+分類+根拠）
- 検討の概要（何を決定する必要があるか）
- 検討の根拠（なぜこの検討が必要か。要件のどの記述から導出されるか）
- 初期的な選択肢（2-3案の概要。この時点での詳細検討は不要）
```

**出力ファイル**: `phase-0.5-1-{視点名}.md`（5ファイル）

**設計意図**:
- カテゴリ名のみの注入により、LLMのアテンションを広く分配しつつ、サブ項目レベルのアンカリングを回避（二段階注入）
- 要件ID紐付き必須により、過剰検出を構造的に抑制（要件から導出不可能な検討事項は「暗黙要件」として明示が必要）
- 各サブエージェントは他の視点の出力を参照しない（コンテキスト独立性: Step 0d §1.4）
- 暗黙要件の[必須]/[推奨]分類により導出の透明性を確保（M-10）

**Step 0dへの対応**:
| バイアス | 対処 |
|---------|------|
| アンカリング効果 | 5サブエージェント独立実行。他の視点の出力を入力に含めない |
| 訓練データ分布バイアス | カテゴリ名ガイド（標準+ドメイン固有）が低頻度ドメインの想起を促進 |
| アテンション配分バイアス | 視点ごとの分離実行により、要件書の記述量偏りの影響を緩和 |

### Phase 0.5-2a: 交差パターンテンプレート照合

**目的**: 要件間の標準的な相互作用パターンを構造的に検出する。

**サブエージェント数**: 1

**入力**:
1. `requirements.md`（全文）
2. 交差パターンテンプレート（§5で定義）

**プロンプト構造**:
```
以下の要件文書について、事前定義された交差パターンを照合し、
要件間の相互作用に起因する設計検討事項を特定してください。

## 入力
{requirements.md の内容}

## 交差パターンテンプレート
{§5のテンプレート}

## 照合方針
{要件数が30件を超える場合（Phase 0.5-0の判定結果を参照）:}
1. まずNFR全件と制約全件を交差パターンの優先照合対象とする
2. SR間の交差は、同一機能グループ内のSR間を優先照合する
3. 全ペアの網羅的照合ではなく、パターンごとに「最も関連性が高い要件ペア」を
   最大10件特定する方式に切り替える

## 出力形式
各パターンについて:
- パターン名
- 判定: 該当 / 非該当 / 判断不能
- 「該当」の場合: 要件の組み合わせ（要件ID）、相互作用内容、導出される設計検討事項
- 「非該当」の場合: 非該当の理由
- 「判断不能」の場合: 判断不能の理由（要件記述の不十分さを具体的に指摘）
```

**出力ファイル**: `phase-0.5-2a-cross-template.md`

**設計意図**:
- 「判断不能」カテゴリにより、要件の不十分さを検出しPhase 0.5-5のユーザー検証で活用（M-11）
- 大規模要件での組み合わせ爆発を効率化ガイドラインで抑制（M-07）

### Phase 0.5-2b: 自由推論交差分析

**目的**: テンプレートに含まれない非標準的な要件間相互作用を検出する。

**サブエージェント数**: 1

**入力**:
1. `requirements.md`（全文のみ。Phase 0.5-1やPhase 0.5-2aの出力は含めない）

**プロンプト構造**:
```
以下の要件文書を読み、要件間の相互作用（トレードオフ、依存関係、
競合関係）を分析し、それぞれに起因する設計検討事項を導出してください。

特に以下に注意:
- 明示的に記述されていない暗黙的な相互作用
- 3つ以上の要件が絡む複合的な相互作用
- 非機能要件同士のトレードオフ

## 入力
{requirements.md の内容}

## 出力形式
各相互作用について:
- 関連する要件ID
- 相互作用の種類（トレードオフ / 依存 / 競合 / 複合）
- 内容の説明
- 導出される設計検討事項（検討事項名、概要、初期選択肢）
```

**出力ファイル**: `phase-0.5-2b-cross-freeform.md`

**設計意図**:
- Phase 0.5-2aの出力を入力に含めないことで、テンプレートパターンへのアンカリングを防止
- requirements.mdのみから自由推論することで、テンプレート外の相互作用を検出

### Phase 0.5-3a: 統合・構造化

**目的**: Phase 0.5-1（5ファイル）とPhase 0.5-2a/2bの出力を統合し、構造化する。

**サブエージェント数**: 1（分割処理Bの場合は§11参照）

**入力**:
1. `requirements.md`
2. Phase 0.5-1の出力（5ファイル）
3. Phase 0.5-2aの出力
4. Phase 0.5-2bの出力

**処理内容**:
1. **重複排除**: 以下の基準で同一性を判定し統合する
2. **横断的関心事の分類**: 以下の基準で分類し境界仕様を生成する
3. **ドメイン固有検討事項の分類**: 5視点外の事項を以下の手順で分類する
4. **要件トレーサビリティマトリクス**: Part A/B/Cの3部構成で生成
5. **構造化**: 統合された検討事項リストを5視点+横断的+交差分析の構造で整理

**重複排除の基準**:
```
以下のいずれかに該当する場合、2つの検討事項を「同一」とみなし統合する:
1. 同一の要件IDに紐付き、かつ検討の対象（何を決定するか）が同一
2. 異なる視点から導出されたが、決定すべき事項が実質的に同一
   （例: 「認証方式の選定」が技術インフラ視点と規制視点の両方で導出）

統合時のルール:
- 最も詳細な検討概要を保持する
- 全ての導出元視点を記録する
- 全ての要件ID紐付きを集約する
- 異なる初期選択肢が挙がっている場合は全てを統合する
```

**横断的関心事の分類基準**:
```
以下のいずれかに該当する検討事項を「横断的関心事」として分類する:
1. 3つ以上の視点から導出された検討事項
2. 2つの視点から導出され、かつarch_designとprocess_designの両方で検討が必要な事項

横断的関心事には境界仕様を生成する:
- arch_designの責務範囲
- process_designの責務範囲
- 相互依存関係
```

**ドメイン固有検討事項の分類**:
```
5視点（技術インフラ、ドメイン、ユーザー体験、運用、規制）のいずれにも
明確に分類できない検討事項は、以下の手順で分類する:
1. 最も関連性の高い視点に分類する（完全一致でなくてよい）
2. 分類の根拠を明記する
3. 「ドメイン固有」タグを付与する
```

**出力ファイル**: `phase-0.5-3a-integrated.md`

**出力構造**:
```
## 1. 検討事項リスト（視点別）
### 1.1 技術インフラ
- [検討事項1]: ...
- [検討事項2]: ...
### 1.2 ドメイン・ビジネスロジック
...
### 1.6 交差分析由来
...

## 2. 横断的関心事の境界仕様

  arch_designの責務: [方針レベルの決定事項]

## 3. 要件トレーサビリティマトリクス

### Part A: 明示要件のカバレッジ

- 視点別件数: ...

- 5視点の要約（各視点2-3文。分割処理A/Bの場合は§11参照）
```

**目的**: 統合結果をリファレンスカテゴリのサブ項目レベルで照合し、カバレッジギャップと内容不足を検出する。

**サブエージェント数**: 1

**入力**:

**処理内容**:

2. **内容充実度チェック**: 各リファレンスカテゴリについて、主要サブ項目ごとに判定する
3. **不足カテゴリの特定**: 「該当・未識別」のサブ項目を持つカテゴリを不足カテゴリとして特定

**サブ項目の判定ガイドライン**:

```
### 「該当・識別済み」の条件
対応する検討事項が以下を全て満たす:
- 検討の対象（何を決定するか）が具体的に記述されている
- 少なくとも2つの選択肢が示されている
- 要件との紐付き根拠が明記されている

### 「該当・未識別」の条件（以下のいずれか）
- 対応する検討事項が存在しない
- 対応する検討事項はあるが、検討の対象が曖昧（例:「セキュリティを考慮する」のみ）
- 対応する検討事項はあるが、選択肢が示されていない

原因ラベル:
- [導出不足]: 要件から導出可能だが検討事項に含まれていない
- [要件曖昧]: 要件自体が曖昧で、具体的な検討事項の導出が困難。
  Phase 0.5-5のユーザー検証で要件の明確化を促す

### 「非該当」の条件
- このプロジェクトの特性上、サブ項目が検討対象にならない
  - 理由は以下のいずれかであること:
    a. 要件書に明示的に除外されている
    b. プロジェクトの技術スタックから論理的に不要
       （例: シングルページアプリでオフライン対応不要）
  - 「要件に記載がない」のみでは「非該当」と判定しない。
    暗黙要件の可能性があるため「該当・未識別」とする

### 「判断不能」の条件
- 要件記述が不十分で、該当性を判定できない
  → Phase 0.5-5で要件の明確化を促す（M-11の交差パターン「判断不能」と整合）
```

3. **逆流の追加検出**: リファレンスカテゴリの各サブ項目について「このサブ項目の検討にarch_designの設計決定が必要か」を追加判定。必要だがarch_designの出力に該当する決定がない場合、追加の逆流として検出する
4. **軽量敵対的検証**: プロンプト末尾に以下を追加:
   ```
   最後に、このプロセス設計のもとで運用上の障害が起きるシナリオを3つ挙げ、
   それぞれについて「このプロセス設計で防止できるか」を評価してください。
   防止できない場合、追加の設計検討事項を特定してください。
   ```

**出力ファイル**: `phase-0.5-3p-integrated.md`

**設計意図**:
- process_designのサブエージェント予算内（5-6回）に収めるため、敵対的検証を独立サブエージェントにせずプロンプト拡張で実施
- 同一コンテキスト内での実施のためアンカリングリスクはあるが、追加的な検出の「機会」として位置づける（L-6参照）
- リファレンスカテゴリ照合時の逆流追加検出により、Phase 0.5-1(p)の品質に依存しない独立した検出機会を確保（M-15）

---

### Phase 0.5-4(p): ユーザー検証

**目的**: process_designの検討事項リストをユーザーに提示し、検証を受ける。

**ユーザーインタラクション数**: 1

**提示内容**:
1. process_designの検討事項リスト（視点別）
2. arch→process順方向検証の結果
3. process→arch逆流検出の結果（arch_designへの差し戻し候補）
4. カバレッジ照合結果の要約
5. 軽量敵対的検証からの追加事項

**逆流への対応ガイダンス**:
```
逆流が検出された場合の推奨対応:
1. 軽微な不足（1-2項目、影響範囲限定）: process_design内で暫定方針を決定し、
   arch_designへの差し戻し候補として記録。process_designを継続
2. 重大な不足（3項目以上、または広範囲に影響）: process_designを一時停止し、
   arch_designの該当部分を補完した上で再開を推奨
3. 不整合（arch_designの設計決定間の矛盾）: arch_designの修正が必要。
   process_designの関連部分は修正完了まで保留

※ 最終的な判断はユーザーに委ねる（スコープ外の対応手順の具体化はしない）
```

### 7.3 複合崩壊リスク

以下の仮定が同時に崩壊した場合、安全網が大幅に弱体化する:

| 複合ケース | 影響 | 発生しやすい条件 |
|-----------|------|----------------|
| A-1（推論力低下）+ A-4（カテゴリ不完全） | 導出層も検証層も劣化。敵対的検証のみが安全網 | 低頻度ドメインのプロジェクト |
| A-4（カテゴリ不完全）+ A-3b（ユーザー欠落検出不能） | 検証層のカバレッジギャップがユーザーにも検出されない | 専門外ドメインのプロジェクトを非専門家が実施 |
| A-1（推論力低下）+ A-3b（ユーザー欠落検出不能） | 導出品質が低く、ユーザーも補完できない。検証層のみが安全網 | 非専門家がLLMに不慣れなドメインで実施 |

**注記**: 複合崩壊ケースの可視化は「ドキュメント化」であり、新たな構造的緩和策は追加していない。個別の仮定に対する構造的緩和（A-4に対するM-03の非該当厳格化、A-1に対するリファレンス照合等）がそれぞれ機能する。構造的不可能性に対して追加の構造的緩和は設計できないため、リスクの可視化として位置づける。

## 11. 大規模入力への対応

### 11.1 arch_designの分割処理

Phase 0.5-0で判定された入力サイズに応じて、以下の処理分割を適用する。

#### 標準処理（要件30件以下、50KB以下）
- 全フェーズでrequirements.md全文を入力。追加対応不要。

#### 分割処理A（要件31-60件、50-100KB）
- **Phase 0.5-1**: 各サブエージェントにrequirements.md全文を入力（そのまま）
- **Phase 0.5-3a**: 入力をPhase 0.5-1の出力要約（各視点のキーポイント）+ 完全な検討事項リスト（構造化）に分割。requirements.md全文は入力しない（Phase 0.5-1で処理済みのため）
- **Phase 0.5-4**: 5視点要約を各視点3-5文に拡張

#### 分割処理B（要件61件以上、100KB超）
- **Phase 0.5-1**: requirements.mdを機能グループ（SR/NFR/制約）に分割し、各サブエージェントに全グループを入力するが、重点処理する機能グループを指定
- **Phase 0.5-3a**: 2サブエージェントに分割
  - Phase 0.5-3a-1: 視点1-3の統合
  - Phase 0.5-3a-2: 視点4-5+交差分析の統合
  - Phase 0.5-3a-3: 2つの出力をマージ（追加1サブエージェント）
    - マージ処理に以下を含む:
      - 横断的関心事の統合（Phase 0.5-3a-1と0.5-3a-2にまたがる検討事項について、横断的関心事基準での再判定）
      - 交差分析由来の検討事項の統合（Phase 0.5-2a/2bの出力はPhase 0.5-3a-3のマージ段階で統合する）
- **Phase 0.5-4**: 5視点要約を各視点5-8文+要件グループ別概要に拡張

### 11.2 process_designの分割処理

分割処理B（要件61件以上）の場合:
- **Phase 0.5-1(p)**: requirements.mdは分割処理Bの機能グループ分割をそのまま使用。arch_designの出力（`arch-design-concerns-final.md`）は全文入力。各サブエージェントは全グループを参照するが、重点処理グループを指定

### 11.3 交差パターン照合の効率化

要件数が30件を超える場合（Phase 0.5-2a）:
1. まずNFR全件と制約全件を交差パターンの優先照合対象とする
2. SR間の交差は、同一機能グループ内のSR間を優先照合する
3. 全ペアの網羅的照合ではなく、パターンごとに「最も関連性が高い要件ペア」を最大10件特定する方式に切り替える

### 11.4 ユーザー提示の段階化

Phase 0.5-5のユーザーインタラクション（検討事項数に応じて段階化。§2 Phase 0.5-5に記載）。
