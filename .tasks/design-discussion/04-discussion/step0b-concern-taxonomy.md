# Step 0b: 検出対象の全体像の確立

## 1. 設計検討事項の分類次元

以下の外部ソースから分類次元を抽出し、統合する。

### 1.1 品質特性（ISO 25010）

| 品質特性 | 設計検討事項の例 |
|---------|----------------|
| 機能適合性 | 機能の完全性、正確性、適切性 |
| 性能効率 | 応答時間設計、スループット設計、リソース利用効率、スケーリング戦略 |
| 互換性 | 共存設計、相互運用性、外部API互換性 |
| 使用性 | アクセシビリティ、操作性、UIレスポンス設計、エラーメッセージ設計 |
| 信頼性 | 成熟性、可用性設計（SLA/SLO）、障害許容設計、回復性設計 |
| セキュリティ | 機密性、完全性、否認防止、責任追跡性、認証・認可設計 |
| 保守性 | モジュール性、再利用性、解析性、修正性、試験性 |
| 移植性 | 適応性、設置性、置換性、マルチ環境対応 |

### 1.2 アーキテクチャビューポイント

| ビュー | 対応する設計検討事項 |
|--------|-------------------|
| 論理ビュー（4+1） | コンポーネント分割、レイヤー構成、ドメインモデル構造、依存関係管理 |
| 開発ビュー（4+1） | パッケージ構成、ビルド構成、モジュール境界、開発環境構成 |
| プロセスビュー（4+1） | 並行処理設計、プロセス間通信、同期/非同期戦略、ジョブ/キュー設計 |
| 物理ビュー（4+1） | デプロイトポロジー、ネットワーク構成、インフラ構成 |
| シナリオ（4+1） | ユースケース横断的な品質属性の検証 |
| コンテキスト（C4） | システム境界、外部連携ポイント、ユーザー種別 |
| コンテナ（C4） | 実行単位の分割、コンテナ間通信、技術スタック選定 |
| コンポーネント（C4） | 内部構造設計、責務分割、インターフェース設計 |

### 1.3 ドメイン・ビジネスロジック

| 領域 | 設計検討事項 |
|------|------------|
| DDD戦略設計 | バウンデッドコンテキストの定義、コンテキストマッピング（関係性の設計）、ユビキタス言語の確立 |
| DDD戦術設計 | 集約境界の設計、エンティティ/値オブジェクト設計、ドメインイベント識別、リポジトリ設計 |
| ビジネスルール構造 | ビジネスルールの配置戦略（ドメイン層 vs アプリケーション層）、ルールエンジンの要否、バリデーション戦略 |
| 状態管理 | ステートフル/ステートレス設計方針、分散状態の同期戦略、セッション管理方式、イベントソーシングの要否 |
| ワークフロー/サガ | 長期実行プロセスの調整方式、補償トランザクション設計、ワークフローエンジンの要否 |

### 1.4 ユーザー・体験設計

| 領域 | 設計検討事項 |
|------|------------|
| レスポンス設計 | 応答時間目標のUI/UXへの影響、ローディング/プログレス表示戦略、楽観的UI更新 |
| オフライン対応 | オフラインモードの要否と方式、データ同期戦略、コンフリクト解決方針 |
| マルチデバイス | レスポンシブ設計方針、デバイス固有機能の活用方針、データの端末間同期 |
| アクセシビリティ | WCAG準拠レベル、スクリーンリーダー対応方針、キーボード操作設計 |
| 国際化/ローカライゼーション | 多言語対応の方式、日時/通貨/数値のフォーマット方針、RTL対応の要否 |

### 1.5 運用・進化

| 領域 | 設計検討事項 |
|------|------------|
| 監視・可観測性 | ログ設計（構造化ログ、ログレベル）、メトリクス設計、トレーシング設計、アラート設計 |
| バックアップ・復旧 | バックアップ戦略（RPO/RTO）、リストア手順設計、災害復旧設計 |
| デプロイ・リリース | デプロイ戦略（Blue-Green、Canary等）、ロールバック方針、フィーチャーフラグ設計 |
| データマイグレーション | スキーマ変更戦略、データ移行方式、ダウンタイム管理、段階的移行パス |
| キャパシティプランニング | 初期キャパシティ設計、スケールアウト/アップ方針、コスト最適化 |
| 既存システム共存 | 移行期間の共存設計、データ同期方式、段階的切替戦略 |

### 1.6 規制・コンプライアンス

| 領域 | 設計検討事項 |
|------|------------|
| データ保護規制 | GDPR/個人情報保護法対応（同意管理、データ削除権、ポータビリティ）、データ保持期間設計 |
| 業界固有規制 | PCI DSS（決済）、HIPAA（医療）、SOX（金融）等への準拠設計 |
| 監査要件 | 監査ログ設計、証跡保全方式、監査対応インターフェース |
| データ所在地 | データレジデンシー要件、クロスボーダーデータ転送制約 |

### 1.7 コスト・経済性

| 領域 | 設計検討事項 |
|------|------------|
| インフラコスト | クラウドリソース選定、リザーブド/スポットインスタンス戦略、マルチクラウド方針 |
| 運用コスト | 自動化レベルの設計、SRE人員要件に影響する設計判断 |
| 開発コスト | 技術的負債の許容範囲設計、ライブラリ/フレームワーク選定のTCO |
| ライセンス | OSS利用方針、商用ライセンスの影響、ライセンスコンプライアンス |

### 1.8 開発プロセス

| 領域 | 設計検討事項 |
|------|------------|
| テスト戦略 | テストピラミッド設計、E2Eテスト範囲、テスト環境設計 |
| CI/CD | パイプライン設計、品質ゲート設計、環境プロモーション戦略 |
| ブランチ戦略 | ブランチモデル選定、マージ方針、リリースブランチ管理 |
| ドキュメント | アーキテクチャ決定記録（ADR）方針、API仕様管理、運用ドキュメント |

## 2. カバレッジマップ

上記の分類次元を5つの視点にマッピングし、偏りがないことを確認する。

| 視点 | 対応する分類次元 | カバレッジの根拠 |
|------|----------------|----------------|
| **技術インフラ** | 1.1(性能効率,互換性,信頼性,保守性,移植性), 1.2(開発ビュー,プロセスビュー,物理ビュー,コンテキスト,コンテナ), 1.5(監視,デプロイ,キャパシティ) | ISO 25010の品質特性 + 4+1 View + C4のインフラ層 |
| **ドメイン・ビジネスロジック** | 1.1(機能適合性), 1.3(全項目), 1.2(論理ビュー,コンポーネント) | DDD戦略/戦術設計 + 4+1 View論理ビュー |
| **ユーザー体験** | 1.1(使用性), 1.4(全項目) | ISO 25010使用性 + UX固有の設計次元 |
| **運用** | 1.5(全項目), 1.7(インフラコスト,運用コスト,開発コスト), 1.8(CI/CD,テスト戦略,ブランチ戦略) | 運用設計 + コスト設計 + 開発プロセス |
| **規制・ガバナンス** | 1.1(セキュリティ), 1.6(全項目), 1.7(ライセンス), 1.8(ドキュメント) | セキュリティ + 規制 + コンプライアンス |

**メタ次元の注記**: シナリオ（4+1 View）は特定の視点に属さず、各視点の検討事項を具体的なユースケースで検証するメタ次元として機能する。カバレッジマップの5視点すべてに横断的に適用される。

### バランス確認

| 視点 | 分類次元の数 | 設計検討事項の例の数 | 評価 |
|------|------------|-------------------|------|
| 技術インフラ | 10+ | 20+ | 最多。これは技術インフラが設計検討事項の中心であることの反映であり、過剰ではない |
| ドメイン・ビジネスロジック | 7 | 15+ | 02-discussionの指摘を受けて追加。DDD由来の次元が充実 |
| ユーザー体験 | 6 | 10+ | 適切。UX固有の設計次元を独立して確保 |
| 運用 | 8 | 15+ | 適切。運用・デプロイ・コスト・プロセスをカバー |
| 規制・ガバナンス | 6 | 10+ | 適切。業界固有規制の多様性を反映 |

**偏り評価**: 技術インフラが最多だが、これは設計検討事項の性質として技術的な決定が多いことの反映であり、他の視点が不当に軽視されているわけではない。02-discussionのcritical-reviewで指摘された「ドメイン・ビジネスロジック」視点の欠落は、1.3セクションの追加により解消されている。

**注記**: 上記のバランス確認は分類枠組みの**記述上のカバレッジ**を検証するものであり、LLMの**実効的な検出能力**のバランスを保証するものではない。LLMの訓練データ分布バイアス（step0-foundations.md セクション2.4 項目1）により、記述量が同等でも視点間の検出有効性には差がある。この差はStep 4c（方式限界分析）で評価する。

## 3. 後続ステップへの適用方法

このカバレッジマップは以下の目的で使用する:

### Step 3b: リファレンスカテゴリの設計時
- 方式が提案するリファレンスカテゴリが、カバレッジマップの**5視点すべて**をカバーしているか照合する
- カバー外の視点がある場合、その理由を明記する（例: 「ユーザー体験視点はarch_designのスコープ外のため非対象」等）

### Step 3b: 飽和スキャン軸の設計時
- スキャン軸が、カバレッジマップの5視点の中で**リファレンスカテゴリでカバーされていない部分**を補完しているか確認する
- 5視点の中でスキャン軸にもリファレンスにもカバーされない部分が残る場合、その領域を明示する

### Step 4c: 方式限界分析
- カバレッジマップのどの領域が方式の安全網の「内側」にあり、どの領域が「外側」にあるかをマッピングする
- 安全網の外にある領域の規模と影響を評価する
