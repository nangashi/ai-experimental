# Step 3b: 具体手順の詳細設計

## 0. 設計の前提と方針

### 0.1 採用方針: リファレンス補強型動的導出

Step 3aで確定した「リファレンス補強型動的導出」方式を具体化する。核心思想:

- **導出層**: 動的推論（方式A）を主構造とし、視点駆動型の独立サブエージェントで検討事項を導出する
- **構造的検証層**: リファレンスカテゴリ（方式B）を検証層として後段に配置し、カバレッジギャップを検出する
- **敵対的検証層**: シナリオベースの障害分析で「未知の未知」を能動的に探索する

### 0.2 Step 3aからの追加決定（A/Bの回答を反映）

| 論点 | Step 3aの方針 | A/Bの回答を踏まえた追加決定 | 根拠 |
|------|-------------|------------------------|------|
| Phase 0.5-1の入力 | 視点レンズのみ | **カテゴリ名ガイドを追加**（Bの二段階注入提案を採用） | カバレッジ向上とアンカリング抑制のバランスが最適。サブ項目は注入しない |
| Phase 0.5-3の構造 | 1サブエージェント | **2サブエージェントに分割**（Aの責務分割提案を採用） | 統合+構造化と照合+充実度チェックを分離し、各サブエージェントの処理負荷を適正化 |
| Phase 0.5-4の入力 | requirements + 5視点要約のみ | **要約 + トレーサビリティギャップ情報を追加**（Aの中間案を採用） | ギャップ指向の検出モードを維持しつつ、アンカリングは抑制 |
| 横断的関心事 | 明示的な位置づけなし | **Phase 0.5-3aで境界仕様を生成**（Aの提案を採用） | arch→processの引き継ぎに必要 |
| process_designの敵対的検証 | なし | **Phase 0.5-3のプロンプト拡張で軽量実施**（Aの提案を採用、追加サブエージェントなし） | コスト制約（5-6回）を維持しつつ暗黙要件検出の機会を確保 |
| 補完導出 | 常時実行 | **条件付き実行**（Bの提案を採用） | カテゴリ名ガイドにより不足カテゴリ数が減少する見込み。不足検出時のみ実行 |

## 2. arch_design: フェーズ別詳細設計

### Phase 0.5-1: 視点駆動型独立導出

**目的**: requirements.mdから、5つの独立した視点で設計検討事項を導出する。

**サブエージェント数**: 5（並列実行）

**各サブエージェントの入力**:
1. `requirements.md`（全文）
2. 担当視点の定義（Step 0b §2のカバレッジマップから該当視点の記述）
3. **担当視点のリファレンスカテゴリ名リスト**（§3で定義、カテゴリ名のみ。サブ項目は含めない）

**プロンプト構造**:
```
あなたは{視点名}の専門家です。以下の要件文書を読み、{視点名}の観点から
設計検討事項を導出してください。

## 入力
{requirements.md の内容}

## 視点の定義
{Step 0b §2 カバレッジマップの該当視点の記述}

## 参考カテゴリ（これらに限定せず、要件に基づいて導出すること）
以下は{視点名}に関連する一般的なカテゴリです。これらはガイドとして参照してください。
これらのカテゴリ外の検討事項も含めてください。
- {カテゴリ名1}
- {カテゴリ名2}
- ...

## 出力形式
各検討事項について以下を記述:
- 検討事項名
- 対応する要件ID（SR-xxx / NFR-xxx / 制約-xxx、または「暗黙要件」）
- 検討の概要（何を決定する必要があるか）
- 検討の根拠（なぜこの検討が必要か。要件のどの記述から導出されるか）
- 初期的な選択肢（2-3案の概要。この時点での詳細検討は不要）
```

**出力ファイル**: `phase-0.5-1-{視点名}.md`（5ファイル）

**設計意図**:
- カテゴリ名のみの注入により、LLMのアテンションを広く分配しつつ、サブ項目レベルのアンカリングを回避（Step 3a advocate-bの「二段階注入」提案）
- 要件ID紐付き必須により、過剰検出を構造的に抑制（要件から導出不可能な検討事項は「暗黙要件」として明示が必要）
- 各サブエージェントは他の視点の出力を参照しない（コンテキスト独立性: Step 0d §1.4）

**Step 0dへの対応**:
| バイアス | 対処 |
|---------|------|
| アンカリング効果 | 5サブエージェント独立実行。他の視点の出力を入力に含めない |
| 訓練データ分布バイアス | カテゴリ名ガイドが低頻度ドメインの想起を促進 |
| アテンション配分バイアス | 視点ごとの分離実行により、要件書の記述量偏りの影響を緩和 |

### Phase 0.5-2b: 自由推論交差分析

**目的**: テンプレートに含まれない非標準的な要件間相互作用を検出する。

**サブエージェント数**: 1

**入力**:
1. `requirements.md`（全文のみ。Phase 0.5-1やPhase 0.5-2aの出力は含めない）

**プロンプト構造**:
```
以下の要件文書を読み、要件間の相互作用（トレードオフ、依存関係、
競合関係）を分析し、それぞれに起因する設計検討事項を導出してください。

特に以下に注意:
- 明示的に記述されていない暗黙的な相互作用
- 3つ以上の要件が絡む複合的な相互作用
- 非機能要件同士のトレードオフ

## 入力
{requirements.md の内容}

## 出力形式
各相互作用について:
- 関連する要件ID
- 相互作用の種類（トレードオフ / 依存 / 競合 / 複合）
- 内容の説明
- 導出される設計検討事項（検討事項名、概要、初期選択肢）
```

**出力ファイル**: `phase-0.5-2b-cross-freeform.md`

**設計意図**:
- Phase 0.5-2aの出力を入力に含めないことで、テンプレートパターンへのアンカリングを防止
- requirements.mdのみから自由推論することで、テンプレート外の相互作用を検出

### Phase 0.5-3a: 統合・構造化

**目的**: Phase 0.5-1（5ファイル）とPhase 0.5-2a/2bの出力を統合し、構造化する。

**サブエージェント数**: 1

**入力**:
1. `requirements.md`
2. Phase 0.5-1の出力（5ファイル）
3. Phase 0.5-2aの出力
4. Phase 0.5-2bの出力

**処理内容**:
1. **重複排除**: 同一の検討事項が複数の視点から導出されている場合を特定し、統合する。統合時は最も充実した記述を保持し、複数の視点からの根拠を集約する
2. **横断的関心事の分類**: 複数の視点にまたがる検討事項を「横断的関心事」として分類し、境界仕様を生成する
3. **要件トレーサビリティマトリクス**: 全要件IDに対応する検討事項の一覧を生成。対応なしの要件IDにはフラグを付与
4. **構造化**: 統合された検討事項リストを5視点+横断的+交差分析の構造で整理

**出力ファイル**: `phase-0.5-3a-integrated.md`

**出力構造**:
```
## 1. 検討事項リスト（視点別）
### 1.1 技術インフラ
- [検討事項1]: ...
- [検討事項2]: ...
### 1.2 ドメイン・ビジネスロジック
...
### 1.6 交差分析由来
...

## 2. 横断的関心事の境界仕様
### [セキュリティ]
  arch_designの責務: [方針レベルの決定事項]
  process_designの責務: [手順レベルの決定事項]
  相互依存: [process側がarch側の何に依存するか]
### [ロギング]
...

## 3. 要件トレーサビリティマトリクス
| 要件ID | 対応する検討事項 | カバレッジ状態 |
...

## 4. 統計サマリ
- 検討事項総数: N件
- 視点別件数: ...
- 暗黙要件由来: M件

### Phase 0.5-3b: リファレンスカテゴリ照合・内容充実度チェック

**目的**: 統合結果をリファレンスカテゴリのサブ項目レベルで照合し、カバレッジギャップと内容不足を検出する。

**サブエージェント数**: 1

**入力**:
1. Phase 0.5-3aの出力（`phase-0.5-3a-integrated.md`）
2. リファレンスカテゴリ定義（§3、サブ項目を含む完全版）

**処理内容**:
1. **カバレッジ照合**: 統合結果の各検討事項をリファレンスカテゴリのサブ項目と対応付ける
2. **内容充実度チェック**: 各リファレンスカテゴリについて、主要サブ項目ごとに以下を判定:
   - **該当**: このプロジェクトで検討が必要なサブ項目である
   - **非該当**: このプロジェクトの特性上、検討不要（理由を明記）
   - **該当・識別済み**: 導出結果に十分な検討事項がある
   - **該当・未識別**: 導出結果に対応する検討事項がない、または表面的すぎる → **不足フラグ**
3. **不足カテゴリの特定**: 「該当・未識別」のサブ項目を持つカテゴリを不足カテゴリとして特定

**出力ファイル**: `phase-0.5-3b-coverage.md`

**出力構造**:
```
## 1. カバレッジ照合結果

### カテゴリ: [アーキテクチャスタイル]（技術インフラ視点）
| サブ項目 | 判定 | 対応する検討事項 | 備考 |
|---------|------|----------------|------|
| モノリス/マイクロサービス選定 | 該当・識別済み | 検討事項#3 | — |
| サービス間通信方式 | 該当・未識別 | — | 不足フラグ |
| APIゲートウェイの要否 | 非該当 | — | 単一サービス構成のため |
...

## 2. 不足カテゴリサマリ
| カテゴリ名 | 視点 | 不足サブ項目数 / 該当サブ項目数 | 不足サブ項目リスト |
...

## 3. 統計
- 全カテゴリ数: 30
- 照合完了: 30
- 不足カテゴリ数: N
- 不足サブ項目総数: M
```

**設計意図**:
- Phase 0.5-1ではカテゴリ名のみを注入し、Phase 0.5-3bでサブ項目を使って検証する「二段階注入」（Step 3a advocate-b提案）
- 固定閾値（50%）ではなく、サブ項目ごとの該当性判定方式（Step 3a論点2の決定）
- 全カテゴリ均一適用（Step 0d §2.3 パターン完全適用）

### Phase 0.5-3c: 補完導出（条件付き実行）

**実行条件**: Phase 0.5-3bで不足カテゴリが検出された場合のみ実行。

**サブエージェント数**: 不足カテゴリ数に応じて1-3（各サブエージェントは不足カテゴリのグループを担当）

**入力**:
1. `requirements.md`
2. Phase 0.5-3bの不足カテゴリサマリ（不足カテゴリ名+不足サブ項目リスト）
3. 担当するリファレンスカテゴリの完全定義（サブ項目含む）

**注意**: Phase 0.5-3aの統合結果（検討事項リストの詳細）は入力に含めない。アンカリング防止のため、不足カテゴリと要件書から独立に導出する。

**プロンプト構造**:
```
以下の要件文書について、指定されたカテゴリの設計検討事項を導出してください。
これらのカテゴリは、先行する導出プロセスでカバレッジが不足していると判定されたものです。

## 入力
{requirements.md}

## 対象カテゴリと不足サブ項目
{不足カテゴリ名、不足サブ項目リスト}

## カテゴリの詳細定義
{リファレンスカテゴリのサブ項目定義}

## 出力形式
各サブ項目について:
- このプロジェクトに該当するか（該当/非該当+理由）
- 該当する場合: 検討事項名、対応する要件ID、検討の概要、根拠、初期選択肢
```

**出力ファイル**: `phase-0.5-3c-supplement-{n}.md`

### Phase 0.5-4: 敵対的検証（シナリオベース障害分析）

**目的**: 導出層と検証層で検出されなかった「未知の未知」を、障害シナリオからの逆推論で探索する。

**サブエージェント数**: 1

**入力**:
1. `requirements.md`（全文）
2. Phase 0.5-3aの「5視点の要約」（各視点2-3文。検討事項の詳細リストは含めない）
3. Phase 0.5-3bの「トレーサビリティギャップ情報」:
   - トレーサビリティ未カバー要件IDリスト
   - 不足カテゴリ名リスト（サブ項目の詳細は含めない）

**プロンプト構造**:
```
あなたはシステム障害分析の専門家です。以下の要件文書に基づくシステムについて、
本番稼働後に発生しうる障害シナリオを生成し、その障害を防ぐために
設計段階で検討すべきだった事項を逆推論で特定してください。

## 要件文書
{requirements.md}

## 現在の検討状況（概要のみ）
以下は各視点で検討された内容の要約です:
{5視点の要約}

## 注力領域
以下の領域は検討が特に不足している可能性があります:
- 要件カバレッジが不足: {未カバー要件IDリスト}
- カテゴリカバレッジが不足: {不足カテゴリ名リスト}

## 出力形式
5-10件の障害シナリオを生成し、それぞれについて:
- シナリオ名
- 障害の具体的な内容（何がどう壊れるか）
- 根本原因（設計段階で何が欠けていたか）
- 導出される設計検討事項（検討事項名、対応する要件ID、概要）
- このシナリオが既存の検討事項でカバーされている可能性（高/中/低）
```

**出力ファイル**: `phase-0.5-4-adversarial.md`

**設計意図**:
- 入力を「要約+ギャップ情報」に制限し、検討事項リストの詳細へのアンカリングを軽減（Step 3a advocate-aの中間案）
- ギャップ情報により「既に薄い領域」への重点配分を維持
- 「帰結→原因」の逆推論モードで、導出層の「原因→帰結」とは異なる認知経路を活用
- 「既存カバレッジ可能性」の自己評価は参考情報であり、Phase 0.5-5のユーザー判断を補助する

### Phase 0.5-5: ユーザー検証

**目的**: 統合された検討事項リストとカバレッジ分析結果をユーザーに提示し、検証を受ける。

**ユーザーインタラクション数**: 1-2

**提示内容（3部構成）**:

**Part 1: 検討事項リスト**
- Phase 0.5-3a/3cで統合・補完された検討事項リスト
- 視点別に整理、要件ID紐付き表示
- 横断的関心事の境界仕様を含む

**Part 2: カバレッジ分析結果**
- Phase 0.5-3bの照合結果の要約:
  ```
  ■ カバレッジ照合結果:
    - 全Nカテゴリ中 Mカテゴリ: 導出層で十分にカバー
    - Kカテゴリで追加検討事項を検出（Phase 0.5-3c由来）:
      - [カテゴリ名1]: [追加された検討事項の要約]
      ...
    - 該当なしと判定されたカテゴリ: L件
  ```
- トレーサビリティ未カバー要件の一覧（ある場合）

**Part 3: 敵対的検証からの追加事項**
- Phase 0.5-4で特定された検討事項のうち、「既存カバレッジ可能性: 低」のもの
- 各項目についてユーザーの追加判断を求める

**ユーザーへの質問**:
1. 検討事項リストに追加すべき項目はあるか
2. 不要と判断する項目はあるか（削除候補）
3. カバレッジ分析で「非該当」と判定されたカテゴリの判断は妥当か

### Phase 0.5-3(p): 統合・リファレンス照合 + 軽量敵対的検証

**目的**: process_designの導出結果を統合し、リファレンスカテゴリと照合する。加えて、軽量な敵対的検証を同一プロンプト内で実施する。

3. **軽量敵対的検証**: プロンプト末尾に以下を追加:
   ```
   最後に、このプロセス設計のもとで運用上の障害が起きるシナリオを3つ挙げ、

## 8. 制約テーブル

本方式が持つ構造的な制約と、その緩和策。

| ID | 制約 | 影響 | 緩和策 | 緩和策の実装箇所 |
|----|------|------|--------|----------------|
| L-1 | 同一LLMの知識源非独立性 | 全サブエージェントが共有する系統的盲点は検出不可 | ユーザー検証（知識源独立な最終安全網）、敵対的検証（認知経路の多様化） | Phase 0.5-4, Phase 0.5-5 |
| L-2 | 低頻度ドメインの検出力低下 | 訓練データに少ないドメインの検討事項が薄くなる | リファレンスカテゴリが外部基準として機能、カテゴリ名ガイドが想起を促進 | Phase 0.5-1（ガイド）, Phase 0.5-3b（照合） |
| L-3 | カテゴリ外検出の不確実性 | リファレンスカテゴリ外の検討事項の検出は動的導出と敵対的検証に依存 | Phase 0.5-1のオープンエンド導出、Phase 0.5-4のシナリオベース探索 | Phase 0.5-1（「限定せず」指示）, Phase 0.5-4 |
| L-4 | 3要件以上の複合的交差 | A-5仮定より、3要件以上が絡む相互作用の検出は困難 | Phase 0.5-2bの自由推論で部分的にカバー。構造的保証は不可 | Phase 0.5-2b |
| L-5 | 補完導出のコンテキスト独立性による重複 | Phase 0.5-3cが既存結果を参照しないため、重複した検討事項を生成する可能性 | Phase 0.5-3aの統合ステップで重複排除可能（ただし3cの結果は3aに戻さず、最終統合で処理） | Phase 0.5-5の最終提示で重複排除 |
| L-6 | process_designの敵対的検証がプロンプト拡張 | 同一コンテキスト内のためアンカリングリスクが残存 | 3シナリオの軽量な検証に限定し影響を最小化。コスト制約との兼ね合い | Phase 0.5-3(p) |
