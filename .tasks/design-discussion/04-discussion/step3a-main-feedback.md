## 方針レベルのハイブリッド案

### 基本構造: 「リファレンス補強型動的導出」

**核心思想**: 導出の主構造は動的推論（方式A）に置き、リファレンスカテゴリ（方式B）を構造的検証層として組み込む。これにより:
- A-1（LLM推論力）とA-4（分類枠組み）の両方を基盤とし、一方の崩壊時にも他方が安全網として機能する
- プロジェクト固有の適応性（方式Aの強み）と検出の安定性・下限保証（方式Bの強み）を両立する

### 論点ごとの方針決定

#### 論点1: 網羅性の根拠構造 → ハイブリッド二重基盤

- **導出層**: 方式Aの視点駆動型独立導出を採用。5視点×独立サブエージェントで構造的独立性を確保
- **構造的検証層**: 方式Bのリファレンスカテゴリを検証層として後段に配置。導出結果をリファレンスカテゴリのサブ項目と照合し、カバレッジギャップを検出
- **相互補完の構造**: 導出層がプロジェクト固有の検討事項を拾い、検証層が「業界標準で検討すべきだが導出から漏れた事項」を捕捉する

**判断根拠**: A-1とA-4の両方に依存するが、一方の崩壊が致命的にならない構造。導出層の出力が薄い領域をリファレンスカテゴリが補完し、リファレンスカテゴリ外の事項は導出層が捕捉する。

#### 論点2: 内容充実度チェック → リファレンスベース直接照合 + シナリオ補完

- **基盤**: 方式Bの内容充実度チェック（リファレンスサブ項目との照合）を全カテゴリに均一適用する。Step 0d §2.3（パターン完全適用）の教訓に直接対応。
- **閾値の修正**: 50%の固定閾値は採用しない。代わりに「主要サブ項目のうち、該当するにもかかわらず未識別のものがないか」という判定に変更。50%/60%のような数値基準ではなく、各サブ項目の該当/非該当をLLMが判定し、「該当するが未識別」を検出する方式。
- **シナリオ補完**: 敵対的検証層（論点3）が追加の深さ検証として機能

**判断根拠**: 直接照合は安定性と再現性で優れ、全カテゴリ均一適用が可能。ただし固定閾値の恣意性（方式Aの指摘）は妥当なため、閾値ではなく該当性判定に変更。

#### 論点3: 敵対的検証層 → 採用（構造改良あり）

- **採用**: 方式AのPhase A4（シナリオベース障害分析）を検証層の一部として採用
- **構造改良**: Phase A4のアンカリング問題に対処するため、Phase A4への入力を「requirements.md + 5視点の要約（検討事項リストの詳細ではなく、各視点の要約のみ）」に制限。導出済みリストの詳細をコンテキストに含めないことで、アンカリングを構造的に軽減
- **位置づけ**: 「未知の未知」への能動的探索メカニズムであり、リファレンスカテゴリ照合（既知の未知への対処）を補完する

**判断根拠**: 1サブエージェント追加のコストで「未知の未知」への検出機会を得る。方式Aのラウンド2で「狭い範囲だが無価値ではない」と主張され、方式Bもこの範囲の無価値性は主張していない。

#### 論点4: 交差分析 → テンプレート + 自由推論のハイブリッド

- **標準パターンの安定検出**: 方式Bの交差パターンテンプレート（事前定義）を基盤とし、標準的な相互作用の検出を構造的に保証
- **探索的分析**: テンプレート照合に加えて、独立サブエージェントによる自由推論の交差分析を並列実行。テンプレートの出力を入力に含めない（アンカリング防止）
- **統合**: テンプレート結果と自由推論結果を統合・重複排除

**判断根拠**: 方式Aのラウンド2で「両方式の統合を検討すべき」と明示的に提案。方式Bの下限安定性と方式Aの探索力を組み合わせる。

#### 論点5: 安全網の設計 → リファレンスカテゴリで防御範囲を最大化

- **方式B寄り**: リファレンスカテゴリ（外部基準ベース）が「既知の未知」をカバーすることで、A-3b（ユーザーの欠落検出能力）への依存範囲を「リファレンスカテゴリ外のみ」に限定
- **補完**: 敵対的検証層（論点3）がリファレンスカテゴリ外の一部を能動的に探索
- **最終安全網**: ユーザー検証を残すが、主にVerification（提示事項の妥当性確認）に位置づけ、Gap Detection（欠落の発見）は構造的メカニズムに委ねる

**判断根拠**: A-3bは原理的に脆弱（Step 0c 指摘4）であり、高依存を避ける方式Bの構造が安全。

#### 論点6: カテゴリ外検出 → 多層の能動的メカニズム

- **導出層**: 方式Aの視点レンズによるオープンエンド推論（カテゴリに限定しない）
- **自由推論の交差分析**: 論点4の探索的分析がカテゴリ外の相互作用を検出する可能性
- **敵対的検証**: 論点3のシナリオベース分析がカテゴリ外の暗黙要件を検出する可能性
- **方式Bの補完指示は不採用**: 同一コンテキスト内でリファレンス走査後に「追加で」と問う構造は、方式Aのラウンド2で指摘されたアテンション配分バイアスの問題が残るため不採用

**判断根拠**: カテゴリ外検出は導出層と敵対的検証層の能動的メカニズムに委ね、検証層（リファレンス照合）にカテゴリ外検出機能を持たせようとしない。役割の明確な分離。

#### 論点7: 過剰検出の管理 → 要件紐付き導出 + 構造化提示

- **導出層**: 方式Aの要件紐付き（要件IDとの対応を必須化）で関連性密度を確保
- **検証層**: 方式Bのカバレッジマトリクス形式で構造化提示。ただし30カテゴリの全走査ではなく、導出結果とリファレンスカテゴリの照合結果のみ提示（不足カテゴリのハイライト）
- **ユーザー提示**: 「プロジェクトに関連する検討事項リスト」+「リファレンス照合で不足が検出されたカテゴリ」+「シナリオベースで追加された事項」の3部構成

## ハイブリッド案のフロー概要

### arch_design

```
Phase 0.5-1: 視点駆動型独立導出（5サブエージェント並列）
  → 5視点それぞれから検討事項を要件紐付きで導出

Phase 0.5-2a: 交差パターンテンプレート照合（1サブエージェント）
Phase 0.5-2b: 自由推論交差分析（1サブエージェント、Phase 1/2aと独立）
  → 2a+2bの結果を統合

Phase 0.5-3: 統合・リファレンスカテゴリ照合（1サブエージェント）
  → 導出結果をリファレンスカテゴリと照合、カバレッジギャップ検出、内容充実度チェック
  → 不足カテゴリの補完導出（別サブエージェント、コンテキスト独立）

Phase 0.5-4: 敵対的検証（1サブエージェント）
  → requirements.md + 5視点要約から障害シナリオ生成、追加検討事項の特定

Phase 0.5-5: ユーザー検証（1インタラクション）
  → 検討事項リスト + カバレッジ照合結果 + シナリオ追加事項を提示

合計: 8〜11サブエージェント、1〜2ユーザーインタラクション
```

### process_design

```
Phase 0.5-1: プロセス視点導出（2-3サブエージェント並列）
Phase 0.5-2: arch→process整合性検証 + process→arch逆流検出（2サブエージェント並列）
Phase 0.5-3: 統合・リファレンス照合（1サブエージェント）
Phase 0.5-4: ユーザー検証（1インタラクション）

合計: 5〜6サブエージェント、1ユーザーインタラクション
```

## A/Bへの質問

1. **方式A**: 敵対的検証層への入力制限（検討事項リストの詳細ではなく要約のみ）はアンカリング軽減として有効か。情報量の低下で検出力が下がるリスクをどう評価するか。
2. **方式B**: リファレンスカテゴリ照合を「導出後の検証」として後段に配置する設計について、リファレンスカテゴリを「導出のガイド」（プロンプト注入）としても使う方がカバレッジが上がるのではないか。
