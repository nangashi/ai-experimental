# 弁論ラウンド1: 方式Aの主張

## 論点1: 網羅性の根拠構造 — LLM推論力 vs 外部基準の分類枠組み

### 方式Aの優位性

方式Aの網羅性は**A-1（LLM推論力）+ 構造的多層検証**に依拠し、A-4（分類枠組みの網羅性）への依存が**構造的に低い**。これは以下の理由で優れた設計判断である。

**1. A-4の脆弱性は本質的である**

Step 0 §4のA-4の定義を見ると、「崩れた場合の影響」は「分類外の検討事項が系統的に見落とされる」である。方式BはA-4への依存を「最高」と自認しており（方式B §6）、これはリファレンスカテゴリ外の検討事項が系統的に検出不能になるリスクを構造的に内包することを意味する。

Step 0bのカバレッジマップ自体がLLMの支援で構築されている以上、このマップが「LLMの系統的盲点」を含む可能性を排除できない。外部基準（ISO 25010等）を根拠とすることで品質を担保しているという方式Bの主張は、ISO 25010等の外部基準自体が**汎用フレームワーク**であり、プロジェクト固有の検討次元（例: 特定のビジネスドメインにおける規制間の相互作用）をカバーしない事実を看過している。

**2. A-1はA-4より検証可能性が高い**

A-1（LLM推論力）は、具体的なプロジェクトのrequirements.mdに対して実行することで実効性を事後的に検証できる。一方、A-4（分類枠組みの網羅性）は「網羅的であること」の証明が原理的に困難である（Step 0 §1.3 条件4の「未知の未知」問題）。方式AがA-1に依存することは、より検証可能な仮定に依存していることを意味する。

**3. 方式Aの三層構造は検出機会を多重化する**

方式Aの導出層（視点駆動推論）→ 交差検証層（トレーサビリティ + 要件間相互作用）→ 敵対的検証層（シナリオベース障害分析）は、それぞれ**異なる認知モード**で漏れを探索する。これは「同一のチェックリストを精密に走査する」方式Bよりも、検出メカニズムの多様性において構造的に優れている。

### 方式Bの弱点

方式Bの網羅性は「リファレンスカテゴリの完全性」に一元的に依存する。Step 0 §1.3 条件4（暗黙要件の明示化）について、方式Bは「リファレンスカテゴリのサブ項目がヒントとして機能し、暗黙要件の検出を促す」と主張するが、これはリファレンスカテゴリ自体に暗黙要件に対応するサブ項目が含まれている場合にのみ機能する。バックアップ戦略のように広く認知された暗黙要件は含まれるだろうが、プロジェクト固有の暗黙要件（特定のビジネスプロセスに伴うデータ整合性要件等）はリファレンスカテゴリに含まれ得ない。

## 論点2: 内容充実度チェックの設計

### 方式Aの優位性

方式AのPhase A4（シナリオベース障害分析）は、内容充実度を**外部的な基準**——「この設計でシステムが実際に障害を起こすか」——で検証する。

**1. 障害シナリオは自己参照的でない**

Step 0d §2.2は自己参照的基準の問題を明確に警告している。方式Bの「主要サブ項目の50%カバレッジ」は、基準自体（サブ項目リスト）がリファレンスカテゴリに依存しており、リファレンスカテゴリの品質に循環的に依存する。一方、「このシステムが障害を起こすシナリオ」は、リファレンスカテゴリとは独立した基準であり、自己参照的循環を構造的に回避する。

**2. 「深さ」の検証として直接的**

検討事項の深さが不足していると、設計判断が曖昧になり、実装段階で問題が発生する。この問題の最も直接的な検証は「曖昧な設計判断のまま進んだ場合に何が起きるか」をシナリオで確認することである。方式Bの「サブ項目の50%カバレッジ」は構造的な代理指標（proxy metric）であり、50%の基準自体に根拠がない。49%でも十分な場合と、50%でも不十分な場合がある。

### 方式Bの弱点

方式Bの内容充実度チェック（§3.4）には以下の構造的問題がある。

**1. 50%閾値の恣意性**: 「主要サブ項目の50%以上」という基準は外部的根拠を持たない。なぜ50%なのか、なぜ40%や60%ではないのかの理由が不明である。この閾値はプロジェクトの性質に依存すべきだが、全カテゴリ均一に適用される。

**2. サブ項目の存在と検討の深さは別次元**: サブ項目が「識別」されることと、そのサブ項目が「十分に検討」されることは異なる。方式Bは「具体的な判断事項として記述されている」という具体性基準を設けているが、この判定自体がLLMの評価に依存し、明確な閾値を持たない。

## 論点3: 敵対的検証層の費用対効果

### 方式Aの優位性

Phase A4（シナリオベース敵対的探索）は、方式A固有の検証メカニズムであり、以下の理由でコスト（1サブエージェント呼び出し）に見合う価値を持つ。

**1. 異なる認知モードによる盲点の補完**

導出層（Phase A1）は「要件 → 検討事項」の順方向推論、交差検証層（Phase A3）は「要件 × 検討事項の突合」という構造的検証を行う。Phase A4は「障害シナリオ → 未識別の原因 → 欠落した検討事項」という**逆方向推論**を行う。この認知モードの切り替えは、Step 0d §1.4の「コンテキスト独立性」とは別の次元での独立性を提供する。知識源は同一でも、推論の方向が異なれば異なる結果を生む可能性がある。

**2. 暗黙要件の検出に対する唯一の構造的手段**

Step 0 §1.3 条件4（暗黙要件の明示化）は「原理的に完全性を保証できない」と明記されている。この困難な条件に対して、方式Aは**唯一の構造的検出手段**としてPhase A4を用意している。方式Bはリファレンスカテゴリのサブ項目提示に依存するが、サブ項目に含まれない暗黙要件に対する構造的検出メカニズムを持たない。

**3. コストは限定的**

Phase A4は1サブエージェント呼び出しであり、全体（arch_design 8回）の12.5%に過ぎない。Phase A1（5回）やPhase A3（1回）と並列実行はできないが（A3の出力が必要）、逐次実行でも追加コストは許容範囲内である。

### 方式Bの弱点

方式Bは敵対的検証層を持たない。この設計判断の理由は「コスト効率の優先」と推定されるが、代償として暗黙要件の検出に対する構造的安全網を欠く。方式Bの課題4（同一LLMの知識源限界）への対策は「リファレンスのサブ項目によるヒント」と「ユーザーへの提示」のみであり、これはPhase A4のようなアクティブな探索メカニズムとは質的に異なる。

方式Bのアプローチは「すでに知っている検討事項の確認」（リファレンスとの突合）に特化しており、「まだ知らない検討事項の発見」に対する構造が弱い。

## 論点4: 交差分析の構造 — 自由推論 vs テンプレート駆動

### 方式Aの優位性

方式AのPhase A2は、要件のみを入力とする独立サブエージェントで交差分析を行う。この設計には以下の利点がある。

**1. プロジェクト固有の相互作用パターンを発見可能**

方式Bの交差パターンテンプレートは10パターンが事前定義されている。しかし、要件間の相互作用はプロジェクトの要件構成に強く依存する。例えば「リアルタイム通知 × データプライバシー × マルチテナンシー」のような3要件の複合的相互作用は、10パターンのテンプレートには含まれない。方式Aの自由推論はこのようなプロジェクト固有の相互作用を原理的に発見可能である。

**2. アンカリング防止の構造的設計**

Phase A2はPhase A1の出力を**一切含めない**独立サブエージェントである。これはStep 0d §1.1（アンカリング効果）への直接的な構造的対策であり、Phase A1の結果に引きずられず、要件文書から独立に相互作用を発見できる。

**3. テンプレート外の交差パターンの構造的検出不能性**

方式Bの交差パターンテンプレートはA-4と同様の構造的脆弱性を持つ。テンプレートに含まれない交差パターンは構造的に検出不能である。方式Bはこの問題を認識し「テンプレート外の交差はLLMの推論能力に依存」（方式B §6, A-5の行）と記しているが、テンプレート駆動の処理内でLLMにテンプレート外の交差を「自由に」発見させることは、アンカリング効果（テンプレートに列挙されたパターンに引きずられる）により実効性が疑わしい。

### 方式Bの弱点

方式Bの交差パターンテンプレートは検出の安定性を提供するが、**検出の天井**を設定してしまう。10パターンの固定リストは、Step 0 §1.3 条件3（交差分析完了）の充足を「10パターンの照合完了」に矮小化するリスクがある。また、テンプレートの検出頻度列（高/中/低）は、LLMの訓練データ分布バイアス（Step 0d §1.2）を反映している可能性が高く、低頻度とラベル付けされたパターン（国際化×ドメイン、アクセシビリティ×性能）が実際には特定プロジェクトで重要である場合にも軽視される構造になっている。

## 論点5: A-3b（ユーザーの欠落検出能力）への依存構造

### 方式Aの優位性

方式AがA-3bへの依存を**高い**と明言していることは、弱点ではなく**誠実な設計分析**である。以下の論拠でこれを主張する。

**1. 方式BのA-3b低依存は条件付きである**

方式Bは「A-3b依存が低い」と主張するが、これは**リファレンスカテゴリ内**に限定された主張である。方式B自身が認めるように（方式B §6, A-3bの行）、「リファレンスカテゴリ外の検討事項の検出はユーザーに依存する」。つまり、方式BのA-3b低依存は「A-4が成立する範囲内でのみ」成立する条件付きの主張であり、A-4が崩れた瞬間にA-3bへの依存が顕在化する。

**2. A-3b依存の実質は両方式で大きく変わらない**

系統的盲点（Step 0 §2.4 項目4「同一盲点の共有」）は、いずれの方式でもLLM内部の多層構造だけでは解消できない。Step 0d §1.4が明示するように、知識源独立性を確保するには「異なるLLMモデルの使用、人間の専門家レビュー」が必要である。方式Bのリファレンスカテゴリも結局はLLMの知識（外部基準の解釈を含む）に依存しており、真の知識源独立性を提供しない。したがって、系統的盲点に対する最終安全網としてのユーザー依存は**両方式に共通の構造的制約**であり、方式Bの「低依存」は外見上の差異に過ぎない。

**3. 方式Aはユーザー検証の効果を最大化する設計を持つ**

方式Aは敵対的検証層（Phase A4）の結果をユーザーに提示する。障害シナリオという具体的な形式は、抽象的なカテゴリリストよりもユーザーのドメイン知識を活性化しやすい。「このシナリオで問題が起きないか？」という問いは、「このカテゴリに追加の検討事項はないか？」という問いよりも、ユーザーの具体的な知識（過去のインシデント経験等）を引き出す効果がある。つまり、A-3bへの依存が高いことを前提に、ユーザーの欠落検出能力を**最大限に引き出す**設計を方式Aは持っている。

### 方式Bの弱点

方式BのA-3b低依存の主張は、カバレッジマトリクスという「安心感」を提供する一方で、リファレンスカテゴリ外の盲点に対する**虚偽の安全感**を生むリスクがある。カバレッジマトリクスが全て「対応済み」を示した場合、ユーザーは「網羅的に検討された」と判断し、カテゴリ外の検討事項を積極的に探索しなくなる可能性がある。これは方式Bの構造が意図せずA-3bの実効性を**低下**させるパラドックスである。

## 論点6: リファレンスカテゴリ外の検討事項の検出メカニズム

### 方式Aの優位性

**1. オープンエンドな推論が主構造である**

方式Aの導出層は「この要件からどのような検討事項が導出されるか」というオープンエンドな問いを主構造とする。視点レンズは推論の**方向**を制約するが、推論の**範囲**を制約しない。「技術インフラ視点から検討事項を導出せよ」という指示は、Step 0bの技術インフラ分類次元に列挙された項目に限定するものではなく、プロジェクト固有の技術的検討事項（例: 特定のレガシーシステムとの互換性要件）も自然に導出される。

**2. 多層構造がカテゴリ外検出の機会を多重化する**

方式Aでは、カテゴリ外の検討事項を発見する機会が複数ある:
- Phase A1: 各視点レンズの範囲内でのオープンエンド推論
- Phase A2: 要件間相互作用からの自由推論
- Phase A4: シナリオベース障害分析からの帰結→原因推論
- Phase A5: ユーザーのドメイン知識

各機会は異なるメカニズムで動作するため、一つの機会で見逃された事項が別の機会で発見される可能性がある。

### 方式Bの弱点

方式Bの「リファレンスカテゴリ外の検討事項があれば追加で挙げよ」という補完指示は、**構造的なアンカリング問題**を抱えている。

Step 0d §1.1が明示するように、LLMは先に提示された情報にアテンションが引きずられる。フェーズA-1のサブエージェントは、まずリファレンスカテゴリ（8カテゴリ、各カテゴリに3-5のサブ項目）を詳細に走査した**後に**、「それ以外の事項はないか」と問われる。この時点でLLMのアテンションはリファレンスカテゴリの枠組みに強くアンカリングされており、枠組み外の事項を自由に発想する能力は大幅に低下している。

これは指示的独立性（Step 0d §1.4）の問題の変形である。「リファレンスカテゴリを走査した上で、カテゴリ外も見よ」という指示は、同一コンテキスト内での「先の結果に依存せず独立に分析せよ」と本質的に同じ構造であり、Step 0d §1.1が明示する「機能しない」パターンに該当する。

つまり、方式Bのカテゴリ外検出メカニズムは**方式の主構造（リファレンス駆動）と構造的に矛盾**しており、補完指示としての実効性は低いと判断せざるを得ない。

## 論点7: 過剰検出のコスト

### 方式Aの優位性

**1. 要件紐付きが構造的フィルタとして機能する**

方式AのPhase A1は、各検討事項に「対応する要件ID」と「なぜこの要件からこの事項が導出されるかの推論根拠」の記載を必須としている。この構造により、要件に紐付かない検討事項は導出段階で自然にフィルタリングされる。結果として、ユーザーに提示される検討事項はプロジェクトに関連するもののみとなり、注意の希釈が生じにくい。

**2. ユーザーの認知負荷が低い**

方式Aが提示する検討事項リストは、要件文書から導出されたプロジェクト固有の事項で構成される。ユーザーは「自分のプロジェクトに関連する事項」のみを確認すればよく、「該当しない汎用事項を却下する」作業が不要である。

Step 0 §3.4（コスト許容範囲）は「ユーザー待ち時間」を明示的に制約としている。ユーザーがPhase A5で確認すべき内容が過剰でないことは、ユーザーインタラクションの実効性を高め、A-3a/A-3bの実効性向上にもつながる。

### 方式Bの弱点

**1. 全カテゴリ走査の構造的コスト**

方式Bのarch_designリファレンスカテゴリは5視点×5-8カテゴリ = 約30カテゴリ、各カテゴリに3-5のサブ項目で、合計100以上のサブ項目に対する判定が必要である。小規模なWebアプリケーションプロジェクトでも、「データ所在地: クロスボーダー転送制約」「ワークフロー/サガ: 補償トランザクション設計」等、明らかに非該当のサブ項目への判定コストが発生する。

**2. 事前フィルタリングの自己矛盾**

方式Bは課題2（過剰検出のコスト）への対策として「プロジェクト特性による事前フィルタリング」を提案しているが、これは本質的に**方式Bの主構造（全カテゴリ網羅的走査）と矛盾**する。フィルタリングでカテゴリを除外することは、除外されたカテゴリの検討事項を構造的に検出不能にする。フィルタリングの判断自体がLLM推論に依存するため、「LLM推論で除外したカテゴリに、LLM推論では気づけない重要な検討事項があった」というシナリオが成立する。

方式Bはこのリスクを認識し「フィルタリング結果をユーザーに提示し確認を得る」（ユーザーインタラクション1回消費）と対策を述べているが、これはA-3b依存の増加であり、方式Bの「A-3b依存が低い」という主張と整合しない。

**3. 注意の希釈効果**

Step 0 §1.2で「過剰検出は致命的リスクを生まない」としつつも「コスト面の制約として間接的に考慮する」とある。過剰検出の間接的リスクとして最も重要なのは**注意の希釈**である。100以上のサブ項目を確認するユーザーは、後半の項目に対する注意力が低下し、重要な「該当あり」判定を見逃すリスクがある。これはStep 0d §1.3（アテンション配分バイアス）のユーザー側への適用であり、LLMだけでなく人間にも作用するバイアスである。
