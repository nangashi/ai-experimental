# Step 3b: 具体手順の詳細設計

## 基本方針（Step 3a統合案 + フィードバック反映）

「要件駆動導出（主） + リファレンス検証（補完）」の2層方式。Step 3aのフィードバックを受けて以下を修正:

1. **リファレンスの粒度**: 「例示のみ」→「確認チェック項目」(3-5個/カテゴリ)に変更。カテゴリ内の具体的な識別漏れを検出可能にする
2. **検証層のバイアス独立性**: リファレンスの各項目を強制提示して確認するプロンプト設計。LLMの自己評価に依存しない
3. **追加導出**: 視点切り替えプロトコルを定義
4. **Step 4/Step 5の分離**: 適用済みバイアスを防ぐため独立実行

---

## 1. リファレンスの構造設計

### 1.1 リファレンスの種別

arch_design用とprocess_design用のリファレンスを分離する。

**arch_design用リファレンス**: アーキテクチャ設計フェーズで検討すべきカテゴリと確認項目
**process_design用リファレンス**: 開発プロセス設計フェーズで検討すべきカテゴリと確認項目

### 1.2 エントリ構造

```yaml
- category: "{カテゴリ名}"
  trigger: "{適用トリガー条件}"
  arch_concerns: "{arch側で決定すべき方針レベルの関心事}"
  process_concerns: "{process側で決定すべき実装レベルの関心事}"
  check_items:
    - "{確認チェック項目1}"
    - "{確認チェック項目2}"
    - "{確認チェック項目3}"
  # check_itemsは「この項目が導出結果に存在するか」の確認に使う
  # 網羅的リストではなく、見落としやすい重要項目を選定する
```

### 1.3 カテゴリ一覧（arch_design用の例示）

以下はStep 3bの設計段階での想定であり、実装時に精査が必要。

| # | カテゴリ | トリガー条件 | 確認チェック項目の例 |
|---|---------|------------|-------------------|
| 1 | アーキテクチャスタイル | 常に適用 | モノリス/マイクロサービス選択、通信パターン、サービス境界 |
| 2 | 実行基盤・インフラ | 常に適用 | ホスティング形態、スケーリング戦略、リージョン設計 |
| 3 | データ管理 | データ永続化あり | データストア選定、データモデリング方針、バックアップ/リカバリ |
| 4 | セキュリティ | 常に適用 | 認証方式、認可モデル、データ保護、通信暗号化 |
| 5 | 可観測性 | バックエンドサービスあり | 監視戦略(SLI/SLO)、ログ集約方針、分散トレーシング採否 |
| 6 | 外部統合 | 外部システム連携あり | インターフェース契約、エラー伝播、リトライ戦略 |
| 7 | 信頼性・可用性 | NFRに可用性要求あり | 冗長構成、障害分離、フェイルオーバー |
| 8 | パフォーマンス | NFRに性能要求あり | キャッシュ戦略、非同期処理、ボトルネック対策 |
| 9 | デプロイ・リリース | 常に適用 | デプロイ戦略、環境構成、リリースプロセス |
| 10 | API・インターフェース設計 | 外部/内部API提供あり | API設計方針、命名規則、バージョニング |

カテゴリ数は10〜15を目安とする（コンテキスト制約内に収まる軽量さ）。

### 1.4 メンテナンス方針

- **更新トリガー**: 検証層で「導出されたがリファレンスにないカテゴリ」が繰り返し発生する場合
- **更新範囲**: カテゴリ追加、トリガー条件の精緻化、確認チェック項目の追加/修正
- **更新頻度**: 設計実績を蓄積し、一定件数ごとにレビュー（頻繁な更新は不要）

### Phase 0: 初期化・要件読込（変更なし）

現行のPhase 0と同一。要件文書の存在・品質検証、構造サマリの抽出を行う。

**出力**: 構造サマリ（sr_list, nfr_list, constraints, actors_and_interfaces等）

### Phase 1-A: 要件駆動の設計次元導出（導出層）

**目的**: 要件文書を起点に、このプロジェクトに必要な設計次元を動的に導出する。

#### Step 1: 直接導出

**入力**: Phase 0の構造サマリ
**処理**:
```
各SR/NFR/制約に対して以下の問いを適用:
「この要件を実現するために、アーキテクチャレベルで決定しなければならない
 方針・技術・構成・インターフェースは何か？」

各要件から0〜N個の設計検討事項（候補）を導出する。
各候補は以下の形式で記録:
  - 対象: 何を決めるか
  - トリガー: どの要件から導出されたか（SR/NFR/制約のID）
  - 判断基準候補: 関連するNFR/制約
```

**粒度正規化ルール**:
- decision point粒度 = 「実装チームが実装前に方針として決定していなければならない事項」
- 過細（例: 変数命名規則）→ process_designへ委譲マーク
- 過粗（例: 「セキュリティ全般」）→ サブ検討事項に分解（「認証方式」「認可モデル」等）

**出力**: 要件トレーサブルな設計検討事項の候補リスト

#### Step 2: 暗黙要件の補完

**入力**: 構造サマリの技術コンテキスト（前提スタック・ドメイン・運用環境） + Step 1の候補リスト
**処理**:
```
問い1:
「このプロジェクトの技術コンテキスト（{ドメイン}, {スタック}, {構成}）を前提とした場合、
 要件に明示されていないが、設計で必ず対処すべき事項は何か？」

問い2:
「以下のNFR間の相互作用で、追加の設計判断が必要になる組み合わせはあるか？
 {nfr_list}」
```

**出力**: 暗黙要件から導出された追加検討事項（トリガーは「技術コンテキスト: {根拠}」）

#### Step 3: 設計次元の整理と依存関係分析

**入力**: Step 1 + Step 2の検討事項候補
**処理**:
```
1. 候補を設計次元にグルーピング
   （同一次元内の検討事項は相互に強く依存する）
2. 各次元の定義:
   - 決定対象
   - 対象外（他の次元で扱う or スコープ外）
   - 依存先（他の次元の決定結果に依存するもの）
3. 次元間の依存関係に基づく議論順序の決定
4. 整合性チェック:
   - 全SR/NFRがいずれかの次元でカバーされるか
   - 次元間の重複がないか
   - 依存に循環がないか
```

**出力**: スコープ分割計画（設計次元一覧 + 依存関係 + 議論順序）

### Phase 1-B: リファレンス検証（検証層）

**目的**: 導出結果をリファレンスと照合し、識別漏れを検出する。導出層とは独立した情報源として機能する。

#### Step 4: リファレンスカテゴリとの照合

**入力**: Phase 1-Aの導出結果 + リファレンス（arch_design用）
**実行**: 導出層と独立したLLM呼び出しで実行する（同一コンテキスト内で導出→検証を連続しない）

**処理**:
```
リファレンスの各カテゴリについて以下を実行:

1. トリガー条件の照合
   「このカテゴリのトリガー条件（{trigger}）はこのプロジェクトの要件に該当するか？」
   → 不該当の場合: 不適用（理由コード NC）として記録し、次のカテゴリへ

2. 確認チェック項目の照合（トリガー該当の場合）
   各確認チェック項目について:
   「以下の検討事項は、導出結果に含まれているか？
    確認項目: {check_item}
    導出結果: {導出された設計検討事項のリスト}」
   → 含まれている: 「確認済み」
   → 含まれていない:
     「この項目はこのプロジェクトで検討が必要か？」
     → 必要: 「導出漏れ」→ Step 4b（追加導出）へ
     → 不要: 不適用（理由コード NC/NR/OOS）として記録
```

**プロンプト設計の重要ポイント（バイアス独立性の確保）**:
- リファレンスの各項目を**1項目ずつ強制提示**して確認する（LLMに「全体として問題ないか」と問わない）
- 導出結果を「このリストは完成している」と前提しない問いの形式にする
- 「含まれていない場合」に即「不適用」とせず、必要性を改めて判断させる

**不適用の理由コード**:
| コード | 定義 | 例 |
|--------|------|-----|
| NC | アーキテクチャパターン/コンテキストが非該当 | モノリスに分散トレーシング |
| NR | 要件で明示的に除外/不要とされている | NFRでSLA不要と明記 |
| OOS | このフェーズの責務範囲外 | 実装詳細はprocess_designで |

**出力**: 照合結果テーブル（各カテゴリ × 各確認項目の判定 + 理由コード）

#### Step 4b: 追加導出（視点切り替えプロトコル）

Step 4で「導出漏れ」と判定された項目について、追加の導出を行う。

**処理**:
```
追加導出プロトコル:
1. 「なぜこの項目が最初の導出で識別されなかったか」の仮説生成
   候補:
   - 要件に直接的な言及がなく、暗黙の技術的要求だった
   - 別の次元の検討事項として統合されていた
   - 技術コンテキストからの推論が不足していた

2. 仮説に基づいて異なる視点から再導出
   視点切り替えオプション:
   - 実装者視点: 「このシステムを実装するエンジニアが、この項目なしで作業できるか」
   - 運用者視点: 「このシステムを運用するチームが、この項目なしで運用できるか」
   - 外部ユーザー視点: 「利用者がこのシステムを使うとき、この項目は影響するか」

3. 再導出結果を導出リストに追加
   - 追加項目にはトリガーとして「検証層Step 4bで追加（視点: {使用した視点}）」を記録
```

**出力**: 追加検討事項 + 更新された導出結果リスト

#### Step 5: 飽和スキャン

**入力**: Phase 1-Aの導出結果 + 要件文書（Step 4の照合結果は入力しない）
**実行**: Step 4とは独立したLLM呼び出しで実行する

**処理**:
```
以下の4軸でスキャンし、導出結果に未検討領域がないか確認:

1. データフロー軸:
   「データの入力→処理→出力→永続化の各段階で、
    設計方針が未定のまま残っている段階はないか？」

2. インフラ軸:
   「実行環境・スケーリング・デプロイ形態で、
    設計方針が未定のまま残っている事項はないか？」

3. 外部連携軸:
   「外部システムとのインターフェース・契約・障害対応で、
    設計方針が未定のまま残っている事項はないか？」

4. 横断的関心事軸:
   「セキュリティ・可観測性・エラー処理・国際化等で、
    設計方針が未定のまま残っている事項はないか？」

各軸について:
- 未検討領域が検出された場合: 追加の検討事項として記録
- 検討済みと判断された場合: 対応する導出結果を参照して確認
```

**出力**: 飽和スキャン結果（各軸の検出事項 or 「飽和」判定）

### Phase 1-C: スコープ分割計画のユーザー確認

Phase 1-A〜1-Bの結果を統合し、最終的なスコープ分割計画をユーザーに提示する。

**処理**:
```
1. Phase 1-Aの導出結果 + Phase 1-Bの追加項目を統合
2. 設計次元一覧を再整理
3. 以下を含むスコープ分割計画を提示:
   - 設計次元テーブル（次元名・決定対象・対象外・依存先・ADR見込み）
   - リファレンス照合結果サマリ（カテゴリ別の適用/不適用）
   - 不適用判定の記録（カテゴリ + 項目 + 理由コード）
   - 飽和スキャン結果
```

**ユーザーに提示するテーブル**:
```
### スコープ分割計画

| # | 設計次元 | 決定対象 | 依存先 | ADR見込み |
|---|---------|---------|--------|----------|
| ... | ... | ... | ... | ... |

### リファレンス照合結果

| カテゴリ | 適用 | 確認済み項目 | 不適用項目（理由） |
|---------|------|------------|------------------|
| ... | ... | ... | ... |

### 飽和スキャン結果

| 軸 | 結果 | 追加事項 |
|----|------|---------|
| ... | ... | ... |
```

ユーザーの承認後、Phase 1-Dへ。

2. 「確定した設計検討事項一覧」をprocess_designの入力として構造化

### Phase 1-A: arch決定事項からのprocess検討事項導出（導出層）

#### Step 1: シードタグ付き項目の処理（優先）

```
arch設計完了記録のシードタグ付き項目それぞれについて:
「このarch決定（{決定内容}）を実装として実現するために、
 チームが揃えるべきプロセス・ツール・規約・自動化は何か？」
→ process検討事項を導出
```

#### Step 2: 全arch決定事項からのprocess検討事項導出

```
シードタグなしのarch決定事項についても同様の問いを適用:
「このarch決定（{決定内容}）は、process設計にどのような影響を与えるか？
 ツール選定・規約・CI/CD・品質ゲートの観点で具体化すべき事項はあるか？」
```

#### Step 3: プロセス固有の暗黙要件補完

```
「このプロジェクトの技術スタック（{tech_stack}）と開発体制を前提に、
 arch決定から導出されないが、開発プロセスとして定めるべき事項は何か？」

対象領域:
- チーム規模/体制起因の規約（ブランチ戦略、コードレビュー等）
- ツールチェーン統合の整合性
- 環境管理方針
```

### Phase 1-B: リファレンス検証（検証層）

#### Step 4: process用リファレンスとの照合

arch_designと同じプロトコルで、process_design用リファレンスとの照合を実施。
不適用判定に理由コードを必須とする。

#### Step 5: ハンドオフ検証

**入力**: arch設計完了記録の「確定した設計検討事項一覧」 + Phase 1-Aの導出結果
**処理**:
```
arch設計完了記録の全決定事項に対して:
「このarch決定（{決定内容}）に対応するprocess検討事項は、
 導出結果に存在するか？」

判定:
- 対応あり: 確認済み
- 対応なし + シードタグあり: 「引き継ぎ漏れ（Handoff Gap）」→ 追加導出
- 対応なし + シードタグなし: 「対応不要の可能性あり」→ 必要性を判断
  → 必要: 追加導出
  → 不要: 不適用（理由コード付き）
```

**出力**: ハンドオフ検証結果テーブル

## 5. 数値パラメータの根拠

| パラメータ | 値 | 根拠 |
|-----------|-----|------|
| リファレンスカテゴリ数 | 10〜15 | カテゴリ目次が全量でコンテキストに収まるサイズ。ISO 25010の品質特性（8項目）+ インフラ・デプロイ・API等の技術横断事項で10〜15程度 |
| 確認チェック項目/カテゴリ | 3〜5 | 「見落としやすい重要項目」に限定。網羅リストではなく検出力と保守コストのバランス |
| 飽和スキャン軸数 | 4 | データフロー・インフラ・外部連携・横断的関心事。アーキテクチャの主要ビューポイントに対応 |
| 不適用理由コード種類 | 3 | NC/NR/OOS。判定理由のカテゴリとして十分かつ簡潔 |

## 7. 統合案の制約と前提

### 明示的な制約

1. **対象は識別漏れ（Identification Gap）**: 検討深度の不足（Depth Gap）は本方式の主要対象外。カテゴリ内の個別項目の検討深度は設計フェーズ（Phase 1-D）とレビューフェーズ（Phase 3）で担保する
2. **リファレンスは補完であり主ではない**: リファレンスの品質不足が即座にシステム全体の品質低下に繋がらない設計。導出層が主であり、検証層がなくても最低限の識別は可能
3. **LLM能力への依存は残る**: 導出層はLLMの推論能力に依存する。多層安全網（暗黙要件補完→リファレンス検証→飽和スキャン）で緩和するが、完全な排除は不可能

### フィードバックループ

検証層で「導出漏れ → 追加導出」が頻発する確認チェック項目を記録する。この記録は:
- リファレンスのトリガー条件精緻化の入力になる
- 導出層のプロンプト改善の示唆になる
- 長期的な方式品質改善に貢献する

記録先: `{work_dir}/feedback-log.md`（各実行後に追記、スキル改善時に参照）
