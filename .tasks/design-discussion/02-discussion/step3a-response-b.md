# Step 3a: 方式B擁護者からのフィードバック

---

## 1. 方式Bの強みが十分に反映されているか

### 十分に反映された点

**論点3（不適用判定）**: 「理由コード付き不適用判定」の採用は、方式Bの核心的な貢献の一つが正当に評価されたと認める。「考慮した上でのスキップ」と「見落とし」の区別は、R1（検証可能性）において本質的であり、統合案がこれを採用したことは評価する。

**論点5（安定層の認識）**: 「リファレンスを抽象カテゴリ層に限定する」という折衷は、方式Bが主張した「安定層と変化層の分離」を実装した形として受け入れられる。品質特性・セキュリティ原則等のカテゴリが安定していることへの評価は適切。

**論点6（ハンドオフ検証）**: シードタグ（方式A）+ 全決定事項スキャン（方式B）の組み合わせは、弁論での議論が正しく反映された統合である。

### 不十分に反映された点

**論点1の判定論理に根本的な問題がある**

統合案は「帰納的推論はリストの穴に制約されない → 範囲の広さが識別に有利 → 方式A寄り」と判定した。しかし方式Bの主張の核心は「範囲の狭さ」ではなく「**修正可能性の非対称性**」だった。

統合案自身が「方式Bの『前提の可視性と修正可能性』はR1（検証可能性）に直接貢献する」と認めながら、この点をR1への貢献として処理して識別の起点の判定から切り離した。

修正可能性の非対称性（リストの穴は検出・修正可能、LLM推論の穴は不可視・モデル更新まで修正不可）は、R1のみならず**長期的な識別品質の改善可能性**に直結する問題である。これを検証可能性の一側面として矮小化したのは、方式Bの主要論点の核心を見落としている。

**リファレンスの「例示のみ」化は方式Bの識別機能を実質的に除去する**

統合案のリファレンス設計:
> 典型的な検討事項の例示（3-5個、網羅的リストではなく例示）

方式Bが主張した識別機能の強みは「リストに存在する検討事項を確実に識別できる安定性」にある。「例示のみ」の設計では、検証層が「このカテゴリは適用されているか」というカテゴリ粒度の確認しかできない。

具体例: セキュリティカテゴリが「適用済み」と判定されても、「認証方式」「セッション管理」「CSRF対策」「レートリミット」のいずれかが検討事項から漏れていた場合、カテゴリレベルの照合では検出できない。方式Bが提供する「**カテゴリ内の具体的な検討事項の識別漏れ検出**」が消滅している。

## 2. 統合案の弱点

### 弱点1: 検証層の系統誤差問題が未解決

統合案の構造:
```
[層1: 導出層] 要件駆動の動的推論（主）
[層2: 検証層] リファレンス照合 + 飽和スキャン（補完）
```

検証層がリファレンス照合と飽和スキャンから構成されているが、この構造では方式Bが指摘した問題が残る。

弁論ラウンドで提起した通り、LLMの推論漏れは訓練データ偏りによる**系統誤差**として発生する。導出層で系統的に見落とされた検討事項は、同じLLMが行うカテゴリレベルの照合（検証層）でも系統的に見落とされる可能性が高い。

「可観測性カテゴリは適用済みと判定した → でも分散トレーシングの必要性を見落としている」という状況は、検証層がカテゴリ粒度に留まる限り検出されない。検証層が導出層の系統誤差を本当に補完できるのは、「導出層の推論バイアスとは独立した情報源」を使う場合のみである。

方式Bの具体的エントリ（「分散システム → 分散トレーシング」等）は、このバイアス独立性を提供していた。「例示のみ」の抽象カテゴリ設計では、この独立性が著しく低下する。

### 弱点2: 「カテゴリ適用済み」判定の定義が曖昧

統合案の検証層:
> 「カバーされていないカテゴリについて『不適用（理由コード）』または『導出漏れ → 追加導出』を判定」

「カテゴリがカバーされている」の判定基準が未定義である。

- 「セキュリティカテゴリ」について「認証方式の検討事項が導出されていればカバー済み」とするか
- 「認証・認可・CSRF・セッション管理・レートリミット・インジェクション対策の全てが識別されてカバー済み」とするか

この判定基準がLLMの推論に委ねられるなら、導出層と同じ系統誤差が判定基準の解釈にも影響する。

### 弱点3: 不適用判定の粒度が粗くなる

論点3で方式B寄りと判定した「理由コード付き不適用判定」が、統合案では「カテゴリレベルの照合時に記録」という形になっている。しかしカテゴリ粒度での不適用記録は粒度が粗すぎる。

方式Bが提案した不適用判定は**エントリ粒度**（「分散トレーシングの検討事項が不適用、理由: このシステムはシングルサービスであるため NC」）であり、この粒度があって初めて「考慮した上でのスキップ」と「見落とし」の区別が意味を持つ。

カテゴリ粒度の不適用記録（「セキュリティカテゴリは適用された」）では、カテゴリ内の個別検討事項の検討状況は見えないままである。

## 3. 具体化への提案

### 提案1: リファレンスの粒度設計を再検討する

「典型的な検討事項の例示（3-5個）」を「例示として参照するだけ」にするのではなく、**確認チェックリストとして機能させる**ことを提案する。

具体的には:
```
カテゴリ: 可観測性
トリガー: バックエンドサービスあり
arch側の関心事: 監視戦略（SLI/SLO設定、ログ集約方針、分散トレーシング採否）
典型的検討事項（チェック用）:
  - メトリクス収集戦略
  - ログ集約と構造化ログ方針
  - 分散トレーシングの採否（分散システムの場合）
  - アラートと異常検知の方針
```

これらの「典型的検討事項」は「この項目が導出されているか」の確認チェックに使う。「3-5個の例示」という設計では「例として思い出す」だけになるが、「確認チェック項目」として使えば、導出漏れを具体的に検出できる。

この設計ならメンテナンスコストも許容範囲に収まる（カテゴリあたり3-5項目、カテゴリ10-15個で計30-75項目）。

### 提案2: 検証層を「導出バイアスと独立した情報源」として位置づける

検証層がリファレンス照合と飽和スキャンで構成される場合、「どちらが導出層とのバイアス独立性を担保するか」を明確にする必要がある。

- **リファレンス照合**: バイアス独立性が高い（事前定義された外部基準）。ただしリファレンスの質に依存
- **飽和スキャン**: バイアス独立性が低い（同じLLMの推論）。ただし異なる推論角度を提供

Step 3bの詳細化では、リファレンス照合において「LLMが自身の導出結果を評価するだけでなく、リファレンスの各項目を強制的に提示して確認する」プロンプト設計にすることを検討すべきである。「導出されたリストとカテゴリの一致確認」ではなく、「リファレンスの各項目を提示して導出結果に存在するか確認」する方向がバイアス独立性を維持する。

### 提案3: 不適用判定の粒度を検証層の粒度設計と整合させる

統合案で採用された「理由コード付き不適用判定」（論点3判定）を有意義にするために、不適用判定の粒度をリファレンスの粒度に合わせる必要がある。

- リファレンスがカテゴリのみ → 不適用判定もカテゴリ粒度（粗い）
- リファレンスが確認チェック項目を持つ → 不適用判定も項目粒度（提案1と整合）

提案1（確認チェックリスト）を採用した場合、各確認項目について「導出済み/導出漏れ（追加導出）/不適用（理由コード）」を記録することで、論点3の方式B貢献が本来の粒度で実現される。

### 提案4: 長期品質改善フィードバックループを設計に組み込む

方式Bの「修正可能性」という利点は、統合案でも活かすことができる。Step 3bでの詳細化において、検証層での「導出漏れ → 追加導出」判定の記録を**リファレンス改善フィードバック**として活用するメカニズムを設計に含めることを提案する。

具体的には:
- 検証層で「リファレンスに存在したが導出されなかった項目」を記録
- これが蓄積されることで「このリファレンス項目は繰り返し導出漏れになる」という実績データが得られる
- 実績データをリファレンスのトリガー条件の精度改善に使う

このメカニズムにより、統合案は「運用するほど導出品質が向上する」特性を持つことができ、方式Bの「修正可能性」の利点が統合案でも実現される。

## 総評

統合案は論点3・5・6において方式Bの貢献を適切に取り込んでいる。しかし論点1の判定において方式Bの核心論点（修正可能性の非対称性）をR1への貢献に矮小化した結果、リファレンスが「検証補完」の役割に格下げされ、検証層の実効性に構造的な弱点が残った。

特にリファレンスの「例示のみ」設計は、方式Bが提供する「カテゴリ内の具体的な識別漏れ検出」機能を実質的に除去しており、検証層がカテゴリ粒度の確認以上の機能を発揮できるかが懸念される。

Step 3bでは「確認チェック項目」の粒度設計と、検証層が「導出層バイアスと独立した情報源」として機能するためのプロンプト設計が最も重要な詳細化ポイントになると考える。
