# Step 4: シナリオ走査による検証

Step 3bの具体手順に従い、6つのシナリオで走査する。手順に書かれていない暗黙の補完は行わない。

---

## シナリオ1: 正常系（標準的なWebアプリプロジェクト）

**設定**: SR 15件、NFR 8件、制約 5件の標準的なWebアプリケーション。フロントエンド(React) + バックエンドAPI(Node.js) + PostgreSQL。認証あり、外部API連携1件。

### Phase 0: 初期化

- 要件文書読込 → 構造サマリ抽出 ✅
- 出力: sr_list(15), nfr_list(8), constraints(5), actors_and_interfaces → 問題なし

### Phase 1-A: 導出層

**Step 1（直接導出）**:
- SR 15件 × 「何を決めるべきか」→ 候補リスト生成
- 想定される導出: フロントエンド構成、APIルーティング、DB設計、認証、外部API連携等
- 粒度正規化: 「フロントエンドの状態管理ライブラリ」→ process委譲マーク ✅
- **情報の流れ**: 構造サマリ → 導出結果 ✅

**Step 2（暗黙要件補完）**:
- 技術コンテキスト: Webアプリ + 認証 → CSRF対策、セッション管理、レートリミット
- NFR間相互作用: 性能 + セキュリティ → 認証のキャッシュ戦略
- **情報の流れ**: 技術コンテキスト + Step 1結果 → 追加候補 ✅

**Step 3（次元整理）**:
- グルーピング → 設計次元化
- 依存関係分析 → 議論順序決定
- 整合性チェック: SR/NFRカバレッジ確認
- **情報の流れ**: Step 1+2の候補 → スコープ分割計画 ✅

### Phase 1-B: 検証層

**Step 4（リファレンス照合）**:
- 10カテゴリのトリガー照合 → 大部分が該当
- 各確認チェック項目を1項目ずつ照合
- 想定結果: ほとんどが「確認済み」。バックアップ/リカバリが導出漏れの可能性
- 不適用判定: 例えば「信頼性/可用性」カテゴリで高可用性構成が不要な場合 → NC
- **情報の流れ**: 導出結果 + リファレンス → 照合結果ファイル ✅

**Step 5（飽和スキャン）**:
- 4軸スキャン。Step 4の結果を入力しない（独立実行） ✅
- **情報の流れ**: 導出結果 + 要件文書 → スキャン結果ファイル ✅

### Phase 1-C: ユーザー確認

- 導出結果 + 照合結果 + スキャン結果を統合して提示 ✅
- 不適用判定記録も含む ✅

### Phase 1-D〜1-E: 設計 + 完了記録

- 対話的設計（通常通り）
- 設計完了記録に構造化フィールド出力 ✅

**走査結果**: ✅ 正常系で情報の流れに途切れなし

## シナリオ2: リスト外項目の検出（AI/MLパイプライン）

**設定**: AI/MLパイプラインを含むプロジェクト。リアルタイム推論サービス + モデル訓練パイプライン + 特徴量ストア。

### Phase 1-A: 導出層

**Step 1**: 要件から「モデルバージョン管理」「推論レイテンシSLO」「特徴量パイプライン設計」を導出。これらはリファレンスに存在しない可能性が高い。

**Step 2**: 技術コンテキスト「ML推論パイプライン」から「モデルドリフト監視」「A/Bテスト基盤」「特徴量ドリフト検出」を補完。

### Phase 1-B: 検証層

**Step 4**: リファレンス照合
- 「可観測性」カテゴリ: 適用 → 確認チェック項目「SLI/SLO設定」「分散トレーシング」は確認済み
- **しかし**: 「モデルドリフト監視」はリファレンスの確認チェック項目に含まれない
- ⚠️ **問題検出**: 確認チェック項目が「分散トレーシング」等の汎用項目のみの場合、ML特有の監視事項は照合対象にならない。Step 4では「リファレンスに存在する項目の照合」のみを行うため、リファレンスに存在しないML固有項目の検証は**Step 4では行われない**。

**Step 5**: 飽和スキャン
- 横断的関心事軸: 「可観測性について未検討事項はないか」→ ここでモデルドリフト監視が検出される可能性がある
- **しかし**: Step 5は「導出結果 + 要件文書」を入力とし、Step 4の照合結果は入力しない設計。導出結果にモデルドリフト監視が既に含まれていれば（Step 2で補完済み）、Step 5は「確認済み」と判断する

**走査結果**: ⚠️ 条件付きOK
- Phase 1-A Step 2（暗黙要件補完）がML特有の項目を導出できるかがLLM能力依存
- リファレンスのStep 4ではML固有項目をカバーできない（設計通り — リスト外項目は導出層の責務）
- Step 5（飽和スキャン）が追加の安全網として機能
- **手順上の曖昧箇所なし** — 処理は一意に定まる

## シナリオ3: Phase 1-AとPhase 1-Bの衝突

**設定**: Phase 1-A（導出層）で「可観測性」関連の検討事項を「パフォーマンス」次元に統合して導出した。Phase 1-B Step 4で「可観測性」カテゴリを照合すると、確認チェック項目のうち一部が「パフォーマンス次元で検討済み」、一部が「未照合」となる。

### 走査

**Step 4の処理**:
```
カテゴリ: 可観測性
トリガー: バックエンドサービスあり → 適用

確認チェック項目「SLI/SLO設定」:
  導出結果に含まれるか？ → 「パフォーマンス」次元内に含まれている
  → 「確認済み」

確認チェック項目「ログ集約方針」:
  導出結果に含まれるか？ → 含まれていない
  → このプロジェクトで検討が必要か？ → 必要
  → 「導出漏れ」→ Step 4b（追加導出）
```

**Step 4bの処理**:
```
1. 仮説: 「ログ集約方針がパフォーマンス次元に統合されなかったのは、
   性能観点ではなく運用観点の要素だったため」
2. 視点切替: 運用者視点で再導出
3. 結果: 「ログ集約方針」を独立した検討事項として追加、
   または既存の次元に配置
```

⚠️ **問題検出**: Step 4の「導出結果に含まれるか」の照合において、**導出結果が設計次元単位でグルーピングされている場合、確認チェック項目との名称・粒度のずれ**が判定を困難にする可能性がある。

例: 導出結果に「パフォーマンス監視の設計方針」があり、リファレンスの確認項目に「SLI/SLO設定」がある。これが「同じ内容」かどうかのマッチングはLLMの判断に依存する。

**修正案**: Step 4のプロンプト設計で、照合は「名称の一致」ではなく「この確認項目の関心事が、導出結果のいずれかの検討事項でカバーされているか」という意味的照合であることを明示する。

**走査結果**: ⚠️ 修正要
- 手順は一意に定まるが、照合の「含まれるか」判定が名称ベースか意味ベースかが**未明示**
- Step 4のプロンプト設計に「意味的照合（名称ではなく関心事のカバレッジ）」を明記する必要あり

## シナリオ4: 分界マップにない横断的関心事

**設定**: 国際化/ローカライゼーション（i18n/l10n）を要件に含むプロジェクト。リファレンスの10カテゴリには「国際化」カテゴリが存在しない。

### Phase 1-A: 導出層

**Step 1**: 要件に「多言語対応」SRがある場合、直接導出で「i18nの設計方針」が候補に上がる ✅
**Step 2**: 技術コンテキストから「翻訳管理戦略」「日時/通貨のロケール処理」を補完 ✅

### Phase 1-B: 検証層

**Step 4**: リファレンスに「国際化」カテゴリがないため、照合対象にならない
- ⚠️ **問題検出**: 導出層で正しく導出されていれば問題ないが、導出漏れの場合に検証層でも捕捉できない

**Step 5**: 飽和スキャン
- データフロー軸: 「入力→処理→出力の各段階で未決定はないか」→ 出力段階でロケール処理が浮上する可能性
- 横断的関心事軸: 国際化が横断的関心事として識別される可能性
- **しかし**: スキャン軸の問いが「セキュリティ・可観測性・エラー処理・**国際化等**」と「等」で含めているため、LLMが国際化を想起するかは不確実

**修正案**:
1. 飽和スキャンの横断的関心事軸の問いに「国際化等」ではなく、「及びプロジェクト要件に関連する横断的関心事」と要件参照を明示
2. または: 飽和スキャンの問いに「Phase 0で抽出したSR/NFRに、まだ設計次元に反映されていない要件はないか」というトレーサビリティチェックを追加（これはPhase 1-A Step 3の整合性チェックと重複するが、検証層として独立に実施する価値がある）

**走査結果**: ⚠️ 修正要
- リファレンスに存在しないカテゴリの検討事項は導出層依存 → 設計通り
- ただし飽和スキャンの問いが特定の横断的関心事を列挙するスタイルだと、列挙外の事項を見落とすリスク
- **飽和スキャンの横断的関心事軸の問いを要件参照型に修正**すべき

## シナリオ5: 要件にもリファレンスにもない検討事項

**設定**: バックアップ/リストア戦略、データマイグレーション計画。これらはNFRに明示されていないが、本番運用で不可欠。

### Phase 1-A: 導出層

**Step 1**: NFRに明示なし → 直接導出では候補に上がらない可能性が高い
**Step 2**: 暗黙要件補完の問い「技術コンテキストから必ず対処すべき事項は何か」→ データベースを使うプロジェクトならバックアップ/リストアが推論される可能性がある。しかし、問いの範囲が「設計で必ず対処すべき事項」であり、「運用で必要な事項」が含まれるかはLLMの解釈依存

### Phase 1-B: 検証層

**Step 4**: リファレンスのカテゴリ「データ管理」の確認チェック項目に「バックアップ/リカバリ」が含まれている場合 → 照合で検出 ✅
- ⚠️ ただし確認チェック項目に含まれていない場合は検出できない

**Step 5**: 飽和スキャン
- インフラ軸: 「実行環境・スケーリング・**デプロイ形態**で未決定はないか」→ バックアップ/リストアはデプロイ形態ではない
- データフロー軸: 「永続化の段階で未決定はないか」→ バックアップが浮上する可能性

⚠️ **問題検出**: 要件にもリファレンスにもない事項の検出は、飽和スキャン軸の問いの範囲とLLMの推論力に依存する。手順上は「スキャン軸の問いに回答するLLMが想起するかどうか」がボトルネック。

**修正案**:
1. リファレンスの「データ管理」カテゴリの確認チェック項目に「バックアップ/リカバリ方針」を含める（これはメンテナンスの一環）
2. 飽和スキャンに「運用者視点」を軸として追加（4軸→5軸）:
   「このシステムを運用開始した後、運用チームが定常的に行う作業で、設計方針が未定のものはないか？」

**走査結果**: ⚠️ 修正要
- 現行設計ではリファレンスのチェック項目品質とLLM推論力に依存
- 飽和スキャンに「運用者視点」軸を追加することで、運用上必要だが要件に明示されない事項の検出力が向上
- リファレンスの「データ管理」にバックアップ/リカバリを追加

## 修正適用

### 修正1: Step 4のプロンプトに意味的照合を明記

Step 4の処理定義に追加:
```
照合の判定基準: 名称の一致ではなく「関心事のカバレッジ」で判断する。
「この確認チェック項目が問題としている関心事（{check_itemの内容}）が、
 導出結果のいずれかの検討事項でカバーされているか」を判断基準とする。
```

### 修正2: 飽和スキャンの横断的関心事軸を要件参照型に修正

修正前:
```
「セキュリティ・可観測性・エラー処理・国際化等で、
 設計方針が未定のまま残っている事項はないか？」
```

修正後:
```
「以下の2つの観点で、設計方針が未定の事項がないか確認:
 1. 一般的な横断的関心事（セキュリティ・可観測性・エラー処理等）
 2. プロジェクト要件（{sr_list}, {nfr_list}）に関連するが、
    まだいずれの設計次元にも割り当てられていない要件」
```

### 修正3: 飽和スキャンに運用者視点軸を追加（4→5軸）

追加軸:
```
5. 運用視点軸:
   「このシステムを本番運用開始した後、運用チームが
    定常的に行う作業（バックアップ、監視、障害対応、
    データマイグレーション等）で、設計方針が未定のものはないか？」
```

### 修正4: リファレンスの確認チェック項目にバックアップ/リカバリを追加

「データ管理」カテゴリの確認チェック項目:
```
- データストア選定
- データモデリング方針
- バックアップ/リカバリ方針  ← 追加
```
