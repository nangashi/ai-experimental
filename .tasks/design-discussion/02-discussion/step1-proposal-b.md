# Step 1: 方式B 提案 — 事前定義方式

## 1. 方式の概要

**事前定義方式**は、設計検討事項のリファレンスリスト（以下「リスト」）を事前に構築しておき、各プロジェクトの設計フェーズでそのリストを「識別のスキャフォールド」として機能させるアプローチである。

### 基本メカニズム

```
リファレンスリスト（事前定義済み）
        ↓
[フィルタリング] 要件コンテキストでカテゴリを絞込
        ↓
[マッピング] 各エントリを「適用/不適用/要確認」に分類
        ↓
[補完] リスト外のプロジェクト固有事項を識別
        ↓
識別済み検討事項セット（検証可能・トレーサブル）
```

LLMが「暗黙知から網羅的に考える」のではなく、**構造化されたリストとの照合**という作業に処理を置き換える。これにより、「何を考慮すべきか」という発散的推論をリスト作成時点に限定し、実行時のLLMはより確実性の高い「マッピング判断」に集中できる。

### リファレンスリストの構造

リストは以下の階層で構成する:

```
カテゴリ（品質特性・機能領域・インフラ横断事項）
  └ 項目（具体的な設計検討事項）
      ├ トリガー条件: どのコンテキストで適用されるか
      ├ arch属性: arch_designで決定すべき戦略・方針
      ├ process属性: process_designで決定すべき実装・規約
      └ 依存: 他項目との関連
```

**トリガー条件の例**:
- `分散システム → 可観測性/分散トレーシング`
- `外部API統合あり → エラーハンドリング/リトライ戦略`
- `マルチテナント → データ分離戦略`

この条件が「フィルタリング」に使われ、大きなリストをコンテキスト制約内に収める。

## 2. arch_design への適用方針

### フロー概要

```
Phase A0: 要件分析 + カテゴリフィルタリング
Phase A1: リスト項目のマッピング（適用/不適用/要確認）
Phase A2: リスト外項目の補完（プロジェクト固有識別）
Phase A3: 検討事項の設計実施（決定 + 根拠 + 判断基準）
Phase A4: マッピング結果の検証（不適用判定の妥当性確認）
```

### Phase A0: 要件分析 + カテゴリフィルタリング

- requirements.md から技術スタック・アーキテクチャパターン・NFRの特性を抽出
- リストのカテゴリ目次（軽量・全量渡し可能なサマリ層）と突合し、関連カテゴリを選択
- 選択されたカテゴリの詳細エントリのみをコンテキストに展開（2段階展開でコンテキスト節約）

### Phase A1: リスト項目のマッピング

- 各エントリのトリガー条件と要件を照合し、3分類に判定:
  - **適用**: このプロジェクトで検討が必要
  - **不適用**: 理由を明示してスキップ（単純除外ではなく根拠付き）
  - **要確認**: 要件から判断できない（追加情報が必要）
- マッピング結果は中間ファイルに書き出し（サブエージェント間共有）

### Phase A2: リスト外項目の補完

- A1の結果を踏まえ、「このプロジェクト要件にはリストに存在しない固有の検討事項があるか」をLLMに明示的に問う
- リスト外項目があれば「カスタム項目」として追加（次ラウンドでリスト本体へのフィードバック候補）
- これにより B1（リストの網羅性限界）を実行時にも補完できる

### Phase A3: 設計実施

- 「適用」判定された全項目（リスト由来 + カスタム）について、戦略・方針レベルの決定を記述
- 各決定に対し: 採用理由、却下した代替案、前提条件 を明示

### Phase A4: 検証

- 「不適用」判定された項目の理由を再審査（「本当に不要か」をセカンドパスで確認）
- SR/NFR とのトレーサビリティマトリクスを生成し、未対応のSR/NFRがないか確認

## 3. process_design への適用方針

### フロー概要

```
Phase P0: arch_designの決定事項インポート + カテゴリフィルタリング
Phase P1: リスト項目のマッピング（arch決定事項を考慮）
Phase P2: リスト外項目の補完
Phase P3: 実装・規約の設計
Phase P4: arch→processハンドオフ検証
```

### Phase P0: インプット処理

- arch_designの全決定事項を読込み
- process_design専用リストのカテゴリをフィルタリング（arch属性の項目は既決定として参照、process属性のみが設計対象）

### Phase P1: マッピング（arch決定事項を考慮）

- arch_designで「可観測性: OpenTelemetry採用」と決定済みならば、process_designのエントリ「計装ライブラリ選定」は「適用（OpenTelemetry SDK）」として初期値付きでマッピング
- arch決定が未解決の項目は「依存待ち」として明示

### Phase P4: ハンドオフ検証

- arch_designの全決定事項に対して、process_designで対応するエントリが存在するか確認
- 対応がない場合は「引き継ぎ漏れ」として明示的に報告（Handoff Gapの防止）

---

## 4. 横断的関心事のarch/process境界の扱い

横断的関心事（可観測性、セキュリティ、エラーハンドリング等）は、リストのエントリに **arch属性** と **process属性** を両方付与することで対処する。

### エントリ構造例（可観測性）

```yaml
category: 可観測性
item: メトリクス収集
trigger: "バックエンドサービス あり"
arch属性:
  - 何を監視するか（SLI/SLO設定）
  - どの監視戦略か（Push型/Pull型）
process属性:
  - どのライブラリで計装するか
  - メトリクス命名規約
  - アラートルール定義プロセス
依存: [デプロイ戦略, サービス境界設計]
```

- arch_designはarch属性のみを処理対象とする
- process_designはprocess属性を処理対象とし、arch属性の決定結果を前提条件として参照する
- **境界が曖昧な事項**（例: 国際化がarch/processどちらか）は、エントリに `ambiguous: true` フラグを立て、マッピング時にLLMが判断するのではなく、フェーズ設計者が事前に帰属を決定してリストに記録する

## 5. 網羅性の保証メカニズム

### 5-1. リスト自体の網羅性

リストの網羅性は実行時に担保するのではなく、**リスト更新プロセス**で担保する:

1. **初期作成**: 既存の設計ガイド、ISO 25010（品質特性）、OWASP、C4モデルのビューポイント等から体系的にシード
2. **実行後フィードバック**: Phase A2/P2で識別された「リスト外項目」を次版候補としてログ
3. **定期レビュー**: リスト外項目の発生頻度が高いカテゴリを優先してリスト拡充（R5保守性対応）

### 5-2. マッピング精度の保証

- 「不適用」判定には**理由コード**を必須とする（単なる除外ではなく根拠付き）
  - `NC（Non-applicable by Context）`: アーキテクチャパターンが異なる
  - `NR（Not Required）`: 要件に明示的に除外されている
  - `OOS（Out of Scope）`: 本フェーズ対象外
- Phase A4/P4の検証パスで、NC/NRの理由コードが妥当かをセカンドチェック

### 5-3. リスト外項目の積極的受け入れ

B1（リストの完全性限界）は解消不能なため、リスト外識別を設計プロセスの正式ステップとして組み込む（Phase A2/P2）。これにより:
- 新規ドメイン（AI/MLパイプライン等）にも対応可能
- リスト外項目の蓄積がリスト改善の入力になる

## 6. 依存する仮定と妥当性

### B1: リファレンスリストを十分な網羅性で事前定義できる

**この方式が最も強く依存する仮定**。

**妥当性の根拠**:
- 設計検討事項の多くは技術横断的なパターン（品質特性・セキュリティ・運用）に収束する傾向があり、ISO 25010等の標準が既存の体系として機能する
- ソフトウェアアーキテクチャの主要ビューポイント（C4、4+1ビュー）も既存の体系
- 完全な網羅性は目標ではなく、「動的導出方式に比べて識別率が高い」ことが比較優位の根拠

**限界の明示**:
- 新しい技術パラダイム（初出のLLMオーケストレーション設計等）はリストに存在しない可能性が高い
- → Phase A2/P2の「リスト外補完」ステップがこれに対応する安全弁

### B2: LLMはリスト項目をプロジェクト要件に正しくマッピングできる

**妥当性の根拠**:
- マッピング判断は「網羅的に考える」推論より制約が明確（入力: トリガー条件 + 要件、出力: 適用/不適用/理由）
- 構造化された判断タスクはLLMが比較的安定して実行できる（C2：構造化プロンプトへの準拠）
- 「不適用」に理由コードを要求することで、誤判定の検出可能性が向上する

**残存リスク**:
- トリガー条件の記述精度がマッピング精度に直接影響する
- → リスト作成時にトリガー条件を曖昧に書かない品質ゲートが必要

### B3: リファレンスリストのメンテナンスコストが許容範囲内

**妥当性の根拠**:
- リストの更新が必要な頻度は「技術的パラダイムシフト」の頻度に依存する。多くの設計パターンは数年単位で安定
- Phase A2/P2で収集された「リスト外項目」を更新候補として整理することで、更新作業の発見コストが低減

**許容できない場合の代替**:
- B3が成立しない場合（技術変化が速い領域）は、リストの対象を「普遍的カテゴリ（品質特性、運用ビュー）」のみに限定し、技術固有層は動的導出に委ねるハイブリッド構成が検討候補

## 7. 想定される課題とその対策

### 課題1: コンテキストウィンドウへの収まりにくさ

リスト全体を一度にコンテキストに渡せない可能性がある。

**対策**: 2段階展開
1. 全カテゴリの目次（1エントリ数行の軽量サマリ）をPhase A0で渡す
2. フィルタリングで選択されたカテゴリのみ詳細を展開

カテゴリ目次を「索引層」として常に保持し、詳細層のコンテキスト消費を最小化する。

### 課題2: 「不適用」判定の粗雑化

LLMが安直に「不適用」と判定して検討をスキップするリスク。

**対策**:
- 理由コード（NC/NR/OOS）を必須とし、理由なしの「不適用」をエラー扱い
- Phase A4でセカンドパス検証を実施（不適用判定の根拠を再確認）
- 「要確認」分類を設けることで、判断不能の場合に即「不適用」に落とさない

### 課題3: リスト自体の設計偏りによる系統的漏れ

リストが特定の技術スタック・ドメインに偏っていると、そのバイアスが全プロジェクトに影響する。

**対策**:
- リスト作成時に複数の標準（ISO 25010、OWASP、TOGAF等）を参照して多角的にシード
- Phase A2/P2の「リスト外補完」を形式的でなく実質的なステップとして設計（LLMに「このリストに何が抜けているか」を明示的に問う）
- リスト外項目の発生頻度を記録し、偏りのあるカテゴリを優先して拡充

### 課題4: 横断的関心事の重複・抜け

arch/process両フェーズにまたがる項目で、どちらも「もう一方が担当する」と誤認するリスク。

**対策**:
- リストの各エントリにarch/process属性を明示的に付与（5. で述べた構造）
- Phase P4のハンドオフ検証を形式的なトレーサビリティチェックとして実施
- 「arch決定済み → process未実装」のギャップを自動検出

### 課題5: リスト外の革新的領域への対応

LLMオーケストレーションやエッジコンピューティング等、リストが想定していない技術領域。

**対策**:
- Phase A2/P2の「リスト外補完」ステップで対応（動的導出との組み合わせ）
- ただし、この補完ステップは動的導出方式と同じリスク（方式Aの仮定A1相当）を局所的に持つことを認識する
- リスト外識別の成功率を監視し、特定技術領域では動的導出を主方式とするハイブリッド判断の余地を残す
