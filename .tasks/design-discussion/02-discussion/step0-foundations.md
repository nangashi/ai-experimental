# Step 0: 前提・定義・制約の確立

## 1. 用語定義

### 設計検討事項 (Design Consideration)

特定のプロジェクト要件（SR/NFR/制約）または設計コンテキストに対して、設計フェーズで判断・決定が必要な事項。以下の要素で構成される:

- **対象**: 何について決めるか（例: 認証方式、デプロイ戦略）
- **トリガー**: なぜ検討が必要か（関連するSR/NFR/制約、または暗黙の技術的要求）
- **判断基準**: 何を基準に決めるか（NFR/制約との対応関係）
- **決定**: 選択した方式とその根拠

### 漏れの種類

| 種類 | 定義 | 例 |
|------|------|-----|
| **識別漏れ (Identification Gap)** | プロジェクトに必要な検討事項が設計プロセスで一切識別されない | 分散システムなのに可観測性が検討されない |
| **検討深度の不足 (Depth Gap)** | 識別されたが重要な側面が検討されない | 認証方式は決めたがセッション管理方針が未検討 |
| **引き継ぎ漏れ (Handoff Gap)** | arch_designの決定がprocess_designに正しく反映されない | アーキテクチャで可観測性戦略を決めたが計装ライブラリ・ログ規約が未設計 |

**本議論の主要フォーカス**: 識別漏れの防止。検討深度と引き継ぎは識別の後工程であり、識別されていない事項は深堀りも引き継ぎもできない。

### 網羅性の評価基準

以下の条件をすべて満たすとき「漏れなく検討できた」とする:

1. **トレーサビリティ**: 全SR/NFR/制約に対して、対応する設計検討事項が存在する
2. **暗黙要件の補完**: 要件に明示されていないが技術的に必要な検討事項（例: WebアプリならCSRF対策）が識別されている
3. **検討深度**: 各検討事項の決定に必要な判断基準が明示されている
4. **引き継ぎ完全性**: arch_designの全決定事項がprocess_designで参照・具体化されている
5. **整合性**: 検討事項間の依存関係が識別され、矛盾がない

## 2. スコープと前提条件

### 入力品質の前提

- requirements.md は確定済み（Confirmed状態）を前提とする
- ただし、**暗黙の技術的要件**（例: Webアプリならセキュリティヘッダー設定、APIならレートリミット）はrequirements.mdに明示されていない可能性がある
- 暗黙の技術的要件の識別は、設計プロセスの責務に含まれる

### 対象の設計フェーズの責務範囲

| フェーズ | 責務 | 決定する事項のレベル |
|---------|------|-------------------|
| **arch_design** | 要件→アーキテクチャ方針・技術選定・コンポーネント構成・インターフェース設計 | 戦略・方針レベル（何を・なぜ） |
| **process_design** | アーキテクチャ→開発プロセス・ツールチェーン・規約・CI/CD | 実装・運用レベル（どう実現するか） |

### 横断的関心事の分界

arch_designとprocess_designの両方に関わる事項（例: 可観測性、セキュリティ、エラーハンドリング）:

- **arch_design側**: 戦略・方針レベルの決定（何を監視するか、どのレベルのセキュリティが必要か）
- **process_design側**: 実装レベルの決定（どのツール/ライブラリで実現するか、どの規約で統一するか）
- **分界の曖昧さ**: 一部の横断的関心事はどちらで扱うべきか判断が分かれる可能性がある（例: 国際化は方針? 実装?）

### LLM実行環境の制約

| 制約 | 詳細 |
|------|------|
| コンテキストウィンドウ | 有限。大量のリファレンスリストを丸ごと渡すのは非効率。サマリ化やフィルタリングが必要 |
| サブエージェントの独立性 | サブエージェントは独立動作。親エージェントとの情報共有はファイル経由 |
| プロンプト依存性 | LLMの推論はプロンプトの構造とコンテンツに依存する。暗黙知に依存した「網羅的に考えよ」は不確実 |
| 冪等性 | 同一プロンプトでも実行ごとに結果が異なりうる。検証・補完の仕組みが必要 |

## 3. 方式が満たすべき条件

| # | 条件 | 定義 | 検証方法 |
|---|------|------|---------|
| R1 | **検証可能性** | 漏れの有無を事後的に検証できるメカニズムを持つ | 設計完了後に別プロセスで漏れを検出できるか |
| R2 | **適応性** | プロジェクト固有の検討事項（汎用リストに存在しない項目）に対応できる | 非標準的なプロジェクト要件（AI/MLパイプライン等）への適用可否 |
| R3 | **実装可能性** | LLMスキルとして実現可能（コンテキスト・処理時間の制約内） | 標準的なプロジェクト（SR 15件、NFR 8件程度）で実行可能か |
| R4 | **コスト効率** | トークン使用量・処理時間が妥当 | 全体の追加トークン量が現行比で許容範囲内か |
| R5 | **保守性** | 方式自体のメンテナンスコストが低い | リファレンスデータの更新頻度・範囲が限定的か |

## 4. 方式が依存する仮定

### 方式A: 動的導出方式が依存する仮定

| ID | 仮定 | リスク | 検証基準 |
|----|------|--------|---------|
| A1 | LLMは要件文書から関連する設計検討事項を高い網羅性で導出できる | 導出漏れが体系的に発生する可能性 | 既知の検討事項リストとの差分が許容範囲内か |
| A2 | LLMは検討事項間の依存関係・横断的関心事を正確に分析できる | 独立して見える事項の関連を見落とす | 横断的関心事の識別率 |
| A3 | 複数回の導出・検証で網羅性を向上できる | イテレーション回数と品質向上の関係が不明確 | 追加ラウンドごとの新規発見数の減少傾向 |

### 方式B: 事前定義方式が依存する仮定

| ID | 仮定 | リスク | 検証基準 |
|----|------|--------|---------|
| B1 | 設計検討事項のリファレンスリストを十分な網羅性で事前定義できる | 新しい技術領域やドメインに対応できない | リスト外項目の発生頻度 |
| B2 | LLMはリファレンスリストの各項目をプロジェクト要件に正しくマッピング（適用/不適用の判断）できる | 誤適用（不要な項目の過剰適用）または見落とし（必要な項目のスキップ） | マッピング精度 |
| B3 | リファレンスリストのメンテナンスコストが許容範囲内 | 技術進化に追従するコストが累積する | 更新頻度と更新あたりの労力 |

### 共通の仮定

| ID | 仮定 | リスク |
|----|------|--------|
| C1 | requirements.mdが十分な品質であれば、設計に必要な情報は明示的または暗黙的に含まれている | 要件自体の品質が低い場合、どの方式でも識別漏れが発生する |
| C2 | LLMは構造化されたプロンプトに従いステップバイステップで処理できる | プロンプトの構造が不適切だと、どの方式でも期待通りに動作しない |
| C3 | 設計検討事項の識別はdecision pointの粒度で行う（設計パターンの詳細レベルではない） | 粒度の認識が提案者間でずれると議論がかみ合わない |
