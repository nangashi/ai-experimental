# Step 6: 最終版 — 設計検討事項の識別漏れ防止方式

## 方式概要

**名称**: 要件駆動導出 + リファレンス検証の2層方式

**基本原理**: 要件文書からの動的導出を主軸とし、事前定義リファレンスによる検証を補完層とする。導出層がプロジェクト固有の検討事項を適応的に識別し、検証層が導出層の系統誤差を外部基準で補完する。

```
[層1: 導出層（主）]
  要件文書を起点に、設計検討事項を動的に導出する
  - 直接導出（要件 → 検討事項）
  - 暗黙要件補完（技術コンテキスト → 検討事項）
  - 依存関係分析と設計次元整理

[層2: 検証層（補完）]
  リファレンスと導出結果を照合し、漏れを検出する
  - 確認チェック項目による識別漏れ検出（バイアス独立の外部基準）
  - 不適用判定の明示記録（理由コード付き）
  - 飽和スキャン（5軸の未検討領域チェック）
```

**方式が満たす条件**:

| 条件 | 充足方法 |
|------|---------|
| R1 検証可能性 | 検証層の照合結果（適用/不適用+理由コード）+ 飽和スキャン結果が記録される |
| R2 適応性 | 導出層が要件駆動のため、リファレンス外の事項も識別可能 |
| R3 実装可能性 | 検証層はカテゴリレベル（10-15カテゴリ、計30-75確認項目）で軽量 |
| R4 コスト効率 | 導出層は現行と同等。検証層は独立サブエージェント2回で追加コスト限定的 |
| R5 保守性 | リファレンスは抽象カテゴリ+確認チェック項目。安定層が主体で更新頻度は低い |

**明示的な制約**:
1. 本方式の主対象は識別漏れ（Identification Gap）。検討深度の不足（Depth Gap）はカテゴリ内チェック項目で部分的に検出するが、主にはレビューフェーズで担保する
2. リファレンスは補完であり主ではない。リファレンス品質が不足しても導出層で最低限の識別は可能
3. LLM能力への依存は残る。多層安全網で緩和するが完全排除は不可能

## 1. リファレンスの構造設計

### 1.1 エントリ構造

```yaml
- category: "{カテゴリ名}"
  trigger: "{適用トリガー条件}"
  arch_concerns: "{arch側で決定すべき方針レベルの関心事}"
  process_concerns: "{process側で決定すべき実装レベルの関心事}"
  check_items:
    - "{確認チェック項目1}"
    - "{確認チェック項目2}"
    - "{確認チェック項目3}"
```

- `check_items`は「見落としやすい重要項目」を選定したチェックリスト（網羅的リストではない）
- 各項目は検証層で「この項目が導出結果に存在するか」の確認に使用する
- カテゴリ数: 10〜15。確認チェック項目: カテゴリあたり3〜5（計30〜75項目）

### 1.2 arch_design用リファレンス（カテゴリ一覧）

| # | カテゴリ | トリガー条件 | 確認チェック項目 |
|---|---------|------------|----------------|
| 1 | アーキテクチャスタイル | 常に適用 | モノリス/マイクロサービス選択、通信パターン、サービス境界 |
| 2 | 実行基盤・インフラ | 常に適用 | ホスティング形態、スケーリング戦略、リージョン設計 |
| 3 | データ管理 | データ永続化あり | データストア選定、データモデリング方針、バックアップ/リカバリ方針 |
| 4 | セキュリティ | 常に適用 | 認証方式、認可モデル、データ保護、通信暗号化 |
| 5 | 可観測性 | バックエンドサービスあり | 監視戦略(SLI/SLO)、ログ集約方針、分散トレーシング採否 |
| 6 | 外部統合 | 外部システム連携あり | インターフェース契約、エラー伝播、リトライ戦略 |
| 7 | 信頼性・可用性 | NFRに可用性要求あり | 冗長構成、障害分離、フェイルオーバー |
| 8 | パフォーマンス | NFRに性能要求あり | キャッシュ戦略、非同期処理、ボトルネック対策 |
| 9 | デプロイ・リリース | 常に適用 | デプロイ戦略、環境構成、リリースプロセス |
| 10 | API・インターフェース設計 | 外部/内部API提供あり | API設計方針、命名規則、バージョニング |

※ process_design用リファレンスは実装時に別途定義（ライブラリ選定、規約、CI/CD等のカテゴリ）

### 1.3 メンテナンス方針

- **更新トリガー**: 検証層で「導出されたがリファレンスにないカテゴリ/項目」が繰り返し発生する場合
- **更新範囲**: カテゴリ追加、トリガー条件精緻化、確認チェック項目の追加/修正
- **フィードバックループ**: 各実行の検証結果（導出漏れ・追加導出の記録）をfeedback-log.mdに蓄積し、リファレンス改善の入力とする

### Phase 1-A: 要件駆動の設計次元導出（導出層）

#### Step 1: 直接導出

**入力**: Phase 0の構造サマリ
**処理**:
```
各SR/NFR/制約に対して:
「この要件を実現するために、アーキテクチャレベルで決定しなければならない
 方針・技術・構成・インターフェースは何か？」

各候補を以下の形式で記録:
  - 対象: 何を決めるか
  - トリガー: 導出元の要件ID（SR-XXX / NFR-XXX / 制約名）
  - 判断基準候補: 関連するNFR/制約
```

**粒度正規化ルール**:
- decision point粒度 = 実装チームが実装前に方針として決定していなければならない事項
- 過細（変数命名規則等）→ process_design委譲マーク
- 過粗（「セキュリティ全般」等）→ サブ検討事項に分解

**出力**: `{work_dir}/derived-considerations.md` に候補リスト形式で保存

#### Step 2: 暗黙要件の補完

**入力**: 構造サマリの技術コンテキスト + Step 1の候補リスト
**処理**:
```
問い1:
「このプロジェクトの技術コンテキスト（{ドメイン}, {スタック}, {構成}）を前提に、
 要件に明示されていないが設計で必ず対処すべき事項は何か？」

問い2:
「以下のNFR間の相互作用で、追加の設計判断が必要になる組み合わせはあるか？
 {nfr_list}」
```

**出力**: 追加候補をderived-considerations.mdに追記（トリガー: 「技術コンテキスト: {根拠}」）

#### Step 3: 設計次元の整理と依存関係分析

**入力**: Step 1 + Step 2の候補リスト
**処理**:
```
1. 候補を設計次元にグルーピング
2. 各次元の定義（決定対象・対象外・依存先）
3. 議論順序の決定（依存関係ベース）
4. 整合性チェック:
   - 全SR/NFRがいずれかの次元でカバーされるか
   - 次元間の重複がないか
   - 依存に循環がないか
   → 循環検出時: 循環に含まれる次元を統合するか、
     依存方向を制約で一方向に限定して解消
```

**出力**:
- `{work_dir}/scope-plan.md` にスコープ分割計画を保存
- `{work_dir}/derived-considerations.md` は候補リスト形式のまま維持（Step 4の入力用に並存管理）

### Phase 1-B: リファレンス検証（検証層）

**目的**: 導出結果をリファレンスと照合し、識別漏れを検出する。導出層とはバイアス独立な情報源として機能する。

#### Step 4: リファレンスとの照合

**実行方式**: 独立したサブエージェントで実行。入力は `derived-considerations.md`（候補リスト形式）+ リファレンスファイル + 要件サマリ。`scope-plan.md` は入力しない。

**処理**:
```
リファレンスの各カテゴリについて:

1. トリガー条件の照合
   「トリガー条件（{trigger}）はこのプロジェクトの要件に該当するか？」
   → 不該当: 不適用（NC）として記録、次のカテゴリへ
   ※ 「常に適用」トリガーのカテゴリは照合不要（自動適用）

2. 確認チェック項目の照合（トリガー該当の場合）
   各check_itemを1項目ずつ強制提示:
   「以下の確認チェック項目の関心事は、導出結果のいずれかの
    検討事項でカバーされているか？（名称の一致ではなく、
    関心事のカバレッジで判断すること）
    確認項目: {check_item}
    導出結果: {derived-considerations.mdの候補リスト}」

   → カバーされている: 「確認済み」（対応する導出結果のIDを記録）
   → カバーされていない: 必要性判断フローへ
```

**必要性判断フロー**（確認チェック項目がカバーされていない場合）:
```
1. このプロジェクトの要件（SR/NFR/制約）のいずれかが
   この関心事を含意するか？
   → Yes: 「導出漏れ」→ Step 4b（追加導出）へ
   → No: 次へ

2. このプロジェクトの技術スタック/ドメインにおいて、
   この関心事は標準的に対処が必要か？
   → Yes: 「導出漏れ（暗黙要件）」→ Step 4b（追加導出）へ
   → No: 不適用（NC）として記録
```

**不適用の理由コード**:
| コード | 定義 |
|--------|------|
| NC | アーキテクチャパターン/コンテキストが非該当、または要件・コンテキスト双方で不要 |
| NR | 要件で明示的に除外/不要とされている |
| OOS | このフェーズの責務範囲外（process_designで対応） |

**出力**: `{work_dir}/reference-check.md` に照合結果テーブルを保存

#### Step 4b: 追加導出（視点切り替えプロトコル）

```
1. 仮説生成: 「なぜこの項目が最初の導出で識別されなかったか」
   候補:
   - 要件に直接的な言及がない暗黙の技術的要求だった
   - 別の次元の検討事項として統合されていた
   - 技術コンテキストからの推論が不足していた

2. 視点切り替えによる再導出:
   - 実装者視点: 「この項目なしで実装チームが作業できるか」
   - 運用者視点: 「この項目なしで運用チームが運用できるか」
   - 外部ユーザー視点: 「この項目は利用者に影響するか」

3. 再導出結果を derived-considerations.md に追加
   トリガー: 「検証層Step 4bで追加（視点: {使用した視点}）」
```

#### Step 5: 飽和スキャン

**実行方式**: Step 4とは独立したサブエージェントで実行。入力は `derived-considerations.md` + 要件文書のみ。`reference-check.md`（Step 4の照合結果）は**入力しない**。

**処理**:
```
以下の5軸でスキャン:

1. データフロー軸:
   「データの入力→処理→出力→永続化の各段階で、
    設計方針が未定のまま残っている段階はないか？」

2. インフラ軸:
   「実行環境・スケーリング・デプロイ形態で、
    設計方針が未定のまま残っている事項はないか？」

3. 外部連携軸:
   「外部システムとのインターフェース・契約・障害対応で、
    設計方針が未定のまま残っている事項はないか？」

4. 横断的関心事軸:
   「以下の2つの観点で設計方針が未定の事項がないか確認:
    a. 一般的な横断的関心事（セキュリティ・可観測性・エラー処理等）
    b. プロジェクト要件に関連するが、まだいずれの設計次元にも
       割り当てられていない要件」

5. 運用視点軸:
   「このシステムを本番運用開始した後、運用チームが定常的に行う作業
    （バックアップ、監視、障害対応、データマイグレーション等）で、
    設計方針が未定のものはないか？」
```

**「検討済み」の判定基準**:
- 当該軸に関連する導出結果が存在し、かつその導出結果の「対象」記述に当該事項が明示的に含まれている場合のみ「確認済み」とする
- 明示的記述がない場合は「潜在的未検討」として記録する

**出力形式**:
```markdown
## 飽和スキャン結果

### {軸名}
- 確認済み: {対応する導出結果のIDリスト}
- 新規発見: {未検討事項の記述}（あれば）
- 潜在的未検討: {明示的カバレッジが確認できない事項}（あれば）
```

保存先: `{work_dir}/saturation-scan.md`

### Phase 1-C: スコープ分割計画のユーザー確認

**入力**: scope-plan.md + reference-check.md + saturation-scan.md

**統合処理**:
```
1. Step 4とStep 5の結果を突合し、重複を除去
   （Step 5で検出した項目がStep 4の不適用記録に含まれる場合、
    Step 5の検出を優先してユーザーに提示する）
2. 追加導出された項目をスコープ分割計画に反映
3. 統合結果をユーザーに提示
```

**ユーザーに提示するテーブル**:
```
### スコープ分割計画

| # | 設計次元 | 決定対象 | 依存先 | ADR見込み |
|---|---------|---------|--------|----------|

### リファレンス照合結果

| カテゴリ | 適用 | 確認済み項目 | 不適用項目（理由コード） |
|---------|------|------------|----------------------|

### 飽和スキャン結果

| 軸 | 確認済み | 新規発見 | 潜在的未検討 |
|----|---------|---------|------------|
```

- 「不適用判定」欄はユーザーに提示し、妥当性を確認する（トリガー誤判定の安全網）
- ユーザー承認後、Phase 1-Dへ

#### Step 5: ハンドオフ検証

**入力**: arch設計完了記録 + Phase 1-Aの導出結果
**実行方式**: Step 4とは独立したサブエージェント。`reference-check.md` は入力しない。

```

「このarch決定（{決定内容}）に対応するprocess検討事項は、
```
