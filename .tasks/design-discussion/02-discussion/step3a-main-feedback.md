# Step 3a: メインエージェントのフィードバックと方針レベルの統合案

## 弁論の総括評価

3ラウンドの弁論を通じて、両方式の本質的な強みと弱みが明確化された。特に最終ラウンドで方式Aが示した「条件付き結論」は議論の成熟を示しており、これを統合の出発点とする。

---

## 各論点への判定

### 論点1: 識別の起点 → **要件駆動導出を主軸とし、リファレンスを検証層に配置**

**判定**: 方式A寄り（ただし方式Bの指摘を反映した修正あり）

**根拠**:
- 方式Aの「帰納的推論はリストの穴に制約されない」は、識別漏れ防止の本来目的に直結する
- 方式Bの「LLMの知識も不透明なリスト」という指摘は妥当だが、帰結が異なる。不透明であることは「範囲が狭い」こととは別の問題であり、範囲の広さは識別に有利に働く
- ただし方式Bの「前提の可視性と修正可能性」は検証可能性（R1）に直接貢献する

**バイアス検証**: 「方式Aの範囲の広さ」を根拠としたが、この論理は方式Bにも適用されるか？→ 方式Bの「リスト照合の安定性」は既知領域に限定されるため、「範囲の広さ」を主張しにくい。非対称性は成立する。

**方針**: 要件文書からの導出を第一ステップとし、リファレンスは「導出結果の検証・補完」に使用する。リファレンスを照合の起点にしない。

---

### 論点2: 暗黙要件の識別 → **コンテキスト駆動推論を主、トリガー条件を検証チェックに**

**判定**: 方式A寄り（方式Bのトリガー条件を検証層に取り込む）

**根拠**:
- 方式Aの「コンテキスト駆動推論」は新規ドメインへの適応性で優位
- 方式Bの「識別と設計実施の分離」は妥当だが、識別の安定性はリストのドメインカバレッジに依存する
- 方式Aが認めた「適用済み記録が補完動機を低下させるリスク」は、リスト照合を起点にした場合の構造的弱点として残る

**バイアス検証**: 方式Bの「前倒し推論の利点（専門家レビュー・再利用）」を軽視していないか？→ この利点は確かに存在するが、リファレンスを「検証層」として配置することで同じ利点を得られる。前倒しの再利用は検証層でも享受可能。

**方針**: Phase内で技術コンテキストからの暗黙要件を推論するステップを配置し、その後リファレンスのトリガー条件で「見落としチェック」を実施する。

---

### 論点3: 不適用判定 → **明示的スキップ記録を採用（方式Bの貢献）**

**判定**: 方式B寄り

**根拠**:
- 方式Bの「理由コード付き不適用判定」はR1（検証可能性）において方式Aの飽和スキャンより構造的に優れる
- 方式Aが認めた通り、飽和スキャン軸も事前定義の一形態であり、「完全にリストフリー」ではない
- 20〜30件の判定コストは許容範囲（方式Aも認めた）
- 「考慮した上でのスキップ」と「見落とし」の区別は設計品質の重要な指標

**バイアス検証**: 方式Aの「スキャン軸は未知発見に強く、リストエントリは既知確認に強い」という補完関係の指摘を無視していないか？→ 無視していない。統合案では飽和スキャン軸（方式A）とリファレンス照合（方式B）の両方を検証層に配置する。

**方針**: 検証層でリファレンスとの照合時に、各項目について「適用/不適用（理由コード付き）」を記録する。飽和スキャンも併用する。

---

### 論点4: リスト外項目への対応 → **要件駆動導出が主であり、リスト外補完は不要になる**

**判定**: 統合案の構造で解消

**根拠**:
- 統合案では要件駆動導出が主（論点1の判定）のため、「リスト外項目」という概念自体が変質する
- リファレンスは検証層であり、「リスト外 = 導出されたがリファレンスにない項目」は正常な状態（新規識別）
- 方式Bの「制約された問いがLLMの精度を上げる」は、検証層で「この要件に対して以下の検討事項が導出されたが、リファレンス上の関連項目 X, Y は含まれていない。これらは検討不要か？」という形で活用する

**方針**: 「リスト外補完」を独立ステップにせず、検証層での差分チェックとして統合する。

---

### 論点5: メンテナンスコスト → **リファレンスを抽象カテゴリ層に限定して保守コストを最小化**

**判定**: 両方式の折衷

**根拠**:
- 方式Bの「安定層（品質特性・セキュリティ原則）は10年単位で安定」という主張は妥当
- 方式Aの「技術変化速度に追従する具体エントリのメンテナンスは困難」という指摘も妥当
- 解決策: リファレンスを「抽象カテゴリ + トリガー条件」のレベルに留め、具体的な検討事項は動的導出に委ねる

**バイアス検証**: 抽象カテゴリのみでは方式Bの「照合の安定性」が失われないか？→ 抽象カテゴリでも「可観測性カテゴリが適用されているか」「セキュリティカテゴリが適用されているか」の検証は有効。具体的な検討事項の識別は導出に委ねるが、カテゴリレベルの漏れチェックは保持される。

**方針**: リファレンスを「カテゴリ（品質特性・技術横断事項）+ トリガー条件」の2層構造にし、具体的な検討事項のエントリは持たない。メンテナンスはカテゴリ追加・トリガー条件更新のみ。

---

### 論点6: 横断的関心事 → **シード + 構造的検証の組み合わせ**

**判定**: 両方式の統合

**根拠**:
- 両方式とも「予防 + 検証の組み合わせが理想」で収束している
- 方式Aのシード（予防的引き渡し）は構造的にHandoff Gapを減らす
- 方式BのP4（全決定事項スキャン）はシードタグ漏れへのバックアップとして有効

**方針**: arch_designでの決定時にprocess_designへのシードタグを付与（方式A）し、process_designの検証フェーズで全arch決定事項との対応を構造的にチェック（方式B）する。

---

## 統合案: 方針レベルの概要

### 基本構造: 「要件駆動導出 + リファレンス検証」の2層方式

```
[層1: 導出層（主）]
  要件文書を起点に、設計検討事項を動的に導出する
  - 直接導出（要件 → 検討事項）
  - 暗黙要件補完（技術コンテキスト → 検討事項）
  - 依存関係分析

[層2: 検証層（補完）]
  リファレンスカテゴリと導出結果を照合し、漏れを検出する
  - カテゴリレベルの網羅性チェック（全関連カテゴリが導出結果に反映されているか）
  - 不適用判定の明示記録（理由コード付き）
  - 飽和スキャン（軸別の未検討領域チェック）
```

### arch_design への適用イメージ

```
Phase 0: 入力読込・メタデータ抽出（現行通り）

Phase 1-前半: 要件駆動の設計次元導出（層1）
  Step 1: 要件からの直接導出
    「各SR/NFR/制約を実現するために、アーキテクチャレベルで何を決める必要があるか」
  Step 2: 暗黙要件の補完
    「技術コンテキスト（ドメイン・スタック・構成）から、要件に明示されていないが必要な検討事項は何か」
  Step 3: 依存関係分析と設計次元の整理

Phase 1-後半: リファレンス検証（層2）
  Step 4: リファレンスカテゴリとの照合
    - 導出された設計次元がリファレンスの各カテゴリをカバーしているか確認
    - カバーされていないカテゴリについて「不適用（理由コード）」または「導出漏れ → 追加導出」を判定
  Step 5: 飽和スキャン
    - データフロー・インフラ・外部連携・横断的関心事の各軸でスキャン

Phase 1-設計: 確定した設計次元に沿った対話的設計（現行通り）
  - 各次元の設計 + ADR提案
  - process_designへのシードタグ付与

Phase 1-検証: 設計完全性ゲート（現行のQG-3に相当）
```

### process_design への適用イメージ

```
Phase 0: 入力読込・arch決定事項のインポート（現行通り）

Phase 1-前半: arch決定事項からのprocess検討事項導出（層1）
  Step 1: シードタグ付き事項の処理（優先）
  Step 2: 全arch決定事項からのprocess検討事項導出
  Step 3: プロセス固有の暗黙要件補完

Phase 1-後半: リファレンス検証（層2）
  Step 4: process用リファレンスカテゴリとの照合
  Step 5: ハンドオフ検証（全arch決定事項 ↔ process検討事項の対応確認）

Phase 1-設計: 対話的設計（現行通り）

Phase 3: 整合性チェック（現行通り）
```

### リファレンスの構造

```
リファレンス = カテゴリ一覧 + 各カテゴリの定義
  - カテゴリ名
  - トリガー条件（どのコンテキストで適用されるか）
  - arch側の関心事（方針レベルで決めること）
  - process側の関心事（実装レベルで決めること）
  - 典型的な検討事項の例示（3-5個、網羅的リストではなく例示）

具体的な検討事項の詳細リストは持たない（動的導出に委ねる）
```

### 方式が満たすべき条件（R1-R5）の充足

| 条件 | 充足方法 |
|------|---------|
| R1 検証可能性 | 検証層の照合結果（適用/不適用+理由コード）が記録される |
| R2 適応性 | 導出層が要件駆動のため、リファレンスカテゴリ外の事項も自然に識別 |
| R3 実装可能性 | 検証層はカテゴリレベル（10-15カテゴリ程度）で軽量 |
| R4 コスト効率 | 導出層は現行の設計次元導出と同等。検証層はカテゴリレベルの照合で追加コスト小 |
| R5 保守性 | リファレンスは抽象カテゴリのみ。メンテナンスはカテゴリ追加/トリガー更新に限定 |
