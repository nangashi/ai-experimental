# Step 1: 方式A 提案 — 動的導出方式

## 1. 方式の概要

### 基本原理

動的導出方式は、事前定義されたリストを参照するのではなく、**プロジェクトの要件文書（requirements.md）を唯一の起点**として設計検討事項を推論・導出するアプローチである。

処理の核心は「要件 → 設計インパクト分析 → 検討事項の識別」という推論チェーンにある。各要件が持つ技術的インパクトを分析し、そのインパクトが設計フェーズでの判断を要求する事項（= 設計検討事項）として結晶化される。

```
requirements.md
    │
    ▼
[Phase 1] 要件構造化
    │   SR/NFR/制約を解析し、設計インパクトの高い次元で分類
    │
    ▼
[Phase 2] 直接導出（requirement-driven）
    │   各要件から「この要件を満たすために何を決めなければならないか」を導出
    │
    ▼
[Phase 3] 暗黙要件の補完（context-driven）
    │   技術スタック・ドメイン・構成パターンから暗黙的に必要な検討事項を補完
    │
    ▼
[Phase 4] 横断的依存関係の分析
    │   検討事項間の依存・影響関係を識別し、分界を確定
    │
    ▼
[Phase 5] 飽和確認（saturation check）
        「もし自分がこの要件を実装するエンジニアなら、さらに何を決める必要があるか」
        という視点で残余スキャン → 追加がなければ飽和と判断
```

### 静的リストとの本質的違い

| 観点 | 動的導出方式 | 事前定義方式 |
|------|------------|------------|
| 起点 | 要件文書 | リファレンスリスト |
| 方向 | 要件 → 検討事項（帰納） | リスト → 要件マッピング（演繹） |
| 適応性 | プロジェクト固有の事項が自然に出現 | リスト外事項の識別は期待できない |
| 漏れの構造 | 推論の穴（LLM能力依存） | リストの穴（定義時の漏れ） |

## 2. arch_design への適用方針

### 責務

要件 → アーキテクチャ方針・技術選定・コンポーネント構成・インターフェース設計の判断事項を識別する。

### Phase/Step 構成

#### Phase A1: 要件構造化

**入力**: requirements.md
**出力**: 構造化要件マップ（SR/NFR/制約の分類と設計インパクト評価）

```
Step A1-1: 要件の型分類
  - SR（機能要件）: 何を実現するか
  - NFR（非機能要件）: どのレベルで実現するか
  - 制約（外部制約）: 何を前提とするか

Step A1-2: 設計インパクト評価
  各要件に対して「アーキテクチャ決定に直接影響する度合い」を High/Mid/Low で評価
  High: アーキテクチャのコアに影響（例: 99.9% 可用性 → 冗長構成が必須）
  Mid:  特定コンポーネントの選定に影響（例: 大量ファイルアップロード → ストレージ戦略）
  Low:  実装詳細レベルで処理可能（例: 文字コードは UTF-8）

Step A1-3: 技術コンテキスト抽出
  requirements.md から「前提技術スタック」「対象ドメイン」「運用環境」を識別
  → 暗黙要件補完の素材として使用
```

#### Phase A2: 直接導出（Requirement-Driven）

**入力**: 構造化要件マップ
**出力**: 要件トレーサブルな設計検討事項リスト

```
Step A2-1: 各要件への「何を決めるべきか」質問の適用
  テンプレート:
  「[要件X] を実現するために、アーキテクチャレベルで決定しなければならない
   方針・技術・構成・インターフェースは何か？」

  → 各要件から 0〜N 個の設計検討事項（候補）を導出
  → 各候補は（対象, トリガー, 判断基準候補）の形式で記録

Step A2-2: 粒度の正規化
  decision point 粒度（=設計パターンの詳細より上、要件より下）に統一
  過細粒度（例: 変数名の命名規則）→ process_design へ委譲マーク
  過粗粒度（例: 「セキュリティ全体」）→ サブ検討事項に分解
```

#### Phase A3: 暗黙要件の補完（Context-Driven）

**入力**: 技術コンテキスト + 識別済み検討事項
**出力**: 補完済み設計検討事項リスト

```
Step A3-1: ドメイン・スタック起因の暗黙要件識別
  「[技術コンテキスト] を前提とした場合、たとえ要件に明示されていなくても
   設計で必ず対処すべき事項は何か？」
  例: Web API → CSRF/認証セッション管理/レートリミット
      分散システム → 分散トレーシング/サーキットブレーカー
      マルチテナント → データ分離戦略

Step A3-2: NFR間の結合から生じる検討事項の識別
  「[NFR-A] と [NFR-B] を同時に満たすために追加で必要な設計判断はあるか？」
  例: 高可用性 + 強整合性 → CAP定理上のトレードオフ決定が必要
```

#### Phase A4: 横断的依存関係の分析

```
Step A4-1: 検討事項間の依存グラフ構築
  「[検討事項X] を決定したとき、それに依存して初めて決定できる検討事項はあるか？」
  → 依存関係を明示的に記録（後続フェーズの順序決定に使用）

Step A4-2: 横断的関心事の識別とフェーズ分界
  対象: セキュリティ / 可観測性 / エラーハンドリング / 国際化 等
  分界規則:
  - arch_design 担当: 「何を」「どの方針で」（例: 認証方式の選択、ログ戦略の選択）
  - process_design 委譲: 「どのツールで」「どの規約で」（例: ライブラリ選定、規約定義）
  → arch側決定事項を「process_design へのシード」としてタグ付け
```

#### Phase A5: 飽和確認

```
「設計完了時点で、実装チームが『何を使って / どう実現するか』の方針が決まっていないまま
 コーディングを開始せざるを得ないような未決定事項が残っていないか？」

以下の軸でスキャン:
  - データフロー軸: データの入力→処理→出力→永続化の各段階で未決定はないか
  - インフラ軸: 実行環境・スケーリング・デプロイ形態で未決定はないか
  - 外部連携軸: 外部システムとのインターフェース・契約で未決定はないか
  - 横断的関心事軸: セキュリティ・可観測性・エラー処理で方針未定はないか
```

## 3. process_design への適用方針

### 責務

arch_design の決定事項 → 開発プロセス・ツールチェーン・規約・CI/CD の判断事項を識別する。

### Phase/Step 構成

#### Phase P1: arch_design 決定事項の継承

```
Step P1-1: arch_design 出力の読み込み
  - 全設計検討事項の決定内容
  - process_design へのシードタグ付き事項（優先処理）

Step P1-2: 決定事項の process インパクト分析
  「[arch決定X] を実装として実現するために、チームが揃えるべき
   プロセス・ツール・規約・自動化は何か？」
  → 各arch決定からprocess検討事項を導出
```

#### Phase P2: プロセス固有の暗黙要件補完

```
Step P2-1: 開発体制・規模起因の暗黙要件
  「[チーム規模/開発体制] を前提に、プロセスとして必ず定めるべき事項は何か？」
  例: 複数チーム → ブランチ戦略、コードレビュー規約
      継続的デリバリー → feature flag 戦略

Step P2-2: arch決定間の結合から生じるプロセス要件
  「[arch決定A] と [arch決定B] を同じコードベースで並行実装するために
   必要なプロセス上の取り決めは何か？」
```

#### Phase P3: 引き継ぎ完全性の検証

```
Step P3-1: arch → process のマッピング確認
  全arch検討事項に対して、対応するprocess検討事項が存在するか確認
  「process_designシード」タグのある事項が全て対応済みかチェック

Step P3-2: 未対応の識別と補完
  対応するprocess事項が存在しないarch決定を特定し、process検討事項を追加導出
```

## 4. 横断的関心事のarch/process境界の扱い

### 基本原則: 関心分離 × 明示的シード

横断的関心事（可観測性・セキュリティ・エラーハンドリング等）は1回の設計で完結させようとするのではなく、**arch_designが戦略的決定を行い、その決定を「シード」として明示的にprocess_designへ引き渡す**という2フェーズ構造で処理する。

```
可観測性の例:
  arch_design:
    → 「何を観測するか」「どのトレース粒度か」「集中 vs 分散ログ集約か」を決定
    → シード: 「構造化ログ+分散トレーシング採用、APM統合必要」

  process_design:
    → シードを受けて「どのライブラリを使うか」「ログフォーマット規約」
       「OpenTelemetry計装ルール」「アラート閾値の設定プロセス」を決定
```

### 境界が曖昧な場合の解決規則

1. **「なぜ」「何を」の判断 → arch_design**
2. **「どのように」「どのツールで」の判断 → process_design**
3. **判断が分かれる場合**: arch側でシードとして記録し、両フェーズで検討事項として保持（重複許容、引き継ぎで解決）

## 5. 網羅性の保証メカニズム

### 多層の安全網

動的導出は単一パスでは網羅性が不十分になるリスクがある。以下の多層構造でこれを補完する。

#### 層1: 要件ベーストレーサビリティ

全SR/NFR/制約に対して「対応する設計検討事項が1つ以上存在する」かチェック。
対応なしの要件が存在する場合、その要件を起点に再導出を実施。

```
検証クエリ:
「以下の要件一覧のうち、設計検討事項リストに未対応のものはどれか？
 [要件一覧] / [設計検討事項リスト]」
```

#### 層2: 軸別飽和スキャン（Phase A5/P3の実体）

設計の各軸（データフロー・インフラ・外部連携・横断的関心事）を独立したスキャン視点として適用。要件非明示の視点から漏れを探す。

軸は固定（変更不要）であり、メンテナンスコストが発生しない。

#### 層3: 検討事項間の依存チェック

依存グラフで「依存先の未決定が依存元に影響しないか」を確認。
例: API設計を決定しようとしているが、認証方式が未決定 → 識別漏れの信号

#### 層4: 実装者視点での残余スキャン

最終パスとして「このリストを受け取った実装エンジニアが、設計上の空白を感じる箇所はないか」という質問を自己適用する。人間の実装者の思考プロセスをシミュレーションすることで、LLMの推論バイアスとは独立した漏れ検出を図る。

### 識別漏れ検出の信号

- 要件から導出した検討事項の判断基準に「未定の前提」が含まれている
- 複数の検討事項が同じ要件を参照しているが、相互に矛盾する
- シードタグのある事項に対応するprocess検討事項が存在しない

## 6. この方式が依存する仮定

### A1への依存（最大の前提）

**仮定**: LLMは要件文書から関連する設計検討事項を高い網羅性で導出できる

**依存の深さ**: 高。本方式の有効性の核心は A1 の成立にかかっている。A1 が成立しない（体系的な導出漏れが発生する）場合、多層の安全網も補完しきれない可能性がある。

**妥当性の根拠**:
- LLMは設計ドキュメント・アーキテクチャパターン・技術文書の大規模コーパスで訓練されており、要件から設計インパクトへの推論を広くカバーしていると考えられる
- ただし、これは一般的な技術領域に対する信頼であり、高度に特化したドメイン（例: 航空宇宙、医療機器安全）では A1 の成立は弱くなる
- **緩和策**: Phase A3/P2（暗黙要件補完）と多層安全網の組み合わせにより、単一推論の穴を部分的に埋める設計としている

### A2への依存（構造化推論の前提）

**仮定**: LLMは検討事項間の依存関係・横断的関心事を正確に分析できる

**依存の深さ**: 中。Phase A4（依存関係分析）の精度がこれに依存するが、依存関係の網羅性よりも「検討事項の識別」が本方式の主目的であるため、A2 の部分的な失敗は許容できる。依存グラフの誤りは後続フェーズでの矛盾として顕在化しやすい。

**妥当性の根拠**:
- 依存関係分析はパターン化されており（A → B の典型パターン例: 認証方式 → セッション管理）、LLMはこれらを高頻度で学習している
- 完全なグラフよりも「見落としが顕在化する仕組み」（層3の検出メカニズム）の方が重要であり、A2 への絶対的依存を緩和している

### A3への依存（反復改善の前提）

**仮定**: 複数回の導出・検証で網羅性を向上できる

**依存の深さ**: 中。本方式は1回の導出で完結させず、多層安全網として複数パスを明示的に設計している（Phase A2→A3→A5、Phase P1→P2→P3）。A3 が成立しない（反復しても新規発見が出続ける）場合はコスト問題として顕在化するが、品質は単調に向上する方向にある。

**妥当性の根拠**:
- 飽和確認（Phase A5）は「軸別スキャン」という異なるアプローチを採用しており、同じ推論パスの繰り返しではない
- 異なる視点（要件起点 vs. 実装者視点）を組み合わせることで、各視点のバイアスを相殺する設計

## 7. 想定される課題とその対策

### 課題1: LLM推論の非決定性

**問題**: 同一要件から異なる実行で異なる検討事項が導出される可能性がある。

**影響**: 実行ごとに設計の出発点が異なり、品質が安定しない。

**対策**:
- 多層安全網の層1（要件トレーサビリティ）により、要件 → 検討事項のマッピングを構造的に記録し、事後検証を可能にする（R1: 検証可能性を担保）
- 出力形式を「対象・トリガー・判断基準候補」の構造化形式に強制することで、推論の揺れを最小化する

### 課題2: 暗黙要件の識別限界

**問題**: 要件に記述のない技術的必要性（例: CSRF対策、レートリミット）の識別は、LLMの学習データに依存する。専門性の高いドメインでは識別漏れが増加する可能性がある。

**影響**: 漏れの種類「識別漏れ」が発生し、安全網でも補完されない。

**対策**:
- Phase A3（ドメイン・スタック起因の暗黙要件識別）を明示的なStepとして組み込み、技術コンテキストからの推論を強制する
- 飽和スキャンの軸（データフロー・インフラ・外部連携・横断的関心事）は、一般的なアーキテクチャ関心事を網羅しており、明示的な「最低限チェックすべき次元」として機能する
- **残存リスク**: 極めて特殊なドメインでは識別漏れが残る可能性がある。その場合、ユーザーが軸に当該ドメイン固有の次元を追加することで対処する

### 課題3: 粒度の揺れ

**問題**: 導出された検討事項の粒度が inconsistent になる（過細・過粗の混在）。

**影響**: process_designへの引き継ぎや検討事項間の依存分析が困難になる。

**対策**:
- Phase A2-2「粒度の正規化」ステップで明示的に粒度調整を実施
- 「decision point粒度」の操作的定義（「実装チームが実装前に決定していなければならない方針・技術の選択」）をプロンプトに組み込み、粒度の基準をLLMに与える

### 課題4: コンテキストウィンドウの限界

**問題**: 大規模なrequirements.md（SR 30件以上、NFR 15件以上）では、全件を一度に処理するとコンテキストウィンドウを圧迫する。

**影響**: R3（実装可能性）・R4（コスト効率）への影響。

**対策**:
- Phase A1で要件を「設計インパクト High/Mid/Low」に分類し、Highのみで最初の導出を実施
- 飽和確認フェーズで Mid/Low を処理することで、処理を分散する
- ただし、これにより処理フローが複雑化するトレードオフがある

### 課題5: 横断的関心事の境界決定コスト

**問題**: arch/process の境界が曖昧な事項が多い場合、境界決定自体がオーバーヘッドになる。

**影響**: R4（コスト効率）への影響。

**対策**:
- 境界規則を「なぜ/何を → arch、どのように/どのツールで → process」と単純化し、曖昧な場合は両フェーズで重複保持を許容する
- 重複の解消は「引き継ぎ完全性検証」（Phase P3-1）で自然に行われる

## 方式Aの強みの総括

動的導出方式の本質的強みは、**プロジェクトの要件が多様であっても、適応コストがゼロ**という点にある。事前定義方式はリストに存在しない事項を見つけられないという構造的限界を持つが、動的導出方式には「リストの穴」は存在しない。存在するのは「推論の穴」であり、これは多層安全網・構造化プロンプト・反復スキャンという方法論的手当てにより継続的に改善可能である。

また、リファレンスリストのメンテナンスコスト（R5: 保守性）が発生しない点も、長期運用においての優位性となる。
