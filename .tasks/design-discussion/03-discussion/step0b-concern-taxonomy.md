# Step 0b: 検出対象の全体像 — 設計検討事項の分類枠組み

## 1. 分類次元の抽出

### 1.1 品質特性（ISO 25010）

| 次元 | 含まれる検討事項の例 |
|------|-------------------|
| 機能適合性 | 機能の完全性、正確性、適切性 |
| 性能効率 | 時間効率（応答時間）、資源効率（CPU/メモリ）、容量（同時接続数、データ量） |
| 互換性 | 共存性（他システムとの同時稼働）、相互運用性（データ交換、API連携） |
| 使用性 | 学習性、運用性、ユーザーエラー防止、アクセシビリティ |
| 信頼性 | 成熟性（障害頻度）、可用性（稼働率）、障害許容性（部分障害時の挙動）、回復性（復旧時間） |
| セキュリティ | 機密性（認証・認可）、完全性（データ改ざん防止）、否認防止（監査ログ）、責任追跡性、真正性 |
| 保守性 | モジュール性、再利用性、解析性（障害調査容易性）、修正性、テスト容易性 |
| 移植性 | 適応性（環境変更対応）、設置性（デプロイ容易性）、置換性（コンポーネント交換） |

### 1.2 アーキテクチャビューポイント

| 次元 | 含まれる検討事項の例 |
|------|-------------------|
| 論理ビュー | コンポーネント分割、レイヤー構成、責務分担、依存関係方向 |
| 開発ビュー | モジュール構成、パッケージ構造、ビルド単位、開発チーム境界 |
| プロセスビュー | 並行処理、同期/非同期、プロセス間通信、スケーリング単位 |
| 物理ビュー | デプロイ構成、ネットワークトポロジー、インフラ配置、リージョン戦略 |
| シナリオビュー | ユースケース実現パス、コンポーネント間のシーケンス、エラーパス |

### 1.3 ドメイン・ビジネスロジック

| 次元 | 含まれる検討事項の例 |
|------|-------------------|
| 集約境界 | トランザクション境界、整合性境界、集約ルート設計 |
| バウンデッドコンテキスト | コンテキスト間の翻訳戦略、共有カーネル vs 腐敗防止層 |
| ドメインイベント | イベント駆動の境界、結果整合性、イベントソーシング要否 |
| ビジネスルール構造 | ルールの配置（ドメイン層 vs アプリ層）、ルール変更頻度、条件分岐の複雑度 |
| 状態管理戦略 | 状態遷移モデル、状態の永続化戦略、楽観的/悲観的ロック |
| ドメインモデリング方針 | リッチドメインモデル vs トランザクションスクリプト、CQRSの要否 |

### 1.4 ユーザー・体験設計

| 次元 | 含まれる検討事項の例 |
|------|-------------------|
| レスポンス設計 | 知覚パフォーマンス、楽観的UI更新、スケルトンスクリーン |
| オフライン対応 | オフライン時の振る舞い、データ同期戦略、コンフリクト解決 |
| マルチデバイス | レスポンシブ設計、デバイス間データ同期、プラットフォーム差異 |
| エラー体験 | ユーザー向けエラーメッセージ、リトライUI、フォールバック表示 |
| 通知・フィードバック | リアルタイム通知、プッシュ通知、メール通知のアーキテクチャ |

### 1.5 運用・進化

| 次元 | 含まれる検討事項の例 |
|------|-------------------|
| バックアップ・リストア | バックアップ頻度・方式、RPO/RTO、リストア手順 |
| 監視・アラート | メトリクス収集、ログ集約、トレーシング、アラート閾値、ダッシュボード |
| 障害対応 | インシデント対応フロー、自動復旧、サーキットブレーカー、ヘルスチェック |
| データマイグレーション | スキーマ変更戦略、データ移行手順、ゼロダウンタイムマイグレーション |
| 段階的移行 | 既存システムとの並行稼働、機能フラグ、カナリアリリース |
| キャパシティプランニング | スケーリングトリガー、リソース見積もり、負荷テスト計画 |
| ローテーション・廃止 | シークレットローテーション、証明書管理、非推奨機能の移行パス |

### 1.6 規制・コンプライアンス

| 次元 | 含まれる検討事項の例 |
|------|-------------------|
| データ保護規制 | GDPR、個人情報保護法、データ最小化、同意管理、忘れられる権利 |
| 業界固有規制 | PCI DSS（決済）、HIPAA（医療）、SOX（財務）、金融規制 |
| データ所在地 | データレジデンシー要件、リージョン制約、越境データ転送 |
| 監査要件 | 監査ログ要件、証跡保持期間、改ざん防止 |
| アクセシビリティ規制 | WCAG準拠レベル、法的義務（障害者差別解消法等） |

### 1.7 コスト・経済性

| 次元 | 含まれる検討事項の例 |
|------|-------------------|
| インフラコスト | ランニングコスト見積もり、スケーリング時のコスト増加、リージョン別コスト差 |
| ライセンスコスト | 商用ライブラリ、SaaS利用費、OSSライセンス互換性 |
| 開発コスト | 技術選定による学習コスト、移行コスト、採用・育成コスト |
| TCO分析 | 初期構築 vs 運用コストのバランス、マネージドサービス vs 自前運用の比較 |

### 1.8 開発プロセス・チーム

| 次元 | 含まれる検討事項の例 |
|------|-------------------|
| 開発環境 | ローカル開発環境構成、dev container、環境変数管理 |
| ブランチ戦略 | Git flow / trunk-based / GitHub flow、マージ戦略 |
| コード品質 | リンター、フォーマッター、型チェック、コードレビュー方針 |
| テスト戦略 | テストピラミッド、E2E/統合/単体の比率、テストデータ管理 |
| CI/CD | パイプライン構成、デプロイ自動化、品質ゲート配置 |
| 依存管理 | ライブラリ更新方針、脆弱性スキャン、ロックファイル管理 |
| ドキュメント | API仕様生成、アーキテクチャ決定記録（ADR）、開発ガイド |

## 2. カバレッジマップ

分類次元を5つの視点にマッピングする。

| 視点 | 対応する分類次元 | arch_design | process_design |
|------|----------------|-------------|---------------|
| **技術インフラ** | 性能効率、互換性、信頼性、移植性、論理ビュー、プロセスビュー、物理ビュー | 主担当 | 補助（技術選定の具体化） |
| **ドメイン・ビジネスロジック** | 機能適合性、集約境界、バウンデッドコンテキスト、ドメインイベント、ビジネスルール構造、状態管理戦略、ドメインモデリング方針、シナリオビュー | 主担当 | — |
| **ユーザー体験** | 使用性、レスポンス設計、オフライン対応、マルチデバイス、エラー体験、通知・フィードバック | 主担当 | 補助（テスト方針での体験検証） |
| **運用** | バックアップ・リストア、監視・アラート、障害対応、データマイグレーション、段階的移行、キャパシティプランニング、ローテーション・廃止 | 方針策定 | 具体化（CI/CD、監視ツール） |
| **規制** | データ保護規制、業界固有規制、データ所在地、監査要件、アクセシビリティ規制 | 方針策定 | 具体化（品質ゲート、テスト） |
| **開発プロセス** | 開発環境、ブランチ戦略、コード品質、テスト戦略、CI/CD、依存管理、ドキュメント、保守性、開発ビュー | 構造設計 | 主担当 |
| **コスト・経済性** | インフラコスト、ライセンスコスト、開発コスト、TCO分析 | 主担当 | 補助（ツール選定のコスト評価） |
| **セキュリティ** | セキュリティ（ISO 25010の副特性: 機密性・完全性・否認防止・責任追跡性・真正性）| 方針策定 | 具体化（品質ゲート、CI） |

### バランス検証

| 視点 | 分類次元数 | 偏りの有無 |
|------|----------|-----------|
| 技術インフラ | 8 | やや多い（ISO 25010の4品質特性 + 4ビューポイント。技術的視点が多いのは設計フェーズの性質上妥当） |
| ドメイン・ビジネスロジック | 7 | 適切（DDD関連が主。ビジネスロジック複雑なプロジェクトでは全て重要） |
| ユーザー体験 | 6 | 適切 |
| 運用 | 7 | 適切 |
| 規制 | 5 | やや少ない（ただし規制は業界依存が大きく、汎用分類としては妥当） |
| コスト・経済性 | 4 | やや少ないが、制約条件として他の視点（技術インフラ、運用等）に横断的に影響するため独立視点としては妥当 |
| 開発プロセス | 9 | やや多い（process_designの主担当領域であり、detail項目が多いのは妥当） |
| セキュリティ | 5（ISO 25010の副特性） | 適切（他の視点にまたがるセキュリティ関連事項は各視点で扱う） |

**バランス評価**: 全7視点が3つ以上の分類次元を持ち、特定の視点に極端に偏っていない。技術インフラと開発プロセスがやや多いが、設計スキルの性質上妥当。

## 3. 後続ステップへの適用方法

### Step 3b: リファレンスカテゴリの設計時

リファレンスカテゴリを設計する際、カバレッジマップの7視点すべてにカテゴリが対応していることを照合する。カバー外の視点がある場合、以下のいずれかを明記する:
- 別のメカニズム（例: 飽和スキャン）でカバーされる
- 該当フェーズの責務外である（理由を明示）
- カバレッジギャップとして認識し、制約として記録する

### Step 3b: 飽和スキャン軸の設計時

スキャン軸がカバレッジマップの各視点を反映しているか照合する。特に:
- 技術インフラ以外の視点（ドメイン、ユーザー体験、運用、規制）がスキャン軸に含まれているか
- arch_designとprocess_designの境界領域（横断的関心事）がスキャン軸でカバーされているか

### Step 4c: 方式限界分析

カバレッジマップの各視点について、安全網（導出層 + 検証層）がカバーする領域とカバーしない領域をマッピングする。特に:
- LLMの系統的盲点と相関する可能性がある視点を特定する
- プロジェクト固有度が高い視点（ドメイン、規制）のカバレッジを検証する
