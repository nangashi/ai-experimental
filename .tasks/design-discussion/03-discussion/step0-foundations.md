# Step 0: 前提・定義・制約の確立

## 1. 用語定義

### 「設計検討事項」(Design Concern)

**定義**: 設計フェーズ（arch_design, process_design）において、要件・制約・品質特性を満たすために意思決定または方針策定が必要な事項。以下の3種を含む:

- **Decision Point**: 複数の選択肢から1つを選ぶ必要がある判断ポイント（例: DBMSの選定、認証方式の選定）
- **Design Concern**: 明示的な選択肢がなくとも、設計上の方針・戦略を定める必要がある事項（例: エラーハンドリング方針、ログ出力規約）
- **Quality Attribute Scenario**: 品質特性を具体的なシナリオとして記述し、設計で対応する必要がある事項（例: 「同時接続1000ユーザーで応答時間2秒以内」）

### 「漏れ」の種類

| 種類 | 定義 | 例 |
|------|------|-----|
| **識別漏れ** (Identification Gap) | 検討すべき事項が設計フェーズで認識されない | 要件にセキュリティNFRがあるのに認証設計が議論されない |
| **検討深度の不足** (Insufficient Depth) | 事項は識別されたが、決定・方針が不十分なまま放置される | 「キャッシュを使う」と決めたがTTL戦略・無効化方針が未定義 |
| **引き継ぎ漏れ** (Handoff Gap) | フェーズ間で決定事項や前提が正しく伝達されない | 下記の双方向を含む |
| **過剰適用** (Over-inclusion) | 当該プロジェクトに不要な検討事項が混入し、設計が肥大化する | リファレンスリストの全項目を無条件に適用する |

**スコープの注記**: 本議論は主にUnder-inclusion（識別漏れ・深度不足・引き継ぎ漏れ）のリスク低減を対象とする。Over-inclusion（過剰適用）は方式比較時の副次的指標として扱い、方式間でFalse Positive/False Negativeのトレードオフが生じる場合に評価する。

#### 引き継ぎ漏れの双方向定義

- **順方向 (arch → process)**: arch_designの決定がprocess_designの入力として反映されない
  - 例: arch_designでマイクロサービスを採用したが、process_designでモノリス前提のCI/CDを設計
- **逆方向 (process → arch)**: process_design中にarch_designの決定不足が判明するが、フィードバックされない
  - 例: process_designでディレクトリ構造を設計中に、コンポーネント間の責務境界が曖昧であることが判明

### 「網羅性」の評価基準

以下の3条件がすべて満たされるとき、「漏れなく検討できた」とする:

1. **識別網羅**: 入力（要件・制約・品質特性）から導出されるべき検討事項がすべて識別されている
2. **深度十分**: 各検討事項について、後続フェーズの入力ファイル（architecture.md等）に決定・方針が記録されており、後続フェーズの担当者が追加情報なしに作業を開始できる状態にある
3. **引き継ぎ完全**: フェーズ間で決定事項の欠落・矛盾がない（双方向）

**評価の限界**: 識別網羅の完全な検証は原理的に困難（未知の検討事項は検証できない）。実用的には、独立した複数の視点からの交差検証で近似する。

## 2. スコープと前提条件

### 入力の品質に関する前提

- **requirements.md は「確認済み」(Confirmed) だが完全ではない可能性がある**: 要件定義書はユーザーが確認済みだが、暗黙の要件・非機能要件の記述不足がありうる。設計フェーズはこれを補完する責務を持つ。
- **architecture.md はarch_designの出力であり、同様に不完全でありうる**: process_designの入力であるarchitecture.mdも、検討漏れがある前提で設計する。

### 対象の設計フェーズの責務範囲

| フェーズ | 入力 | 責務 | 出力 |
|---------|------|------|------|
| arch_design | requirements.md | アーキテクチャスタイル、技術選定、コンポーネント設計、インターフェース設計、横断的関心事 | architecture.md + ADR群 |
| process_design | architecture.md + requirements.md | ライブラリ選定、プロジェクト規約、ディレクトリ構造、ランタイム管理、品質ツール、品質ゲート、CI/CD | process-design.md + ADR群 |

**責務境界の曖昧領域**: 横断的関心事（ログ、エラーハンドリング、監視）は両フェーズにまたがる。arch_designでは「方針」を、process_designでは「具体的な実装規約」を定める。

**フィードバックパスの扱い**: 逆方向引き継ぎ（process → arch）が判明した場合の対応も方式の評価対象に含める。ただし、arch_designの再実行手順そのものは本議論のスコープ外とし、「逆方向の検出メカニズム」の有無を方式比較の評価軸とする。

### LLMの実行環境の制約

1. **コンテキストウィンドウ**: 有限。大量の情報を同時に参照することに限界がある
2. **サブエージェントの独立性**: 各サブエージェントは独立したコンテキストで動作し、親エージェントの文脈を直接共有しない
3. **ファイル経由の情報共有**: サブエージェント間のデータ受け渡しはファイル経由で行う（コンテキスト中継は非効率）
4. **ツール制約**: `allowed-tools` で利用可能なツールが制限される（両スキルとも Glob, Grep, Read, Write, Edit, Task）

### LLMの系統的特性

5. **知識源の非独立性**: 同一のLLMモデルを使用するため、すべてのサブエージェントが同一の訓練データ・知識ベースに依存する。コンテキストを分離しても「知識源の独立性」は得られない
6. **系統的盲点**: 訓練データに含まれない領域や、訓練データで過小評価されている設計パターンは、すべてのサブエージェントで同様に見落とされる可能性がある
7. **構造的バイアス**: LLMは頻出パターン（Webアプリ、REST API等）に偏る傾向があり、ドメイン固有・業界固有の検討事項を見落としやすい

## 3. 方式が満たすべき条件

### 必須条件

| 条件 | 定義 | 検証方法 |
|------|------|---------|
| **検証可能性** | 漏れの有無を事後的に検証する手段がある | 方式の出力に、検証に必要な中間成果物（チェックリスト照合結果、カバレッジマップ等）が含まれる |
| **適応性** | プロジェクト固有の検討事項に対応できる | 事前定義されたリストに含まれない検討事項も識別・検討できる |
| **実装可能性** | LLMスキルとして実現可能 | 使用するツール（Glob, Grep, Read, Write, Edit, Task）の範囲内で実現でき、コンテキストウィンドウに収まる |

### 望ましい条件

| 条件 | 定義 |
|------|------|
| **コスト効率** | サブエージェント呼び出し回数・コンテキスト消費量が合理的な範囲に収まる |
| **透明性** | 検討事項の導出過程がユーザーに可視化される |
| **段階的改善** | 方式の効果を測定し、フィードバックに基づいて改善できる |

### コスト許容範囲

- サブエージェント呼び出し: 1フェーズあたり10-20回程度（現状の並列レビュー5回 + 統合1回 + 修正1回の約7回と比較して、2-3倍程度まで許容）
- 追加のユーザーインタラクション: 最小限（既存の対話フローに自然に統合できること）

## 4. 方式が依存する仮定の明示

### 方式A（動的導出方式）が依存しうる仮定

| ID | 仮定 | 検証基準 |
|----|------|---------|
| A-1 | LLMは要件文書から関連する設計検討事項を十分に導出できる | 定性的なパイロット評価（複数のサンプルプロジェクトでの試行結果）で評価。定量的な全体集合の定義は困難なため、独立レビュアー群の合意セットを暫定正解とする |
| A-2 | LLMは交差分析（複数の要件を組み合わせて横断的関心事を導出）ができる | 2つ以上の要件の組み合わせから生じる検討事項を識別できる |
| A-3 | 導出プロンプトの品質が導出結果の網羅性を決定する | プロンプト改善により識別率が向上する |

### 方式B（事前定義方式）が依存しうる仮定

| ID | 仮定 | 検証基準 |
|----|------|---------|
| B-1 | 設計検討事項は事前に十分な粒度でカテゴライズ可能 | カテゴリ外の重要な検討事項が5%未満 |
| B-2 | LLMはリファレンスリストの各項目を正確に解釈し、プロジェクトへの該当性を判断できる | 該当性判断の正確性90%以上 |
| B-3 | リファレンスリストはプロジェクトの多様性に対して十分な汎用性を持つ | 異なるドメイン（Web/モバイル/データパイプライン/組込み等）で有効 |

### 共通仮定

| ID | 仮定 | 検証基準 |
|----|------|---------|
| C-1 | LLMは同一のリストに対して繰り返し実行しても概ね同じ結果を返す（再現性） | 同一入力に対する結果のばらつきが20%未満 |
| C-2 | サブエージェントによる検証は、プロンプト構成・役割設定の差異から部分的に異なる観点を提供する。ただしLLMの系統的盲点（制約5）と重複する領域では独立性は成立しない | 検証で新たに発見される検討事項がある（系統的盲点に属さない範囲で） |
| C-3 | ファイル経由の情報共有で、必要な文脈が十分に伝達される | 重要な文脈の欠落が検証で検出されない |
