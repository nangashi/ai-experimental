# Step 3b: 具体手順の詳細設計

## A/Bフィードバックの反映判断

| フィードバック | 提案元 | 判断 | 理由 |
|--------------|--------|------|------|
| 導出根拠記録（要件紐付き）の追加 | A | **採用** | 検証可能性・深度検出・過剰適用制御の3機能。コスト小 |
| 高リスク視点0件時のユーザー確認自動発火 | A | **採用** | 系統的盲点への具体的対策として必要 |
| フェーズ境界尊重の過剰適用制御明記 | A | **採用** | 明記のみ。実装変更なし |
| 逆方向検出の統合方法の具体化 | A | **採用** | 実装の曖昧さ解消に必要 |
| 項目レベル「不確定→該当扱い」の復活 | B | **部分採用** | 「不確定」ステータスの記録は採用するが、自動「該当扱い」は不採用。理由: 項目レベルの不確定はユーザー確認項目として提示する方が精度が高い |
| 視点内アンカー項目の追加 | B | **採用** | 動的導出の品質安定化に有効。3-5個の代表的検討観点を各視点に付与 |
| フェーズ間整合性照合の明示 | B | **採用** | 引き継ぎ検証の一部として組み込む |
| 適用割合監視 | B | **不採用** | 動的導出ではチェックリストの全項目照合を行わないため、「該当率90%超」の指標が適用不可。代替として導出件数の視点間偏りチェックを導入 |

## 方式の全体構造

```
[Phase 0: 初期化] — 既存の Phase 0 と同一
    ↓
[Phase 0.5: 検討事項スキャン] — 新規挿入フェーズ
    Step 1: 8視点フレーム提示 + 視点別動的導出
    Step 2: 交差分析
    Step 3: カバレッジ検証
    Step 4: 深度検証
    Step 5: 境界判定 + 引き継ぎ記録
    ↓
[Phase 1: 対話的設計] — 既存 Phase 1 にスキャン結果を入力として統合
    ↓
[Phase 2以降] — 既存と同一
```

Phase 0.5 はarch_design / process_design の両方に挿入する。process_designではPhase 0.5の冒頭に「引き継ぎ検証」を追加する。

### Step 2: 交差分析

#### 処理

Step 1の結果を踏まえ、サブエージェント（`Task`, `model: "sonnet"`）に交差分析を委譲する。

**プロンプト構成**:
```
Step 1で導出された検討事項と入力文書を基に、以下の交差パターンで
複数要件の組み合わせから生じる横断的検討事項を導出してください。

## 交差パターン
1. 性能要件 × 機能要件 → キャッシュ/非同期/バッチ処理の設計要否
2. セキュリティ要件 × データ要件 → 暗号化粒度/アクセス制御粒度
3. 可用性要件 × コスト制約 → 冗長化方式の選択肢制約
4. 規制要件 × アーキテクチャ → コンプライアンス実現可能性

上記は例示。入力文書の要件から他の有意な交差パターンも発見すること。

## 出力フォーマット
各横断的検討事項について:
- 検討事項名
- 交差元: {要件A} × {要件B}
- 導出根拠: なぜこの組み合わせが検討事項を生むのか
- 影響: Step 1のどの視点の検討事項に影響するか
```

#### 出力

`{work_dir}/concern-scan.md` に「交差分析結果」セクションとして追記。

#### サブエージェントの返答

`result` + `cross_concerns_count`

### Step 3: カバレッジ検証

メインエージェントが実行する（サブエージェント不要）。

#### 処理

1. `{work_dir}/concern-scan.md` を読み込む
2. 視点別件数を確認する
3. **0件視点の処理**:
   - 通常視点（技術インフラ、ドメイン、ユーザー体験、運用、コスト、開発プロセス、セキュリティ）:
     - 「意図的除外」の理由をユーザーに提示し、妥当性を確認する
   - 高リスク視点（規制、業界固有ドメイン）:
     - **ユーザー確認を自動発火**: 「{視点名}に関連する検討事項が0件です。このプロジェクトに{アンカー項目の例}に関する要件はありませんか？」と質問する
4. **視点間偏りチェック**:
   - 特定視点に検討事項が集中している場合（全体の50%超が1視点に集中）、偏りの理由を記録する
5. **項目レベルの不確定チェック**:
   - 導出根拠が曖昧な検討事項（要件IDへの紐付きが弱い、または「一般的なベストプラクティス」のみを根拠とする）を抽出する
   - 不確定項目にはステータス「要確認」を付与し、ユーザーへの確認リストに追加する
   - ただし自動的に「該当」とはしない（ユーザーの判断を待つ）

#### 出力

テキスト出力（ユーザーに提示）:
```
### 検討事項スキャン結果

| 視点 | 件数 | 備考 |
|------|------|------|
| ... | | |

**合計**: {N}件（交差分析含む）

{0件視点がある場合: 除外理由 or 確認質問}
{要確認項目がある場合: 確認リスト}

修正があればお知らせください。問題なければ「ok」と回答してください。
```

- 承認: Step 4 へ
- 修正指示: concern-scan.md を更新して再提示

### Step 4: 深度検証

メインエージェントが実行する。

#### 処理

各検討事項に対して以下のチェックを行う:

| チェック項目 | 合格基準 | 種別対応 |
|------------|---------|---------|
| 決定の明確性 | 後続担当者が判断不要な形で記述されている | DP: 選択肢と選択理由。DC: 方針の記述。QAS: 数値目標と対策 |
| 前提条件の明示 | 決定がどの前提に依存するか | 全種別共通 |
| 除外の理由 | 不採用の選択肢に理由がある | DP のみ |
| 副作用の記録 | 他の検討事項への影響 | 全種別共通 |

**注**: Phase 0.5の時点ではまだ対話的設計（Phase 1）が行われていないため、ここでの深度検証は「検討事項の識別と初期定義の深度」を確認するものであり、最終的な設計決定の深度検証はPhase 1以降で行う。

具体的には、各検討事項について「Phase 1で何を決定する必要があるか」が明確に定義されているかを確認する。

#### 出力

深度不足フラグが付いた検討事項リストを `{work_dir}/concern-scan.md` に追記。

---

### Step 5: 境界判定 + 引き継ぎ記録

#### 処理

**Step 5a: ラベル付与**

各検討事項に以下のラベルを付与する:
- `arch-own`: arch_designで完結
- `process-own`: process_designで完結
- `arch-to-process`: arch_designで方針策定 → process_designで具体化
- `process-to-arch`: process_designが起点だがarch_designへのフィードバック要

ラベル付与の判断基準:
- 検討事項が「方針・アーキテクチャ判断」レベルなら `arch-own` or `arch-to-process`
- 検討事項が「実装規約・ツール選定」レベルなら `process-own`
- 横断的関心事（ログ・エラーハンドリング・監視等）: デフォルトで `arch-to-process`

**過剰適用制御（フェーズ境界尊重）**: arch_designフェーズで「実装規約レベル」の検討事項が導出された場合、`process-own` にラベリングしてprocess_designに委譲する。arch_designフェーズの検討事項数を制御する。

**Step 5b: 引き継ぎ記録の生成**

`arch-to-process` ラベルの検討事項について構造化テンプレートを生成する:

```markdown
## 引き継ぎ事項: {検討事項名}

- **arch_designでの決定**: {方針の内容}
- **process_designで必要な具体化**: {何を決める必要があるか}
- **前提条件**: {process_designが参照すべきarch_designの決定}
- **充足条件（方針層）**: {arch_designが満たすべき深度基準}
- **充足条件（規約層）**: {process_designが満たすべき深度基準}
```

#### 出力

`{work_dir}/concern-scan.md` にラベルと引き継ぎ記録を追記。
`{work_dir}/handoff-record.md` に引き継ぎ記録のみを抽出して保存（process_designの入力用）。

## process_design 固有の追加処理

### Phase 0.5 冒頭: 引き継ぎ検証

process_designのPhase 0.5では、Step 1の前に以下を実行する:

**Step 0: 引き継ぎ検証 + 方針整合性照合**

1. `{work_dir}/handoff-record.md`（arch_designが生成）を読み込む
2. architecture.md の全セクションを走査し、引き継ぎ記録に含まれていない具体化必要箇所がないか確認する
3. **方針整合性照合**: architecture.md の設計決定と、process_designの前提が論理的に整合するか検証する
   - 例: architecture.mdがマイクロサービスを採用 → process_designがモノリス前提のCI/CDを設計していないか
4. 不整合が検出された場合:
   - 「architecture.md の{セクション}と{前提}が矛盾しています」とユーザーに提示
   - 解決を待ってから続行

**逆方向検出（Phase 0.5 全体を通じて）**:

process_designのPhase 0.5のStep 1-5を実行する中で、architecture.mdの記述が不十分で具体化できない箇所を発見した場合:
1. 当該検討事項に `process-to-arch` ラベルを付与
2. 「ブロック状態」としてマーク（他の非ブロック項目は続行）
3. 不足している方針決定の内容を「arch_design再検討リスト」に記録
4. Phase 0.5完了時にブロック項目をユーザーに提示し、対応を求める

## Phase 1 への統合方法

### arch_design の場合

Phase 0.5の検討事項スキャン結果は、Phase 1 Step 0（スコープ分割計画）に以下の形で統合する:

1. `{work_dir}/concern-scan.md` の検討事項を設計次元にマッピングする
   - 既存の設計次元（アーキテクチャスタイル、実行基盤等）に該当する検討事項を割り当てる
   - 既存の次元に収まらない検討事項は、新しい次元として追加する
2. スコープ分割計画の「整合性チェック」で、Phase 0.5の検討事項がいずれかの次元にカバーされていることを確認する

### process_design の場合

Phase 0.5の検討事項スキャン結果は、Phase 1の7項目設計に以下の形で統合する:

1. 各検討事項を7項目（ライブラリ、規約、ディレクトリ、ランタイム、品質ツール、品質ゲート、CI/CD）にマッピングする
2. いずれの項目にも収まらない検討事項がある場合、追加項目として設計する

## カバレッジマップとの照合

Step 0bのカバレッジマップの8視点と、本方式のカバーメカニズムの対応:

| 視点 | カバーメカニズム | カバレッジの限界 |
|------|----------------|----------------|
| 技術インフラ | Step 1のアンカー4項目 + 動的導出 + Step 2の交差分析 | LLMの構造的バイアスによる非標準パターンの見落とし |
| ドメイン・ビジネスロジック | Step 1のアンカー4項目 + SR/UCからの動的導出 | 業界固有のドメイン知識の不足（系統的盲点） |
| ユーザー体験 | Step 1のアンカー4項目 + NFR/SRからの動的導出 | 要件書にUX要件が薄い場合の導出困難 |
| 運用 | Step 1のアンカー4項目 + NFRからの動的導出 | 要件書に運用要件が明示されにくい（Step 3でユーザー確認） |
| 規制 [高リスク] | Step 1のアンカー4項目 + 業界推定プロンプト + Step 3の自動確認 | LLMの系統的盲点。自動確認で緩和 |
| コスト・経済性 | Step 1のアンカー4項目 + 制約からの動的導出 | 具体的な金額見積もりは不可。検討要因の識別のみ |
| 開発プロセス | Step 1のアンカー4項目 + architecture.mdからの動的導出 | process_designのみ。arch_designでは対象外 |
| セキュリティ | Step 1のアンカー4項目 + NFR/制約からの動的導出 + Step 2の交差分析 | 業界固有の脅威モデルの見落とし |

**カバー外の領域**:
- LLMの訓練データに含まれない規制・業界固有パターン → Step 3の高リスク視点ユーザー確認で緩和
- 要件書に記載されていない暗黙の要件 → Step 3の0件チェック + 項目レベル不確定チェックで検出を試みるが、完全な検出は保証されない
