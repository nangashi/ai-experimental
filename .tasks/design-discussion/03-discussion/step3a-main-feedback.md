# Step 3a: メインエージェントのフィードバックと方針案

## 弁論の総括

3ラウンドの弁論を通じて、以下の構造的な結論が明らかになった。

### 議論で合意が形成された点

1. **純粋な単一方式は最適ではない**: 両擁護者が論点1・2でハイブリッドが最適と評価した
2. **LLMの系統的盲点は両方式共通の制約**: ドメイン固有・規制領域で、導出もリスト照合もLLMの知識に依存する限界がある
3. **引き継ぎには動的要素が不可欠**: アーキテクチャ決定に依存する引き継ぎ内容は静的チェックリストでは表現できない
4. **検証可能性は二面的**: 「何を確認したか」（方式B）と「なぜ必要か」（方式A）は代替ではなく補完関係

### 争点が残る点

1. **Over/Under-inclusionのバランス点**: 方式Aは「関連性フィルタリング」、方式Bは「不確定→該当扱い」。Under-inclusion抑制が主目標である以上、両者のどちらの強度で運用するかの判断が必要
2. **コスト制御**: 方式Bの「不確定率に比例するコスト増」と方式Aの「導出品質に依存するコスト変動」のどちらが管理しやすいか

---

## バイアス低減の検証

メインエージェントとして、以下の観点でバイアスがないか検証する。

### 方式Aに有利な判断の検証

**論点3（Over/Under-inclusion）で方式A優位と評価される根拠**:
- 方式Bの「不確定→該当扱い」のコスト爆発リスクは具体的（要件書が薄いほどコスト増大）
- しかし方式Aの「関連性フィルタリング」も、LLMの判定精度に依存するという同構造の問題を持つ
- **検証結果**: 方式Aの優位性は「条件付き」であり、プロンプト品質が前提。この条件が明示されているため、バイアスではなく条件付き判断として妥当

**論点4（引き継ぎ）で方式A優位と評価される根拠**:
- 「イベントソーシング固有の引き継ぎ」の具体例は説得力がある
- しかし方式Bの「ブロック状態マーク」も逆方向検出の構造を持つ
- **検証結果**: 「僅差」という評価は妥当。方式Bの構造的安定性（実行時揺れの少なさ）も認めるべき

### 方式Bに有利な判断の検証

**論点5（検証可能性）で引き分けと評価される根拠**:
- 方式Bの「非該当の根拠記録」による除外追跡は方式Aにない機能
- 方式Aの「因果構造記録」は方式Bにない機能
- **検証結果**: 引き分け評価は妥当

## 方針レベルのハイブリッド案

弁論と検証を踏まえ、以下の方針を提案する。

### 基本構造: 「構造化フレーム + 動的導出」

```
[Phase 1: 構造化フレーム提示]
  カバレッジマップの8視点を強制的に検討対象として提示
      ↓
[Phase 2: 視点別動的導出]
  各視点内で、入力文書から検討事項をLLMが動的に導出
  + 視点間の交差分析で横断的検討事項を補完
      ↓
[Phase 3: カバレッジ検証]
  各視点の識別件数を確認。0件視点には「意図的除外」の明示記録を要求
  + 適用判定テーブルの生成（「何を確認したか/しなかったか」の証跡）
      ↓
[Phase 4: 深度検証]
  各検討事項の充足条件チェック
      ↓
[Phase 5: 境界判定と引き継ぎ記録]
  arch/process境界の動的ラベリング + 構造化引き継ぎ記録
```

### 方式A/Bからの採用判断

| 要素 | 採用元 | 理由 |
|------|--------|------|
| 8視点の強制検討フレーム | 方式B | 視点単位の構造的カバレッジ保証。「何も出ない」リスクの構造的排除 |
| 視点内の検討事項導出 | 方式A | プロジェクト固有の検討事項への適応力。チェックリスト汎用性の限界を回避 |
| 交差分析 | 方式A | 複数要件の組み合わせから生じる横断的関心事の識別。方式Bにはない機能 |
| カバレッジ検証（0件チェック） | 方式A+B統合 | 0件視点の「意図的除外記録」（方式A）+ 適用判定テーブル形式の証跡（方式B） |
| 深度検証基準 | 方式A+B統合 | 方式Aの4項目チェックリスト + 方式Bの3種別（Decision Point/Design Concern/QA Scenario）別基準 |
| 境界判定 | 方式A | 動的ラベリング（arch-own/process-own/arch-to-process/process-to-arch） |
| 引き継ぎ記録 | 方式A | 構造化テンプレート（前提条件フィールド含む） |
| 逆方向検出 | 方式A+B統合 | 明示的フェーズ（方式A）+ ブロック状態記録（方式B）の統合 |
| 過剰適用制御 | 方式A | 関連性フィルタリング（要件紐付き要求）。「不確定→該当扱い」は不採用 |
| 非該当記録 | 方式B | 除外判定の根拠記録は検証可能性に寄与 |

### 「不確定→該当扱い」の不採用理由

方式Bの保守的ルールは、要件書が薄いプロジェクトでコスト爆発リスクがあり、step0のコスト許容範囲（10-20回/フェーズ）を超える可能性がある。代わりに以下のアプローチを採用する:

- 各視点で0件の場合: 「意図的除外」の理由記録を要求（サイレント通過の防止）
- 要件書から導出できない検討事項: ユーザーへの確認項目として提示（LLMが勝手に「該当」と判定しない）
- 除外判定: 根拠を記録し、事後検証を可能にする

### arch_design / process_design への適用

**arch_design**: Phase 0（初期化）後、Phase 1（対話的設計）の前に「検討事項スキャン」フェーズを挿入。8視点 × 動的導出で検討事項を識別し、スコープ分割計画に反映する。

**process_design**: Phase 0（初期化）後、Phase 1（対話的設計）の前に「引き継ぎ検証 + 検討事項スキャン」フェーズを挿入。arch_designの引き継ぎ記録を検証し、process固有の検討事項を8視点で導出する。

### 横断的関心事の扱い

方式Bの「方針層/規約層の2段階充足条件」は、静的チェックリストとしてではなく、導出された検討事項の**分類基準**として採用する:
- arch_designで識別された横断的関心事には「方針層充足」を要求
- process_designでは「方針層の存在確認」+「規約層充足」を要求
- 方針層が未充足の場合は逆方向フィードバック候補として記録
