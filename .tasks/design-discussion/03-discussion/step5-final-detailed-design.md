# 最終詳細設計: 「構造化フレーム + 動的導出」ハイブリッド方式

> 本文書はstep3b（詳細設計）にstep5（最終サマリ）の全16件の対応方針を反映した統合版である。
> 各修正箇所には `[修正N]` タグを付与する。

---

## 1. A/Bフィードバックの反映判断

| フィードバック | 提案元 | 判断 | 理由 |
|--------------|--------|------|------|
| 導出根拠記録（要件紐付き）の追加 | A | **採用** | 検証可能性・深度検出・過剰適用制御の3機能。コスト小 |
| 高リスク視点0件時のユーザー確認自動発火 | A | **採用** | 系統的盲点への具体的対策として必要 |
| フェーズ境界尊重の過剰適用制御明記 | A | **採用** | 明記のみ。実装変更なし |
| 逆方向検出の統合方法の具体化 | A | **採用** | 実装の曖昧さ解消に必要 |
| 項目レベル「不確定→該当扱い」の復活 | B | **部分採用** | 「不確定」ステータスの記録は採用するが、自動「該当扱い」は不採用。項目レベルの不確定はユーザー確認項目として提示する |
| 視点内アンカー項目の追加 | B | **採用** | 動的導出の品質安定化に有効。3-5個の代表的検討観点を各視点に付与 |
| フェーズ間整合性照合の明示 | B | **採用** | 引き継ぎ検証の一部として組み込む |
| 適用割合監視 | B | **不採用** | 動的導出ではチェックリストの全項目照合を行わないため「該当率90%超」の指標が適用不可。代替として導出件数の視点間偏りチェックを導入 |

## 2. 方式の全体構造

```
[Phase 0: 初期化] — 既存 Phase 0 + 構造化質問（業界・組織固有制約の確認）[修正13]
    ↓
[Phase 0.5: 検討事項スキャン] — 新規挿入フェーズ
    Step 1: 8視点フレーム提示 + 視点別動的導出（ドメイン適応指示付き、1視点10件上限）[修正7,10]
    Step 2: 交差分析（独立再走査パス + 統合指示 + 4パターン必須確認）[修正6,8,11]
    Step 3: カバレッジ検証（重複検出 + 高リスク視点ポジティブ確認）[修正3,5,12]
    Step 4: 初期深度検証（Phase 0.5用基準に限定）[修正1]
    Step 5: 境界判定（デフォルトルール付き）+ 引き継ぎ記録 [修正4,16]
    ↓
[Phase 1: 対話的設計] — メインエージェントがStep 0で統合マッピング実行 [修正9]
    ↓
[Phase 2以降] — 既存と同一
```

Phase 0.5 はarch_design / process_design の両方に挿入する。process_designではPhase 0.5の冒頭に「引き継ぎ検証」（Step 0）を追加する。

### 高リスク視点の定義 [修正3]

高リスク視点は以下の2視点とし、本文書内で**視点番号で一意に参照**する:
- **視点2: ドメイン・ビジネスロジック**（業界固有ドメインを含む）
- **視点5: 規制**

Step 1のプロンプト、Step 3の処理、その他すべての参照箇所で同一の視点番号を使用する。

## 3. Phase 0 への追加: 構造化質問 [修正13]

Phase 0の既存処理（要件抽出、QG-1等）完了後、以下の構造化質問をユーザーに提示する。

```
### 業界・組織固有制約の確認

以下の質問に回答してください。該当しない場合は「なし」と回答してください。

1. **業界規制**: このプロジェクトが準拠すべき業界固有の規制・標準はありますか？
   （例: PCI DSS、HIPAA、SOX、金融庁ガイドライン、GDPR等）

2. **組織固有制約**: 技術選定やアーキテクチャに関する組織固有のルール・制約はありますか？
   （例: 承認済みクラウドプロバイダー、禁止技術、必須ミドルウェア等）

3. **データ分類**: このシステムが扱うデータの中で、特別な保護が必要なカテゴリはありますか？
   （例: 個人情報、決済情報、医療情報、営業秘密等）

4. **外部連携**: 連携先の外部システムが課す制約はありますか？
   （例: API仕様のバージョン、認証方式、SLA等）
```

回答結果はPhase 0.5 Step 1の入力に含める。

**方式の制約（L-1）**: この構造化質問は要件書の品質が低い場合の補完措置であるが、ユーザーが認識していない制約は回答に含まれないため、完全な補完は保証されない。

### Step 4: 初期深度検証 [修正1]

メインエージェントが実行する。

#### 処理

Phase 0.5の時点ではPhase 1（対話的設計）が未実施であるため、確認するのは**「検討事項の定義の完全性」のみ**とする。設計決定の深度検証はPhase 1以降で行う。

| チェック項目 | 合格基準 |
|------------|---------|
| 検討事項の明確性 | Phase 1で「何を」「どのレベルで」決定すべきかが明記されている |
| 導出根拠の存在 | 少なくとも1つの要件ID（SR/NFR/IR/制約）への紐付きがある、または「要確認」ステータスが付与されている |
| 視点への帰属 | 8視点のいずれかに分類されている |

不合格の検討事項には「定義不完全」フラグを付与し、Phase 1での優先的な深掘り対象とする。

#### 出力

深度不足フラグが付いた検討事項リストを `{work_dir}/concern-scan.md` に追記。

### Step 5: 境界判定 + 引き継ぎ記録

#### 処理

**Step 5a: ラベル付与**

各検討事項に以下のラベルを付与する:
- `arch-own`: arch_designで完結
- `process-own`: process_designで完結
- `arch-to-process`: arch_designで方針策定 → process_designで具体化
- `process-to-arch`: process_designが起点だがarch_designへのフィードバック要

ラベル付与の判断基準:
- 検討事項が「方針・アーキテクチャ判断」レベルなら `arch-own` or `arch-to-process`
- 検討事項が「実装規約・ツール選定」レベルなら `process-own`
- 横断的関心事（ログ・エラーハンドリング・監視等）: デフォルトで `arch-to-process`
- **デフォルトルール** [修正4]: 上記基準で判断が曖昧な場合は `arch-to-process` とする（安全側: 方針層と規約層の両方で検討される）
- **例外** [修正4]: 明確にツール固有の設定（ESLint設定、Dockerfile等）は `process-own`

**過剰適用制御（フェーズ境界尊重）**: arch_designフェーズで「実装規約レベル」の検討事項が導出された場合、`process-own` にラベリングしてprocess_designに委譲する。arch_designフェーズの検討事項数を制御する。

**Step 5b: 引き継ぎ記録の生成**

`arch-to-process` ラベルの検討事項について構造化テンプレートを生成する:

```markdown
## 引き継ぎ事項: {検討事項名}

- **arch_designでの決定**: {方針の内容}
- **process_designで必要な具体化**: {何を決める必要があるか}
- **前提条件**: {process_designが参照すべきarch_designの決定}
- **充足条件（方針層）**: {以下の固定基準で評価} [修正16]
- **充足条件（規約層）**: {以下の固定基準で評価} [修正16]
```

**充足条件の固定基準** [修正16]:

## 5. process_design 固有の追加処理

### Phase 0.5 冒頭: 引き継ぎ検証

process_designのPhase 0.5では、Step 1の前に以下を実行する:

**Step 0: 引き継ぎ検証 + 方針整合性照合**

1. `{work_dir}/handoff-record.md`（arch_designが生成）を読み込む

2. architecture.md の全セクションを走査し、引き継ぎ記録に含まれていない具体化必要箇所がないか確認する

   **「具体化必要箇所」の判定ルール** [修正2]:
   以下の構造的特徴のいずれかに該当するセクションを「具体化必要箇所」と判定する:
   - architecture.mdに「方針」「ガイドライン」「原則」等の記述があり、具体的なツール名・設定値・手順が伴っていない
   - 「process_designで具体化」「詳細はプロセス設計で決定」等の明示的な委譲記述がある
   - ADR（Architecture Decision Record）でstatusが「proposed」または「決定済み・要具体化」となっている
   - 横断的関心事（ログ、エラーハンドリング、監視等）で方針のみ記述され、実装規約が未定義

3. **方針整合性照合**: architecture.md の設計決定と、process_designの前提が論理的に整合するか検証する
   - 例: architecture.mdがマイクロサービスを採用 → process_designがモノリス前提のCI/CDを設計していないか
   - 例: architecture.mdが特定の認証方式を決定 → process_designのセキュリティテスト方針が整合しているか

4. 不整合が検出された場合:
   - 「architecture.md の{セクション}と{前提}が矛盾しています」とユーザーに提示
   - 解決を待ってから続行

**逆方向検出（Phase 0.5 全体を通じて）**:

process_designのPhase 0.5のStep 1-5を実行する中で、architecture.mdの記述が不十分で具体化できない箇所を発見した場合:
1. 当該検討事項に `process-to-arch` ラベルを付与
2. 「ブロック状態」としてマーク（他の非ブロック項目は続行）
3. 不足している方針決定の内容を「arch_design再検討リスト」に記録
4. Phase 0.5完了時にブロック項目をユーザーに提示し、対応を求める

## 8. カバレッジマップとの照合

Step 0bのカバレッジマップの8視点と、本方式のカバーメカニズムの対応:

| 視点 | カバーメカニズム | カバレッジの限界 |
|------|----------------|----------------|
| 1. 技術インフラ | Step 1のアンカー4項目 + 動的導出 + Step 2の交差分析 | LLMの構造的バイアスによる非標準パターンの見落とし |
| 2. ドメイン・ビジネスロジック [高リスク] | Step 1のアンカー4項目 + SR/UCからの動的導出 + ドメイン適応 [修正10] + ポジティブ確認 [修正12] | 業界固有のドメイン知識の不足（系統的盲点）。ポジティブ確認で緩和 |
| 3. ユーザー体験 | Step 1のアンカー4項目 + NFR/SRからの動的導出 | 要件書にUX要件が薄い場合の導出困難 |
| 4. 運用 | Step 1のアンカー5項目（シークレット管理追加 [修正15]）+ NFRからの動的導出 | 要件書に運用要件が明示されにくい（Step 3でユーザー確認） |
| 5. 規制 [高リスク] | Step 1のアンカー4項目 + 業界推定プロンプト + ポジティブ確認 [修正12] | LLMの系統的盲点。ポジティブ確認で緩和 |
| 6. コスト・経済性 | Step 1のアンカー4項目 + 制約からの動的導出 | 具体的な金額見積もりは不可。検討要因の識別のみ |
| 7. 開発プロセス | Step 1のアンカー4項目 + architecture.mdからの動的導出 | process_designのみ。arch_designでは対象外 |
| 8. セキュリティ | Step 1のアンカー4項目 + NFR/制約からの動的導出 + Step 2の交差分析 | 業界固有の脅威モデルの見落とし |

**カバー外の領域**:
- LLMの訓練データに含まれない規制・業界固有パターン → Step 3の高リスク視点ポジティブ確認で緩和 [修正12]
- 要件書に記載されていない暗黙の要件 → Phase 0の構造化質問 [修正13] + Step 3の0件チェック + 項目レベル不確定チェックで検出を試みるが、完全な検出は保証されない
- アンカー外かつ要件外の事項 → アンカー項目の段階的拡充で部分緩和 [修正15]

## 9. 方式の制約・限界

本方式の原理的な制約を以下に記載する。これらは方式の設計上解消が困難であり、運用上の緩和策で対処する。

| ID | 制約 | 影響 | 緩和策 |
|----|------|------|--------|
| L-1 | 要件書品質への線形依存 [修正13,14] | 要件書が薄いほど識別精度が低下 | Phase 0の構造化質問で補完。ただし完全な補完は不可 |
| L-2 | 同一LLMの系統的盲点に対する独立検証なし [修正14] | 訓練データに含まれない領域は全ステップで見落とされる | 高リスク視点のポジティブ確認でユーザーの知識を活用。外部レビューの推奨。業界キーワード検出時の高リスク視点拡張 |
| L-3 | アンカー外かつ要件外の事項は原理的に検出困難 [修正15] | 運用上重要だが要件書に記載されない事項（シークレット管理等）の検出が保証されない | アンカー項目の段階的拡充で部分緩和（運用視点にシークレット管理を追加済み） |
| L-4 | 規制の網羅性は保証されない | 業界固有・地域固有の規制の一部が検出されない | ユーザー確認の自動発火 + ポジティブ確認。専門家レビューの推奨 |
| L-5 | 導出品質はプロンプト品質に依存 | プロンプト品質が低い場合、識別精度が大幅に低下 | パイロット評価による反復改善。アンカー項目による品質の底上げ |
