# 提案A: 動的導出方式

## 1. 方式の概要

動的導出方式とは、事前に固定されたチェックリストに依存せず、**入力となる要求（requirements.md）の内容を分析し、そこから設計上決定すべき事項を都度導出する**方式である。

基本的な考え方は以下の通り:

- 要求の中に含まれる機能要件(SR)、非機能要件(NFR)、インターフェース要件(IR)、制約の各項目が、それぞれどのような設計判断を必要とするかを推論する
- 要件間の関係性（依存、競合、相互作用）から、単一要件からは見えない横断的な検討事項を浮き彫りにする
- プロジェクト固有の特性に応じて検討の粒度と範囲を適応的に調整する

つまり、「何を検討すべきか」自体を要求から動的に生成するアプローチである。

## 2. arch_designへの適用方針

### 検討次元の導出方法

アーキテクチャ設計における検討次元は、要求を以下の3段階で分析して導出する:

**Stage 1: 要件駆動の検討事項抽出**

各要件を個別に走査し、その要件が暗示するアーキテクチャ上の決定事項を抽出する。

- SR（機能要件）→ ドメインモデル、サービス境界、データフロー、外部連携の検討が必要か
- NFR（非機能要件）→ 性能設計、可用性設計、セキュリティ設計、スケーラビリティ設計が必要か
- IR（インターフェース要件）→ API設計、プロトコル選定、認証方式の検討が必要か
- 制約 → 技術スタック制約、環境制約、コスト制約がどの設計判断に影響するか

**Stage 2: 要件間交差分析**

複数の要件を組み合わせたときに初めて現れる検討事項を導出する。

例: 「リアルタイム通知（SR）」×「99.9%可用性（NFR）」→ メッセージングインフラの冗長化設計が必要、という検討事項は個別要件からは見えにくい。

**Stage 3: アーキテクチャ決定の依存グラフ構築**

導出された検討事項間の依存関係を分析し、検討順序を決定する。

```
アーキテクチャスタイル選定
  → コンポーネント分割
    → コンポーネント間通信方式
      → データ一貫性戦略
```

依存関係がない検討事項は並行して決定できる。循環依存がある場合は、仮決定→検証→確定のイテレーションで解消する。

### トリガーとなる情報の例

| 要件の特徴 | トリガーされる検討次元 |
|---|---|
| 複数の外部システム連携がある | 統合パターン（API Gateway, イベント駆動等）の選定 |
| 高い可用性要件がある | 冗長化戦略、フェイルオーバー方式 |
| マルチテナントの記述がある | テナント分離方式（DB分離、スキーマ分離、行レベル分離） |
| 大量データ処理の記述がある | バッチ処理アーキテクチャ、データパイプライン設計 |
| リアルタイム性の要件がある | 通信方式（WebSocket, SSE, Polling）、イベント駆動設計 |

これらは例示であり、実際にはLLMが要件の意味を解釈して検討次元を動的に生成する点が本方式の核心である。

## 3. process_designへの適用方針

### 実装レベル検討事項の導出

process_designでは、**arch_designの出力（architecture.md）と元の要求（requirements.md）の両方**をインプットとして、実装レベルで決定すべき事項を導出する。

導出の観点:

1. **アーキテクチャ決定の実装詳細化**: architecture.mdで決定された各アーキテクチャ要素について、「これを実装するために何を決める必要があるか」を導出する
   - 例: 「RESTful API」が選定された → HTTPフレームワーク選定、ルーティング規約、リクエスト/レスポンスフォーマット規約、バリデーション方式
2. **要件から直接導出される実装方針**: 要求に含まれる非機能要件のうち、実装レベルで対応するものを抽出する
   - 例: 「監査ログ」要件 → ロギングライブラリ選定、ログフォーマット規約、ログレベル運用方針
3. **開発プロセス関連事項**: プロジェクトの規模・チーム構成・制約から開発プロセスに関する決定事項を導出する
   - 例: コーディング規約、ブランチ戦略、CI/CDパイプライン構成、テスト戦略

### arch_designの出力の活用

architecture.mdの各セクションを走査し、実装で具体化すべき箇所を特定する:

- **コンポーネント定義** → 各コンポーネントの実装技術選定、ディレクトリ構成
- **通信方式** → 具体的なライブラリ・プロトコルバージョン、シリアライゼーション形式
- **データ設計** → ORM/クエリビルダ選定、マイグレーション戦略、シーディング方針
- **セキュリティ設計** → 認証ライブラリ、暗号化実装、シークレット管理方式

### ライブラリ選定・規約・CI/CDの特定

これらは固定的なカテゴリではなく、要件とアーキテクチャから導出される:

- アーキテクチャで「イベント駆動」が選定されている → メッセージブローカーの具体的な製品選定とクライアントライブラリ選定が必要
- アーキテクチャで「コンテナデプロイ」が決定されている → コンテナオーケストレーション、CI/CDパイプラインのコンテナビルド・デプロイステップ設計が必要
- 制約に「チーム規模3名」がある → ブランチ戦略はトランクベース開発が候補、重厚なレビュープロセスは不要

## 4. arch_design⇔process_designの横断的関心事の扱い

### 基本原則: 抽象度による分割

横断的関心事は「抽象度」で分割する:

| 関心事 | arch_designで決めること | process_designで決めること |
|---|---|---|
| ログ | ログの集約先、保持期間、可観測性戦略 | ロギングライブラリ、ログフォーマット、ログレベル規約 |
| エラーハンドリング | エラー伝播戦略、障害分離境界、リトライポリシー | エラークラス設計、例外処理規約、エラーコード体系 |
| 監視 | 監視対象メトリクス、アラート閾値、SLO定義 | 監視ライブラリ、メトリクス収集実装、ヘルスチェック実装 |
| 認証・認可 | 認証方式（OAuth2, OIDC等）、認可モデル（RBAC, ABAC等） | 認証ライブラリ、トークン管理実装、ミドルウェア構成 |

### 動的導出による境界決定

事前にテーブルを固定するのではなく、各横断的関心事について以下を動的に判定する:

1. **この関心事はシステム全体の構造に影響するか？** → Yes ならarch_design
2. **この関心事は特定の技術選定や実装パターンの問題か？** → Yes ならprocess_design
3. **両方に該当するか？** → arch_designで方針を決定し、process_designで実装詳細を決定する

この判定自体を要件の文脈に応じて行うことで、プロジェクト固有の事情（例: ログが規制要件で特に重要なプロジェクトでは、ログの扱いがアーキテクチャレベルの関心事になる）に適応できる。

### 一貫性の確保

process_designの導出時にarch_designの出力を参照することで、アーキテクチャ決定との一貫性を構造的に保証する。具体的には:

- process_designの各決定事項に対して、関連するarch_designの決定を紐づける
- 矛盾が検出された場合（例: arch_designでステートレス方針なのにprocess_designでセッションライブラリを選定）はフラグを立てる

## 5. 網羅性の保証メカニズム

### なぜ動的導出で漏れが防げるのか

**本質的な理由: 検討事項は要件から決まるものであり、事前に全パターンを列挙することは原理的に不可能だから。**

事前定義方式の限界:
- チェックリストは過去の経験の集積であり、未経験のドメインや要件パターンに対して盲点がある
- 汎用チェックリストは「どのプロジェクトにも当てはまる」ものに留まり、プロジェクト固有の検討事項をカバーしない
- チェックリストの項目が多すぎると、該当しない項目のスキップ判断自体がコストになる

動的導出方式の優位性:
- **要件の意味を解釈する**ため、明示されていない暗黙の検討事項（例: 「ユーザー登録」機能からメール送信インフラの検討を導出）も浮き彫りにできる
- **要件間の相互作用を分析する**ため、単一要件のチェックリスト走査では見逃す複合的な検討事項を捕捉できる
- **プロジェクトの文脈に適応する**ため、不要な検討事項でノイズを生じさせない

### 具体的なチェック・検証メカニズム

動的導出の結果が十分であることを保証するために、以下の多層検証を行う:

**検証1: 要件トレーサビリティ検証**

全ての要件（SR, NFR, IR, 制約）に対して、少なくとも1つの設計決定が紐づいていることを検証する。紐づいていない要件がある場合、その要件に対する設計検討が漏れている可能性がある。

**検証2: 決定完全性検証**

導出された各検討事項について、「この事項が未決定のまま実装に進んだ場合、実装者が独自判断を迫られるか？」を検証する。Yesの場合、その事項は設計段階で決定すべきである。

**検証3: 横断的関心事の網羅性検証**

導出された検討事項を、arch_designとprocess_designの両方の視点から走査し、一方でのみ扱われている横断的関心事がないか検証する。

**検証4: 自己批判的レビュー**

導出結果に対して「この要件の組み合わせで、他に考慮すべきことはないか？」という反省的な問いを立て、追加の検討事項がないか最終確認する。

## 6. 想定される課題とその対策

### 課題1: 導出の再現性

**問題**: LLMによる動的導出は、実行ごとに異なる検討事項が生成される可能性がある。

**対策**:
- 導出プロセスを構造化する（Stage 1→2→3の段階的分析）ことで、導出の方向性を安定させる
- 検証メカニズム（特にトレーサビリティ検証）により、漏れがあれば検出・補完する
- 導出結果をユーザーに提示し、レビューを受けるステップを設けることで、人間の知見で補完する

### 課題2: 過剰導出

**問題**: LLMが過度に多くの検討事項を導出し、不要な検討にコストをかける可能性がある。

**対策**:
- 導出された各検討事項に対して、「この事項を決定しないと実装にどのような影響があるか」の影響度を評価し、影響度が低い事項は除外する
- 制約（技術スタック、チーム規模、スケジュール等）を導出のフィルタとして活用し、検討の必要がない事項を早期に刈り込む

### 課題3: 導出の品質がLLMの能力に依存

**問題**: 特殊なドメイン（医療、金融規制等）では、LLMの知識が不足して重要な検討事項を見落とす可能性がある。

**対策**:
- ドメイン固有の制約や規制要件は、requirements.mdに明記されていることを前提とする（要求定義フェーズの責務）
- 要件に「準拠すべき規格・規制」が含まれている場合、その規格が要求する設計上の考慮事項を明示的に導出対象に含める

### 課題4: arch_designとprocess_designの境界判定の曖昧さ

**問題**: 「アーキテクチャレベルか実装レベルか」の判定がLLMの解釈に依存し、一貫しない可能性がある。

**対策**:
- 判定基準を明確に定義する:「複数コンポーネントに影響する、または変更時にシステム全体の再設計が必要になる決定はアーキテクチャレベル」「単一コンポーネント内で完結し、他への影響なく変更可能な決定は実装レベル」
- 境界が曖昧な場合は、arch_designで方針を決定し、process_designで詳細化するという原則を適用する（漏れるより重複する方が安全）
