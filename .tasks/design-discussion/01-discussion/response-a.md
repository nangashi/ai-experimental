# 動的導出方式の立場からのハイブリッド方式への意見

## 1. 核心的な利点が保たれているかの評価

### 保たれている利点

**要件駆動の動的導出（Phase 2）が独立したフェーズとして確保されている**のは良い設計である。動的導出方式の最大の利点は「要件間交差分析によるリスト外の検討事項の発見」であり、Phase 2がこれを担うことで、事前定義のリストに閉じない探索能力が維持されている。

**多層検証（Phase 5）が組み込まれている**のも適切。要件トレーサビリティ検証はリファレンスリスト走査とは独立した検証軸であり、両方を行うことで検証の厚みが増す。

**横断的関心事の境界シフトが明示的に記録される設計**は、議論で浮かび上がった「デフォルト逸脱時の管理方法の不在」（論点5での指摘）を解消しており、動的判定とBの契約的アプローチの良い統合になっている。

### 懸念: Phase 1→2の順序による導出バイアス

ただし、**Phase 1（リファレンスリスト照合）をPhase 2（動的導出）の前に置く構造には懸念**がある。

人間の認知にもLLMの推論にも「アンカリング効果」がある。Phase 1でリファレンスリストの項目を先に走査すると、Phase 2の動的導出がリストの項目に**引きずられる**リスクがある。例えば:

- Phase 1でA6-A10の品質属性を走査した後、Phase 2で「リストにない品質属性はないか？」と問うと、LLMは既に見た5項目を基準に差分を考える。これはリストがなければ自由に導出できた検討事項を見落とす方向に作用する。
- Phase 1で「スキップ」と判定した項目について、Phase 2で改めて検討する動機が弱くなる。Phase 1のスキップ判定が正しかったか、要件間交差の観点から再検証されない。

**対策案: Phase 1とPhase 2を並列実行し、Phase 3で統合する**

```
Phase 0: 入力分析
  ↓（並列に分岐）
Phase 1: リファレンスリスト照合     Phase 2: 要件駆動の動的導出
  ↓                                  ↓
Phase 3: マージ・差分分析・依存関係解決
  - Phase 1の結果とPhase 2の結果を照合
  - Phase 2で導出されたがPhase 1でスキップ判定だった項目を重点検証
  - Phase 2で導出されずPhase 1にのみある項目の必要性を再評価
  ↓
Phase 4: 対話的設計
  ↓
Phase 5: 多層検証
```

この並列構造により:
- 動的導出がリファレンスリストのアンカリングを受けない
- Phase 3の差分分析で「リストにあるが要件から導出されなかった項目」「要件から導出されたがリストにない項目」が明示的に可視化される
- 差分自体が検証材料になる（「リストの項目が導出されなかった理由」を分析することで、スキップ判定の妥当性が強化される）

## 2. ハイブリッド方式への懸念と改善提案

### 懸念1: リファレンスリストの「ガイドライン」としての位置づけの曖昧さ

メインエージェントはリファレンスリストを「固定のチェックリストではなく、導出のガイドラインとして機能する」と位置づけている。しかし、Phase 1で「各項目について検討要/スキップ（理由付き）を判定」し、Phase 5で「リファレンスリスト走査完了の確認」を行う以上、**実質的にはチェックリストとして運用される**。

「ガイドライン」と「チェックリスト」の実質的な違いは何か？ を明確にすべきである。

**改善提案**: リファレンスリストの役割を以下のように明確に定義する:

- **Phase 1での役割**: 「この項目を検討せよ」ではなく「この観点から要件を再読せよ」というプロンプト。項目ごとに「検討要/スキップ」を判定するのではなく、リスト全体を俯瞰して「この要件セットで重要な検討次元は何か」を判断する材料として使用する。
- **Phase 5での役割**: 「リストの全項目を走査したか」ではなく「リストの各項目が導出結果にカバーされているか、カバーされていない場合は意図的か」を検証する。

この定義であれば、リファレンスリストは「見落とし防止の参照資料」として機能し、検討のスコープを制約する「チェックリスト」にはならない。

### 懸念2: Phase 3（マージと依存関係解決）の複雑さ

Phase 1とPhase 2の結果を統合するPhase 3は、実質的に**2つの異なる視点から生成された検討事項リストの整合を取る**作業であり、非自明な判断を多数含む:

- 同じ関心事がPhase 1とPhase 2で異なる粒度・名称で表現されている場合の同定
- Phase 2で導出された新規項目のリファレンスリスト項目との依存関係の設定
- 横断的関心事のarch/process境界の統合判定

**改善提案**: Phase 3の具体的な統合手順を定義する。特に:
- Phase 1の項目とPhase 2の導出結果の**マッピングルール**（名寄せの基準）
- 競合する場合の**優先順位**（例: Phase 2の要件駆動導出を優先し、Phase 1のリファレンスリスト項目は補完的に扱う）
- 依存関係の統合方法（Phase 2の依存グラフにPhase 1の項目をどう組み込むか）

### 懸念3: 検証の二重構造が冗長にならないか

Phase 5の検証が「リスト走査完了の確認（B由来）」と「要件トレーサビリティ検証（A由来）」の二重構造になっている。これは網羅性を高めるが、**2つの検証が矛盾した結果を出す場合の扱い**が未定義である。

例えば:
- リスト走査: 「A9（性能設計方針）がスキップされている」→ 問題あり
- トレーサビリティ: 「全要件に設計決定が紐づいている。性能に関するNFRが存在しないためA9に対応する決定は不要」→ 問題なし

この場合、どちらの検証結果を優先するか？

**改善提案**: トレーサビリティ検証を上位（primary）、リスト走査を下位（secondary）とする。理由は、トレーサビリティ検証が**プロジェクト固有の要件に基づく検証**であり、リスト走査は**汎用的な参照に基づく検証**だから。リスト走査で「漏れ」が検出されても、トレーサビリティ検証で対応するNFRが存在しないことが確認されれば、真の漏れではない。逆に、トレーサビリティ検証で「紐づけなし」の要件が検出された場合は、リスト走査の結果に関わらず問題とすべきである。

## 3. リファレンスリストの粒度と項目数

### 推奨する粒度

リファレンスリストの項目は**「設計上の関心事カテゴリ」レベル**に留めるべきである。Bの提案のようにA1-A15の個別項目レベルまで詳細化すると、チェックリスト的な運用に引きずられる。

**適切な粒度の例**:
```
アーキテクチャ設計のリファレンスカテゴリ:
1. システム構造（境界、スタイル、コンポーネント分割）
2. データ（ストア、フロー、所有権、一貫性）
3. 通信・統合（プロトコル、同期/非同期、外部システム連携）
4. 品質属性（要件に応じて具体化: 性能、可用性、セキュリティ、スケーラビリティ等）
5. 横断的関心事（ログ、エラーハンドリング、構成管理、監視、テスト）
6. デプロイメント（環境、パイプライン、リリース戦略）
```

```
実装方針設計のリファレンスカテゴリ:
1. 開発基盤（言語、フレームワーク、ディレクトリ構成）
2. 実装パターン（arch_designの各決定に対する技術選定）
3. 横断的関心事の実装（arch_designからの引き継ぎ項目）
4. 開発プロセス（規約、バージョン管理、CI/CD、テスト）
```

### 推奨する項目数

- **arch_design: 6カテゴリ程度**（上記の例）
- **process_design: 4カテゴリ程度**（上記の例）

各カテゴリは内部にサブ項目を持つが、サブ項目レベルは固定せず、カテゴリの説明として「含まれうる検討事項の例示」を記載するに留める。

### 粒度設計の根拠

1. **アンカリングの最小化**: カテゴリが粗いほど、LLMが要件に応じて自由に具体化できる。「品質属性」というカテゴリと「A6: スケーラビリティ戦略、A7: 可用性・耐障害性設計、A8: セキュリティアーキテクチャ...」という列挙では、後者のほうがLLMの思考を制約する。
2. **保守コストの最小化**: カテゴリレベルは技術トレンドの変化に対して安定しており、頻繁な更新が不要。
3. **「二等市民」問題の解消**: カテゴリの粒度が粗ければ、動的導出で追加された項目も既存カテゴリに自然に統合できる。「モデルサービング設計」は「システム構造」や「デプロイメント」カテゴリの一部として扱える。
4. **チェックリスト化の防止**: 6カテゴリ程度であれば「全カテゴリを走査する」のは参照行為であり、15項目の「検討要/スキップ判定」のような事務的処理にはならない。

### 各カテゴリに含めるべき情報

各カテゴリの定義には以下を含める:
- **カテゴリの説明**: 1-2文でこのカテゴリが扱う範囲を記述
- **典型的な検討事項の例示**: 3-5個の具体例（ただし「これに限られない」ことを明記）
- **arch/processのデフォルト分界**: このカテゴリ内でarchとprocessの責務が分かれる場合の一般的な基準

サブ項目の網羅的列挙、スキップ条件、依存順序は含めない（これらは動的導出とPhase 3の統合で決定する）。
