# アプローチB（事前定義方式）からのハイブリッド案への意見

## 1. 事前定義の核心的利点が保たれているか

### 保たれている利点

**リファレンスリスト照合（Phase 1）が独立フェーズとして存在する点は高く評価する。** 事前定義方式の最も重要な利点は「一般的な設計次元の検討漏れを構造的に防ぐ」ことであり、Phase 1がこの安全網として機能する設計になっている。

**検証の二重構造（Phase 5）も有効。** 「リファレンスリスト走査完了の確認」と「要件トレーサビリティ検証」の組み合わせにより、事前定義方式の「閉じた空間の検証」とAの「要件起点の検証」が相補的に機能する。これは単独のどちらの方式よりも堅牢な検証となる。

**既知/未知の区別の明示性**も保たれている。Phase 1由来の項目とPhase 2由来の項目を区別することで、「何がリファレンスリストでカバーされ、何が動的に導出されたか」が追跡可能であり、リファレンスリストの改善フィードバックループが維持される。

### 損なわれている、または弱体化している利点

**依存関係の事前定義による順序保証が弱体化している。** Phase 3で「依存関係に基づいて検討順序を決定」とあるが、これはPhase 1のリファレンスリスト項目とPhase 2の動的導出項目を統合した後に動的に順序付けする設計であり、事前定義方式の「層構造による順序保証」が失われている。

事前定義方式の立場としては、**リファレンスリスト内の項目には事前定義された順序を持たせ、動的導出された項目のみを依存分析で挿入位置を決定する**のが望ましい。つまり、既知の項目間の順序は安定させつつ、未知の項目の位置だけを動的に決めるハイブリッドが適切である。

## 2. ハイブリッド方式への懸念と改善提案

### 懸念1: Phase 1とPhase 2の役割境界の曖昧さ

現在の提案では、Phase 1は「リファレンスリスト照合」、Phase 2は「要件駆動の動的導出」と分かれているが、Phase 2で「導出された検討事項をリファレンスリストの項目と照合」するとある。この照合の結果、リファレンスリスト内の項目が「Phase 1ではスキップ判定されたがPhase 2で必要と判定される」ケースが起こりうる。

**改善提案**: Phase 1のスキップ判定はPhase 2の終了まで暫定（tentative）とし、Phase 2の結果によってスキップ判定が覆る場合を明示的に許可する。ただし、覆った場合は「Phase 1でスキップ→Phase 2で復活（理由: ○○）」と記録し、追跡可能性を維持する。

### 懸念2: リファレンスリストの位置づけが「ガイドライン」に弱体化するリスク

メインエージェントの提案には「固定のチェックリストではなく、導出のガイドラインとして機能する」「各項目は『検討範囲のヒント』を含むが、実際の検討範囲は要件に応じてLLMが判断する」とある。

事前定義方式の立場から、これは**リファレンスリストの拘束力を過度に弱める**方向である。リファレンスリストの項目は「検討するか明示的にスキップするか」の二択を強制すべきであり、「ガイドライン」として参照する/しないをLLMの裁量に委ねるべきではない。

**改善提案**: リファレンスリストの各項目に対して、Phase 1で必ず「検討要/スキップ（理由付き）」のいずれかを記録することを必須とする。「参照しなかった」という状態を許容しない。これにより、リストの走査完了が検証可能な性質を維持できる。

### 懸念3: arch/process境界のシフト管理

「デフォルト分界マップ + arch_designからの逸脱記録」という設計は良いが、**逸脱が許容される範囲の定義が不足している**。制限なく境界をシフトできるなら、デフォルト分界マップの存在意義が薄れる。

**改善提案**: 分界マップの各項目に「シフト許容度」を定義する。

- **固定（FIXED）**: arch/processの分担は変更不可（例: デプロイメントアーキテクチャのarch側責務）
- **調整可能（ADJUSTABLE）**: 要件に応じてarch側の責務範囲を拡大/縮小できる（例: ログ戦略。規制要件がある場合にarch側を拡大）
- **委譲可能（DELEGABLE）**: 要件によってはarch側の検討を省略しprocess側に全面委譲できる（例: テスト戦略。テスト要件が軽微な場合）

これにより、境界シフトの判断も構造化され、LLMの恣意的な判断に対するガードレールとなる。

### 懸念4: Phase 2の導出スコープの制御

Phase 2の「要件間交差分析でリファレンスリスト外の検討事項を導出」は、Aの動的導出方式をそのまま適用する形だが、**導出範囲が無制限になるリスクがある**。LLMが過剰に検討事項を導出した場合、Phase 3のマージが複雑化し、プロセス全体のコストと時間が不安定になる。

**改善提案**: Phase 2の導出に上限を設ける。例えば「リファレンスリスト外の項目は最大5件まで」とし、それを超える場合はユーザーに優先順位付けを求める。これにより、動的導出の利点を取り込みつつ、プロセスの予測可能性を維持できる。

## 3. リファレンスリストの粒度と項目数

### 粒度について

事前定義方式の立場から、リファレンスリストの項目粒度は以下の原則に従うべきである。

**原則: 「スキップ判断が一意に行えるレベル」**

項目が粗すぎると、スキップ判断が曖昧になる（例: 「品質属性設計」は粗すぎて、性能は不要だがセキュリティは必要というケースに対応できない）。項目が細かすぎると、項目数が膨大になりスキップ判断のコストが増大する。

適切な粒度は**「1つの設計判断カテゴリが1つの項目に対応する」**レベルである。具体的には:

- 良い粒度: 「スケーラビリティ戦略」「認証・認可方式」「ログ集約方式」
- 粗すぎる: 「品質属性」「横断的関心事」
- 細かすぎる: 「水平スケーリング方式」「垂直スケーリング方式」「オートスケーリングポリシー」

### 項目数について

arch_designとprocess_designそれぞれについて:

**arch_design: 12-18項目を推奨**

提案Bの15項目（A1-A15）は妥当な範囲にある。議論を通じて以下の調整が望ましい:
- 第1層（基盤決定）: 5-6項目。提案Bの5項目が適切。
- 第2層（品質属性設計）: 5-8項目。提案Bの5項目はやや少なく、ISO 25010の主要カテゴリを参考に互換性・保守性を加えてもよい。ただし項目追加のトレードオフ（Aが指摘した粒度ジレンマ）を考慮し、8項目を上限とする。
- 第3層（横断的関心事）: 4-6項目。提案Bの5項目が適切。

**process_design: 12-16項目を推奨**

提案Bの16項目（P1-P16）は上限に近い。arch_designの出力に応じたスキップが適切に機能するなら、実際に検討する項目数は10前後になると想定され、過剰検討のリスクは低い。

### 層構造の維持を推奨

ハイブリッド方式においても、リファレンスリスト内の項目には**層構造（基盤→品質属性→横断的関心事）を維持すべき**である。Phase 3でのマージ時に、リファレンスリスト項目は層構造に従った順序を保ち、動的導出項目は適切な層に挿入する形が望ましい。これにより:

- 基盤的な決定（アーキテクチャスタイル、コンポーネント分割等）が必ず先に検討される安全性を維持
- 動的導出項目も文脈に応じた適切な位置で検討される柔軟性を確保

## 総括

ハイブリッド方式は事前定義方式の核心的利点の大部分を保持しつつ、動的導出の適応性を取り込んだ合理的な設計である。上記の改善提案は、事前定義方式が提供する**構造性・検証可能性・予測可能性**をハイブリッド方式でも確実に維持するためのガードレールであり、ハイブリッド方式の基本方向性には賛同する。

最も重要な点は、リファレンスリストを「任意参照のガイドライン」ではなく「全項目に対して判定必須のチェックポイント」として位置づけることである。これが崩れると、事前定義方式の検証可能性という核心的利点が失われる。
