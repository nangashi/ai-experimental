# improve_seq スキル — メカニズム詳細

## 1. 構造理解（Phase 1）

### 目的

レビューの前に対象を深く理解することで、表面的・部分最適な指摘を構造的に防ぐ。

### 構造理解が果たす役割

| 後続Phase | 構造理解の使われ方 |
|---|---|
| Phase 2 レビュー | 全体の目的・構造に照らした指摘を行う文脈として |
| Phase 3 トリアージ | 「この対象にとって本当に重要か」の判断基準として |
| Phase 4 改善適用 | 修正が全体の一貫性を壊さないことの確認基準として |

### 構造理解の品質基準（QG-1）

以下のいずれかに該当する場合、構造理解が不十分と判断し再実行を指示する:

- 目的が「〜を行う」のような1行の記述で、背景・動機が欠落している
- フローの記述がない（要素の列挙のみ）
- 制約が1つも特定されていない
- 現状の達成度に「不足していること」の記載がない
- レビュー観点が対象の種類に対して汎用的すぎる（例: コードに対して「可読性」のみ）

---

## 2. レビュー観点の設計

### 固定観点

| ID | 名称 | 何を見るか |
|----|------|-----------|
| R0 | 要件充足の妥当性 | 対象が達成しようとしている目的に対して、最もシンプルで妥当な手段を取っているか。過剰な抽象化、不要な機能、要件との乖離がないか |

### 動的観点

構造理解の結果から 2-3 個を導出する。

導出時の制約:
- 固定観点（R0）と重複しないこと
- 「一般的に良い」ではなく「この対象の目的・構造から見て重要」な観点であること
- 各観点は独立していること（同じ問題を別角度で見るのではなく、異なる問題領域を見る）
- 観点ごとに「なぜこの対象にとって重要か」の根拠を明記すること

---

## 3. Finding（指摘）の構造

```
### Finding-{観点ID}-{連番}: {タイトル}

- **問題点**: 何が問題か（事実ベースで記述。構造理解のどの部分と矛盾するか明示）
- **修正方針**: 具体的にどう修正するか（抽象的な「改善する」ではなく、変更内容を明記）
- **対応しないリスク**: この問題を放置した場合の影響（頻度・深刻度を含む）
- **見込める改善**: 修正により得られる具体的な効果
- **トレードオフ**: 修正により失われるもの、増える複雑性（なければ「なし」と明記）
- **核心の問い**: この指摘の判断に必要な、最も本質的な問い
```

Finding の品質基準:
- 問題点は対象の具体的な箇所を参照していること
- 問題点は構造理解（target-analysis.md）の内容と紐づけられていること
- 修正方針は実行可能なレベルで具体的であること
- 「より良い」「改善される」等の曖昧な表現を含まないこと

---

## 4. トリアージの判定ロジック

### 4.1 有効性チェック

各指摘に対し3つのチェックを行う:

**複雑化チェック**
- 修正方針を適用した場合、対象の構造的複雑さが増すか
- 判定基準: 行数の純増、新しい抽象レイヤーの追加、新しい依存関係の追加 → 複雑化あり
- 例外: 複雑化しても、それ以上のリスク低減が見込める場合は別途判断

**効果チェック**
- 「見込める改善」が具体的・検証可能か
- 判定基準: 改善効果が「〜が防げる」「〜が不要になる」等の具体的帰結で表現されているか
- NG例: 「可読性が向上する」「より良くなる」 → 効果不明確

**トレードオフチェック**
- 「トレードオフ」が「なし」でないか
- 「なし」以外の場合、人間の価値判断が必要と見なす

### 4.2 分類マトリクス

| 複雑化 | 効果 | トレードオフ | → 分類 |
|--------|------|-------------|--------|
| なし | 明確 | なし | **auto-apply** |
| なし | 明確 | あり | **human-decision** |
| あり | 明確 | なし | **human-decision** |
| あり | 明確 | あり | **human-decision** |
| - | 不明確 | - | **skip** |

### 4.3 トリアージの限界と対策

トリアージの判定者もAIであるため、「効果チェック」をAI判定の自信度に依存させない。判定基準は構造的な条件（行数増減、抽象レイヤー追加の有無、「なし」という文字列の有無）に限定する。

## 5. 人間判断の記録と再利用

### decisions.md の構造

```markdown
# 判断記録

## Decision-1: {タイトル}
- **判断**: 対応しない
- **理由**: パフォーマンスより可読性を優先する方針
- **論点キーワード**: パフォーマンス, 可読性, トレードオフ

## Decision-2: ...
```

### 照合ロジック

Phase 3 Step 2 で、各 Finding に対して:

1. Finding のタイトル・問題点・核心の問いから論点を抽出
2. decisions.md の各 Decision の論点キーワードと照合
3. 同一論点と判断される場合、前回の判断に従う（skip）

照合は完全一致ではなくセマンティックな類似性で行う（親エージェントが判定）。

### 判断の蓄積価値

decisions.md はセッションをまたいで保持される（同一 key で再実行した場合）。これにより:
- ユーザーの価値観・優先度が蓄積される
- 繰り返しの確認が不要になる
- 「このプロジェクトでは X より Y を優先する」といった方針が暗黙的に学習される

---

## 6. 過剰改善の防止策

反復ループを排除したことで過剰改善のリスクは大幅に低減されるが、単一パスでも以下の防止策を設ける:

1. **構造理解による文脈化**: レビュアーが「この対象にとって何が重要か」を把握した上で指摘するため、些末な指摘が減る
2. **Finding の品質基準**: 曖昧な改善は生成段階で排除（構造理解との紐づけを要求）
3. **効果チェック**: トリアージで具体性のない指摘を skip
4. **既判断スキップ**: decisions.md により同じ論点の再提起を防止

---

## 7. コンテキスト節約

- 親エージェント（SKILL.md 実行者）は対象ファイルの全文をコンテキストに保持しない
- 構造理解はファイル（target-analysis.md）経由で後続サブエージェントに渡す
- レビュー結果は findings ファイル経由でトリアージに渡す
- 改善適用はサブエージェントが直接対象ファイルを読み書きする
