# improve_seq スキル — 設計上の検討事項

## Q1: 対象ファイルのスコープ

**現在の想定**: 単一ファイルを対象とする

**代替案**: ディレクトリ（複数ファイル）も対象にできるようにする

**トレードオフ**:
- 単一ファイル: シンプル。構造理解のフォーカスが明確。差分追跡が容易
- 複数ファイル: 実用的だが構造理解の範囲が広がり、分析の質が下がるリスク

**素案の方針**: 単一ファイルで開始。有効性が確認できた後に拡張を検討する

---

## Q2: 構造理解の深さの調整

**現在の想定**: 全対象に対して同一の深さで構造理解を行う

**検討事項**:
- 小規模なファイル（数十行のコード等）に対して、フロー分析・制約特定・達成度分析のフルセットは過剰な可能性
- 一方で、構造理解の深さを対象サイズに応じて動的に変えると、スキルの複雑性が増す

**素案の方針**: フルセットで固定。過剰でもコストは構造理解の1サブエージェント分であり、レビュー品質への影響の方が大きい

---

## Q3: レビュー実行時のモデル選択

**現在の想定**: general-purpose サブエージェント（親と同一モデル）

**検討事項**:
- 生成と評価のモデル分離を厳密に行うなら、レビューに異なるモデルを使うべき
- Claude Code 環境ではモデル切り替えが限定的（sonnet/opus/haiku）
- haiku はレビューの深さが不足する可能性

**素案の方針**: まずは同一モデルで開始。「初めて見る人の視点で」のフレーミングで自己選好バイアスに対抗する。有効性の検証結果を見てモデル分離を検討

---

## Q4: 既存スキル（agent_audit等）との関係

**agent_audit との違い**:
- agent_audit: エージェント定義の静的分析に特化。2次元分析（基準有効性 + スコープ整合性）
- improve_seq: 対象を限定しない汎用改善。構造理解 → レビュー → 改善適用

**共存の方針**:
- agent_audit はエージェント定義に特化した深い分析ツールとして維持
- improve_seq は汎用改善ツールとして、agent_audit がカバーしない対象に使用

---

## Q5: セッション再開時の振る舞い

**現在の想定**: 同一 key で再実行した場合、decisions.md のみ引き継いで最初から実行

**検討事項**:
- 前回の target-analysis.md を再利用するか
- 対象ファイルが前回から変更されていた場合の扱い

**素案の方針**:
- decisions.md のみ引き継ぐ
- target-analysis.md は毎回再生成（対象が変更されている可能性があるため）
- 前回のファイルは上書きされる（履歴管理は git に委ねる）

---

## Q6: auto-apply の安全性

**現在の想定**: 「複雑化なし + 効果明確 + トレードオフなし」の指摘は人間確認なしで自動適用

**検討事項**:
- トリアージの判定者がAIであるため、auto-apply の条件を満たしていると誤判定するリスク
- auto-apply を完全に廃止して全件 human-decision にすると、手間削減の目的が達成できない

**素案の方針**: auto-apply を維持するが、Phase 5 のサマリで auto-apply された全件を一覧表示する。事後確認の機会を提供する
