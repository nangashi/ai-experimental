# improve_seq スキル — ワークフロー

## 全体フロー

```
Phase 0 (初期化)
  → Phase 1 (構造理解)
  → Phase 2 (レビュー実行)
  → Phase 3 (トリアージ)
  → Phase 4 (改善適用)
  → Phase 5 (完了サマリ)
```

単一パス。反復しない。深い理解に基づく1回のレビューで有効な改善を適用する。

---

## Phase 0: 初期化

**目的**: ワーキングディレクトリの準備と前回状態の復元

1. 引数パース（key, target_path）
2. `{work_dir}` = `.skill_output/improve_seq/${key}/` を作成
3. `{work_dir}/decisions.md` が存在すれば前回の判断記録として読み込む
4. 対象ファイルを読み込み、`{work_dir}/target-snapshot-before.md` として保存（差分追跡用）

---

## Phase 1: 構造理解

**目的**: 対象の構造・目的・制約を深く分析し、レビューの基盤を作る

**サブエージェント委譲** — `{skill_dir}/templates/analyze-target.md`

入力:
- 対象ファイルの内容

処理:
1. **対象の種類判定**: コード / スキル定義 / 設計文書 / プロンプト / その他
2. **目的の特定**: この対象は何を達成するために存在するか
3. **構造の分析**:
   - フロー（処理の流れ、データの流れ、依存関係）
   - 構成要素とその役割
   - 各要素間の関係性
4. **制約の特定**: この対象が守るべき制約（技術的制約、設計方針、互換性など）
5. **現状の達成度**: 目的に対してどこまで達成できていて、何が不足しているか
6. **レビュー観点の導出**: 構造理解から、この対象にとって重要なレビュー観点を導出する

出力: `{work_dir}/target-analysis.md`

```markdown
# 対象分析: {ファイル名}

## 種類
{コード / スキル定義 / 設計文書 / ...}

## 目的
{この対象が達成しようとしていること}

## 構造
### フロー
{処理・データの流れ}

### 構成要素
{主要な構成要素とその役割}

### 要素間の関係
{依存関係、呼び出し関係、データの受け渡し}

## 制約
{守るべき制約の一覧}

## 現状の達成度
- 達成できていること: ...
- 不足していること: ...

## レビュー観点
### R1: {観点名}
- 何を見るか: ...
- なぜこの対象にとって重要か: ...

### R2: ...
```

### レビュー観点の導出ルール

- 固定観点として「要件充足の妥当性」（対象が求めている要件に対して最もシンプルで妥当な解決方法を実現しているか）を常に含める
- 動的観点は構造理解の結果から 2-3 個導出する
- 「一般的に良い」ではなく「この対象の目的・構造から見て重要」な観点であること
- 各観点は独立していること

**品質ゲート QG-1**: target-analysis.md の内容を親が確認する。構造理解が表面的（目的が曖昧、フローの記述がない、制約の特定がない）な場合、再実行を指示する。

---

## Phase 2: レビュー実行

**目的**: 構造理解を基盤として対象をレビューし、改善候補を抽出する

**サブエージェント委譲（並列）** — `{skill_dir}/templates/review.md`

各観点ごとに1サブエージェントを起動。

各サブエージェントへの入力:
- 対象ファイルの内容
- `{work_dir}/target-analysis.md`（構造理解の全文）
- 担当する観点の定義
- `{work_dir}/decisions.md`（過去の判断記録。存在する場合）

レビュー指示:
- 「構造理解を踏まえ、この対象を初めて見る批判的なレビュアーとして、代替案を含めて評価してください」
- 「全体の目的・構造に照らして部分最適にならない指摘を行ってください」
- 過去の decisions.md に記載された判断済みの論点は指摘しない

各指摘（Finding）の出力フォーマット:

```markdown
### Finding-{観点ID}-{連番}: {タイトル}

- **問題点**: 何が問題か（事実ベースで記述。構造理解のどの部分と矛盾するか明示）
- **修正方針**: 具体的にどう修正するか（変更内容を明記）
- **対応しないリスク**: 修正しない場合に何が起こるか（頻度・深刻度を含む）
- **見込める改善**: 修正により得られる具体的な効果
- **トレードオフ**: 修正により失われるもの、増える複雑性（なければ「なし」と明記）
- **核心の問い**: この指摘について判断すべき最も重要な問い
```

出力: `{work_dir}/findings-{perspective_id}.md`

---

## Phase 3: トリアージ

**目的**: 各指摘の有効性を判定し、自動適用 / 人間判断 / スキップに分類する

親エージェントが直接処理する（人間への提示を含むため）。

### Step 1: 重複・類似の統合

複数の観点から同一の指摘がある場合、統合する。

### Step 2: 決定ログとの照合

`{work_dir}/decisions.md` を参照し、過去に判断済みの指摘（同一または類似の論点）をスキップする。

### Step 3: 有効性チェック

各指摘に対し以下を判定する:

| チェック項目 | 判定基準 |
|---|---|
| 複雑化チェック | 修正により対象の行数・抽象度・依存関係が増加するか |
| 効果チェック | 改善が具体的・測定可能か。曖昧な「より良い」レベルではないか |
| トレードオフチェック | 修正方針の「トレードオフ」が「なし」以外か |

### Step 4: 分類

| 分類 | 条件 | アクション |
|---|---|---|
| **auto-apply** | 複雑化なし + 効果明確 + トレードオフなし | 自動で Phase 4 へ |
| **human-decision** | トレードオフあり OR 複雑化の懸念あり | 人間に提示して判断を仰ぐ |
| **skip** | 効果不明確 OR 既に判断済み | スキップ |

### Step 5: 人間への提示（human-decision の場合）

per-item approval に従い、1件ずつ提示する:

```
## Finding: {タイトル}

**問題点**: ...
**修正方針**: ...
**対応しないリスク**: ...
**見込める改善**: ...
**トレードオフ**: ...

### 判断すべき核心の問い
{核心の問い}

→ 対応する / 対応しない / 修正方針を変更する
```

人間の判断結果を `{work_dir}/decisions.md` に記録:

```markdown
## Decision-{N}: {タイトル}
- **判断**: 対応する / 対応しない / 修正方針を変更（内容: ...）
- **理由**: {人間が述べた理由、あれば}
- **論点キーワード**: {類似判断の照合用キーワード}
```

### Step 6: トリアージ結果の出力

出力: `{work_dir}/triage.md`

---

## Phase 4: 改善適用

**目的**: auto-apply + 人間が承認した指摘を対象に適用する

**サブエージェント委譲** — `{skill_dir}/templates/apply-improvements.md`

入力:
- 対象ファイル
- `{work_dir}/target-analysis.md`（構造理解）
- `{work_dir}/triage.md`（適用すべき指摘リスト）

処理:
- 各指摘の修正方針に従い、対象ファイルを修正
- 構造理解を参照し、修正が全体の一貫性を壊さないことを確認
- 修正は対象ファイルに直接適用（Edit ツール使用）

出力:
- 対象ファイルの更新
- `{work_dir}/changes.md`（適用した変更のサマリ）
- `{work_dir}/target-snapshot-after.md`（適用後のスナップショット）

---

## Phase 5: 完了サマリ

**目的**: 改善の全体像を記録する

親エージェントが直接処理する。

出力: `{work_dir}/summary.md`

```markdown
# 改善サマリ: ${key}

## 対象
- パス: {target_path}
- 種類: {target_type}

## 構造理解の要約
- 目的: {1行}
- 主要な制約: {箇条書き}

## レビュー観点
- {各観点の一覧}

## トリアージ結果
| 指摘数 | auto-apply | human-decision | skip |
|---|---|---|---|
| ... | ... | ... | ... |

## 適用した改善一覧
- {各改善の1行サマリ}

## 人間の判断記録
- {各判断の1行サマリ}

## スキップした指摘
- {各スキップの1行サマリと理由}
```
