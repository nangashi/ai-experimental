## 各項目の詳細

### 1. ログ戦略

- **何を決めるか**: アプリケーションログの形式（構造化/テキスト）、ログレベルの使い分け、収集・保管先、保持期間。

- **選択肢**:
  | 要素 | 選択肢 | 備考 |
  |------|--------|------|
  | ログ形式 | 構造化ログ(JSON) / テキストログ | 構造化ログが現在の標準 |
  | ロガー | Pino (Node.js最速) / Winston (多機能) / Bunyan / console.log(非推奨) | パフォーマンス要件で選択 |
  | 収集基盤 | OpenTelemetry Collector / Fluentd / Fluent Bit / CloudWatch Logs | ベンダー中立性で選択 |
  | 保管・分析 | Better Stack Logs / Grafana Loki / Elasticsearch / CloudWatch Logs Insights | コストとクエリ性能で選択 |
  | ログレベル | FATAL / ERROR / WARN / INFO / DEBUG / TRACE | 環境別に出力レベルを変更 |

- **選定基準**:
  - 構造化ログ(JSON形式)は必須。検索・集計・アラート設定が容易
  - ログとトレースの相関（Trace ID/Span IDの付与）ができるか
  - 保管コスト（ログ量に応じたストレージコスト、ティアリング対応）
  - ローカル開発時の可読性（構造化ログの人間可読な出力モード）

- **トレードオフ・注意点**:
  - 構造化ログは人間には読みにくい → 開発環境ではpretty print、本番ではJSON出力に切り替え
  - ログに個人情報(PII)を含めない。マスキング/リダクション機能の設計が必要
  - DEBUGレベルのログは本番では無効化（パフォーマンスへの影響とストレージコスト）
  - リクエストIDを全ログに付与し、リクエスト横断でのトレーサビリティを確保
  - ログ保持期間を定義（例: HOT 7日 / WARM 30日 / COLD 90日 / 削除）

- **2025-2026年のトレンド**:
  - OpenTelemetryがログ・メトリクス・トレースの統一標準としてデファクトに。2026年にはログのstable化が進む見込み
  - ログとトレースの自動相関（OTel SDKが自動的にTrace ID/Span IDをログに付与）
  - インテリジェントデータティアリング: ログの重要度に応じて保管先・保持期間を自動分類しコスト最適化
  - AI支援のログ分析: 異常パターンの自動検出、根本原因の推定
  - 構造化ログ + セマンティック規約(OpenTelemetry Semantic Conventions)による標準化

### 2. エラートラッキング

- **何を決めるか**: ランタイムエラー（未処理例外、APIエラー、フロントエンドエラー）の検知・集約・通知方法。

- **選択肢**:
  | サービス | 特徴 | 無料枠 | 適用場面 |
  |---------|------|--------|---------|
  | Sentry | 業界標準。エラー集約、スタックトレース、ソースマップ、パフォーマンス監視 | 5Kイベント/月 | 一般的なWeb/モバイルアプリ |
  | Better Stack | ログ+稼働監視+インシデント管理の統合。SQL-firstアプローチ | 無料枠あり | ログ中心の監視体制 |
  | GlitchTip | Sentry互換のOSS。セルフホスト可能 | セルフホスト(無料) | コスト重視、Sentry互換が欲しい場合 |
  | Rollbar | AI支援のデバッグ、柔軟なグルーピング | 5Kイベント/月 | エラーの根本原因分析を重視 |
  | LogRocket | セッションリプレイ + エラートラッキング | 1Kセッション/月 | フロントエンドのUX問題とエラーの相関分析 |

- **選定基準**:
  - フロントエンド/バックエンド両方のエラーを一元管理できるか
  - ソースマップ対応（ミニファイされたコードのスタックトレース復元）
  - アラート連携（Slack, Email等への即時通知）
  - エラーのグルーピング精度（同一原因のエラーをまとめる能力）
  - コスト予測の容易さ（イベントベース課金のバースト対策）

- **トレードオフ・注意点**:
  - Sentryはイベントベース課金のため、バグ多発時にコストが急増するリスク
  - Sentryの2024年のAIトレーニングデータポリシー変更で一部ユーザーが代替を検討
  - セルフホスト(GlitchTip)は運用コストが発生するが、データ主権を確保できる
  - エラーの優先度付け（Critical/Warning/Info）と通知ルールの設計が重要

- **2025-2026年のトレンド**:
  - AI支援のエラー分析・根本原因推定が各サービスで標準機能化
  - OpenTelemetryとの統合が進み、エラーとトレース/メトリクスの相関分析が容易に
  - Better Stack等の統合型プラットフォーム（ログ + エラー + 稼働監視 + インシデント管理）の台頭
  - セッションリプレイとエラートラッキングの統合（ユーザーが何をしていたかの文脈でエラーを理解）

### 3. APM (アプリケーションパフォーマンスモニタリング)

- **何を決めるか**: アプリケーションのパフォーマンス計測方法、ボトルネック検知、分散トレーシングの導入方針。

- **選択肢**:
  | ツール | 特徴 | コスト | 適用場面 |
  |--------|------|-------|---------|
  | Sentry Performance | エラートラッキングと統合。トランザクション追跡 | 無料枠内 | Sentry利用時に追加コスト0で導入 |
  | New Relic | フルスタック可観測性。AI支援の分析 | 100GB/月無料 | 包括的な監視が必要な場合 |
  | Datadog | APM+インフラ+ログの統合。広範なインテグレーション | 有料（高コスト） | エンタープライズ、複数サービス |
  | Grafana + Tempo | OSSスタック。分散トレーシング | 無料(セルフホスト) / Grafana Cloud無料枠 | コスト重視、OSSエコシステム |
  | Vercel Analytics | Next.js/Vercel最適化。Web Vitals自動計測 | 無料枠あり | Vercelデプロイ時 |

  | 計測対象 | メトリクス例 | 目安 |
  |---------|-----------|------|
  | レスポンスタイム | P50, P95, P99 | P95 < 200ms (API), P95 < 1s (ページ) |
  | エラー率 | 5xxレスポンス割合 | < 0.1% |
  | スループット | リクエスト数/秒 | アプリケーション依存 |
  | Web Vitals | LCP, INP, CLS | LCP < 2.5s, INP < 200ms, CLS < 0.1 |
  | DB クエリ | スロークエリ検出、N+1検出 | クエリ実行時間 < 100ms |

- **選定基準**:
  - エラートラッキングツールとの統合（Sentry利用なら追加でSentry Performanceが最小コスト）
  - 分散トレーシングの要否（マイクロサービス → 必須、モノリス → あると便利）
  - Web Vitals(Core Web Vitals)の計測要否（フロントエンドパフォーマンス重視なら必須）
  - コスト（フルスタックAPMは高コスト。必要な機能に絞る）

- **トレードオフ・注意点**:
  - フルスタックAPMは高コスト。小規模プロジェクトではSentry Performance + Vercel Analytics等の組み合わせで十分な場合が多い
  - APMのオーバーヘッド（計測によるパフォーマンス低下）。サンプリングレートの調整が必要
  - メトリクスの「ゴールデンシグナル」（レイテンシ、トラフィック、エラー、飽和度）を基本にする
  - ダッシュボードは作り過ぎない。見る人がいないダッシュボードは無駄

- **2025-2026年のトレンド**:
  - OpenTelemetryによるベンダー非依存のテレメトリデータ収集が標準化
  - AI/ML支援の異常検知（ベースラインからの逸脱を自動検出）
  - eBPFベースの低オーバーヘッド計測（アプリケーションコード変更不要の自動計測）
  - 「3 pillars of observability」(ログ・メトリクス・トレース)の統合プラットフォームへの集約
  - コスト意識の高まり: テレメトリデータの取捨選択とティアリングが重要課題に

### 4. ヘルスチェック・稼働監視

- **何を決めるか**: サービスの死活監視方法、ヘルスチェックエンドポイントの設計、外形監視の導入。

- **選択肢**:
  | 種類 | 内容 | ツール例 |
  |------|------|---------|
  | ヘルスチェックエンドポイント | `/health`, `/ready` でアプリの状態を公開 | 自前実装（フレームワーク組み込み） |
  | 外形監視 (Synthetic Monitoring) | 外部からHTTPリクエストを送信し応答を確認 | Better Stack Uptime / UptimeRobot / Checkly |
  | SSL証明書監視 | SSL証明書の有効期限を監視 | Better Stack / UptimeRobot |
  | ステータスページ | サービスの稼働状況を公開 | Instatus / Better Stack / Cachet |

  | エンドポイント | チェック内容 | 用途 |
  |-------------|-----------|------|
  | `/health` | アプリプロセスの生存確認 | ロードバランサー/オーケストレーター |
  | `/health/ready` | DB接続、外部依存の疎通確認 | デプロイ時のReadinessチェック |
  | `/health/live` | プロセスが応答可能か | Kubernetes Liveness Probe |

- **選定基準**:
  - 監視頻度（1分 / 5分 / 30秒）
  - 監視拠点の地理的分散（複数リージョンからの監視）
  - アラート連携（Slack / Email / SMS / 電話）
  - ステータスページの要否（ユーザー向けの稼働状況公開）

- **トレードオフ・注意点**:
  - ヘルスチェックにDB接続チェックを含めると、DB障害時にロードバランサーが全インスタンスをunhealthyにするリスク → `/health`(shallow)と`/health/ready`(deep)を分離
  - ヘルスチェックエンドポイントは認証不要にするが、内部情報の漏洩に注意（バージョン番号等は含めない、又は認証必須にする）
  - 外形監視は「ユーザー目線」の監視。内部メトリクスでは検知できない問題（DNS障害、CDN障害等）を補完
  - 監視間隔が短すぎるとコスト増、長すぎると検知遅延

- **2025-2026年のトレンド**:
  - Synthetic Monitoring（外形監視）+Real User Monitoring（RUM）の組み合わせが標準構成に
  - Checkly等のMonitoring as Code（テストコードで監視を定義し、CI/CDパイプラインに統合）
  - ステータスページの自動化（障害検知 → 自動的にステータスページ更新 → 復旧で自動解除）
  - マルチリージョン監視によるCDN/エッジ障害の検知

### 5. アラート設定方針

- **何を決めるか**: 何をアラートにするか、通知先、重大度の定義、アラート疲れの防止策。

- **選択肢**:
  | 要素 | 選択肢 | 備考 |
  |------|--------|------|
  | 通知チャネル | Slack / Email / SMS / 電話 / PagerDuty / OpsGenie | 重大度に応じて使い分け |
  | アラートソース | エラートラッキング / 稼働監視 / メトリクス閾値 / ログベース | 多層で設定 |
  | 重大度レベル | P1(緊急) / P2(高) / P3(中) / P4(低) | 対応時間目標と連動 |

  | 重大度 | 定義 | 通知先 | 対応目標 |
  |--------|------|--------|---------|
  | P1 (緊急) | サービス全体停止、データ損失リスク | SMS + Slack + Email | 即時対応 |
  | P2 (高) | 主要機能の障害、パフォーマンス重度劣化 | Slack + Email | 1時間以内 |
  | P3 (中) | 一部機能の不具合、警告レベルの異常 | Slack | 翌営業日 |
  | P4 (低) | 情報レベル、改善提案 | 週次レポート | 次スプリント |

- **選定基準**:
  - 「アクション可能な」アラートのみ発報する（情報提供のみのアラートは避ける）
  - ノイズの少なさ（誤検知が多いとアラート疲れ → 真のアラートを見逃す）
  - エスカレーションルール（一定時間未対応の場合、上位に自動エスカレーション）
  - 個人開発/小規模チーム向けにはSlack + Email通知で十分

- **トレードオフ・注意点**:
  - アラート疲れ(Alert Fatigue)が最大のリスク。頻繁に発報するが対応不要なアラートは即時修正か削除
  - 閾値ベースのアラートは適切な値の設定が難しい → 最初は緩めに設定し、運用しながら調整
  - SLOベースのアラート（エラーバジェットのバーンレートが閾値を超えたら発報）がより効果的
  - 個人開発では深夜アラートの対応ポリシーを事前に決めておく

- **2025-2026年のトレンド**:
  - SLOベースのアラート: エラーバジェットのバーンレートに基づくアラートが主流に（Google SREの実践が普及）
  - AIOps: AI/MLによる異常検知、ノイズ抑制、根本原因の自動推定
  - 「known/unknown/next update」構造のインシデント通知テンプレートが推奨
  - アラートのコード管理（Terraform/Pulumi等でアラートルールをIaCとして管理）

### 6. バックアップ・災害復旧

- **何を決めるか**: データバックアップの方式、頻度、復旧目標(RPO/RTO)、テスト方針。

- **選択肢**:
  | 要素 | 選択肢 | 備考 |
  |------|--------|------|
  | バックアップ方式 | マネージド自動バックアップ / 手動スナップショット / 論理バックアップ(pg_dump等) | コストと復旧速度のトレードオフ |
  | 復旧方式 | PITR (Point-in-Time Recovery) / スナップショット復元 / リストア | 復旧粒度の違い |
  | 保管先 | 同一リージョン / クロスリージョン / 別クラウド | 災害の規模に応じて選択 |

  | 目標 | 定義 | 個人開発の目安 | 商用サービスの目安 |
  |------|------|-------------|----------------|
  | RPO (Recovery Point Objective) | 最大データ損失許容時間 | 24時間 | 1時間以下 |
  | RTO (Recovery Time Objective) | 最大復旧所要時間 | 数時間 | 1時間以下 |

- **選定基準**:
  - 使用DBのマネージドバックアップ機能（Supabase / PlanetScale / Neon等は自動バックアップ付き）
  - RPO/RTOの要件（個人開発なら緩め、商用なら厳格に）
  - バックアップ復元テストの実施頻度
  - ファイルストレージ（アップロード画像等）のバックアップも忘れない

- **トレードオフ・注意点**:
  - 「バックアップを取っている」と「復旧できる」は別。定期的な復元テストが必須
  - マネージドDBのバックアップに依存する場合、ベンダーロックインのリスク。論理バックアップ(pg_dump等)も併用
  - クロスリージョンバックアップはコストが高い。リスクとコストのバランスを検討
  - バックアップデータの暗号化（保管時・転送時）も必要

- **2025-2026年のトレンド**:
  - マネージドDB(Supabase, PlanetScale, Neon等)がデフォルトでPITR対応。設定不要で自動バックアップ
  - Infrastructure as Code(IaC)によるインフラ全体の再構築可能性。バックアップ対象はデータのみに集中
  - 災害復旧のゲームデイ（定期的な復旧訓練）の実施が推奨されているが、小規模チームでは四半期に1回のテストが現実的
  - イミュータブルバックアップ（ランサムウェア対策として変更・削除不可のバックアップ）の需要増

### 7. インシデント対応フロー

- **何を決めるか**: 障害発生時の対応手順、役割分担、ポストモーテムの実施方針。個人開発/小規模チーム向けのシンプルなフローを設計する。

- **選択肢**:
  | 要素 | 選択肢 | 備考 |
  |------|--------|------|
  | ランブック管理 | Markdown + Gitリポジトリ / Notion / Confluence | バージョン管理と検索性 |
  | ステータスページ | Instatus / Better Stack / Cachet(OSS) / GitHub Pages | ユーザーへの状況公開 |
  | ポストモーテム | テンプレートベース / 5 Whys / タイムライン分析 | 再発防止に繋げる |

  | フェーズ | 内容 | 個人開発での実践 |
  |---------|------|----------------|
  | 1. 検知 | アラート受信、影響範囲の初期評価 | Slackアラートの確認 |
  | 2. トリアージ | 重大度の判定、対応優先度の決定 | P1-P4の重大度判定 |
  | 3. 対応 | 応急処置（ロールバック、スケール等） | 直前バージョンへのロールバック |
  | 4. 復旧 | 根本原因の修正、復旧確認 | 修正コードのデプロイ |
  | 5. 振り返り | ポストモーテム、ランブック更新 | 簡易ポストモーテム記録 |

- **選定基準**:
  - チーム規模に応じたシンプルさ（個人開発 → 簡易フロー、チーム → 役割分担あり）
  - ランブックの検索性・アクセス性（障害時にすぐ見つけられるか）
  - ポストモーテムの「blame-free（非難しない）」文化
  - 発生頻度の高いインシデントから優先的にランブックを整備

- **トレードオフ・注意点**:
  - 過度に詳細なフローは障害時に読まれない。重要なのは「最初の3ステップ」
  - ランブックはGitリポジトリで管理しPRベースで更新（変更履歴の追跡）
  - ポストモーテムの最大の価値は「ランブックの改善PR」に繋げること
  - 個人開発でも最低限のインシデント記録（日時、影響範囲、原因、対応内容）は残す

- **2025-2026年のトレンド**:
  - ランブックのMarkdown + Gitリポジトリ管理が標準化。PRベースの更新で監査証跡を確保
  - NIST/DORA/SEC規制への対応としてインシデント対応のドキュメント化が義務化の方向
  - 「5 A's」(Actionable, Accessible, Accurate, Authoritative, Adaptable)なランブック設計が推奨
  - ポストモーテム → ランブック改善PRのサイクルを四半期ごとに回す

### 8. メンテナンスウィンドウの方針

- **何を決めるか**: 計画メンテナンスの実施方針、ダウンタイムの許容度、ユーザーへの通知方法。

- **選択肢**:
  | 方針 | 内容 | 適用場面 |
  |------|------|---------|
  | ゼロダウンタイムデプロイ | ブルーグリーン/ローリングデプロイで無停止更新 | SaaS、24時間稼働が必要なサービス |
  | 定期メンテナンスウィンドウ | 週次/月次の決まった時間帯にメンテナンス | トラフィックの少ない深夜帯に実施 |
  | オンデマンドメンテナンス | 必要時にアナウンス → メンテナンス実施 | 不定期な大規模変更 |
  | メンテナンスモード | アプリ側でメンテナンス画面を表示 | DBマイグレーション等のダウンタイム必要時 |

- **選定基準**:
  - サービスのダウンタイム許容度（SLOとの整合）
  - DBマイグレーションの方式（オンラインDDL対応か）
  - ユーザーへの事前通知リードタイム
  - デプロイ頻度（高頻度 → ゼロダウンタイム必須）

- **トレードオフ・注意点**:
  - ゼロダウンタイムデプロイは実装コストが高い（DBマイグレーションの後方互換性確保が必要）
  - マイグレーション時のダウンタイムは避けられない場合がある（大規模なスキーマ変更等）
  - メンテナンスモードの実装はシンプルに（環境変数やフィーチャーフラグで制御）
  - ユーザーへの通知は最低48時間前が理想

- **2025-2026年のトレンド**:
  - ゼロダウンタイムデプロイが標準化。PaaS(Vercel, Fly.io等)がデフォルトで対応
  - DBマイグレーションのゼロダウンタイム対応ツール（gh-ost, pt-online-schema-change, PlanetScaleのブランチ機能等）の成熟
  - フィーチャーフラグによる段階的リリースがメンテナンスウィンドウの代替手段に
  - GitOps + ArgoCD等による宣言的デプロイで、ロールバックが容易に

### 9. コスト監視

- **何を決めるか**: クラウドリソース、API利用料、SaaSツールの利用コストの把握・最適化方法。

- **選択肢**:
  | 要素 | 選択肢 | 備考 |
  |------|--------|------|
  | クラウドコスト | AWS Cost Explorer / GCP Billing / Azure Cost Management | クラウドネイティブツール |
  | マルチクラウド/IaC | Infracost (IaCのコスト見積もり) / CloudHealth | PRレベルでコスト影響を確認 |
  | API利用料 | 利用量ダッシュボード / 予算アラート | OpenAI / Stripe等のAPI |
  | SaaSツール | 月次棚卸し / スプレッドシート管理 | 小規模なら手動管理でも十分 |

  | 監視対象 | チェック内容 | 頻度 |
  |---------|-----------|------|
  | クラウドリソース | 未使用インスタンス、過剰プロビジョニング | 週次 |
  | API利用料 | 月間上限に対する消費ペース | 日次アラート |
  | ストレージ | 不要データの蓄積、ログ保持期間 | 月次 |
  | 全体予算 | 月間予算に対する実績 | 日次アラート |

- **選定基準**:
  - 予算アラート設定の容易さ（閾値超過で即時通知）
  - コスト配分の粒度（プロジェクト別、環境別、リソース別）
  - コスト最適化の推奨機能（リザーブドインスタンス推奨等）
  - 個人開発ではクラウドの無料枠超過検知が最優先

- **トレードオフ・注意点**:
  - 「クラウド支出の30-35%は無駄」というのが業界データ。定期的な棚卸しが重要
  - 個人開発では「月額上限アラート」の設定が必須（予期せぬ高額請求の防止）
  - 無料枠の期限切れに注意（AWS Free Tierの12ヶ月制限等）
  - API利用料は特にLLM APIで急増する可能性がある。利用量の上限設定を必ず行う

- **2025-2026年のトレンド**:
  - FinOpsの普及（エンジニアリング + ファイナンスの協業によるクラウドコスト最適化）
  - AI/LLM APIコストの管理が新しい課題として浮上。トークン使用量の監視・制限が必要
  - Infracost等のIaCコスト見積もりツール（PRマージ前にコスト影響を確認）
  - ユニットエコノミクス（ユーザーあたりコスト、トランザクションあたりコスト）での管理が推奨
  - AI駆動のコスト最適化（自動的なリソースサイジング提案、最大30%のコスト削減実績）

### 10. SLO/SLI設定

- **何を決めるか**: サービスの信頼性目標(SLO)、その計測指標(SLI)、エラーバジェットの管理方針。

- **選択肢**:
  | SLI (計測指標) | 定義 | 計測方法 |
  |---------------|------|---------|
  | 可用性 | 成功したリクエストの割合 | (成功リクエスト数 / 総リクエスト数) * 100 |
  | レイテンシ | リクエストの応答時間 | P50, P95, P99 のレスポンスタイム |
  | エラー率 | エラー(5xx)レスポンスの割合 | (5xxレスポンス数 / 総リクエスト数) * 100 |
  | スループット | 単位時間あたりの処理能力 | リクエスト数/秒 |

  | SLO (目標) | 個人開発の目安 | 商用サービスの目安 | 許容ダウンタイム/月 |
  |-----------|-------------|----------------|-----------------|
  | 99.0% | 適切 | 低め | 約7.3時間 |
  | 99.5% | やや高め | 一般的 | 約3.6時間 |
  | 99.9% | 不要 | 高め | 約43分 |
  | 99.95% | 不要 | 高い | 約22分 |

- **選定基準**:
  - ユーザージャーニーの重要度に基づいて設定（「ログインできるか」「データを見れるか」「重要アクションを完了できるか」の2-3指標に集中）
  - 現在の実績値をベースラインとして設定し、段階的に改善
  - インフラの能力に見合ったSLO（99.99%を目標にするなら冗長構成が必須）
  - エラーバジェットの消費ルール（バジェット消費が速い場合は機能リリースを停止する等）

- **トレードオフ・注意点**:
  - CPUやメモリ使用率はSLIにしない。ユーザーが体感する指標（ページロード時間、トランザクション成功率）を選ぶ
  - SLOを高くし過ぎると開発速度が犠牲に（リリースごとにSLO違反のリスク）
  - エラーバジェット = 1 - SLO。99.95%のSLOなら月に0.05%分の不具合が「許容」される
  - 個人開発で99.9%以上のSLOは過剰。インフラコストと運用負荷が不釣り合い

- **2025-2026年のトレンド**:
  - SLOベースの運用がGoogle SRE本の実践として広く普及
  - エラーバジェットをビジネスサイクルに連動（アーキテクチャ刷新期は緩め、繁忙期は厳格に）
  - ユーザージャーニーベースのSLI設定（技術メトリクスではなくユーザー行動の成功率で計測）
  - LLMゲートウェイ、ベクトルストア等の新しい依存関係が信頼性のサーフェスエリアを拡大
  - SLOのコード管理（SLOの定義をIaCとしてリポジトリで管理）

### 11. デプロイモニタリング

- **何を決めるか**: デプロイ後の異常を早期検知する仕組み、ロールバックの判断基準と実行方法。

- **選択肢**:
  | デプロイ戦略 | 特徴 | リスク | 適用場面 |
  |------------|------|-------|---------|
  | ローリングデプロイ | 順次インスタンスを更新。ダウンタイムなし | 新旧バージョンが一時的に共存 | 標準的なWebアプリ |
  | ブルーグリーンデプロイ | 新環境を構築し、一括切り替え | インフラコスト2倍（一時的） | 確実なロールバックが必要な場合 |
  | カナリアデプロイ | 少数ユーザーに先行リリース。段階的に拡大 | 実装が複雑 | リスクの高い変更 |
  | フィーチャーフラグ | コードはデプロイ済み、フラグで機能ON/OFF | フラグ管理の複雑化 | A/Bテスト、段階的リリース |

  | 監視項目 | 閾値例 | ロールバック判断 |
  |---------|--------|----------------|
  | エラー率 | デプロイ前の2倍以上 | 5分間持続で自動ロールバック |
  | レイテンシ(P95) | デプロイ前の1.5倍以上 | 10分間持続でアラート |
  | ヘルスチェック失敗 | 3回連続失敗 | 即座にロールバック |
  | ビジネスメトリクス | コンバージョン率の有意な低下 | 手動判断 |

- **選定基準**:
  - デプロイ頻度（高頻度 → 自動化・カナリア推奨）
  - ロールバックの即時性要件（「ロールバックは1アクションで完了」が理想）
  - インフラコスト（ブルーグリーンは2倍のリソース）
  - チームのデプロイ経験・成熟度

- **トレードオフ・注意点**:
  - カナリアデプロイは効果的だが実装が複雑。PaaSの機能やサービスメッシュが必要な場合が多い
  - ロールバック手順は事前にテストしておく（本番で初めてロールバックを試すのは危険）
  - 「ロールバックが1スイッチ/1コマンドで完了する」状態を目標にする
  - DBマイグレーションを含むデプロイのロールバックは特に注意（マイグレーションの後方互換性）
  - デプロイ後の「安定確認期間」(bake time)を設ける（最低15-30分）

- **2025-2026年のトレンド**:
  - PaaS(Vercel, Fly.io等)のデプロイプレビュー・自動ロールバック機能が成熟
  - Argo Rollouts / Flagger等のKubernetes向けプログレッシブデリバリーツールの普及
  - 自動プロモーションゲート: パフォーマンスメトリクス、エラー率が基準を満たさないとトラフィック拡大を自動ブロック
  - デプロイとオブザーバビリティの統合（デプロイイベントをメトリクスダッシュボードにマーカーとして表示）
  - フィーチャーフラグ + カナリアデプロイの組み合わせが最も安全なリリース戦略として認識
  - 対数的トラフィック増加（1% → 10% → 50% → 100%）が推奨パターン
