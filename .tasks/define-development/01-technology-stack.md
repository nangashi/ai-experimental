# 技術スタック選定ガイド

Webアプリケーション新規開発における技術スタック選定で決めるべき事項を網羅的に整理したドキュメント。各項目について選択肢、選定基準、トレードオフ、2025-2026年のトレンドを記載する。

## 決定事項一覧（サマリテーブル）

| # | 決定事項 | 主な選択肢 | 選定の主軸 |
|---|---------|-----------|-----------|
| 1 | プログラミング言語 | TypeScript, JavaScript, Go, Rust, Python | チーム習熟度、型安全性、エコシステム |

## 各項目の詳細

### 1. プログラミング言語（フロントエンド・バックエンド）

- **何を決めるか**: フロントエンドとバックエンドそれぞれで使用する言語。同一言語で統一するか、用途別に分けるか。
- **選択肢**:
  | 言語 | フロントエンド | バックエンド | 特徴 |
  |------|:---:|:---:|------|
  | TypeScript | o | o | 型安全性、JS互換、最大エコシステム |
  | JavaScript | o | o | 学習コスト最小、型なし |
  | Go | x | o | 高並行性、低レイテンシ、シンプルな言語仕様 |
  | Rust | x | o | 最高パフォーマンス、メモリ安全性、学習曲線急 |
  | Python | x | o | ML/AI連携、豊富なライブラリ、動的型付け |
- **選定基準**:
  - チームの既存スキルセット（習熟度が性能差より重要）
  - フロント・バック間でコード共有が必要か（バリデーション、型定義等）
  - 型安全性の要求レベル
  - エコシステムの充実度（ライブラリ、ツール、採用市場）
- **トレードオフ・注意点**:
  - TypeScriptでフロント・バック統一すると、コード共有（型定義、バリデーションスキーマ等）の恩恵が大きいが、バックエンドの並行処理性能はGo/Rustに劣る
  - Go/Rustをバックエンドに採用すると性能は向上するが、フロントとの型共有に追加の仕組み（OpenAPI生成等）が必要
  - Pythonはバックエンド単体では問題ないが、フロントとの統一性は得られない
- **2025-2026年のトレンド**:
  - TypeScriptがフルスタック開発のデファクト標準として定着。フロント・バック統一の主流言語
  - GoはAPI/マイクロサービスのバックエンドとして安定した人気
  - RustはWasm経由でフロントエンドにも進出しつつあるが、Web開発での採用はまだ限定的

---

### 2. フロントエンドフレームワーク

- **何を決めるか**: UIコンポーネントの構築基盤となるフレームワーク/ライブラリ。
- **選択肢**:
  | フレームワーク | 仮想DOM | バンドルサイズ | エコシステム | 学習曲線 |
  |--------------|:---:|------|------|------|
  | React | あり | 中 | 最大 | 中 |
  | Vue | あり | 小 | 大 | 低 |
  | Svelte | なし（コンパイル） | 最小 | 中 | 低 |
  | Angular | あり | 大 | 大 | 高 |
  | Solid | なし（リアクティブ） | 最小 | 小 | 中 |
  | Preact | あり | 最小 | 中（React互換） | 低 |
- **選定基準**:
  - エコシステムの規模（UIライブラリ、サードパーティ統合の充実度）
  - パフォーマンス要件（バンドルサイズ、ランタイム速度）
  - チームの既存経験
  - 長期メンテナンスの見通し（コミュニティ規模、企業バッキング）
  - メタフレームワーク（SSR/SSG対応）の選択肢
- **トレードオフ・注意点**:
  - Reactはエコシステム最大だが、状態管理・ルーティング等の選択肢が多く「決定疲れ」が生じやすい
  - Vueは学習曲線が緩やかだが、大規模エンタープライズでの採用事例がReactほど多くない
  - Svelteはバンドルサイズ最小でDX良好だが、エコシステムがReact/Vueに比べ小さい
  - Angularはフルスタック的な機能を内蔵するが、学習コストが高くバンドルも大きい
- **2025-2026年のトレンド**:
  - React 19のServer Components/Server Actionsが本格普及。サーバーファーストの開発モデルへ移行
  - Svelte 5のRunes（リアクティビティシステム刷新）により、大規模開発での採用が増加
  - Solidが高パフォーマンス用途で注目度上昇中
  - Signal ベースのリアクティビティが各フレームワークで共通トレンド（Angular Signals、Vue Reactivity、Svelte Runes）

---

### 3. バックエンドフレームワーク

- **何を決めるか**: API/サーバーサイドロジックの構築基盤。フルスタックフレームワークを採用しない場合に特に重要。
- **選択肢**:
  | フレームワーク | 言語 | パフォーマンス | 構造化 | エッジ対応 | 特徴 |
  |--------------|------|:---:|:---:|:---:|------|
  | Express | Node.js | 低 | 低 | x | 最大エコシステム、ミドルウェア豊富 |
  | Fastify | Node.js | 高 | 中 | x | Express互換、スキーマバリデーション |
  | Hono | マルチランタイム | 高 | 中 | o | エッジファースト、Web Standards準拠 |
  | NestJS | Node.js | 中 | 高 | x | エンタープライズ向け、DI内蔵、Angular風 |
  | Elysia | Bun | 最高 | 中 | o | Bun最適化、型安全、End-to-end type safety |
  | Nitro | マルチランタイム | 高 | 中 | o | Nuxt内部エンジン、単独利用可 |
- **選定基準**:
  - パフォーマンス要件（リクエスト/秒、レイテンシ）
  - コードの構造化レベル（チーム規模が大きいほど高い構造化が有利）
  - エッジデプロイの必要性
  - ミドルウェア/プラグインのエコシステム
  - 型安全性（入出力のバリデーション、APIスキーマ）
- **トレードオフ・注意点**:
  - Expressは最も普及しているが、パフォーマンスが他より劣り非同期エラーハンドリングに課題がある
  - NestJSはエンタープライズグレードの構造を提供するが、学習コストが高くボイラープレートが多い
  - Honoはエッジ対応で軽量だが、Expressほどのミドルウェアエコシステムはない
  - ElysiaはBunに依存するため、Bunのエコシステム成熟度がリスク
- **2025-2026年のトレンド**:
  - Honoがエッジコンピューティング対応の軽量フレームワークとして急成長
  - Web Standards API準拠のフレームワークが増加（Hono, Elysia等）
  - Express後継としてのFastifyの地位が安定
  - NestJSがエンタープライズTypeScriptバックエンドの標準的選択肢として定着

---

### 4. フルスタックフレームワークの採否

- **何を決めるか**: フロントエンドとバックエンドを統合するメタフレームワークを採用するかどうか。採用する場合、どのフレームワークを使うか。
- **選択肢**:
  | フレームワーク | ベース | SSR | SSG | ISR | RSC | ストリーミング |
  |--------------|------|:---:|:---:|:---:|:---:|:---:|
  | Next.js | React | o | o | o | o | o |
  | Nuxt | Vue | o | o | o | - | o |
  | SvelteKit | Svelte | o | o | - | - | o |
  | Remix | React | o | - | - | - | o |
  | Astro | マルチ | o | o | - | - | o |
  | **採用しない** | - | - | - | - | - | - |
- **選定基準**:
  - SSR/SSG/ISRの必要性（SEO要件、初期表示速度）
  - デプロイプラットフォームとの親和性（Vercel, Cloudflare, AWS等）
  - APIルートの統合（フロントと同一リポジトリでAPIを管理するか）
  - フロントエンドフレームワークの選択との整合性
  - フロント・バックの独立デプロイが必要か
- **トレードオフ・注意点**:
  - フルスタックFW採用のメリット: 開発速度向上、ルーティング/データフェッチの統合、デプロイ簡素化
  - フルスタックFW採用のデメリット: フレームワークへの強い依存、フロント・バックの独立スケーリング困難、ベンダーロックイン（特にVercelとNext.js）
  - 採用しない場合: フロント・バック完全分離で独立デプロイ可能だが、認証・セッション管理等の連携コスト増
  - Next.jsはVercelとの結合度が高く、セルフホストでの一部機能制限がある
  - Remixはweb標準に忠実でシンプルだが、SSGやISRは非対応
  - Astroはコンテンツ中心サイトに最適だが、高度なインタラクティブアプリには不向き
- **2025-2026年のトレンド**:
  - Next.js 15がReact Server Componentsを本格統合、App Routerが標準に
  - SvelteKitがSvelte 5対応で注目度上昇。バンドルサイズ削減で差別化
  - Remixの「web標準ファースト」アプローチが支持を拡大（Shopify Admin等の大規模採用事例）
  - サーバーファースト開発がフレームワーク全体のトレンド（Server Components, Server Actions）
  - AI SDK統合（AI生成コンテンツのストリーミングレスポンス対応）が各フレームワークで進行中

### 5. データベース

- **何を決めるか**: データ永続化層の種別（RDB vs NoSQL）と具体的なプロダクト。ホスティング形態（マネージド vs セルフホスト）。
- **選択肢**:
  | DB | 種別 | スケーラビリティ | 運用コスト | 特徴 |
  |-----|------|:---:|------|------|
  | PostgreSQL | RDB | 高 | 中 | 最も多機能なOSS-RDB、JSON対応、拡張機能豊富 |
  | MySQL | RDB | 高 | 中 | 広く普及、レプリケーション成熟 |
  | SQLite | RDB（組込） | 低 | 最低 | ファイルベース、サーバーレス、運用ゼロ |
  | MongoDB | ドキュメントDB | 高 | 中〜高 | スキーマレス、水平スケーリング |
  | DynamoDB | Key-Value/ドキュメント | 最高 | 従量課金 | AWS依存、無制限スケール |
  | PlanetScale | MySQL互換 | 高 | 従量課金 | ブランチ機能、サーバーレス |
  | Turso | SQLite互換 | 中 | 低 | エッジ対応SQLite、libSQL |
  | Neon | PostgreSQL | 高 | 従量課金 | サーバーレスPostgreSQL、ブランチ機能 |
- **選定基準**:
  - データ構造の複雑さ（リレーション多 → RDB、非構造化 → NoSQL）
  - スケーラビリティ要件（読み取り/書き込みの比率、データ量の成長見込み）
  - 運用コスト・運用負荷（マネージドサービスの無料枠）
  - エッジデプロイとの親和性
  - バックアップ・リカバリの要件
  - ORM/クエリビルダーの対応状況
- **トレードオフ・注意点**:
  - PostgreSQLは最も汎用的だが、セルフホストの運用負荷が高い → Supabase/Neon等のマネージドサービスで軽減
  - SQLiteは運用コストゼロだが、同時書き込みに制限がある。個人開発・小規模には最適
  - MongoDBはスキーマレスで初期開発が速いが、データ整合性の担保が開発者責任になる
  - サーバーレスDB（Neon, Turso, PlanetScale）はコールドスタートのレイテンシに注意
  - RDB選択後にNoSQLへの移行（またはその逆）はコストが非常に高い
- **2025-2026年のトレンド**:
  - PostgreSQLがWebアプリのデフォルトDBとしての地位を強固に。Supabase等BaaSの基盤としても普及
  - SQLiteのエッジ対応（Turso/libSQL）により、エッジコンピューティングでのRDB利用が現実的に
  - サーバーレスDB（Neon, PlanetScale）がスタートアップ/個人開発で人気。従量課金で初期コストゼロ
  - MongoDBの採用は減少傾向。PostgreSQLのJSON機能がドキュメントDB的用途をカバー

### 6. ORM / クエリビルダー

- **何を決めるか**: データベースアクセス層の抽象化レベルと使用するライブラリ。
- **選択肢**:
  | ライブラリ | 種別 | バンドルサイズ | 型安全性 | SQL透明性 | マイグレーション |
  |-----------|------|:---:|:---:|:---:|:---:|
  | Prisma | ORM | 大（バイナリ依存） | 高（コード生成） | 低 | 自動生成（優秀） |
  | Drizzle | ORM/QB | 最小（~7KB） | 高（型推論） | 高 | 手動/自動 |
  | Kysely | クエリビルダー | 小 | 高（型推論） | 最高 | 手動 |
  | TypeORM | ORM | 大 | 中 | 低 | デコレータベース |
  | Knex.js | クエリビルダー | 中 | 低 | 高 | 手動 |
  | 生SQL | - | なし | なし | 最高 | 完全手動 |
- **選定基準**:
  - DX重視 vs パフォーマンス重視
  - バンドルサイズの制約（サーバーレス/エッジでは重要）
  - マイグレーション管理の自動化レベル
  - SQLの学習・理解を前提とするか
  - 型チェックのアプローチ（ビルド時コード生成 vs ランタイム型推論）
- **トレードオフ・注意点**:
  - Prismaはマイグレーション管理が最も優秀だが、バイナリ依存でバンドルサイズが大きく、サーバーレスのコールドスタートに影響
  - Drizzleは軽量でサーバーレス最適だが、Prismaほどのマイグレーション自動化はない
  - KyselyはSQL透明性最高だが、ORM的な機能（リレーション自動解決等）は提供しない
  - TypeORMはメンテナンス停滞気味で新規採用は非推奨
  - Prismaの型チェックはビルド時生成のため高速、Drizzleはランタイム推論のため大規模スキーマで型チェックが遅くなる場合がある
- **2025-2026年のトレンド**:
  - DrizzleがPrismaに次ぐ人気を獲得。サーバーレス/エッジ環境での採用が急増
  - 「Prisma for DX, Drizzle for performance」という使い分けが定着
  - Kyselyがクエリビルダーとして高評価を維持（SQL熟練者向け）
  - TypeORMからPrisma/Drizzleへの移行が進行中

### 7. ランタイム

- **何を決めるか**: JavaScript/TypeScriptの実行環境。
- **選択肢**:
  | ランタイム | エンジン | npm互換性 | パフォーマンス | エコシステム成熟度 | TypeScript |
  |-----------|---------|:---:|:---:|:---:|:---:|
  | Node.js | V8 | 100% | ベースライン | 最高 | 要トランスパイル |
  | Deno 2 | V8 | ~95% | 高 | 中 | ネイティブ |
  | Bun | JavaScriptCore | ~98% | 最高 | 低 | ネイティブ |
- **選定基準**:
  - 既存npmパッケージへの依存度
  - パフォーマンス要件（HTTP throughput、コールドスタート）
  - セキュリティ要件（Denoの権限モデル）
  - 本番環境での実績・安定性
  - TypeScriptのネイティブサポート要否
- **トレードオフ・注意点**:
  - Node.jsは最も安定・成熟しているが、TypeScriptサポートがネイティブでない（ただしNode.js 22以降で実験的サポート開始）
  - Bunは最高パフォーマンス（Node.jsの2-3.5倍のHTTPスループット）だが、本番環境の実績がまだ限定的
  - Deno 2はnpm互換性が大幅改善されたが、一部パッケージで互換性問題が残る
  - BunとDenoはランタイム + パッケージマネージャ + バンドラーを一体提供
  - ホスティングプラットフォームのランタイムサポート状況を確認する必要がある
- **2025-2026年のトレンド**:
  - Node.jsが依然として本番環境のデファクト標準。Node.js 22でESM完全対応
  - Bunがグリーンフィールドプロジェクトでの採用増。特にパフォーマンス重視の場面
  - Deno 2がnpm互換性改善で実用的な選択肢に。セキュリティ重視のプロジェクトで採用
  - コールドスタート: Bun 8-15ms, Deno 40-60ms, Node.js 60-120ms

### 8. パッケージマネージャ

- **何を決めるか**: 依存関係の管理・インストールツール。
- **選択肢**:
  | PM | インストール速度 | ディスク効率 | モノレポ | ロックファイル | 特徴 |
  |----|:---:|:---:|:---:|------|------|
  | npm | 遅い | 低 | Workspaces | package-lock.json | Node.js同梱、追加インストール不要 |
  | pnpm | 速い | 最高（~70%削減） | 優秀 | pnpm-lock.yaml | ハードリンク、厳格な依存解決 |
  | Yarn (Berry) | 中 | 高（PnP時） | 優秀 | yarn.lock | PnP、ゼロインストール |
  | bun | 最速 | 高 | あり | bun.lockb | Bunランタイム統合 |
- **選定基準**:
  - CI/CDでのインストール速度
  - ディスク使用量（開発マシンのストレージ、CI環境）
  - モノレポの必要性
  - チームの既存ツールチェーンとの整合性
  - ランタイム選択との一貫性（Bun使用時はbun PMが自然）
- **トレードオフ・注意点**:
  - npmは追加セットアップ不要だが、速度・ディスク効率で他に劣る
  - pnpmは厳格な依存解決により「ファントム依存」を防止するが、一部パッケージで互換性問題が起きることがある
  - Yarn Berry（PnP）はnode_modulesを排除するが、一部ツールとの互換性に注意が必要
  - bunは最速だが、Bunランタイムを使わない場合は導入意義が薄い
  - ロックファイル形式が異なるため、PM変更時は移行コストが発生
- **2025-2026年のトレンド**:
  - pnpmがnpmに次ぐシェアを獲得。OSS/エンタープライズ双方で採用拡大
  - npmは依然として最も広く使われているが、速度面での改善は限定的
  - Yarn Berryは採用が頭打ち。Classic Yarnからの移行が進まず
  - bunが新規プロジェクトでの採用を伸ばしている（Bunランタイム採用時のセット利用）
  - ベンチマーク（Next.jsアプリ ~1.1kパッケージ）: bun 8.6s, pnpm 31.9s, npm 57.4s, yarn 138s

### 9. APIプロトコル

- **何を決めるか**: フロントエンド-バックエンド間、またはサービス間の通信プロトコル。
- **選択肢**:
  | プロトコル | 型安全性 | HTTPキャッシュ | 学習コスト | クライアント多様性 | 適用範囲 |
  |-----------|:---:|:---:|:---:|:---:|------|
  | REST | なし（OpenAPIで補完） | 優秀 | 低 | 最高 | 汎用、公開API |
  | GraphQL | スキーマ定義 | 困難 | 高 | 高 | 複雑なデータ要件 |
  | tRPC | 最高（TypeScript推論） | なし | 低 | TS限定 | TypeScript monorepo |
  | gRPC | Protocol Buffers | なし | 高 | 高（多言語） | マイクロサービス間通信 |
- **選定基準**:
  - クライアントの多様性（Webのみ vs モバイル・サードパーティ含む）
  - フロント・バックが同一言語（TypeScript）か
  - データフェッチの複雑さ（ネストしたリレーション、部分取得の必要性）
  - キャッシュ戦略の重要度
  - 公開APIとしての利用予定
- **トレードオフ・注意点**:
  - RESTはHTTPキャッシュと自然に統合できるが、over-fetching/under-fetchingの問題がある
  - GraphQLはクライアント主導のデータ取得が可能だが、HTTPキャッシュが困難（POST+単一エンドポイント）で、N+1問題への対処が必要
  - tRPCはTypeScript monorepoで最高のDXを提供するが、TypeScript以外のクライアントには使えない
  - gRPCはブラウザ直接通信に制約がある（gRPC-Webが必要）
  - ブラウザ-サーバー間の通信では、REST/GraphQL/tRPCのパフォーマンス差はほぼ無視できる
- **2025-2026年のトレンド**:
  - tRPCがTypeScriptフルスタック開発（特にT3 Stack）で急成長
  - RESTが依然として最も広く採用。OpenAPI 3.1 + コード生成で型安全性を補完
  - GraphQLの採用は安定期に入り、新規採用よりも既存システムでの利用が主
  - Server Actionsの普及により、フルスタックフレームワーク内でのAPI層の必要性が低下するケースも

### 10. 認証ライブラリ/サービス

- **何を決めるか**: ユーザー認証・認可の実装方式。自前実装 vs マネージドサービス。
- **選択肢**:
  | サービス | コスト（月額） | 無料MAU | データ所有 | 実装時間 | UI提供 |
  |---------|------|------|:---:|------|:---:|
  | Auth.js (NextAuth) | 無料（OSS） | 無制限 | 完全 | 2-5日 | なし |
  | Clerk | $25〜 | 10,000 | サービス側 | 1-3日 | あり |
  | Supabase Auth | $0-25 | 50,000 | 完全（Postgres） | 2-4日 | 一部 |
  | Firebase Auth | 従量課金 | 50,000 | Google管理 | 1-3日 | あり（FirebaseUI） |
  | Auth0 | $23〜 | 7,500 | サービス側 | 2-4日 | あり |
  | Lucia Auth | 無料（OSS） | 無制限 | 完全 | 3-5日 | なし |
  | 自前実装 | 無料 | 無制限 | 完全 | 5-14日 | なし |
- **選定基準**:
  - コスト（MAU増加時のスケーリングコスト）
  - データ所有権・プライバシー要件
  - 実装速度（Time-to-market）
  - ソーシャルログイン対応の必要性
  - フレームワークとの統合度（Next.js + Auth.js等）
  - コンプライアンス要件（SOC 2, HIPAA, GDPR等）
- **トレードオフ・注意点**:
  - Auth.js（旧NextAuth）は無料・データ所有だが、UI構築とカスタマイズが必要
  - Clerkは最高のDXだが、MAU課金でスケール時のコスト増とベンダーロックイン
  - Supabase AuthはDBと統合されRLS連携が強力だが、Supabaseプラットフォームへの依存
  - 自前実装はセキュリティリスクが高く、OWASP対応等の専門知識が必要
  - 認証サービスの移行は非常にコストが高い（ユーザーデータ、セッション管理の再構築）
- **2025-2026年のトレンド**:
  - Auth.js v5（旧NextAuth v5）がフレームワーク非依存化、Next.js以外でも利用可能に
  - Clerkが「認証のStripe」として急成長。プリビルトUIの品質で差別化
  - PasskeyベースのパスワードレスSS認証が各サービスで標準対応
  - Supabase AuthのRLS（Row Level Security）統合が、認証+認可の統合ソリューションとして評価

### 11. BaaS（Backend-as-a-Service）の採否

- **何を決めるか**: バックエンド機能（DB、認証、ストレージ、リアルタイム等）をBaaSに委譲するかどうか。する場合、どのサービスを使うか。
- **選択肢**:
  | BaaS | DB種別 | 認証 | ストレージ | リアルタイム | 無料枠 | OSS |
  |------|------|:---:|:---:|:---:|------|:---:|
  | Supabase | PostgreSQL | o | o | o | 500MB DB, 1GB Storage | o |
  | Firebase | NoSQL (Firestore) | o | o | o | 1GB DB, 5GB Storage | x |
  | Appwrite | MariaDB | o | o | o | 無制限（セルフホスト） | o |
  | PocketBase | SQLite | o | o | o | 無制限（セルフホスト） | o |
  | Convex | 独自 | o | o | o | 1GB DB | x |
  | **BaaS不使用** | - | - | - | - | - | - |
- **選定基準**:
  - 開発速度 vs カスタマイズ性
  - ベンダーロックインの許容度
  - 無料枠の充実度（個人開発/スタートアップで特に重要）
  - データ所有権・ポータビリティ
  - セルフホストの可否
  - カスタムバックエンドロジックの必要性
- **トレードオフ・注意点**:
  - BaaS採用のメリット: バックエンド開発工数の大幅削減、インフラ運用不要、認証/ストレージ/リアルタイム等の統合
  - BaaS採用のデメリット: ベンダーロックイン、カスタムロジックの制限、料金体系への依存
  - Supabaseはオープンソースで移行可能性が高いが、クラウド版固有機能の利用には注意
  - Firebaseは読み取り/書き込み課金のため、予測困難なコスト構造
  - PocketBaseは単一バイナリで最もシンプルだが、水平スケーリング不可
  - BaaS不使用の場合: 完全な制御が可能だが、認証・ストレージ等の基盤構築に時間がかかる
- **2025-2026年のトレンド**:
  - SupabaseがFirebaseの最有力代替として急成長。PostgreSQL基盤のオープンソースアプローチが支持
  - PocketBaseが個人開発・小規模プロジェクトで人気急上昇。単一バイナリのシンプルさが評価
  - Appwriteがセルフホスト可能なフルスタックBaaSとして成熟
  - 「BaaS + Edge Functions」の組み合わせで、カスタムロジックの制約を緩和するパターンが普及
  - Firebaseの新規採用は減少傾向。NoSQL（Firestore）の制約とコスト予測困難性が要因

### 技術選定時の一般原則

1. **チームスキル優先**: 理論上の性能差よりも、チームの既存スキルと生産性を優先する
2. **退出戦略の確保**: ベンダーロックインの度合いを評価し、移行パスを事前に確認する
3. **段階的採用**: 新しい技術は小さな範囲で検証してから全面採用する
4. **YAGNI原則**: 将来の要件を過度に予測した技術選定を避ける。現時点の要件に適した選択をする
5. **エコシステムの健全性**: GitHub Stars数よりも、メンテナンス頻度、Issue対応速度、コントリビュータ数を重視する
