## 各項目の詳細

### 1. Gitワークフロー

- **何を決めるか**: プロジェクト全体のブランチ運用モデル。どのブランチをいつ作り、どのタイミングでマージし、どのブランチからデプロイするかのルール。
- **選択肢**:
  - **GitHub Flow**: `main` ブランチ + 短命なfeatureブランチ。PRベースでマージ。シンプルで小〜中規模チーム向き。
  - **Trunk-based Development (TBD)**: 全員が `main` に直接コミット、またはごく短命（1日以内）のブランチを使用。Feature Flagで未完成機能を制御。CI/CDとの親和性が最も高い。
  - **Git Flow**: `main` / `develop` / `feature` / `release` / `hotfix` の5種ブランチ。複数バージョンの並行保守が必要なプロダクト向き。
- **選定基準**:
  - チーム規模（個人〜小規模ならGitHub FlowまたはTBD、大規模ならTBD推奨）
  - デプロイ頻度（高頻度→TBD、低頻度→Git Flow）
  - 複数バージョンの並行保守の要否（必要→Git Flow）
  - CI/CDの成熟度（高い→TBD、まだ構築中→GitHub Flow）
- **トレードオフ・注意点**:
  - Git Flowは現代のWebアプリ開発ではオーバーヘッドが大きく、継続的デリバリーとの相性が悪い
  - TBDはFeature Flag基盤やCI/CDの整備が前提。インフラが未整備な段階ではGitHub Flowから始めるのが現実的
  - GitHub Flowは単一リリーストラックのプロダクトに最適だが、ステージング環境との連携パターンを別途設計する必要がある
- **2025-2026年のトレンド**:
  - TBDへの移行が加速。DORA メトリクス（リードタイム、デプロイ頻度、変更失敗率、MTTR）とTBDの強い相関が広く認知されている
  - 「2026年に勝つチームは最も洗練されたブランチ戦略を持つチームではなく、シンプルにする勇気を持つチーム」という認識が主流
  - Merge Queue の普及により、TBDの実用性が向上（PRをキューに入れ、CIがスタックした変更を統合テストしてからマージ）
  - Git Flowからの移行パス: まずGitHub Flowに移行し、CI/CDが成熟したらTBDへ段階的に移行するアプローチが推奨

### 2. ブランチ戦略（命名規則・保護ルール・マージ戦略）

- **何を決めるか**: ブランチの命名規則、保護ブランチの設定、PRマージ時の戦略（squash / rebase / merge commit）。
- **選択肢**:
  - **命名規則**:
    - `feature/` / `fix/` / `chore/` / `docs/` + 説明的な名前（例: `feature/add-payment-gateway`）
    - Issue番号を含める形式（例: `feature/123-add-payment-gateway`）
    - Conventional Commitsに合わせた prefix（`feat/`, `fix/`, `refactor/`）
  - **保護ルール**:
    - `main` ブランチへの直接pushを禁止
    - PRマージ前にCIパス必須
    - 承認レビュー必須（1人以上 / 2人以上）
    - force push禁止
    - ブランチの最新化（up-to-date）必須
  - **マージ戦略**:
    - **Squash and Merge**: featureブランチの全コミットを1コミットに圧縮。履歴がクリーン。
    - **Rebase and Merge**: 各コミットを個別にベースブランチに追加。線形履歴を維持。
    - **Merge Commit**: マージコミットを作成。ブランチの統合ポイントが明示的。
- **選定基準**:
  - コミット履歴の粒度の要求（クリーンさ重視→squash、詳細追跡→merge commit）
  - `git bisect` の使用頻度（使う→merge commitまたはrebase）
  - チームのGitスキルレベル（初〜中級→squash、上級→rebase）
  - revert のしやすさ（squash: 1コミットrevertで完了、merge commit: マージ全体をrevert可能）
- **トレードオフ・注意点**:
  - Squash: WIPコミットが消えるためPR単位での変更追跡になる。大きなPRをsquashすると1コミットが巨大になる
  - Rebase: コンフリクト解消時にコードを意図せず失うリスクがある。共有ブランチでのrebaseは禁忌
  - Merge Commit: 履歴が冗長になるが、真の開発履歴を保持。bisectでのデバッグが複雑になる場合がある
  - 「特定の戦略が100%正解ということはない」のが現在のコンセンサス
- **2025-2026年のトレンド**:
  - Squash and Merge が最も広く採用されている。特にGitHub FlowやTBDとの組み合わせで主流
  - Conventional Commits（`feat:`, `fix:`, `chore:`等）の採用が拡大。semantic-releaseとの統合で自動バージョニングに直結
  - 「ローカルではrebaseで整理、チームレベルではmergeで統合」というフェーズ統合モデルの推奨
  - GitHub の Merge Queue 機能の活用が増加し、マージ前の統合テストが自動化

### 3. コードレビュープロセス

- **何を決めるか**: PRテンプレートの内容、レビュー観点のチェックリスト、承認ルール、レビューのSLA（応答時間目標）。
- **選択肢**:
  - **PRテンプレート**: 変更概要、関連Issue、テスト方法、スクリーンショット、チェックリスト
  - **レビュー観点**:
    - 機能正確性、セキュリティ、パフォーマンス、可読性、テストカバレッジ
    - リスクベースの優先度付け（認証・暗号化などは深くレビュー、ドキュメント修正は軽くレビュー）
  - **承認ルール**:
    - 個人開発: セルフレビューチェックリスト + AI支援レビュー
    - 小規模チーム: 1人の承認必須
    - 中〜大規模: 2人の承認必須 + CODEOWNERS による自動アサイン
  - **レビューツール**: GitHub PR Review / AI-assisted review（GitHub Copilot, CodeRabbit等）
- **選定基準**:
  - チーム規模と利用可能なレビューアーの数
  - プロダクトのリスクレベル（金融・医療→厳格なレビュー、社内ツール→軽量レビュー）
  - 開発速度とのバランス（過剰なレビューはボトルネックになる）
  - PRサイズの目安: 200〜400行が業界標準
- **トレードオフ・注意点**:
  - レビュー待ち時間が開発速度のボトルネックになりやすい。タイムリーなフィードバックが鍵
  - 個人開発でもセルフレビュー（PRを作成→一晩置いて自分でレビュー）は品質向上に有効
  - 自動化できるチェック（lint、型チェック、テスト）はCIに委譲し、人間は設計・ロジックに集中
  - 建設的なフィードバック文化の構築が重要。提案は質問形式で行い、対話を促す
  - CONTRIBUTING.md にレビュー基準を明文化し、リビングドキュメントとして維持
- **2025-2026年のトレンド**:
  - AI支援レビューの普及（GitHub Copilot code review, CodeRabbit, Graphite等）。定型チェックの自動化が進む
  - リスクベースレビューの採用拡大。全変更を同じ深度でレビューするのではなく、リスクに応じて濃淡をつける
  - レビュー指標のトラッキング（レビュー応答時間、サイクルタイム）による継続改善
  - PR サイズの小型化と頻繁なマージが推奨される傾向がさらに強まっている

### 4. Issue管理

- **何を決めるか**: Issue管理ツールの選定、Issueテンプレート、ラベル体系、ワークフロー（ステータス遷移）。
- **選択肢**:
  - **GitHub Issues + GitHub Projects**: GitHubに統合。コードとの連携が自然。無料。Project Boards でカンバン/テーブルビュー。
  - **Linear**: 高速でモダンなUI。キーボード操作に最適化。GitHub/GitLab統合あり。無料プラン（250アクティブIssueまで）。
  - **Jira**: エンタープライズ向け。高度なカスタマイズ、スプリント管理、レポーティング。10ユーザーまで無料。
  - **Notion + GitHub Issues**: Notionでロードマップ管理、GitHub Issuesで技術タスク管理のハイブリッド
- **選定基準**:
  - 既存のツールチェーンとの統合（GitHubメインならGitHub Issues、複数ツール連携ならLinear）
  - ワークフローの複雑さ（シンプル→GitHub Issues、複雑なスプリント管理→Jira）
  - チーム規模（個人〜小規模→GitHub Issues or Linear、大規模→Jira）
  - コスト（GitHub Issues: 無料、Linear: 無料〜$8/user/月、Jira: 無料〜$8.15/user/月）
- **トレードオフ・注意点**:
  - GitHub Issues は機能がシンプルだが、GitHub Projects (v2) でカンバン・ロードマップ機能が強化されている
  - Linear は DX（開発者体験）が優れるが、カスタマイズ性はJiraに劣る
  - Jira は学習コストとセットアップの複雑さが高い。個人開発や小規模チームにはオーバースペック
  - ツール間のコンテキストスイッチを最小化することが生産性に直結する
- **2025-2026年のトレンド**:
  - GitHub Projects の採用が2025年に20%増加。GitHub内で完結するワークフローへの志向
  - Linear の急成長。スタートアップやモダンな開発チームでの採用が増加
  - Jira は大企業で依然主流だが、小〜中規模チームでは代替ツールへの移行が進む
  - AI統合（IssueからのPR自動生成、コンテキスト要約）が各ツールで導入されている

### 5. リリースプロセス

- **何を決めるか**: バージョニング方式、changelog生成方法、リリースノート作成、リリース頻度・タイミング。
- **選択肢**:
  - **バージョニング方式**:
    - **SemVer (Semantic Versioning)**: `MAJOR.MINOR.PATCH`（例: `2.1.3`）。APIやライブラリに適切。
    - **CalVer (Calendar Versioning)**: `YYYY.MM.DD` や `YYYY.MINOR`（例: `2025.4`）。Webアプリやサービスに適切。
    - **ビルド番号ベース**: CI/CDのビルド番号をそのまま使用。
  - **changelog生成**:
    - **semantic-release**: Conventional Commitsから全自動でバージョン決定、changelog生成、タグ作成、リリース公開。
    - **release-please** (Google): PR駆動のリリースフロー。Release PRを自動作成。
    - **changesets**: monorepo対応。パッケージ単位でのバージョン管理。
    - **手動**: CHANGELOG.md を人が書く。
  - **リリース方式**:
    - 継続的デプロイ（mainマージ=リリース）
    - 定期リリース（毎週/隔週）
    - 手動トリガーリリース
- **選定基準**:
  - プロダクトの種類（ライブラリ→SemVer必須、Webアプリ→CalVerまたは自動）
  - デプロイ頻度（高頻度→自動リリース、低頻度→手動承認付きリリース）
  - ステークホルダーへの変更通知の要否
  - Conventional Commitsの採用可否（採用→semantic-release連携可能）
- **トレードオフ・注意点**:
  - semantic-release は Conventional Commits への厳格な準拠が必要。コミットメッセージの規律が求められる
  - CalVer は互換性情報を伝えないため、ライブラリには不向き。Webサービスには十分
  - 自動リリースはCI/CDパイプラインの信頼性が前提。テスト不十分な状態で自動リリースすると障害に直結
  - LTS（Long-term Support）バージョンの定義も長期プロダクトでは検討が必要
  - breaking change 時のマイグレーションガイド提供が重要
- **2025-2026年のトレンド**:
  - Conventional Commits + semantic-release の組み合わせがデファクトスタンダード化
  - release-please (Google) の採用増加。PR駆動のリリースフローがチームレビューと統合しやすい
  - GitHub Release Notes の自動生成機能（PRラベルベース）の活用が拡大
  - Webサービスでは「リリース」という概念自体が薄れ、継続的デプロイが主流に
  - AI によるリリースノート自動生成（変更内容の要約）が登場

### 6. タスク分割・見積もり手法（個人開発向け）

- **何を決めるか**: タスクの粒度、見積もり手法、進捗管理の方法。
- **選択肢**:
  - **タスク分割手法**:
    - ユーザーストーリーマッピング: ユーザー視点で機能を分解
    - 技術タスク分解: フロントエンド/バックエンド/インフラ等のレイヤーで分解
    - Vertical Slice: 1つのユーザー価値を縦断的に実装（UI〜DB）
  - **見積もり手法**:
    - Tシャツサイズ（XS/S/M/L/XL）: 相対的な工数感を素早く把握
    - 時間見積もり: 直感的だが精度が低い
    - タスク数カウント: 同じ粒度に分解した上でタスク数で管理
    - #NoEstimates: 見積もりを行わず、タスクを十分小さく分割して均一化
  - **進捗管理**:
    - カンバンボード（Todo / In Progress / Done）
    - GitHub Milestones（期限付きゴール）
    - 週次レビュー（振り返り + 翌週計画）
- **選定基準**:
  - 個人開発ではオーバーヘッドの小さい手法を選ぶ
  - 外部ステークホルダーへの報告義務の有無
  - プロジェクトの期間と規模
- **トレードオフ・注意点**:
  - 個人開発では厳密な見積もりよりもタスクの小型化（1日以内で完了する粒度）が有効
  - Vertical Slice は「動くもの」が早く出来るため、モチベーション維持に効果的
  - 見積もりの精度に時間をかけすぎない。ザックリ分類（小/中/大）で十分なケースが多い
  - 1週間〜2週間のイテレーションで区切り、定期的に優先度を見直す
- **2025-2026年のトレンド**:
  - #NoEstimates の考え方が浸透。タスクを十分小さくすれば見積もりが不要になるという実践
  - AI によるタスク分解支援（要件から自動的にタスクリストを生成）
  - GitHub Projects の自動化（IssueのステータスをPRのマージに連動して更新）
  - 「Ship small, ship often」の文化がさらに強化

### 7. ドキュメント管理方針

- **何を決めるか**: ドキュメントの種類・保管場所・更新タイミング・フォーマット。
- **選択肢**:
  - **ADR (Architecture Decision Records)**:
    - コードリポジトリ内の `docs/adr/` に Markdown で格納
    - 1ファイル = 1意思決定。ステータス（Proposed / Accepted / Deprecated / Superseded）を管理
    - ツール: adr-tools (CLI), Log4brains (静的サイト生成), Backstage ADR plugin
  - **設計ドキュメント**:
    - リポジトリ内 `docs/` ディレクトリ
    - Notion / Confluence などの外部Wiki
    - GitHub Wiki
  - **APIドキュメント**:
    - コードからの自動生成（OpenAPI / Swagger, TypeDoc, Storybook）
    - 手動メンテナンス
  - **更新タイミング**:
    - コードと同時更新（同一PRに含める）
    - 定期的なドキュメントレビュー（月次 / 四半期）
    - ADR: 意思決定時に即時作成
- **選定基準**:
  - ドキュメントの対象読者（自分のみ / チーム / 外部）
  - メンテナンスコストの許容度
  - コードとの同期の重要性（同期必須→リポジトリ内、参考情報→外部可）
- **トレードオフ・注意点**:
  - ADR は「なぜその決定をしたか」の記録に特化。詳細すぎると書かれなくなり、簡潔すぎると意味をなさない
  - リポジトリ内のドキュメントはコードレビューで品質管理できる利点がある
  - 外部Wikiはリッチな表現が可能だが、コードとの同期が取れなくなりがち
  - 「コードに書けることはコードに書く」（型定義、テスト、コメント）が基本。ドキュメントは「なぜ」に集中
  - ADR は遅れて書くと文脈が失われる。意思決定の直後に書くことが重要
- **2025-2026年のトレンド**:
  - ADR の採用が拡大。AWS、Google Cloud、Microsoftが公式にベストプラクティスとして推奨
  - Docs-as-Code（ドキュメントもコードと同じワークフローで管理）の普及
  - AI によるドキュメント自動生成・更新（コード変更からドキュメント差分を提案）
  - Living Documentation（テストやコードから自動生成されるドキュメント）への志向

### 8. 技術的負債の管理方法

- **何を決めるか**: 技術的負債の検出方法、優先度付け、対処のタイミング・割合。
- **選択肢**:
  - **検出方法**:
    - 静的解析ツール（SonarQube, Code Climate, ESLint/Biome）
    - コードレビューでの指摘
    - TODO/FIXME/HACK コメントの追跡
    - 依存パッケージの脆弱性・非推奨チェック（Dependabot, Renovate）
  - **管理方法**:
    - 専用ラベル（`tech-debt`）でIssue管理し、バックログに統合
    - 定期的な「テックデット棚卸し」（月次 / 四半期）
    - 開発サイクルの一定割合（10〜20%）を負債返済に充てる
    - Boy Scout Rule: 触ったコードは少し綺麗にして返す
  - **優先度付け**:
    - 影響範囲 x 発生頻度 のマトリクスで評価
    - ビジネスリスク（セキュリティ脆弱性 > パフォーマンス劣化 > コード可読性）
- **選定基準**:
  - プロジェクトのフェーズ（初期はある程度の負債を許容、成長期に返済）
  - 負債がユーザー体験やセキュリティに直結するか
  - 返済に必要な工数と得られる効果のバランス
- **トレードオフ・注意点**:
  - 「負債ゼロ」を目指すのは非現実的。意図的な負債（速度のためのトレードオフ）と無意識の負債を区別する
  - 個人開発では負債が蓄積しやすい。定期的な棚卸しの仕組みが特に重要
  - 依存パッケージの更新を放置すると、セキュリティリスクとアップグレードコストが指数関数的に増大
  - Renovate/Dependabot による自動PR + マージで依存の鮮度を維持するのが効果的
- **2025-2026年のトレンド**:
  - AI によるテックデット検出と自動修正の普及（Devin, GitHub Copilot Workspace等）
  - AI ツールがJira/Linear のテックデットチケットを自動でピックアップしてPRを作成する事例が増加
  - Renovate/Dependabot の組み合わせによる依存更新の自動化がデファクト
  - 「負債を返す専用スプリント」ではなく「各イテレーションで少しずつ返す」インクリメンタルアプローチが主流
  - コード品質メトリクスのダッシュボード化（SonarQube Quality Gates等）で可視化
