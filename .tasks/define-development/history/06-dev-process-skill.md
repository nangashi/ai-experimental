# 06: dev_process スキル設計

## 日付

2026-02-25

## 目的

classification.md の P3（開発プロセス設計）に対応する新スキル `dev_process` を設計する。P3 のプロセス項目を固定カタログとして保持し、P1 の設計決定（architecture.md）と P2 の規約（standards.md）に基づいて対話的にプロセスを設計する。

## 入力

- `classification.md`（P3 項目定義の参照元）
- `history/04-arch-design-skill-redesign.md`（ワークフロー・品質ゲートのパターン参照）
- `history/05-standards-design-skill.md`（P2 スキル設計、パターン参照）
- 現行 `.claude/skills/process_design/`（参考、テンプレートの流用元）
- `.claude/instructions/`（設計原則）

## スコープ

### 新規作成

| 観点 | 設計 |
|------|------|
| スキル名 | `dev_process` |
| 入力 | `architecture.md` + `requirements.md` + `standards.md` |
| 出力 | `docs/project-definition/development-process.md` |
| 決定項目 | P3 カタログから固定 |
| ADR 要否 | カタログに事前定義済み（6 項目） |
| 議論グループ | 5 固定グループ（依存順） |

### process_design との関係

- `process_design` は現行のまま残す（P2+P3 をカバーする旧スキル）
- `standards_design`（P2）+ `dev_process`（P3）で `process_design` の責務を分割
- 新規プロジェクトでは `arch_design` → `standards_design` → `dev_process` の順で使用

## P3 決定項目と議論グループ

P3 の項目を依存関係に基づき 5 議論グループに編成する。

### P1・P2 との性質の違い

| 観点 | P1（arch_design） | P2（standards_design） | P3（dev_process） |
|------|-------------------|----------------------|-------------------|
| 決定の性質 | 「どれを選ぶか」型 | 「どう書くか」型 | 「どう回すか」型 |
| ADR 密度 | 高（29/39） | 低（2/33） | 中（6/33） |
| 主な出力物 | 設計書 + ADR 群 | 規約文書 | プロセス定義 + パイプライン設計 |
| 対話パターン | 項目ごと議論 → ADR | グループ一括提案 | ツール選定は ADR、フロー定義は一括提案 |

### グループ編成の根拠

- **Git → テスト → CI/CD の依存チェーン**: classification.md のレビュー（history/03）で確認された依存順序。CI パイプラインがテスト戦略に依存し、テスト戦略が Git ワークフロー（ブランチ戦略）に依存する
- **データ管理・セキュリティの統合**: 項目数が少なく（3+2=5）、どちらもインフラ運用に関わるため統合。監視・運用（2 項目）とクロスデバイス（1 項目）も吸収
- **DX・AI の統合**: 開発環境（4 項目）と AI 統合プロセス（3 項目）は開発者の作業環境に関わる。AI 部分は省略可能

### Group 1: Git ワークフロー・開発プロセス（5 項目）

コードの流し方と管理方法を決める。全後続グループの前提。

| # | 項目 | ADR | 手戻り | 備考 |
|---|------|:---:|:---:|------|
| 1 | Git ワークフロー | 要 | 中 | GitHub Flow/Trunk-based 等 |
| 2 | ブランチ戦略 | — | 中 | 命名規則、保護ルール、マージ戦略。#1 に依存 |
| 3 | Issue 管理 | 要 | 小 | GitHub Issues/Linear 等 |
| 4 | リリースプロセス | — | 中 | SemVer/CalVer、自動リリース。#1 に依存 |
| 5 | ドキュメント管理方針 | — | 小 | ADR、Docs-as-Code 等 |

**Group 内の議論順序**: #1（Git ワークフロー）→ #2, #4（ブランチ戦略、リリースプロセス）→ #3, #5（Issue 管理、ドキュメント）。#1 が #2, #4 の前提。

### Group 2: テスト戦略（7 項目）

品質担保の仕組みを決める。P1（技術スタック）と P2（テスト関連規約）に依存。Group 1（ブランチ戦略）が CI でのテスト実行戦略に影響。

| # | 項目 | ADR | 手戻り | 備考 |
|---|------|:---:|:---:|------|
| 1 | テスト投資配分とレベル別責務 | — | 中 | ピラミッド/トロフィー選択 + ユニット/結合/E2E の責務 |
| 2 | テストフレームワーク選定 | 要 | 中 | Vitest/Jest、Playwright/Cypress。P1 技術スタックに依存 |
| 3 | テストカバレッジ目標 | — | 小 | 数値目標と測定方法 |
| 4 | テストデータ管理 | — | 小 | フィクスチャ/ファクトリー方式 |
| 5 | モック戦略 | — | 中 | MSW 等の選定と方針。テスト用・開発用モックサーバーを含む |
| 6 | スナップショットテストの採否 | — | 小 | 採用/限定採用/不採用 |
| 7 | CI でのテスト実行戦略 | — | 小 | 全テスト/差分実行/ステージ分割。Group 3 の CI 設計の入力 |

### Group 3: CI/CD・環境管理（6 項目）

自動化パイプラインとデプロイ環境を決める。Group 1（Git ワークフロー）、Group 2（テスト戦略）、P1（ホスティング）、P2（Linter/Formatter）に依存。

| # | 項目 | ADR | 手戻り | 備考 |
|---|------|:---:|:---:|------|
| 1 | CI/CD プラットフォーム | 要 | 中 | GitHub Actions 等。P1 ホスティングと連動 |
| 2 | CI パイプライン構成 | — | 中 | lint→test→build→deploy の設計。Group 2 テスト戦略に依存 |
| 3 | CD の戦略 | — | 中 | 自動デプロイ/手動承認/ロールバック |
| 4 | IaC の採否 | 要 | 中 | Terraform/Pulumi/SST 等 |
| 5 | 環境管理 | — | 中 | dev/staging/prod の分離戦略 |
| 6 | シークレット管理（CI/CD 内） | — | 中 | GitHub Secrets 等 |

**Group 内の議論順序**: #1（CI/CD プラットフォーム）→ #2（CI 構成）→ #3, #5, #6（CD、環境、シークレット）→ #4（IaC）。#1 が他の全項目の前提。#4 は環境管理（#5）と連動。

### Group 4: インフラ・運用プロセス（8 項目）

データ管理、セキュリティ、監視、クロスデバイスの運用プロセス。P1 決定（DB/ORM、ホスティング、PWA 採否）に依存するが、Group 1-3 への依存は弱い。

| 3 | データバックアップ・リストア戦略 | — | 中 | RPO/RTO 設定、バックアップ方式、復旧手順 |
| 4 | シークレット管理 | — | 中 | 環境変数/Secret Manager 等 |
| 5 | 依存関係の脆弱性管理 | — | 小 | Snyk/npm audit 等の CI 組込 |
| 6 | エラートラッキング | 要 | 中 | Sentry 等。初期から入れるべき |

**統合の根拠**: データ管理プロセス（3 項目）、セキュリティプロセス（2 項目）、監視・運用（2 項目）、クロスデバイス（1 項目）はいずれも項目数が少なく、独立グループにするとチェックポイント過多になる。「運用インフラ」というテーマで凝集。

### 議論グループ間の依存関係

```
P1 (architecture.md) + P2 (standards.md)
  ├→ Group 1（Git ワークフロー）
  │    ├→ Group 2（テスト戦略）
  │    │    └→ Group 3（CI/CD）← Group 1, 2 両方に依存
  │    └→ Group 3
  ├→ Group 4（インフラ・運用プロセス）
  └→ Group 5（開発環境・AI）
```

Group 1→2→3 は強い依存チェーン。Group 4, 5 は P1/P2 に依存するが Group 1-3 とは独立。議論順序は 1 → 2 → 3 → 4 → 5 とする。

### Phase 1: 対話的プロセス設計

**目的**: P3 カタログの各グループを依存順に議論し、全項目のプロセス設計を確定する

#### グループ議論のフロー

含まれるグループを順に以下のパターンで議論する:

##### 1. グループ提示

`{skill_dir}/references/p3-decision-catalog.md` から当該グループの項目テーブルを参照し、提示する。既存 ADR で解決済みの項目があればマークする。

##### 2. プロセスの提案

グループ内の全項目について、P1 決定（`{tech_stack}`, `{hosting}`, `{deploy_strategy}`）、P2 規約（`{linter_formatter}`, `{quality_enforcement}`）、およびプロジェクト文脈（個人開発、MVP）に基づき、プロセスを提案する。各項目について:

- **直接決定**: P1/P2 決定や制約から選択肢が 1 つに絞られる場合。根拠を示して決定
- **ADR 提案**: 複数の有力な候補がありトレードオフ分析が必要な場合
- **N/A**: P1 の決定により該当しない場合（例: PWA 不採用時の Service Worker 設定）。スキップ理由を明示

##### 3. ADR の提案とグルーピング

ADR が必要な項目について:

```
### 技術選定: {選定ポイント名}

この判断は選択肢のトレードオフ分析が必要です。
以下のコマンドを **別セッション** で実行してください:

`/adr_create {選定ポイント名}。スコープ: {対象項目リスト}。制約: {確定済み決定から導出した制約}`

完了したら「完了」と回答してください。
```

ADR グルーピングの判断基準:
- 同一 ADR にまとめるべき: 選択肢が相互に制約し合う項目（例: Git ワークフロー + ブランチ戦略は連動するが、ブランチ戦略は ADR 不要のため Git ワークフローの ADR スコープに含める）
- 別 ADR にすべき: 独立に判断でき、組み合わせの影響が小さい項目（例: テスト FW と CI/CD プラットフォーム）

##### 4. ADR 読込と整合性検証（ADR がある場合）

ユーザーが「完了」と回答したら:

1. `{adr_dir}` を Glob し、新 ADR を特定
2. 「決定」「受け入れたトレードオフ」セクションのみ Read
3. **QG-2**: 既存決定・他 ADR・`{constraints}` との矛盾検証
   - 矛盾検出時: ユーザーに提示し解決方法を相談
4. 決定を追加
5. **後続影響チェック**: ADR の決定が同グループ・後続グループの項目に影響するか確認
   - 例: GitHub Actions 採用 → IaC の選択肢が変化（SST/Pulumi が候補に）
   - 例: Trunk-based 採用 → ブランチ戦略が簡素化、リリースプロセスが変化

##### 5. グループサマリと確認

グループ内の全項目を議論後:

```
### {グループ名} サマリ

| # | 項目 | 決定内容 | ADR | 根拠 |
|---|------|---------|-----|------|
| 1 | {項目名} | {確定した内容の要約} | {ADR-NNNN or —} | {根拠の要約} |
| ... | | | | |

修正があればお知らせください。問題なければ「ok」と回答してください。
```

**ユーザーチェックポイントの根拠**: Group 1→2→3 は強い依存チェーン。各グループの決定が後続の前提になる。

#### Group 3 固有: 品質チェック配置サマリの作成

Group 3（CI/CD）の議論完了時、P2 の `{quality_enforcement}` と Group 2（テスト戦略）の結果を統合して品質チェック配置サマリを作成する:

```
### 品質チェック配置サマリ

| チェック | 保存時 | commit 時 | push 時 | CI | CD | 備考 |
|---------|-------|----------|---------|-----|-----|------|
| {チェック名} | {○/—} | {○/—} | {○/—} | {○/—} | {○/—} | {備考} |

設計原則:
- commit 時: 高速なチェックのみ（変更ファイル対象の lint/format）
- push 時: 中量のチェック（型チェック等）
- CI: 全量チェック（テスト全量、ビルド検証、セキュリティスキャン）
- CD: デプロイ前ゲート（E2E テスト等）
- 重複排除: 同一チェックが複数タイミングに配置されていないことを確認

修正があればお知らせください。問題なければ「ok」と回答してください。
```

品質チェック配置サマリは Group 3 サマリとは別のユーザーチェックポイントとする。**根拠**: P2 規約（lint/format）と P3 プロセス（テスト/CI/CD）の統合地点であり、配置ミスはCI の信頼性に直結する。

#### QG-3: プロセス完全性ゲート（Phase 1 完了前）

全グループ議論後:

1. P3 カタログの全項目に対して、プロセスが確定 or N/A 判定されているか
2. `{cross_cutting}` と `{p3_handoff}` の P3 引き継ぎ事項がプロセスに反映されているか

```
### プロセス完全性チェック

| チェック | 結果 | 詳細 |
|---------|------|------|
| P3 カタログ網羅 | {M}/{N} | 全項目にプロセスが確定しているか |
| P1 引き継ぎ反映 | {M}/{N} | architecture.md の P3 引き継ぎ事項が反映されているか |
| P2 引き継ぎ反映 | {M}/{N} | standards.md の P3 引き継ぎ事項が反映されているか |
```

## 設計判断の根拠

### スキル名: `dev_process`（`process_design` ではなく）

**判断**: 既存の `process_design` と異なる名前で新規作成する

**根拠**:
- `process_design` はすでに存在し、P2+P3 をカバーする旧スキル
- 同名で上書きするとスキルの参照が混乱する
- `dev_process` は簡潔で、`arch_design` / `standards_design` と並んで分かりやすい

### 5 議論グループの編成

**判断**: 9 カテゴリ（classification.md の原分類）を 5 グループに統合

**根拠**:
- 9 グループだとユーザーチェックポイントが過多になる（workflow-structure-design: ユーザーチェックポイント配置）
- 統合基準: (1) 依存関係が近い、(2) 項目数が少ないカテゴリ（監視 2 項目、クロスデバイス 1 項目）は隣接グループに吸収
- Group 1→2→3 の依存チェーンは classification.md レビュー（history/03）で「Git→テスト→CI/CD」の順序が確認済み
- Group 4 への統合: データ管理（3）+ セキュリティ（2）+ 監視（2）+ クロスデバイス（1）= 8 項目。「運用インフラ」として凝集度が高い

### 3 入力ファイル体制

**判断**: P3 は architecture.md + standards.md + requirements.md の 3 ファイルを入力とする

**根拠**:
- P3 のプロセスは P1（技術スタック、ホスティング）と P2（Linter/Formatter、品質ツール）の両方に依存する
- 特に CI パイプライン設計は P2 の Linter/Formatter 選定と P1 のホスティング選定の両方が必要
- 品質チェック配置サマリは P2 の自動強制サマリとの統合が必須

**トレードオフ**:
- Phase 0 のコンテキスト消費がやや増加 → 構造サマリのみ抽出し、規約本文は保持しないことで緩和

### 品質チェック配置サマリの独立チェックポイント化

**判断**: Group 3 の中で品質チェック配置サマリをグループサマリとは別のチェックポイントにする

**根拠**:
- 品質チェック配置サマリは P2（保存時/commit 時の自動強制）と P3（CI/CD のチェック）の統合地点
- 配置の重複・抜け漏れは CI の信頼性と開発速度に直結する
- Group 3 のプロセス全体の確認とは別に、チェック配置の精査が必要

### 整合性チェック（5 観点設計レビューではなく）

**判断**: P2 と同様、整合性チェックを採用

**根拠**:
- P2 と同じ理由: プロセス文書は設計文書ではないため、security/performance/reliability の設計レビューは的外れ
- プロセス文書の品質リスクは「P1/P2 との矛盾」「プロセス間の矛盾」「過剰プロセス」であり、整合性チェックで検出可能
- P3 固有のリスク「プロジェクト文脈の適合性」は QG-3b で事前に検証済み

### QG-3b に「プロジェクト文脈の適合性」を追加（P2 にはない観点）

**判断**: P3 の QG-3b に 4 つ目の観点「プロジェクト文脈の適合性」を追加する

**根拠**:
- P3 はプロセスの「重さ」が直接的に開発負荷に影響する
- classification.md のレビュー（history/03）で「個人開発/MVP 文脈での過剰」が複数件指摘された（コードレビュープロセス削除、ローカル環境セットアップ実装時移動等）
- P2 の規約はツールで自動化可能だが、P3 のプロセスは手動運用が多いため過剰感が強くなる
- P1 でも QG-3b 観点 2「制約との適合性」で類似チェックをしているが、P3 ではプロセスの運用コストに焦点を当てた文言に変更
