# 01: 開発前決定事項の網羅的調査

## 日付

2026-02-25

## 目的

新規開発プロジェクト（クロスデバイス対応リーディングリスト管理ツール）を安定した開発プロセスに乗せる前に、何を決める必要があるかを網羅的に調査する。発散フェーズとして、プロジェクト固有の実装や既存データとは独立に、一般的なベストプラクティスを広く収集する。

## 入力

- `docs/project-definition/problem-statement.md` — プロジェクトの問題定義・ユースケース・スコープ・非機能要求

## 作業内容

### Step 1: カテゴリ分割と作業計画

開発の側面を15カテゴリに分解し、作業計画を `work-plan.md` に記載した。

### Step 2: 7グループ並列リサーチ

15カテゴリを7グループに分け、各グループをサブエージェントに並列で委譲した。各エージェントはWeb検索で2025-2026年の最新トレンドを含めて調査し、結果をファイルに直接書き出した。

| Group | 担当エージェント | 調査対象 | 出力ファイル |
|-------|----------------|---------|-------------|
| A | Tech Stack | 技術スタック・システムアーキテクチャ | 01, 02 |
| B | Project Structure | プロジェクト構成・コーディング規約・DX | 03, 04, 05 |
| C | Dev Process | 開発プロセス・CI/CD・インフラ | 06, 07 |
| D | Testing & Data | テスト戦略・データ管理 | 08, 09 |
| E | Security & Monitoring | セキュリティ・監視・運用 | 10, 11 |
| F | Frontend & AI | フロントエンド設計・クロスデバイス・AI統合 | 12, 13, 14 |
| G | Failure Lessons | 失敗事例・教訓 | 15 |

### Step 3: 横断レビュー

全15ファイルの調査結果を以下の観点で横断レビューした:

1. **arch_design / process_design スキルとの照合** — 既存スキルの出力テンプレート（`references/output-template.md`）がカバーする設計項目と調査カテゴリのカバレッジを検証
2. **失敗事例からの不足検出** — `15-failure-lessons.md` の33事例から、調査内容で考慮不足のリスクを特定
3. **決定順序の整理** — 上流→下流の依存関係に基づく推奨決定順序を策定
4. **必須 vs 段階的の分類** — 「開発開始前に必須」と「開発中に決めてよい」を分類

## 発見した課題（次のアクション候補）

### 追加調査が必要な領域

| 優先度 | 領域 | 理由 |
|--------|------|------|
| 高 | UX設計プロセス | 技術面はカバー済みだが、デザインツール・ワイヤーフレーム・プロトタイプのプロセスが欠落 |
| 中 | 運用移行計画 | 既存のRSS+Xブックマーク運用から新システムへの移行計画が未検討 |
| 中 | 開発フェーズ分割 | MVPのUC実装順序、各フェーズの完了条件が未定義 |

### 失敗事例から特に注意すべき点

- AI生成コードのセキュリティ脆弱性（人間比2.74倍）→ CI内セキュリティスキャン必須
- LLM APIコスト暴走（見積もり10倍）→ コスト上限の事前設定
- スコープクリープ → MVP範囲の厳格な管理

## 備考

- 各調査ファイルの構造は統一フォーマット: 決定事項一覧（サマリテーブル）→ 各項目の詳細（何を決めるか / 選択肢 / 選定基準 / トレードオフ / 2025-2026年トレンド）
- 調査はプロジェクト固有の実装（既存ADR等）とは独立に実施。収束フェーズで既存の技術選定結果と突合する必要がある
