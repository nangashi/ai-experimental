# 02: フェーズ分類の検討

## 日付

2026-02-25

## 目的

01の調査で発散した約140の決定事項を、スキル化を見据えたフェーズに分類する。フェーズ分けの基準と構造を確定する。

## 入力

- `00-summary.md` — 調査サマリ（既存Phase 1-7、必須vs段階的の分類）
- `01-technology-stack.md` 〜 `15-failure-lessons.md` — 各カテゴリの詳細調査

## 検討内容

### フェーズ境界の判断基準

| # | 基準 | 説明 |
|---|------|------|
| 1 | 未決定時の手戻りコスト | 決めずに実装を始めた場合、後からどれだけのコードを書き直すか |
| 2 | ADR比較検討の要否 | 複数の有力な選択肢があり、トレードオフを明示的に比較して記録すべきか |
| 3 | 依存関係 | 上流の決定がないと選択肢が定まらない/意味のある判断ができない |
| 4 | フェーズに適切な名前がつくか | 凝集度の高い、一言で説明できるまとまりになっているか |

### 既存Phase 1-7の課題

00-summary.mdのPhase 1-7は「決定の推奨順序」としては妥当だが、スキル化の単位として以下の課題があった:

- Phase 1（基盤決定）とPhase 2（技術スタック確定）の境界が不明瞭。一度に決められることは統合すべき
- Phase 3（プロジェクト基盤）に設計（ディレクトリ構成）とプロセス（Gitワークフロー）が混在
- 横断的関心事（セキュリティ、エラーハンドリング等）が単一フェーズに押し込まれているが、実際は複数フェーズにまたがる
- Phase 4以降の「詳細設計」はスキル化の必要性が低い（機能単位で実装時に行えばよい）

### 確定したフェーズ構造: 3フェーズ

```
Phase 1: アーキテクチャ・技術選定
  ├ ADR比較検討が中心
  ├ 未決定→全コードに手戻り
  └ 出力: 設計書 + ADR群
         ↓
Phase 2: 開発規約設計
  ├ ポリシー策定が中心
  ├ 未決定→コードの一貫性が崩壊
  └ 出力: 規約ドキュメント + 設定ファイル群
         ↓
Phase 3: 開発プロセス設計
  ├ ワークフロー定義が中心
  ├ 未決定→開発の回し方が属人化
  └ 出力: プロセス定義 + CI/CDパイプライン定義
         ↓
  実装開始（詳細設計は機能単位で実施）
```

### 各フェーズの定義

#### Phase 1: アーキテクチャ・技術選定

- **一言で**: システムの骨格と使う道具を決める
- **性質**: ADR駆動。複数候補の比較検討が主な作業。決定の多くは「どれを選ぶか」型
- **手戻りコスト**: 最大。アーキテクチャパターンの変更はほぼ全書き直し
- **含まれるもの**:
  - アーキテクチャパターン、レイヤー分離、FE-BE分離戦略
  - 技術スタック全般（言語、FW、DB、ORM、ランタイム、ホスティング等）
  - 認証方式・サービス、BaaS採否
  - API設計方針、状態管理アーキテクチャ
  - 横断的関心事の方針（エラーハンドリング全体方針、キャッシュ戦略、DI方針）
- **既存スキルとの関係**: `arch_design` がほぼ対応

#### Phase 2: 開発規約設計

- **一言で**: コードの書き方・構成のルールを決める
- **性質**: ポリシー策定。ADRが必要なのは一部のツール選定（Linter/Formatter）のみ
- **手戻りコスト**: 高。命名規則の変更は全ファイルに影響。ただしPhase 1ほど構造的な書き直しにはならない
- **含まれるもの**:
  - プロジェクト構成・ディレクトリ構造
  - コーディング規約（命名、インポート順序、コメント方針、TypeScript設定）
  - Linter/Formatter選定・設定
  - エラーハンドリングパターン（コードレベル）
  - ログ出力規約
  - テスト命名規則・構造規約
  - セキュリティコーディング規約（入力検証パターン、CSP/CORS方針）
- **Phase 1への依存**: FWが決まらないとディレクトリ構造が決まらない。言語が決まらないとLinter候補が定まらない
- **既存スキルとの関係**: `process_design` が部分的に対応

#### Phase 3: 開発プロセス設計

- **一言で**: コードの流し方と品質担保の仕組みを決める
- **性質**: ワークフロー定義。「どう回すか」の仕組みを作る作業
- **手戻りコスト**: 中〜高。Git戦略の変更はブランチ再構成が必要だが、コードの書き直しにはならない
- **含まれるもの**:
  - Gitワークフロー・ブランチ戦略
  - CI/CDパイプライン設計
  - テスト戦略（ピラミッド/トロフィー、カバレッジ目標、モック戦略、実行戦略）
  - 品質ゲート定義
  - リリースプロセス
  - 開発環境セットアップ
  - 依存関係更新戦略
  - コードレビュー方針
- **Phase 1-2への依存**: テストFWは技術スタックに依存。CI/CDはホスティング先に依存。品質ゲートはLinter/Formatter選定に依存
- **既存スキルとの関係**: `process_design` が部分的に対応

### Phase 2と3の分割根拠

| 比較軸 | Phase 2（開発規約） | Phase 3（開発プロセス） |
|--------|-------------------|---------------------|
| 決めること | 静的なルール（コードがどうあるべきか） | 動的なフロー（コードがどう流れるか） |
| 成果物の形 | 規約ドキュメント + 設定ファイル | プロセス定義 + パイプライン設定 |
| 変更の影響 | 既存コード全体に波及 | コードの中身には直接影響しない |
| ADR密度 | 低（ツール選定のみ） | 低（大半はポリシー決定） |

### 横断的関心事の扱い

同一トピックが複数フェーズにまたがるパターン:

| トピック | Phase 1 | Phase 2 | Phase 3 |
|---------|---------|---------|---------|
| セキュリティ | 認証方式/サービス選定 | 入力検証パターン、CSP/CORS方針 | 依存脆弱性スキャン(CI)、シークレット管理 |
| エラーハンドリング | 全体方針(Result型 vs 例外) | コーディングパターン | エラー追跡サービス導入 |
| テスト | — | テスト命名/構造規約 | テスト戦略/ピラミッド、FW選定、CI統合 |
| データ管理 | DB選定、ORM選定 | スキーマ命名規則 | マイグレーション戦略、バックアップ |
| 監視・ログ | — | ログ出力規約、計装規約 | 監視基盤、アラート、ログ集約 |
