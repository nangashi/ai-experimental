## P1 決定項目と議論グループ

39 項目を依存関係に基づき 5 議論グループに編成する。

### グループ編成の根拠

- **依存順**: 上流の決定が下流の選択肢を制約するため、依存順で議論する
- **認証方式を Group 1 に含める**: 認証方式（JWT/セッション/パスキー等）はアーキテクチャレベルの判断であり、技術スタック選定（認証ライブラリ）の前提になる
- **フロントエンド統合**: FE ライブラリ + クロスデバイス + i18n は FE FW 選定後に一括で議論可能

### Group 1: アーキテクチャ基盤（9 項目）

システムの骨格と認証モデルを決める。全後続グループの前提。

| # | 項目 | ADR | 手戻り | 備考 |
|---|------|:---:|:---:|------|
| 1 | アーキテクチャパターン | 要 | 極大 | モノリス/サーバーレス等 |
| 2 | レイヤー分離パターン | 要 | 大 | Clean/Hexagonal 等 |
| 3 | FE-BE 分離戦略 | 要 | 大 | SPA+API/SSR 等。FW 選定に直結 |
| 4 | API 設計方針 | 要 | 大 | REST/GraphQL/tRPC。プロトコル選択含む |
| 5 | 状態管理アーキテクチャ | 要 | 大 | サーバー/クライアントステートの分離方針 |
| 6 | キャッシュ戦略（方針） | 要 | 中 | 多層キャッシュの全体設計 |
| 7 | エラーハンドリング全体方針 | 要 | 大 | Result 型/例外等。コードパターンは P2 |
| 8 | 認証方式 | 要 | 極大 | JWT/セッション/パスキー等。全レイヤーに影響 |
| 9 | セッション管理 | 要 | 大 | 認証方式と連動 |

| 2 | プログラミング言語 | 要 | 極大 | 全コードに影響 |
| 3 | フルスタック FW 採否 | 要 | 極大 | FE/BE 分離戦略と連動 |
| 4 | フロントエンド FW | 要 | 極大 | UI 全体の書き方を規定 |
| 5 | バックエンド FW | 要 | 極大 | API 実装全体を規定 |
| 6 | ランタイム | 要 | 大 | Node.js/Deno/Bun |

### Phase 1: 対話的アーキテクチャ設計

**目的**: P1 カタログの各グループを依存順に議論し、全項目の設計を確定する

#### グループ議論のフロー

含まれるグループを順に以下のパターンで議論する:

##### 1. グループ提示

`{skill_dir}/references/p1-decision-catalog.md` から当該グループの項目テーブルを参照し、提示する。既存 ADR で解決済みの項目があればマークする。

##### 2. 項目ごとの判断

各項目について、要件・制約・既存決定から以下を判断する:

- **直接決定**: 制約から選択肢が 1 つに絞られる場合。根拠を示して決定する。決定内容が後続グループの項目の ADR 要否や N/A 判定に影響する場合は更新する（例: フルスタック FW 採用 → Group 3 のルーティング設計が N/A に）
- **ADR 提案**: 複数の有力な候補があり、トレードオフ分析が必要な場合
- **N/A**: 要件上該当しない場合（例: フルスタック FW 採用時のルーティング設計）。スキップ理由を明示

##### 3. ADR の提案とグルーピング

ADR が必要な項目は、関連する項目をまとめて 1 つの ADR 提案にする。

```
### 技術選定: {選定ポイント名}

この判断は選択肢のトレードオフ分析が必要です。
以下のコマンドを **別セッション** で実行してください:

`/adr_create {選定ポイント名}。スコープ: {対象項目リスト}。制約: {確定済み決定から導出した制約}`

完了したら「完了」と回答してください。
```

ADR グルーピングの判断基準:
- 同一の ADR で解決すべき: 選択肢の組み合わせが相互に制約し合う項目（例: 言語 + FW + ランタイム）
- 別の ADR にすべき: 独立に判断でき、組み合わせの影響が小さい項目（例: DB 選定と FE FW 選定）

##### 4. ADR 読込と整合性検証

ユーザーが「完了」と回答したら:

1. `{adr_dir}` を Glob し、新 ADR を特定
2. 「決定」「受け入れたトレードオフ」セクションのみ Read
3. **QG-2**: 既存決定・他 ADR・`{constraints}` との矛盾検証
   - 矛盾検出時: ユーザーに提示し解決方法を相談
4. 決定を `{tech_decisions}` に追加
5. **後続影響チェック**: ADR の決定内容が後続グループの項目に影響するか確認する。影響がある場合は、当該項目の ADR 要否・N/A 判定を更新し、ユーザーに提示する
   - 例: Cloudflare Access 採用 → 認証ライブラリの選択肢が変化
   - 例: BaaS 採用 → 複数グループの項目が N/A またはスコープ変更

##### 5. グループ確認

グループ内の全項目を議論後、グループ決定サマリを提示:

```
### {グループ名} サマリ

| 項目 | 決定 | ADR | 根拠 |
|------|------|-----|------|
| ... | ... | ... | ... |

修正があればお知らせください。問題なければ「ok」と回答してください。
```

**ユーザーチェックポイントの根拠**: 各グループの決定は後続グループの前提になるため、方向転換コストが高い。

#### QG-3b: 技術選定の整合性検証（Phase 1 完了前、QG-3 後）

QG-3 通過後、確定した全決定の組み合わせに対して以下の 3 観点を検証する:

## 設計判断の根拠

### 動的次元導出の廃止

**判断**: P1 カタログの固定グループに置き換える

**根拠**:
- classification.md で 39 項目が 2 ラウンドのレビューを経て確定済み
- 動的導出は「導出ルールの品質」に依存するが、ルールの網羅性を保証する手段がなかった（workflow-structure-design: 外部基準による接地）
- 固定カタログは classification.md という「外部基準」に接地されており、カバレッジが構造的に担保される
- 項目の N/A 判定で適応性を確保する（不要な項目をスキップ可能）

**トレードオフ**:
- classification.md が存在しないプロジェクトでは使えない → この制約を受け入れる（汎用性よりカバレッジ品質を優先）

### 5 議論グループの編成

**判断**: 9 カテゴリ（classification.md の原分類）を 5 グループに統合

**根拠**:
- 9 グループだとユーザーチェックポイントが過多になり「ok 疲労」を招く（workflow-structure-design: ユーザーチェックポイント配置）
- 統合基準: (1) 依存関係が近い、(2) 項目数が少ない（2 項目以下）カテゴリは隣接グループに吸収
- 認証方式 + セッション管理を Group 1 に移動: 認証モデルはアーキテクチャレベルの判断であり、認証ライブラリ（Group 2）の前提

### ADR グルーピング

**判断**: スキル実行時にエージェントが判断する（事前に ADR 単位を固定しない）

**根拠**:
- ADR の粒度は制約とプロジェクト文脈に依存する（例: フルスタック FW 採用時は言語・FE/BE FW・ランタイムが 1 ADR で解決）
- 事前に ADR 単位を固定すると、不要な ADR が発生するか、必要な分離ができない
- 判断基準のみ提示し、実行時の柔軟性を確保する（prompt-design-guidelines: 剛性的ステップ指定の回避）

### Phase 1 内のグループ議論フロー

**判断**: 項目テーブルの提示 → 項目ごとの判断（直接決定/ADR/N/A）→ グループサマリ確認。グループ内の詳細な議論順序は固定しない

**根拠**:
- prompt-design-guidelines: 「剛性的ステップ指定の回避」「カテゴリ別分解の優先」
- Group 2 のように内部に自然な連鎖がある場合、エージェントが文脈に応じて最適な順序で議論する方が品質が高い
- Group 2 内の推奨議論順序（BaaS → 言語/FW → DB/ORM → 認証ライブラリ → ホスティング）はカタログに備考として記載するが、強制はしない

## 横断的関心事の取り扱い
