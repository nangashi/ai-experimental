# 03: フェーズ分類のレビューと修正

## 日付

2026-02-25

## 目的

02で確定した3フェーズ構造に基づく分類結果（classification.md）を、3観点のサブエージェントレビューで検証・修正する。

## 入力

- `classification.md` — 約150項目の3フェーズ+実装時分類（初版）
- `01-technology-stack.md` 〜 `14-ai-integration.md` — 各カテゴリの詳細調査

## レビュー観点

1. **決定項目の粒度** — 粗すぎ/細かすぎ/重複/不均一
2. **決定の依存関係と順番の矛盾** — 後方依存/前方参照欠落/循環依存
3. **事前決定すべき項目か** — 個人開発・MVP文脈での過剰/不足/不要

## 実施内容

### Round 1: 初版レビュー（3エージェント並列）

28件の指摘（高6, 中13, 低9）。高・中の16件について対応判断を実施。

#### 適用した修正（16件）

**重複統合（4件）**
- P1: 認証ライブラリ/サービス(01) と 認証サービス/ライブラリ(10) → 01側に統合
- P1: APIプロトコル(01) と API設計方針(02) → 02側に統合
- P2: 入力バリデーション(10) と データバリデーション戦略(09) → 「バリデーション戦略」に統合
- P3: バックアップ・災害復旧(11) と データバックアップ・リストア(09) → 09側に統合

**個人開発/MVP文脈で過剰（7件）**
- P1→実装時: 認可モデル、ドメインモデリング方針、プッシュ通知の採否
- P1→FW選定に吸収: DI方針
- P2→削除: SQLインジェクション対策（ORM注記に吸収）、OWASP Top 10（横断チェックリスト化）
- P3→実装時: 技術的負債の管理方法

**個人開発で不要（2件）**
- P3から削除: コードレビュープロセス、タスク分割・見積もり手法

**依存関係修正（2件）**
- P3セクション順: Git→CI/CD→テスト を Git→テスト→CI/CDに変更（CIパイプラインのテスト依存を解消）
- P3: 開発用モックサーバーをテスト戦略セクションに移動（モック戦略との密結合）

**備考追記（1件）**
- P2: ディレクトリ構成⇔コンポーネント設計の連動を明記

### Round 2: 修正版レビュー（3エージェント並列）

10件の指摘（高3, 中5, 低2）。高・中の8件について対応判断を実施。

#### 適用した修正（8件）

**粒度統合（4件）**
- P1: PWAの採否 + ネイティブアプリとの比較検討 → 「クライアント配信方式の選択」に統合（包含関係の解消）
- P3: テストピラミッド/トロフィー選択 + テストレベル別の方針 → 「テスト投資配分とレベル別責務」に統合
- P3: 開発用モックサーバー/データ → モック戦略に統合（備考に「テスト用・開発用を含む」）
- P1→P2: レスポンシブデザイン方針 + ブレイクポイント設計 → P2「レスポンシブ設計」に統合（手戻りコスト中でP1基準に合わない）

**循環依存解消（1件）**
- P2: コンポーネント設計→ディレクトリ構成の依存方向を明確化（「連動して決定」→一方向依存に）

**実装時に移動（3件）**
- P3→実装時: ローカル開発環境セットアップ（個人開発で環境統一不要）
- P3→実装時: テストの命名規則・構造（FW慣習に従えば十分）
- P3→実装時: デプロイモニタリング（PaaSデフォルトで開始）
