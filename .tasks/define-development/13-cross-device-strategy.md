## 各項目の詳細

### 1. レスポンシブデザイン方針

- **何を決めるか**: デザイン・実装の起点をモバイル画面にするか、デスクトップ画面にするか
- **選択肢**:
  - **モバイルファースト（推奨）**: 最小画面から設計し、min-widthメディアクエリで段階的に拡張。モバイルトラフィックが過半数を占める2025年の標準
  - **デスクトップファースト**: 大画面から設計し、max-widthメディアクエリで縮小適応。管理画面・BIツール等デスクトップ主体のアプリ向け
  - **コンテンツファースト**: 特定デバイスに依存せず、コンテンツの読みやすさを起点に設計
- **選定基準**:
  - 主要ユーザーの利用デバイス比率
  - アプリの性質（情報閲覧主体 vs 入力・編集主体）
  - パフォーマンス要件（モバイルファーストはCSSの削減に有利）
- **トレードオフ・注意点**:
  - モバイルファーストは「機能の優先順位付け」を強制するため、情報設計の質が上がる
  - デスクトップファーストは既存の業務アプリの移行時に自然だが、モバイル対応が後回しになりやすい
  - CSSの記述順序がパフォーマンスに影響（モバイルファーストではベースCSSがモバイル向けのため軽量）
- **2025-2026年のトレンド**:
  - モバイルファーストが業界標準として完全に定着
  - Container Queriesの実用化により、「ビューポートベース」から「コンテナベース」のレスポンシブへ移行しつつある
  - Fluid Typography（clamp関数）の活用でブレイクポイント依存を削減する手法が普及

### 2. ブレイクポイント設計

- **何を決めるか**: レイアウトの切替ポイントの数値と、設計哲学
- **選択肢**:
  - **標準ブレイクポイント（Tailwind CSS準拠）**:
    - sm: 640px（大きめのスマートフォン）
    - md: 768px（タブレット縦）
    - lg: 1024px（タブレット横/小型デスクトップ）
    - xl: 1280px（デスクトップ）
    - 2xl: 1536px（大型デスクトップ）
  - **最小ブレイクポイント**: 480px / 768px / 1024px の3段階で運用
  - **コンテンツ駆動型**: レイアウトが崩れるポイントをブレイクポイントとする。デバイス幅に依存しない
- **選定基準**:
  - 採用するCSSフレームワーク（Tailwind CSSならそのブレイクポイントに従うのが自然）
  - レイアウトのバリエーション数
  - テスト・QAの負荷（ブレイクポイントが多いほどテストケース増加）
- **トレードオフ・注意点**:
  - デバイス固定のブレイクポイントは将来のデバイス変化に脆弱。コンテンツが崩れるポイントで設定するのが原則
  - 3〜4段階のブレイクポイントが実用上の最適解。過剰なブレイクポイントは保守コスト増
  - Tailwind CSS利用時はカスタムブレイクポイントの追加も可能だが、標準値で大半のケースに対応
  - Container Queriesとの併用で、コンポーネントレベルのレスポンシブが可能に
- **2025-2026年のトレンド**:
  - Container Queries（@container）が全主要ブラウザでサポートされ実用段階に
  - Tailwind CSS v4がContainer Queriesをネイティブサポート
  - 「ビューポートブレイクポイントは全体レイアウト用、Container Queriesはコンポーネント用」の二層戦略が推奨
  - Fluid Design（clamp, min, max関数）の活用でブレイクポイント自体を減らすアプローチ

### 3. PWA（Progressive Web App）の採否

- **何を決めるか**: PWAとして実装するか。する場合、どの機能を含めるか
- **選択肢**:
  - **フルPWA**: Service Worker + マニフェスト + オフライン対応 + インストール可能 + プッシュ通知
  - **部分PWA**: マニフェスト + 基本的なキャッシュ。インストール可能だがオフライン対応は限定的
  - **不採用**: 通常のWebアプリとして配信。ブラウザのみで利用
- **選定基準**:
  - モバイルでの利用頻度（高頻度ならインストール性が重要）
  - オフライン利用の必要性
  - プッシュ通知の必要性
  - 開発・保守コストの許容範囲
  - App Store配信の必要性（PWAはストア不要だが、TWA経由でGoogle Playも可能）
- **トレードオフ・注意点**:
  - PWAはネイティブアプリの68%低い開発コストで同等のパフォーマンスを実現可能
  - iOSのPWAサポートは改善傾向だが、プッシュ通知（iOS 16.4+）やバックグラウンド処理に制約あり
  - HTTPS必須（開発時はlocalhost例外）
  - PWAの最小要件: マニフェスト + Service Worker + HTTPS
  - アプリストアからの流入は得られないため、Webからの発見性を確保する必要がある
- **2025-2026年のトレンド**:
  - PWAが「実験的技術」から「本番標準」に成熟。主要ブラウザのコアPWA API完全サポート
  - ハイブリッド戦略（PWA + 必要に応じてネイティブモジュール追加）が増加
  - Workbox 7がVite/webpack/Next.jsとネイティブ統合
  - Project Fugu APIの拡充により、PWAでアクセスできるデバイス機能が大幅拡大

### 4. Service Worker設定

- **何を決めるか**: Service Workerの実装方法とキャッシュ戦略
- **選択肢**:
  - **Workbox（推奨）**: Google製。宣言的なキャッシュ戦略設定。プリキャッシュ・ランタイムキャッシュ対応。v7でVite/webpack統合
  - **next-pwa / @serwist/next**: Next.js向けWorkboxラッパー。設定ベースで簡易導入
  - **手動実装**: fetch イベントリスナーを直接記述。完全な制御だが保守コスト高
- **選定基準**:
  - フレームワークとの統合（Next.js → @serwist/next、Vite → vite-plugin-pwa）
  - キャッシュ戦略の複雑さ
  - Service Workerの更新・バージョン管理の要件
- **トレードオフ・注意点**:
  - キャッシュ戦略の使い分け:
    - **Cache First**: 静的アセット（JS/CSS/画像）。変更頻度が低いリソース
    - **Network First**: API応答、動的コンテンツ。最新データ優先
    - **Stale While Revalidate**: セミ静的コンテンツ。即座に表示しつつバックグラウンド更新
  - Service Workerの更新タイミング制御を誤ると、古いキャッシュが残り続けるバグの原因に
  - キャッシュサイズの上限設定（maxEntries, maxAgeSeconds）を忘れないこと
  - デバッグはChrome DevTools > Application > Service Workers で確認
- **2025-2026年のトレンド**:
  - Workbox 7がビルドツールとのネイティブ統合を強化
  - Background Sync API / Periodic Background Sync APIの活用が増加
  - Navigation Preloadによるナビゲーションパフォーマンス改善
  - Service Workerのライフサイクル管理ツール（skipWaiting + clients.claim パターン）の標準化

### 5. マニフェスト設定

- **何を決めるか**: Web App Manifest（manifest.json / manifest.webmanifest）の設定項目
- **選択肢**:
  - **最小構成**: name, short_name, icons, start_url, display, background_color, theme_color
  - **フル構成**: 上記 + description, categories, screenshots, shortcuts, share_target, file_handlers, protocol_handlers
- **選定基準**:
  - インストール体験の品質要件
  - OS統合機能の必要性（共有ターゲット、ファイルハンドラー等）
  - ストア公開の予定（TWA向けにはscreenshots等が重要）
- **トレードオフ・注意点**:
  - display モードの選択:
    - **standalone**: 最も一般的。ブラウザUIなし、ステータスバーあり
    - **fullscreen**: ゲーム・没入型コンテンツ向け
    - **minimal-ui**: ナビゲーション最小限で表示
    - **browser**: 通常のブラウザタブ
  - アイコンは複数サイズ必須（192x192, 512x512が最低限。maskableアイコンも推奨）
  - scope設定でPWAのURL範囲を制限可能
  - マニフェスト変更後、Androidでは再インストールが必要な場合がある（キャッシュの影響）
- **2025-2026年のトレンド**:
  - display_override で複数のdisplayモードをフォールバック指定可能に
  - shortcuts（ホーム画面長押しメニュー）の活用が増加
  - launch_handler でPWAの起動動作を制御（既存ウィンドウへのフォーカス等）
  - ブラウザのRicher Install UIにより、descriptionsとscreenshotsの重要性が増加

### 6. インストールプロンプト

- **何を決めるか**: PWAのインストール（ホーム画面追加）を促すUIの方針
- **選択肢**:
  - **カスタムプロンプト**: beforeinstallprompt イベントをキャプチャし、アプリ内の適切なタイミングでUI表示
  - **ブラウザデフォルト**: ブラウザが自動表示するインストールバナーに依存
  - **非表示**: インストールを積極的に促さない。設定画面等にインストール方法のガイドを配置
- **選定基準**:
  - ユーザーのリピート率向上の重要度
  - UXへの侵入度合いの許容範囲
  - iOSの制約（beforeinstallpromptイベント非対応、手動で「ホーム画面に追加」が必要）
- **トレードオフ・注意点**:
  - 初回訪問でのインストールプロンプトはUXを損なう。ユーザーが価値を理解した後（2-3回目の訪問等）に表示するのが効果的
  - beforeinstallpromptは一度dismissされると再表示に制約がある
  - iOS向けにはカスタムUIで「Safari > 共有 > ホーム画面に追加」の手順をガイドする必要がある
  - インストール率のトラッキング（appinstalled イベント）を実装して効果測定
- **2025-2026年のトレンド**:
  - 「コンテキスト型プロンプト」が推奨。ユーザーが特定のアクション後に自然な形で表示
  - getInstalledRelatedApps() APIでネイティブアプリとの重複インストールを検出可能に
  - ブラウザのRicher Install UIの改善により、アプリ説明・スクリーンショット表示が充実

### 7. プッシュ通知の採否

- **何を決めるか**: Web Push通知を実装するか。する場合のタイミングと対象
- **選択肢**:
  - **Web Push実装**: Push API + Notifications API。バックエンドにプッシュサーバー必要
  - **不採用**: メール通知やアプリ内通知で代替
  - **段階的導入**: 初期リリースでは不採用、ユーザーフィードバック後に導入検討
- **選定基準**:
  - リアルタイム通知の必要性（タスク期限、メッセージ受信等）
  - ユーザーのプッシュ通知への許容度（過度な通知は離脱の原因）
  - バックエンドインフラの複雑さ許容度
  - iOSでのWebプッシュ対応状況（iOS 16.4+、ホーム画面追加が前提）
- **トレードオフ・注意点**:
  - 通知許可のタイミングが重要。ページ表示直後の許可リクエストは高確率で拒否される
  - VAPID（Voluntary Application Server Identification）キーの管理が必要
  - プッシュサービス（FCM / APNs Web Push）のセットアップが必要
  - 通知の頻度・内容の制御が不適切だとユーザー離脱を招く
  - Service Worker内でpushイベントを処理する実装が必要
- **2025-2026年のトレンド**:
  - iOS/SafariのWeb Push対応が安定し、クロスプラットフォーム通知が現実的に
  - Notification Triggers API（スケジュール通知）が一部ブラウザで対応
  - ユーザーの「通知疲れ」への配慮が設計原則として重視される
  - サーバーレス環境（Vercel、Cloudflare Workers）でのプッシュ通知配信パターンが成熟

### 8. Android共有機能連携（Share Target API）

- **何を決めるか**: AndroidのOS共有メニューからWebアプリにデータを受け取る機能を実装するか
- **選択肢**:
  - **Share Target API実装**: manifest.jsonのshare_target設定 + サーバーサイドのPOSTハンドラー
  - **Web Share API（発信側のみ）**: アプリからOS共有メニューを呼び出す機能のみ実装
  - **不採用**: ブラウザのコピー&ペーストで対応
- **選定基準**:
  - Androidユーザーの比率とモバイルワークフローの重要度
  - 共有されるデータの種類（テキスト、URL、画像、ファイル）
  - PWAとしてインストールされることが前提か
- **トレードオフ・注意点**:
  - Share Target APIはPWAがインストール済みの場合のみ動作
  - Androidでは url フィールドが空で、URLが text フィールドに入る場合がある。サーバー側でのパース処理が必要
  - マニフェストのshare_target.actionには絶対URLを指定すること（相対URLだとAndroidで動作しない場合がある）
  - マニフェスト変更後はPWAの再インストールが必要（Androidのキャッシュが更新されないため）
  - ファイル共有にはPOSTメソッド + multipart/form-data の enctype 設定が必要
- **2025-2026年のトレンド**:
  - Share Target APIが安定し、Chrome on Androidでの実装実績が蓄積
  - Web Share API Level 2（ファイル共有対応）が標準化
  - Open WebUI等のAIアプリでShare Target実装の事例が増加
  - iOS/Safariでの対応は引き続き限定的

### 9. ネイティブアプリとの比較検討

- **何を決めるか**: Web技術（PWA）で十分か、ネイティブ/ハイブリッドアプリが必要か
- **選択肢**:
  - **PWA**: Web技術のみ。ストア不要。開発コスト最小。デバイスAPI制約あり
  - **React Native**: ネイティブUI。JSバンドル。iOS/Android同時対応。学習コストあり
  - **Flutter**: Dart言語。独自レンダリングエンジン。高パフォーマンス。Webと技術スタック分断
  - **Capacitor**: Web技術をネイティブシェルでラップ。既存Webアプリの移植に最適
  - **Tauri Mobile**: Rust + WebView。軽量バイナリ。モバイル対応は発展途上
- **選定基準**:
  - 必要なデバイスAPI（カメラ、Bluetooth、NFC、バイオメトリクス等）
  - App Store/Google Play配信の必要性
  - パフォーマンス要件（60fps アニメーション等）
  - 開発チームのスキルセット
  - 開発・保守コストの許容範囲
- **トレードオフ・注意点**:
  - PWAが対応できないネイティブ機能: Bluetooth、NFC、高度なカメラ制御、バックグラウンド位置情報、ヘルスキット連携等
  - PWAでカバーできる機能が年々拡大（Project Fugu）。2025年時点で大半のユースケースに対応
  - Capacitorは既存のWeb技術資産を最大限活用できるが、ネイティブブリッジのパフォーマンスオーバーヘッドがある
  - React Nativeは「Learn Once, Write Anywhere」の思想だが、プラットフォーム固有のコードが必要な場面もある
  - ネイティブアプリはストアの審査プロセスがリリースのボトルネックになる
- **2025-2026年のトレンド**:
  - PWA + 必要時ネイティブ拡張（Capacitor）のハイブリッド戦略が主流
  - React Native New Architecture（Fabric + TurboModules）が安定し、パフォーマンスが大幅向上
  - Expo（React Native）がWeb対応を強化し、「一つのコードベースで全プラットフォーム」に近づく
  - Tauri Mobileが正式リリースに向けて開発進行中

### 10. タッチ操作とマウス操作の両立

- **何を決めるか**: タッチデバイスとマウスデバイスの両方で快適に操作できるインタラクション設計
- **選択肢**:
  - **ポインターイベント統一（推奨）**: Pointer Events API（pointerdown, pointermove, pointerup）でタッチ/マウス/ペンを統一処理
  - **デバイス検出分岐**: User-Agent / メディアクエリ（hover, pointer）で分岐。デバイス固有の最適化
  - **タッチ優先設計**: タッチ操作を基本とし、マウスは追加的に対応
- **選定基準**:
  - 対象デバイスの範囲（モバイルのみ / デスクトップのみ / 両対応）
  - インタラクションの複雑さ（ドラッグ&ドロップ、ジェスチャー等）
  - テスト・QAの負荷
- **トレードオフ・注意点**:
  - タッチターゲットの最小サイズ: 44x44px（Apple HIG）/ 48x48dp（Material Design）/ 24x24px（WCAG 2.2最低要件）
  - hover状態はタッチデバイスで不安定。@media (hover: hover) で分岐し、hover効果はマウスデバイスのみに適用
  - ドラッグ&ドロップはタッチでの実装が複雑。dnd-kit等のライブラリがタッチ/マウス両対応
  - 300msタップ遅延はモダンブラウザでは解消済み（touch-action: manipulationで明示的に除去）
  - テキスト選択とスワイプジェスチャーの競合に注意
- **2025-2026年のトレンド**:
  - Pointer Events APIが全ブラウザで安定。タッチ/マウス統一の標準手法に
  - CSS media features（hover, pointer, any-hover, any-pointer）の活用が一般的に
  - スワイプジェスチャー（戻る操作等）とアプリのジェスチャーの競合問題への対処が重要テーマ
  - Haptic Feedback API（バイブレーション）のPWA対応が進行中

### 11. オフライン対応の方針

- **何を決めるか**: ネットワーク接続がない/不安定な場合のアプリの振る舞い
- **選択肢**:
  - **オフラインファースト**: ローカルデータを正として操作し、オンライン時にサーバーと同期。IndexedDB + Service Worker
  - **オンライン前提 + グレースフルデグラデーション（推奨多数）**: 基本はオンラインだが、一時的なオフラインでもクラッシュしない。キャッシュデータの読み取りは可能
  - **オンライン専用**: オフライン時はエラー表示。最もシンプル
- **選定基準**:
  - ユーザーの利用環境（安定したWi-Fi主体 vs 不安定なモバイル回線）
  - データの同期要件（コンフリクト解決の複雑さ）
  - 実装コストの許容範囲
  - アプリの性質（読み取り主体 vs 書き込み主体）
- **トレードオフ・注意点**:
  - オフラインファーストはコンフリクト解決（同一データの同時編集）が最大の課題
  - ストレージの使い分け:
    - **Cache Storage**: HTTP応答（HTML/CSS/JS/画像）のキャッシュ。Service Workerから利用
    - **IndexedDB**: 構造化データ（ユーザーデータ、アプリ状態）の保存。検索・インデックス対応
    - **localStorage**: 小さなキーバリューデータ。同期APIのためメインスレッドブロックに注意
  - オンライン前提でも、最低限のオフラインページ（offline.html）の提供は推奨
  - Background Sync APIで、オフライン時の操作をキューに入れ、オンライン復帰時に自動送信
- **2025-2026年のトレンド**:
  - SQLite in the Browser（OPFS経由のwa-sqlite等）が実用段階に。IndexedDBの代替として注目
  - CRDTベースの同期エンジン（Yjs、Automerge）がオフラインファーストのコンフリクト解決に
  - Service Workerのバックグラウンド同期がクロスブラウザで安定
  - 「オフライン対応」は信頼性の一要素として、ネットワーク不安定環境への耐性として位置づけ

### 12. ビューポート・Safe Area対応

- **何を決めるか**: ノッチ、ダイナミックアイランド、ソフトウェアキーボード等のデバイス固有の表示領域への対処
- **選択肢**:
  - **viewport-fit=cover + CSS env()**: `<meta name="viewport" content="viewport-fit=cover">` でコンテンツを画面全体に拡張し、env(safe-area-inset-*)でSafe Area内にコンテンツを配置
  - **viewport-fit=auto（デフォルト）**: ブラウザが自動的にSafe Area内にコンテンツを制限。フルスクリーン体験は得られない
  - **固定レイアウト**: 固定のヘッダー/フッターで物理的にSafe Areaを確保
- **選定基準**:
  - PWAとしてのフルスクリーン体験の重要度
  - 対象デバイスの多様性（iPhoneのノッチ/ダイナミックアイランド対応の必要性）
  - 下部ナビゲーションバーの有無（iOSのホームインジケーターとの重なり防止）
- **トレードオフ・注意点**:
  - env(safe-area-inset-bottom) は特にPWA standaloneモードで重要。下部の固定UIがホームインジケーターと重なる
  - ソフトウェアキーボード表示時のビューポート変更への対応:
    - visualViewport APIで実際の可視領域を取得
    - interactive-widget=resizes-content / resizes-visual でキーボード時の挙動を制御
  - iPhone横向き時のノッチ/ダイナミックアイランド対応
  - Android の cutout（パンチホール）対応も env() で統一的に処理可能
- **2025-2026年のトレンド**:
  - env(safe-area-inset-*) が全ブラウザで安定。PWA設計の必須要素に
  - Virtual Keyboard API（navigator.virtualKeyboard）でキーボードの表示制御が可能に
  - dvh (dynamic viewport height) 単位の活用により、モバイルブラウザのアドレスバー表示/非表示に対応
  - foldable device（折りたたみスマホ）への対応が新しいテーマに（Viewport Segments API）
