## 各項目の詳細

### 1. テストピラミッド/テストトロフィーの選択

- **何を決めるか**: テスト全体の投資配分モデル。ユニット・結合・E2Eの比率と重点をどこに置くか
- **選択肢**:
  - **テストピラミッド（Martin Fowler）**: ユニット（多） > 結合（中） > E2E（少）。低コストなユニットテストを基盤に、上位レベルは最小限に抑える
  - **テストトロフィー（Kent C. Dodds）**: 静的解析 > ユニット（少） > 結合（多） > E2E（少）。結合テストに最も投資し、ユーザーの実際の使い方に近いテストを重視する
  - **テストダイヤモンド**: ユニット（少） > 結合（多） > E2E（中）。バックエンドAPI中心のアプリに適用
  - **ハイブリッド**: フロントエンドはトロフィー、バックエンドはピラミッドのように領域ごとに使い分け
- **選定基準**:
  - フロントエンド中心: トロフィーが適合（コンポーネント結合テストの投資対効果が高い）
  - バックエンドAPI中心: ピラミッドが適合（ビジネスロジックのユニットテストが効果的）
  - フルスタック: ハイブリッドアプローチが現実的
- **トレードオフ・注意点**:
  - ピラミッドはバグ発見コストが最小だが、実際のユーザー操作との乖離が大きい
  - トロフィーは信頼性が高いが、結合テストの実行時間・セットアップコストが増加する
  - どちらを選んでもモデルは指針であり、プロジェクト特性に応じた調整が必要
- **2025-2026年のトレンド**: テストトロフィーの影響が拡大しつつも、テストピラミッドの経済原則（バグ発見コストは上位レベルほど指数的に増加する）は依然として有効とされる。実務では、フロントエンドにトロフィー、バックエンドにピラミッドを適用するハイブリッドアプローチが主流になりつつある

### 2. テストレベル別の方針（ユニットテスト、結合テスト、E2Eテスト）

- **何を決めるか**: 各テストレベルの責務・対象範囲・作成基準
- **選択肢**:
  - **ユニットテスト**: 純粋関数・ビジネスロジック・ユーティリティ関数を対象。外部依存なし
  - **結合テスト**: コンポーネント間連携・API-DB連携・フック+コンポーネント連携を対象
  - **E2Eテスト**: ユーザーの主要操作フロー（ログイン、購入、データ登録等）を対象
- **選定基準**:
  - ユニットテスト: ビジネスロジック・バリデーション・データ変換・状態管理のロジック層
  - 結合テスト: コンポーネントのユーザーインタラクション・APIエンドポイント・データフロー
  - E2Eテスト: クリティカルパス（収益に直結するフロー）に限定
- **トレードオフ・注意点**:
  - ユニットテストだけでは実際のバグを見逃す（結合部分のバグが多い）
  - E2Eテストは壊れやすく、保守コストが高い。数を絞ってクリティカルパスに集中する
  - 結合テストは範囲の定義が曖昧になりやすい。テスト対象の境界を明文化する
- **2025-2026年のトレンド**: Testing Libraryの普及により、フロントエンドの結合テスト（コンポーネントテスト）の書きやすさが向上。「実装の詳細ではなく、ユーザーの振る舞いをテストする」というアプローチが標準化

### 3. テストフレームワーク選定

- **何を決めるか**: ユニット/結合テスト、E2Eテスト、コンポーネントテストで使用するフレームワーク
- **選択肢**:
  - **ユニット/結合テスト**: Vitest / Jest
  - **E2Eテスト**: Playwright / Cypress
  - **コンポーネントテスト**: Testing Library（React Testing Library 等）
- **選定基準**:

  | 観点 | Vitest | Jest |
  |------|--------|------|
  | 速度 | Viteベース、ESMネイティブ、コールドラン最大4倍高速 | やや遅い（特にESMプロジェクト） |
  | 設定 | Viteプロジェクトならゼロコンフィグ | 設定が必要な場合あり（ESM、TypeScript） |
  | エコシステム | 急速に成長中、Vue/Svelte公式採用 | 成熟、情報量が最も多い |
  | 互換性 | Jest互換API、移行容易 | React Nativeで優位 |
  | 推奨 | 新規プロジェクト全般 | レガシー、React Native |

  | 観点 | Playwright | Cypress |
  |------|-----------|---------|
  | ブラウザ対応 | Chrome, Firefox, Safari(WebKit) | Chromium, Firefox のみ |
  | 並列実行 | ネイティブ対応、無料 | 有料（Cypress Cloud）or 外部ツール |
  | 言語 | JS/TS, Python, C#, Java | JS/TSのみ |
  | 速度 | ヘッドレス35-45%高速（並列時） | シリアル実行がデフォルト |
  | DX | CLI中心、強力なコード生成 | GUI優秀、初心者向け |
  | 推奨 | マルチブラウザ、大規模、CI重視 | 小規模、チームの学習コスト重視 |

- **トレードオフ・注意点**:
  - Vitestは新しいため、一部のエッジケースでJestより情報が少ない
  - Playwrightは高機能だが、Cypressに比べてGUIデバッグ体験がやや劣る
  - Testing Libraryはフレームワーク非依存だが、React/Vue/Svelte用の個別パッケージが必要
- **2025-2026年のトレンド**: 新規プロジェクトではVitestがデファクトスタンダードになりつつある。E2EテストではPlaywrightがCypressを逆転し、マルチブラウザ対応と無料の並列実行で優位。Testing Libraryは「実装ではなくユーザー行動をテストする」哲学と共にフロントエンドの標準として定着

### 4. テストカバレッジの目標値と測定方法

- **何を決めるか**: カバレッジの目標値、測定対象、品質ゲートとしての使い方
- **選択肢**:
  - **全体目標**: 60%（最低ライン） / 75-80%（推奨） / 90%（高品質）
  - **レベル別目標**: ユニット90% / 結合80% / E2E（カバレッジ対象外）
  - **新規コード基準**: 新規コードは80%以上を必須とし、既存コードは段階的に向上
  - **メトリクス種別**: 行カバレッジ / ブランチカバレッジ / 関数カバレッジ
- **選定基準**:
  - Google基準: 60%が許容、75%が良好、90%が優秀
  - 業界コンセンサス: 75-80%が現実的かつ効果的
  - 重要なのは数値よりクリティカルパスのカバー
  - ブランチカバレッジが行カバレッジより実質的な品質指標
- **トレードオフ・注意点**:
  - 100%を追求すると、テストの質が下がりやすい（数値のためのテストになる）
  - 高カバレッジでも、アサーションが弱ければバグは見逃す
  - カバレッジが低いコードは「テスト不足」、高いコードでも「テスト十分」とは限らない
  - CIの品質ゲートとして使う場合、新規コードのみに適用するのが現実的
- **2025-2026年のトレンド**: カバレッジ数値よりも「ミューテーションテスト」による実質的なテスト品質の測定が注目されている。また、新規コードに対するカバレッジ閾値（80%）をCI品質ゲートに組み込むアプローチが広く採用されている

### 5. テストデータ管理（フィクスチャ、ファクトリー、シード）

- **何を決めるか**: テストで使用するデータの生成・管理方法
- **選択肢**:
  - **フィクスチャ（Fixture）**: 静的なJSONファイルやオブジェクトとして事前定義。シンプルだが柔軟性が低い
  - **ファクトリー（Factory）**: Fishery, Factory.tsなどのライブラリでデータを動的生成。柔軟でテスト間の独立性が高い
  - **シード（Seed）**: DB初期データ投入スクリプト。開発環境セットアップ、E2Eテストの前提データに使用
  - **Builder パターン**: テストごとに必要な属性だけ上書きする、ファクトリーの発展形
- **選定基準**:
  - ユニットテスト: ファクトリー（テストごとに独立データ生成）
  - 結合テスト: ファクトリー + シード（DB状態の初期化が必要）
  - E2Eテスト: シード（一貫した初期状態）+ APIベースのセットアップ
  - 開発環境: シードスクリプト（`prisma db seed` 等）
- **トレードオフ・注意点**:
  - フィクスチャは理解しやすいが、テスト間の暗黙的な依存が生まれやすい
  - ファクトリーは柔軟だが、複雑なリレーションの生成は学習コストがかかる
  - シードデータは環境構築を容易にするが、テストの前提条件が暗黙的になる
  - テストデータはテストコードと同じリポジトリで管理し、バージョン管理する
- **2025-2026年のトレンド**: ファクトリーパターンが主流。TypeScriptプロジェクトではPrismaのスキーマから型安全なファクトリーを自動生成するアプローチも登場。E2Eテストでは、テストごとにAPIを通じてデータをセットアップし、テスト後にクリーンアップする「APIベースのテストデータ管理」が推奨される

### 6. モック戦略（外部API、DB、時間のモック）

- **何を決めるか**: 外部依存のモック範囲・方法・レイヤー
- **選択肢**:
  - **ネットワークレイヤーモック（MSW）**: Service Worker/Node.jsインターセプトで実際のHTTPリクエストをモック。クライアント実装に非依存
  - **モジュールモック（jest.mock / vi.mock）**: 特定モジュールの関数をモック。単体テスト向け
  - **テストコンテナ（Testcontainers）**: 実際のDB/Redis/外部サービスをDockerコンテナで起動。結合テスト向け
  - **時間モック（vi.useFakeTimers / jest.useFakeTimers）**: Date, setTimeout等を制御
  - **インメモリDB**: SQLite等でDBをモック。軽量だが本番DBとの差異リスク
- **選定基準**:
  - 外部API: MSW（ネットワークレイヤー）を第一選択。フロントエンド・バックエンド・Storybook・E2Eで再利用可能
  - DB: ユニットテストではリポジトリ層をモック、結合テストではTestcontainersで実DB使用
  - 時間: vi.useFakeTimersで制御（タイムアウト、日付依存ロジック）
  - 外部サービス: 開発環境はMSW、CI結合テストはTestcontainers
- **トレードオフ・注意点**:
  - モックが多すぎると実際の動作との乖離が広がる（偽の安心感）
  - MSWはネットワークレイヤーで動くため、アプリケーションコードに一切影響しない
  - Testcontainersは高い信頼性だが、CI実行時間が増加する
  - モックの保守コスト: API仕様変更時にモックデータの更新が必要
- **2025-2026年のトレンド**: MSW v2が成熟し、外部APIモックの業界標準として定着。開発・テスト・Storybookでモック定義を一元化する「Single Source of Truth」アプローチが推奨。Testcontainersもバックエンドの結合テストで広く採用されている

### 7. スナップショットテストの採否

- **何を決めるか**: スナップショットテストを導入するか、導入する場合の適用範囲
- **選択肢**:
  - **採用（広範囲）**: 全コンポーネントのレンダリング結果をスナップショット
  - **限定採用**: API応答スキーマ、設定ファイル、特定のシリアライズ可能な出力に限定
  - **不採用**: 明示的なアサーションのみ使用
- **選定基準**:
  - UIコンポーネントの変更頻度が低い場合は有効
  - 大規模・動的コンテンツが多い場合は不向き
  - レガシーコード引き継ぎ時のリグレッション検出に有効
  - チームがスナップショットの差分レビューを確実に行える体制があるか
- **トレードオフ・注意点**:
  - 脆い（fragile）: UIの小さな変更でも大量のスナップショット更新が発生
  - 意味のないテスト: 「変わっていないこと」を確認するだけで、「正しいこと」を確認しない
  - 巨大なスナップショットファイルはレビュー不能になり、盲目的に`-u`で更新される
  - 導入するなら小さく焦点を絞ったスナップショットにすること
- **2025-2026年のトレンド**: UIコンポーネントに対するスナップショットテストは減少傾向。代わりにTesting Libraryによる行動ベースのテストやビジュアルリグレッションテストが推奨される。ただし、APIレスポンスの構造やCLI出力など、非UIのシリアライズ可能な出力には引き続き有効

### 8. ビジュアルリグレッションテストの採否

- **何を決めるか**: ビジュアルリグレッションテストを導入するか、導入する場合のツールと対象範囲
- **選択肢**:
  - **Chromatic**: Storybook連動、コンポーネントレベル。デザインシステム・UIライブラリに最適。無制限並列実行
  - **Percy（BrowserStack）**: ページレベル、マルチブラウザ/デバイス対応。AIによる偽陽性フィルタリング（40%削減）
  - **Playwright Screenshots**: 無料、CI組み込み。ただし比較UIは自前で構築が必要
  - **不採用**: コスト・保守負荷を考慮して見送り
- **選定基準**:
  - デザインシステム/UIライブラリを持つ: Chromaticが最適
  - ページ全体・マルチブラウザ検証: Percyが最適
  - コストを抑えたい: Playwrightスクリーンショット比較
  - UIが少ない/変更頻度が低い: 不採用も妥当
- **トレードオフ・注意点**:
  - SaaSツール（Chromatic, Percy）は有料。無料枠の範囲を確認する
  - 偽陽性（フォントレンダリング差異、アンチエイリアシング等）への対応が必要
  - Storybook未使用のプロジェクトではChromaticの恩恵が薄い
  - テスト対象をクリティカルなUIコンポーネントに絞ることで費用対効果を上げる
- **2025-2026年のトレンド**: Percy が2025年後半にAI搭載のVisual Review Agentをリリースし、偽陽性を40%削減。ChromaticはStorybook公式チームによる開発で信頼性が高い。コスト許容できるなら「Chromatic（コンポーネント）+ Percy（ページ）」の併用が最強の組み合わせとされている

### 9. アクセシビリティテストの方針

- **何を決めるか**: アクセシビリティテストの自動化範囲、適合レベル、実施方法
- **選択肢**:
  - **自動テスト（axe-core）**: eslint-plugin-jsx-a11y（静的解析） + @axe-core/react（開発時） + axe-playwright/cypress-axe（E2Eテスト）
  - **手動テスト**: スクリーンリーダー（VoiceOver, NVDA）、キーボードナビゲーション
  - **併用（推奨）**: 自動テストをCIに組み込み + 定期的な手動テスト
- **選定基準**:
  - 自動テストでカバーできるのはWCAG基準の30-40%のみ
  - 法的要件（ADA、EU指令等）がある場合は手動テスト必須
  - 適合レベル: WCAG 2.1 AA が標準的な目標
  - 重要ページ（ホーム、ログイン、決済）はCIで毎回テスト
- **トレードオフ・注意点**:
  - 自動テストのパスは「アクセシブル」の証明にならない（偽の安心感に注意）
  - 手動テストは工数がかかるが、最もインパクトのある問題を発見できる
  - アクセシビリティの専門知識がチームにない場合、外部監査の活用も検討
  - 段階的導入: まず自動テストから始め、手動テストを徐々に追加
- **2025-2026年のトレンド**: axe-coreが事実上の業界標準。Playwright + axe-coreの組み合わせがCIでのアクセシビリティテストの推奨パターン。EUアクセシビリティ法（European Accessibility Act, 2025年6月施行）の影響で、欧州市場向けサービスではWCAG 2.1 AA準拠が実質必須に

### 10. パフォーマンステストの方針

- **何を決めるか**: フロントエンド/バックエンドのパフォーマンステスト手法とツール
- **選択肢**:
  - **フロントエンド**: Lighthouse CI（Web Vitals測定、CIゲート） / k6 browser（実ブラウザ負荷テスト）
  - **バックエンド**: k6（API負荷テスト） / Artillery（YAML定義の負荷テスト）
  - **統合**: k6（フロントエンド + バックエンド統合テスト）
- **選定基準**:
  - Lighthouse CI: PR単位でWeb Vitals（LCP, CLS, INP）の回帰検出。設定容易、無料
  - k6: API負荷テスト + ブラウザベースのパフォーマンステスト。スクリプトはJavaScript
  - 開発初期はLighthouse CIのみで十分。負荷テストはリリース前に追加
  - パフォーマンスバジェットの設定: LCP < 2.5s, CLS < 0.1, INP < 200ms
- **トレードオフ・注意点**:
  - Lighthouse CIは合成テスト（Synthetic）であり、実ユーザーのパフォーマンスとは異なる
  - k6の負荷テストは本番に近い環境で実施しないと意味が薄い
  - パフォーマンスバジェットを厳しくしすぎるとCIが頻繁に失敗する
  - RUM（Real User Monitoring）との併用で、合成テストの限界を補完する
- **2025-2026年のトレンド**: k6のブラウザモジュールによりフロントエンド+バックエンドの統合パフォーマンステストが1ツールで実現可能に。Core Web Vitalsの指標がINP（Interaction to Next Paint）に更新され、インタラクティビティの測定が重視されている

### 11. テストの命名規則・構造

- **何を決めるか**: テストファイルの配置、テストケースの命名規則、テストコードの構造
- **選択肢**:
  - **命名パターン**:
    - `describe("ComponentName") > it("should do X when Y")` : 最も一般的
    - `describe("ComponentName") > describe("when X") > it("does Y")` : ネストによるコンテキスト分離
    - `test("ComponentName: does X when Y")` : フラット構造
  - **テスト内部構造**:
    - **AAA（Arrange-Act-Assert）**: テストコードを準備・実行・検証に分離
    - **GWT（Given-When-Then）**: BDDスタイル、ビジネス仕様寄りの記述
  - **ファイル配置**:
    - コロケーション: `src/components/Button/Button.test.tsx`（ソースコードと同じディレクトリ）
    - 分離: `__tests__/components/Button.test.tsx`（テスト専用ディレクトリ）
- **選定基準**:
  - AAA: 技術寄りのユニットテストに適合。シンプルで汎用的
  - GWT: ビジネスロジックのテスト、E2Eテストに適合。非エンジニアにも読みやすい
  - コロケーション: テスト対象との距離が近く、テスト漏れに気づきやすい
  - チーム内でどちらかに統一し、ESLintルールで強制する
- **トレードオフ・注意点**:
  - ネストが深すぎると可読性が下がる（最大2-3レベル）
  - 命名で実装の詳細を記述しない（「calls setState」ではなく「updates the display name」）
  - テストファイル名はソースファイル名と一致させる（`Button.tsx` -> `Button.test.tsx`）
- **2025-2026年のトレンド**: コロケーション（テストファイルをソースコードと同じ場所に置く）が主流。AAA構造が最も広く使われているが、AAAとGWTは実質同じ構造であり、チームの好みで選択。ESLintの`testing-library/*`ルールによるテスト品質の自動チェックが普及

### 12. テスト実行の高速化手法（並列実行、テスト分割）

- **何を決めるか**: テスト実行の高速化手法と適用方針
- **選択肢**:
  - **フレームワーク内並列実行**: Vitest（ワーカースレッド並列）/ Jest（`--workers`）
  - **CIシャーディング**: GitHub Actions matrix strategyでテストを分割実行。Jest `--shard 1/4`, Playwright `--shard 1/4`
  - **テスト分割ツール**: Knapsack Pro等でテスト実行時間に基づく最適分割
  - **差分実行**: 変更ファイルに関連するテストのみ実行（`--changed` / `--related`）
  - **キャッシュ活用**: Vitestのファイル変更検知キャッシュ、CIのテスト結果キャッシュ
- **選定基準**:
  - テストスイート < 5分: フレームワーク内並列実行で十分
  - テストスイート 5-15分: CIシャーディング（2-4分割）を導入
  - テストスイート > 15分: シャーディング + 差分実行 + キャッシュの組み合わせ
  - E2Eテスト: Playwrightのシャーディングで並列化（最も効果大）
- **トレードオフ・注意点**:
  - シャーディングは分割が不均等だと効果が薄い（均等な実行時間分割が理想）
  - 差分実行はリグレッションを見逃すリスクがある（定期的な全テスト実行と併用）
  - GitHub Actionsの並列ジョブ数制限: 無料20、Pro 40、Team 60
  - テスト間の状態共有がある場合、並列実行で不安定になる（テスト分離が前提）
- **2025-2026年のトレンド**: Jest `--shard`により19分が3分未満に短縮された事例が報告されている。89%の高パフォーマンスチームが並列テスト実行をCI/CDに不可欠と回答。テスト実行時間の可視化と継続的な最適化が重視されている

### 13. CIでのテスト実行戦略

- **何を決めるか**: PR・マージ・デプロイ各段階でのテスト実行内容と品質ゲート
- **選択肢**:
  - **PR作成時**: 静的解析（lint, type-check） + ユニット/結合テスト + カバレッジレポート
  - **マージ時（main）**: 全テスト実行（ユニット + 結合 + E2E） + カバレッジゲート
  - **デプロイ前**: E2Eスモークテスト（本番環境のクリティカルパス）
  - **定期実行（Nightly）**: 全テスト + パフォーマンステスト + アクセシビリティテスト
  - **段階的実行（Fast-first）**: 高速テスト先行 -> 通過後に低速テスト実行
- **選定基準**:
  - PR時のフィードバックは10分以内が目標（開発者体験を損なわない）
  - E2EテストはPRでは差分関連のみ、mainマージ時に全実行
  - 品質ゲート: カバレッジ閾値（新規コード80%）、テスト全パス、lint/type-checkパス
  - fail-fast: 最初のテスト失敗で早期フィードバック（ただしシャーディング時はfalse）
- **トレードオフ・注意点**:
  - 全テスト毎回実行はCI時間・コスト増。段階的実行で最適化
  - E2EテストをPRで毎回実行すると、フレーキーテストによるブロックが頻発
  - Nightlyテストの失敗は翌朝対応になるため、クリティカルな問題は検出が遅れる
  - テスト結果のアーカイブ（GitHub Actions Artifacts）で失敗分析を容易にする
- **2025-2026年のトレンド**: GitHub Actionsのmatrix strategyとシャーディングを活用した並列CIが標準。テスト結果の可視化（Allure Report等）やフレーキーテスト検出の自動化が普及。AIを活用したテスト影響分析（変更に影響されるテストの自動特定）が登場し始めている
