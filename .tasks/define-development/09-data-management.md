## 各項目の詳細

### 1. スキーマ設計方針（正規化レベル、ER図の作成）

- **何を決めるか**: データベーススキーマの正規化レベル、設計ドキュメント、命名規約
- **選択肢**:
  - **第3正規形（3NF）**: データ冗長性を最小化。書き込み性能・データ整合性を重視
  - **BCNF（ボイス・コッド正規形）**: 3NFをさらに厳密化。アカデミックに正確だが実用上は3NFで十分な場合が多い
  - **選択的非正規化**: 読み取りが多いテーブルや集計データに対して意図的に非正規化。パフォーマンス最適化
  - **ER図**: Mermaid（コードベース管理可能） / dbdiagram.io / draw.io
  - **命名規約**: snake_case / camelCase、単数形 / 複数形、プレフィックス有無
- **選定基準**:
  - OLTP（トランザクション処理）: 3NFを基本とし、必要に応じて非正規化
  - 読み取り負荷が高い: レポート/ダッシュボード用にマテリアライズドビューや非正規化テーブルを追加
  - チーム規模が小さい: 過度な正規化より開発速度を優先
  - ER図はMermardでコード管理し、スキーマ変更時に同期更新
- **トレードオフ・注意点**:
  - 正規化が不十分: データ不整合、更新異常のリスク
  - 正規化が過度: JOIN増加によるクエリ性能低下、開発の複雑化
  - 非正規化はパフォーマンス改善の「最後の手段」として扱い、先にインデックスやクエリ最適化を検討
  - 命名規約はORM（Prisma/Drizzle）の慣例に合わせると摩擦が少ない
- **2025-2026年のトレンド**: Prisma/Drizzleの普及により、スキーマ定義がTypeScriptコードとして管理されるケースが増加。ERD自動生成ツール（prisma-erd-generator等）でコードからER図を生成する運用が広まっている。JSONBカラムの活用で柔軟なスキーマ（Semi-structured data）と正規化の中間的なアプローチも増加

### 2. マイグレーション戦略（ツール選定、バージョン管理、ロールバック）

- **何を決めるか**: スキーマ変更の管理方法、ツール、ロールバック方針、本番適用フロー
- **選択肢**:
  - **Prisma Migrate**: 宣言的スキーマ定義（.prismaファイル）から差分マイグレーション自動生成。成熟したエコシステム
  - **Drizzle Kit**: TypeScriptスキーマ定義からSQLマイグレーション生成。軽量、SQLに近い
  - **TypeORM Migrations**: コードファーストまたはSQLベースのマイグレーション。NestJSとの統合が強い
  - **手動SQL + バージョン管理**: SQLファイルを手動作成・管理。最大の制御力だが工数が多い
- **選定基準**:

  | 観点 | Prisma Migrate | Drizzle Kit |
  |------|---------------|-------------|
  | アプローチ | 宣言的（スキーマファースト） | コードファースト（TypeScript定義） |
  | SQL制御 | 低い（自動生成、カスタム困難な場合あり） | 高い（生成SQLを直接編集可能） |
  | バンドルサイズ | 大きい（Prisma Engine含む） | 軽量（~7.4kb） |
  | 学習コスト | Prisma独自のスキーマ言語の習得が必要 | SQL/TypeScriptの知識で十分 |
  | エコシステム | 成熟（ドキュメント、ツール豊富） | 急成長中 |
  | サーバーレス | やや不利（コールドスタート） | 有利（軽量） |
  | 推奨用途 | SaaS、チーム開発、迅速なプロトタイピング | パフォーマンス重視、エッジ/サーバーレス |

- **トレードオフ・注意点**:
  - ロールバック: 「downマイグレーション」は実務で使われにくい。代わりに「前方修正（forward-fix）」で新しいマイグレーションを作成するアプローチが安全
  - 破壊的変更（カラム削除、型変更）は段階的に実施: (1) 新カラム追加 -> (2) データ移行 -> (3) アプリ切替 -> (4) 旧カラム削除
  - マイグレーションファイルは一度適用したら編集しない（イミュータブル）
  - 本番適用前にステージング環境で検証。大規模テーブルの変更はダウンタイムに注意
- **2025-2026年のトレンド**: Prismaが依然として最も広く使われているが、DrizzleがTypeScript開発者の間で急成長。サーバーレス/エッジ環境ではDrizzleのバンドルサイズ優位性が評価されている。TypeORMとSequelizeは新規プロジェクトでの採用が減少傾向。Prisma/Drizzle共にスキーマの型安全性がTypeScriptプロジェクトでの選定理由として最も重視されている

### 3. シードデータの管理

- **何を決めるか**: 開発・テスト・ステージング環境の初期データ投入方法と管理方針
- **選択肢**:
  - **スクリプトベース**: `prisma db seed` / カスタムスクリプト。環境構築時に自動実行
  - **環境別分離**: 開発用（ダミーデータ）/ テスト用（最小データ）/ ステージング用（本番に近いデータ）
  - **ファクトリーベース**: ファクトリー関数でシードデータを動的生成。スキーマ変更への追従が容易
  - **SQLダンプ**: 本番データのマスキング済みダンプをステージングに適用
- **選定基準**:
  - 開発環境: 主要なユースケースを網羅するダミーデータ（マスターデータ + サンプルデータ）
  - テスト環境: テストケースに必要な最小データ。テストごとにセットアップ/ティアダウン
  - ステージング: 本番に近い量とバリエーション（個人情報はマスキング）
  - シードスクリプトはバージョン管理し、マイグレーションと連動させる
- **トレードオフ・注意点**:
  - シードデータが肥大化すると環境構築時間が増加
  - スキーマ変更時にシードデータの更新を忘れると環境構築が壊れる
  - 本番データの利用は個人情報保護の観点から慎重に（マスキング必須）
  - 冪等性: シードスクリプトは何度実行しても同じ結果になること（upsert使用）
- **2025-2026年のトレンド**: Prismaの`prisma db seed`が標準的なシード方法。ファクトリーパターンとシードの統合（ファクトリーでシードデータを生成）が推奨。Faker.jsによるリアリスティックなダミーデータ生成が広く使われている

### 4. データバリデーション戦略（DB制約 vs アプリ層 vs 両方）

- **何を決めるか**: データ整合性を保証するレイヤーと方法
- **選択肢**:
  - **DB制約のみ**: NOT NULL, UNIQUE, CHECK, FOREIGN KEY。最終防衛線として機能
  - **アプリ層のみ**: Zod, Yup, class-validator等。ユーザーフレンドリーなエラーメッセージ
  - **両方（推奨）**: アプリ層で早期検証 + DB制約で最終保証。多層防御
- **選定基準**:
  - **必ずDB制約で保証すべきもの**: 一意性（UNIQUE）、外部キー参照整合性（FK）、NOT NULL、CHECK制約（範囲、ENUM値）
  - **アプリ層で検証すべきもの**: 入力フォーマット（メール、電話番号等）、ビジネスルール（年齢制限等）、クロスフィールドバリデーション
  - **Zodの位置づけ**: APIの入力バリデーション + フォームバリデーションの共通スキーマ定義。コンパイル時型チェックをランタイムに拡張

  | レイヤー | 役割 | ツール例 |
  |---------|------|---------|
  | フロントエンド | UXフィードバック（即時） | Zod + React Hook Form |
  | API層 | 入力サニタイズ・型検証 | Zod（フロントと共有可能） |
  | ビジネスロジック層 | ビジネスルール検証 | ドメインモデルのメソッド |
  | DB層 | データ整合性の最終保証 | 制約（NOT NULL, FK, CHECK, UNIQUE） |

- **トレードオフ・注意点**:
  - DB制約のみではエラーメッセージがユーザーフレンドリーでない
  - アプリ層のみではDB直接操作やバグによるデータ不整合リスク
  - バリデーションルールの重複管理: Zodスキーマからフロントエンド・バックエンドの両方で使用できる設計にする
  - DB制約違反のエラーハンドリング: ORMのエラーを適切にキャッチしてユーザー向けメッセージに変換
- **2025-2026年のトレンド**: ZodがTypeScriptプロジェクトのバリデーション標準として定着。フロントエンド（React Hook Form + Zod）とバックエンド（API入力検証）でスキーマを共有する「Schema-first validation」が主流。Zod v4のパフォーマンス改善（2倍高速化）も注目されている

### 5. インデックス設計方針

- **何を決めるか**: インデックスの設計アプローチ、適用基準、保守方法
- **選択肢**:
  - **ワークロード駆動（推奨）**: 遅いクエリを観測し、必要に応じてインデックスを追加
  - **事前設計**: スキーマ設計時にクエリパターンを予測してインデックスを定義
  - **ハイブリッド**: 主要なクエリパターンは事前設計、その他はワークロード駆動で追加
- **選定基準**:
  - **必ずインデックスを作成**: PRIMARY KEY（自動）、FOREIGN KEY、頻繁にWHERE/ORDER BYで使われるカラム、UNIQUE制約カラム
  - **複合インデックス**: 複数カラムの組み合わせで頻繁にフィルタされる場合。選択性の高いカラムを先頭に配置
  - **カバリングインデックス**: SELECT句のカラムもインデックスに含め、テーブルアクセスを回避
  - **部分インデックス**: WHERE句付きインデックスでサイズを削減（例: `WHERE deleted_at IS NULL`）
  - **インデックスタイプ**: B-tree（デフォルト、範囲/等値）、GIN（配列、全文検索、JSONB）、GiST（地理情報）
- **トレードオフ・注意点**:
  - インデックスは読み取りを高速化するが、書き込み（INSERT/UPDATE/DELETE）を遅くする
  - 過剰なインデックスはストレージ消費とVACUUM/ANALYZE時間を増加させる
  - 大規模テーブルへのインデックス追加は`CREATE INDEX CONCURRENTLY`で無停止実施
  - EXPLAIN ANALYZEで実行計画を確認し、インデックスが実際に使われているか検証
  - 未使用インデックスは定期的に削除（pg_stat_user_indexesで確認）
- **2025-2026年のトレンド**: HypoPG（仮想インデックス）による「実際に作成せずにEXPLAINで効果を検証」するアプローチが推奨されている。また、Supabase等のマネージドPostgreSQLサービスでインデックスアドバイザー機能が充実。ワークロード駆動の「計測 -> 仮説 -> 検証 -> 適用」サイクルがベストプラクティスとして確立

### 6. データバックアップ・リストア戦略

- **何を決めるか**: バックアップ方法、頻度、保持期間、リストア手順、テスト方針
- **選択肢**:
  - **フルバックアップ**: DB全体のスナップショット。復旧が容易だがストレージ消費大
  - **増分バックアップ**: 前回からの差分のみ保存。ストレージ効率が高いが復旧が複雑
  - **PITR（Point-in-Time Recovery）**: WAL（Write-Ahead Log）ベースで任意時点に復旧。PostgreSQLネイティブ機能
  - **マネージドサービス**: AWS RDS自動バックアップ / Supabase自動バックアップ / PlanetScale等
  - **3-2-1ルール**: 3コピー、2種類のメディア、1つはオフサイト
- **選定基準**:
  - RPO（Recovery Point Objective）: 許容できるデータ損失時間。RPO 1時間なら1時間ごとのバックアップ
  - RTO（Recovery Time Objective）: 許容できる復旧時間。RTO 30分なら30分以内にリストア完了
  - マネージドサービス利用時は自動バックアップ機能を活用（設定のみ）
  - 小規模: マネージドサービスの自動バックアップ + 日次エクスポート
  - 大規模: PITR + 増分バックアップ + 3-2-1ルール
- **トレードオフ・注意点**:
  - バックアップは定期的にリストアテストを実施しないと、いざという時に使えない
  - 暗号化: バックアップデータの暗号化（at rest + in transit）は必須
  - リージョン障害に備えたクロスリージョンバックアップの検討
  - バックアップ保持期間はコンプライアンス要件に従う（例: 7年保持）
  - マネージドサービスの自動バックアップ保持期間を確認（AWS RDSは最大35日）
- **2025-2026年のトレンド**: マネージドDBサービス（Supabase, PlanetScale, Neon等）の自動バックアップ機能が充実し、小-中規模プロジェクトではバックアップ運用の大部分が自動化。PITRはPostgreSQL系サービスで標準機能として提供。バックアップのリストアテスト自動化（CI/CDパイプラインへの組み込み）が推奨されている

### 7. データライフサイクル（論理削除 vs 物理削除、アーカイブ）

- **何を決めるか**: データ削除の方法、アーカイブ方針、データ保持期間
- **選択肢**:
  - **論理削除（Soft Delete）**: `deleted_at` タイムスタンプまたは `is_deleted` フラグで非表示化。復元可能
  - **物理削除（Hard Delete）**: SQLのDELETE文で完全削除。復元不可（バックアップ除く）
  - **ハイブリッド（推奨）**: データ種別に応じて使い分け + 定期的なアーカイブ/パージ
  - **アーカイブ**: 一定期間後にコールドストレージ（S3等）に移動。本体テーブルの軽量化
- **選定基準**:

  | データ種別 | 推奨方式 | 理由 |
  |-----------|---------|------|
  | ユーザーアカウント | 論理削除 + 一定期間後の物理削除 | 復元可能期間を提供しつつ、GDPR/CCPA対応 |
  | 注文/取引履歴 | 論理削除（長期保持） | 税務監査、BI分析、法的要件 |
  | ログ/アクティビティ | 物理削除（TTLベース） | ストレージ節約、古いログの保持不要 |
  | セッション/一時データ | 物理削除（TTLベース） | ストレージ節約 |
  | 個人情報（GDPR対象） | 物理削除（削除権の行使） | 法的義務 |

- **トレードオフ・注意点**:
  - 論理削除のクエリ影響: 全クエリに`WHERE deleted_at IS NULL`が必要。ORMのデフォルトスコープで対応
  - 論理削除のDB肥大化: 削除済みレコードが蓄積し、クエリ性能に影響。部分インデックスで対策
  - 物理削除の参照整合性: 外部キー制約によるカスケード削除に注意。関連テーブルのデータが消える
  - GDPR/CCPAの「忘れられる権利」: 論理削除では不十分。個人識別情報の物理削除が必要
  - アーカイブ: ホットデータとコールドデータを分離し、ホットテーブルの性能を維持
- **2025-2026年のトレンド**: 多くの実務プロジェクトではハイブリッドアプローチが標準。論理削除にはPrismaの`@@map`やDrizzleのフィルタデフォルト機能で透過的に対応。GDPR/CCPAの影響で「データ削除ポリシーの明文化」が必須事項に。「個人情報は物理削除、ビジネスデータは匿名化して保持」のパターンが確立

### 8. 外部データのスクレイピング/取得方針（OGP、記事メタデータ取得）

- **何を決めるか**: 外部サイトからのデータ取得方法、キャッシュ、法的考慮、エラーハンドリング
- **選択肢**:
  - **OGPメタデータ取得**: HTMLのmeta ogタグを解析。リンクプレビュー生成に使用
  - **APIベース取得**: 公開API（oEmbed, OpenGraph.io等）を利用。構造化されたデータ取得
  - **HTMLパース**: Cheerio等でHTMLを解析。APIがない場合の代替手段
  - **ヘッドレスブラウザ**: Puppeteer/Playwrightで動的コンテンツを取得。コスト高
- **選定基準**:
  - OGPメタデータ: サーバーサイドでHTTPリクエスト + HTML解析。軽量で十分な場合が多い
  - SPA/動的サイト: ヘッドレスブラウザが必要（ただしコストと複雑性が増す）
  - 頻繁なアクセス: キャッシュ必須（Redis or DBキャッシュ、TTL設定）
  - 法的リスク: robots.txt遵守、利用規約確認、著作権への配慮
- **トレードオフ・注意点**:
  - 外部サイトのHTML構造変更で取得が壊れるリスク（エラーハンドリング必須）
  - レート制限: 外部サイトへの過剰なリクエストを避ける（キュー処理、バックオフ）
  - キャッシュの鮮度: OGPデータの更新頻度に応じたTTL設定（例: 24時間 ~ 7日）
  - フォールバック: 取得失敗時のデフォルト表示（URL表示、エラーメッセージ）
  - User-Agentの設定: 適切なUser-Agentを設定し、ボットとして識別可能にする
- **2025-2026年のトレンド**: Unfurl, link-preview-js 等の軽量ライブラリが充実。Vercelの`@vercel/og`のようなOGP画像動的生成も普及。サーバーサイドでの取得 + Redisキャッシュ + 非同期更新のパターンが標準的。LLMを活用したメタデータ補完（タイトルやサマリの自動生成）も登場

### 9. キャッシュ戦略（DBクエリキャッシュ、アプリケーションキャッシュ）

- **何を決めるか**: キャッシュのレイヤー、ツール、無効化戦略、キャッシュ対象
- **選択肢**:
  - **インメモリキャッシュ**: Node.jsプロセス内（node-cache, lru-cache）。最速だがプロセス間共有不可
  - **分散キャッシュ（Redis）**: プロセス間・サーバー間共有。セッション、APIレスポンス、DBクエリ結果
  - **CDNキャッシュ**: 静的アセット、SSGページ。グローバル配信
  - **ブラウザキャッシュ**: Cache-Control, ETag, Service Worker。クライアント側
  - **多層キャッシュ**: CDN -> Redis -> インメモリの多段構成
- **選定基準**:

  | キャッシュ対象 | 推奨レイヤー | TTL目安 |
  |--------------|------------|--------|
  | 静的アセット（CSS/JS/画像） | CDN + ブラウザ | 長期（1年、contenthash使用） |
  | SSGページ | CDN | 数時間 ~ ISR（On-demand） |
  | APIレスポンス（読み取り多） | Redis | 数分 ~ 数時間 |
  | DBクエリ結果 | Redis / インメモリ | 数秒 ~ 数分 |
  | セッション | Redis | セッション有効期間 |
  | ユーザー固有データ | キャッシュしない（最新性重視） | - |

- **キャッシュ無効化戦略**:
  - **TTL（Time-To-Live）**: 一定時間後に自動失効。シンプルだが鮮度にラグ
  - **Write-Through**: 書き込み時にキャッシュも同期更新。一貫性高いが書き込み遅延
  - **Write-Behind**: 書き込みはキャッシュのみ、非同期でDBに反映。高速だがデータ損失リスク
  - **Pub/Sub invalidation**: データ変更イベントでキャッシュ無効化。リアルタイム性が必要な場合
  - **タグベース無効化**: 関連するキャッシュエントリをタグでグループ化し一括無効化
- **トレードオフ・注意点**:
  - キャッシュは複雑性を増す。「キャッシュが本当に必要か」を先に検討
  - キャッシュ不整合: 古いデータの表示リスク。ビジネス要件に応じたTTL設計
  - キャッシュスタンピード: TTL切れ時に大量リクエストがDBに直撃。ミューテックスロックで防止
  - キャッシュウォームアップ: デプロイ後のコールドキャッシュ対策
  - 監視: キャッシュヒット率、メモリ使用量、エビクション率を計測
- **2025-2026年のトレンド**: Next.jsのISR（Incremental Static Regeneration）やon-demand revalidationによるキャッシュ制御が普及。Redisはキャッシュ + セッション + Pub/Subの統合ソリューションとして依然として主流。Vercel KV（Redisベース）やUpstash（サーバーレスRedis）等のマネージドRedisサービスの利用が増加。Redis + ローカルキャッシュの多層構成がパフォーマンスと整合性のバランスとして推奨

### 10. トランザクション管理方針

- **何を決めるか**: 同時実行制御の方式、トランザクション境界、分散トランザクションの扱い
- **選択肢**:
  - **楽観ロック（Optimistic Locking）**: バージョンカラム（`version` or `updated_at`）で競合検出。コミット時に競合判定
  - **悲観ロック（Pessimistic Locking）**: `SELECT ... FOR UPDATE`でレコードロック。アクセス中は他トランザクションをブロック
  - **MVCC（Multi-Version Concurrency Control）**: PostgreSQLのデフォルト。読み取りはロックなし、書き込み時のみ競合検出
  - **サガパターン**: マイクロサービス間の分散トランザクション。各サービスのローカルトランザクション + 補償トランザクション
- **選定基準**:

  | シナリオ | 推奨方式 | 理由 |
  |---------|---------|------|
  | Webアプリの一般的なCRUD | 楽観ロック | HTTP statelessとの相性、低競合率 |
  | 在庫管理、座席予約 | 悲観ロック | 高頻度競合、ビジネス上のシリアライズ必須 |
  | 読み取り多・書き込み少 | MVCC（DB任せ） | PostgreSQLデフォルトで十分 |
  | マイクロサービス間 | サガパターン | 分散トランザクションの代替 |
  | ユーザーの同時編集 | 楽観ロック + 競合解決UI | UXを考慮した競合ハンドリング |

- **トランザクション境界の設計**:
  - サービス層（ユースケース層）でトランザクションを開始・終了
  - 1つのユースケース = 1つのトランザクション境界が原則
  - ORMのトランザクションAPI（`prisma.$transaction()`, Drizzleの`db.transaction()`）を使用
  - ネストされたトランザクションは避ける（予期しないコミット/ロールバックの原因）
- **トレードオフ・注意点**:
  - 楽観ロック: 競合時にユーザーに再試行を求める必要がある（UX設計が必要）
  - 悲観ロック: デッドロックリスク。タイムアウト設定が必須
  - 長時間トランザクション: DBコネクションの枯渇、他トランザクションのブロック。できるだけ短くする
  - Webアプリケーション（HTTP stateless）では悲観ロックは不適切な場合が多い
  - 読み取り一貫性: トランザクション分離レベル（READ COMMITTED がPostgreSQLデフォルト）の理解が必要
- **2025-2026年のトレンド**: モノリシックなWebアプリケーションでは楽観ロック + PostgreSQL MVCCの組み合わせが主流。Prismaの`$transaction()`やDrizzleのトランザクションAPIにより、ORMレベルでのトランザクション管理が容易に。分散トランザクションが必要なマイクロサービス構成では、サガパターンよりもイベント駆動の最終的一貫性（Eventual Consistency）が採用される傾向
