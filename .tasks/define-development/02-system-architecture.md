# システムアーキテクチャ選定ガイド

Webアプリケーション新規開発におけるシステムアーキテクチャの決定事項を網羅的に整理したドキュメント。各項目について選択肢、選定基準、トレードオフ、2025-2026年のトレンドを記載する。

## 決定事項一覧（サマリテーブル）

| # | 決定事項 | 主な選択肢 | 選定の主軸 |
|---|---------|-----------|-----------|
| 1 | アーキテクチャパターン | モノリス, モジュラーモノリス, マイクロサービス, サーバーレス | チーム規模、スケール要件、運用負荷 |
| 2 | レイヤー分離パターン | レイヤード, Clean, Hexagonal, Onion | ドメインの複雑さ、テスタビリティ、チーム規模 |
| 3 | フロントエンド-バックエンド分離戦略 | SPA+API, SSR, SSG, ISR, ハイブリッド | SEO要件、初期表示速度、インタラクティブ性 |
| 4 | API設計方針 | RESTful設計、エラーハンドリング規約、バージョニング | 一貫性、拡張性、ドキュメンテーション |
| 5 | 状態管理アーキテクチャ | サーバーステート/クライアントステート分離、統合管理 | データの所在、同期頻度、オフライン要件 |
| 6 | キャッシュ戦略 | HTTPキャッシュ, CDN, Service Worker, アプリケーション層 | パフォーマンス要件、データ鮮度、オフライン要件 |
| 7 | エラーハンドリング全体方針 | 境界パターン、Result型、例外伝播 | ユーザー体験、デバッグ容易性、一貫性 |
| 8 | ドメインモデリング方針 | DDD採用/不採用、Anemic/Rich Domain Model | ドメインの複雑さ、チーム経験、開発規模 |
| 9 | 依存性注入（DI）方針 | DIコンテナ, 手動DI, モジュールパターン | テスタビリティ、疎結合度、フレームワーク依存 |

## 各項目の詳細

### 1. アーキテクチャパターン

- **何を決めるか**: システム全体の構造パターン。サービスの分割粒度とデプロイ単位。
- **選択肢**:

  | パターン | デプロイ単位 | チーム規模 | 運用負荷 | スケーラビリティ | 初期コスト |
  |---------|:---:|------|:---:|:---:|:---:|
  | モノリス | 1 | 1-5人 | 低 | 低 | 最低 |
  | モジュラーモノリス | 1 | 3-15人 | 低 | 中 | 低 |
  | マイクロサービス | 多数 | 10人以上 | 高 | 最高 | 高 |
  | サーバーレス | 関数単位 | 1-10人 | 最低 | 高（自動） | 低 |
  | Jamstack | 静的+API | 1-5人 | 低 | 高（CDN） | 低 |

  **各パターンの詳細**:

  **モノリス**: 全機能が単一のコードベース・デプロイ単位に含まれる。小規模チームでの初期開発に最適。コード間の結合度が高いため、規模拡大に伴い変更影響範囲の把握が困難になる。

  **モジュラーモノリス**: モノリスの内部をドメイン別モジュールに分割し、モジュール間の依存を明示的に管理する。モノリスの開発・デプロイの容易さを保ちつつ、将来のサービス分割への移行パスを確保する。

  **マイクロサービス**: 各サービスが独立してデプロイ・スケール可能。サービス間通信（HTTP/gRPC/メッセージキュー）、分散トランザクション、サービスディスカバリ等のインフラが必要。組織のスケーリング（コンウェイの法則）に対応する。

  **サーバーレス**: 関数単位でデプロイ。インフラ管理が不要で従量課金。コールドスタート、実行時間制限、ステートレス制約がある。イベント駆動処理や不定期実行のバッチ処理に適する。

  **Jamstack**: 静的サイト生成 + API/サーバーレス関数。CDNでの配信により高いパフォーマンスとスケーラビリティ。動的コンテンツが多いアプリには不向き。

- **選定基準**:
  - チーム規模と組織構造（コンウェイの法則）
  - スケーラビリティ要件（トラフィック量、成長見込み）
  - 運用チームの有無・能力
  - デプロイ頻度と独立デプロイの必要性
  - コスト構造（固定費 vs 従量課金）
  - 開発速度の優先度
- **トレードオフ・注意点**:
  - 「マイクロサービスから始める」は典型的なアンチパターン。小規模チームでは運用負荷がメリットを上回る
  - モノリスからマイクロサービスへの移行は可能だが、逆は困難。モジュラーモノリスが中間的な退出戦略
  - サーバーレスはコールドスタート（数百ms〜数秒）が低レイテンシ要件と衝突する場合がある
  - ハイブリッド構成（モノリス + サーバーレス関数で補助タスク）も有効
  - マイクロサービスの分散トランザクション管理（Sagaパターン等）は複雑さが大きい
- **2025-2026年のトレンド**:
  - 「モジュラーモノリスから始めて、必要に応じて分割」がベストプラクティスとして定着
  - サーバーレスとエッジコンピューティングの融合（Cloudflare Workers, Vercel Edge Functions）
  - 「マイクロサービス疲れ」からの揺り戻しで、シンプルなアーキテクチャへの回帰傾向
  - ハイブリッドアプローチ: コアサービス（マクロサービス） + イベント駆動サーバーレス関数
  - サイドカーレスサービスメッシュ（Istio Ambient Mesh等）によるマイクロサービスのオーバーヘッド削減

### 2. レイヤー分離パターン

- **何を決めるか**: アプリケーション内部のコード構造とレイヤー間の依存関係ルール。
- **選択肢**:

  | パターン | 依存方向 | 複雑度 | テスタビリティ | 適用規模 |
  |---------|---------|:---:|:---:|------|
  | レイヤードアーキテクチャ | 上→下（一方向） | 低 | 中 | 小〜中規模 |
  | Clean Architecture | 外→内（依存性逆転） | 高 | 最高 | 中〜大規模 |
  | Hexagonal (Ports & Adapters) | 外→内（ポート/アダプタ） | 高 | 最高 | 中〜大規模 |
  | Onion Architecture | 外→内（同心円） | 高 | 最高 | 中〜大規模 |
  | Feature-based (垂直スライス) | 機能単位 | 中 | 高 | 中規模 |

  **各パターンの詳細**:

  **レイヤードアーキテクチャ（3層/N層）**:
  - Presentation → Business Logic → Data Access の一方向依存
  - 最もシンプルで理解しやすい
  - ビジネスロジック層がデータアクセス層に依存するため、DB変更時にビジネスロジックへの影響がある

  **Clean Architecture**:
  - Entities → Use Cases → Interface Adapters → Frameworks & Drivers の同心円構造
  - 依存性の方向は常に外側から内側（依存性逆転の原則）
  - ビジネスルールがフレームワーク・DB・UI から完全に独立
  - Uncle Bob（Robert C. Martin）が提唱

  **Hexagonal Architecture（Ports & Adapters）**:
  - アプリケーションコアが「ポート」（インターフェース）を定義し、外部は「アダプタ」で接続
  - 主要ポート（入力: Driving）と従属ポート（出力: Driven）を区別
  - テスト時にアダプタをモックに差し替え可能
  - Alistair Cockburnが提唱

  **Onion Architecture**:
  - Domain Model → Domain Services → Application Services → Infrastructure の同心円
  - Clean Architectureと概念的にほぼ同一。実装詳細が異なる
  - Jeffrey Palermoが提唱

  **Feature-based（垂直スライス）**:
  - 機能単位でコードを分割（レイヤー横断）
  - 各機能が独立して変更可能
  - レイヤー分離と組み合わせて使うことも可能

- **選定基準**:
  - ドメインの複雑さ（複雑なビジネスルールがあるか）
  - テスタビリティの要件（単体テストの独立性）
  - チームのアーキテクチャパターンへの習熟度
  - 長期メンテナンスの見通し（コードベースの成長予測）
  - フレームワーク・DB変更の可能性
- **トレードオフ・注意点**:
  - Clean/Hexagonal/Onionは概念的にほぼ同一（依存性逆転 + ドメイン中心）。実装レベルでの差異に固執する意味は薄い
  - これらの高度なパターンは、小規模プロジェクトではオーバーエンジニアリングになりやすい
  - レイヤードアーキテクチャは理解しやすいが、ビジネスロジックが薄いCRUDアプリでは「通過するだけの層」が生まれやすい
  - 垂直スライスは機能追加が容易だが、横断的関心事（認証、ログ等）の配置に工夫が必要
  - TypeScript/JSのエコシステムではインターフェース（ポート）の実装がJava等より弱い点に注意
  - フロントエンドにClean Architectureを適用する試みもあるが、コストに見合うケースは限定的
- **2025-2026年のトレンド**:
  - 「必要十分なレイヤー分離」の実用主義が主流。パターンの教条的適用からの脱却
  - Vertical Slice Architecture（垂直スライス）がレイヤード + Clean Architectureのハイブリッドとして注目
  - フルスタックフレームワーク（Next.js等）の規約に合わせたレイヤー分離（Server Components = Presentation、Server Actions = Application Layer等）
  - Feature-basedディレクトリ構造がNext.js App Router等で標準的に

### 3. フロントエンド-バックエンド分離戦略

- **何を決めるか**: ページのレンダリング方式と、フロント・バックの結合度。
- **選択肢**:

  | 戦略 | 初期表示 | SEO | インタラクティブ性 | サーバー負荷 | 複雑度 |
  |------|:---:|:---:|:---:|:---:|:---:|
  | SPA + API | 遅い | 弱い | 最高 | 低 | 中 |
  | SSR (Server-Side Rendering) | 速い | 強い | 高 | 高 | 高 |
  | SSG (Static Site Generation) | 最速 | 最強 | 低〜中 | 最低 | 低 |
  | ISR (Incremental Static Regeneration) | 速い | 強い | 中 | 低 | 中 |
  | ストリーミングSSR | 段階的 | 強い | 高 | 中 | 高 |
  | ハイブリッド（ページ単位選択） | ページ依存 | ページ依存 | ページ依存 | 中 | 高 |

  **各戦略の詳細**:

  **SPA（Single Page Application）+ API**:
  - フロントエンドとバックエンドを完全に分離。フロントはAPIを呼び出してデータ取得
  - フロント・バックが独立してデプロイ・スケール可能
  - 初回ロードでJSバンドル全体をダウンロードするため、初期表示が遅い
  - SEOはクローラーのJS実行能力に依存（改善傾向だが完全ではない）

  **SSR（Server-Side Rendering）**:
  - サーバーでHTMLを生成してクライアントに送信。その後クライアントでハイドレーション
  - 初期表示が速く、SEOに強い
  - サーバーへの負荷が増加。リクエストごとにレンダリングが必要
  - Time-to-First-Byte（TTFB）がサーバー処理時間に依存

  **SSG（Static Site Generation）**:
  - ビルド時にHTMLを事前生成。CDNから配信
  - 最速の初期表示、最強のSEO
  - コンテンツ更新にはリビルドが必要。頻繁に更新されるデータには不向き

  **ISR（Incremental Static Regeneration）**:
  - SSGの拡張。バックグラウンドで静的ページを再生成
  - SSGの速度とSSRの動的性のバランス
  - Next.js固有機能（他フレームワークでは類似機能の対応状況が異なる）

  **ストリーミングSSR**:
  - HTMLをチャンク単位で段階的に送信。重要なコンテンツを先に表示
  - React Server Components + Suspenseで実現
  - TTFBを短縮しつつ、動的データの表示も可能

- **選定基準**:
  - SEO要件（検索エンジンからの流入の重要度）
  - 初期表示速度の要件（LCP、FCP等のCore Web Vitals）
  - コンテンツの更新頻度（静的 vs 動的）
  - フロント・バックの独立デプロイの必要性
  - オフライン対応の必要性（PWA要件）
  - インフラコスト（SSRはサーバーコスト増）
- **トレードオフ・注意点**:
  - SPA + APIはフロント・バック完全分離で柔軟だが、SEOと初期表示速度に課題
  - SSRは万能に見えるが、サーバーコストとTTFBのトレードオフがある
  - SSGはパフォーマンス最高だが、動的コンテンツの多いアプリには不適合
  - ハイブリッド（ページ単位でSSR/SSG/CSRを使い分け）が現実的だが、メンタルモデルが複雑になる
  - フルスタックフレームワーク採用時は、フレームワークの規約に沿ったレンダリング戦略が推奨
  - SPA + APIでフロント・バック分離する場合、CORSの設定・管理が必要
- **2025-2026年のトレンド**:
  - React Server Componentsによる「サーバーファースト」開発モデルの普及
  - ストリーミングSSR（Suspense + streaming）が標準的な選択肢に
  - ページ単位のハイブリッドレンダリングがフレームワーク標準
  - Partial Prerendering（PPR）: 静的シェル + 動的パーツのストリーミング（Next.js 15）
  - エッジSSR（Cloudflare Workers等）でSSRのレイテンシ問題を軽減

### 4. API設計方針

- **何を決めるか**: APIの設計原則、エラーハンドリング規約、バージョニング戦略。
- **選択肢**:

  **RESTful設計原則**:

  | 項目 | 決定内容 |
  |------|---------|
  | URL設計 | リソース指向（名詞、複数形）。例: `/api/users/{id}/bookmarks` |
  | HTTPメソッド | GET（取得）、POST（作成）、PUT（全体更新）、PATCH（部分更新）、DELETE（削除） |
  | ステータスコード | 成功: 200/201/204、クライアントエラー: 400/401/403/404/422、サーバーエラー: 500/503 |
  | ページネーション | Cursor-based（推奨）vs Offset-based |
  | フィルタリング | クエリパラメータ。例: `?status=active&sort=-created_at` |
  | レスポンス形式 | JSON。envelope（`{data, meta, errors}`）の採否 |

  **エラーハンドリング規約**:

  | アプローチ | 概要 | 特徴 |
  |-----------|------|------|
  | RFC 7807 (Problem Details) | 標準化されたエラーレスポンス形式 | `type`, `title`, `status`, `detail`, `instance` フィールド |
  | カスタムエラーフォーマット | プロジェクト固有の形式 | 柔軟だが標準化の恩恵なし |
  | GraphQLエラー | `errors` 配列 + `extensions` | GraphQL仕様に準拠 |

  ```json
  // RFC 7807 例
  {
    "type": "https://example.com/errors/validation",
    "title": "Validation Error",
    "status": 422,
    "detail": "The 'email' field is not a valid email address.",
    "instance": "/api/users",
    "errors": [
      { "field": "email", "message": "Invalid email format" }
    ]
  }
  ```

  **バージョニング戦略**:

  | 方式 | 実装例 | メリット | デメリット |
  |------|------|---------|-----------|
  | URLパスベース | `/api/v1/users` | 明示的、キャッシュ容易 | URL汚染 |
  | ヘッダーベース | `Accept: application/vnd.api.v1+json` | URL clean | ブラウザテスト困難 |
  | クエリパラメータ | `/api/users?version=1` | 簡易 | キャッシュキー複雑化 |
  | バージョニングしない | API進化（非破壊的変更のみ） | シンプル | 破壊的変更時に困る |

- **選定基準**:
  - API利用者の範囲（社内のみ vs サードパーティ公開）
  - クライアントの多様性（Web、モバイル、外部サービス）
  - 後方互換性の維持期間
  - ドキュメンテーションの自動生成要件（OpenAPI/Swagger）
  - エラー情報のデバッグ容易性
- **トレードオフ・注意点**:
  - 厳密なRESTful設計（HATEOAS含む）は理論的には美しいが、実装・消費の両面でコストが高い。実用的なREST（Level 2）が一般的
  - URLパスベースのバージョニングが最も広く採用されているが、マイクロバージョンの管理が課題
  - 内部APIのみなら「バージョニングしない」（非破壊的進化）が最もシンプル
  - エラーレスポンスの一貫性は開発初期に決めないと、後から統一するコストが高い
  - OpenAPI仕様からのクライアントコード自動生成を活用すると、APIとクライアントの型の乖離を防げる
  - ページネーションはCursor-basedが推奨（Offset-basedはデータ挿入時にズレる問題がある）
- **2025-2026年のトレンド**:
  - OpenAPI 3.1 + コード生成（orval, openapi-typescript等）による型安全なAPI開発
  - RFC 9457（RFC 7807の後継）によるエラーレスポンスの標準化
  - tRPCやServer ActionsによるAPI定義の自動化（明示的なAPI設計が不要になるケース）
  - API-first開発（仕様定義 → コード生成）のワークフローが成熟

### 5. 状態管理アーキテクチャ

- **何を決めるか**: アプリケーション状態の分類方法と、それぞれの管理方式。
- **選択肢**:

  **状態の分類**:

  | 状態種別 | 定義 | 例 | 管理方式 |
  |---------|------|-----|---------|
  | サーバーステート | サーバーに永続化され、APIで取得するデータ | ユーザー情報、記事一覧、設定 | TanStack Query, SWR, Apollo Client |
  | クライアントステート（UI） | UI表示に関わる一時的な状態 | モーダルの開閉、フォーム入力中 | useState, useReducer, Zustand, Jotai |
  | クライアントステート（グローバル） | 複数コンポーネントで共有する状態 | テーマ設定、認証状態 | Context API, Zustand, Redux |
  | URLステート | URLに反映される状態 | 検索クエリ、フィルタ、ページ番号 | URLSearchParams, nuqs |
  | フォームステート | フォームの入力値・バリデーション状態 | 入力値、エラー、送信状態 | React Hook Form, Formik |
  | 永続化ステート | ブラウザに保存する状態 | ユーザー設定、下書き | localStorage, IndexedDB |

  **管理ライブラリの選択肢（React系）**:

  | ライブラリ | 種別 | バンドルサイズ | 特徴 |
  |-----------|------|:---:|------|
  | TanStack Query | サーバーステート | ~12KB | キャッシュ、再フェッチ、楽観的更新 |
  | SWR | サーバーステート | ~4KB | stale-while-revalidate、軽量 |
  | Zustand | クライアントステート | ~1KB | ミニマル、ボイラープレート少 |
  | Jotai | クライアントステート | ~3KB | アトミック、ボトムアップ |
  | Redux Toolkit | 両方 | ~11KB | 大規模向け、DevTools充実 |
  | Context API | クライアントステート | 0（組込） | 軽量、再レンダリング注意 |

- **選定基準**:
  - サーバーステートとクライアントステートの比率
  - キャッシュ戦略の要件（stale-while-revalidate、楽観的更新）
  - オフライン対応の必要性
  - 状態のデバッグ要件（DevTools、タイムトラベルデバッグ）
  - バンドルサイズの制約
  - チームの習熟度
- **トレードオフ・注意点**:
  - 最大の設計判断は「サーバーステートとクライアントステートの分離」。この分離を行うだけで、状態管理の複雑さの大部分が解消される
  - TanStack Query/SWR導入により、従来Reduxで管理していたサーバーステートの大部分が不要になる
  - Zustandは軽量だが、大規模になるとストア分割の設計が必要
  - Context APIは手軽だが、高頻度更新の状態を持つと不要な再レンダリングが発生
  - Redux Toolkitは機能豊富だが、小規模プロジェクトではオーバーキル
  - Server Components環境ではクライアントステートの必要量自体が減少する
- **2025-2026年のトレンド**:
  - **TanStack Query + Zustand** の組み合わせが2025年の主流パターンとして定着
  - サーバーステート管理ライブラリ（TanStack Query, SWR）の必須化
  - Reduxの新規採用は減少傾向。既存プロジェクトではRedux Toolkit + RTK Queryが標準
  - React Server Componentsの普及で、クライアント側状態管理の必要量が減少
  - Jotai/Valtioなどのアトミック状態管理が、特定のUI複雑性を持つケースで採用増
  - 40%のバンドルサイズ削減事例あり（Redux → Zustand + TanStack Query移行）

### 6. キャッシュ戦略

- **何を決めるか**: 各レイヤーでのキャッシュ方針と、データ鮮度と性能のバランス。
- **選択肢**:

  **キャッシュレイヤー**:

  | レイヤー | 制御主体 | 粒度 | 有効範囲 | 主な用途 |
  |---------|---------|------|---------|---------|
  | ブラウザHTTPキャッシュ | サーバー（ヘッダー） | リソース単位 | ユーザーローカル | 静的アセット、API応答 |
  | CDNキャッシュ | インフラ設定 | URL単位 | グローバル | 静的ファイル、SSG/ISRページ |
  | Service Workerキャッシュ | フロントエンドコード | プログラマブル | ユーザーローカル | オフライン対応、高速再表示 |
  | アプリケーション層キャッシュ | バックエンドコード | ドメイン単位 | サーバー内 | DB負荷軽減、計算結果 |
  | データフェッチライブラリキャッシュ | フロントエンドライブラリ | クエリ単位 | ユーザーローカル | API応答のキャッシュ |
  | DBクエリキャッシュ | DB設定 | クエリ単位 | DB内 | 頻出クエリの高速化 |

  **キャッシュ戦略パターン**:

  | 戦略 | 動作 | 適用場面 |
  |------|------|---------|
  | Cache-First | キャッシュ優先、なければネットワーク | 変更頻度が低いリソース |
  | Network-First | ネットワーク優先、失敗時にキャッシュ | 最新データが重要なリソース |
  | Stale-While-Revalidate | キャッシュを即時返却 + バックグラウンド更新 | 鮮度と速度のバランス |
  | Cache-Only | キャッシュのみ | オフラインリソース |
  | Network-Only | ネットワークのみ | 常に最新が必要 |

  **HTTPキャッシュヘッダー設計**:

  | リソース種別 | Cache-Control | ETag | 備考 |
  |-------------|--------------|:---:|------|
  | 静的アセット（ハッシュ付） | `max-age=31536000, immutable` | 不要 | ファイル名にハッシュを含む前提 |
  | HTML | `no-cache` または `max-age=0, must-revalidate` | 必要 | 常にサーバーで検証 |
  | API応答（変更少） | `max-age=60, stale-while-revalidate=300` | 推奨 | 用途に応じて調整 |
  | API応答（リアルタイム） | `no-store` | 不要 | キャッシュしない |
  | 認証済みデータ | `private, no-cache` | 推奨 | CDNにキャッシュしない |

- **選定基準**:
  - データの更新頻度（リアルタイム vs 日次 vs 静的）
  - オフライン対応の要件
  - ユーザーに対するデータ鮮度の許容範囲
  - インフラコスト（CDN料金、サーバー負荷）
  - デバッグ容易性（キャッシュの問題は特定が困難）
- **トレードオフ・注意点**:
  - キャッシュは「表示が古い」「更新が反映されない」というUX問題の主要原因。戦略の設計と無効化の仕組みが重要
  - 複数レイヤーのキャッシュが重なると、無効化の伝播が複雑になる（CDN + Service Worker + ブラウザキャッシュ）
  - Service Workerキャッシュは強力だが、デバッグが困難。更新の即時反映が難しい場合がある
  - `stale-while-revalidate`は速度と鮮度のバランスが良いが、一時的に古いデータが表示される
  - 認証済みデータをCDNにキャッシュすると、他ユーザーのデータが漏洩するリスクがある（`private`ヘッダー必須）
  - キャッシュキーの設計ミスにより、異なるデータが同じキャッシュエントリを共有する問題に注意
- **2025-2026年のトレンド**:
  - TanStack Query/SWRによるデータフェッチ層のキャッシュが標準化
  - `stale-while-revalidate`がAPIキャッシュの標準戦略として定着
  - Service Worker + Workbox（vite-plugin-pwa等）によるPWAキャッシュの実装が容易に
  - エッジキャッシュ（Cloudflare等）によるSSRレスポンスのキャッシュが普及
  - CDNのAPIベースキャッシュパージ（更新時にプログラマティックに無効化）

### 7. エラーハンドリングの全体方針

- **何を決めるか**: フロントエンド・バックエンド・DB間のエラーの分類、伝播、表示の方針。
- **選択肢**:

  **エラー分類体系**:

  | カテゴリ | 例 | 対処 | ユーザーへの表示 |
  |---------|-----|------|----------------|
  | バリデーションエラー | 入力値不正、形式エラー | フィールド単位で表示 | 具体的な修正指示 |
  | 認証エラー | 未ログイン、トークン期限切れ | ログインページへリダイレクト | ログイン促進 |
  | 認可エラー | 権限不足 | 権限不足の通知 | アクセス権限の案内 |
  | ビジネスロジックエラー | 在庫不足、重複登録 | ドメイン固有のハンドリング | 状況に応じた代替案 |
  | インフラエラー | DB接続失敗、外部API障害 | リトライ + フォールバック | 一般的なエラーメッセージ |
  | 予期しないエラー | 未処理例外、バグ | ログ記録 + アラート | 一般的なエラーメッセージ |

  **エラー伝播パターン**:

  | パターン | 実装方式 | メリット | デメリット |
  |---------|---------|---------|-----------|
  | 例外ベース | try/catch + throw | 言語標準、慣れた開発者が多い | 制御フローが追いにくい、型安全でない |
  | Result型 | `Result<T, E>` / `Either` | 明示的、型安全、コンパイル時チェック | ボイラープレート、TSでのサポート弱い |
  | エラーコードベース | エラーコード + メッセージ | シンプル、言語非依存 | エラー処理の漏れが起きやすい |
  | Error Boundary（フロント） | React Error Boundary | UIクラッシュ防止、フォールバックUI | クラス コンポーネント必須（Reactの場合） |

  **レイヤー別エラーハンドリング方針**:

  | レイヤー | 責務 | 実装 |
  |---------|------|------|
  | フロントエンド（UI） | ユーザーフレンドリーなエラー表示 | Error Boundary、トースト通知、インラインエラー |
  | フロントエンド（データ取得） | リトライ、フォールバック、キャッシュ利用 | TanStack Query のerror/retry設定 |
  | APIレイヤー | エラーレスポンスの標準化 | RFC 9457形式、一貫したステータスコード |
  | ビジネスロジック層 | ドメインエラーの定義と発生 | カスタムエラークラス / Result型 |
  | データアクセス層 | DB固有エラーの変換 | 制約違反 → ビジネスエラーへの変換 |
  | インフラ層 | 外部サービスエラーのラップ | リトライ、サーキットブレーカー |

- **選定基準**:
  - エラーの可視性（開発者/ユーザーそれぞれへの情報提供レベル）
  - デバッグ容易性（本番環境でのエラー追跡）
  - ユーザー体験（エラー発生時のリカバリパス提供）
  - コードの一貫性（全レイヤーで統一されたエラー処理）
  - 型安全性（コンパイル時のエラー処理漏れ検出）
- **トレードオフ・注意点**:
  - Result型はTypeScriptでは言語レベルのサポートがないため、neverthrow等のライブラリが必要。チーム全体の理解が前提
  - 例外ベースは馴染み深いが、「どの関数がどの例外を投げるか」が型レベルで表現できない（JSの制約）
  - エラーメッセージの国際化（i18n）を考慮する場合、エラーコードベースが有利
  - 本番環境ではスタックトレースをユーザーに露出しない。内部エラー詳細はログのみ
  - グローバルエラーハンドラ（uncaughtException等）は最終防衛線としてのみ使用。通常フローでの利用は避ける
  - バリデーションはフロント・バック双方で行う（フロントはUX、バックはセキュリティ）
- **2025-2026年のトレンド**:
  - React Server Componentsの`error.tsx`によるルートレベルエラーハンドリングの標準化
  - Zodなどのスキーマバリデーションライブラリでフロント・バック共通のバリデーション
  - Result型パターンの採用が増加（neverthrow, ts-results等）
  - 構造化ロギング（JSON形式のログ）+ エラートラッキングサービス（Sentry等）の組み合わせ
  - RFC 9457（Problem Details for HTTP APIs）の採用拡大

### 8. ドメインモデリング方針

- **何を決めるか**: ビジネスロジックの表現方法とドメイン駆動設計（DDD）の採用度合い。
- **選択肢**:

  | アプローチ | ドメインの複雑さ | 実装コスト | 適用規模 | 特徴 |
  |-----------|:---:|:---:|------|------|
  | トランザクションスクリプト | 低 | 最低 | 小規模CRUD | 手続き的、サービス層に全ロジック |
  | Anemic Domain Model | 中 | 低 | 中規模 | データ保持のみのエンティティ + サービスにロジック |
  | Rich Domain Model | 高 | 中 | 中〜大規模 | エンティティ自身がビジネスロジックを持つ |
  | DDD（戦略的+戦術的） | 最高 | 高 | 大規模、複雑ドメイン | Bounded Context, Aggregate, Value Object等 |

  **DDDの主要パターン（採用する場合）**:

  | パターン | 定義 | 用途 |
  |---------|------|------|
  | Entity | IDで識別される可変オブジェクト | ユーザー、注文 |
  | Value Object | 値で等価判定される不変オブジェクト | メールアドレス、金額 |
  | Aggregate | 整合性境界を持つエンティティ群 | 注文 + 注文明細 |
  | Repository | 永続化の抽象化 | DBアクセスの隠蔽 |
  | Domain Service | エンティティに属さないドメインロジック | 料金計算、権限判定 |
  | Domain Event | ドメインで発生した事象 | 「注文確定」イベント |
  | Bounded Context | ドメインモデルの適用範囲 | 「販売」コンテキスト vs 「配送」コンテキスト |

  **DDDの段階的採用**:

  | レベル | 採用要素 | コスト | 効果 |
  |-------|---------|:---:|:---:|
  | Level 0 | 不採用（トランザクションスクリプト） | 最低 | - |
  | Level 1 | Value Objectのみ | 低 | 型安全性向上 |
  | Level 2 | Entity + Value Object + Repository | 中 | テスタビリティ向上 |
  | Level 3 | 上記 + Aggregate + Domain Service | 高 | ドメインの明確な表現 |
  | Level 4 | フルDDD（Bounded Context含む） | 最高 | 大規模組織での整合性 |

- **選定基準**:
  - ドメインの複雑さ（ビジネスルールの多さ・変更頻度）
  - チームのDDD経験
  - プロジェクトの規模と寿命
  - ドメインエキスパートとの協業の有無
  - テスタビリティの要件
- **トレードオフ・注意点**:
  - CRUDが主体のアプリにフルDDDを適用するのは典型的なオーバーエンジニアリング
  - TypeScriptではValue Objectの表現が他言語（Java, C#等）より弱い（branded types等で補完）
  - Anemic Domain Modelは「DDD的にはアンチパターン」とされるが、多くのWebアプリでは実用的
  - DDDの戦略的パターン（Bounded Context, Context Map）はマイクロサービス設計と親和性が高い
  - Value Objectの導入（Level 1）は低コストで効果が高く、最初の一歩として推奨
  - ドメインモデルとORMのエンティティを同一にするか分離するかは重要な設計判断
- **2025-2026年のトレンド**:
  - 「実用的DDD」: フルDDDではなく、Value Object + Repository パターンの部分採用が主流
  - TypeScriptでのDDD実践ノウハウの蓄積が進む（branded types, readonly等の活用）
  - Event Sourcing + CQRSの組み合わせは大規模システムで採用増だが、小規模では過剰
  - フルスタックフレームワークのServer Actions/Server Componentsとドメインモデルの統合パターンが議論中

### 9. 依存性注入（DI）の方針

- **何を決めるか**: コンポーネント/モジュール間の依存関係の管理方法。DIコンテナの採否。
- **選択肢**:

  | アプローチ | 複雑度 | テスタビリティ | フレームワーク依存 | 学習コスト |
  |-----------|:---:|:---:|:---:|:---:|
  | DIコンテナ（InversifyJS） | 高 | 最高 | あり | 高 |
  | DIコンテナ（tsyringe） | 中 | 高 | あり | 中 |
  | NestJS内蔵DI | 中 | 高 | NestJS限定 | 中（NestJS利用時は低） |
  | 手動DI（コンストラクタ注入） | 低 | 高 | なし | 低 |
  | モジュールパターン（関数ベース） | 最低 | 中 | なし | 最低 |
  | Factory関数パターン | 低 | 高 | なし | 低 |

  **各アプローチの詳細**:

  **DIコンテナ（InversifyJS, tsyringe）**:
  - デコレータベースで依存関係を宣言的に定義
  - ライフサイクル管理（Singleton, Transient, Request Scope）
  - TypeScriptのデコレータ機能に依存（experimentalDecorators）
  - テスト時にモック注入が容易
  - InversifyJS: 機能豊富（条件付きバインディング、マルチインジェクション等）
  - tsyringe: 軽量、Microsoftメンテ、シンプルなAPI

  **手動DI（コンストラクタ注入）**:
  ```typescript
  // 依存をコンストラクタで受け取る
  class UserService {
    constructor(private readonly userRepo: UserRepository) {}
  }
  // 組み立て（Composition Root）
  const userRepo = new PostgresUserRepository(db);
  const userService = new UserService(userRepo);
  ```
  - フレームワーク不要、純粋なTypeScript
  - 依存グラフが小さい場合は最もシンプル
  - 規模が大きくなるとComposition Rootが肥大化

  **Factory関数パターン**:
  ```typescript
  // 関数クロージャで依存を閉じ込める
  function createUserService(deps: { userRepo: UserRepository }) {
    return {
      getUser: (id: string) => deps.userRepo.findById(id),
    };
  }
  ```
  - クラス不要、関数型スタイルと親和性高い
  - TypeScriptの型推論と自然に統合
  - テスト時は deps をモックオブジェクトに差し替え

  **モジュールパターン（依存関係直接import）**:
  ```typescript
  // 直接importで依存を解決
  import { userRepo } from './repositories/userRepo';
  export const getUser = (id: string) => userRepo.findById(id);
  ```
  - 最もシンプルだが、テスト時のモック差し替えが困難
  - vi.mock / jest.mock で対処可能だが、型安全性が低い

- **選定基準**:
  - テスタビリティの要件（モック注入の頻度）
  - チームの規模とアーキテクチャの複雑さ
  - フレームワーク（NestJS等）がDIを内蔵しているか
  - TypeScriptのデコレータ機能への許容度
  - 関数型 vs オブジェクト指向のスタイル選好
- **トレードオフ・注意点**:
  - DIコンテナはJava/C#エコシステムでは標準だが、TypeScript/JSでは「オーバーキル」と見なされることが多い
  - NestJS使用時はフレームワーク内蔵DIを使うのが自然。別途DIコンテナを導入する必要はない
  - 手動DIは小〜中規模で十分実用的。DIコンテナは依存グラフが複雑になった場合に検討
  - Factory関数パターンは関数型アプローチと相性が良く、TypeScriptの型推論が自然に効く
  - テストでのモック差し替えが不要なら（例: 依存が少ない、E2Eテスト中心）、DIの仕組み自体が不要な場合もある
  - デコレータベースのDIコンテナは、TypeScript 5.0のデコレータ仕様変更に影響を受ける可能性がある
- **2025-2026年のトレンド**:
  - TypeScriptのWebアプリでは「手動DI」または「Factory関数パターン」が主流。DIコンテナの採用は限定的
  - NestJS採用時はフレームワーク内蔵DIが標準
  - 関数型アプローチ（Effect-TS等）によるDI代替パターンの実験的採用
  - React Server Components環境では、サーバー側のDI設計がより重要に（コンポーネント単位でのデータ取得）
  - tsyringeがMicrosoft維持で安定。InversifyJSはより高機能だが、採用のハードルが高い

### アーキテクチャ決定のフロー

以下の順序で決定を行うことを推奨する。上流の決定が下流の選択肢を制約するため。

```
1. アーキテクチャパターン（モノリス/マイクロサービス/サーバーレス）
   ↓
2. フロントエンド-バックエンド分離戦略（SPA+API / SSR / フルスタックFW）
   ↓
3. レイヤー分離パターン（レイヤード / Clean / Hexagonal）
   ↓
4. ドメインモデリング方針（DDD採否、モデル設計）
   ↓
5. 以下は並行して決定可能:
   - API設計方針
   - 状態管理アーキテクチャ
   - キャッシュ戦略
   - エラーハンドリング方針
   - DI方針
```

### 規模別の推奨アーキテクチャ構成

| 規模 | アーキテクチャ | レイヤー分離 | ドメインモデリング | DI |
|------|-------------|------------|-----------------|-----|
| 個人開発/MVP | モノリス or サーバーレス | レイヤードまたは Feature-based | トランザクションスクリプト | 手動DI or モジュール |
| 小規模チーム（2-5人） | モノリス or モジュラーモノリス | レイヤード + 部分的Clean | Value Object + Repository | 手動DI or Factory |
| 中規模チーム（5-15人） | モジュラーモノリス | Clean or Hexagonal | Rich Domain Model | DIコンテナ or 手動DI |
| 大規模チーム（15人以上） | マイクロサービス or モジュラーモノリス | Clean/Hexagonal per service | フルDDD | DIコンテナ |

### アーキテクチャ選定時の一般原則

1. **シンプルさから始める**: 最もシンプルなアーキテクチャから始め、必要に応じて複雑さを追加する
2. **コンウェイの法則を意識する**: アーキテクチャはチーム構造を反映する。チーム構造に合ったアーキテクチャを選ぶ
3. **可逆性を重視する**: 後から変更困難な決定（DB選択等）は慎重に、変更容易な決定（ライブラリ選択等）は素早く決める
4. **YAGNI**: 「将来必要になるかもしれない」でアーキテクチャを複雑化しない
5. **境界を明確にする**: どのパターンを採用するにせよ、モジュール/レイヤー間の境界を明確に定義する
6. **段階的移行パスを確保する**: モノリス → モジュラーモノリス → マイクロサービスの段階的移行が可能な設計にする
