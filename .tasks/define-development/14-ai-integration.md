## 各項目の詳細

### 1. LLM APIの統合パターン

- **何を決めるか**: フロントエンド/バックエンドからLLM APIをどのように呼び出すか。直接呼び出しかSDK/ラッパー経由か
- **選択肢**:
  - **公式SDK（@anthropic-ai/sdk, openai等）**: 型定義完備。ストリーミング・ツール使用の抽象化。ベンダー固有
  - **AI SDK（Vercel AI SDK）**: マルチプロバイダー対応。useChat/useCompletion hooks。Next.js/React統合。ストリーミングUI対応
  - **直接API呼び出し（fetch/axios）**: 最大の柔軟性。依存関係最小。抽象化なし
  - **LangChain / LlamaIndex**: 高レベル抽象化。RAG・エージェント構築。学習コスト高
- **選定基準**:
  - ベンダーロックインの許容度（マルチプロバイダー対応が必要か）
  - フレームワークとの統合（Next.js → Vercel AI SDKが自然）
  - 必要な機能の複雑さ（単純な生成 vs RAG/エージェント）
  - TypeScript型安全性の要求度
- **トレードオフ・注意点**:
  - **バックエンドファースト原則**: APIキーをフロントエンドに露出させない。必ずバックエンド（API Route / Server Action）経由でLLM APIを呼び出す
  - Vercel AI SDKはストリーミングUIを極めて簡潔に実装できるが、Vercelエコシステムへの依存度が上がる
  - 公式SDKは最新API機能に最速対応するが、ベンダー切替時にコード変更が大きい
  - LangChainは抽象化が深く、デバッグが困難になりやすい。シンプルなユースケースにはオーバーヘッド
  - APIキーの管理: 環境変数 + シークレット管理サービス。決してクライアントコードに含めない
- **2025-2026年のトレンド**:
  - Vercel AI SDK（ai パッケージ）が急成長。streamText / generateText APIでマルチプロバイダー統一が容易
  - Anthropic公式SDKが型定義・ストリーミング・Tool Use対応を強化
  - 「薄いラッパー + 公式SDK」パターンが中規模プロジェクトのベストプラクティス
  - AI Gateway（Cloudflare AI Gateway等）によるプロキシ・キャッシュ・レート制限の一元管理が台頭

### 2. プロンプト管理

- **何を決めるか**: プロンプトテンプレートの保存場所、編集方法、バージョン管理
- **選択肢**:
  - **コード内管理（TypeScript定数/テンプレートリテラル）**: 最もシンプル。型安全。コードレビューで変更管理
  - **テンプレートファイル（.md / .txt / .yaml）**: プロンプトをコードから分離。非エンジニアも編集可能
  - **プロンプト管理プラットフォーム（PromptLayer, Langfuse等）**: Web UI、バージョン管理、A/Bテスト、分析。運用コスト発生
  - **データベース管理**: 動的なプロンプト切替。API経由で取得。デプロイ不要の変更
- **選定基準**:
  - プロンプト変更の頻度（低頻度 → コード内、高頻度 → DB/プラットフォーム）
  - チーム構成（エンジニアのみ → コード内、非エンジニア参画 → UI付きプラットフォーム）
  - A/Bテスト・段階的ロールアウトの必要性
  - プロジェクト規模（個人/小規模 → コード内 or ファイル、大規模 → プラットフォーム）
- **トレードオフ・注意点**:
  - ハードコードは「関心の分離」に反するが、小規模プロジェクトでは最も開発効率が高い
  - テンプレートファイルはGit管理が自然で、PRレビューでプロンプト変更を追跡可能
  - プロンプト管理プラットフォームは強力だが、外部依存・コスト・学習コストが発生
  - プロンプト内の変数展開（ユーザー名、コンテキスト等）の仕組みを統一すること
  - プロンプトのイミュータビリティ: 一度作成したバージョンは変更せず、新バージョンを作成
- **2025-2026年のトレンド**:
  - プロンプトを「コードと同等のアーティファクト」として管理する考え方が定着
  - PromptLayer、Langfuse等のプラットフォームがオブザーバビリティ統合を強化
  - Gitベースのプロンプト管理 + CIでの自動評価パイプラインが小中規模プロジェクトの主流
  - プロンプトのコンポーザビリティ（部品化して組み合わせる）がベストプラクティスに

### 3. AI処理の実行タイミング

- **何を決めるか**: LLM APIの呼び出しを同期的に行うか、非同期的に行うか
- **選択肢**:
  - **同期（リクエスト-レスポンス + ストリーミング）**: ユーザーの操作に対してリアルタイムで応答。チャットUI、インライン補完等
  - **非同期（ジョブキュー）**: リクエストを受け付けて即座にレスポンス。処理完了後に通知。長時間処理向け
  - **バッチ処理**: 複数リクエストをまとめて処理。コスト効率重視。リアルタイム性不要な場合
  - **ハイブリッド**: 短い処理は同期+ストリーミング、長い処理は非同期ジョブ
- **選定基準**:
  - ユーザーの期待するレスポンス速度
  - 処理時間の予測可能性（数秒以内 → 同期、分単位 → 非同期）
  - APIのレート制限への対応
  - コスト最適化の必要性（Batch APIは通常50%割引）
- **トレードオフ・注意点**:
  - 同期+ストリーミングはUXが最も良いが、接続維持のサーバーリソースコストがある
  - 非同期ジョブはキューの管理（Redis/SQS等）とジョブ状態のポーリング/Webhook通知が必要
  - Batch API（Anthropic / OpenAI対応）は結果取得まで最大24時間。バックグラウンド処理・データ分析向け
  - タイムアウト設定: LLM応答は通常5〜60秒。ゲートウェイ/ロードバランサーのタイムアウトを適切に設定
  - AbortControllerでユーザーキャンセルを実装し、不要なAPI消費を防止
- **2025-2026年のトレンド**:
  - ストリーミングが同期処理のデフォルトに。ストリーミングなしの同期呼び出しは例外的
  - Anthropic Batch APIの活用が増加。大量のプロンプト処理を50%割引で実行
  - サーバーレスファンクション（Vercel Functions、Cloudflare Workers）でのストリーミング対応が安定
  - エッジコンピューティングでのLLM推論（小型モデル）も選択肢に

### 4. レスポンスキャッシュ戦略

- **何を決めるか**: LLM APIの応答をキャッシュするか。する場合の方式と条件
- **選択肢**:
  - **完全一致キャッシュ**: 同一プロンプト（ハッシュ）に対してキャッシュ応答を返す。確定的な質問向け
  - **セマンティックキャッシュ**: 意味的に類似したプロンプトに対してキャッシュ応答を返す。ベクトル検索必要
  - **Anthropic Prompt Caching**: APIレベルのプロンプトプレフィックスキャッシュ。長いシステムプロンプトのコスト削減
  - **キャッシュなし**: 毎回新規生成。最新性・多様性重視
- **選定基準**:
  - プロンプトの重複率（FAQ的な質問が多いか、すべてユニークか）
  - コスト削減の優先度
  - レスポンスの鮮度要件
  - 実装の複雑さの許容度
- **トレードオフ・注意点**:
  - 完全一致キャッシュはシンプルだが、ヒット率が低い場合が多い
  - セマンティックキャッシュはヒット率が高いが、ベクトルDB（Pinecone等）が必要で実装・運用コスト増
  - Anthropic Prompt Cachingは長いシステムプロンプト/コンテキストで90%のコスト削減が可能
  - キャッシュのTTL（有効期限）設定: コンテンツの鮮度要件に応じて設定
  - ユーザー固有の文脈を含むプロンプトはキャッシュヒットしにくい
  - キャッシュキーの設計: プロンプト本文 + モデル名 + temperature等のパラメータを含める
- **2025-2026年のトレンド**:
  - Anthropic Prompt Cachingが標準的な最適化手法に。長いシステムプロンプトのプロジェクトでは必須
  - Cloudflare AI Gatewayによるトランスペアレントキャッシュが注目
  - セマンティックキャッシュはRAG文脈で採用が増加
  - 「キャッシュ可能な処理」と「キャッシュ不可能な処理」の明確な分類が設計時の重要ポイント

### 5. コスト管理・モニタリング

- **何を決めるか**: LLM API利用コストの可視化、予算管理、最適化の仕組み
- **選択肢**:
  - **APIプロバイダーのダッシュボード**: Anthropic/OpenAIの管理画面。基本的な使用量確認
  - **カスタムモニタリング**: トークン数・コストをアプリ内でトラッキング。Grafana/Datadog連携
  - **AI Gateway（Cloudflare AI Gateway等）**: プロキシとしてリクエスト/レスポンスを記録。キャッシュ・レート制限も統合
  - **LLMオブザーバビリティツール（Langfuse, Helicone等）**: トークン使用量、レイテンシ、プロンプト/出力の記録
- **選定基準**:
  - 予算規模と超過リスクの許容度
  - チームの運用体制（専任SREがいるか）
  - 詳細な分析の必要性（プロンプト別コスト分析等）
  - プライバシー要件（リクエスト内容を外部サービスに送信可能か）
- **トレードオフ・注意点**:
  - 予算アラートの設計: 50%/80%/100%の段階的アラート + 日次の変化率検知（3倍超で警告）
  - トークン数の事前見積もり: tiktoken等でリクエスト前にトークン数を計算し、上限超過を防止
  - モデルの使い分け: 簡単なタスクには安価なモデル（Haiku等）、複雑なタスクには高性能モデル（Opus等）
  - ユーザー単位の使用量制限を検討（日次/月次のAPI呼び出し上限）
  - ループ・リトライの暴走を防ぐ: 最大リトライ回数とサーキットブレーカーの実装
- **2025-2026年のトレンド**:
  - LLMオブザーバビリティが専門分野として確立。Langfuse/Helicone等のツールが成熟
  - 「トークン単価 x 使用量」の単純計算から、「タスク単位のROI」ベースのコスト管理へ
  - Anthropic/OpenAIのAPIで使用量上限（spend limit）の設定が可能に
  - AIコスト最適化（モデルルーティング、プロンプト圧縮、キャッシュ活用）が専門知識として定着

### 6. レート制限・リトライ戦略

- **何を決めるか**: APIのレート制限への対処方法とエラー時のリトライポリシー
- **選択肢**:
  - **指数バックオフ + ジッター（推奨）**: リトライ間隔を指数的に増加 + ランダムなずれ。サンダリングハード問題を回避
  - **トークンバケット**: クライアント側でリクエストレートを制御。バースト許容 + 平均レート制限
  - **キュー制御**: リクエストをキューに入れ、レート制限内で順次処理
- **選定基準**:
  - 同時ユーザー数とリクエスト頻度
  - APIプロバイダーのレート制限仕様（RPM, TPM）
  - ユーザーへのレスポンス速度の要求
  - システムの複雑さの許容度
- **トレードオフ・注意点**:
  - Anthropic APIのレート制限: Tier別にRPM（リクエスト/分）とTPM（トークン/分）が設定
  - HTTP 429（Too Many Requests）レスポンスの Retry-After ヘッダーを尊重する
  - リトライ上限の設定: 最大3〜5回。無限リトライは禁止（コスト暴走の原因）
  - 冪等性の考慮: LLM APIの呼び出しは基本的に冪等でない（毎回異なる応答）。リトライ時の重複処理に注意
  - サーキットブレーカーパターン: 連続失敗時にリクエストを一時停止し、システム全体の安定性を保護
  - クライアント側のレート制限（デバウンス/スロットル）で不要なリクエストを防止
- **2025-2026年のトレンド**:
  - AI Gatewayによるレート制限の一元管理（アプリコードからの分離）
  - APIプロバイダーのレスポンスヘッダー（x-ratelimit-remaining等）を活用したアダプティブレート制御
  - リクエスト優先度に基づくキュー管理（有料ユーザー優先等）
  - マルチプロバイダーフォールバック（Anthropicがレート制限に達したらOpenAIに切替等）

### 7. フォールバック戦略

- **何を決めるか**: AI機能が利用不可（API障害、レート制限超過、タイムアウト等）の場合の振る舞い
- **選択肢**:
  - **代替モデルフォールバック**: プライマリモデル（Claude Sonnet）が応答しない場合、セカンダリモデル（GPT-4o等）に切替
  - **機能縮退（グレースフルデグラデーション）**: AI機能を無効化し、手動操作で代替可能なUIを提供
  - **キャッシュ応答**: 類似リクエストのキャッシュ応答を返す（セマンティックキャッシュ）
  - **エラー表示 + リトライUI**: 「現在利用できません。後でお試しください」+ リトライボタン
  - **ハイブリッド**: 段階的にフォールバック（代替モデル → キャッシュ → 機能縮退）
- **選定基準**:
  - AI機能のクリティカル度（コア機能 vs 付加価値機能）
  - ベンダー多様化のコスト・複雑さの許容度
  - ユーザーへの影響の許容度
  - 復旧時間の目標
- **トレードオフ・注意点**:
  - AI機能がコア機能の場合、代替モデルフォールバックが必須。ただしマルチベンダー対応のコストが発生
  - AI機能が付加価値の場合、機能縮退が最もシンプルでユーザー影響も限定的
  - フォールバック先のモデルでもプロンプトが正しく動作するかテストが必要（モデル間の互換性）
  - フォールバック発生時のアラート通知を設定し、根本原因の調査を促す
  - ユーザーにAI処理が失敗したことを透明に伝える（黙ってフォールバックしない）
- **2025-2026年のトレンド**:
  - マルチプロバイダールーティング（OpenRouter、LiteLLM等）がフォールバックの標準手段に
  - Vercel AI SDKが複数プロバイダーの切替をコードレベルで容易に
  - 「AI非依存の基本機能 + AIによる拡張」という設計原則が浸透
  - カオスエンジニアリング的なAI障害テスト（AIレスポンスの意図的遅延・エラー注入）の導入

### 8. ストリーミングレスポンス実装

- **何を決めるか**: LLMの応答をリアルタイムで表示するための技術的実装方式
- **選択肢**:
  - **Server-Sent Events（SSE）（推奨）**: HTTP/1.1+。サーバーからクライアントへの一方向ストリーム。LLM APIの標準。最もシンプル
  - **WebSocket**: 双方向通信。チャットアプリで入力中表示等の双方向機能が必要な場合
  - **Fetch Streaming（ReadableStream）**: HTTP/2+。fetch APIのレスポンスボディをストリームとして読み取り
- **選定基準**:
  - 双方向通信の必要性（不要ならSSEが最適）
  - インフラ制約（一部のPaaS/CDNでWebSocketに制約あり）
  - フレームワークとの統合（Vercel AI SDKはSSEベース）
  - 接続の安定性要件
- **トレードオフ・注意点**:
  - SSEの実装:
    - Content-Type: text/event-stream
    - サーバー: テキストチャンクをdata:フィールドで送信
    - クライアント: EventSourceまたはfetchのReadableStreamで受信
  - レンダリングのパフォーマンス最適化:
    - トークンのバッチレンダリング（30〜60msごと、または20〜60文字ごと）でリフロー回避
    - requestAnimationFrameの活用
    - Markdownのインクリメンタルパース（react-markdown + remarkプラグイン）
  - AbortControllerで中断機能を実装。ユーザーが停止ボタンを押した場合にストリームを即座に切断
  - TTFT（Time to First Token）の最適化: 300〜700ms以内が目標
  - エラーハンドリング: ストリーム中断時の部分レスポンスの処理方針を決める
- **2025-2026年のトレンド**:
  - SSEがLLMストリーミングのデファクトスタンダード。Anthropic/OpenAI/Google全てがSSEベース
  - Vercel AI SDKのuseChat/streamTextがストリーミングUIの実装を極めて簡潔に
  - Chrome DevToolsのAI向けストリーミングデバッグ機能が強化
  - サーバーレスエッジ（Cloudflare Workers、Vercel Edge Functions）でのストリーミングが安定

### 9. プロンプトのバージョン管理

- **何を決めるか**: プロンプトの変更履歴管理、ロールバック、A/Bテストの仕組み
- **選択肢**:
  - **Git管理（コード/ファイルと同リポジトリ）**: PRレビューで変更管理。CIで自動テスト。小中規模向け
  - **プロンプトレジストリ（PromptLayer, Langfuse等）**: Web UI管理。バージョン比較。A/Bテスト。トレーシング統合
  - **データベース管理（自前テーブル）**: バージョン番号 + プロンプト本文 + メタデータ。API経由で取得
- **選定基準**:
  - プロンプト変更のデプロイフロー（コードデプロイと同期 vs 独立）
  - A/Bテスト・段階的ロールアウトの必要性
  - プロンプトのトレーサビリティ要件（どのプロンプトバージョンがどの応答を生成したか）
  - プロジェクト規模とチーム構成
- **トレードオフ・注意点**:
  - **イミュータビリティ原則**: 作成済みのプロンプトバージョンは変更しない。変更時は新バージョンを作成
  - Git管理はエンジニアには自然だが、プロンプト変更にコードデプロイが必要
  - レジストリはデプロイ不要でプロンプト変更が可能だが、外部依存・コストが発生
  - トレーサビリティ: リクエストごとにプロンプトバージョンIDをログに記録し、問題発生時の原因追跡を可能にする
  - ロールバック手順を事前に定めておく（新バージョンで品質劣化が発生した場合）
- **2025-2026年のトレンド**:
  - 「Prompt as Code」の考え方が普及。プロンプトもCI/CDパイプラインの対象に
  - Langfuseがオープンソースのプロンプト管理 + オブザーバビリティとして台頭
  - プロンプトの自動評価（CI内でテストセットに対するスコアを算出）が標準プラクティスに
  - プロンプトのA/Bテストフレームワーク（トラフィック分割 + メトリクス比較）が成熟

### 10. AIレスポンスの品質保証・テスト

- **何を決めるか**: LLMの出力品質を担保するためのテスト方法と評価基準
- **選択肢**:
  - **自動評価パイプライン（CI統合）**: テストセット + 期待出力 + 自動スコアリング。デプロイ前チェック
  - **LLM-as-Judge**: 別のLLMで出力品質を評価。人間に近い判定。コスト発生
  - **人手評価**: ドメインエキスパートによるレビュー。最高精度だがスケールしない
  - **ハイブリッド（推奨）**: 自動評価でゲート + 重要変更時に人手レビュー
- **選定基準**:
  - 品質基準の定量化可能性（事実確認 → 自動化しやすい、創造性 → 人手評価が必要）
  - テスト実行の頻度（毎デプロイ → 自動化必須、週次 → 人手可能）
  - コスト・時間の許容範囲
  - 品質リスクの大きさ（医療・法律等は高い品質保証が必要）
- **トレードオフ・注意点**:
  - LLMの出力は非決定的。同一プロンプトでも異なる応答が返る。テストは確率的に評価する必要がある
  - temperature=0 でもモデルの挙動が完全に決定的とは限らない
  - テストセットの設計:
    - 多様な入力シナリオ（正常系、エッジケース、敵対的入力）
    - 期待される出力の定義（完全一致ではなく、含むべき要素/含むべきでない要素）
    - 定期的なテストセットの更新（モデルの世代交代に対応）
  - 評価メトリクス:
    - **正確性**: 事実に基づいているか
    - **一貫性**: 同様の質問に一貫した応答を返すか
    - **ハルシネーション率**: 事実でない情報を生成する割合
    - **安全性**: 有害・不適切なコンテンツを生成しないか
  - リグレッションテスト: プロンプト変更後に既存の品質が劣化しないことを確認
- **2025-2026年のトレンド**:
  - DeepEval、Braintrust等の自動評価フレームワークが成熟
  - LLM-as-Judgeの手法が標準化（評価プロンプトのベストプラクティスが確立）
  - プロンプト変更のCI/CDゲート（スコアが閾値を下回ったらデプロイブロック）が実用段階
  - RAG評価（RAGAs等）がRAGパイプライン品質保証の標準に
  - プロダクション監視（ドリフト検知、品質劣化アラート）がLLMOpsの重要要素に

### 11. ユーザー体験設計（AI応答のUX）

- **何を決めるか**: AI処理中のユーザー体験（ローディング、部分表示、エラー）の設計方針
- **選択肢**:
  - **ストリーミング表示（推奨）**: トークンが生成される都度、テキストを逐次表示。ChatGPT/Claudeと同様のUX
  - **スケルトンUI + 完全応答**: 処理中はスケルトン表示、完了後に一括表示
  - **プログレスバー / ステップ表示**: 処理段階を表示（「分析中...」→「生成中...」→「完了」）
  - **バックグラウンド処理 + 通知**: リクエスト送信後に他の作業が可能。完了通知で結果表示
- **選定基準**:
  - 処理時間の長さ（数秒 → ストリーミング、数十秒〜数分 → バックグラウンド）
  - ユーザーの待機許容度
  - 処理結果の表示形式（テキスト → ストリーミング向き、構造化データ → 完全応答向き）
- **トレードオフ・注意点**:
  - ストリーミング表示のUXベストプラクティス:
    - TTFT（最初のトークンまでの時間）を300〜700ms以内に
    - テキストのちらつき防止: バッチレンダリング（30〜60ms間隔）
    - スクロールの自動追従: 新しいテキストが表示されたらスクロール。ただしユーザーが手動スクロールした場合は追従停止
    - 停止ボタン: ストリーミング中の中断を常に可能に
    - Markdownのインクリメンタルレンダリング: 不完全なMarkdownの処理
  - ローディング状態のアクセシビリティ:
    - aria-live="polite" でスクリーンリーダーに状態変化を通知
    - aria-busy="true" で処理中を明示
  - エラー状態の設計: 「何が起きたか」「ユーザーに何ができるか」を明確に伝える
  - コピーボタン: AI応答テキストのコピー機能は必須級のUX要素
- **2025-2026年のトレンド**:
  - ChatGPT/Claude UIのパターンがユーザーの期待値を形成。ストリーミング表示が「当然」の体験に
  - Chrome DevToolsがLLMストリーミングのレンダリングベストプラクティスを公式ガイドとして公開
  - AI応答のCitation UI（出典表示）がRAGアプリの標準UXに
  - マルチモーダル応答（テキスト + 画像 + コード）の表示パターンが確立

### 12. Claude API固有の検討事項

- **何を決めるか**: Anthropic Claude APIを利用する場合の、モデル選択・機能活用方針
- **選択肢**:

  **モデル選択**:
  - **Claude Opus 4.6（claude-opus-4-6）**: 最高性能。複雑な推論・分析・コード生成。高コスト
  - **Claude Sonnet 4（claude-sonnet-4-20250514）**: 高性能かつコスト効率良好。多くのユースケースに最適
  - **Claude Haiku 3.5（claude-3-5-haiku-20241022）**: 最速・最安。シンプルなタスク、分類、抽出向け

  **Extended Thinking**:
  - 有効化: 複雑な推論タスクで精度向上。thinking content blocksでモデルの思考過程が見える
  - 無効化: シンプルなタスクではオーバーヘッド。レイテンシ・コスト増

  **Tool Use（Function Calling）**:
  - 有効化: 外部API呼び出し、データ取得、計算処理をモデルに委譲
  - 無効化: テキスト生成のみのシンプルなユースケース

- **選定基準**:
  - タスクの複雑さ（複雑 → Opus/Sonnet + Extended Thinking、シンプル → Haiku）
  - レイテンシ要件（リアルタイム → Haiku/Sonnet、バッチ → Opus）
  - コスト予算
  - 外部ツール連携の必要性
- **トレードオフ・注意点**:
  - **モデルルーティング**: タスクの複雑さに応じてモデルを動的に切替。分類タスク → Haiku、要約 → Sonnet、分析 → Opus
  - **Extended Thinking**:
    - tool_choiceは "auto" または "none" のみ対応。"any"/"tool"指定はエラー
    - thinkingブロックをAPIに返送する必要がある（マルチターン会話時）
    - effortパラメータ（low/medium/high）でthinking深度を制御
    - ストリーミング時はthinking_deltaイベントで思考過程を逐次受信
  - **Tool Use**:
    - ツール定義はJSON Schemaで記述
    - ツール実行結果をtool_resultとしてAPIに返送するループが必要
    - Fine-grained tool streamingが全モデルでGA（ベータヘッダー不要）
  - **Prompt Caching**:
    - 長いシステムプロンプト（1024トークン以上）でキャッシュが有効
    - キャッシュヒット時、入力トークンコストが90%削減
    - cache_controlパラメータでキャッシュ対象を指定
  - **Batch API**:
    - 非リアルタイム処理で50%コスト削減
    - 結果取得まで最大24時間
    - 大量のデータ処理・評価タスクに最適
- **2025-2026年のトレンド**:
  - effortパラメータ（adaptive thinking）がGA。budget_tokensの代替として推奨
  - Claude Opus 4.6でinterleaved thinking（ツール使用間の自動思考）がデフォルト有効
  - Fine-grained tool streaming がGA。ストリーミング中のツール呼び出し結果を逐次受信
  - Claude API のmodel ID命名規則: claude-{tier}-{version}形式に統一
  - マルチモーダル入力（画像・PDF）の活用が実用段階に
  - Computer Use（画面操作）APIが特定ユースケースで実用化
