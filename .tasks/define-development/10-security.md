## 各項目の詳細

### 1. 認証方式の選定

- **何を決めるか**: ユーザーの本人確認に使用するプロトコル・方式。アプリケーションの特性（SPA/SSR/モバイル対応）に応じて最適な方式を選ぶ。

- **選択肢**:
  | 方式 | 特徴 | 適用場面 |
  |------|------|---------|
  | セッションベース | サーバー側でセッション状態を保持。Cookie経由でセッションIDを送信 | SSRアプリ、シンプルな構成、即時セッション無効化が必要な場合 |
  | JWT (JSON Web Token) | ステートレス。トークンにユーザー情報を含む。サーバー側で状態管理不要 | SPA、マイクロサービス、スケーラブルな構成 |
  | OAuth 2.0 / OIDC | 外部IdPに認証を委譲。ソーシャルログイン等 | Google/GitHub等のソーシャルログイン、SSO連携 |
  | パスキー (WebAuthn) | FIDO2ベースの生体認証/デバイス認証。パスワード不要 | セキュリティ重視、UX重視の新規アプリ |
  | ハイブリッド | JWT+サーバーサイド検証、セッション+JWTの組み合わせ | JWTの利便性とセッションの即時無効化を両立したい場合 |

- **選定基準**:
  - アプリケーションアーキテクチャ（SPA / SSR / ハイブリッド）
  - セッション即時無効化の要否（JWTはブラックリスト無しでは即時無効化不可）
  - スケーラビリティ要件（JWTはステートレスでスケールしやすい）
  - セキュリティ要件（金融・医療系はセッションベースが推奨される傾向）
  - モバイル/ネイティブアプリとの共存

- **トレードオフ・注意点**:
  - JWTは即時無効化ができない（ブラックリスト管理が必要になり、ステートレスの利点が減少）
  - セッションベースはサーバー間でセッション共有が必要（Redis等の集中ストアが追加要件）
  - パスキーはフォールバック手段（メール/SMS）との組み合わせが必要
  - OAuth 2.0はIdP側の障害がサービスに直結するリスクがある

- **2025-2026年のトレンド**:
  - パスキー(WebAuthn)の本格普及。主要ブラウザ・OSが完全サポートし、金融機関も導入開始。W3C WebAuthn Level 3仕様が策定中
  - ハイブリッドアプローチ（JWTの利便性 + サーバーサイドセッションの安全性を組み合わせる）が実務的な最適解として台頭（Clerk等が採用）
  - パスキー導入により「パスワードリセットチケット32%削減」「SMS OTPコスト削減」「ログイン成功率向上」等の実績報告あり
  - OAuth3の検討が始まっているが、まだドラフト段階

### 2. 認可モデル

- **何を決めるか**: 「認証済みユーザーが何を操作できるか」を制御するモデル。ロールベース、属性ベース、またはシンプルなフラグベースのいずれかを選択する。

- **選択肢**:
  | モデル | 特徴 | 適用場面 |
  |--------|------|---------|
  | RBAC (Role-Based Access Control) | ロール(admin, editor, viewer等)に権限を割当。ユーザーにロールを付与 | 権限体系が明確で変更頻度が低いアプリ |
  | ABAC (Attribute-Based Access Control) | ユーザー属性・リソース属性・環境条件の組み合わせで制御 | 複雑な条件分岐が必要な場合（部署×リソース種別×時間帯等） |
  | 簡易フラグベース | isAdmin等のbooleanフラグで制御 | 小規模アプリ、管理者/一般ユーザーの2区分で十分な場合 |
  | ReBAC (Relationship-Based Access Control) | リソース間のリレーションで制御（「この文書の所有者のみ編集可」等） | Google Drive型の共有モデル、SNS |

- **選定基準**:
  - 権限パターンの複雑さ（2-3ロール → フラグ/簡易RBAC、多段階 → RBAC、条件分岐 → ABAC）
  - 将来の拡張性（ロール追加の頻度、マルチテナント化の可能性）
  - 実装コスト（RBAC < ABAC の順に実装が複雑）
  - デフォルト拒否（deny by default）の原則を採用できるか

- **トレードオフ・注意点**:
  - RBACは「ロール爆発」のリスク（組み合わせが増えると管理が困難）
  - ABACはポリシー定義が複雑になりデバッグが難しい
  - 簡易フラグは手軽だがスケールしない（権限追加のたびにスキーマ変更が必要）
  - OWASP Top 10 2025でも「Broken Access Control」が引き続き第1位。認可のテスト自動化が重要

- **2025-2026年のトレンド**:
  - OWASP Top 10 2025で「Broken Access Control」が引き続きA01（第1位）を維持。認可設計の重要性がさらに認識されている
  - 「デフォルト拒否」原則の徹底、最小権限の原則の強化が推奨されている
  - 認可のテスト自動化ツール・フレームワークの充実（OWASP ZAP等でのアクセス制御テスト）
  - Policy-as-Code（OPA/Rego等）によるポリシー管理のコード化が進展

### 3. 認証サービス/ライブラリの選定

- **何を決めるか**: 認証機能を自前実装するか、外部BaaS(Backend as a Service)を利用するか。利用する場合はどのサービスを選ぶか。

- **選択肢**:
  | サービス | 無料枠 | 特徴 | 適用場面 |
  |---------|--------|------|---------|
  | Clerk | 10,000 MAU | Next.js最適化、プリビルトUI、パスキー対応、40%高速な実装 | Next.js SaaSアプリ |
  | Auth0 (Okta) | 25,000 MAU | エンタープライズSSO、HIPAA対応、豊富な連携先 | エンタープライズ向け、コンプライアンス要件あり |
  | Firebase Auth | 50,000 MAU | Google連携、モバイル対応、匿名認証 | Firebase/GCPエコシステム利用時 |
  | Supabase Auth | 50,000 MAU | PostgreSQL連携、RLS(Row Level Security)統合 | Supabase利用時、RLSベースの認可 |
  | Better Auth | OSS (無料) | TypeScriptネイティブ、セルフホスト、高カスタマイズ性 | 自前制御を重視、ベンダーロックイン回避 |
  | 自前実装 | - | 完全制御、学習目的 | 特殊要件、学習目的（非推奨） |

- **選定基準**:
  - フレームワーク互換性（Next.js → Clerk優勢、Firebase利用中 → Firebase Auth等）
  - 無料枠のMAU数とスケール時のコスト
  - MFA/パスキー/ソーシャルログインの対応状況
  - ベンダーロックインの許容度
  - 監査ログ・Webhook等のエンタープライズ機能の要否

- **トレードオフ・注意点**:
  - 外部サービスはIdP障害時にサービス全体が停止するリスク
  - 自前実装はセキュリティホールのリスクが高い（パスワードハッシュ、トークン管理等）
  - Auth0はソーシャルコネクション数に制限あり（下位プランでは2-3接続）
  - Firebase Authはカスタムクレーム管理がやや煩雑

- **2025-2026年のトレンド**:
  - Clerkが急成長。Next.js App Router/Server Componentsとの統合が最も進んでいる
  - Better Auth等のOSS認証ライブラリの台頭（セルフホスト型でベンダーロックイン回避）
  - 全サービスがパスキー(WebAuthn)対応を進めている
  - 認証サービスのAI活用（不正アクセス検知、リスクベース認証）が始まっている

### 4. パスワードポリシー（またはパスワードレス認証）

- **何を決めるか**: パスワードを使用する場合の要件、またはパスワードレス認証への移行方針。

- **選択肢**:
  | 方針 | 内容 | メリット | デメリット |
  |------|------|---------|-----------|
  | 従来型パスワード | 最低8文字、複雑性要件、定期変更等 | ユーザーに馴染みがある | 漏洩リスク、リスト型攻撃、UX低下 |
  | NIST SP 800-63B準拠 | 最低8文字、複雑性要件なし、定期変更なし、漏洩パスワードチェック | セキュリティとUXのバランス | 漏洩DBとの照合が必要 |
  | パスキー優先 | パスキーをデフォルト、パスワードをフォールバック | フィッシング耐性、UX向上 | パスキー非対応環境への対応が必要 |
  | パスワードレス完全移行 | マジックリンク/パスキー/ソーシャルログインのみ | パスワード関連攻撃を完全排除 | メール/SMS依存、デバイス紛失時の回復手段 |

- **選定基準**:
  - ターゲットユーザー層のリテラシー（パスキー対応デバイスの普及率）
  - セキュリティ要件のレベル（フィッシング耐性が必要か）
  - アカウント回復手段の設計容易性
  - 準拠すべき規格・基準の有無

- **トレードオフ・注意点**:
  - NIST SP 800-63Bは定期的なパスワード変更を「推奨しない」と明記（ユーザーが弱いパスワードを選ぶ原因になるため）
  - パスキーのフォールバック手段がないとアカウントロックアウトのリスク
  - マジックリンクはメール到達性に依存し、UXの遅延がある
  - ソーシャルログインのみの場合、IdPアカウント削除時にサービスアカウントも消失するリスク

- **2025-2026年のトレンド**:
  - パスキーの急速な普及。2026年が「パスワードレスの転換点」と言われている
  - 「Identifier-First」UXパターン（メールアドレス入力 → パスキー有無を自動判定 → 適切な認証方式を提示）が推奨
  - ログイン直後にパスキー設定を促すプロンプトで採用率102%向上の実績
  - FIDO AllianceがCXP(Credential Exchange Protocol)策定中。パスキーのプロバイダー間移行が容易に

### 5. セッション管理

- **何を決めるか**: 認証済みセッションの有効期限、更新方式、セキュリティ要件。

- **選択肢**:
  | 方式 | 有効期限の目安 | 更新方法 | 特徴 |
  |------|-------------|---------|------|
  | サーバーサイドセッション | 15-30分（アイドルタイムアウト） | リクエストごとにスライド延長 | 即時無効化可能、サーバー負荷あり |
  | JWT + リフレッシュトークン | アクセストークン: 15分、リフレッシュ: 7-30日 | リフレッシュトークンで再発行 | ステートレス、即時無効化にはブラックリスト要 |
  | ハイブリッド | アクセストークン: 5-15分 | サーバー側でリフレッシュ検証 | 両方の利点を併せ持つ |

- **選定基準**:
  - セッション即時無効化の要否（不正アクセス検知時にログアウトさせたいか）
  - スケーラビリティ（サーバーサイドセッションは集中ストアが必要）
  - ユーザー体験（ログインの持続時間、再認証の頻度）
  - リフレッシュトークンの安全な保管場所（HttpOnly Cookie推奨）

- **トレードオフ・注意点**:
  - JWTアクセストークンの有効期限が長いと、トークン漏洩時の被害期間が延びる
  - リフレッシュトークンのローテーション（使用ごとに新トークンを発行）が推奨されるが実装が複雑
  - セッションストア（Redis等）の可用性がサービス全体の可用性に影響
  - モバイルアプリではセッション期間を長めに設定するのが一般的

- **2025-2026年のトレンド**:
  - ハイブリッドアプローチ（短寿命JWT + サーバーサイドリフレッシュ検証）が主流化
  - リフレッシュトークンローテーションの標準化
  - デバイスバインド・セッション（セッションを特定デバイスに紐付け）の普及
  - アダプティブ・セッション管理（リスクレベルに応じて有効期限を動的に変更）

### 6. CSRF対策

- **何を決めるか**: クロスサイトリクエストフォージェリ攻撃への防御策。

- **選択肢**:
  | 対策 | 内容 | 有効範囲 |
  |------|------|---------|
  | SameSite Cookie属性 | Cookie送信をサイト間で制限 (Strict/Lax/None) | 主要ブラウザ（Chrome/EdgeはLaxがデフォルト） |
  | CSRFトークン (Synchronizer Token Pattern) | フォームにランダムトークンを埋め込み、サーバーで検証 | 全ブラウザ対応、フォームベースアプリ |
  | Double Submit Cookie | Cookieとリクエストボディに同一トークンを送信 | SPA向け |
  | Fetch Metadata (Sec-Fetch-*ヘッダー) | リクエストの発信元コンテキストをサーバーで検証 | 対応ブラウザのみ |
  | カスタムヘッダー要求 | APIリクエストにカスタムヘッダー(X-Requested-With等)を必須化 | SPA + API構成 |

- **選定基準**:
  - アプリケーション形態（MPA/SSR → CSRFトークン、SPA → カスタムヘッダー/SameSite）
  - ブラウザサポート要件
  - 既存フレームワークの組み込み機能（Next.js, Django, Rails等の標準CSRF保護）
  - 多層防御の方針

- **トレードオフ・注意点**:
  - SameSite=Strictは外部サイトからの正規リンクでもCookieが送られないためUXが低下する場合がある
  - SameSite=Laxはデフォルトで安全だが、GET経由の副作用がある設計では不十分
  - CSRFトークンはステートレスアーキテクチャと相性が悪い
  - SameSite単独ではCSRF対策として不完全。多層防御が推奨される

- **2025-2026年のトレンド**:
  - Chrome/EdgeがSameSite=Laxをデフォルト化済み。Secure属性が第三者Cookie送信の必須条件に
  - Fetch Metadata APIの普及が進み、サーバー側でリクエストの発信元コンテキストを検証可能に
  - SPA+API構成ではSameSite Cookie + Bearer Token（Authorization ヘッダー）の組み合わせが主流
  - フレームワーク組み込みのCSRF保護が成熟し、手動実装の必要性が低下

### 7. XSS対策

- **何を決めるか**: クロスサイトスクリプティング攻撃への多層防御戦略。

- **選択肢**:
  | 対策 | 層 | 内容 |
  |------|---|------|
  | CSP (Content Security Policy) | ブラウザ | スクリプト読み込み元を制限。nonce/hashベースが推奨 |
  | 出力エスケープ | アプリケーション | テンプレートエンジンの自動エスケープ機能を活用 |
  | 入力サニタイズ | アプリケーション | ユーザー入力からスクリプトタグ等を除去（DOMPurify等） |
  | HttpOnly Cookie | ブラウザ/サーバー | JavaScriptからCookieアクセスを禁止 |
  | Trusted Types | ブラウザ | DOM XSSを防止するブラウザAPI |

- **選定基準**:
  - ユーザー生成コンテンツ（UGC）の有無と種類（プレーンテキスト / リッチテキスト / HTML）
  - 使用フレームワークの自動エスケープ機能（React/Vue/Angularは標準で自動エスケープ）
  - インラインスクリプトの使用有無（CSP nonce/hashの導入コストに影響）
  - サードパーティスクリプト（Analytics, 広告SDK等）の利用状況

- **トレードオフ・注意点**:
  - CSPのallowlistベースは管理が煩雑でバイパスされやすい。nonce/hashベース（Strict CSP）が推奨
  - `unsafe-inline`と`unsafe-eval`のCSPディレクティブは使用を避ける
  - リッチテキストエディタを利用する場合はサニタイズライブラリ（DOMPurify）が必須
  - CSPはXSS対策の「唯一の防御策」にしてはいけない。開発プラクティスとの多層防御が原則
  - CSP導入時はまず`Content-Security-Policy-Report-Only`で影響を確認してから強制モードに移行

- **2025-2026年のトレンド**:
  - Strict CSP（nonce/hashベース）が標準的な推奨に。allowlistベースからの移行が進む
  - Trusted Types APIの普及（DOM XSSの根本防止）
  - CSPレポーティング（report-to/report-uriディレクティブ）によるリアルタイム違反検知
  - モダンフレームワーク(React/Next.js等)の自動エスケープ + CSPの多層防御が基本構成

### 8. CORS設定方針

- **何を決めるか**: 異なるオリジンからのリクエストの許可/拒否ポリシー。

- **選択肢**:
  | 方針 | 設定 | 適用場面 |
  |------|------|---------|
  | 同一オリジン限定 | CORSヘッダーなし（デフォルト拒否） | フロントとバックエンドが同一オリジンの場合 |
  | ホワイトリスト方式 | `Access-Control-Allow-Origin`に許可オリジンを明示列挙 | SPA + 別オリジンAPI、複数フロントエンド |
  | 動的オリジン検証 | リクエストのOriginヘッダーを検証し、許可リストに含まれる場合のみ応答 | 複数環境（dev/staging/prod）のサポート |
  | ワイルドカード (`*`) | 全オリジンを許可 | 公開API（認証不要のエンドポイントのみ） |

- **選定基準**:
  - フロントエンドとバックエンドのデプロイ構成（同一オリジン / 別オリジン）
  - 認証Cookieを使用するか（`credentials: include`利用時はワイルドカード不可）
  - 公開APIの有無
  - プリフライトリクエスト（OPTIONS）のパフォーマンス影響

- **トレードオフ・注意点**:
  - ワイルドカード(`*`)と`credentials: include`は同時使用不可（ブラウザが拒否）
  - Origin値の検証が不十分だと、サブドメインテイクオーバー等で悪用されるリスク
  - プリフライトリクエストのキャッシュ（`Access-Control-Max-Age`）で不要なOPTIONSリクエストを削減
  - `Access-Control-Allow-Headers`と`Access-Control-Allow-Methods`も必要に応じて適切に設定

- **2025-2026年のトレンド**:
  - BFF(Backend for Frontend)パターンの普及により、同一オリジン構成が増加しCORS設定が不要になるケースが増えている
  - Private Network Access（旧CORS-RFC1918）のブラウザ実装が進み、ローカルネットワークへのアクセスにもCORS的な制御が必要に
  - `Access-Control-Allow-Private-Network`ヘッダーの導入

### 9. 入力バリデーション戦略

- **何を決めるか**: ユーザー入力の検証をどのレイヤーで、どのように実施するか。

- **選択肢**:
  | レイヤー | 目的 | 主なツール/手法 |
  |---------|------|---------------|
  | フロントエンド | UX向上（即座にフィードバック） | Zod + React Hook Form, Yup, Valibot |
  | バックエンド(API) | セキュリティ強制（信頼境界） | Zod, joi, class-validator, JSON Schema |
  | データベース | データ整合性の最終防御 | NOT NULL, CHECK制約, UNIQUE, 外部キー |
  | スキーマ共有 | フロント/バック間のバリデーション一貫性 | Zod (型推論 + バリデーション一体化) |

- **選定基準**:
  - 多層バリデーション（フロントエンド + バックエンド + DB）が必須。フロントエンドのみは不可
  - バリデーションスキーマの共有可能性（フロントとバックエンドで同一定義を使えるか）
  - 型安全性（TypeScript型との統合）
  - バリデーションエラーメッセージのカスタマイズ性

- **トレードオフ・注意点**:
  - フロントエンドバリデーションは容易にバイパスされるため、セキュリティ目的では使えない
  - バリデーションスキーマの重複管理は保守コストが高い。Zod等で共有するのが推奨
  - 過度に厳しいバリデーションはUXを損なう（国際化対応の氏名フィールド等）
  - ホワイトリスト方式（許可する形式を定義）がブラックリスト方式（拒否する形式を定義）より安全

- **2025-2026年のトレンド**:
  - Zodの事実上の標準化（TypeScript型推論 + バリデーション + シリアライゼーションの統合）
  - Valibotの台頭（Zodと同等の機能でバンドルサイズが小さい）
  - Server ActionsのバリデーションにZodが標準的に使用される（Next.js App Router）
  - OpenAPIスキーマからのバリデーションコード自動生成

### 10. SQLインジェクション対策

- **何を決めるか**: データベースクエリの安全な構築方法。

- **選択肢**:
  | 対策 | 実装方法 | 適用場面 |
  |------|---------|---------|
  | ORM (Object-Relational Mapping) | Prisma, Drizzle, TypeORM等 | 標準的なCRUD操作 |
  | クエリビルダー | Knex.js, Kysely等 | 複雑なクエリ、ORM非対応のパターン |
  | プリペアドステートメント | ドライバレベルのパラメータ化 | 生SQLが必要な場合 |
  | ストアドプロシージャ | DB側でクエリロジックを定義 | パフォーマンス重視、複雑なビジネスロジック |

- **選定基準**:
  - 使用データベースとの互換性
  - 型安全性（TypeScript連携）
  - クエリの複雑さ（単純CRUD → ORM、複雑な集計 → クエリビルダー/生SQL）
  - パフォーマンス要件

- **トレードオフ・注意点**:
  - ORMを使用しても、rawクエリ機能を使う場合はインジェクションリスクが残る
  - テンプレートリテラルでのクエリ組み立ては絶対に避ける
  - OWASP Top 10 2025でInjection(A05)は第5位に下がったが、依然として重大なリスク
  - ORM/クエリビルダーのエスケープ機能に依存しつつ、入力バリデーションも併用する多層防御が推奨

- **2025-2026年のトレンド**:
  - Prisma/Drizzleの普及によりTypeScript型安全クエリが標準化。生SQLを書く機会が減少
  - ORMのデフォルトのパラメータバインディングにより、意図せずインジェクション脆弱性を作る確率が低下
  - OWASP Top 10 2025でInjectionがA03からA05に下がった（フレームワークの自動対策が効果を発揮）
  - それでもrawクエリやdynamic queryには引き続き注意が必要

### 11. シークレット管理

- **何を決めるか**: APIキー、データベース接続文字列、暗号化キー等の機密情報の保管・配布方法。

- **選択肢**:
  | 方式 | 特徴 | 適用場面 |
  |------|------|---------|
  | 環境変数 (.env) | 最もシンプル。プロセス環境変数として注入 | 小規模プロジェクト、ローカル開発 |
  | クラウドシークレットマネージャー | AWS Secrets Manager / GCP Secret Manager / Azure Key Vault | クラウドネイティブアプリ、本番環境 |
  | HashiCorp Vault | 動的シークレット生成、自動ローテーション、監査ログ | エンタープライズ、マルチクラウド |
  | プラットフォーム組み込み | Vercel Environment Variables, Fly.io Secrets, Railway Variables | PaaS利用時 |
  | SOPS (Secrets OPerationS) | Git暗号化管理。KMS等で暗号化した状態でリポジトリに保管 | GitOps、IaC |

- **選定基準**:
  - デプロイ先のインフラ（クラウド → マネージドサービス、PaaS → プラットフォーム組み込み）
  - シークレットのローテーション要件（自動 vs 手動）
  - アクセス制御の粒度（誰がどのシークレットにアクセスできるか）
  - 監査要件（アクセスログの必要性）
  - コスト（Vault/クラウドマネージャーは有料、.envは無料）

- **トレードオフ・注意点**:
  - .envファイルをGitにコミットしない（.gitignoreに追加必須）
  - 環境変数はプロセスの`/proc/PID/environ`から読み取られるリスクがある
  - クラウドシークレットマネージャーはAPIコールが必要なためレイテンシが増加する場合がある（キャッシュで対策）
  - 全シークレットをAES-256で暗号化保存、TLS 1.2以上で転送暗号化が基本

- **2025-2026年のトレンド**:
  - 「シークレットを持たない」アーキテクチャへの移行。Workload Identity Federation（クラウドIAMロール）で静的クレデンシャルを排除
  - Just-In-Time(JIT)アクセス: 短寿命の動的シークレット生成が標準に
  - Vault Secrets Operator（Kubernetes環境でのシークレット同期）の普及
  - .envファイルからの脱却が推奨されている。「シークレットは環境変数ではなく、専用ストアに」が原則に

### 12. 依存関係の脆弱性管理

- **何を決めるか**: サードパーティパッケージの脆弱性検出・修正プロセス。

- **選択肢**:
  | ツール | 特徴 | コスト |
  |--------|------|-------|
  | npm audit | npm組み込み。GitHub Advisory Database参照 | 無料 |
  | GitHub Dependabot | PRによる自動更新提案。GitHub組み込み | 無料（GitHub利用時） |
  | Snyk | 包括的な脆弱性DB、修正提案、優先度付け | 無料枠あり（200テスト/月） |
  | Socket | 振る舞い分析によるマルウェア検知。PRに自動コメント | 無料枠あり |
  | Renovate | 依存関係の自動更新Bot。カスタマイズ性が高い | 無料（OSS） |

- **選定基準**:
  - 検出精度（既知脆弱性 vs 未知マルウェア/振る舞い検知）
  - CI/CDパイプラインへの統合容易性
  - 修正提案の品質（バージョンアップの互換性情報の提供）
  - コスト（個人/小規模チーム → 無料ツール組み合わせ、企業 → Snyk等の有料プラン）

- **トレードオフ・注意点**:
  - npm audit単独ではdevDependenciesのノイズが多い。`--omit=dev`で本番依存のみに絞る
  - npm auditは振る舞いベースのマルウェア検知ができない
  - Dependabotは更新PRが大量に生成されることがあり、マージ戦略の設計が必要
  - 脆弱性の重大度だけでなく、実際のアプリケーションでの到達可能性(reachability)も考慮すべき

- **2025-2026年のトレンド**:
  - OWASP Top 10 2025に「Software Supply Chain Failures」がA03として新設（第3位）。サプライチェーンセキュリティの重要性が急上昇
  - 2024年だけでSnykが3,000以上の悪意あるnpmパッケージを検出。SHA1-Huludワーム等の実例
  - 多層防御: npm audit(ベースライン) + Socket(振る舞い分析) + Snyk/Dependabot(脆弱性管理)の組み合わせが推奨
  - Lockfileの整合性検証、installスクリプトの無効化(`--ignore-scripts`)が標準プラクティスに
  - SBOMの生成・管理が規制要件として浮上

### 13. HTTPセキュリティヘッダー

- **何を決めるか**: ブラウザのセキュリティ機能を有効化するレスポンスヘッダーの設定。

- **選択肢**:
  | ヘッダー | 目的 | 推奨値 |
  |---------|------|--------|
  | `Strict-Transport-Security` (HSTS) | HTTPS強制 | `max-age=31536000; includeSubDomains; preload` |
  | `Content-Security-Policy` (CSP) | XSS防止、リソース読み込み制限 | nonce/hashベースのStrict CSP |
  | `X-Content-Type-Options` | MIMEスニッフィング防止 | `nosniff` |
  | `X-Frame-Options` | クリックジャッキング防止 | `DENY` または `SAMEORIGIN` |
  | `Referrer-Policy` | リファラー情報の漏洩防止 | `strict-origin-when-cross-origin` |
  | `Permissions-Policy` | ブラウザ機能の制限（カメラ、マイク等） | 必要な機能のみ許可 |
  | `X-XSS-Protection` | ブラウザXSSフィルター | `0`（非推奨、CSPで代替） |
  | `Cross-Origin-Opener-Policy` (COOP) | クロスオリジンウィンドウ参照の制限 | `same-origin` |
  | `Cross-Origin-Resource-Policy` (CORP) | クロスオリジンリソース読み込みの制限 | `same-origin` |

- **選定基準**:
  - 最低限のセット: HSTS + X-Content-Type-Options + X-Frame-Options + Referrer-Policy（低リスク、高効果）
  - 推奨セット: 上記 + CSP + Permissions-Policy
  - フルセット: 上記 + COOP + CORP + COEP
  - アプリケーションの要件（iframe埋め込みの要否、第三者スクリプトの利用等）に応じて調整

- **トレードオフ・注意点**:
  - HSTSは一度preloadリストに登録すると解除が困難。段階的に導入すべき（max-ageを短くして検証 → 延長 → preload申請）
  - CSPは厳格すぎるとサードパーティスクリプト（Analytics等）が動作しなくなる
  - X-Frame-OptionsはCSP `frame-ancestors`で代替可能（新しいブラウザ向け）だが、後方互換性のため両方設定するのが安全
  - Permissions-Policyは不要な機能を明示的に無効化することで攻撃面を縮小

- **2025-2026年のトレンド**:
  - CSP frame-ancestorsによるX-Frame-Optionsの代替が進むが、レガシーブラウザ対応で両方設定が推奨
  - COOP/CORP/COEPの「クロスオリジン分離」ヘッダー群が重要性を増している（SharedArrayBuffer等の利用に必要）
  - セキュリティヘッダーのスキャン・監査ツール（securityheaders.com等）の普及
  - Next.js/Vercel等のプラットフォームがデフォルトでセキュリティヘッダーを提供する動きが加速

### 14. レート制限

- **何を決めるか**: API・エンドポイントへのリクエスト数制限の方式と設定値。

- **選択肢**:
  | アルゴリズム | 特徴 | 適用場面 |
  |------------|------|---------|
  | 固定ウィンドウ (Fixed Window) | 一定時間枠ごとにカウンタリセット | シンプルな実装、ウィンドウ境界でバーストの可能性 |
  | スライディングウィンドウ (Sliding Window) | 直近N秒のリクエスト数をカウント | より正確なレート制御、メモリ使用量がやや大きい |
  | トークンバケット (Token Bucket) | 一定レートでトークン補充、トークン消費でリクエスト許可 | バースト許容が必要な場合 |
  | リーキーバケット (Leaky Bucket) | 一定レートでリクエスト処理 | トラフィックの平滑化が必要な場合 |

  | 実装レイヤー | ツール/サービス | 適用場面 |
  |------------|----------------|---------|
  | CDN/エッジ | Cloudflare Rate Limiting, AWS WAF | グローバルな防御、DDoS対策 |
  | APIゲートウェイ | Kong, AWS API Gateway, nginx | サービス全体のレート制限 |
  | アプリケーション | express-rate-limit, upstash/ratelimit | エンドポイント単位の細かい制御 |

- **選定基準**:
  - 保護対象（ログイン試行、API全体、特定の高コスト操作）
  - 識別子（APIキー / JWT / IPアドレス / ユーザーID）
  - 分散環境でのカウンター共有の要否（Redis等）
  - レート超過時のレスポンス（429 Too Many Requests + Retry-After ヘッダー）

- **トレードオフ・注意点**:
  - IPアドレスベースはNAT/共有IPで正規ユーザーが制限されるリスク
  - レート制限が緩すぎると攻撃防止にならず、厳しすぎると正規ユーザーのUXを損なう
  - 429レスポンスにはRetry-Afterヘッダーを含め、クライアントに再試行タイミングを伝える
  - ログイン試行への制限は特に重要（ブルートフォース対策）

- **2025-2026年のトレンド**:
  - 多層レート制限: WAF(エッジ) + APIゲートウェイ + アプリケーションレベルの3層構成が推奨
  - セッションベースのレート制限（IPではなくトークン/セッション単位）への移行
  - AIベースの異常検知との連携（通常パターンからの逸脱を検出してレート制限を動的に適用）
  - Upstash等のサーバーレスレート制限サービスの普及（エッジでのRedis不要のレート制限）

### 15. OWASP Top 10 2025への対応方針

- **何を決めるか**: OWASP Top 10 2025の各カテゴリに対する具体的な対策方針。

- **OWASP Top 10 2025 一覧と対応**:

  | 順位 | カテゴリ | 2021年からの変化 | 主な対策 |
  |------|---------|----------------|---------|
  | A01 | Broken Access Control | 維持(#1) | 認可モデルの設計、デフォルト拒否、アクセス制御テスト |
  | A02 | Security Misconfiguration | 上昇(#5→#2) | セキュリティヘッダー、不要機能の無効化、デフォルト設定の変更 |
  | A03 | Software Supply Chain Failures | **新設** | 依存関係の脆弱性管理、SBOM、サプライチェーン検証 |
  | A04 | Cryptographic Failures | 下降(#2→#4) | TLS 1.2+強制、適切な暗号アルゴリズム選択、シークレット管理 |
  | A05 | Injection | 下降(#3→#5) | ORM/パラメータ化クエリ、入力バリデーション、エスケープ |
  | A06 | Insecure Design | 下降(#4→#6) | 脅威モデリング、セキュアバイデザイン原則 |
  | A07 | Authentication Failures | 維持(#7) | 強力な認証方式、MFA/パスキー、セッション管理 |
  | A08 | Data Integrity Failures | 維持 | CI/CDパイプラインの保護、署名検証 |
  | A09 | Security Logging & Alerting Failures | 維持(#9) | 構造化ログ、セキュリティイベント監視、アラート |
  | A10 | Mishandling of Exceptional Conditions | **新設** | エラーハンドリング設計、フェイルセーフ/フェイルセキュア |

- **選定基準**:
  - プロジェクトのリスクプロファイルに応じて優先度を設定
  - A01(アクセス制御)とA03(サプライチェーン)は全プロジェクトで必須対応
  - OWASP ASVSレベル1を最低基準とし、必要に応じてレベル2/3に拡張

- **トレードオフ・注意点**:
  - 全15項目を完璧に対応するのは現実的でない。リスクベースで優先順位を決める
  - 新設のA03(Software Supply Chain Failures)は従来見落とされがちだった領域。対策の導入コストは低い（ツール導入）
  - A10(Mishandling of Exceptional Conditions)は24のCWEを含む広範なカテゴリ。エラー処理設計の見直しが必要

- **2025-2026年のトレンド**:
  - OWASP Top 10が2025年版に更新。A03「Software Supply Chain Failures」とA10「Mishandling of Exceptional Conditions」が新設
  - SSRF(Server-Side Request Forgery)はA01(Broken Access Control)に統合
  - Security Misconfigurationが第2位に急上昇。クラウド環境の設定ミスが深刻化
  - Injectionが第5位に下降。フレームワークの自動対策が効果を発揮
  - OWASP ASVS 5.0.0が2025年5月にリリース。セキュリティ要件の検証標準が更新
