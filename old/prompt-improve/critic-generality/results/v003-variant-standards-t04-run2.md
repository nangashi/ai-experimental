### Generality Critique Results

#### Critical Issues (Domain over-dependency)
- **Item 1: カート操作の整合性** - "カート" はEC (e-commerce) 業界特有の概念。他業界では適用不可。
- **Item 4: 配送先情報の検証** - "配送先" "郵便番号" は物理商品を扱うEC/物流業界に特化。デジタルサービス、SaaS、内部ツール等には無関係。
- **Problem Bank全体** - 全4問題が "カート" "商品" "在庫" "配送先" "注文" などEC用語で構成され、業界依存度が極めて高い。

#### Scope Item Generality
| Item | Classification | Failed Dimensions | Proposal |
|------|----------------|------------------|----------|
| 1. カート操作の整合性 | Domain-Specific | Industry (e-commerce specific) | 汎用化: "一時的データ操作の整合性" - 一時保存データ (セッション、ドラフト等) の追加・削除・変更が正確に反映されるか |
| 2. 在庫引当のタイミング | Conditional | Industry (inventory management systems) | 汎用化: "リソース引当のタイミング" - 限りあるリソース (座席、枠、ライセンス等) の引当タイミングが明確か |
| 3. 決済処理のエラーハンドリング | Conditional | Industry (payment-enabled systems) | 部分修正: "クレジットカード決済" → "決済処理" に変更。決済機能を持つシステム (EC, SaaS課金, チケット販売等) に適用可能。 |
| 4. 配送先情報の検証 | Domain-Specific | Industry (physical goods delivery), Technology (postal system specific) | 汎用化: "外部連携データの検証" - 外部システム/サービスとの連携データ (住所、ID、コード等) の整合性チェックが実装されているか |
| 5. 注文ステータスの状態遷移 | Conditional | Industry (order processing systems) | 汎用化: "処理ステータスの状態遷移" - ワークフロー処理の状態遷移 (申請→承認→完了等) が正しく設計されているか |

#### Problem Bank Generality
- Generic: 0
- Conditional: 0
- Domain-Specific: 4 (全て)
  - "カートに追加した商品が消失する" → "一時保存したデータが消失する"
  - "在庫切れ商品が購入可能になっている" → "引当済みリソースが重複予約されている"
  - "配送先の郵便番号が不正でもエラーにならない" → "外部連携データの検証エラーが検出されない"
  - "キャンセル済み注文が発送されてしまう" → "キャンセル済み処理が実行されてしまう"

#### Improvement Proposals
- **観点全体の再設計を推奨**: 特定領域依存が2項目以上 (Item 1, 4)、条件付き汎用が3項目 (Item 2, 3, 5) で、観点名 "注文処理の正確性観点" 自体がEC特化。
- **観点名変更**: "注文処理の正確性観点" → "業務処理の整合性観点" または "トランザクション整合性観点"
- **Item 1汎用化**: カートという具体概念を削除し、セッションデータ・ドラフト・一時保存などの抽象概念に置換。
- **Item 2汎用化**: 在庫→リソース (座席予約、会議室、ライセンス枠等の一般概念) に拡張。
- **Item 3部分修正**: "クレジットカード決済" という特定決済手段の明示を削除し、"決済処理" として汎用化。
- **Item 4汎用化**: 配送先・郵便番号という物理配送特化の概念を、外部システム連携の一般的検証パターンに置換。
- **Item 5汎用化**: 注文→処理、発送→実行などワークフロー一般に適用可能な用語に変更。
- **Problem Bank全面改訂**: 全問題例からEC用語を削除し、業界中立な表現に置換 (上記参照)。

#### Positive Aspects
- 項目2「リソース引当」、項目5「状態遷移」は抽象化すれば多くの業界に適用可能な概念
- 項目3「エラーハンドリング」と項目5「状態遷移」はアーキテクチャの一般的パターン
- 観点の基本構造は適切で、用語の汎用化で広く再利用可能になる
