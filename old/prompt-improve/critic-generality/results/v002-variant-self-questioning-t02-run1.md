### Generality Critique Results

#### Critical Issues (Domain over-dependency)
- [項目1: HIPAA準拠の患者データ保護]: 医療業界特有の規制要件に依存
- [項目2: GDPR対応の同意管理]: 地域特有の規制（EU）に依存
- [問題バンク全体]: PHI・患者・診療記録・処方箋など、医療業界特化の用語で構成

#### Scope Item Generality
| Item | Classification | Failed Dimensions | Proposal |
|------|----------------|------------------|----------|
| 1. HIPAA準拠の患者データ保護 | Domain-Specific | Industry, Regulation Dependency | 「個人情報の保護方針」に汎用化。根拠: HIPAAは医療業界特有の規制、「患者」も医療特化用語 |
| 2. GDPR対応の同意管理 | Conditionally Generic | Regulation Dependency | 「ユーザー同意の取得・管理設計」に汎用化。根拠: GDPRはEU地域規制だが、同意管理の概念自体は広く適用可能 |
| 3. アクセス権限の最小化原則 | Generic | なし | 維持。根拠: 最小権限の原則はB2C/内部ツール/OSSライブラリ全てで適用可能 |
| 4. データ保持期間の明確化 | Generic | なし | 維持。根拠: データライフサイクル管理は業界非依存の普遍的概念 |
| 5. 匿名化・仮名化の実装 | Generic | なし | 維持。根拠: プライバシー保護技術は技術スタック・業界非依存 |

#### Problem Bank Generality
- Generic: 0
- Conditional: 0
- Domain-Specific: 3 (list: 「PHIが暗号化されていない」「患者の診療記録が無期限に保存」「処方箋情報が複数部署で共有」)

**問題バンクの評価**: 全3件が医療用語を含む。汎用化例:
- 「PHI」→「個人情報」「機密データ」
- 「患者の診療記録」→「ユーザーの個人データ」
- 「処方箋情報」→「機密情報」

#### Improvement Proposals
- [項目1の汎用化]: 「HIPAA準拠の患者データ保護」→「個人情報の保護方針（機密性・完全性・可用性の設計）」に変更
- [項目2の汎用化]: 「GDPR対応の同意管理」→「個人データ処理に対するユーザー同意の取得・管理設計」に変更
- [問題バンクの全面改訂]: 医療用語を排除し、「個人情報が暗号化されていない」「個人データが無期限保存」「機密情報の共有範囲が未制御」等の汎用表現に変更
- [観点全体の再設計判断]: 特定領域依存が2つ以上（項目1, 2）あり、問題バンクも医療業界に偏重しているため、観点全体の抜本的見直しを推奨

#### Positive Aspects
- 最小権限・データ保持期間・匿名化など、プライバシー保護の基本原則は汎用的に記述されている
- 5項目中3項目が業界・規制非依存の概念で構成されており、基盤は健全
