## ステップ1: 観点の理解

### 主要目的（1文要約）
観点定義の有効性を評価し、レビュー品質への実際の寄与度と既存観点との境界明確性を5段階スコアで定量評価すること。

### 評価スコープ（5項目）
1. **寄与度の分析**: 観点が見逃し防止に貢献する問題を3つ以上列挙し、修正可能で実行可能な改善に繋がるか検証
2. **寄与度スコア（5段階）**: 検出問題の数・重要性・実行可能性に基づく定量評価
3. **境界明確性の検証**: 評価スコープと既存観点の重複箇所を特定し、相互参照の正確性を確認
4. **境界明確性スコア（5段階）**: 重複の程度・相互参照の整合性・境界ケース処理の適切性を定量評価
5. **スコープ適切性スコア（5段階）**: スコープのフォーカス度合い・5項目の一貫性を定量評価

### スコープ外項目と参照先
- 明示的なスコープ外項目の記載なし（評価対象はperspective定義ファイル自体）

## ステップ2: 寄与度の分析

### この観点（プロンプト）がなかった場合に見逃される問題

#### 問題1: 定量評価基準の欠如による評価一貫性の低下
- **内容**: 明確な5段階スコアリング基準がないと、「重大な問題/改善提案/確認」の判定が主観的になり、評価者間の判断が大きくブレる
- **修正可能性**: ✓ スコアリングルーブリックを導入することで一貫性が向上
- **実行可能な改善**: ✓ 具体的な数値基準（「3つ以上の重要な問題」「軽微な重複」など）を明示

#### 問題2: 評価プロセスの構造化不足
- **内容**: ステップ1-4の段階的分析プロセスがないと、評価者が「何を確認すべきか」の認識が曖昧になり、評価漏れが発生
- **修正可能性**: ✓ 評価手順を明示化することで網羅性が向上
- **実行可能な改善**: ✓ 「観点の主要目的を1文で要約」「スコープ5項目を列挙」などの具体的チェックリスト

#### 問題3: 判定根拠の欠如
- **内容**: 各スコアに対応する理由が記載されないと、「なぜこの観点が寄与度3なのか」が不明確になり、改善アクションに繋がらない
- **修正可能性**: ✓ 出力フォーマットに「[1-5]/5 - [理由]」を義務付け
- **実行可能な改善**: ✓ スコアと根拠のセット記述を強制

### スコープの適切性評価
- ステップ1-4は論理的に連結しており、「理解→分析→検証→結論」の流れは妥当
- 5段階スコアリング基準（寄与度・境界明確性・スコープ適切性）は観点評価の主要側面をカバー
- 出力フォーマットはスコア・問題・提案・確認の4要素を包含し、十分にフォーカスされている

## ステップ3: 境界明確性の検証

### 既存観点との照合
**既存観点情報**: 提供されていないため、プロンプト内の記載から推定
- `{existing_perspectives_summary}` 変数が存在 → 他の批評観点（clarity, completeness など）との連携を想定
- 評価例で "Code Quality" / "Security" 観点への言及あり → 一般的なレビュー観点との境界を意識

### 重複の検証
- **ステップ3で明示的に境界確認を要求**: 「評価スコープの5項目と既存観点のスコープを照合」「参照先の観点が実際にその項目をスコープに含んでいるか検証」
- **境界明確性スコアで重複を定量化**: レベル2「重大な重複」、レベル4「軽微な重複」など、重複の許容度を段階的に定義
- **評価例2で具体的な重複ケースを提示**: "Reliability" と "Security" のエラーハンドリング重複を例示

### 相互参照の正確性
- プロンプト内に `{perspective_path}` 変数があり、評価対象のperspectiveファイルを動的に読み込む設計
- `{existing_perspectives_summary}` でコンテキスト情報を提供する前提
- ステップ3で「参照先の観点が実際にその項目をスコープに含んでいるか検証」を明示

### ボーナス/ペナルティ指針のカバレッジ
- プロンプト自体にボーナス/ペナルティ指針は記載なし
- 境界明確性スコアのレベル4で「ボーナス/ペナルティ指針で明確化されている」場合は許容される旨を記載
- 評価例2で「ボーナス判定で両観点で重複する可能性」を指摘 → 境界ケースの認識あり

## ステップ4: 結論の導出

### スコア

#### 寄与度（独自性）スコア: 5/5
**理由**: このプロンプトがなかった場合、以下3つの重要な問題が見逃される。各問題は修正可能で実行可能な改善に繋がる。
1. 定量評価基準の欠如による一貫性低下 → スコアリングルーブリック導入で解決
2. 評価プロセスの構造化不足 → ステップ1-4の明示化で解決
3. 判定根拠の欠如 → 出力フォーマットの標準化で解決

#### 境界明確性スコア: 4/5
**理由**: スコープは明確で、ステップ3で既存観点との重複を明示的に検証する手順が組み込まれている。ただし、`{existing_perspectives_summary}` の具体的なフォーマットや含まれる情報が不明確なため、実際の照合精度に依存する。評価例で境界ケースの扱いを示しているため、実用上は十分に機能する見込み。

#### スコープ適切性スコア: 5/5
**理由**: スコープが適切にフォーカスされ、ステップ1-4および3つのスコア基準すべてが観点の目的（有効性の定量評価）に直接貢献している。出力フォーマットも必要十分な情報を含み、冗長性がない。

### 改善提案（品質向上に有効）

#### 提案1: `{existing_perspectives_summary}` のフォーマット明示化
**理由**: ステップ3で「既存観点のスコープを照合」とあるが、`{existing_perspectives_summary}` が単なるテキスト要約なのか、構造化データ（JSON/表形式）なのか不明。以下の情報を含む標準フォーマットを定義すべき：
- 既存観点名
- 各観点のスコープ5項目（箇条書き）
- スコープ外項目と参照先
これにより、照合精度が向上し、境界明確性スコアの信頼性が高まる。

#### 提案2: スコア算出ロジックの明確化
**理由**: 現在、3つのスコア（寄与度・境界明確性・スコープ適切性）がそれぞれ独立して評価されるが、総合判定（重大な問題/改善提案/確認）との関係が不明確。例えば：
- 「寄与度1または境界明確性1なら『重大な問題』と判定」
- 「すべてのスコアが4以上なら『確認（良い点）』のみ記載」
などの判定ルールを追加すると、出力の一貫性が向上する。

#### 提案3: 評価例の拡充
**理由**: 現在の評価例は2件（明確な寄与度、境界曖昧）のみ。以下のケースを追加すると、評価者の理解が深まる：
- **スコープ不適切ケース**: 5項目中3項目が観点の目的と乖離している例
- **高スコアケース**: すべてのスコアが5の理想的な観点定義
- **境界ケースの詳細例**: ボーナス/ペナルティ指針で重複を解消した成功例

### 確認（良い点）

#### 評価点1: 段階的分析プロセスの明確化
ステップ1-4の構造化により、評価者が「何を確認すべきか」が明確になり、評価漏れを防ぐ設計。特にステップ2で「修正可能で実行可能な改善に繋がるか確認」を明示している点が優れている。

#### 評価点2: 5段階スコアリング基準の具体性
各スコアのレベル1-5に対して、定量的な基準（「3つ以上の重要な問題」「軽微な重複」など）を明示。これにより評価者間の判断ブレが抑制される。

#### 評価点3: 出力フォーマットの標準化
スコア・問題・提案・確認の4要素を含む構造化出力により、レビュー結果の比較可能性が向上。特に「[1-5]/5 - [理由]」の形式で根拠を義務付けている点が評価できる。

#### 評価点4: 評価例による実践的ガイダンス
例1（基本ケース）と例2（困難ケース）で、異なるスコアパターンと判定根拠を示すことで、評価者が具体的な判断方法を理解できる。
