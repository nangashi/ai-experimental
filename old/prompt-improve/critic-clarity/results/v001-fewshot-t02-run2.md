# T02 評価結果

## Input
パフォーマンス観点の観点定義（処理効率、リソース利用、レスポンス性のスコープ、問題バンク含む）

## Analysis Process

### A. 表現の曖昧性
- 主観的表現の多用: 「適切」（計算量、キャッシュ活用）、「最適化されている」（クエリ）、「妥当」（メモリ使用量）、「効果的に」（並列処理）、「損なわない」（応答時間）
- カッコ内の具体的判断基準が一切ない
- 各項目について「何をもって適切/最適とするか」の基準が未定義

### B. AIの動作一貫性
- 「アルゴリズムの計算量が適切か」: AIごとに解釈が分かれる（O(n^2)を許容するAI、O(n log n)を求めるAI等）
- 「データベースクエリが最適化されているか」: インデックス使用を見るAI、N+1問題を見るAI、実行計画を見るAI等、確認対象が一意に定まらない
- 「ユーザー体験を損なわない応答時間」: 数値基準がなく、AIが独自の閾値（100ms、1秒、3秒等）を設定する可能性
- 問題バンクも同様に曖昧（「応答時間が非常に遅い」「一部のコードに改善の余地がある」）

### C. 評価基準の実行可能性
- 評価スコープの各項目が抽象的すぎて、具体的な検出パターンとして機能しない
- 問題バンクの「非常に遅い」「改善の余地がある」は深刻度判定の参考にならない
- 「不要なライブラリ」「不要な同期処理」の「不要」判断基準が不明

## Output

### 明確性批評結果

#### 重大な問題（AIの動作に大きなブレが生じる）
- 「アルゴリズムの計算量が適切か」: 「適切」の判断基準が未定義。AIごとに異なる計算量要件でレビューする可能性大 → 「アルゴリズムの計算量がO(n log n)以下か、またはO(n^2)以上の場合は明示的な性能要件との照合根拠があるか」
- 「データベースクエリが最適化されているか」: 「最適化」の基準が不明確。AIごとに確認観点が異なる可能性 → 「クエリにインデックスが使用されているか（EXPLAIN実行計画で確認）、N+1問題が発生していないか（ループ内クエリ実行の検出）」
- 「ユーザー体験を損なわない応答時間か」: 数値基準なし。AIが独自の許容値を設定する → 「APIエンドポイントの応答時間が200ms以内か（p95）、または設計文書に明記された要件を満たしているか」
- 「メモリ使用量が妥当か」: 「妥当」の判断不可能。測定方法も不明 → 「メモリリークが検出されないか（長時間実行時のメモリ増加傾向チェック）、単一リクエストでのメモリ使用量が想定データ量の10倍以内か」
- 問題バンク「応答時間が非常に遅い」: 「非常に」の基準なし → 「応答時間が3秒以上（ユーザー操作）または30秒以上（バッチ処理）」等の具体値を追加

#### 改善提案
- 「キャッシュが適切に活用されているか」→「キャッシュ可能なデータ（マスタ、設定等）がキャッシュされているか、キャッシュTTLが設定されているか」
- 「並列処理が効果的に実装されているか」→「CPU密集処理が並列化されているか、並列度がCPUコア数と対応しているか」
- 「重い処理がバックグラウンド化されているか」→「応答時間が1秒以上の処理が非同期処理またはジョブキューで実装されているか」
- 問題バンク「一部のコードに改善の余地がある」は軽微問題の例として不適切（曖昧すぎる）→削除または具体例に置換（「ループ内でのオブジェクト生成が繰り返される」等）

#### 確認（良い点）
- スコープ外が明確（詳細なベンチマーク測定、特定ハードウェア最適化を除外）
- 評価対象領域（処理効率、リソース、レスポンス性）の分類は適切
