# T03 評価結果

## Input
コード品質観点の観点定義（可読性、保守性のスコープ、ボーナス/ペナルティ、問題バンク含む）

## Analysis Process

### A. 表現の曖昧性
- 主観的表現: 「意味を表している」（変数名）、「複雑なロジック」、「適度に削減」（重複コード）、「十分なカバレッジ」
- カッコ内に具体的補足があるのは「ネストが深すぎないか（3階層以内を推奨）」のみ
- ボーナス/ペナルティの「高い設計」「過度な使用」も判断基準が不明

### B. AIの動作一貫性
- 「変数名・関数名が意味を表しているか」: AIごとに判断基準が異なる（辞書に存在する単語か、ドメイン用語か、英語として自然か等）
- 「複雑なロジック」: 何をもって複雑とするか不明（ネスト深度、分岐数、行数等）
- 「関数が単一責任を守っているか」: 単一責任の粒度がAIごとに異なる可能性
- 「適度に削減」「バランスを考慮」: AIが独自のバランス感覚で判断する
- 「十分なカバレッジ」: 数値基準なし
- 境界ケースの問題: 「3階層以内を推奨」と「4-5階層は中程度問題」の間（ちょうど3階層、3.5階層のネスト）の扱いが不明確
- 条件付き基準「テストコードがある場合、カバレッジが十分か」: テストがない場合とある場合でAIの判断が分岐し、一貫性が低下

### C. 評価基準の実行可能性
- 「意味を表している」は検出パターンとして機能しない（主観的）
- 「複雑なロジック」も具体的な閾値がないと検出不可能
- 「適度に」「バランスを」「十分な」は実行不可能
- 問題バンクの「関数が200行を超える」「ネストが4-5階層」は具体的で実行可能（良い例）
- 「1文字変数名（ただしループカウンタを除く）」の例外処理がAI間で一貫するかは不確実（forループのi, j以外の例外判断）

## Output

### 明確性批評結果

#### 重大な問題（AIの動作に大きなブレが生じる）
- 「変数名・関数名が意味を表しているか」: 「意味を表す」の判断基準が未定義。AIごとに異なる基準でレビューする可能性 → 「変数名・関数名が英単語または一般的なドメイン用語で構成され、省略語を使う場合はコメントで説明されているか」
- 「複雑なロジックにコメントがあるか」: 「複雑」の定義なし → 「ネストが3階層以上、条件分岐が5箇所以上、またはループ内にさらにループがあるロジックにコメントがあるか」
- 「適度に削減されているか（完全なDRY原則ではなく、バランスを考慮）」: 「適度」「バランス」が曖昧。AIが独自の判断基準を設定する → 「同一処理が3箇所以上で繰り返される場合に関数化されているか、ただし2箇所のみの重複は許容する」
- 「十分なカバレッジ」: 数値基準なし → 「主要な機能（公開API、ビジネスロジック）のカバレッジが80%以上か」
- 境界ケース曖昧性: 「3階層以内を推奨」と「4-5階層は中程度問題」の間のケース（ちょうど3階層、または3階層と4階層の境界）で判断が一致しない可能性 → 「ネストが3階層以下を推奨、4階層以上は改善対象とする」と明確化
- 条件付き基準「テストコードがある場合、カバレッジが十分か」: テストの有無で判断が分岐 → 「テストコードが存在する場合はカバレッジを評価、存在しない場合は『テストが全く存在しない（重大問題）』として扱う」と明示

#### 改善提案
- ボーナス「型安全性が高い設計」: 「高い」の基準不明 → 「TypeScriptでstrictモード有効、またはPythonで型ヒント使用率80%以上」
- ペナルティ「グローバル変数の過度な使用」: 「過度」の基準不明 → 「グローバル変数が5個以上、または複数ファイルから変更される」
- 「1文字変数名（ただしループカウンタを除く）」: 例外範囲が曖昧（i, j, k以外の例外はあるか）→「1文字変数名（for/whileループのi, j, k, n以外）」と明確化

#### 確認（良い点）
- 問題バンクの「関数が200行を超える」「ネストが4-5階層」等の数値基準は明確で、AI間で一貫した判断が可能
- スコープ外が明確（パフォーマンス最適化、チーム固有のルール）
- 「テストが全く存在しない」は検出可能で明確な問題パターン
