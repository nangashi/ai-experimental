# 審議サマリー

## 決定ステートメント

**議題**: Cloudflare Pages + Cloudflare Workers（Hono+TypeScript）構成のリーディングリスト管理PWAにおいて、フロントエンドフレームワークとしてどの技術を採用するか

**スコープ**:
- 対象: フロントエンドのUIフレームワーク/ライブラリの選定。PWA対応（Web Share Target API含む）、レスポンシブ対応（Android+PC）、Cloudflare Pagesとの互換性、学習目的としての適性を評価する
- 対象外: バックエンド技術（Cloudflare Workers + Hono + TypeScript で確定済み — ADR-0001）、ホスティング基盤（Cloudflare Pages で確定済み — ADR-0001）、CSSフレームワーク/UIコンポーネントライブラリ、データベース選定、認証方式

**可逆性**: medium — フレームワーク変更はUIコード全体の書き直しが必要だが、バックエンドAPI（Hono）には影響しない。初期段階（CRUD UIが小規模）での変更コストは比較的低い。

---

## 関連する既存決定

- **ADR-0001: PWA+REST API構成のホスティング基盤選定** — Cloudflare Pages（フロントエンド）+ Cloudflare Workers（バックエンド）を採用する決定。フロントエンドのデプロイ先が Cloudflare Pages に確定しているため、採用するフロントエンドフレームワークは静的ファイル形式（HTML/CSS/JS）としてビルド可能であることが前提条件となる。また、バックエンドが Cloudflare Workers + Hono + TypeScript に確定しているため、フロントエンドはそのREST APIと通信する構成を前提とする。

## 前提条件（CSD）

### 確実（Certainty）

- **C1**: フロントエンドのデプロイ先は Cloudflare Pages に確定済みである — 根拠: ADR-0001 で決定済み
- **C2**: バックエンドは Cloudflare Workers + Hono + TypeScript（REST API）で確定済みである — 根拠: ADR-0001 で決定済み
- **C4**: 月額コストは 0 円（Cloudflare 無料ティア内）の制約が適用される — 根拠: ADR-0001 の C3 より。主要候補フレームワークは全て OSS であり、この制約で除外される候補は実質ない
- **C6**: 画面表示 3 秒以内の要件が引き続き適用される — 根拠: ADR-0001 の C5 より
- **C7**: PWA 対応（マニフェスト + Service Worker）および Web Share Target API のサポートが必要 — 根拠: 決定ステートメントのスコープに明記
- **C8**: レスポンシブ対応（Android + PC）が必要 — 根拠: 決定ステートメントのスコープに明記。CSS レベルの実装であり、主要候補フレームワークはいずれも妨げない
- **C10**: Web Share Target API の受け取りエンドポイントは Cloudflare Workers（Hono）側で実装する — 根拠: research.md の調査結果

### 仮定（Supposition）

- **S1**: 開発者（1名）は React の基本的な知識を持つが、Svelte / SolidJS の実務経験はない — 根拠: 「学習目的としての適性を評価する」という記述から推定 — 検証方法: 開発者に直接確認
- **S2**: アプリの規模は小規模（CRUD UI が中心）であり、長期的に大きく拡張する可能性は低い — 根拠: リーディングリスト管理という用途特性 — 検証方法: 今後のロードマップを確認
- **S3**: 主要候補フレームワーク（React / Vue / Svelte / SolidJS）はいずれも Vite ベースの構成で Cloudflare Pages にデプロイ可能である — 根拠: research.md の調査結果。SvelteKit v2 も Vite を標準採用しており、Vite ベースと見なせる — 検証方法: 選択したフレームワークの公式ドキュメントで確認
- **S4**: vite-plugin-pwa が選択するフレームワークで安定的に動作する — 根拠: vite-plugin-pwa は Vite ベースフレームワークを広くサポート。ただし SvelteKit は専用パッケージ（@vite-pwa/sveltekit）が必要であり、設定方法・統合難易度が異なる可能性がある — 検証方法: 実際の Cloudflare Pages へのデプロイで検証
- **S5**: 選択するフレームワークのメジャーバージョンは今後 1-2 年で大きく変化しない — 根拠: Svelte 5 安定版、React 19 安定版、Vue 3.x 安定。SolidStart は移行期でリスクが高い — 検証方法: 各フレームワークのロードマップを確認
- **S6**: 現構成では静的出力を前提とするが、Cloudflare Pages Functions を利用すれば SSR 構成も実現可能である — 根拠: レッドチーム指摘。SvelteKit 等は @sveltejs/adapter-cloudflare で SSR 対応可能 — 検証方法: SSR を検討する場合、Pages Functions の制約（CPU時間等）を確認
- **S7**: 現時点はシングルユーザー前提だが、D2（チーム化の可能性）により変化しうる — 根拠: レッドチーム指摘。D2 が現実化すると崩れる — 検証方法: D2 の解消に伴い再評価
- **S8**: バックエンドが TypeScript であり、フロントエンドも TypeScript を採用することが強く望ましい — 根拠: バックエンドとの言語統一は合理的だが、フロントエンドへの TypeScript 強制は明示されていない — 検証方法: 決定者に確認
- **S9**: 評価対象は React / Vue / Svelte / SolidJS に限定する — 根拠: Preact / Qwik / Lit 等は暗黙的に除外。主要フレームワークに絞ることで評価コストを抑える判断 — 検証方法: 除外候補が有力な場合は追加検討
- **S10**: Service Worker の実装は vite-plugin-pwa（または同等プラグイン）に委譲できる — 根拠: 手動実装が必要になるシナリオは未検討 — 検証方法: 候補フレームワークでの PWA 実装事例を確認

### 不確実（Doubt）

- **D1**: 開発者の「学習目的」の優先度が、生産性（既存スキルの活用）とどの程度トレードオフになるか明確ではない — 影響範囲: フレームワーク選定の重要な評価軸（OBJ-2 と OBJ-3）に直接影響 — リスク: 学習コストを過小/過大評価した選定
- **D2**: プロジェクトが個人利用を超えてチーム開発に拡張される可能性がある — 影響範囲: エコシステムの大きさ・採用市場でのスキル汎用性の評価（OBJ-5）に影響。S7 と関連 — リスク: 将来のチーム参加時にフレームワーク変更が必要になる可能性
- **D3**: SolidStart 2.0 の API 変更が既存の学習リソースを陳腐化させる速度が不明 — 影響範囲: SolidJS/SolidStart を採用した場合のみ影響 — リスク: ドキュメント混在による学習効率の著しい低下

## 判断基準

優先度は決定者（人間）が設定する。以下は定義のみ。

- **OBJ-1: PWA要件の実現容易性** — マニフェスト・Service Worker・Web Share Target APIを含むPWA要件を、過剰なハックや手動実装なしに実現できること
  - 評価の視点: vite-plugin-pwa（または同等プラグイン）との統合サポートの成熟度 / Web Share Target APIのフロントエンド受け取り処理との相性 / Service Workerのキャッシュ戦略をカスタマイズできる柔軟性 / 公式または信頼できるPWA実装事例・ドキュメントの充実度

- **OBJ-2: 初期立ち上げ速度と開発継続性** — 開発者が既存スキルを活かしつつアプリを完成させられること、または新しいスキルを習得しながらも開発を止めずに進められること
  - 評価の視点: 開発者の既存React知識との連続性（学習コストの初期障壁） / スターターテンプレート・CLIツールの充実度 / エラー時の問題解決情報の網羅度 / 小規模CRUD UIを構築するまでに必要な概念数の少なさ

- **OBJ-3: 新しい概念・パターンの学習価値** — このプロジェクトを通じて、既存スキル（React）と異なるアプローチやアーキテクチャパターンを学習できること
  - 評価の視点: Reactと異なる設計思想（リアクティビティモデル・コンパイル時最適化・シグナルベースなど）の体験可能性 / Web標準に近い書き方を学べるか / 習得した知識の汎用性 / 学習リソースの質と最新性

- **OBJ-4: Cloudflare Pagesとの統合安定性** — Cloudflare Pagesへの静的ビルド・デプロイが安定して動作し、将来的なSSR移行オプションも確保できること
  - 評価の視点: Viteベースの静的出力ビルドの実績 / Cloudflare Pages向けのアダプター・設定の存在と成熟度 / SSR移行が必要になった場合のCloudflare Pages Functions対応状況 / ビルド設定の複雑さ

- **OBJ-5: エコシステムの長期的安定性** — 採用後1-3年間、フレームワーク自体のAPI破壊的変更やコミュニティの縮小によって追加コストが発生しないこと
  - 評価の視点: メジャーバージョンの安定性 / npmダウンロード数・GitHubスター数などのコミュニティ規模 / 大手企業またはOSS財団のサポート有無 / 既存学習リソースの量と陳腐化リスク

## 検討した選択肢

### ALT-1: React（Vite + React 19）

既存スキルを最大限活用できる最大エコシステムのフレームワーク。Vite ベース SPA として Cloudflare Pages に静的デプロイ。SSR 移行時は React Router v7 / OpenNext 経由が必要。
提案元: OBJ-1（情報量の多さ）、OBJ-2（既存スキル活用）、OBJ-4（静的デプロイの安定性）、OBJ-5（最大エコシステム・Meta 支援）

### ALT-2: Vue 3（Vite + Vue）

React 経験者に学習障壁が低く、Composition API で React hooks に近い開発体験を提供。Nuxt が Cloudflare Workers GA に含まれ、SSR 移行パスも存在。
提案元: OBJ-1（React に次ぐ事例数）、OBJ-2（React との近似性による低い移行コスト）、OBJ-3（リアクティビティの違いを学べる）、OBJ-5（安定した 3.x 系）

### ALT-3: Svelte 5 / SvelteKit

コンパイル時リアクティビティという根本的に異なる設計思想。Svelte 5 の Runes システムで現代的なシグナルパターンを体験可能。`@sveltejs/adapter-cloudflare` による公式 Cloudflare 対応が最も手厚い。
提案元: OBJ-1（@vite-pwa/sveltekit 専用パッケージ）、OBJ-3（最高の学習価値）、OBJ-4（Cloudflare GA 対応・SSR 移行が最もシームレス）、OBJ-5（高い開発者満足度）

### ALT-4: SolidJS / SolidStart

React に似た JSX 構文で仮想 DOM を使わないシグナルベースのリアクティビティ。学習価値は高いが、SolidStart 2.0 移行期によるドキュメント混在リスクとエコシステムの小ささが課題。
提案元: OBJ-3（シグナルベースのリアクティビティの純粋な実装）

### 除外した選択肢

なし（S9 により Preact / Qwik / Lit 等は評価対象外としたが、制約による除外ではなく評価コスト削減のための限定）

**ALT-2 × OBJ-1（○）**: vite-plugin-pwa 対応済み。事例数は React に次ぐ規模。実装難易度に本質的な差はない。

**ALT-2 × OBJ-2（○）**: Composition API は React hooks に類似。React 経験者の移行コストが低い。日本語ドキュメント充実。テンプレート構文の学習が初期に必要。1-2 日で基本的な CRUD が書けるようになると想定。

**ALT-2 × OBJ-3（○）**: Proxy ベースリアクティビティ・SFC 構造は React と異なる概念を学べる。ただし React との概念的距離が Svelte より小さく、根本的な設計思想の違いは体験しにくい。

**ALT-2 × OBJ-4（○）**: Vite SPA として CF Pages 静的デプロイが確立。Nuxt が CF Workers GA。SSR 移行時は Nuxt への追加学習が必要だが、React より移行パスが明確。

**ALT-2 × OBJ-5（○）**: 安定した 3.x 系。採用率 17.6% で 2 位。Vue 2→3 の大規模移行を完了済みで近い将来の同規模変更は低確率。React に比べ採用市場・LLM データカバレッジが小さく、大企業の直接支援なし。

**ALT-3 × OBJ-1（○）**: @vite-pwa/sveltekit 専用パッケージ。SvelteKit の navigation preloading が PWA 体験を補完。専用パッケージの設定が React/Vue より複雑で Svelte 5 後の統合事例がまだ少ない（S4 依存）。

**ALT-3 × OBJ-2（△）**: Svelte 構文は HTML/CSS/JS に近い。Svelte 5 Runes は React hooks に近い設計。ただし実務経験なし（S1）、Svelte 5 ドキュメント混在、SvelteKit のファイル構造が複雑で採用率 7.2%。初期学習コストが React・Vue より高い。

**ALT-3 × OBJ-3（◎）**: コンパイル時リアクティビティという根本的に異なる設計思想。Runes で現代的シグナルパターン。Web 標準に近い構文。全評価視点を高いレベルで満たし、転用可能な汎用概念の習得に最適。

**ALT-3 × OBJ-4（◎）**: @sveltejs/adapter-cloudflare で公式対応。CF Workers GA 認定。静的/SSR をアダプター切替のみで対応可能。Cloudflare 公式ドキュメントにデプロイガイドが存在し、SSR 移行が最もシームレス。

**ALT-3 × OBJ-5（△）**: Svelte 5 安定リリース済み。開発者満足度最高。Vercel の支援。採用率 7.2%、Svelte 5 Runes で既存リソース陳腐化リスク、D2（チーム化）時にスキル保有者が少ない（D3 依存）。

**ALT-4 × OBJ-1（△）**: vite-plugin-pwa は SolidJS にも対応。SolidStart 移行期でドキュメント混在、エコシステムが小さく事例が乏しい（D3 依存）。

**ALT-4 × OBJ-2（×）**: JSX 構文で React 経験者に表面的な親和性あり。リアクティビティモデルの根本的な違いで React 経験が「負の転用」に。SolidStart 移行期。エコシステム最小で問題解決コストが最も高い。

**ALT-4 × OBJ-3（△）**: シグナルベースのリアクティビティの純粋な実装。仮想 DOM なし JSX で概念を解体。SolidStart 移行期のドキュメント混在が学習体験を損ない、エコシステムが小さく学習リソース不足。

**ALT-4 × OBJ-4（△）**: start-cloudflare-pages アダプター経由でデプロイ可能。CF Workers サポートがベータ段階、SolidStart 2.0 で設定変更の可能性、CF Workers GA 一覧に未掲載（D3 依存）。

**ALT-4 × OBJ-5（×）**: SolidStart 1.0 は正式リリース済みでシグナルは業界トレンド。採用率 2% 未満、大手企業サポートなし、SolidStart 2.0 移行期で API 変化予想。コミュニティ規模・学習リソース量で他候補と圧倒的な差。

## 判断ガイド

以下は条件付きの帰結の提示であり、推奨ではない。
最終的な優先順位の決定は判断者に委ねられる。

- **OBJ-2（初期立ち上げ速度と開発継続性）を最優先するなら** → ALT-1（React）が適している。ただし OBJ-3（新しい概念の学習）はほぼ得られない。

- **OBJ-3（新しい概念・パターンの学習価値）を最優先するなら** → ALT-3（Svelte 5 / SvelteKit）が適している。ただし OBJ-2（初期立ち上げ速度）にコストがかかり、OBJ-5（長期安定性）にリスクがある。

- **OBJ-4（Cloudflare Pagesとの統合安定性）を最優先するなら** → ALT-3（Svelte 5 / SvelteKit）が適している。ただし OBJ-2（立ち上げ速度）の初期コストとOBJ-5（エコシステム安定性）のリスクを受け入れる必要がある。

- **OBJ-5（エコシステムの長期的安定性）を最優先するなら** → ALT-1（React）が適している。ただし OBJ-3（学習価値）の機会を失い、OBJ-4（SSR移行）に将来コストが発生する可能性がある。

- **OBJ-2 と OBJ-3 のバランスを取るなら** → ALT-2（Vue 3）が適している。React との類似性で立ち上げコストを抑えつつ、異なるリアクティビティモデルを学べる。ただし OBJ-3 の学習深度は ALT-3 より浅く、OBJ-5 は ALT-1 より小さいエコシステムを受け入れることになる。

## 不確実性の影響

以下の不確実性が判明した場合、選択肢の評価が変化する可能性がある:

- **D1（学習目的の優先度）が判明した場合**: 「生産性優先」と判明した場合 → ALT-1（React）の相対的な評価が上昇し、ALT-3（Svelte）の OBJ-2 の低評価がより重大なデメリットになる。「学習優先」と判明した場合 → ALT-3（Svelte）の OBJ-3/OBJ-4 の高評価が正当化され、OBJ-2 の初期コストは許容範囲となる。

- **D2（チーム化の可能性）が判明した場合**: チーム化が確実になった場合 → OBJ-5（エコシステム安定性・採用市場）の重要度が高まり、ALT-1（React）の ◎ 評価が決定的な優位性を持つ。ALT-3（Svelte）の採用率 7.2% は採用市場での制約になりうる。チーム化が不要と確定した場合 → OBJ-5 の評価軸における「採用市場」要素の重みが下がり、ALT-3 の相対的評価が改善する。
