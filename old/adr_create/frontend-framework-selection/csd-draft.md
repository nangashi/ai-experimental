# CSD分類

## 確実（Certainty）

- C1: フロントエンドのデプロイ先は Cloudflare Pages に確定済みである — 根拠: ADR-0001 で決定済み。変更の余地はない
- C2: バックエンドは Cloudflare Workers + Hono + TypeScript（REST API）で確定済みである — 根拠: ADR-0001 で決定済み。フロントエンドはこの REST API と通信する構成を前提とする
- C3: フロントエンドのビルド成果物は Cloudflare Pages にデプロイ可能な静的ファイル形式（HTML/CSS/JS）であること — 根拠: ADR-0001 のホスティング基盤選定に伴う必須要件。ファイルフォーマットの変更は Cloudflare Pages との互換性を壊す
- C4: 月額コストは 0 円（Cloudflare 無料ティア内）の制約が適用される — 根拠: ADR-0001 の C3 より。採用するフレームワークは商用ライセンスや実行時課金を発生させてはならない
- C5: シングルユーザー構成であり、大規模スケーリングは不要 — 根拠: ADR-0001 の C4 より。パーソナル用途の PWA であり、同時接続数・トラフィック規模は最小
- C6: 画面表示 3 秒以内の要件が引き続き適用される — 根拠: ADR-0001 の C5 より。フレームワークのバンドルサイズ・初期ロード速度がこの要件を満たせること
- C7: PWA 対応（マニフェスト + Service Worker）および Web Share Target API のサポートが必要 — 根拠: 決定ステートメントのスコープに明記。リーディングリスト管理アプリとして URL 共有を受け取る機能が要件
- C8: レスポンシブ対応（Android + PC）が必要 — 根拠: 決定ステートメントのスコープに明記。CSSフレームワークは対象外だが、フレームワークがレスポンシブ実装を妨げてはならない
- C9: TypeScript による型安全な開発が前提である — 根拠: バックエンドが TypeScript で統一されており、フロントエンドも同一言語に統一することは明示された前提（スコープの「Hono+TypeScript」記述より）
- C10: Web Share Target API の受け取りエンドポイントは Cloudflare Workers（Hono）側で実装する — 根拠: research.md 観点3より。フロントエンドフレームワークの選定対象ではないことが確認されている

## 仮定（Supposition）

- S1: 開発者（1名）は React の基本的な知識を持つが、Svelte / SolidJS の実務経験はない — 根拠: 決定ステートメントに「学習目的としての適性を評価する」とあり、新規学習を前提としている可能性が高い。ただし具体的なスキルセットは確認されていない — 検証方法: 開発者に直接確認する
- S2: アプリの規模は小規模（CRUD UI が中心）であり、長期的に大きく拡張する可能性は低い — 根拠: 決定ステートメントに「初期段階（CRUD UIが小規模）」とある。リーディングリスト管理という用途の特性上、機能が爆発的に増加する可能性は低い — 検証方法: 今後のロードマップを確認する
- S3: 採用するフレームワークは Vite をビルドツールとして使用できる — 根拠: research.md より、React / Vue / Svelte / SolidJS はいずれも Vite ベースの構成で Cloudflare Pages にデプロイ可能であることが確認されている。ただし SvelteKit は Vite ではなく独自ビルドシステムを持つ点に注意 — 検証方法: 選択したフレームワークの公式ドキュメントで確認
- S4: vite-plugin-pwa が選択するフレームワークで安定的に動作する — 根拠: research.md 観点3より、vite-plugin-pwa が全 Vite ベースフレームワークを統一サポートすることは確認済み。SvelteKit 向けには @vite-pwa/sveltekit が提供されている — 検証方法: 実際の Cloudflare Pages へのデプロイで検証
- S5: 選択するフレームワークのメジャーバージョンは今後 1-2 年で大きく変化しない — 根拠: Svelte 5 は安定版リリース済み、React 19 は安定版リリース済み、Vue 3.x は安定。SolidStart は 1.x → 2.0 移行期にあるためリスクが高い — 検証方法: 各フレームワークのロードマップを確認

## 不確実（Doubt）

- D1: 開発者の「学習目的」の優先度が、生産性（既存スキルの活用）とどの程度トレードオフになるか明確ではない — 影響範囲: フレームワーク選定の重要な評価軸（エコシステム成熟度 vs 新技術習得）に直接影響する。React は即戦力だが学習価値が低く、Svelte は学習価値が高いが習得コストが発生する — リスク: 学習コストを過小/過大評価した選定が後から問題になる可能性
- D2: プロジェクトが個人利用を超えてチーム開発に拡張される可能性がある — 影響範囲: エコシステムの大きさ・採用市場でのスキルの汎用性の評価に影響する。チーム化する場合は React のコミュニティ規模が有利になる — リスク: 現時点でシングルユーザー前提で選定した場合、将来のチーム参加時にフレームワーク変更が必要になる可能性
- D3: SolidStart 2.0 の API 変更が既存の学習リソースを陳腐化させる速度が不明 — 影響範囲: SolidJS/SolidStart を採用した場合のみ影響。学習リソースの混在（1.x 系と 2.x 系）により学習効率が著しく低下する可能性がある — リスク: 採用後に主要 API が変更され、書き直しまたはドキュメント混在による混乱が生じる
