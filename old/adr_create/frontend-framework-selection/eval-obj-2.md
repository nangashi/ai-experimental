# OBJ-2: 初期立ち上げ速度と開発継続性 — 評価結果

## 提案した代替案

- **React (Vite + React)**: 開発者がすでに基本的な React 知識を持つ（S1）ため、学習コストの初期障壁がゼロに近い。`create vite` テンプレートが充実しており、Stack Overflow・GitHub Issues・LLM の学習データ量がすべてのフレームワーク中最大。小規模 CRUD UI を構築するために必要な概念（useState・useEffect・props）は習得済みであり、最速でアプリを完成させられる選択肢。
- **Vue 3 (Vite + Vue)**: JSX を使わないテンプレート構文は React 経験者には学習コストがあるが、その量は小さい。`npm create vue@latest` による公式スターターが充実しており、日本語ドキュメントも整備されている。React 知識の転用度は高く、コンポーネントの概念・ライフサイクル・props/emit パターンは近似している。

## 評価

### React (Vite + React)

- 評価: ◎
- 利点:
  - 開発者の既存 React 知識（S1）が即座に活用できる。フレームワーク学習コストがほぼゼロ
  - `create vite --template react-ts` による TypeScript テンプレートが公式提供されており、ゼロからの立ち上げが 5 分以内に完了する
  - エラー時の問題解決情報が全フレームワーク中最大。Stack Overflow の React タグは 40 万件超、LLM の学習データへの網羅度も最高水準
  - CRUD UI に必要な概念（コンポーネント・useState・props・useEffect・fetch）はすでに習得済みであり、概念習得コストがかからない
- 欠点:
  - 既存スキルの延長であるため、OBJ-3（新しい概念・パターンの学習価値）とのトレードオフが存在する
  - React 19 の新機能（Server Components・Actions）を学ぶ場合は、追加の概念習得が必要になる可能性がある（ただし小規模 SPA では不要）
- 根拠:
  - S1 に基づき、React は開発者の既存スキルと完全に連続している唯一の選択肢
  - research.md の観点1によると、React の npm ダウンロード数・Stack Overflow 採用率（44.7%）はすべての候補中最大であり、問題解決情報の入手容易性が際立っている
- CSD依存:
  - S1（開発者が React の基本知識を持つ）に依存。S1 が崩れ、開発者が React の実務経験がない場合、この評価は ○ に低下する可能性がある
  - D1（学習目的の優先度）に依存しない——OBJ-2 は「開発継続性」を問う観点であり、学習コストの低さは OBJ-2 において純粋なプラスとして評価する

---

### Vue 3 (Vite + Vue)

- 評価: ○
- 利点:
  - コンポーネントベース・props/emit・ライフサイクルなど、React と設計思想が近いため React 経験者には学習障壁が相対的に低い
  - `npm create vue@latest` による公式スターターが充実しており、TypeScript テンプレートも含まれる
  - 日本語ドキュメントが充実しており、日本語話者にとって問題解決情報を入手しやすい
  - Options API / Composition API の両方が使えるため、React の hooks 経験者は Composition API で違和感なく移行できる
- 欠点:
  - テンプレート構文（`<template>` / `v-if` / `v-for`）は React（JSX）と異なる。初期に若干の学習が必要
  - LLM の学習データ量は React に次ぐが、英語圏の Stack Overflow 密度は React より薄い
  - 開発者の既存スキルとの連続性は React に及ばないため、最初の数日間に学習コストが発生する
- 根拠:
  - research.md の観点1によると Vue の採用率は 17.6% で 2 位。エコシステムは十分に成熟しているが、問題解決情報量では React に劣る
  - Composition API の設計は React hooks に類似しており、React 経験者であれば 1〜2 日程度で基本的な CRUD が書けるようになると想定される
- CSD依存:
  - S1（開発者が React の基本知識を持つ）に依存。React 経験者であることが「Vue の学習障壁が低い」という評価の前提

---

### Svelte 5 (SvelteKit)

- 評価: △
- 利点:
  - Svelte の構文は HTML/CSS/JavaScript に近く、「フレームワーク固有の概念を最小化する」設計思想を持つ。中長期的には学習コストが低いとされる
  - `npm create svelte@latest` による公式スターターが提供されており、TypeScript テンプレートも含まれる
  - Svelte 5 の `runes`（$state, $derived, $effect）は React hooks に意図的に近い構文を採用しており、移行障壁を下げようとする設計がある
- 欠点:
  - 開発者は Svelte の実務経験がない（S1）。Svelte の設計思想（コンパイル時リアクティビティ・$runes・SvelteKit のルーティング規約）は React と根本的に異なるため、初期学習コストが発生する
  - Svelte 5 は 2024 年末に安定リリースされたばかりであり、Svelte 5 固有のドキュメント・Stack Overflow の回答・LLM 学習データが Svelte 4 ベースのものと混在している（research.md 観点4）。問題解決時に情報の鮮度を確認する追加コストがかかる
  - SvelteKit の File-based ルーティング・`load` 関数・`+page.svelte` / `+page.server.ts` のファイル命名規約は、React + Vite のシンプルな SPA と比べてファイル構造が複雑であり、初期段階での概念習得コストが高い
  - 採用率 7.2%（research.md 観点1）であり、エラー時の問題解決情報量は React・Vue に比べて少ない
- 根拠:
  - S1 に基づき、Svelte の実務経験がない開発者にとっての初期学習コストは React・Vue より高い
  - Svelte 5 ドキュメント混在リスクは、小規模プロジェクトで問題解決に時間を要するシナリオを生む可能性がある
  - 開発継続性の観点では、立ち上げ直後の数週間に最も学習投資が集中するため、△ とする
- CSD依存:
  - S1（開発者が Svelte の実務経験なし）に依存。S1 が崩れ、開発者に Svelte 経験がある場合は ○ に上昇する可能性がある
  - D1（学習目的の優先度）とも関連するが、OBJ-2 の文脈では「学習コストが開発継続を妨げるか」の観点で評価しており、学習目的自体は OBJ-3 で評価する

---

### SolidJS (SolidStart)

- 評価: ×
- 利点:
  - JSX 構文を採用しており、React 経験者にとって構文面での親和性がある
  - `createSignal` / `createEffect` などのプリミティブは React hooks の構造と表面上似ており、概念の一部は転用できる
- 欠点:
  - React と同じ JSX を使うが、リアクティビティモデルが根本的に異なる（細粒度リアクティビティ・再レンダリングなしの更新）。React 経験者が React と同じ考え方でコードを書くと予期しないバグが発生しやすく、React 経験が「負の転用」になるリスクがある
  - SolidStart 1.x → 2.0.0-alpha 移行期（research.md 観点5）にあり、学習リソース・ドキュメントが 1.x と 2.x で混在している。問題解決情報の信頼性が低下している
  - npm 週次ダウンロード数は約 108〜149 万件で React の 1/10〜1/20（research.md 観点1・5）。LLM の学習データ量・Stack Overflow の回答密度ともに最低水準であり、エラー時の問題解決情報が最も入手しにくい
  - エコシステムの規模が小さいため、依存するサードパーティライブラリ（ルーティング・状態管理・フォーム等）の選択肢が少なく、ライブラリ選定の調査コストも発生する
- 根拠:
  - research.md 観点5 の「SolidStart 2.0 移行期によるドキュメント混在リスク」が、OBJ-2 の「エラー時の問題解決情報の入手容易性」に直接的に悪影響を与える
  - D3（SolidStart 2.0 の API 変更が学習リソースを陳腐化させる速度が不明）が現実化すると、OBJ-2 の評価は × を超えた著しいリスクになる
  - 開発を止めずに進めるという OBJ-2 の定義に照らすと、最も問題解決コストが高い選択肢
- CSD依存:
  - D3（SolidStart 2.0 の API 変更による学習リソース陳腐化速度が不明）に強く依存。D3 が悪化方向に解決すると評価はさらに低下する
  - S5（メジャーバージョンの安定性）に依存。SolidStart については「移行期でリスクが高い」と既に分類されており、S5 の適用対象として最もリスクが高い候補
