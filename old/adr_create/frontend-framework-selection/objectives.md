# 判断基準（Objectives）

## OBJ-1: PWA要件の実現容易性

- 種別: fundamental（根本目的）
- 定義: マニフェスト・Service Worker・Web Share Target APIを含むPWA要件を、過剰なハックや手動実装なしに実現できること
- 評価の視点:
  - vite-plugin-pwa（または同等プラグイン）との統合サポートの成熟度
  - Web Share Target APIのフロントエンド受け取り処理との相性（受け取りエンドポイントはWorkers側だが、フロントエンド側でのルーティング・UI統合の容易さ）
  - Service Workerのキャッシュ戦略をカスタマイズできる柔軟性
  - 公式または信頼できるPWA実装事例・ドキュメントの充実度

## OBJ-2: 初期立ち上げ速度と開発継続性

- 種別: fundamental（根本目的）
- 定義: 開発者が既存スキルを活かしつつアプリを完成させられること、または新しいスキルを習得しながらも開発を止めずに進められること
- 評価の視点:
  - 開発者の既存React知識との連続性（学習コストの初期障壁）
  - スターターテンプレート・CLIツールの充実度（ゼロからの立ち上げの容易さ）
  - エラー時の問題解決情報（Stack Overflow、GitHub Issues、LLMの学習データへの網羅度）
  - 小規模CRUD UIを構築するまでに必要な概念数の少なさ

## OBJ-3: 新しい概念・パターンの学習価値

- 種別: fundamental（根本目的）
- 定義: このプロジェクトを通じて、既存スキル（React）と異なるアプローチやアーキテクチャパターンを学習できること
- 評価の視点:
  - Reactと異なる設計思想（リアクティビティモデル・コンパイル時最適化・シグナルベースなど）の体験可能性
  - Web標準に近い書き方を学べるか（将来の汎用性）
  - 習得した知識が他のフレームワーク・プロジェクトに転用できる汎用性
  - 学習リソース（公式チュートリアル、コミュニティ）の質と現時点での最新性

## OBJ-4: Cloudflare Pagesとの統合安定性

- 種別: fundamental（根本目的）
- 定義: Cloudflare Pagesへの静的ビルド・デプロイが安定して動作し、将来的なSSR移行オプションも確保できること
- 評価の視点:
  - Viteベースの静的出力ビルドの実績（デプロイ互換性）
  - Cloudflare Pages向けのアダプター・設定の存在と成熟度
  - SSR移行が必要になった場合のCloudflare Pages Functions対応状況（S6の対処）
  - ビルド設定の複雑さ（設定ファイルの量・カスタマイズの難易度）

## OBJ-5: エコシステムの長期的安定性

- 種別: fundamental（根本目的）
- 定義: 採用後1-3年間、フレームワーク自体のAPI破壊的変更やコミュニティの縮小によって追加コストが発生しないこと
- 評価の視点:
  - メジャーバージョンの安定性（直近のリリース履歴・破壊的変更の頻度）
  - npmダウンロード数・GitHubスター数・コントリビューター数などのコミュニティ規模
  - バックに大手企業またはOSS財団のサポートがあるか
  - 既存学習リソース（書籍・チュートリアル・LLM学習データ）の量と陳腐化リスク（D3との関連）

## 手段目的→根本目的の変換記録

- 手段: 「TypeScriptを採用したい（フロントエンドもTypeScript）」 → 根本: OBJ-2「開発継続性」のうち「バックエンド（Hono+TypeScript）との型共有による型安全なAPI呼び出し」として組み込み済み — 理由: S8（TypeScript採用は強く望ましい仮定）に分類済みのため、フレームワーク評価軸としてではなく前提として扱う
- 手段: 「vite-plugin-pwaを使いたい」 → 根本: OBJ-1「PWA要件の実現容易性」に統合 — 理由: プラグイン選択は手段であり、PWA要件を実現できることが根本目的
- 手段: 「Reactを使いたい」 → 根本: OBJ-2「初期立ち上げ速度と開発継続性」に統合 — 理由: Reactの選好は既存スキルの活用（立ち上げ速度）と学習目的（OBJ-3）のトレードオフの文脈に吸収される
