# レッドチーム検証結果

## 再分類提案

- C3: Certainty → Supposition — 「静的ファイル形式（HTML/CSS/JS）であること」は ADR-0001 のホスティング選定に伴う要件として自明に見えるが、Cloudflare Pages は Workers Sites や Pages Functions を通じて SSR 的な挙動も実現できる。SvelteKit のように SSG モードと SSR モードを切り替えられるフレームワークを採用する場合、「静的出力のみ」を Certainty として扱うと選択肢を不当に絞る可能性がある。根拠としている「ファイルフォーマットの変更は Cloudflare Pages との互換性を壊す」は正確ではなく、Cloudflare Pages Functions を使えば動的レスポンスも可能である。再分類提案: Supposition「現構成では静的出力を前提とするが、Pages Functions 利用により SSR 構成も実現可能であることに注意が必要」とし、検証方法として「SvelteKit + Cloudflare Pages の公式アダプター（@sveltejs/adapter-cloudflare）のドキュメントを確認する」を付記する

- C5: Certainty → Supposition — 「シングルユーザー構成であり、大規模スケーリングは不要」は現時点の前提だが、D2（チーム開発への拡張可能性）が Doubt として別途挙げられているにもかかわらず C5 が Certainty に置かれることで矛盾が生じている。D2 の不確実性が現実化すると C5 は崩れる。C5 を Certainty として扱うことでスケーラビリティ評価を評価軸から除外するリスクがある。再分類提案: Supposition「現時点はシングルユーザー前提だが、D2 の不確実性により変化しうる」とし、D2 との参照関係を明示する

- C9: Certainty → Supposition — 「TypeScript による型安全な開発が前提」はバックエンドとの統一という合理的な根拠があるが、「フロントエンドも同一言語に統一することは明示された前提」という記述はスコープの「Hono+TypeScript」記述を根拠にしており、これはバックエンドの記述であってフロントエンドへの強制を明示したものではない。決定ステートメントを直接確認したが、フロントエンドへの TypeScript 強制が「確実」と言える明示的な根拠は薄い。再分類提案: Supposition「バックエンドが TypeScript で統一されており、フロントエンドも TypeScript を採用することが強く望ましいが、技術的には必須ではない。決定者に直接確認することを推奨」とし、検証方法を付記する

## 暗黙の仮定の検出

- 「vite-plugin-pwa を PWA 実装に使用する」という仮定が S4 に含まれているが、SvelteKit 向けには別パッケージ（@vite-pwa/sveltekit）が提供されているにも関わらず、S4 の記述が「全 Vite ベースフレームワークを統一サポートする」と表現している。SvelteKit は Vite を内部的に使用するが独自のプラグインシステムを持ち、vite-plugin-pwa の設定方法が大きく異なる（または互換性問題が生じる）可能性がある。推奨分類: Doubt、理由: SvelteKit を選択した場合の PWA 実装の手順が他フレームワークと大きく異なり、「安定的に動作する」という S4 の表現が実態より楽観的な可能性がある。実際の動作検証なしには確認不可能

- 「Vite 以外のビルドツールは考慮不要」という暗黙の仮定がある。S3 に「SvelteKit は Vite ではなく独自ビルドシステムを持つ点に注意」とあるが、SvelteKit が候補に含まれる場合、この前提が一貫していない。SvelteKit の実態は Vite ベース（SvelteKit v2 から Vite を標準採用）であり、「独自ビルドシステム」という記述は現在の実態と乖離している可能性がある。推奨分類: Supposition、理由: SvelteKit のビルドシステムに関する記述の正確性を確認すべき。SvelteKit 公式ドキュメントで現在の Vite 統合状況を検証することを推奨

- 「採用候補は React / Vue / Svelte / SolidJS の4択」という暗黙の仮定がある。CSD 全体を通じてこれらのみが言及されているが、Preact（React 互換・超軽量）、Qwik（再開可能なフレームワーク）、Lit（Web Components ベース）等の選択肢が暗黙的に除外されている。特に C6（3秒以内表示要件）および C4（コスト0円）の観点からは Preact が有力な選択肢になりうる。推奨分類: Supposition「評価対象は React / Vue / Svelte / SolidJS に限定する」とし、選択肢を絞る理由を明示する

- 「Service Worker の実装は vite-plugin-pwa（または同等プラグイン）に委譲できる」という暗黙の仮定がある。C7 で PWA 対応が必要とされているが、Service Worker の実装難易度やフレームワークとの統合複雑性については CSD 上で言及がない。手動実装が必要になるシナリオは検討されていない。推奨分類: Supposition、理由: vite-plugin-pwa を前提とした実装が全候補フレームワークで同等の難易度で実現できるという仮定は検証が必要

## 検証不要と判断した項目

- C1: ADR-0001 で明示的に決定済みの事項であり、今回の検討スコープでの再検討は対象外。Cloudflare Pages の使用は確定事項として扱うことが妥当。検証済み、分類維持

- C2: ADR-0001 で明示的に決定済みの事項であり、バックエンドスタック（Cloudflare Workers + Hono + TypeScript）は確定事項。フロントエンドがこれと通信する構成は現状の最も合理的な前提。検証済み、分類維持

- C4: ADR-0001 の C3 から直接導出されており、Cloudflare の無料ティアを前提とした月額0円の制約は ADR レベルで固定されている。フレームワーク自体は全候補がオープンソース・無料であるため、この制約が評価に与える影響は実質ゼロ（除外されるフレームワークは存在しない）。ただし「商用ライセンスや実行時課金を発生させてはならない」という記述は現在の主要候補（React/Vue/Svelte/SolidJS）全てに適用されないため、制約としての有効性は低い。検証済み、分類維持（ただし有効性の低さを後続フェーズで認識すること）

- C6: ADR-0001 の C5 から引き継がれた要件であり、画面表示3秒以内は明示された非機能要件。この要件はフレームワーク評価の重要な軸（バンドルサイズ・初期ロード速度）として機能する。検証済み、分類維持

- C7: 決定ステートメントのスコープに明記された要件であり、PWA 対応と Web Share Target API サポートは今回の選定目的そのもの。C10 で Web Share Target のエンドポイント実装は Workers 側と明記されており、フロントエンド側の要件は適切に限定されている。検証済み、分類維持

- C8: 決定ステートメントのスコープに明記された要件。レスポンシブ対応は CSS レベルの実装事項であり、主要候補フレームワークのいずれもこれを妨げない。制約として有効。検証済み、分類維持

- C10: research.md の調査結果に基づく分類であり、Web Share Target API のエンドポイント実装を Workers 側で行うことは技術的事実として適切に記録されている。検証済み、分類維持

- S2: 「アプリ規模は小規模で長期的に大きく拡張しない」はリーディングリスト管理の用途特性から合理的な仮定。D2（チーム化の可能性）とは別の軸（機能量）での仮定であり、共存可能。Supposition 分類は適切。ただし D2 との関係（スケールの軸が異なる）を後続フェーズで明示することを推奨。検証済み、分類維持

- S5: 各フレームワークの安定性評価（Svelte 5 安定、React 19 安定、SolidStart 移行期リスク）は research.md の知見と整合しており、妥当な仮定。ただし SolidStart のリスクが Supposition ではなく実質的には Doubt に近い可能性がある（「リスクが高い」と記述されているため）。SolidJS/SolidStart を候補に残すかどうかの判断に影響するため、後続フェーズで SolidStart を採用候補から除外するか否かを明示することを推奨。検証済み、分類維持（SolidStart リスクの扱いは Phase 2 以降で判断）

- D1: 「学習目的の優先度と生産性のトレードオフ」は決定者の価値観に依存する不確実性であり、外部から解消できない典型的な Doubt 項目。分類は適切。検証済み、分類維持

- D2: 「チーム化の可能性」は現時点では判断不能な不確実性であり Doubt 分類は適切。C5 との矛盾については C5 の再分類提案（上記）で対応済み。検証済み、分類維持

- D3: 「SolidStart 2.0 の API 変更速度」は外部要因であり、現時点では判断不能な不確実性。Doubt 分類は適切。SolidJS を候補から除外する根拠の一つとなりうる情報として後続フェーズに引き継ぐことを推奨。検証済み、分類維持
