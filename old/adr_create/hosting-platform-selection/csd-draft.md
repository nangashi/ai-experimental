# CSD分類

## 確実（Certainty）

- C1: 対象はフロントエンド（PWA）とバックエンド（REST API）の2コンポーネント — 根拠: 決定ステートメントのスコープに明記されている。アプリケーション構成として確定している
- C2: データベースサービス・CI/CDパイプライン詳細・ドメイン取得/DNS設定・アプリケーションフレームワークはこの決定のスコープ外 — 根拠: 決定ステートメントの「対象外」に明記されており、今回の選定で考慮しない
- C3: Vercel Hobby プラン（無料）は非商用・個人利用に限定されている — 根拠: Vercel 公式ドキュメント（research.md 観点4）に明示されており、商用プロジェクトでは利用不可
- C4: Railway・Fly.io に永続的な無料枠は存在しない — 根拠: research.md 観点2 に明記。Railway は30日間のトライアルクレジットのみ、Fly.io も無料プランなし
- C5: Render.com 無料枠のウェブサービスは約15分非アクティブでスリープし、コールドスタート時に約25秒の遅延が発生する — 根拠: research.md 観点3 に記載された計測値。REST APIの実用上の問題となりえる事実

## 仮定（Supposition）

- S1: フロントエンドとバックエンドは別プラットフォームに分離して運用できる — 根拠: PWA+REST API の疎結合構成であればCORS設定等で対応可能と考えられる。ただし認証フロー・セッション管理の設計によっては同一プラットフォームの利点が生じる可能性がある。検証方法: アーキテクチャ設計段階でCORS・認証方式を確認する
- S2: アプリケーションコードは標準的なPWA+REST API構成であり、特定プラットフォームへの強い依存はない — 根拠: 決定ステートメントの可逆性評価（medium）の根拠として記載されている。ただし今後の実装でプラットフォーム固有機能を活用した場合は崩れる。検証方法: 実装フェーズでベンダー固有APIの採用状況を追跡する
- S3: プロジェクトは小規模〜中規模のトラフィックを想定しており、大規模な水平スケーリングは直近では不要 — 根拠: ホスティング選定の文脈（個人・小規模向けサービスの比較が中心）から推定。ただし明示されていない。検証方法: プロジェクト要件書や想定ユーザー数を確認する
- S4: 各プラットフォームの現時点の料金・無料枠条件が意思決定期間中（数週間〜数ヶ月）は大きく変わらない — 根拠: research.md の調査結果（2026年2月時点）を前提とする。ただし「料金体系は頻繁に変化」との指摘もある（research.md サマリー5）。検証方法: 決定後も定期的に各社料金ページを確認する

## 不確実（Doubt）

- D1: プロジェクトの商用利用の有無・収益目的の有無 — 影響範囲: Vercel Hobby（無料）の利用可否に直接影響する。商用であれば月$20/ユーザーのVercel Proが必要になり、コスト試算が変わる。リスク: 誤って非商用前提で選定すると、商用化時に追加コストや移行が発生する
- D2: バックエンドの常時稼働要件（コールドスタート許容可否） — 影響範囲: Render.com 無料枠の採用可否に直接影響する。コールドスタート25秒が許容できない場合は有料プラン（$7/月〜）または別サービスが必要。リスク: ユーザー体験要件が不明なまま無料枠前提で選定すると、後から有料プランへの切り替えが必要になる
- D3: チームの想定する予算上限（月次コスト許容額） — 影響範囲: 有料プランの選択可否・複数サービス組み合わせの選択肢に影響する。リスク: 予算制約が厳しい場合、コールドスタートを受け入れた無料構成が唯一の選択肢となる一方、それが要件を満たさない可能性がある
- D4: Netlify・Railway・Render の無料枠における商用利用可否 — 影響範囲: research.md 観点4で「個別確認が必要」とされており、商用利用前提の場合に採用可能なサービスが絞られる。リスク: 商用利用を禁じる条項が判明した場合、選択肢が大幅に制限される
