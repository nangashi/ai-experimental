# OBJ-5: ベンダーロックインリスクの最小化 — 評価結果

## 提案した代替案

OBJ-5（特定プラットフォームへの強い依存を避け、将来的に別サービスへ移行できる余地を持つこと）を最適化する観点から、以下の代替案を提案する。

- **Cloudflare Pages（フロントエンド）+ Cloudflare Workers（バックエンド）**: Cloudflareは業界標準のWAAPI（Workers API）を利用するが、Workers固有の実行環境（Node.js非互換の独自ランタイム）が移行コストを生む。ただし静的ファイル配信（PWA）はプラットフォーム非依存。この目的においてCloudflareの固有APIへの依存度が問題の中心となる。
- **Netlify（フロントエンド）+ Render.com（バックエンド）**: フロントエンドはNetlify（静的配信、標準的なCI連携）、バックエンドはRender.com（標準的なDockerコンテナまたはNode.jsランタイム）。両者ともプラットフォーム固有APIの採用強制度が低く、標準的な構成で動作する。Dockerベースのバックエンドであれば移行が特に容易。
- **Vercel（フロントエンド）+ Render.com（バックエンド）**: Vercel はNext.js等の自社エコシステムと深く統合しているが、純粋な静的PWAホスティングとして使う場合の固有API依存度は低い。Render.comのバックエンドは標準ランタイムに基づくため移行容易。

## 評価

### Vercel（フロントエンド）+ Railway（バックエンド）

- 評価: △
- 利点:
  - Vercelの静的ファイル配信（PWA）自体はプラットフォーム固有依存が少なく、移行時にアプリケーションコードの変更はほぼ不要
  - Railwayはコンテナベースまたは標準ランタイムを採用しているため、バックエンドコードの移行コストは低い
- 欠点:
  - VercelはNext.js等の自社フレームワークと密結合したエコシステムを持つ。純粋なPWA（例: React + Vite）であればロックインリスクは低いが、Vercelの提供する付加機能（Edge Functions、Analytics等）を採用した場合は移行コストが増大する
  - Railwayには永続的無料枠が存在しない（S4）ため、コスト制約（C3）との矛盾によりRailwayを選択できない可能性が高い。評価の前提条件が不安定である
- 根拠: S2（アプリケーションコードの独立性を仮定）に基づけば移行コストは低いが、Vercelの付加機能採用可能性（S2の崩壊リスク）を考慮すると条件付きの評価となる
- CSD依存: S2（アプリコードのプラットフォーム非依存性）に依存。S4（Railwayの無料枠なし）はコスト制約との整合性を脅かし、この組み合わせ自体の実現可能性に影響する

---

### Vercel（フロントエンド）+ Render.com 無料枠（バックエンド）

- 評価: ○
- 利点:
  - Vercelの静的PWAホスティングはHTMLファイルの配置のみであり、プラットフォーム固有の実装強制はない（標準のビルド成果物をそのまま利用）
  - Render.comのバックエンドは標準的なNode.js / Python等のランタイムで動作し、Dockerコンテナにも対応。アプリケーションコードの変更なしに他プラットフォームへの移行が可能
  - データ・設定（環境変数、デプロイ設定）はrender.yamlなどの標準的なファイルでエクスポートできる
- 欠点:
  - Vercel固有の付加機能（Edge Middleware、ISR等）を採用した場合、移行時のコードリライトが必要になる。純粋な静的配信として使うことがロックイン回避の前提条件
  - Render.comの無料枠はスリープ仕様（C7, S5）があり、OBJ-2（レスポンス速度）との両立が困難。この問題を解決するためにping設定等のRender.com依存の運用回避策を採用すると、事実上そのプラットフォームへの依存が深まる
- 根拠: S2に基づき、標準技術のみで構成すればベンダーロックインリスクは低い。ただしVercel/Renderそれぞれで固有機能を使い始めた場合のリスクは中程度
- CSD依存: S2（アプリコードのプラットフォーム非依存性）に依存。D3（フロントエンド・バックエンド分離構成の選択）が確定していることを前提とする

---

### Netlify（フロントエンド）+ Render.com 無料枠（バックエンド）

- 評価: ○
- 利点:
  - Netlifyは静的サイト・SPAのホスティングにおいてプラットフォーム固有の依存強制度が低い。標準的なビルド成果物（HTML/CSS/JS）を配置するだけで動作し、他サービスへの移行はファイルコピーで完了する
  - Render.comのバックエンドはDockerコンテナ/標準ランタイムに対応しており、同様に移行コストが低い
  - NetlifyとVercelを比較した場合、Netlifyは独自フレームワークへの誘導が弱く、純粋な静的ホスティングに近い立ち位置にある
  - 環境変数・リダイレクト設定等はnetlify.tomlで管理でき、設定のポータビリティが高い
- 欠点:
  - Render.com無料枠のスリープ問題は同様に存在する（C7, S5）。OBJ-2との兼ね合いで有料プランへの移行が必要になった場合、Renderへの依存が高まる可能性がある
  - Netlify Functionsを採用した場合、バックエンドロジックがNetlify固有のAPI（Lambda互換）に依存し、移行コストが増大する。ただしRender.comを別途採用するこの構成では関係しない
- 根拠: フロントエンドのNetlifyはロックインリスクが低い選択肢の一つ。Render.comとの組み合わせもほぼ同等の評価となる
- CSD依存: S2に依存。D3（分離構成）を前提とする

---

### Cloudflare Pages（フロントエンド）+ Cloudflare Workers（バックエンド）

- 評価: △
- 利点:
  - フロントエンド（静的ファイル配信）はプラットフォーム非依存。PWAの静的ファイルはどこにでも移行可能
  - Cloudflareは企業規模・実績が大きく、サービス継続性リスクは低い（OBJ-4の観点では優位だが、OBJ-5の直接的な評価には影響しない）
- 欠点:
  - Cloudflare Workersは独自のランタイム（V8ベース、Node.js APIの一部非対応）を持つ。標準的なNode.js/Express APIをそのまま動かすことができず、Workers固有のAPIや制約に合わせたコード記述が必要になるケースがある
  - バックエンドをWorkers上で実装した場合、他プラットフォームへの移行にはコードの再実装・修正が必要になる可能性が高い。このOBJ-5の観点では最もロックインリスクが大きい選択肢
  - KV、D1、R2等のCloudflare独自サービスと組み合わせた場合、移行コストはさらに増大する（今回はDBがスコープ外だが、将来の選択に波及する）
- 根拠: フロントエンドのロックインリスクは低いが、バックエンドをWorkersで実装することで深刻なプラットフォーム依存が生じる。OBJ-5の観点では二面性がある選択肢
- CSD依存: S2（アプリコードの非依存性）が崩壊するリスクが他の選択肢より高い。Workers固有ランタイムへの適応がS2の仮定を実質的に無効にする

---

### Cloudflare Pages（フロントエンド）+ Render.com（バックエンド）

- 評価: ◎
- 利点:
  - Cloudflare Pagesによる静的PWA配信はプラットフォーム固有のAPI採用強制なし。標準的なHTML/CSS/JSのデプロイのみで、他サービスへの移行はビルド成果物の再配置で完了する
  - Render.comのバックエンドは標準ランタイム（Node.js/Python/Docker）に対応し、アプリケーションコードの変更なしに他プラットフォームへの移行が可能
  - フロントエンドとバックエンドで異なるプラットフォームを採用する分離構成のため、一方を移行しても他方に影響しない（部分的な移行が容易）
  - Cloudflare Pagesは無料枠での静的サイト配信において商用利用の制限が緩く（D1・D2の影響を受けにくい）、将来的なコスト変化への耐性もある
- 欠点:
  - 2プラットフォームを組み合わせる運用複雑性はある（OBJ-3の観点では不利）
  - Render.com無料枠のスリープ問題（C7, S5）が存在し、OBJ-2との両立のために有料プランへ移行した場合、Renderへの経済的依存が生じる
- 根拠: 両プラットフォームとも標準技術ベースであり、プラットフォーム固有APIの採用強制度が最も低い組み合わせ。移行時のアプリケーションコード変更量を最小化できる
- CSD依存: S2（アプリコードの非依存性）が前提。D3（分離構成）を採用している場合の評価

---

### 単一PaaS統合（例: Railway のみ / Render のみで両コンポーネント）

- 評価: △
- 利点:
  - 単一プラットフォームに集約した場合、プラットフォーム内のUI・設定・環境変数管理が統一される（OBJ-3の観点では有利）
  - 移行時も同一プラットフォーム内での再設定のみで完結する場合がある
- 欠点:
  - 単一プラットフォームへの依存度が最も高い。そのプラットフォームが料金変更・サービス終了した場合の影響がフロントエンド・バックエンド両方に波及する
  - フロントエンド（静的ファイル配信に最適化）とバックエンド（常時稼働API）の要件の違いを、単一プラットフォームが最適に解決できない場合がある
  - Railway・Fly.ioに永続的無料枠が存在しない（S4）ため、C3（月額0円）制約下では選択できない可能性が高い
- 根拠: 単一ベンダーへの集約はOBJ-5の定義に直接反する。コスト制約（C3）との整合性も低い
- CSD依存: S4（Railway/Fly.ioの無料枠なし）に強く依存。コスト制約C3と矛盾するため実現可能性が低い
