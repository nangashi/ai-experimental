# 審議サマリー

## 決定ステートメント

**議題**: PWA＋REST API構成のWebアプリケーションのホスティング基盤として、どのサービス/方式を採用するか

**スコープ**:
- 対象: フロントエンド（PWA）ホスティング、バックエンド（REST API）ホスティングのサービス選定
- 対象外: データベースサービスの選定、デプロイ方式の設計、アプリケーションフレームワークの選定、CI/CDパイプラインの詳細設計、ドメイン取得・DNS設定

**可逆性**: medium — ホスティング基盤の変更にはデプロイ設定やDB移行が必要だが、アプリケーションコード自体は標準的なPWA+REST API構成であれば大幅な変更は不要。

---

## 前提条件（CSD）

### 確実（Certainty）

- **C1**: 対象はフロントエンド（PWA）とバックエンド（REST API）の2コンポーネント — 根拠: 決定ステートメントのスコープに明記
- **C2**: データベースサービス・デプロイ方式・CI/CD・ドメイン/DNS・フレームワークはスコープ外 — 根拠: 決定ステートメントの「対象外」に明記
- **C3**: 月額予算は0円（無料ティアで完結すること） — 根拠: ユーザー制約として明示
- **C4**: シングルユーザー構成 — 根拠: ユーザー制約として明示。大規模スケーリングは不要
- **C5**: 画面表示3秒以内が必要 — 根拠: ユーザー制約として明示。コールドスタート25秒等は許容不可
- **C6**: Vercel Hobbyプランは非商用・個人利用限定の規約が存在する — 根拠: Vercel公式ドキュメント。ただしこの決定への影響はD1（商用利用の有無）に依存する
- **C7**: Render.com無料枠にはスリープ仕様が存在する — 根拠: 調査結果。非アクティブ時にスリープし、復帰に遅延が発生する仕組みがある

### 仮定（Supposition）

- **S1**: フロントエンドとバックエンドは別プラットフォームに分離して運用できる — 根拠: PWA+REST APIの疎結合構成であればCORS設定等で対応可能。検証方法: アーキテクチャ設計段階でCORS・認証方式を確認
- **S2**: アプリケーションコードは特定プラットフォームへの強い依存はない — 根拠: 決定ステートメントの可逆性評価（medium）の根拠。検証方法: 実装フェーズでベンダー固有API採用状況を追跡
- **S3**: 各プラットフォームの料金・無料枠条件が意思決定期間中は大きく変わらない — 根拠: 2026年2月時点の調査を前提。ただし「料金体系は頻繁に変化」との指摘あり。検証方法: 決定後も定期確認
- **S4**: 2026年2月時点でRailway・Fly.ioに永続的無料枠は存在しない — 根拠: 調査結果。ただし料金体系は変動しうるため時点限定の事実。検証方法: 決定実行前に公式ページ再確認
- **S5**: Render.comのコールドスタート遅延は約25秒 — 根拠: 調査計測値。ただしアプリ構成や仕様変更で変動しうる
- **S6**: 選択肢はパブリッククラウドのPaaS/サーバーレスサービスを主な対象とする — 根拠: 暗黙の前提。VPS・セルフホストは月額0円・運用負荷の観点から除外と推定
- **S7**: 選定サービスは安定的に存続する — 根拠: 暗黙の前提。新興PaaS各社にはサービス終了・買収リスクがある

### 不確実（Doubt）

- **D1**: プロジェクトの商用利用の有無 — 影響範囲: Vercel Hobby（C6）の利用可否に直接影響。商用であれば有料プラン必須でC3（月額0円）と矛盾する
- **D2**: Netlify・Render等の無料枠における商用利用条件の詳細 — 影響範囲: D1が商用の場合、採用可能なサービスが制限される。リスク: 個別確認が必要
- **D3**: フロントエンドとバックエンドを単一プラットフォームに統一するか、別々のサービスを組み合わせるか — 影響範囲: 運用複雑性と最適化のトレードオフ。静的配信とAPI常時稼働では要件の性質が異なる

## 判断基準

優先度は決定者（人間）が設定する。以下は評価に用いた基準の一覧である。

| ID | 目的名 | 定義 |
|----|--------|------|
| OBJ-1 | コスト制約の遵守 | 月額費用を0円（無料ティア内）で維持できること。有料プランへの移行なしに継続運用できること |
| OBJ-2 | レスポンス速度の確保 | ユーザーが操作した際に3秒以内に画面が表示されること。コールドスタートを含む初回アクセスでも許容範囲内であること |
| OBJ-3 | 運用負荷の最小化 | シングルユーザー個人運用として、デプロイ・監視・メンテナンスに必要な作業量が最小であること |
| OBJ-4 | サービス継続性の確保 | 採用したプラットフォームが安定して存続し、突然のサービス終了・大幅な仕様変更リスクが低いこと |
| OBJ-5 | ベンダーロックインリスクの最小化 | 特定プラットフォームへの強い依存を避け、将来的に別サービスへ移行できる余地を持つこと |

## 検討した選択肢

### ALT-1: Cloudflare Pages（FE）+ Cloudflare Workers（BE）

フロントエンド・バックエンドをともにCloudflareのエッジネットワーク上で統一運用。Pages は静的PWA配信、Workers はエッジランタイムでREST APIを実行。コールドスタートが原理的に発生しにくい構成。OBJ-1、OBJ-2、OBJ-3、OBJ-4、OBJ-5 の各観点から提案。主なリスクは Workers が V8 Isolate ベースの独自ランタイムである点（Node.js 非完全互換）とCPU時間制限（無料: 10ms）。

### ALT-2: Netlify（FE）+ Render.com 無料枠（BE）

フロントエンドに Netlify 無料プラン、バックエンドに Render.com 無料枠を組み合わせる分離構成。両者ともGit連携自動デプロイ対応、標準ランタイム（Node.js/Docker）に対応。OBJ-1、OBJ-2、OBJ-3、OBJ-4、OBJ-5 の各観点から提案。主なリスクは Render.com 無料枠のスリープ仕様（C7）によるコールドスタート問題。

### ALT-3: Vercel（FE）+ Render.com 無料枠（BE）

フロントエンドに Vercel Hobby プラン、バックエンドに Render.com 無料枠を組み合わせる分離構成。OBJ-1、OBJ-3、OBJ-4、OBJ-5 の各観点から提案。主なリスクは Vercel Hobby の非商用限定規約（C6）であり、D1（商用利用の有無）次第でC3（月額0円）と矛盾する。

### ALT-4: Cloudflare Pages（FE）+ Render.com 無料枠（BE）

フロントエンドに Cloudflare Pages、バックエンドに Render.com 無料枠を組み合わせる分離構成。標準技術ベースでベンダーロックインリスクが最も低い。OBJ-3、OBJ-5 の各観点から提案。主なリスクは Render.com のコールドスタート問題（ALT-2/3 と同様）と2プラットフォーム管理の運用負荷。

### ALT-5: Vercel（FE）+ Netlify Functions（BE）

フロントエンドに Vercel、バックエンドに Netlify Functions（AWS Lambda ベースのサーバーレス関数）を組み合わせる構成。スリープ仕様はないがコールドスタートは発生しうる。OBJ-2 の観点から提案。主なリスクはコールドスタートの不確実性と D1 による Vercel 利用リスク。

### 評価根拠の詳細

**ALT-1: Cloudflare Pages + Cloudflare Workers**
- OBJ-1 ◎: 両コンポーネントとも永続無料枠。商用利用制限も緩い。月10万リクエスト/日の上限はシングルユーザーでは十分
- OBJ-2 ◎: エッジランタイムでコールドスタートが原理的に発生しにくい。フロント・バック間通信も同一ネットワーク内で低レイテンシ
- OBJ-3 ◎: 単一プラットフォーム統一で管理先が一箇所。Git連携自動デプロイ、SSL自動管理。スリープ回避策も不要
- OBJ-4 ◎: Cloudflare は創業2009年の大規模CDN企業。無料枠廃止実績なし。企業基盤が極めて安定
- OBJ-5 △: Workers の独自ランタイム（V8 Isolate、Node.js 非完全互換）がバックエンドのベンダーロックインリスクを生む。移行時にコード再実装が必要になりうる

**ALT-2: Netlify + Render.com（無料枠）**
- OBJ-1 ○: 両者とも永続無料枠。Netlify は商用利用制限が Vercel より緩い（D2依存）
- OBJ-2 △: Render 無料枠のスリープ仕様（C7）によるコールドスタート問題（S5: 約25秒）。C5（3秒以内）の達成が不確実
- OBJ-3 ○: 両プラットフォームともGit連携自動デプロイ対応。ただし2プラットフォーム管理とスリープ回避設定の追加負荷
- OBJ-4 ○: Netlify（2014年設立）、Render（2019年設立）ともに市場で確立した地位
- OBJ-5 ○: 標準ランタイム（Node.js/Docker）対応。プラットフォーム固有API依存が低く、移行容易

**ALT-3: Vercel + Render.com（無料枠）**
- OBJ-1 △: Vercel Hobby は非商用限定（C6）。D1（商用利用の有無）次第で有料プラン必須（$20/月）となりC3と矛盾
- OBJ-2 △: Render のコールドスタート問題は ALT-2 と同様
- OBJ-3 ○: ALT-2 と同等。Vercel の UI は直感的
- OBJ-4 ○: Vercel（2015年設立）は財務基盤安定。ただし D1 依存により利用継続自体がリスク
- OBJ-5 ○: 純粋な静的PWAホスティングとして使えばロックイン低。Next.js 等の付加機能採用でリスク増

**ALT-4: Cloudflare Pages + Render.com（無料枠）**
- OBJ-1 ○: CF Pages は永続無料枠・商用利用制限緩い。Render 無料枠と合わせて月額0円達成
- OBJ-2 △: フロントエンドは高速CDN配信。バックエンドは Render のコールドスタート問題が残存
- OBJ-3 ○: 2プラットフォーム管理だが両者とも自動化が充実。Render スリープ回避の運用負荷あり
- OBJ-4 ○: CF は企業基盤が極めて安定。Render は中規模新興企業で若干のリスク
- OBJ-5 ◎: 両プラットフォームとも標準技術ベース。プラットフォーム固有API依存が最小。部分移行が容易

**ALT-5: Vercel + Netlify Functions**
- OBJ-1 ○: 両者とも無料枠利用可能。ただし D1 による Vercel リスクは ALT-3 と同様
- OBJ-2 △: Lambda ベースのコールドスタートが発生しうる（数百ms〜数秒）。C5 達成の安定性は不確実
- OBJ-3 △: FE（Vercel）と BE（Netlify）が異なるプラットフォーム。サーバーレス関数のデバッグ・ローカル開発環境の習熟コスト
- OBJ-4 ○: 両社ともサービス存続リスクは低い
- OBJ-5 ○: 標準的な構成だが、Netlify Functions は Lambda 互換で独自API依存が若干ある

## 判断ガイド

以下は条件付きの帰結の提示であり、推奨ではない。最終的な優先順位の決定は判断者に委ねられる。

- **OBJ-1（コスト制約）と OBJ-2（レスポンス速度）の両方を最優先するなら** → ALT-1（CF Pages + CF Workers）が適している。ただし OBJ-5（ベンダーロックインリスク）に懸念が残り、Workers の独自ランタイムへの依存が将来の移行コストを高める可能性がある。
- **OBJ-5（ベンダーロックインリスク）を最優先するなら** → ALT-4（CF Pages + Render）が適している。ただし OBJ-2（レスポンス速度）に重大なリスクがあり、Render のコールドスタート（約25秒）が C5（3秒以内）と矛盾する可能性が高い。
- **OBJ-2（レスポンス速度）の絶対要件（C5: 3秒以内）を厳格に守る必要があるなら** → コールドスタートが構造的に存在する ALT-2/3/4/5 は除外され、ALT-1 のみが現実的な選択肢となる。
- **OBJ-5（ロックインリスク）を許容しつつ、コスト・速度・運用のバランスを取るなら** → ALT-1。ただし Workers ランタイムの制約（CPU 10ms制限）がアプリケーション要件を満たすかを実装前に検証することが重要。
- **商用利用（D1）が確定している場合** → ALT-3 と ALT-5 は Vercel Hobby の規約により除外。残る選択肢は ALT-1、ALT-2、ALT-4 に絞られる。

## 不確実性の影響

以下の不確実性が判明した場合、選択肢の評価が変化する可能性がある:

- **D1（商用利用の有無）が「商用あり」と判明した場合**: ALT-3（Vercel Hobby利用不可→月$20コスト増→OBJ-1 ×）および ALT-5（同様）が実質的に除外される。ALT-1、ALT-2、ALT-4 の比較に絞られ、OBJ-1 の評価は Cloudflare（商用利用制限が緩い）を有利にする方向に動く。
- **D2（Netlify・Render等の商用利用条件の詳細）が「商用利用不可」と判明した場合**: ALT-2（Netlify 商用利用制限→OBJ-1 △に悪化）および ALT-4（Render の商用利用不可→同様）の評価が下がる。ALT-1 の相対的優位性がさらに高まる。
- **D3（単一/分離プラットフォームの選好）が「単一統一を好む」と判明した場合**: ALT-1 のみが統一構成であり、OBJ-3（運用負荷）の観点での評価が他の分離構成と比べて明確に有利になる。「分離構成を好む」場合でも、コールドスタート問題の解決策がない限り OBJ-2 の制約は残存する。
