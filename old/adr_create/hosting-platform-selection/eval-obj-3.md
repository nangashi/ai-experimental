# OBJ-3: 運用負荷の最小化 — 評価結果

## 提案した代替案

- **Cloudflare Pages（フロント）+ Cloudflare Workers（バックエンド）**: 単一プラットフォーム（Cloudflare）でフロントエンドとバックエンドを統合運用できる。Git連携による自動デプロイ、SSL自動管理、CDN配信がすべてCloudflare管理下に統一されるため、複数プラットフォームにまたがる設定管理が不要になる。無料枠でREST APIをWorkers（エッジ関数）として運用可能。
- **Vercel（フロントエンド）+ Render.com 無料枠（バックエンド）**: フロントエンドはVercel（Git push → 自動デプロイ、SSL自動）、バックエンドはRender.com無料枠（Git連携あり、インフラ管理不要）。どちらもGit連携で自動デプロイが可能であり、サーバー設定・SSL証明書更新の手動作業は不要。ただし2プラットフォームにまたがる管理が必要。
- **Netlify（フロント）+ Render.com 無料枠（バックエンド）**: VercelをNetlifyに替えた構成。Netlifyも同様にGit連携・自動デプロイ・SSL自動管理を提供。Netlify Functionsを利用すれば軽量なAPIを同一プラットフォームで動かす選択肢もあるが、REST APIの全機能をFunctionsで賄うかはアプリ要件次第。
- **Cloudflare Pages（フロント）+ Render.com 無料枠（バックエンド）**: フロントエンドをCloudflare Pages、バックエンドをRender.com無料枠に分離。Cloudflare PagesはGit連携・自動デプロイ・グローバルCDN配信に優れる。2プラットフォーム管理の複雑性はあるが、どちらもインフラ設定・SSL更新の手動作業は不要。

## 評価

### ALT-A: Cloudflare Pages（フロント）+ Cloudflare Workers（バックエンド）

- 評価: ◎
- 利点:
  - フロントエンド・バックエンドが単一プラットフォーム（Cloudflare）に統一されるため、ダッシュボード・デプロイ先・SSL管理・ログ確認が一箇所で完結する
  - Git push による自動デプロイ（Cloudflare Pages）でフロントエンドのデプロイ作業がゼロに近い
  - SSL証明書・CDN・セキュリティ設定はCloudflareが自動管理し、手動更新作業は不要
  - Workersはエッジで常時起動（コールドスタートなし）のためバックエンド監視負荷が低い
- 欠点:
  - Workersは従来のREST APIフレームワーク（Express, FastAPI等）とは実行環境が異なる（Service Worker API相当）。既存のバックエンドコードの移植コストがかかりうる
  - Workersのデバッグ・ローカル開発環境（wrangler）の習熟コストが発生する
  - 無料枠のWorkers制限（1日10万リクエスト、CPU時間制限）に注意が必要
- 根拠: 単一プラットフォーム統合により「複数プラットフォームにまたがる設定管理」という最大の運用負荷要因を除去できる。インフラ管理・SSL更新・デプロイいずれも自動化されており、シングルユーザー個人運用の文脈で運用負荷を最も小さくできる構成
- CSD依存:
  - S6（対象はPaaSに限定）: Workersはサーバーレスエッジ関数であり、S6の「PaaS/サーバーレス」範囲に含まれる — 影響なし
  - D3（フロント・バックを統一か分離か）: この選択肢は統一構成であり、D3が「統一」に確定した場合に最適。ただし既存コードの移植要否（S2）次第で評価は変動する

### ALT-B: Vercel（フロントエンド）+ Render.com 無料枠（バックエンド）

- 評価: ○
- 利点:
  - VercelはGit push → 自動デプロイ、SSL自動管理、プレビューURL生成が標準搭載で、フロントエンドの運用負荷が極めて低い
  - Render.comもGit連携による自動デプロイを提供。サーバー設定・SSL更新の手動作業は不要
  - 両プラットフォームともUIが直感的で、設定変更・ログ確認の習熟コストが低い
- 欠点:
  - フロントエンド（Vercel）とバックエンド（Render）の2プラットフォームにまたがる管理が必要。デプロイ状態・ログ・設定の確認先が分散する
  - Render無料枠のスリープ仕様（C7）により、バックエンドが非アクティブ状態から復帰した際の挙動監視やスリープ回避設定（Uptime Robot等）が追加の運用負荷となりうる（S5: コールドスタート約25秒）
  - Vercel Hobby プランの商用利用制限（C6）は、D1（商用利用の有無）次第では運用継続に影響する
- 根拠: 主要なインフラ管理・SSL作業は自動化されており、基本的な運用負荷は低い。ただし2プラットフォーム管理と、Renderのスリープ回避設定がOBJ-3の観点で懸念点となる
- CSD依存:
  - D3（単一か分離か）: この構成は「分離」前提。D3が「統一」に確定した場合はプラットフォームが変わる
  - S5（Renderのコールドスタート約25秒）: スリープ回避設定の要否に影響。Uptime Robot設定という追加作業が発生するかが運用負荷に関わる
  - C7（Render無料枠のスリープ仕様）: スリープ仕様の存在は確定事実だが、OBJ-3への影響度はD3・S5に依存

### ALT-C: Netlify（フロント）+ Render.com 無料枠（バックエンド）

- 評価: ○
- 利点:
  - NetlifyもGit連携・自動デプロイ・SSL自動管理を提供し、フロントエンドの運用負荷はVercel同等に低い
  - Netlify Functionsを使えば軽量なAPIロジックを同一プラットフォーム内で管理できる余地がある
  - Render.comとの組み合わせではALT-Bと同様のGit連携自動デプロイが利用可能
- 欠点:
  - ALT-Bと同様に2プラットフォーム管理の複雑性がある
  - Netlify Functionsによる統一化を選んだ場合、Netlify FunctionsのREST API対応範囲・制限（無料枠では月125,000リクエスト、実行時間10秒）の把握が必要となり、アプリ要件との適合確認コストが発生する
  - Render無料枠のスリープ問題はALT-Bと同様に残る
- 根拠: 基本的な自動化レベルはVercelと同等。Netlify Functionsによる部分的な統一の可能性があるが、それを選ぶ場合は制限確認という運用コストが追加される。全体的な運用負荷はALT-Bとほぼ同等
- CSD依存:
  - D3（単一か分離か）: ALT-Bと同様
  - S5、C7: ALT-Bと同様

### ALT-D: Cloudflare Pages（フロント）+ Render.com 無料枠（バックエンド）

- 評価: ○
- 利点:
  - Cloudflare PagesはGit連携・自動デプロイ・グローバルCDN・SSL自動管理を提供し、フロントエンドの運用負荷が低い
  - Render.comとの組み合わせでもGit連携自動デプロイが可能
  - Cloudflareの管理画面はシンプルで、フロントエンド側の運用は直感的
- 欠点:
  - ALT-B/Cと同様に2プラットフォーム管理の複雑性がある
  - Render無料枠のスリープ問題はALT-B/Cと同様に残る
  - バックエンドとフロントエンドのデプロイ・ログ確認が2箇所に分散する
- 根拠: Cloudflare Pagesのフロントエンドホスティングとしての自動化レベルはVercel・Netlifyと同等。バックエンドにRenderを使う限り、Renderのスリープ問題という運用負荷はALT-B/Cと同様に存在する。ALT-Bとの差異は主にフロントエンドプラットフォームの選択であり、OBJ-3の観点では差異は軽微
- CSD依存:
  - D3、S5、C7: ALT-B/Cと同様
