# OBJ-4: サービス継続性の確保 — 評価結果

## 提案した代替案

OBJ-4（サービス継続性の確保）を最適化する観点から、以下の代替案を提案する。

- **Cloudflare Pages（フロントエンド）+ Cloudflare Workers（バックエンド）**: Cloudflare は大規模・老舗のグローバルネットワーク企業。無料枠の廃止リスクが低く、両コンポーネントを同一ベンダーで統一できる。サービス継続性の観点では最も安定した選択肢の一つ。ただしバックエンドに Workers を採用するとプラットフォーム固有の実装（OBJ-5と競合）が増える懸念がある。

- **Netlify（フロントエンド）+ Render.com（バックエンド）**: 両者ともに設立年数が比較的長く、フロントエンド・バックエンドのホスティング市場で確立した地位を持つ。Netlify は2014年設立・Render は2019年設立で、Railway（2020年）・Fly.io（2017年）よりも企業としての信頼実績がある。無料枠の変更履歴は存在するが、廃止ではなく条件変更にとどまってきた。

- **Vercel（フロントエンド）+ Render.com（バックエンド）**: Vercel は2015年設立・IPO前後のフェーズで財務基盤が安定。フロントエンドホスティング市場のリーダー的存在であり、突然のサービス終了リスクは低い。バックエンドは Render がカバー。組み合わせとして業界での実績が豊富。

## 評価

### Cloudflare Pages + Cloudflare Workers

- 評価: ◎
- 利点:
  - Cloudflare は創業2009年、世界最大規模のCDN・セキュリティ企業の一つで事業基盤が極めて安定している
  - 無料枠（Pages: 無制限静的配信、Workers: 月100,000リクエスト）は永続的に維持されており、廃止の実績がない
  - 単一ベンダーでフロントエンド・バックエンドを統合するため、片方が継続して片方が終了するというリスクを排除できる
  - 大企業顧客への依存度が高く、無料枠を引き上げに使うビジネスモデルの安定性が高い
- 欠点:
  - Workers はエッジコンピューティング特有の制約（Node.js API の一部非対応等）があり、移行難易度がやや高い（OBJ-5 と競合）
  - 単一ベンダー統合はベンダーロックインリスクとトレードオフ（OBJ-5 への影響）
- 根拠: Cloudflare の企業規模・財務基盤・無料枠の維持実績（2009年〜）はOBJ-4の評価軸「運営実績・規模」で他の選択肢を大きく上回る
- CSD依存: S7（選定サービスの安定的存続）はこの選択肢では特に低リスク。D3（単一/分離運用）はこの選択肢では単一統合として解消される

---

### Netlify（フロントエンド）+ Render.com（バックエンド）

- 評価: ○
- 利点:
  - Netlify（2014年設立）は静的サイト・Jamstack ホスティング市場の草分けで認知度・実績が高い
  - Render（2019年設立）は近年急成長しており、Herokuの無料枠廃止（2022年）後の移行先として多くのプロジェクトに採用されている
  - 両者ともに無料枠を有料化した実績はなく（条件変更は存在）、継続方針が比較的安定している
  - 移行の容易さという観点でも、標準的な設定で他プラットフォームに移れる構造になっている
- 欠点:
  - Render は Railway（2020年）より古いが、依然として中規模の新興企業であり、大企業による買収やサービス縮小のリスクがゼロではない
  - バックエンドの Render 無料枠はコールドスタート問題（S5: 約25秒、C7、C5と矛盾）があり、OBJ-2 との兼ね合いで採用可能か別途判断が必要
  - フロントエンドとバックエンドで別ベンダーを使うため、どちらか一方が仕様変更した場合の対応が必要になる
- 根拠: 業界での認知度・実績においては中程度の評価。継続リスクはゼロではないが、主要ホスティングサービスとして市場で一定の地位を確立している
- CSD依存: S7（安定的存続）の仮定に依存。S5（コールドスタート遅延）がOBJ-2の制約（C5）と衝突するため、無料枠バックエンドでこの組み合わせを採用するにはスリープ回避策が必要

---

### Vercel（フロントエンド）+ Render.com（バックエンド）

- 評価: ○
- 利点:
  - Vercel（2015年設立）はNext.jsの開発元であり、フロントエンドホスティング市場で広く認知されている。IPO検討段階とされる企業規模で財務基盤が比較的安定
  - サービス廃止よりも有料プラン誘導の方向性であり、突然の終了リスクは低い
  - Render.com の継続性評価は上記と同じ
- 欠点:
  - Vercel は商用利用制限（C6）があり、D1（商用利用の有無）が確定していないため、採用自体がリスクを持つ。D1が商用と判明すれば OBJ-1（コスト制約）との矛盾が生じる
  - Hobby プランの規約変更（非商用限定の強化など）が過去にも行われており、無料枠の条件安定性にやや懸念がある
  - Render バックエンド無料枠のコールドスタート問題は上記と同様
- 根拠: Vercel のサービス存続リスクは低いが、商用利用の不確実性（D1）によって「現状そのまま使い続けられるか」が揺らぐ。この点でCloudflare・Netlifyと比べると条件付きの○評価
- CSD依存: D1（商用利用の有無）に強く依存。D1が商用と判明した場合、Vercel Hobby の利用自体が規約違反となりサービス継続不可となる可能性がある

---

### Railway（バックエンド候補として）

- 評価: △
- 利点:
  - 2020年設立だが急成長しており、開発者コミュニティでの評価が高い
  - 移行時のアプリケーション変更は比較的少なく、移行容易性はある
- 欠点:
  - 永続的無料枠が存在しない（S4）。バックエンドのコストが月$5以上発生するため C3（月額0円）と矛盾し、OBJ-1 の制約を充足できない。OBJ-4 の観点では「継続して使えるか」以前に無料での継続自体が不可能
  - 2020年設立と比較的新しく、サービス終了・条件変更のリスクが老舗と比べて高い
  - バックエンド単体の選択肢のため、フロントエンドとの組み合わせが別途必要
- 根拠: S4（永続的無料枠なし）と C3（月額0円制約）の組み合わせにより、サービス継続性の観点以前にコスト制約を満たせない。フロントエンド無料＋バックエンド有料の構成では月額コストが発生する
- CSD依存: S4（Railway・Fly.ioに永続的無料枠なし）に依存。S4が崩れ無料枠が提供された場合は評価が上昇する可能性がある

---

### Fly.io（バックエンド候補として）

- 評価: △
- 利点:
  - 2017年設立で、PaaS市場では比較的長い実績がある
  - エッジデプロイの仕組みが確立しており、技術基盤は安定
- 欠点:
  - 永続的無料枠が存在しない（S4）。Railway と同様に C3（月額0円）と矛盾し、無料での継続運用が不可能
  - 2026年1月からストレージ課金が追加されるなど、コスト変更の頻度が高い（S3の崩れリスク）
  - 料金体系の変更頻度が高いことは、サービス継続性・条件安定性の観点から不安要素
- 根拠: 永続的無料枠の欠如と料金変更頻度の高さにより、OBJ-4（サービス継続性）の視点でも評価が下がる。無料で継続できないことはサービス継続性のリスクそのもの
- CSD依存: S4（永続的無料枠なし）と S3（料金・無料枠条件の変動リスク）に依存
