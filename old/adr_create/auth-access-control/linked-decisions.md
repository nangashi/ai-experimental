# 連鎖する意思決定

## 関連する将来の決定

### LD-1: マルチユーザー化・チーム共有機能の追加
- 想定時期: 1〜2年後（要件の変化次第で早まる可能性あり）
- 影響:
  - ALT-1: 選択肢が広がる — Cloudflare Access はユーザー数・グループポリシーの追加がダッシュボード設定のみで完結。IdP側のグループ管理と連携することで、認証コードの変更なしにマルチユーザー対応が可能。ただし無料枠（50ユーザー）を超えると $7/席/月の課金が発生し、コスト設計の再検討が必要になる。
  - ALT-2: 中立 — ユーザーモデルの拡張（ユーザーテーブルの追加・KV のキー設計変更等）が必要になるが、認証方式自体の変更は最小限。既存のセッション管理コードに権限チェックを追加するだけで対応可能。
  - ALT-3: 選択肢が狭まる — ステートレスJWTのペイロードにロール情報を持たせる場合、発行済みトークンの更新・無効化が困難。ユーザー追加・権限変更が即時反映されない問題が表面化しやすく、KVによるトークン無効化リストの導入が必要になる可能性が高い（ステートレスの利点が薄れる）。

### LD-2: マイクロサービス化・複数WorkersサービスにまたがるAPI分離
- 想定時期: 1〜3年後（アプリ規模の拡大次第）
- 影響:
  - ALT-1: 選択肢が広がる — Cloudflare Access は複数のアプリケーション・Workersサービスに対して同一のポリシーを適用可能。新サービスを追加する際もダッシュボードで保護対象に追加するだけで、サービス間の認証連携コードが不要。Service Token を使ったサービス間認証も Cloudflare Zero Trust の枠組みで統一できる。
  - ALT-2: 中立 — サービスごとに KV 名前空間とセッション管理コードを実装する必要があるが、Hono のミドルウェアを共通ライブラリ化することで対応可能。Workers 間でセッション情報を共有する場合は KV の設計変更が必要。
  - ALT-3: 選択肢が広がる — ステートレスJWTは複数サービス間でトークン検証の共有鍵（または公開鍵）さえ配布すれば、サービスごとに認証コードを書かずに済む。ただしシークレット管理（Workers Secrets の複数サービスへの配布）の設計が必要になる。

### LD-3: モバイルアプリ・ネイティブクライアントへの対応
- 想定時期: 1〜2年後（PWAから発展した場合）
- 影響:
  - ALT-1: 選択肢が狭まる — ネイティブアプリからは Cloudflare Access の認証フロー（ブラウザリダイレクト）を利用できない。ネイティブクライアント向けに別の認証方式を並行導入するか、認証方式全体を変更する必要があり、移行コストが最も高い。
  - ALT-2: 中立 — セッションCookieはネイティブアプリでは扱いが複雑になるが、APIトークン（長期セッションID）の発行を追加実装することで対応可能。Cookie ベースを維持しながら、ネイティブ向けにセッションIDをヘッダーで受け付けるハイブリッド実装が必要。
  - ALT-3: 選択肢が広がる — JWTはネイティブアプリとの相性が良い。Cookie から Authorization ヘッダーへの切り替えや、両方のサポートが容易。既存のJWT署名検証ロジックを流用できるため、移行コストが低い。

### LD-4: CI/CD・ステージング環境の整備
- 想定時期: 6ヶ月〜1年後（開発規模の拡大次第）
- 影響:
  - ALT-1: 選択肢が狭まる — ステージング環境ごとに別の Cloudflare Access アプリケーションを登録する必要がある。テスト時の認証バイパスがダッシュボード設定に依存するため、自動化テスト（E2Eテスト等）での認証回避が困難になる場合がある。Service Token を使ったCI/CD統合は可能だが設定が必要。
  - ALT-2: 中立 — 環境ごとに KV 名前空間を分けることで独立したセッション管理が可能。テスト用のセッション生成スクリプトも実装可能。自前実装のため環境ごとの設定変更がコードレベルで管理できる。
  - ALT-3: 選択肢が広がる — テスト用のJWTをシークレットを使って生成・検証できるため、E2Eテストやステージング環境での認証回避が最も容易。Workers Secrets で環境別のシークレットを管理するだけで良く、コード変更不要。

### LD-5: アクセスログ・監査証跡の要件発生
- 想定時期: 1〜3年後（コンプライアンス要件や障害調査ニーズの発生次第）
- 影響:
  - ALT-1: 選択肢が広がる — Cloudflare Access は認証ログ・アクセスログを自動で Cloudflare のダッシュボードに記録。Logpush で外部SIEMへの転送も可能。Workers 側での追加実装なしに監査証跡が取得できる。
  - ALT-2: 中立 — KV にセッション情報を保存しているため、セッションIDに紐づいたアクセス記録を Workers のログに出力する実装を追加することで監査証跡を実現可能。実装は必要だが難易度は低い。
  - ALT-3: 選択肢が狭まる — ステートレスJWTはサーバーサイドに状態を持たないため、「誰がいつ何にアクセスしたか」の記録を取るには Workers 側での明示的なログ実装が必要。トークンの発行・無効化の記録も別途管理が必要。KVなしでは監査証跡のインフラが存在しない状態。

## 可逆性評価

| 代替案 | 可逆性 | 理由 |
|--------|--------|------|
| ALT-1: Cloudflare Access | 部分可逆 | Workers側の認証コードはゼロのため、Cloudflare Access を外して別方式に移行する際に既存コードへの影響は最小限。ただしダッシュボード設定・IdP連携・ポリシー設定は別途移行作業が必要。Cloudflare Zero Trust に強く依存しており、サービス廃止・プラン変更時の代替手段確保が課題。移行コスト: 中（設定移行 + 新認証実装）。 |
| ALT-2: Cookie セッション + Workers KV | 部分可逆 | Hono ミドルウェアの差し替えで認証方式を変更可能（decision-statement.md に記載）。KV のセッションデータはシングルユーザーのため無効化のみで良く、ユーザーデータのマイグレーションは不要。KV を使用し続ける場合はデータ構造の変更が必要。移行コスト: 低〜中（ミドルウェア差し替え + KVクリーンアップ）。 |
| ALT-3: 署名付き Cookie / JWT in Cookie | 可逆 | Hono JWT ミドルウェアの差し替えのみで移行可能。ステートレスのためサーバーサイドのデータマイグレーション不要。Workers Secrets のシークレットを変更し、旧トークンを無効化（有効期限切れを待つ）するだけで移行完了。移行コスト: 低（ミドルウェア差し替え + シークレット更新）。 |
