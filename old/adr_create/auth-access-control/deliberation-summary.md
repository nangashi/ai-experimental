# 審議サマリー

## 決定ステートメント

**議題**: シングルユーザーWebアプリの認証・アクセス制御として、どの認証方式を採用するか

**スコープ**:
- 対象: 認証方式（Cloudflare Access / 自前JWT / Basic認証等）の選定、セッション管理方式、Web Share Target APIからのPOSTリクエストとの互換性
- 対象外: 認可（権限管理）の詳細設計（シングルユーザーのため不要）、アプリケーション層のビジネスロジック

**可逆性**: medium — 認証方式はミドルウェアレベルで全エンドポイントに影響するが、シングルユーザーのためユーザーデータのマイグレーションは不要。Hono のミドルウェア差し替えで認証方式を変更可能。

---

## 関連する既存決定

- **ADR-0001: PWA+REST API構成のホスティング基盤選定** — Cloudflare Pages（フロントエンド）+ Cloudflare Workers（バックエンド）を採用。有料の認証サービスは採用不可、バックエンドは V8 Isolate ランタイムの Hono、シングルユーザー構成、画面表示 3 秒以内の要件が引き続き適用される。Cloudflare Access が利用可能な構成でもある。
- **ADR-0002: フロントエンドフレームワーク選定** — React（Vite + React 19）を採用。認証フローは React で実装。Cloudflare Access 採用時はフロントエンドでの認証処理が不要になる一方、自前実装では React 側でのトークン管理が必要。
- **ADR-0003: データベース選定** — Cloudflare D1 を採用。D1 には組み込み認証機能がなく、「認証を自前実装する」ことを明示的にトレードオフとして受け入れている。セッション管理ストレージとして Workers KV 等の追加実装が示唆されている。

## 前提条件（CSD）

### 確実（Certainty）

- **C1**: 有料の認証サービスは採用しない — 根拠: ADR-0001 C3 より確定済み。Auth0 有料プラン・Supabase Pro 等は選択不可。
- **C2**: バックエンドは Cloudflare Workers（V8 Isolate ランタイム）上の Hono で動作する — 根拠: ADR-0001 の決定内容より確定済み。Node.js 専用ライブラリや OS 依存の認証ライブラリは利用不可。
- **C3**: シングルユーザー構成であり、マルチユーザー向けの複雑な認可設計は不要 — 根拠: ADR-0001 C4 より確定済み。
- **C4**: フロントエンドは React（Cloudflare Pages にデプロイされた静的 SPA）であり、サーバーサイドレンダリングは行わない — 根拠: ADR-0002 より確定済み。
- **C5**: 画面表示 3 秒以内の要件が存在する — 根拠: ADR-0001 C5 より確定済み。認証処理のレイテンシも要件の対象。
- **C6**: Cloudflare D1 採用済みであり、D1 には組み込み認証機能がない — 根拠: ADR-0003 より確定済み。Cloudflare Access を採用する場合は Workers 側での認証実装は不要になりうる。
- **C7**: Web Share Target API の POST リクエストに Authorization ヘッダーはブラウザから自動付与されない — 根拠: MDN・Chrome for Developers。Bearer JWT 認証方式は Web Share Target API との互換性に根本的な課題を持つ。
- **C8**: Web Share Target API の利用には PWA としてのインストールが前提となる — 根拠: MDN 仕様。
- **C9**: HTTPS 通信が保証されている — 根拠: Cloudflare Pages および Workers はデフォルトで HTTPS を強制する。Cookie の Secure 属性・JWT の署名検証等の前提。

### 仮定（Supposition）

- **S1**: アプリを利用するのは作者本人のみであり、将来もマルチユーザー化しない — 崩れる可能性: ADR-0003 に「将来のマルチユーザー化」への言及あり。検証方法: プロダクトの将来計画を確認する。
- **S2**: iOS Safari を含む複数のブラウザ・OS 環境からの利用が想定される — 崩れる可能性: 利用端末が限定されている場合は範囲が変わる。検証方法: 利用端末の環境を確認する。
- **S3**: パスワード等のシークレットは Cloudflare Workers の Secrets 機能で安全に管理できる — 崩れる可能性: 設定ミス・ローテーション未実施で漏洩リスクが生じる。検証方法: Workers Secrets の設定・ローテーション方針を確認する。
- **S4**: Hono JWT ミドルウェアは継続的にメンテナンスされ、セキュリティ修正が適時提供される — 崩れる可能性: 過去に脆弱性報告あり。メンテナンスが停止した場合は自前対応が必要。検証方法: GitHub Security Advisory の継続的モニタリング。
- **S5**: Cloudflare Access の料金プラン・機能（無料 50 ユーザー枠）は現状維持される — 崩れる可能性: Cloudflare のサービス条件は変更されうる。検証方法: Cloudflare の料金ページを定期的に確認する。
- **S6**: セッション管理のための Workers KV が必要に応じて利用可能（無料枠の範囲内で十分）— 崩れる可能性: 無料枠の変更・廃止の可能性。検証方法: セッション発行・検証頻度が無料枠内に収まるか試算する。
- **S7**: 認証方式の変更はミドルウェア差し替えで対応可能（中可逆性）— 崩れる可能性: Cloudflare Access と自前実装の間の移行はインフラ層の変更を伴う場合がある。検証方法: 採用する方式と他方式間の移行パスを確認する。
- **S8**: このアプリはインターネット公開されており、作者本人以外のアクセスは不正アクセスとして拒否する必要がある — 崩れる可能性: 脅威モデルの前提（悪意ある攻撃者対策か誤操作防止レベルか）が変わる場合。検証方法: 脅威モデルを確認する。
- **S9**: 認証は全 API エンドポイントに一律の方式で適用する — 崩れる可能性: C7 の制約（Web Share Target POST で Authorization ヘッダー非自動付与）を踏まえると、エンドポイント別のハイブリッド構成も代替案として検討しうる。検証方法: エンドポイント別に異なる認証方式を許容するか確認する。

### 不確実（Doubt）

- **D1**: iOS Safari スタンドアロン PWA モードでの Cookie 共有問題が、このアプリの実際の使用シナリオで問題になるか — 影響範囲: iOS で Cookie ベースのセッション管理を利用する場合、初回認証フローで追加実装が必要になる可能性。リスク: Web Share Target API からの投稿が認証エラーで失敗する。
- **D2**: 自前 JWT 実装における設定ミスのリスクを開発者が許容できるか — 影響範囲: Hono JWT ミドルウェアの設定ミスがセキュリティ脆弱性に直結。リスク: 個人用途ゆえセキュリティレビューが省かれやすく、発見が遅れる可能性。
- **D3**: Web Share Target API 経由の POST リクエスト時のセッション Cookie の実際の動作（Cloudflare Workers + Hono 環境での動作未検証）— 影響範囲: Cookie の Secure/SameSite 属性・ドメインマッチ等がエッジ環境で想定通りに動作するか未検証。リスク: 互換性要件を満たせない場合がある。

## 判断基準

*優先度は決定者（人間）が設定*

- **OBJ-1: Web Share Target API との互換性** — 認証方式が、ブラウザから Authorization ヘッダーを自動付与できない Web Share Target API の POST リクエストと共存できること。評価の視点: Web Share Target API 経由の POST リクエストで認証エラーなく機能するか、追加の実装コストなしに互換性を確保できるか、iOS Safari スタンドアロン PWA モードでの Cookie 挙動（D1・D3）に左右されないか。
- **OBJ-2: セキュリティ強度と設定ミスリスクの低さ** — インターネット公開された個人用ツールとして、第三者の不正アクセスを確実に遮断できること。かつ、開発者単独による実装・設定ミスでセキュリティホールが生まれにくいこと。評価の視点: 既知の攻撃ベクターへの耐性、設定ミスが致命的なセキュリティ脆弱性に直結するリスク（D2）、セキュリティ修正の適時提供・メンテナンス継続性（S4）。
- **OBJ-3: Cloudflare Workers 環境での実装・運用の簡潔さ** — V8 Isolate ランタイムの制約下で、認証ロジックの実装・デプロイ・長期運用が最小の負担で行えること。評価の視点: Node.js 非互換ライブラリへの依存なく実装できるか（C2）、認証基盤のセットアップ・設定変更・トラブルシューティングの煩雑さ、外部サービス依存度（S5）。
- **OBJ-4: 認証処理のレイテンシの低さ** — 画面表示 3 秒以内の要件（C5）を満たすうえで、認証処理がボトルネックにならないこと。評価の視点: 認証処理が追加するネットワークラウンドトリップ数、Cloudflare Workers のエッジで完結するか、セッション検証のコスト（Workers KV 参照回数など）。
- **OBJ-5: コスト持続可能性** — 個人用ツールとして、認証基盤の運用コストが長期的に無料または最小限であること。評価の視点: 採用する認証基盤の無料枠と想定使用量のマージン（S5・S6）、無料枠の変更・廃止リスク（S5）、有料プランへの強制移行が生じた場合の代替手段の有無。

## 検討した選択肢

### ALT-1: Cloudflare Access（Zero Trust エッジ認証）

Cloudflare Zero Trust の認証レイヤーを Workers の前段に配置。IdP（GitHub・Google等）経由で認証し、cf_authorization Cookie で透過的にセッション管理。Workers 側に認証コードを書かない。Cookie ベースの透過的認証により Web Share Target POST との互換性が確保され（OBJ-1 ◎）、認証ロジックの自前実装ゼロで設定ミスの表面積を最小化する（OBJ-2 ◎）。

### ALT-2: Cookie セッション + Workers KV（ステートフル）

Hono カスタムミドルウェアでセッション管理を自前実装。セッションIDを HttpOnly/Secure/SameSite=Strict Cookie で発行し、Workers KV にセッション情報を保存。即時ログアウト・セッション無効化が可能。Workers 内完結で外部サービス依存なし（OBJ-3 ○）だが、リクエストごとの KV ラウンドトリップがレイテンシに影響（OBJ-4 △）。

### ALT-3: 署名付き Cookie / JWT in Cookie（ステートレス）

JWT を Cookie として保持し、Hono JWT ミドルウェアで Workers 内完結の署名検証を行うステートレス方式。Workers KV 不要。Workers Secrets にシークレットを保存。外部 I/O なしで認証処理が完結するため最小レイテンシ（OBJ-4 ◎）かつ追加サービス依存ゼロでコスト最小（OBJ-5 ◎）。ただし JWT 設定ミスリスク（D2）を内包（OBJ-2 △）。

### 除外した選択肢

- **JWT Bearer 認証（Authorization ヘッダー）**: C7（Web Share Target POST で Authorization ヘッダー非自動付与）に直接抵触。OBJ-1 で × 評価。
- **Basic 認証**: C7 と同様の Authorization ヘッダー非互換問題。OBJ-1 で ×、OBJ-2 で × 評価。業界非推奨。
- **エンドポイント別ハイブリッド構成**: 実装複雑性が高く、他の方式（ALT-1〜3）で Web Share Target 互換性を確保可能。OBJ-1 で △ 評価。

## トレードオフ行列

| 判断基準 | ALT-1: Cloudflare Access | ALT-2: Cookie+KV | ALT-3: JWT in Cookie |
|----------|--------------------------|-------------------|----------------------|
| OBJ-1: Web Share Target互換性 | ◎ | ○ | ○ |
| OBJ-2: セキュリティ強度 | ◎ | ○ | △ |
| OBJ-3: 実装・運用簡潔さ | ◎ | ○ | ○ |

### OBJ-1（Web Share Target 互換性）の詳細

- **ALT-1 ◎**: cf_authorization Cookie がブラウザに保持され、Web Share Target POST 時にも自動付与される。Workers 側での認証処理が不要で透過的に互換性確保。欠点: D1（iOS Safari PWA Cookie 共有問題）の影響を受ける可能性あり。
- **ALT-2 ○**: セッション Cookie が Web Share Target POST 時にブラウザから自動付与される。欠点: D1・D3 の残存リスクあり。Cookie の SameSite/Secure 属性設定次第で POST が失敗する可能性。
- **ALT-3 ○**: JWT を格納した Cookie が Web Share Target POST 時にブラウザから自動付与される。欠点: ALT-2 と同様に D1・D3 の影響を受ける。自前設定のリスクが加わる。

### OBJ-2（セキュリティ強度）の詳細

- **ALT-1 ◎**: 認証ロジックの自前実装がゼロ。設定ミスの表面積が最小。Cloudflare による DDoS 耐性・ブルートフォース対策・レート制限が自動提供。欠点: Workers 側での CF-Access-Jwt-Assertion 検証省略のリスクは残る。
- **ALT-2 ○**: HttpOnly/Secure/SameSite=Strict Cookie で XSS・CSRF リスクを低減。サーバーサイドセッションで即時無効化可能。欠点: Cookie 属性の設定ミスが脆弱性に直結。開発者のセキュリティ知識に依存（D2）。
- **ALT-3 △**: Cookie（HttpOnly/Secure）で JWT を配信するため XSS による窃取リスクは低い。欠点: Hono JWT ミドルウェアの過去の脆弱性（アルゴリズム混同・aud 未検証）への対応が必要（S4 依存）。ステートレスのため発行済みトークンの即時無効化が困難。設定ミスリスクが最も高い（D2）。

### OBJ-3（実装・運用簡潔さ）の詳細

- **ALT-1 ◎**: Workers 側に認証コードを書かない。ダッシュボードで認証ポリシー管理。V8 Isolate の制約に非依存。欠点: 初期セットアップ（テナント・アプリ登録・ポリシー設定）の学習コスト。
- **ALT-2 ○**: Workers 内完結。外部サービス依存なし。欠点: セッション ID 生成・保存・検証・失効のフル実装が必要。KV 読み書きコードとライフサイクル管理の保守負担。
- **ALT-3 ○**: Hono 公式 JWT ミドルウェア利用で実装量が少ない。KV 不要のステートレス設計でアーキテクチャがシンプル。欠点: セキュリティ設定の自前設定が必要。OBJ-1 対応のため Cookie 配信にする追加実装が必要。

### OBJ-4（レイテンシ）の詳細

- **ALT-1 ○**: 認証処理はエッジで Workers 呼び出し前に完結。欠点: Workers 内での CF-Access-Jwt-Assertion の署名検証に JWKS 公開鍵取得が必要（初回/再検証時に外部 I/O の可能性）。
- **ALT-2 △**: リクエストごとに Workers KV への読み取りラウンドトリップが発生。KV の読み取りレイテンシは通常 1〜数十ms だがコールドキャッシュ時に増加しうる。C5（3秒要件）への余裕が少ない。
- **ALT-3 ◎**: JWT 署名検証が Workers 内（V8 Isolate）で完結。外部ネットワークラウンドトリップなし。HMAC-SHA256 検証は数マイクロ秒〜数ミリ秒。

### OBJ-5（コスト持続可能性）の詳細

- **ALT-1 ○**: 現時点ではシングルユーザーで無料枠（50 ユーザー）内。欠点: S5（Cloudflare Access 無料枠の現状維持）に強く依存。51 ユーザー目から $7/席/月の急激な課金構造。
- **ALT-2 ○**: Workers KV 無料枠（100,000 読み取り/日・1,000 書き込み/日）でシングルユーザー用途は十分。欠点: S6（KV 無料枠継続）に依存。認証で KV 枠を消費するため他用途への枠が減少。
- **ALT-3 ◎**: Workers Secrets（無料）のみ使用。追加サービス・ストレージ依存ゼロ。料金変更リスクなし。欠点: トークン即時無効化のために KV を使う場合は S6 依存が生じる。

## リスク分析

### ALT-1: Cloudflare Access の失敗シナリオ

**シナリオ 1: Cloudflare Access 無料枠の突然の廃止または条件変更**
- 状況: 1年後、Cloudflare が Zero Trust の無料プランを変更し、有料プランへの移行が必要になる。C1（有料の認証サービスは採用しない）に抵触し、認証方式全体の見直しが迫られる。
- 原因: S5（Cloudflare Access の料金プラン・機能の現状維持）の崩壊。出口戦略を準備しなかった。
- 深刻度: high / 発生確率: low
- 軽減策: Cloudflare の料金ページを四半期ごとにモニタリング。ALT-2 または ALT-3 への移行パスをあらかじめ設計し、手順書を作成しておく。

**シナリオ 2: iOS Safari PWA モードでの認証フロー断絶**
- 状況: 1年後、Web Share Target API 経由の共有が iOS Safari スタンドアロン PWA モードから頻繁に失敗するようになる。cf_authorization Cookie がブラウザコンテキストとスタンドアロンコンテキスト間で共有されず、認証リダイレクトが正しく完了しない。
- 原因: D1（iOS Safari PWA モードでの Cookie 共有問題）の悪化シナリオ。検証を後回しにした。
- 深刻度: high / 発生確率: medium
- 軽減策: 開発初期段階で iOS Safari PWA モードと Cloudflare Access の認証リダイレクトの挙動を実機検証する。

**シナリオ 3: Workers 側での CF-Access-Jwt-Assertion 検証省略によるバイパス**
- 状況: 1年後、Workers に直接アクセスするエンドポイントへの不正アクセスが発生する。Cloudflare Access をトラフィックの前段に置くだけで Workers 側の JWT 検証を省いていたため、Access をバイパスした直接リクエストを検証できなかった。
- 原因: 「Cloudflare Access を設定すれば安全」という誤った前提。S8（インターネット公開環境での第三者からの保護）の重要性を過小評価した。
- 深刻度: high / 発生確率: medium
- 軽減策: Workers 側で CF-Access-Jwt-Assertion ヘッダーの署名検証を必ず実装する。Access ポリシーの設定と Workers の検証は「二重防御」として設計する。

**シナリオ 4: Cloudflare Access の仕様変更・API 互換性の破壊**
- 状況: 1年後、Cloudflare が cf_authorization Cookie の仕様を変更した結果、Workers 側の検証ロジックが機能しなくなる。
- 原因: S5（料金プラン・機能の現状維持）の部分的崩壊。外部マネージドサービスの仕様変更に対する追跡コストを過小評価した。
- 深刻度: medium / 発生確率: low
- 軽減策: Cloudflare の変更履歴を定期的にモニタリング。Workers 側の JWT 検証ロジックを抽象化し、設定で切り替えられるようにしておく。

### ALT-3: JWT in Cookie の失敗シナリオ

**シナリオ 1: JWT 設定ミスによるセキュリティ脆弱性の放置**
- 状況: 1年後、Hono JWT ミドルウェアのアルゴリズム検証設定が不適切であったことが判明する。アルゴリズム混同攻撃（none アルゴリズムや RS/HS 混同）により、自前実装の認証が実質的にバイパスされていた。
- 原因: D2（自前 JWT 実装における設定ミスのリスク）の顕在化。個人開発ゆえにセキュリティレビューが省かれ、S4（Hono JWT ミドルウェアのメンテナンス継続）への依存。
- 深刻度: high / 発生確率: medium
- 軽減策: JWT 設定時に alg を明示的に指定（例: HS256 固定）。aud クレームの検証を有効化。Hono JWT ミドルウェアの GitHub Security Advisory を定期的に確認する。

**シナリオ 2: ステートレス方式によるセッション無効化不能による被害拡大**
- 状況: 1年後、認証情報が漏洩またはデバイスを紛失した際に、発行済みの JWT を即時無効化できないことが問題となる。
- 原因: S8（インターネット公開環境での第三者からの保護必要性）との乖離。ステートレス方式のコスト・シンプルさを優先し、緊急無効化フローを設計しなかった。
- 深刻度: high / 発生確率: low
- 軽減策: JWT の有効期限を短く設定（例: 1〜2時間）し、Workers Secrets のシークレットローテーションによる「全トークン一括無効化」フローを整備する。

**シナリオ 3: iOS Safari PWA モードでの Cookie 挙動による Web Share Target 失敗**
- 状況: 1年後、iOS Safari スタンドアロン PWA モードで Web Share Target API 経由の POST が断続的に認証エラーで失敗するようになる。
- 原因: D1（iOS Safari PWA Cookie 共有問題）および D3（Cloudflare Workers + Hono 環境での Cookie 動作未検証）の顕在化。
- 深刻度: high / 発生確率: medium
- 軽減策: 開発初期段階で iOS 実機を用いた Web Share Target + Cookie 認証の動作確認を実施する。

**シナリオ 4: Workers Secrets のシークレット漏洩によるトークン全件偽造**
- 状況: 1年後、Workers Secrets に保存した JWT 署名シークレットが外部に漏洩する。攻撃者は任意の JWT を偽造でき、認証が完全に無効化される。
- 原因: S3（Workers Secrets での安全なシークレット管理）の崩壊。ローカル開発環境への誤記録リスクを認識していなかった。
- 深刻度: high / 発生確率: low
- 軽減策: ローカル開発用シークレットは .dev.vars に格納し、.gitignore に必ず追加する。wrangler.toml にはシークレット値を記載しない。

## 将来の意思決定への影響

### LD-1: マルチユーザー化・チーム共有機能の追加（想定: 1〜2年後）

- **ALT-1**: 選択肢が広がる — ユーザー数・グループポリシーの追加がダッシュボード設定のみで完結。ただし無料枠（50ユーザー）を超えると $7/席/月の課金発生。
- **ALT-2**: 中立 — ユーザーモデルの拡張（ユーザーテーブルの追加・KV のキー設計変更等）が必要だが、認証方式自体の変更は最小限。
- **ALT-3**: 選択肢が狭まる — 発行済みトークンの更新・無効化が困難。ユーザー追加・権限変更が即時反映されない問題が表面化しやすく、KVによるトークン無効化リストの導入が必要になる可能性が高い。

### LD-2: マイクロサービス化・複数Workers サービスにまたがるAPI分離（想定: 1〜3年後）

- **ALT-1**: 選択肢が広がる — 複数のアプリケーション・Workersサービスに対して同一のポリシーを適用可能。Service Token を使ったサービス間認証も Zero Trust の枠組みで統一できる。
- **ALT-2**: 中立 — サービスごとに KV 名前空間とセッション管理コードを実装する必要があるが、Hono のミドルウェアを共通ライブラリ化することで対応可能。
- **ALT-3**: 選択肢が広がる — ステートレスJWTは複数サービス間でトークン検証の共有鍵さえ配布すれば、サービスごとに認証コードを書かずに済む。

### LD-3: モバイルアプリ・ネイティブクライアントへの対応（想定: 1〜2年後）

- **ALT-1**: 選択肢が狭まる — ネイティブアプリからは Cloudflare Access の認証フロー（ブラウザリダイレクト）を利用できない。移行コストが最も高い。
- **ALT-2**: 中立 — セッションCookieはネイティブアプリでは扱いが複雑になるが、APIトークン発行の追加実装で対応可能。
- **ALT-3**: 選択肢が広がる — JWTはネイティブアプリとの相性が良い。Cookie から Authorization ヘッダーへの切り替えや両方のサポートが容易。

### LD-4: CI/CD・ステージング環境の整備（想定: 6ヶ月〜1年後）

- **ALT-1**: 選択肢が狭まる — ステージング環境ごとに別の Cloudflare Access アプリケーションを登録する必要がある。E2Eテストでの認証回避が困難になる場合がある。
- **ALT-2**: 中立 — 環境ごとに KV 名前空間を分けることで独立したセッション管理が可能。テスト用のセッション生成スクリプトも実装可能。
- **ALT-3**: 選択肢が広がる — テスト用のJWTをシークレットを使って生成・検証できるため、E2Eテストやステージング環境での認証回避が最も容易。

### LD-5: アクセスログ・監査証跡の要件発生（想定: 1〜3年後）

- **ALT-1**: 選択肢が広がる — Cloudflare Access は認証ログ・アクセスログを自動で記録。Logpush で外部SIEMへの転送も可能。Workers 側での追加実装なしに監査証跡が取得できる。
- **ALT-2**: 中立 — KV にセッション情報を保存しているため、セッションIDに紐づいたアクセス記録を Workers のログに出力する実装を追加することで監査証跡を実現可能。
- **ALT-3**: 選択肢が狭まる — ステートレスJWTはサーバーサイドに状態を持たないため、アクセス記録を取るには Workers 側での明示的なログ実装が必要。KVなしでは監査証跡のインフラが存在しない。

### 可逆性評価

| 代替案 | 可逆性 | 理由 |
|--------|--------|------|
| ALT-1: Cloudflare Access | 部分可逆 | Workers側の認証コードはゼロのため移行時の既存コードへの影響は最小限。ただしダッシュボード設定・IdP連携・ポリシー設定は別途移行作業が必要。移行コスト: 中。 |
| ALT-2: Cookie セッション + Workers KV | 部分可逆 | Hono ミドルウェアの差し替えで認証方式を変更可能。シングルユーザーのためユーザーデータのマイグレーションは不要。移行コスト: 低〜中。 |
| ALT-3: 署名付き Cookie / JWT in Cookie | 可逆 | Hono JWT ミドルウェアの差し替えのみで移行可能。ステートレスのためサーバーサイドのデータマイグレーション不要。移行コスト: 低。 |

## 判断ガイド

以下は条件付きの帰結の提示であり、推奨ではない。
最終的な優先順位の決定は判断者に委ねられる。

- **OBJ-1（Web Share Target との互換性）または OBJ-2（セキュリティ強度）を最優先するなら** → ALT-1（Cloudflare Access）が適している。ただし S5（Cloudflare Access 無料枠の現状維持）への依存と、Workers 側での二重防御実装（CF-Access-Jwt-Assertion 検証）が必須になる点に注意。また D1（iOS Safari PWA Cookie 共有問題）の早期検証を要する。
- **OBJ-4（レイテンシ）または OBJ-5（コスト持続可能性）を最優先するなら** → ALT-3（JWT in Cookie）が適している。ただし OBJ-2（セキュリティ強度）で最も劣る（△）ため、JWT 設定の厳密な実装（アルゴリズム明示指定・aud 検証）と D2（設定ミスリスク）の許容が必要になる。即時無効化が必要になった場合はステートレスの利点が薄れる。
- **バランスを取るなら** → ALT-2（Cookie セッション + Workers KV）は OBJ-1〜3・OBJ-5 で ○ 評価を維持するが、OBJ-4（レイテンシ）で △ となる。KV への強制 I/O が C5（3秒要件）に影響するかを使用パターンで確認することが重要。
- **将来のマルチユーザー化・マイクロサービス化を重視するなら** → ALT-1 が LD-1・LD-2 で「選択肢が広がる」評価。ただし LD-3（ネイティブクライアント）・LD-4（CI/CD）では「選択肢が狭まる」ため、将来シナリオの優先順位で判断が変わる。

## 不確実性の影響

以下の不確実性が判明した場合、選択肢の評価が変化する可能性がある:

- **D1（iOS Safari スタンドアロン PWA モードでの Cookie 共有問題が実際の使用シナリオで問題になる）が判明した場合**: ALT-1・ALT-2・ALT-3 はいずれも Cookie ベースであるため、全選択肢が追加実装（スタンドアロンモードでの再認証フロー等）を要することになる。ALT-1 はリダイレクトベースの認証フローが追加的に複雑化する可能性があり、ALT-2・ALT-3 は Cookie の SameSite/Path 設定の調整が必要になる。深刻な場合は Cookie ベース以外のアプローチへの再検討が必要になりうる。

- **D2（自前 JWT 実装における設定ミスのリスクを開発者が許容できない）が判明した場合**: ALT-3 の評価が OBJ-2 でさらに低下し、事実上除外に近い状況となる。ALT-2 も Cookie 属性設定に関するリスクがあるため慎重さが増す。結果として ALT-1（認証ロジックの自前実装ゼロ）の優位性が相対的に高まる。

- **D3（Web Share Target API 経由の POST リクエスト時のセッション Cookie の Cloudflare Workers + Hono 環境での動作が未検証）が判明した場合**: Cookie の Secure/SameSite 属性・ドメインマッチがエッジ環境で想定通りに動作しない場合、ALT-2・ALT-3 は OBJ-1 の評価が ○ から △ または × に低下する可能性がある。ALT-1 も同様の影響を受けるが、Cloudflare 自身が管理する Cookie であるため ALT-2・ALT-3 より影響が少ない可能性がある。この場合、Authorization ヘッダーを使わない別のアプローチ（URL パラメータベースのトークン等）の検討が必要になりうる。
