# OBJ-4: 認証処理のレイテンシの低さ — 評価結果

## 提案した代替案

- **Hono JWT ミドルウェア（Cookie 渡し）**: JWT をセッション Cookie として保持し、Hono の JWT ミドルウェアで Workers ランタイム内で完結して検証する。外部サービスへのネットワークラウンドトリップが発生せず、V8 Isolate 上での署名検証（HMAC-SHA256）のみで認証が完了するため、追加レイテンシは最小限（数ミリ秒以下）になる。OBJ-4 を最適化する場合にこの選択肢が最も適している。
- **Cloudflare Access（Zero Trust）**: Cloudflare のエッジで認証を完結させる外部委託型の認証レイヤー。Workers へのリクエストが到達する前にエッジで処理されるが、JWT 検証のため Workers は Cloudflare Access が発行した JWT の署名を検証する必要がある（公開鍵取得のキャッシュが必要）。
- **Cookie ベースのセッション管理（Workers KV バックエンド）**: セッション ID を Cookie として保持し、Workers KV にセッション情報を保存する方式。リクエストごとに Workers KV への読み取り（1 ラウンドトリップ）が発生する。KV の読み取りレイテンシは通常 1〜数十ミリ秒だが、コールドキャッシュ時に増加しうる。
- **Basic 認証（Service Worker ヘッダー注入）**: Service Worker がすべてのリクエストに Authorization ヘッダーを注入する方式。Service Worker の処理は同一ランタイム内で完結し、追加ネットワークラウンドトリップは発生しない。ただし Service Worker のアクティブ化待ちが初回リクエストで発生する可能性がある。

## 評価

### Hono JWT ミドルウェア（Cookie 渡し）

- 評価: ◎
- 利点:
  - JWT の署名検証は Workers ランタイム（V8 Isolate）内で完結するため、外部ネットワークラウンドトリップが一切発生しない
  - Cloudflare Workers の Web Crypto API（SubtleCrypto）を使用した HMAC-SHA256 検証は数マイクロ秒〜数ミリ秒で完了する
  - エッジロケーションで処理されるため、地理的なレイテンシも最小化される
  - セッション状態をサーバーサイドに持たないステートレス構成により、KV 参照などの追加 I/O が不要
- 欠点:
  - JWT の有効期限管理（ローテーション・失効処理）を自前で実装する必要があり、失効した JWT を即座に無効化するにはブラックリスト管理（KV 参照）が必要になり、その場合はラウンドトリップが発生する
  - 依存: S4（Hono JWT ミドルウェアのメンテナンス継続性）に依存
- 根拠: ステートレス検証であり、外部 I/O なしで完結するため、画面表示 3 秒以内要件（C5）に対して最も余裕を持って適合する
- CSD依存: S4（Hono JWT ミドルウェアのメンテナンス継続性）。メンテナンスが停止した場合、代替実装への移行が必要になる。ただしレイテンシ評価自体には影響しない

### Cloudflare Access（Zero Trust）

- 評価: ○
- 利点:
  - 認証処理は Cloudflare エッジで Workers の呼び出し前に完結するため、Workers の処理負荷に追加しない
  - Workers 内では Cloudflare Access が付与した JWT ヘッダー（CF-Access-JWT-Assertion）を検証するだけでよい
  - エッジでの処理はグローバルに分散されており、地理的レイテンシが最小化される
- 欠点:
  - Workers 内での Cloudflare Access JWT の署名検証には公開鍵の取得が必要（初回または再検証時に外部 JWKS エンドポイントへのアクセスが発生する可能性がある）
  - Cloudflare Access の認証フロー（IdP へのリダイレクト等）はインタラクティブな初回ログインで数百ミリ秒〜数秒の追加レイテンシが発生しうるが、これはセッション確立後は不要
  - セッション確立済みの通常リクエストでのレイテンシへの影響は小さいが、JWT 検証の実装方法によっては Workers KV を使ったキャッシュが必要になる場合がある
- 根拠: 通常の認証済みリクエストのレイテンシは低いが、JWKS の公開鍵取得という外部 I/O の可能性が残るため ◎ より低い評価とした
- CSD依存: S5（Cloudflare Access の料金プラン・機能の現状維持）。サービス仕様変更によりレイテンシ特性が変化する可能性があるが、現時点では評価に影響しない

### Cookie ベースのセッション管理（Workers KV バックエンド）

- 評価: △
- 利点:
  - セッション情報を Workers KV に保存するため、セッション失効・強制ログアウトが即座に反映される
  - Cookie の送受信はブラウザが自動処理するため、フロントエンド実装コストは低い
- 欠点:
  - リクエストごとに Workers KV への読み取りラウンドトリップが発生する。Workers KV の読み取りレイテンシは通常 1〜数十ミリ秒だが、グローバルレプリケーションのタイミングやコールドキャッシュ時に増加しうる（S6 依存）
  - 高頻度アクセス時（画像シェア等のバースト）に KV 読み取りがボトルネックになりうる
  - 画面表示 3 秒以内要件（C5）の観点では、KV 参照コストがページロードのクリティカルパス上に乗るため、累積するとリスクになる
- 根拠: KV への強制 I/O がリクエストごとに発生するため、他の選択肢と比較してレイテンシ面で不利。画面表示 3 秒以内要件（C5）に対しての余裕が少ない
- CSD依存: S6（Workers KV 無料枠の十分性）。KV の利用量が無料枠を超えると料金が発生するだけでなく、スロットリングによるレイテンシ悪化の可能性がある

### Basic 認証（Service Worker ヘッダー注入）

- 評価: △
- 利点:
  - Service Worker がヘッダーを注入する処理は同一オリジン内で完結し、追加ネットワークラウンドトリップは原則発生しない
  - Workers 側の検証は Authorization ヘッダーの文字列比較のみで完結するため、検証処理コストは最小限
- 欠点:
  - Service Worker のインストール・アクティブ化が完了していない初回アクセス時は、ヘッダーが付与されずに認証が失敗する可能性があり、追加のフォールバック処理が必要になる
  - Service Worker の更新・キャッシュ無効化時に、サービスが一時的に利用不能になるリスクがある（このダウンタイムはレイテンシではなく可用性の問題だが、ユーザー体験に影響する）
  - PWA としてのインストール（C8）が前提であり、インストール前はシェアターゲットとして動作しない（OBJ-1 との絡みで評価が変わる場合がある）
  - Web Share Target API からの POST リクエストに対して Service Worker がヘッダーを確実に注入できるかは未検証の部分がある（D3 依存）
- 根拠: 定常状態のレイテンシは低いが、Service Worker ライフサイクルによる初回アクセスの遅延リスクがある。また、D3（Web Share Target API 経由の Cookie 動作の未検証）に近い不確実性を持つ
- CSD依存: D3（Web Share Target API 経由 POST リクエストでの実際の動作未検証）。Service Worker が介在するエッジケースでの動作が未検証であり、実際の動作は確認が必要
