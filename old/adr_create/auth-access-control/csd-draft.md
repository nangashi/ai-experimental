# CSD分類

## 確実（Certainty）

- C1: 月額コスト 0 円（Cloudflare 無料ティア内） — 根拠: ADR-0001 C3 より確定済み（既存ADRから取り込み）。有料の認証サービス（Auth0 有料プラン・Supabase Pro 等）は選択不可。Cloudflare Access はシングルユーザーであれば無料枠（50 ユーザー上限）内に収まることをリサーチで確認済み（research.md 観点1）。影響範囲: 採用可能な認証方式を無料で利用できるものに限定する。
- C2: バックエンドは Cloudflare Workers（V8 Isolate ランタイム）上の Hono で動作する — 根拠: ADR-0001 の決定内容より確定済み（既存ADRから取り込み）。Node.js 専用ライブラリや OS 依存の認証ライブラリは利用不可。影響範囲: 認証ミドルウェアは Workers ランタイム互換が必要。
- C3: シングルユーザー構成であり、マルチユーザー向けの複雑な認可設計は不要 — 根拠: ADR-0001 C4 より確定済み（既存ADRから取り込み）。ユーザー管理・ロール管理の実装は対象外。影響範囲: シンプルな認証方式（単一パスワード・単一トークン等）が適合する。
- C4: フロントエンドは React（Cloudflare Pages にデプロイされた静的 SPA）であり、サーバーサイドレンダリングは行わない — 根拠: ADR-0002 より確定済み（既存ADRから取り込み）。認証コールバックの受け取り等は React のルーティングで処理する必要がある。影響範囲: 認証フローのリダイレクト処理・セッション維持方式がフロントエンドの制約を受ける。
- C5: 画面表示 3 秒以内の要件が存在する — 根拠: ADR-0001 C5 より確定済み（既存ADRから取り込み）。認証処理のレイテンシも要件の対象となる。影響範囲: 認証レイテンシが高い方式（毎リクエストで外部 IdP に問い合わせ等）は要件を満たせない可能性がある。
- C6: Cloudflare D1 採用済みであり、D1 には組み込み認証機能がない — 根拠: ADR-0003 より確定済み（既存ADRから取り込み）。ADR-0003 では「認証を自前実装する」ことをトレードオフとして既に受け入れている。影響範囲: JWT 発行・検証・セッション管理を Workers 側で実装する必要があるという前提が確定している。
- C7: Hono ミドルウェア差し替えで認証方式の変更が可能であり、今回の決定は中可逆性（medium）である — 根拠: decision-statement.md の可逆性評価と ADR-0001 の確定内容より。シングルユーザーのためユーザーデータのマイグレーションは不要。影響範囲: 認証方式は将来変更可能だが、全エンドポイントに影響するため変更コストは存在する。
- C8: Web Share Target API の POST リクエストに Authorization ヘッダーはブラウザから自動付与されない — 根拠: research.md 観点3の調査結果（MDN ドキュメント・Chrome for Developers 等）。ブラウザは Web Share Target API 経由の POST 送信時に Authorization ヘッダーを自動付与しない。影響範囲: Bearer JWT 認証方式は Web Share Target API との互換性に根本的な課題を持つ。Cookie ベースのセッション管理または透過的な認証レイヤー（Cloudflare Access 等）が必要。
- C9: Web Share Target API の利用には PWA としてのインストールが前提となる — 根拠: research.md 観点2の調査結果（MDN・Chrome for Developers）。インストール前はシェアターゲットとして登録されない。影響範囲: 認証方式の選択に加え、PWA インストール体験（インストールプロンプト・マニフェスト設定）を考慮する必要がある。

## 仮定（Supposition）

- S1: アプリを利用するのは作者本人のみであり、将来もマルチユーザー化しない — 根拠: decision-statement.md にシングルユーザーと明示されているが、ADR-0003 に「将来のマルチユーザー化」への言及もある。現時点では正しい仮定だが変わりうる。検証方法: プロダクトの将来計画を確認する。影響範囲: マルチユーザー化が決定した場合、Cloudflare Access では 51 ユーザー目から全席分 $7/シート/月が発生するコスト構造（research.md 観点1）となり、認証方式の再選定が必要になりうる。
- S2: iOS Safari を含む複数のブラウザ・OS 環境からの利用が想定される — 根拠: Web Share Target API を利用するユースケース（モバイルからのシェア）として iOS 環境が主要な対象と推測されるが、decision-statement.md に明示はない。検証方法: 利用端末の環境（iOS のみ / Android も対象 / デスクトップのみ等）を確認する。影響範囲: iOS Safari ではスタンドアロン PWA モードとブラウザモードで Cookie が共有されない問題があり（research.md 観点3）、iOS 対応が必須の場合は認証フロー設計に追加制約が生じる。
- S3: パスワード等のシークレットは Cloudflare Workers の Secrets 機能で安全に管理できる — 根拠: Cloudflare Workers Secrets は標準的なシークレット管理機能として利用可能。ただし適切な運用設定が行われることを前提とする。検証方法: Workers Secrets の設定・ローテーション方針を確認する。影響範囲: シークレット管理の不備は自前 JWT・Basic 認証のセキュリティに直結する。
- S4: Hono JWT ミドルウェアは継続的にメンテナンスされ、セキュリティ修正が適時提供される — 根拠: 現時点では活発にメンテナンスされているが（research.md 観点4）、過去にアルゴリズム混同・Audience 未検証の脆弱性（GHSA-f67f-6cw9-8mq4 等）が報告されており、将来も脆弱性が発見される可能性がある。検証方法: 採用後は GitHub Security Advisory の継続的なモニタリングが必要。影響範囲: 最新版への追従を怠ると自前 JWT 実装のセキュリティリスクが高まる。
- S5: Cloudflare Access の料金プラン・機能（無料 50 ユーザー枠）は現状維持される — 根拠: 現時点で無料 50 ユーザー枠が存在することはリサーチで確認済み（research.md 観点1）。ただし Cloudflare のサービス条件は変更されうる。検証方法: Cloudflare の料金ページを定期的に確認する。影響範囲: 無料枠の条件が変更された場合、Cloudflare Access の採用コスト前提が崩れる。
- S6: セッション管理のための Workers KV が必要に応じて利用可能（無料枠の範囲内で十分）— 根拠: ADR-0003 の関連記述から Workers KV の利用が示唆されているが、具体的な採用決定はない。Workers KV 無料枠は 100,000 読み取り/日・1,000 書き込み/日。検証方法: セッション発行頻度（書き込み）・検証頻度（読み取り）が無料枠内に収まるか試算する。影響範囲: 書き込み上限を超える場合は有料プランへの移行または KV を使わない設計が必要。

## 不確実（Doubt）

- D1: iOS Safari スタンドアロン PWA モードでの Cookie 共有問題が、このアプリの実際の使用シナリオで問題になるか — 影響範囲: iOS を主要な利用環境とする場合、Cookie ベースのセッション管理方式でも初回認証フローで追加実装が必要になる可能性がある（research.md 観点3）。リスク: iOS 対応が不十分な場合、Web Share Target API からの投稿機能がモバイルで機能しない（認証エラーが発生する）。Cloudflare Access がこの問題を透過的に回避できるかも未確認。
- D2: 自前 JWT 実装における設定ミスのリスクを開発者が許容できるか、または十分にケアできるか — 影響範囲: Hono JWT ミドルウェアにはアルゴリズム明示指定・Audience 検証等の設定が必要であり、設定ミスがセキュリティ上の脆弱性になる（research.md 観点4）。リスク: 個人用途のシングルユーザーアプリゆえにセキュリティレビューが省かれやすく、設定ミスの発見が遅れる可能性がある。
- D3: Web Share Target API 経由の POST リクエスト時のセッション Cookie の実際の動作（Cloudflare Workers + Hono 環境での動作未検証）— 影響範囲: 理論上はセッション Cookie が自動付与されるが、Cloudflare のエッジネットワーク・Workers 環境での実際の動作が未検証。リスク: エッジでの Cookie 処理（SameSite 属性・Secure 属性・ドメインマッチ等）が想定通りに動作しない場合、Web Share Target API との互換性要件を満たせない。Cloudflare Access の場合は独自トークン（Cf-Access-Jwt-Assertion ヘッダー）の挙動も確認が必要。
