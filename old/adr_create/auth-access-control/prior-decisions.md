# 関連する既存決定

## 関連ADR

### ADR-0001: PWA+REST API構成のホスティング基盤選定
- 決定: Cloudflare Pages（フロントエンド）+ Cloudflare Workers（バックエンド）を採用する。
- 今回の議題との関連: ホスティング基盤が Cloudflare に確定しているため、認証方式の選択肢が技術的に制約される。具体的には、Cloudflare Access（エッジ認証）が利用可能な構成であること、Workers 上で JWT 発行・検証などの自前認証を実装可能であること、Web Share Target API からの POST リクエストは Workers（Hono）で受け取ることが決定済みであり認証方式との互換性を確保する必要があること。
- 制約への影響:
  - 月額コスト 0 円（無料ティア内）の制約が引き続き適用される（ADR-0001 C3 より）。有料の認証サービスは採用できない。
  - バックエンドは Cloudflare Workers（V8 Isolate ランタイム）で動作するため、認証ミドルウェアは Workers ランタイムと互換性を持つ必要がある。
  - シングルユーザー構成（ADR-0001 C4 より）。マルチユーザー向けの複雑な認可設計は不要。
  - 画面表示 3 秒以内の要件（ADR-0001 C5 より）。認証処理のレイテンシも考慮が必要。
  - フレームワークは Hono であることが確定済み（ADR-0001 の決定内容より）。Hono のミドルウェア差し替えで認証方式変更が可能（decision-statement.md の可逆性評価の根拠）。

### ADR-0002: フロントエンドフレームワーク選定
- 決定: React（Vite + React 19）を採用する。
- 今回の議題との関連: フロントエンドが React で確定しているため、認証フロー（ログイン画面の実装、セッション/トークンのクライアントサイド管理）は React で実装することになる。Cloudflare Access を採用する場合はフロントエンド側での認証処理が不要になる一方、自前 JWT を採用する場合はフロントエンドのトークン保存・送信処理が必要になる。
- 制約への影響:
  - フロントエンドは Cloudflare Pages にデプロイされた静的 SPA であり、サーバーサイドレンダリングは現時点では行わない（S6 として仮定）。認証コールバックの受け取り等は React のルーティングで処理する必要がある。

### ADR-0003: データベース選定
- 決定: Cloudflare D1 を採用する。
- 今回の議題との関連: ADR-0003 において「認証を自前実装する」ことを明示的にトレードオフとして受け入れている（受け入れたトレードオフの (1)）。これは今回の決定において「D1 には組み込み認証機能がないため、JWT 発行・検証・セッション管理を Workers 側で実装する必要がある」という前提が既に確定していることを意味する。また、セッション管理のストレージとして Workers KV 等の追加実装が必要とされており（ADR-0003 ALT-1 OBJ-2 詳細より）、今回の認証方式の選定においてその具体的実装方法を決定する。
- 制約への影響:
  - Supabase Auth のような外部 BaaS 認証サービスを認証基盤として採用することは、ADR-0003 でデータベースを D1 に確定した判断と必ずしも矛盾しないが、ADR-0003 では「シングルユーザーのため認証の複雑さは限定的であり実装コストを受け入れる」とされており、過度に複雑な認証基盤の採用は今回のスコープと齟齬が生じる可能性がある。
  - Workers KV はセッション/トークン情報の一時保存先として今回の決定に関連する可能性がある（Cloudflare インフラ統一の観点）。
