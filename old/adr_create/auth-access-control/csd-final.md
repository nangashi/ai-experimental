# CSD分類

## 確実（Certainty）

- C1: 有料の認証サービスは採用しない — 根拠: ADR-0001 C3 より確定済み。有料の認証サービス（Auth0 有料プラン・Supabase Pro 等）は選択不可。※コスト0の継続性はS5に委譲。
- C2: バックエンドは Cloudflare Workers（V8 Isolate ランタイム）上の Hono で動作する — 根拠: ADR-0001 の決定内容より確定済み。Node.js 専用ライブラリや OS 依存の認証ライブラリは利用不可。
- C3: シングルユーザー構成であり、マルチユーザー向けの複雑な認可設計は不要 — 根拠: ADR-0001 C4 より確定済み。
- C4: フロントエンドは React（Cloudflare Pages にデプロイされた静的 SPA）であり、サーバーサイドレンダリングは行わない — 根拠: ADR-0002 より確定済み。
- C5: 画面表示 3 秒以内の要件が存在する — 根拠: ADR-0001 C5 より確定済み。認証処理のレイテンシも要件の対象。
- C6: Cloudflare D1 採用済みであり、D1 には組み込み認証機能がない — 根拠: ADR-0003 より確定済み。ADR-0003 では「認証を自前実装する」ことをトレードオフとして受け入れている。ただし Cloudflare Access を採用する場合は Workers 側での認証実装は不要になりうる。
- C7: Web Share Target API の POST リクエストに Authorization ヘッダーはブラウザから自動付与されない — 根拠: research.md 観点3（MDN・Chrome for Developers）。Bearer JWT 認証方式は Web Share Target API との互換性に根本的な課題を持つ。
- C8: Web Share Target API の利用には PWA としてのインストールが前提となる — 根拠: research.md 観点2。
- C9: HTTPS 通信が保証されている — 根拠: Cloudflare Pages および Workers はデフォルトで HTTPS を強制する。NFR-4（HTTPS必須）とも整合。Cookie の Secure 属性・JWT の署名検証等の前提。

## 仮定（Supposition）

- S1: アプリを利用するのは作者本人のみであり、将来もマルチユーザー化しない — 根拠: decision-statement.md にシングルユーザーと明示。ただしADR-0003に「将来のマルチユーザー化」への言及あり。検証方法: プロダクトの将来計画を確認する。
- S2: iOS Safari を含む複数のブラウザ・OS 環境からの利用が想定される — 根拠: Web Share Target API のモバイルからのシェアユースケースとして iOS が主要対象と推測。検証方法: 利用端末の環境を確認する。
- S3: パスワード等のシークレットは Cloudflare Workers の Secrets 機能で安全に管理できる — 根拠: Workers Secrets は標準的なシークレット管理機能として利用可能。検証方法: Workers Secrets の設定・ローテーション方針を確認する。
- S4: Hono JWT ミドルウェアは継続的にメンテナンスされ、セキュリティ修正が適時提供される — 根拠: 現時点では活発にメンテナンスされているが、過去に脆弱性報告あり（research.md 観点4）。検証方法: GitHub Security Advisory の継続的モニタリング。
- S5: Cloudflare Access の料金プラン・機能（無料 50 ユーザー枠）は現状維持される — 根拠: research.md 観点1で確認済み。ただし Cloudflare のサービス条件は変更されうる。検証方法: Cloudflare の料金ページを定期的に確認する。
- S6: セッション管理のための Workers KV が必要に応じて利用可能（無料枠の範囲内で十分）— 根拠: ADR-0003 の関連記述。Workers KV 無料枠は 100,000 読み取り/日・1,000 書き込み/日。検証方法: セッション発行・検証頻度が無料枠内に収まるか試算する。
- S7: 認証方式の変更はミドルウェア差し替えで対応可能（中可逆性）— 根拠: Hono のミドルウェアアーキテクチャ。ただし Cloudflare Access と自前実装の間の移行はインフラ層（DNS・テナント設定等）の変更を伴い、ミドルウェア差し替えのみでは完結しない場合がある。検証方法: 採用する方式と他方式間の移行パスを確認する。
- S8: このアプリはインターネット公開されており、作者本人以外のアクセスは不正アクセスとして拒否する必要がある — 根拠: Cloudflare Pages/Workers はデフォルトでインターネット公開。シングルユーザー個人用ツールだが、悪意ある第三者からの保護が必要。検証方法: 脅威モデル（誤操作防止レベルか攻撃者対策か）を確認する。
- S9: 認証は全 API エンドポイントに一律の方式で適用する — 根拠: 暗黙の前提として検出。ただし C7 の制約（Web Share Target POST で Authorization ヘッダー非自動付与）を踏まえると、エンドポイント別のハイブリッド構成も代替案として検討しうる。検証方法: エンドポイント別に異なる認証方式を許容するか確認する。

## 不確実（Doubt）

- D1: iOS Safari スタンドアロン PWA モードでの Cookie 共有問題が、このアプリの実際の使用シナリオで問題になるか — 影響範囲: iOS で Cookie ベースのセッション管理を利用する場合、初回認証フローで追加実装が必要になる可能性。リスク: Web Share Target API からの投稿が認証エラーで失敗する。
- D2: 自前 JWT 実装における設定ミスのリスクを開発者が許容できるか — 影響範囲: Hono JWT ミドルウェアの設定ミスがセキュリティ脆弱性に直結。リスク: 個人用途ゆえセキュリティレビューが省かれやすく、発見が遅れる可能性。
- D3: Web Share Target API 経由の POST リクエスト時のセッション Cookie の実際の動作（Cloudflare Workers + Hono 環境での動作未検証）— 影響範囲: Cookie の Secure/SameSite 属性・ドメインマッチ等がエッジ環境で想定通りに動作するか未検証。リスク: 互換性要件を満たせない場合がある。
