# プレモーテム分析

## ALT-1: Cloudflare Access（Zero Trust エッジ認証）

### 失敗シナリオ 1: Cloudflare Access 無料枠の突然の廃止または条件変更

- 状況: 1年後、Cloudflare が Zero Trust の無料プランの変更を発表し、シングルユーザー構成であっても有料プランへの移行が必要になる。C1（有料の認証サービスは採用しない）に抵触するため、認証方式全体の見直しが迫られる。
- 原因: Cloudflare Access の料金プランは外部サービスの事業判断に依存しており、開発者側でコントロールできない。S5（無料枠の現状維持）を自明の前提として扱い、代替の出口戦略を準備しなかった。
- CSD関連: S5（Cloudflare Access の料金プラン・機能の現状維持）の崩壊
- 深刻度: high
- 発生確率: low
- 軽減策: Cloudflare の料金ページを四半期ごとにモニタリングする。ALT-2 または ALT-3 への移行パスをあらかじめ設計し、認証ロジックを Workers 側に移植する際の手順書を作成しておく。Cloudflare Access への依存を隔離するレイヤー（Workers 側でのトークン検証）を整備して移行コストを低減する。

### 失敗シナリオ 2: iOS Safari PWA モードでの認証フロー断絶

- 状況: 1年後、Web Share Target API 経由での共有が iOS Safari スタンドアロン PWA モードから頻繁に失敗するようになる。cf_authorization Cookie がブラウザコンテキストとスタンドアロンコンテキスト間で共有されず、認証リダイレクトがスタンドアロンモードで正しく完了しない。
- 原因: D1（iOS Safari PWA モードでの Cookie 共有問題）を「可能性あり」と認識しながら検証を後回しにした。Cloudflare Access の認証フロー（リダイレクトベース）が iOS のスタンドアロン PWA と組み合わさる挙動を事前に確認しなかった。
- CSD関連: D1（iOS Safari スタンドアロン PWA モードでの Cookie 共有問題）の悪化シナリオ
- 深刻度: high
- 発生確率: medium
- 軽減策: 開発初期段階で iOS Safari PWA モードと Cloudflare Access の認証リダイレクトの挙動を実機検証する。問題が確認された場合は、スタンドアロンモードでの再認証フロー（例: インブラウザでの認証後に Cookie を引き継ぐ仕組み）を検討する。

### 失敗シナリオ 3: Workers 側での CF-Access-Jwt-Assertion 検証省略によるバイパス

- 状況: 1年後、Workers に直接アクセスするエンドポイント（Cloudflare Access を迂回可能なルート）への不正アクセスが発生する。Cloudflare Access をトラフィックの前段に置くだけで、Workers 側での JWT 検証を省いていたため、Access をバイパスした直接リクエストを検証できなかった。
- 原因: 「Cloudflare Access を設定すれば安全」という誤った前提で Workers 側の検証実装を省いた。インフラ設定（Access ポリシー）のみに依存し、コードレベルの検証を行わなかった。S8（インターネット公開環境での第三者からの保護）の重要性を過小評価した。
- CSD関連: S8（不正アクセス防止の必要性）との関連。なし（実装上の見落としによる想定外の失敗）
- 深刻度: high
- 発生確率: medium
- 軽減策: Workers 側で CF-Access-Jwt-Assertion ヘッダーの署名検証を必ず実装する。Cloudflare の公式ドキュメントに従い、JWKS エンドポイントから取得した公開鍵で JWT を検証する。Access ポリシーの設定と Workers の検証は両方必須の「二重防御」として設計する。

### 失敗シナリオ 4: Cloudflare Access の仕様変更・API 互換性の破壊

- 状況: 1年後、Cloudflare が cf_authorization Cookie の仕様（名称・形式・有効期限ロジック）を変更した結果、Workers 側の検証ロジックが機能しなくなる。または Cloudflare Access ダッシュボードの UI が大幅に変更され、既存のポリシー設定が無効化される。
- 原因: 外部マネージドサービスの仕様変更に対する追跡コストを過小評価した。Workers 側に認証コードを書かないことの利点を優先した結果、サービス変更への追随が受動的になった。
- CSD関連: S5（Cloudflare Access の料金プラン・機能の現状維持）の部分的崩壊
- 深刻度: medium
- 発生確率: low
- 軽減策: Cloudflare の変更履歴（ブログ・Changelog・GitHub Issues）を定期的にモニタリングする。Workers 側の JWT 検証ロジックを抽象化し、公開鍵取得元・検証アルゴリズムを設定で切り替えられるようにしておく。

## ALT-3: 署名付き Cookie / JWT in Cookie（ステートレス）

### 失敗シナリオ 1: JWT 設定ミスによるセキュリティ脆弱性の放置

- 状況: 1年後、Hono JWT ミドルウェアのアルゴリズム検証設定が不適切（alg: any またはデフォルト許容）であったことが、外部のセキュリティ報告または侵害後調査で判明する。アルゴリズム混同攻撃（none アルゴリズムや RS/HS 混同）により、自前実装の認証が実質的にバイパスされていた。
- 原因: D2（自前 JWT 実装における設定ミスのリスクを開発者が許容できるか）を軽視した。個人開発ゆえにセキュリティレビューが省かれ、Hono JWT ミドルウェアの設定をデフォルト値のまま使用した。S4（Hono JWT ミドルウェアの継続的メンテナンス）を前提としながら、過去の脆弱性情報（アルゴリズム混同・aud 未検証）を実際の設定に反映しなかった。
- CSD関連: D2（設定ミスリスクの開発者許容）の顕在化、S4（Hono JWT ミドルウェアのメンテナンス継続）への依存
- 深刻度: high
- 発生確率: medium
- 軽減策: JWT 設定時に alg を明示的に指定する（例: HS256 固定）。aud クレームの検証を有効化する。Hono JWT ミドルウェアの GitHub Security Advisory を定期的に確認し、脆弱性修正リリース後は速やかに更新する。実装後にセキュリティチェックリストを用いてレビューを行う（外部ツール利用や OWASP JWT Security Cheat Sheet を参照）。

### 失敗シナリオ 2: ステートレス方式によるセッション無効化不能による被害拡大

- 状況: 1年後、本人の認証情報が漏洩するか、デバイスを紛失した際に、発行済みの JWT を即時無効化できないことが問題となる。JWT の有効期限（例: 7日間）が切れるまでの間、攻撃者がアプリを操作し、個人データの漏洩または外部へのデータ送信が発生する。
- 原因: 「シングルユーザーだから即時無効化は不要」という S1（マルチユーザー化しない）と S8（不正アクセス防止の必要性）の組み合わせから生まれた油断。ステートレス方式のコスト・シンプルさの利点を優先し、緊急無効化フローを設計しなかった。
- CSD関連: S8（インターネット公開環境での第三者からの保護必要性）の重要性との乖離。なし（設計上の意図的選択が問題化した想定外の失敗）
- 深刻度: high
- 発生確率: low
- 軽減策: JWT の有効期限を短く設定する（例: 1〜2時間）とともに、Workers Secrets のシークレットローテーションによる「全トークン一括無効化」フローを手順書として整備する。デバイス紛失・漏洩時の対応手順（シークレット即時ローテーション → 再ログイン強制）を事前に準備する。必要になれば KV によるブラックリストを追加実装できるように設計しておく。

### 失敗シナリオ 3: iOS Safari PWA モードでの Cookie 挙動による Web Share Target 失敗

- 状況: 1年後、iOS Safari スタンドアロン PWA モードで Web Share Target API 経由の POST が断続的に認証エラーで失敗するようになる。JWT を格納した Cookie が、ブラウザコンテキストとスタンドアロンコンテキスト間で正しく共有されず、認証が通らない。
- 原因: D1（iOS Safari PWA Cookie 共有問題）および D3（Cloudflare Workers + Hono 環境での Cookie 動作未検証）を許容可能なリスクとして扱い、実機での検証を実施しなかった。SameSite 属性や Path 設定が Workers + Pages の構成で想定通りに機能しないケースを事前に確認しなかった。
- CSD関連: D1（iOS Safari PWA Cookie 共有問題）の顕在化、D3（Workers 環境での Cookie 動作未検証）の顕在化
- 深刻度: high
- 発生確率: medium
- 軽減策: 開発初期段階で iOS 実機を用いた Web Share Target + Cookie 認証の動作確認を実施する。SameSite=Lax への緩和やドメイン設定の調整など、プラットフォーム特有の対処法を確認しておく。問題が確認された場合の代替フロー（例: 認証後の URL パラメータ経由での Share 再試行）を設計しておく。

### 失敗シナリオ 4: Workers Secrets のシークレット漏洩によるトークン全件偽造

- 状況: 1年後、Workers Secrets に保存した JWT 署名シークレットが何らかの経路（ソースコードへの誤コミット、CI/CD パイプラインのログ漏洩等）で外部に漏洩する。HMAC-SHA256 の署名シークレットが漏洩した場合、攻撃者は任意の JWT を偽造でき、認証が完全に無効化される。
- 原因: S3（Workers Secrets の安全なシークレット管理）を過信した。Secrets の設定・ローテーション方針が文書化されておらず、ローカル開発環境での wrangler.toml や .dev.vars ファイルへの誤記録リスクを認識していなかった。
- CSD関連: S3（Workers Secrets でのシークレット安全管理）の崩壊
- 深刻度: high
- 発生確率: low
- 軽減策: ローカル開発用シークレットは .dev.vars に格納し、.gitignore に必ず追加する。wrangler.toml にはシークレット値を記載しない。シークレット漏洩を検知した場合の即時ローテーション手順を事前に整備する。Workers Secrets の定期的なローテーションポリシーを設定する。
