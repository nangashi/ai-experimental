# OBJ-3: Cloudflare Workers 環境での実装・運用の簡潔さ — 評価結果

## 提案した代替案

- **Cloudflare Access（Zero Trust）**: Cloudflare インフラの一部として提供される認証レイヤーを Workers の前段に配置し、Workers 側での認証ロジックを不要にする。V8 Isolate ランタイムの制約を完全に回避でき、Hono ミドルウェアへの認証コードの組み込みが不要。設定はダッシュボード操作で完結し、コードの変更なく認証要件を管理できる点で実装・運用負担が最小。
- **Cookie セッション（Workers KV バックエンド）**: Hono のカスタムミドルウェアとして Cookie ベースのセッション管理を自前実装する。サインイン時に署名付きセッション ID を発行・KV に保存し、以降のリクエストで Cookie を検証する方式。Hono・Cloudflare Workers と完全に互換し、Node.js 依存ゼロで実装可能。ただし KV 読み書きコードとセッションライフサイクル管理が必要となる。
- **Hono JWT ミドルウェア（ステートレス）**: Hono 公式 JWT ミドルウェアをそのまま利用し、ステートレス認証を実装する。KV などの外部ストレージが不要で、Hono に組み込まれた実装であるため Workers ランタイムとの互換性が保証される。ただしセキュリティ設定（アルゴリズム明示・aud 検証等）の自前設定が必要。

## 評価

### Cloudflare Access（Zero Trust）

- 評価: ◎
- 利点:
  - Workers 側に認証コードを一切記述しない。ミドルウェア不要でエンドポイントのビジネスロジックに集中できる。
  - Cloudflare ダッシュボードによる設定変更が可能で、コードのデプロイなしに認証ポリシーの更新（IP制限・デバイス証明書等）ができる。
  - セキュリティパッチ・アップデートは Cloudflare 側が担当するため、ライブラリのバージョン管理が不要。
  - トラブルシューティングは Cloudflare ダッシュボードのログ・分析ツールで行え、Workers のログとは独立して確認できる。
  - V8 Isolate ランタイムの制約（Node.js 非互換・実行時間・メモリ制限）に完全に非依存であり、ランタイム制約を考慮する必要がない（C2 制約をクリア）。
- 欠点:
  - Cloudflare Access の設定（テナント・アプリケーション登録・ポリシー設定）に初期セットアップコストが発生する。Hono JWT に比べると初期設定の学習コストが高い。
  - Web Share Target API との互換性確認やエンドポイント別挙動の調査は別途必要（ただし OBJ-3 の範囲外）。
  - Cloudflare のサービス仕様変更がインフラ側の設定変更を直接要求する場合があり、アプリのコードと設定が分離されている分、変更追跡が複雑になりうる（S5 依存）。
- 根拠: 認証ロジックの Workers への実装ゼロという特性により、Workers 上の実装・運用の簡潔さを最高水準で達成する。Hono ミドルウェアの保守・セキュリティ設定・KV 管理が一切不要であり、長期的な運用負担が最小。
- CSD依存: S5（Cloudflare Access の料金プラン・機能の現状維持）。サービス仕様変更時にはダッシュボード設定の見直しが必要になりうる。

### Cookie セッション（Workers KV バックエンド）

- 評価: ○
- 利点:
  - Hono のミドルウェアとして実装するため、Workers ランタイムとの互換性は完全に自前で制御できる。外部サービスへの依存がない（Cloudflare KV は同一インフラ内）。
  - セッション管理のロジックがコードとして明示的に存在するため、動作をコードベースで把握・変更可能。
  - KV はシングルユーザーの使用量では無料枠（100,000 読み取り/日・1,000 書き込み/日）に収まる可能性が高く（S6 依存）、外部サービス依存がない分の安定性がある。
  - セッション失効・強制ログアウトなどのセッションライフサイクル管理をコードで柔軟に制御できる。
- 欠点:
  - セッション ID の生成・保存・検証・失効の実装が必要であり、Hono JWT ミドルウェア利用に比べてコード量が増える。
  - KV の読み書きごとにラウンドトリップが発生するため、認証処理がレイテンシに影響する可能性がある（ただし OBJ-4 の範囲）。
  - Workers KV の障害・レート制限が認証機能の可用性に直結するリスクがある（Cloudflare インフラ内であるため低リスクだが存在する）。
  - セッション管理の実装バグ（セッション固定・CSRF等）のリスクは自前実装ゆえ存在する（ただし OBJ-2 の範囲）。
- 根拠: Workers 内に完結する実装であり、V8 Isolate ランタイムとの互換性問題は発生しない（C2 制約クリア）。ただしセッションライフサイクル管理の実装が必要なため、Cloudflare Access に比べると実装・運用コストは増加する。許容範囲内の実装コストで OBJ-3 を達成できる。
- CSD依存: S6（Workers KV が無料枠内に収まること）。KV 使用量が想定を超えた場合のコスト増加リスクがある。

### Hono JWT ミドルウェア（ステートレス）

- 評価: ○
- 利点:
  - Hono 公式ミドルウェアであり、Workers ランタイムとの互換性が公式にサポートされている。Node.js 依存なし（C2 制約クリア）。
  - ステートレス設計により KV などの外部ストレージが不要。Workers の処理だけで完結し、アーキテクチャがシンプル。
  - JWT の発行・検証は Workers 内の標準的な暗号 API（Web Crypto API）で行えるため、ランタイム制約を回避できる。
  - コードとしての実装量は Cookie セッションより少なく、ミドルウェアを適用するだけの簡潔な実装が可能。
- 欠点:
  - セキュリティ設定（アルゴリズム明示・`aud` クレーム検証・シークレット管理）を自前で設定する必要があり、設定ミスが発生しやすい（D2 依存）。過去に Hono JWT に脆弱性報告があり（GHSA-f67f-6cw9-8mq4 等）、継続的なバージョン監視が必要（S4 依存）。
  - Authorization ヘッダーを使う Bearer トークン方式では Web Share Target API と非互換（C7 制約）。OBJ-1 の制約を満たすためには Cookie を使ったトークン送付やエンドポイント別ハイブリッド構成が必要となり、その場合は実装が複雑化する。
  - ステートレスなため、トークン無効化（ログアウト・強制失効）の仕組みを別途実装しないとできない（ブラックリスト KV 等が必要になり、「ステートレス」の利点が薄れる可能性）。
- 根拠: Hono 公式ミドルウェアを利用する分、Workers ランタイムとの互換性は良好で実装の出発点として簡潔。しかし OBJ-1（Web Share Target 互換性）の制約を満たすための追加実装が運用の複雑さを引き起こすため、OBJ-3 単体での評価としては「十分に達成する」水準にとどまる。
- CSD依存: D2（設定ミスのリスクを開発者が許容できるか）、S4（Hono JWT ミドルウェアのメンテナンス継続性）。これらが崩れた場合、OBJ-3 の評価は△方向に下がりうる。
