# 判断基準（Objectives）

## OBJ-1: Web Share Target API との互換性

- 種別: fundamental（根本目的）
- 定義: 認証方式が、ブラウザから Authorization ヘッダーを自動付与できない Web Share Target API の POST リクエストと共存できること
- 評価の視点:
  - Web Share Target API 経由の POST リクエストで、認証エラーなく機能するか
  - 追加の実装コスト（ワークアラウンド）なしに互換性を確保できるか
  - iOS Safari スタンドアロン PWA モードでの Cookie 挙動（D1・D3）に左右されないか

## OBJ-2: セキュリティ強度と設定ミスリスクの低さ

- 種別: fundamental（根本目的）
- 定義: インターネット公開された個人用ツールとして、第三者の不正アクセスを確実に遮断できること。かつ、開発者単独による実装・設定ミスでセキュリティホールが生まれにくいこと
- 評価の視点:
  - 既知の攻撃ベクター（セッションハイジャック・トークン窃取・CSRF等）への耐性
  - 設定ミスが致命的なセキュリティ脆弱性に直結するリスクの大きさ（D2）
  - セキュリティ修正の適時提供・メンテナンス継続性（S4）

## OBJ-3: Cloudflare Workers 環境での実装・運用の簡潔さ

- 種別: fundamental（根本目的）
- 定義: V8 Isolate ランタイムの制約下で、認証ロジックの実装・デプロイ・長期運用が最小の負担で行えること
- 評価の視点:
  - Node.js 非互換ライブラリへの依存なく実装できるか（C2）
  - 認証基盤のセットアップ・設定変更・トラブルシューティングの煩雑さ
  - 外部サービス依存度（サービス仕様変更・停止リスク：S5）

## OBJ-4: 認証処理のレイテンシの低さ

- 種別: fundamental（根本目的）
- 定義: 画面表示 3 秒以内の要件（C5）を満たすうえで、認証処理がボトルネックにならないこと
- 評価の視点:
  - 認証処理が追加するネットワークラウンドトリップ数
  - Cloudflare Workers のエッジで完結するか、外部サービスへの通信が必要か
  - セッション検証のコスト（Workers KV 参照回数など）

## OBJ-5: コスト持続可能性

- 種別: fundamental（根本目的）
- 定義: 個人用ツールとして、認証基盤の運用コストが長期的に無料または最小限であること
- 評価の視点:
  - 採用する認証基盤の無料枠と想定使用量のマージン（S5・S6）
  - 無料枠の変更・廃止リスク（S5）
  - 有料プランへの強制移行が生じた場合の代替手段の有無

---

## 手段目的→根本目的の変換記録

- 手段: 「JWT を採用する」 → 根本: 「実装の簡潔さ・Workers 環境との適合性」（OBJ-3）— 理由: JWT はステートレス処理という手段であり、真の目的は実装・運用負荷の最小化
- 手段: 「Cookie ベースのセッション管理にする」 → 根本: 「Web Share Target API との互換性確保」（OBJ-1）— 理由: Cookie はトークン送付の手段であり、真の目的は Share Target POST リクエストとの共存
- 手段: 「Cloudflare Access を使う」 → 根本: 「セキュリティ強度と設定ミスリスクの低さ」（OBJ-2）および「実装・運用の簡潔さ」（OBJ-3）— 理由: Cloudflare Access 採用は外部化による手段であり、真の目的は安全性確保と実装コスト削減
