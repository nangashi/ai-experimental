# レッドチーム検証結果

## 再分類提案

- C1（月額コスト 0 円）: Certainty の一部 → Supposition へ格下げ提案 — C1 の根拠は「ADR-0001 C3 より確定済み」だが、これは「有料サービスを採用しない」という方針の確定であって、「将来にわたってコスト 0 が保証される」という事実の確定ではない。Cloudflare Access の無料枠継続性は S5 で Supposition として分類されており、S5 が崩れると C1 の「コスト 0」という主張も崩れる。具体的な反証シナリオ: Cloudflare が無料枠を廃止または条件変更した場合、C1 の前提が崩壊する（S5 と同一の崩壊条件を持つ）。提案: C1 の記述を「有料の認証サービスは採用しない（ADR-0001 C3 より確定）」という方針の Certainty として再定式化し、「現時点でのコスト 0 という状態の継続性」については S5 に委譲する形で記述を分離する。

- C7（Hono ミドルウェア差し替えで認証方式の変更が可能、中可逆性）: Certainty → Supposition へ格下げ提案 — 「Hono ミドルウェア差し替えで変更可能」という性質は、選択する代替案によって成立するかどうかが変わる。自前 JWT 実装どうしの切り替えであれば妥当だが、Cloudflare Access を採用している場合に自前 JWT へ移行する（またはその逆）際は、Cloudflare Access のテナント設定・DNS 設定・ルーティング変更等のインフラ層の変更を伴うため、「ミドルウェア差し替えのみで完結する」は成立しない。具体的な反証シナリオ: Cloudflare Access → 自前 JWT への移行時に、Access の認証ルールの削除・Workers の認証ミドルウェアの追加・全エンドポイントのルーティング変更が必要になり、「中可逆性」の見積もりが楽観的すぎる場合がある。また可逆性評価は本来 Objectives フェーズで評価する判断基準であり、CSD の Certainty に分類するのは過剰な確実性付与である。提案: C7 を Supposition に格下げし、「選択する認証方式によってミドルウェア差し替えのみで変更可能な場合と、インフラ層の変更を伴う場合がある」と再記述する。

## 暗黙の仮定の検出

- 「このアプリはインターネットに公開されており、不特定の第三者からのアクセスから保護する必要がある」という仮定 — 推奨分類: Supposition — 理由: 認証が必要という前提全体は、アプリが外部ネットワークから到達可能であり、悪意ある第三者からの保護が必要であることを前提にしている。Cloudflare Pages/Workers はデフォルトでインターネット公開されるため実際上はほぼ確実だが、「どの程度のセキュリティ強度が必要か」（誤操作防止レベルか、悪意ある攻撃者への対策か）が明示されていない。シングルユーザーのみが利用する個人用ツールという性質上、脅威モデルが曖昧なまま認証方式の評価が行われると、セキュリティ強度の評価軸が各エバリュエーターで統一されない恐れがある。Supposition として「このアプリはインターネット公開されており、作者本人以外のアクセスは不正アクセスとして拒否する必要がある」と明示することを推奨する。

- 「HTTPS 通信が保証されている」という仮定 — 推奨分類: Certainty（追加項目として記録推奨）— 理由: Cookie の Secure 属性・JWT の署名検証・Bearer トークンの通信路保護はすべて HTTPS を前提とする。Cloudflare Pages および Workers はデフォルトで HTTPS を提供するため実際には確実だが、CSD に明示されていない。D3「Web Share Target API 経由の POST リクエスト時のセッション Cookie の実際の動作」の議論においても、HTTPS が前提として明示されていないと Cookie の Secure 属性に関する考察が不完全になる。Certainty の追加項目として「Cloudflare Pages および Workers はデフォルトで HTTPS を強制する」を明示することを推奨する。

- 「認証は全 API エンドポイントに一律の方式で適用する」という仮定 — 推奨分類: Supposition — 理由: Web Share Target API への POST エンドポイントと通常の管理画面 GET エンドポイントを、同一の認証方式で保護することを暗黙に前提にしている。しかし C8 では「Web Share Target API の POST で Authorization ヘッダーが自動付与されない」という固有の制約が確実として分類されており、エンドポイントによって制約が異なる。エンドポイント別に認証方式を分けるハイブリッドアーキテクチャの可能性が代替案として検討されていないため、この仮定が明示されていないと Objective Evaluator が「全エンドポイント統一方式」のみを評価し、ハイブリッド構成を見落とす恐れがある。

## 検証不要と判断した項目

- C2（バックエンドは Cloudflare Workers + Hono）: ADR-0001 による確定済みの技術スタック。今回の審議で覆ることは想定されない。検証済み、分類維持。
- C3（シングルユーザー設計が前提）: ADR-0001 C4 の確定事項として現在の設計要件を記述しており、「現時点の設計方針はシングルユーザー向け」という意味では確実。S1（将来もマルチユーザー化しない）と役割が重複していないため両者の共存は妥当。検証済み、分類維持。
- C4（フロントエンドは React SPA on Cloudflare Pages）: ADR-0002 による確定済み。検証済み、分類維持。
- C5（画面表示 3 秒以内）: ADR-0001 C5 による確定済みの非機能要件。検証済み、分類維持。
- C6（D1 に組み込み認証機能なし・自前実装前提）: D1 の技術仕様として事実。ただし「Workers 側で JWT 発行・検証・セッション管理を実装する必要がある」という記述は選択肢依存の帰結を含む（Cloudflare Access を採用する場合は Workers 側での実装は不要になる）。分類（Certainty）は維持しつつ、記述の精度向上を推奨する。検証済み、分類維持（記述修正推奨）。
- C8（Web Share Target API で Authorization ヘッダー自動付与なし）: MDN ドキュメントおよび Chrome for Developers の仕様記述に基づく確認済みの技術的事実。ブラウザの振る舞いとして仕様に定義されており、短期間で変化する性質のものではない。検証済み、分類維持。
- C9（Web Share Target API 利用には PWA インストールが前提）: Web Share Target API の仕様要件として確認済み。検証済み、分類維持。
- S1〜S6 の全項目: 各仮定に「検証方法」が具体的に記述されており、検証可能な形式として適切。Doubt への格下げが必要な項目は検出されなかった。検証済み、分類維持。
- D1〜D3 の全項目: 各不確実性は未検証・未確認の事実に関するものであり Doubt として適切に分類されている。検証済み、分類維持。
