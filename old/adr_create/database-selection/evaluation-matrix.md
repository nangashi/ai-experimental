# トレードオフ行列

## マトリクス

| 判断基準 | ALT-1: Cloudflare D1 | ALT-2: Supabase | ALT-3: TiDB Serverless | ALT-4: Turso (libSQL) |
|----------|:---:|:---:|:---:|:---:|
| OBJ-1: データアクセスの応答性能 | ◎ | △ | △ | ○ |
| OBJ-2: 認証機能の統合性 | △ | ◎ | △ | △ |
| OBJ-3: バックアップ・リストアの実行可能性 | △ | ○ | △ | △ |
| OBJ-4: 運用負荷の最小化 | ◎ | ○ | △ | ○ |
| OBJ-5: 長期的なコスト安定性 | ○ | △ | ○ | ○ |

## 各セルの詳細

### ALT-1: Cloudflare D1

#### ALT-1 × OBJ-1（◎）
- 評価: ◎
- 利点: Binding 接続は接続確立コストが実質ゼロ。Workers と同一エッジインフラで動作しネットワークレイテンシが構造的に最小
- 欠点: D1 リージョンは作成時に固定。Workers Edge との物理距離が離れる場合あり
- 根拠: C5（コネクションプールなし）制約下で最も有利な接続方式
- CSD依存: D3（Workers-D1 リージョン間レイテンシ）。C3 によりユーザーは開発者1名のため初期設定で軽減可能

#### ALT-1 × OBJ-2（△）
- 評価: △
- 利点: Workers KV をセッションストアとして組み合わせ可能。Cloudflare エコシステム内で完結
- 欠点: 認証機能なし。JWT 発行・検証・パスワードハッシュ等を完全自前実装。セッション管理に KV 等の追加実装が必要
- 根拠: OBJ-2 の「低コストで認証を実現」の観点で自前実装の工数がかかる
- CSD依存: C3（シングルユーザー）。複数ユーザー拡張時に実装工数が大幅増加

#### ALT-1 × OBJ-3（△）
- 評価: △
- 利点: `wrangler d1 export` で SQL ダンプ取得可能。SQLite 形式でローカル確認が容易
- 欠点: FTS5 仮想テーブルを含む DB はエクスポート非サポート。リストア後の FTS5 インデックス再構築手順が必要
- 根拠: FTS5 不使用（LIKE 検索のみ）なら○相当だが、FTS5 採用時のエクスポート制約が障壁
- CSD依存: S2（LIKE 検索で充足するか）、D1（FTS5 バックアップ制約の許容可能性）

#### ALT-1 × OBJ-4（◎）
- 評価: ◎
- 利点: 同一プラットフォーム統合で管理対象追加ゼロ。`wrangler` CLI 一元管理。Binding 方式で環境変数不要
- 欠点: GUI ツールの充実度は Supabase に劣る。FTS5 使用時のバックアップ手順整備が追加負荷
- 根拠: プラットフォーム統合による管理一元化が個人開発の運用負荷を最小化
- CSD依存: S4（D1 Free プラン継続）

#### ALT-1 × OBJ-5（○）
- 評価: ○
- 利点: Free プラン 500 MB/DB は個人用途で十分。超過時は従量課金で予測可能。Cloudflare は大企業でインフラとしての安定性が高い
- 欠点: 2025-02-10 に無料枠超過挙動が変更されており、制限変更の予見性がない
- 根拠: 大企業のインフラサービスとして継続性は高いが、プラン変更リスクは残存
- CSD依存: S4、D2

### ALT-2: Supabase

#### ALT-2 × OBJ-1（△）
- 評価: △
- 利点: リージョン選択可能（東京等）。PostgreSQL の高度なクエリ最適化
- 欠点: 外部 HTTP API 呼び出しで毎リクエスト TCP+TLS のオーバーヘッド。D1 Binding より構造的に不利
- 根拠: 3秒要件超過リスクは低いが、接続方式として構造的に D1 より劣る
- CSD依存: S6（CRUD+検索のみ）。複雑クエリが必要になれば PostgreSQL の優位が発揮される可能性

#### ALT-2 × OBJ-2（◎）
- 評価: ◎
- 利点: Supabase Auth（GoTrue ベース）がメール/パスワード・ソーシャルログイン・マジックリンクを即座に提供。JWT 管理自動化。RLS との統合
- 欠点: 外部サービス依存による認証リクエストのレイテンシ。`@supabase/supabase-js` の Workers 互換性確認要
- 根拠: 組み込み認証機能の成熟度で他の選択肢を大幅に上回る
- CSD依存: Supabase 無料枠（5万MAU）の継続

#### ALT-2 × OBJ-3（○）
- 評価: ○
- 利点: PostgreSQL ベースで FTS 特有のエクスポート制約なし。`pg_dump` 互換。REST API 経由の JSON エクスポートも可能
- 欠点: 無料プランで自動バックアップ・PITR なし。手動操作はローカル環境から
- 根拠: 手順の確実性は D1 より高い。S3（手動スナップショットで十分）が成立すれば十分
- CSD依存: S3、S5 相当（Supabase 無料枠継続）

#### ALT-2 × OBJ-4（○）
- 評価: ○
- 利点: 充実した GUI ダッシュボード。Auth 統合で認証運用が簡略化。Supabase CLI でローカル開発可能
- 欠点: 環境変数2つ（URL, ANON_KEY）の管理。外部アカウント管理。Free プランの非アクティブプロジェクト自動停止仕様
- 根拠: GUI と認証統合は有利だが、外部サービスの管理オーバーヘッドとポーズ仕様が個人開発で摩擦
- CSD依存: Supabase Free プランのポーズ仕様は変更されうる

#### ALT-2 × OBJ-5（△）
- 評価: △
- 利点: 500 MB ストレージ・5万 MAU。シングルユーザーでは MAU は問題なし
- 欠点: 非アクティブプロジェクト自動停止仕様。VC 資金調達型スタートアップで料金変更リスクが相対的に高い。有料移行時 $25/月
- 根拠: 自動停止仕様と料金変更リスクがコスト安定性の観点で不利
- CSD依存: Supabase 固有の無料枠継続に関する未分類の不確実性

---

### ALT-3: TiDB Serverless

#### ALT-3 × OBJ-1（△）
- 評価: △
- 利点: MySQL 互換で幅広い SQL 機能。HTTP ドライバー経由で Workers 接続可能
- 欠点: 外部 HTTP 接続のオーバーヘッド。低頻度個人利用でのコールドスタート問題。分散アーキテクチャの恩恵を受けにくい規模
- 根拠: 接続オーバーヘッド + コールドスタートで応答時間が不安定になるリスク
- CSD依存: S1（数千レコード）。規模拡大時は分散アーキテクチャの優位が発揮される可能性

#### ALT-3 × OBJ-2（△）
- 評価: △
- 利点: MySQL 互換でユーザーテーブル作成は技術的に可能
- 欠点: 認証機能なし。Workers との統合エコシステムが未整備で自前実装 + 調査コスト
- 根拠: 自前実装工数に加え、Workers 統合の調査コストも発生
- CSD依存: S6（CRUD+検索のみの場合、認証セッション管理がオーバースペック）

#### ALT-3 × OBJ-3（△）
- 評価: △
- 利点: MySQL 互換で `mysqldump` 相当の操作が原理的に可能。ダンプファイルの汎用性が高い
- 欠点: Serverless 構成の接続制約により標準ツールでの直接ダンプの実行可能性が未確認。無料プランのバックアップ機能詳細が不明
- 根拠: 手順の確実性に不確実性が高い
- CSD依存: 無料プランのバックアップ機能有無が未確認（Doubt 相当）

#### ALT-3 × OBJ-4（△）
- 評価: △
- 利点: Serverless でインフラ管理は自動化。MySQL ツールチェーンが利用可能
- 欠点: Workers との公式統合が薄い。接続設定が手動。分散 SQL の管理機能が個人用途にはオーバースペック
- 根拠: トラブルシューティングコストが運用負荷として顕在化するリスク
- CSD依存: C1（Workers 互換接続の安定動作確認が必要）

#### ALT-3 × OBJ-5（○）
- 評価: ○
- 利点: 無料枠 5 GiB は D1 の10倍。RU ベース課金で軽量アクセスでは超過しにくい。PingCAP は商用 DB 企業で事業継続性リスクが比較的低い
- 欠点: RU 単位の粒度が不明確で消費量の予測が難しい。日本市場での認知が少なく変更の予兆を察知しにくい
- 根拠: 広い無料枠だが RU の予測可能性に不確実性
- CSD依存: TiDB 無料枠継続に関する未分類の不確実性

---

### ALT-4: Turso (libSQL)

#### ALT-4 × OBJ-1（○）
- 評価: ○
- 利点: エッジレプリカによる自動ルーティングで Workers 起動ロケーションに最近接レプリカへ接続。物理的レイテンシを構造的に最小化
- 欠点: 外部 HTTP API 呼び出しで TCP+TLS オーバーヘッドが毎リクエスト発生。C3（ユーザー1名・ロケーション固定）ではエッジ分散の恩恵が限定的
- 根拠: 外部 HTTP 接続だがエッジレプリカにより Supabase/TiDB より優位
- CSD依存: S5（Turso 無料プラン継続）。C3 が崩れ複数ロケーションからのアクセスになれば評価向上

#### ALT-4 × OBJ-2（△）
- 評価: △
- 利点: SQLite 互換で認証関連テーブルを通常の SQL で扱える。HTTP ドライバーが Workers でネイティブ動作し認証チェックのレイテンシが低い
- 欠点: 認証機能なし。完全自前実装が必要。認証情報（`LIBSQL_DB_AUTH_TOKEN`）の管理が追加で必要
- 根拠: D1・TiDB と同様に認証は自前実装。Supabase Auth と比較して明確に劣る
- CSD依存: C3（シングルユーザー）、S5（Turso 無料プラン継続）

#### ALT-4 × OBJ-3（△）
- 評価: △
- 利点: SQLite 互換で `turso db shell` + `.dump` による SQL ダンプが原理的に可能。D1 の FTS5 エクスポート制約が libSQL 固有の制約として明示されていない
- 欠点: バックアップ手順の公式ドキュメントが薄い。Workers からの直接バックアップ不可。無料プランの自動バックアップ有無が不明確
- 根拠: 手順の確実性に不確実性が残る
- CSD依存: S3（手動スナップショットで十分）、S5（Turso 無料プラン継続）

#### ALT-4 × OBJ-4（○）
- 評価: ○
- 利点: Cloudflare Integrations Marketplace 公式パートナー。環境変数自動設定導線あり。Drizzle ORM が libSQL サポート。Turso CLI でDB操作可能
- 欠点: wrangler + turso CLI の2ツール管理。認証情報2つの管理。初期セットアップが複数ステップ
- 根拠: 公式統合パートナーで情報源は確保されているが、D1 の一元管理には劣る
- CSD依存: S5（Turso 無料プラン継続）

#### ALT-4 × OBJ-5（○）
- 評価: ○
- 利点: 無料枠 5 GB・5億ロウ読み取り/月で個人用途では長期超過しにくい。有料移行時 $4.99/月と低コスト
- 欠点: 独立スタートアップで事業継続リスクが相対的に高い。過去に無料枠条件の見直しあり
- 根拠: 広い無料枠と低い有料移行コストは魅力だが、スタートアップリスクは残存
- CSD依存: S5（Turso 無料プラン継続）
