# OBJ-2: 認証機能の統合性 — 評価結果

## 提案した代替案

- **Cloudflare D1 + Workers KV（セッションストア）**: 認証トークン（JWT またはセッションID）のストレージとして Cloudflare KV を組み合わせる構成。D1 をデータストアとして使いつつ、認証セッションの保持を KV で賄うことで、Workers エコシステム内に完結させる。ただし KV はデータベース本体ではなくセッションストアとしての用途に限定。
- **Supabase（Auth 統合）**: Supabase は PostgreSQL ベースのデータストアに加え、Supabase Auth という成熟した認証機能を組み込みで提供する。GoTrue（Auth サーバー）をベースとした JWT ベースの認証・セッション管理・ソーシャルログインが即座に使用可能。Workers からは Supabase JS クライアント（HTTP ベース）で接続可能。

## 評価

### ALT-1: Cloudflare D1

- 評価: △
- 利点:
  - Workers と同一エコシステムのため、Cloudflare Access や Workers 内の JWT 検証との統合は技術的に可能
  - D1 自体は SQLite の汎用データベースであり、ユーザーテーブルを自前で作成してセッション情報を格納できる
  - Cloudflare KV（既存の無料枠内で利用可能）をセッションストアとして組み合わせることで、ステートレスな JWT + KV によるセッション管理が実現できる
- 欠点:
  - D1 自体には認証機能は存在しない。認証機能は Workers 側で完全に自前実装が必要（JWT 発行・検証・リフレッシュ、パスワードハッシュ処理など）
  - パスワード認証を実装する場合、bcrypt/argon2 等のハッシュライブラリが V8 Isolate 環境で動作するかを確認する必要がある（S6 の仮定に影響）
  - セッション無効化（ログアウト・強制セッション失効）を実装するには KV 等の外部ストアとの連携が必須で、実装工数が増大する
  - シングルユーザーかつ個人開発（C3）という制約はあるが、それでも認証実装の自前開発は相応のコストを伴う
- 根拠: D1 はデータストアに特化しており、認証機能は提供しない。シングルユーザーの個人開発（C3）という制約があるため、認証の複雑さを抑えることが実用的に重要。完全自前実装は選択肢として存在するが、OBJ-2 の「低コストで実現できること」「追加工数・複雑さへの影響が小さいこと」の観点では他の選択肢に劣る。
- CSD依存: C3（シングルユーザー構成）— 認証の複雑さの要件は利用者が1名であることに依存。複数ユーザーに拡張する場合は実装工数が大幅に増加する。

### ALT-2: Supabase

- 評価: ◎
- 利点:
  - Supabase Auth（GoTrue ベース）が標準で統合されており、メール/パスワード認証・ソーシャルログイン（GitHub、Google 等）・マジックリンクが即座に利用可能
  - JWT ベースのセッション管理が自動的に処理され、Workers 側ではトークン検証（supabase-js または JWT ライブラリ）のみ実装すれば良い
  - Supabase JS クライアント（`@supabase/supabase-js`）が HTTP ベースで動作するため、C1（Workers からの HTTP 接続可能性）を満たす
  - Row Level Security（RLS）と Auth の統合により、DB レベルでのアクセス制御も可能（シングルユーザーでは不要だが、将来の拡張に有利）
  - 無料枠（5万 MAU）はシングルユーザー（C3）には十分すぎる余裕がある
  - Supabase ダッシュボードから認証ユーザーの確認・管理が視覚的に行える
- 欠点:
  - Supabase は外部サービスであり、Workers から HTTP で接続するため、認証処理を含むリクエストに追加のネットワークレイテンシが発生する可能性がある（OBJ-1 のトレードオフ）
  - `@supabase/supabase-js` ライブラリの Workers 環境での動作確認が必要（一般的には動作するが、バンドルサイズ・互換性に注意）
  - Supabase に認証データ（ユーザー情報）が保存されるため、将来の移行時には認証データとアプリデータの両方の移行が必要
- 根拠: OBJ-2 の評価視点「DBサービスが提供する組み込み認証機能の有無・成熟度」において、Supabase Auth は最も直接的かつ成熟した回答を提供する。シングルユーザーの個人開発（C3）において認証実装工数をゼロに近づけられる点は、OBJ-2 の定義「低コストで実現できること」に合致する。
- CSD依存: S5（Turso の無料プランが継続して提供される）— 直接関係なし。Supabase 固有の前提として「Supabase 無料枠（5万MAU）が継続提供される」という仮定に依存するが、現時点での情報では S4/S5 と類似の性質を持つ（Supposition）。

### ALT-3: TiDB Serverless

- 評価: △
- 利点:
  - TiDB Serverless は MySQL 互換であり、一般的な MySQL ドライバーやクライアントライブラリが使用可能
  - TiDB Cloud のダッシュボードから SQL 操作・ユーザー管理が行える
  - MySQL 互換のため、ユーザーテーブルを自前で作成してセッション情報を格納することは技術的に可能
- 欠点:
  - TiDB Serverless 自体には認証機能（アプリケーションレベルのユーザー認証）は存在しない。D1 と同様に完全自前実装が必要
  - Workers からの接続は HTTP API または TCP（`cloudflare:sockets`）が必要だが、MySQL 互換接続の Workers 対応は D1 や Supabase と比較してエコシステムが整備されていない（リサーチ結果より）
  - TiDB Serverless 向けの Cloudflare Workers 統合事例が少なく、認証連携の実装コストが予測しにくい
  - 自前認証実装の工数に加え、TiDB Serverless 固有の接続方法の調査コストも発生する
- 根拠: TiDB Serverless は分散データベースとして設計されており、アプリケーション認証機能の提供は本来のスコープ外。OBJ-2 の「認証実装の追加工数・複雑さへの影響」の観点では、認証自前実装の工数に加えて、Workers との統合に必要な調査・実装コストも生じるため、評価は△となる。
- CSD依存: S6（データアクセスパターンが CRUD + 検索のみで十分）— 認証セッション管理のアクセスパターンが TiDB Serverless の設計コンセプト（大規模分散・高スループット）と乖離しており、オーバースペックの可能性がある。

### ALT-4: Turso (libSQL)

- 評価: △
- 利点:
  - Turso は SQLite 互換であり、ユーザーテーブルやセッションテーブルを通常の SQL で作成・管理できる。認証関連データを主データと同一 DB に集約することが可能
  - `@libsql/client/web` HTTP ドライバーが Workers 環境でネイティブに動作するため、認証フロー内での DB アクセス（セッション検証・トークン保存）の接続確立コストが低い（D1 Binding と同等に近い）
  - グローバルエッジレプリカにより認証処理の物理的レイテンシが低く抑えられるため、認証チェックを行う毎リクエストの処理でも応答性能要件（C4）を満たしやすい
  - Cloudflare Workers 公式統合パートナーであり、ドキュメントや実装事例が整備されている
- 欠点:
  - Turso 自体にはアプリケーション層の認証機能（ユーザー認証・セッション管理・JWT 発行）は存在しない。D1・TiDB Serverless と同様に完全自前実装が必要（JWT 発行・検証・リフレッシュ、パスワードハッシュ処理など）
  - セッション無効化（ログアウト・強制失効）を実現するには、セッションテーブルまたは外部ストア（Cloudflare KV 等）との連携が必要で、設計の工数が増加する
  - 外部サービスへの依存となるため、Turso の認証情報（`LIBSQL_DB_AUTH_TOKEN`）の管理が必要。認証トークンが漏洩した場合のリスク管理も考慮が必要
  - Supabase Auth と比較して、認証機能のための追加実装量が多い
- 根拠: Turso はデータストアに特化したサービスであり、認証機能はスコープ外。OBJ-2 の「DBサービスが提供する組み込み認証機能の有無・成熟度」の観点では ×（機能なし）だが、「認証を自前実装する場合のセッション/トークン管理ストレージとしての適性」の観点では SQLite 互換・HTTP ドライバーの安定性から一定の適性がある。ALT-1（D1）と同様に完全自前実装が必要である点は共通しており、評価は△とする。D1 との差分としては、Workers との統合がエコシステム的に整備されている点と、外部サービス依存が生じる点が挙げられる。
- CSD依存: C3（シングルユーザー構成）— 認証の複雑さの要件は利用者が1名であることに依存。シングルユーザーであれば自前実装でも管理コストは許容範囲内だが、複数ユーザーへの拡張が生じた場合には実装工数が大幅に増加する。S5（Turso の無料プランが継続して提供される）— 認証データ（セッションテーブル等）を Turso 上に保持する場合、無料プランの制限変更がデータ保持の継続性に影響する。
