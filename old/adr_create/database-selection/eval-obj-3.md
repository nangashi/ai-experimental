# OBJ-3: バックアップ・リストアの実行可能性 — 評価結果

## 提案した代替案

この目的（バックアップ・リストアの実行可能性）を最適化する観点から、以下の代替案を提案する。ただし、今回のユーザー提供代替案（ALT-1〜ALT-3）の中に十分な候補が含まれているため、追加提案は最小限にとどめる。

- **Cloudflare D1（ALT-1）**: SQLite ベースであり、`wrangler d1 export` による SQL ダンプのエクスポートが可能。ただし FTS5 仮想テーブルを含む場合はエクスポート非対応（D1 の制約）。FTS5 を使わない場合、または FTS5 仮想テーブルを別途再構築する手順を整備する場合は、バックアップ自体はスクリプト化可能。
- **Supabase（ALT-2）**: PostgreSQL ベースの BaaS。`pg_dump` 互換のエクスポート機能、Supabase ダッシュボードからのバックアップ取得、Storage API 経由のデータエクスポートなど、複数の手段を持つ。無料プランではポイントインタイムリカバリ（PITR）は非対応だが、手動エクスポートは可能。
- **TiDB Serverless（ALT-3）**: MySQL 互換のため `mysqldump` 相当の操作が想定されるが、Serverless 構成では接続方式が HTTP/TCP に依存するため、標準ツールでのダンプ取得の容易さが不明確。公式ドキュメントのバックアップ機能に依存する部分が大きい。

## 評価

### ALT-1: Cloudflare D1

- 評価: △
- 利点:
  - `wrangler d1 export` コマンドで SQL ダンプの取得が可能。CLI ベースでスクリプト化しやすい
  - SQLite 形式のため、ローカル環境での確認・インポートが容易（`sqlite3` コマンドで操作可能）
  - `wrangler d1 execute --file` でインポートも可能。手順が比較的明確
- 欠点:
  - **FTS5 仮想テーブルを含むデータベースは `wrangler d1 export` が非サポート**（リサーチ観点4・D1 の確認済み制約）。スコープに「キーワード検索」が含まれ、S2 の仮定に従えば FTS5 または LIKE 検索が想定されるが、FTS5 を採用した場合はエクスポートが制限される
  - FTS5 を使用する場合、リストア後に仮想テーブルを別途再構築するスクリプトを整備する必要があり、手順の複雑さが増す
  - リストア後に FTS5 インデックスの再構築を忘れるリスクがある（D1 として明示されている不確実性）
  - Cloudflare のフリープランではポイントインタイムリカバリは提供されない
- 根拠: FTS5 を使わない場合（LIKE 検索のみ）は○相当まで評価が上がるが、スコープの「キーワード検索」要件においてFTS5使用の可能性が高く（S2 に依存）、FTS5 採用時のエクスポート非サポートはこの目的における実質的な障壁となる。リサーチで確認された制約（観点4）をそのまま反映する。
- CSD依存:
  - S2（キーワード検索が LIKE 検索で充足できるかの仮定）: LIKE 検索のみの場合、FTS5 エクスポート非サポートの問題は回避できるため、評価は○相当に改善する
  - D1（FTS5 仮想テーブルを含む場合のエクスポート非サポートの実運用上の許容可能性）: 許容できないと判断された場合、ALT-1 のこの目的に対する評価は× に格下げとなる

### ALT-2: Supabase

- 評価: ○
- 利点:
  - PostgreSQL ベースの BaaS として、ダッシュボードから手動バックアップのエクスポートが可能
  - PostgreSQL の標準ツール（`pg_dump`）に準じたエクスポート形式であり、インポートの確実性が高い
  - キーワード検索に PostgreSQL の全文検索（`tsvector`/`tsquery`）や `pg_trgm` を使用する場合も、これらは通常の PostgreSQL テーブル・インデックスとして扱われるためエクスポートに追加の障壁がない
  - REST API（PostgREST）経由でデータを JSON 形式で取得するスクリプトを書くことも可能（ポータブルなバックアップ手段）
- 欠点:
  - 無料プランではポイントインタイムリカバリ（PITR）は提供されず、自動バックアップのスケジュールも有料プランの機能
  - バックアップの取得・リストア手順が Supabase 独自の手順に依存するため、将来のサービス変更リスクがある
  - `pg_dump` を直接実行するためには PostgreSQL 接続情報（ホスト・ポート）が必要であり、Workers 環境からの直接バックアップではなく、別途ローカル環境でのオペレーションが必要
  - Supabase の無料プランにおけるダッシュボードのバックアップ機能の制限状況は今後変化する可能性がある（S5 と類似の性質）
- 根拠: PostgreSQL ベースのため FTS5 仮想テーブルのような特殊なエクスポート制約がなく、手順の確実性は ALT-1 より高い。ただし無料プランでの自動バックアップ非対応は手動オペレーションを前提とする。個人開発・手動スナップショット運用（S3）の仮定が成立するなら十分に許容できる水準。
- CSD依存:
  - S3（バックアップは手動スナップショットで十分という仮定）: 手動バックアップが前提のため、S3 が崩れ（高頻度な自動バックアップが必要になった場合）、評価は△に格下げとなる
  - S5（Supabase の無料プランが継続して提供される仮定）: バックアップ機能の無料提供範囲が変更された場合、手順・コストに影響する

### ALT-3: TiDB Serverless

- 評価: △
- 利点:
  - MySQL 互換のため、`mysqldump` や MySQL Workbench 等の標準ツールによるバックアップが原理的に可能
  - TiDB Cloud コンソールからのバックアップ取得機能が提供されている（有料プランでは自動バックアップあり）
  - MySQL 互換のダンプファイルは汎用性が高く、将来の移行にも使いやすい
- 欠点:
  - **Cloudflare Workers からの接続が HTTP ベース（TiDB Cloud の Data Service API）に依存するため、直接 `mysqldump` 相当の操作を Workers から実行することはできない**。バックアップはローカル環境または TiDB Cloud コンソールから行う必要がある
  - 無料プランでのバックアップ機能の詳細（自動バックアップの有無、リテンション期間等）がリサーチ結果に含まれておらず、不確実性が高い
  - Serverless 構成では接続が HTTP API 経由となるため、標準 MySQL ツールによる直接ダンプが可能かどうかは接続エンドポイントの設定に依存し、手順の確実性が未確認
  - リストア手順（どのフォーマットで、どのツールを使ってリストアするか）がドキュメント化された手順として把握しにくく、実運用での確実性に不安がある
- 根拠: MySQL 互換という観点では手順の汎用性が高いが、Serverless 構成特有の接続制約により、標準ツールによるバックアップの実行可能性が不明確。リサーチでも TiDB Serverless のバックアップ手順を直接確認していないため、評価は不確実性込みで△とする。Supposition/Doubt への依存度が高い。
- CSD依存:
  - 不確実性: TiDB Serverless の無料プランにおけるバックアップ機能の有無・手順が未確認。具体的な手順確認なしに確実性を評価することができない（Doubt 相当の不確実性が存在する）
  - C1（Workers からの接続方式制約）: Workers からの直接バックアップは実質不可能。バックアップ操作はWorkers外（ローカル/コンソール）で行うことを前提とする必要がある（この目的の評価には直接影響しないが、運用上の制約として明記）

### ALT-4: Turso (libSQL)

- 評価: △
- 利点:
  - SQLite 互換のため、`sqldump` 相当のエクスポートや `sqlite3` コマンドによるダンプが原理的に使用可能
  - Turso CLI（`turso db shell` + `.dump` コマンド）でデータベースの SQL ダンプを取得できる手段が存在する
  - SQLite 互換形式のダンプファイルはポータビリティが高く、ローカルでの確認・検証が容易（`sqlite3` で直接インポート可能）
  - FTS5 仮想テーブルを含む場合でも、D1 のようなエクスポート制限は libSQL の仕様としては明示されておらず、Turso 独自のエクスポート制約は現時点では確認されていない
- 欠点:
  - **Turso CLI 経由のダンプ取得手順が、D1 の `wrangler d1 export` や PostgreSQL の `pg_dump` と比較して文書化が薄く、実運用での確実性が未確認**。公式ドキュメントでのバックアップ手順の明示度がリサーチ結果に含まれていない
  - Cloudflare Workers から Turso への接続は HTTP ベース（`@libsql/client/web` ドライバー）であるため、Workers プロセス自体からバックアップを実行することはできない。バックアップはローカル環境または Turso CLI から別途実施する必要がある
  - 外部サービス（Turso）のデータを完全に手元にエクスポートするための公式バックアップ機能（スケジュール自動バックアップ・PITR 等）が無料プランに存在するかが不明確
  - FTS5 仮想テーブルを使用した場合のリストア手順の確実性（バックアップされた FTS5 仮想テーブルが正常にリストアされるか）について、Turso/libSQL 固有の検証が取れていない
- 根拠: SQLite 互換という観点では、D1 のエクスポート制限（FTS5 非サポート）よりも制約が少ない可能性があるが、Turso 固有のバックアップ手順の文書化・確実性がリサーチ結果から確認できていない。ALT-2（Supabase）が PostgreSQL の確立されたバックアップ手順を持つのに対し、Turso はリサーチ結果の観点5でバックアップ手順を直接確認しておらず、不確実性が残る。Turso CLI による手動ダンプの実行可能性は認められるものの、その手順の信頼性・再現性については追加検証が必要な状態のため△とする。
- CSD依存:
  - S3（バックアップは手動スナップショットで十分という仮定）: 手動バックアップを前提とする場合、Turso CLI による手動ダンプ取得で対応可能であり、評価は維持または○に改善しうる。S3 が崩れ（自動バックアップが必要になった場合）、評価は×に格下げとなる
  - S5（Turso の無料プランが継続して提供される仮定）: Turso のバックアップ機能が有料プランに移行した場合、C2（月額コスト 0 円）の制約に抵触し、バックアップ手段が制限される
  - 不確実性: Turso CLI のバックアップ手順の具体的な動作確認が取れていない。FTS5 仮想テーブルを含むデータベースのダンプ・リストアの確実性は追加検証なしに断言できない
