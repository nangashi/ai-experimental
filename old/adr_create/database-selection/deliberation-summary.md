# 審議サマリー

## 決定ステートメント

### 議題
Cloudflare Workers + Hono + TypeScript バックエンドのリーディングリスト管理アプリにおいて、データベースとしてどの技術を採用するか

### スコープ
- 対象: データベースサービスの選定。記事URL管理（数百〜数千レコード）、タイトル・URLのキーワード検索、バックアップ・リストア対応を評価する
- 対象外: バックエンドフレームワーク（Cloudflare Workers + Hono + TypeScript で確定済み — ADR-0001）、ホスティング基盤（Cloudflare で確定済み — ADR-0001）、フロントエンドフレームワーク（React で確定済み — ADR-0002）、テーブル設計・スキーマ設計、マイグレーション方式

### 可逆性
medium — データベース変更にはスキーマ移行・データ移行・ORM/クエリ層の書き換えが必要だが、小規模データ（数千レコード）のため移行コストは限定的。初期段階での変更は比較的容易。

## 関連する既存決定

### ADR-0001: PWA+REST API構成のホスティング基盤選定
- 決定: Cloudflare Pages（フロントエンド）+ Cloudflare Workers（バックエンド）を採用する。バックエンドは Cloudflare Workers 上で Hono + TypeScript による REST API として動作する。
- 今回の議題との関連: バックエンドが Cloudflare Workers（V8 Isolate ランタイム）上で動作するため、採用するデータベースは Workers ランタイムと互換性を持つ必要がある。
- 今回の決定に引き継がれる制約:
  - データベースは Cloudflare Workers（V8 Isolate ランタイム）から接続可能な方式でなければならない（HTTP/HTTPS ベースの API または Cloudflare Bindings に限定）
  - 月額コストは 0 円（無料ティア内）の制約が引き続き適用される
  - シングルユーザー構成であり、大規模スケーリングは不要
  - 画面表示 3 秒以内の要件が引き続き適用される
  - Cloudflare Bindings（D1 等）の採用は ADR-0001 で受け入れ済みのベンダーロックインを拡大するものとして位置づける

### ADR-0002: フロントエンドフレームワーク選定
- 決定: React（Vite + React 19）を採用する。
- 今回の議題との関連: フロントエンドはバックエンド API を経由してデータにアクセスするため、データベース選定に対する制約なし。

## 前提条件（CSD）

### 確実（Certainty）

| # | 内容 | 根拠 |
|---|------|------|
| C1 | データベースは Cloudflare Workers（V8 Isolate ランタイム）から接続可能な方式でなければならない | ADR-0001 で確定。HTTP/HTTPS ベースの API または Cloudflare Bindings が必要 |
| C2 | 月額コストは 0 円（無料ティア内）でなければならない | ADR-0001 の C3 で確定 |
| C3 | シングルユーザー構成であり、実運用ユーザーは開発者自身の1名のみ | ADR-0001 の C4 で確定。ユーザーのロケーションは開発者の居住地に固定 |
| C4 | データベースアクセスのレイテンシを含む画面表示 3 秒以内の要件がある | ADR-0001 の C5 で確定 |

| # | 内容 | 検証方法 |
|---|------|---------|
| S1 | 管理するレコード数は将来的にも数千レコード（最大でも数万レコード）程度に収まる | 1年後にデータ件数・サイズを実測し、D1 Free プラン制限（500 MB）の 10% を超えていないか確認 |
| S2 | キーワード検索の要件は、タイトル・URL に対する部分一致検索で充足できる | 実際の検索利用パターンを観察し、性能・機能的に十分かを確認 |
| S3 | バックアップは日次〜週次程度の手動スナップショットで十分であり、頻繁な実行は不要 | バックアップ頻度・復元要件を利用開始後に確認 |
| S4 | Cloudflare D1 は現時点で GA 済みでプロダクション利用可能であり、Free プランが継続提供される | Cloudflare D1 リリースノート・プライシングページを定期確認 |
| S5 | Turso の無料プランが継続して提供される | Turso の料金ページを定期確認 |
| S6 | データアクセスパターンは CRUD + キーワード検索のみで十分である | 機能拡張検討時にデータベースの集計クエリ・リレーション設計への影響を評価 |

| # | 内容 | 影響範囲 | リスク |
|---|------|---------|--------|
| D1 | D1 の FTS5 仮想テーブルを含むデータベースが wrangler d1 export の対象外であることの、実運用上の許容可能性 | D1 で FTS5 を使用した場合のバックアップ・リストア | 手順の複雑化、リストア後の FTS5 インデックス再構築忘れリスク |

## 判断基準

| ID | 名前 | 定義 |
|----|------|------|
| OBJ-1 | データアクセスの応答性能 | データベースへのアクセスが、画面表示3秒以内の要件を安定して満たせること。接続方式（Binding vs HTTP API）、リージョン距離、コネクションプールなし環境での応答時間を評価する |
| OBJ-2 | 認証機能の統合性 | ユーザー認証を低コストで実現できること。DBサービス自体が認証機能を提供しているか、またはWorkers側での認証実装とDBの統合がスムーズか |
| OBJ-3 | バックアップ・リストアの実行可能性 | データの損失リスクに対して、現実的な手順でバックアップを取得・リストアできること。エクスポート機能、手順の複雑さ、リストアの確実性を評価する |
| OBJ-4 | 運用負荷の最小化 | シングルユーザーの個人開発として、データベースのセットアップ・維持管理に要する労力が最小であること。初期セットアップの容易さ、Cloudflare Workers との統合、管理ツールの充実度を評価する |
| OBJ-5 | 長期的なコスト安定性 | 無料ティア内での継続利用が現実的に維持できること。無料プランの制限内容、持続可能性、超過時のコスト予測可能性を評価する |

## 検討した選択肢

### ALT-1: Cloudflare D1
- 概要: Cloudflare のエッジ SQLite データベース。Workers との同一プラットフォーム統合。SQL クエリ・LIKE 検索・FTS 対応。
- 無料枠: 5GB、読み取り500万回/日、書き込み10万回/日

### ALT-2: Supabase
- 概要: PostgreSQL ベースの BaaS。RESTful API・リアルタイム機能・認証機能を統合提供。外部サービスとして Workers から HTTP 接続。
- 無料枠: 500MB、5万MAU

### ALT-3: TiDB Serverless
- 概要: MySQL 互換の分散 NewSQL データベース。サーバーレス構成で自動スケーリング。外部サービスとして Workers から HTTP/TCP 接続。
- 無料枠: 5GiB ストレージ、5,000万 Request Units/月

### ALT-4: Turso (libSQL)
- 概要: SQLite 互換の分散エッジデータベース。Cloudflare 公式パートナー。`@libsql/client/web` HTTP ドライバーで Workers から接続可能。エッジレプリカによる低レイテンシ。
- 無料枠: 5GB、5億ロウ読み取り/月

| OBJ-1: データアクセスの応答性能 | ◎ | △ | △ | ○ |
| OBJ-2: 認証機能の統合性 | △ | ◎ | △ | △ |
| OBJ-3: バックアップ・リストアの実行可能性 | △ | ○ | △ | △ |
| OBJ-4: 運用負荷の最小化 | ◎ | ○ | △ | ○ |
| OBJ-5: 長期的なコスト安定性 | ○ | △ | ○ | ○ |

### 評価の凡例
- ◎: 非常に優れている
- ○: 概ね優れている
- △: 懸念点あり・条件付き
- ×: 大きな問題あり

### 主要なトレードオフの解説

**ALT-1 (Cloudflare D1)** は応答性能と運用負荷で突出した強みを持つ一方、認証機能とバックアップに制約がある。Binding 接続による接続オーバーヘッドゼロは C5 の制約下で構造的に最も有利。ただし FTS5 使用時のエクスポート非サポート（D1）が実運用上の懸念として残る。

**ALT-2 (Supabase)** は認証機能の統合性とバックアップ実行可能性で優位に立つが、外部 HTTP 接続による応答性能の構造的不利と、Free プランの自動停止仕様・料金変更リスクが課題。認証実装を自前でゼロから行いたくない場合に有力な選択肢。

**ALT-3 (TiDB Serverless)** はすべての観点で△評価であり、今回の規模・要件への適合性が低い。エンタープライズ向けの機能が個人用途では過剰であり、Workers との統合エコシステムの薄さがトラブルシューティングコストを高める。

**ALT-4 (Turso)** はエッジレプリカによる応答性能と公式統合パートナーとしての信頼性が強みだが、C3（シングルユーザー・固定ロケーション）の条件下ではエッジ分散の恩恵が限定的。スタートアップとしての事業継続リスクも残る。

## 判断ガイド

以下は条件付きの帰結の提示であり、推奨ではない。
最終的な優先順位の決定は判断者に委ねられる。

- **OBJ-1（応答性能）を最優先するなら** → ALT-1（Cloudflare D1）が適している。Binding 接続により接続オーバーヘッドが構造的にゼロであり、C5（コネクションプールなし）の制約下で最も有利。ただし OBJ-3（バックアップ）では FTS5 使用時に制約が生じる（D1）。

- **OBJ-2（認証機能の統合性）を最優先するなら** → ALT-2（Supabase）が適している。Supabase Auth により認証実装を自前で行う必要がなく、実装工数を大幅に削減できる。ただし OBJ-1（応答性能）では外部 HTTP 接続の構造的不利があり、OBJ-5（コスト安定性）では Free プランの自動停止仕様と料金変更リスクに注意が必要。

- **OBJ-3（バックアップの確実性）を最優先するなら** → ALT-2（Supabase）が適している。PostgreSQL ベースで FTS 特有のエクスポート制約がなく、`pg_dump` 互換の手順が利用できる。ただし OBJ-5 のコスト安定性リスクに注意が必要。

- **OBJ-4（運用負荷の最小化）を最優先するなら** → ALT-1（Cloudflare D1）が適している。既存の Cloudflare + wrangler エコシステムに完全統合され、管理対象の追加がゼロ。ただし認証実装は自前となる（OBJ-2 が △）。

- **OBJ-5（コスト安定性）を最優先するなら** → ALT-1、ALT-3、ALT-4 がいずれも ○ 評価。ALT-1 は大企業インフラとしての継続性が高いが D2 の不確実性がある。ALT-4 は有料移行コストが最低（$4.99/月）だがスタートアップリスクが残る。

- **応答性能と運用負荷のバランスを取るなら** → ALT-1（Cloudflare D1）が有力。認証を自前実装する工数と、バックアップ手順の整備コストを許容できるかが採否の鍵となる。

- **認証実装コストを避けつつバランスを取るなら** → ALT-2（Supabase）が有力。応答性能の不利を 3 秒要件との観点で許容できるか（CRUD + 検索の単純なクエリでは実用上問題ない可能性が高い）、および Free プランの自動停止仕様への対処（定期アクセスによる稼働維持等）を検討する必要がある。

## 不確実性の影響

以下の不確実性が判明した場合、選択肢の評価が変化する可能性がある:

- **D1（D1 の FTS5 バックアップ制約の許容可能性）が判明した場合**:
  - 「LIKE 検索のみを使い FTS5 を使用しない」と決定した場合 → ALT-1 の OBJ-3 評価が △ から ○ 相当に向上し、ALT-1 の優位性がさらに高まる
  - 「FTS5 の制約は許容できない」と判断した場合 → ALT-1 の OBJ-3 評価が実質的に問題化し、バックアップ手順の整備コストが追加の運用負荷として顕在化する。ALT-2 の相対的な優位が高まる

- **D2（Cloudflare D1 の Free プラン変更リスク）が現実化した場合**:
  - D1 の無料枠が縮小・有料化された場合 → ALT-1 の OBJ-5 評価が ○ から △ に低下し、コスト面での優位が失われる。その時点での他選択肢への移行コストは「可逆性: medium」の記述の通り限定的であり、ALT-4（Turso）への移行が最も容易（SQLite 互換のため）

- **D3（Workers-D1 リージョン間レイテンシ）が問題化した場合**:
  - 開発者の居住地と D1 リージョンの距離が 3 秒要件に実測上影響することが判明した場合 → ALT-1 の OBJ-1 評価が ◎ から ○ または △ に低下する可能性がある。ただし C3 によりユーザーは1名かつ固定ロケーションのため、D1 リージョンを適切に選択することでほぼ確実に軽減できる。深刻な問題に発展するリスクは低い
