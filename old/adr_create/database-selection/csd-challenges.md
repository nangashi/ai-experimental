# レッドチーム検証結果

## 再分類提案

- C5: Certainty → Supposition — 理由: C5 は「D1 が現在 GA 済みでプロダクション利用可能」という事実を記述しているが、これは今この瞬間の製品状態であり、将来にわたって保証される制約ではない。Cloudflare は製品のプライシング・機能・制限を変更した実績があり（D2 でも無料枠変更リスクを Doubt に挙げている）、C5 の GA 状態や無料枠の可用性も同様のリスクにさらされている。「現時点で D1 は利用可能である」という陳述は Supposition（現時点では正しいと思われるが変わりうる前提）に再分類すべき。検証方法: Cloudflare の D1 リリースノート・プライシングページを定期確認する。なお D2 と内容が重複するため、D2 と統合して Doubt に格下げることも検討に値する。

- S4: Supposition の対象を限定すべき — 理由: S4 は「Drizzle ORM は継続的にメンテナンスされる」という仮定を置いているが、今回の意思決定の議題は「データベース選択」であり、ORM の選定はスコープ外の別の意思決定事項である。Drizzle ORM を使うこと自体がまだ確定していない前提であれば、S4 は「データベース選択 ADR に対して時期尚早な仮定」として記録すべきである。もしすでに Drizzle ORM の採用が別の場所で決定済みであれば、その根拠（ADR番号など）を明示し、Supposition ではなく Certainty に格上げするか、または prior-decisions の Certainty として取り込むべきである。いずれの場合もスコープ整合性の観点から再検討が必要。

## 暗黙の仮定の検出

- 「データアクセスパターンは CRUD + キーワード検索のみ」という仮定 — 推奨分類: Supposition — 理由: csd-draft.md 全体を通じて、必要なデータ操作がレコードの作成・読み取り・更新・削除およびキーワード検索に限られることが暗黙の前提になっている。しかしリーディングリストに将来的に「タグによるフィルタリング」「読了率・ステータス管理」「外部サービス連携（RSS等）との同期」などの機能が追加される可能性は排除できない。これらの機能追加は、データベースの集計クエリ・リレーション設計・スキーマ変更の容易さに影響し、データベース選択の評価基準を変えうる。S2 の「将来的な検索機能拡張」の記述はあるが、検索以外のアクセスパターン拡張についての仮定が明示されていない。

- 「データベース接続は毎リクエスト確立・切断（コネクションプールなし）を前提とする」という仮定 — 推奨分類: Certainty — 理由: Cloudflare Workers の V8 Isolate モデルはリクエストをまたいだ持続的接続を持たない。この特性はデータベース選択において重要な制約であり、接続オーバーヘッドの低いデータベース（D1 Binding や HTTP API 型）が有利になる根拠となる。C1 の「HTTP/HTTPS ベースの API または Bindings に限定」は半分この制約を表現しているが、「コネクションプールが使えない」という点が明示的な Certainty として記述されていない。明示的に C として追加することで、以降の評価フェーズで各選択肢の接続モデルを正確に評価できる。

- 「個人用途のため実運用ユーザーは事実上1名（開発者自身）」という仮定 — 推奨分類: Certainty — 理由: D3 の中で「個人用途のため、実運用ユーザーは限定的」と括弧書きで言及されているが、この事実は C3（シングルユーザー構成）の補足として Certainty に明示すべき重要な前提である。ユーザーのロケーションが D1 データベースリージョンに近い（または設定可能）ことが D3 のリスク低減理由として機能するため、意思決定の透明性のために Certainty として明記することが望ましい。

## 検証済み、分類維持と判断した項目

- C1: Workers の TCP 制限に関して、Cloudflare は 2023 年以降 `cloudflare:sockets` API による一部 TCP 接続をサポートしているが、これは生の TCP であり、PostgreSQL 等の従来型データベースプロトコルへの直接接続は依然として一般的に推奨されない。HTTP API または Bindings への限定という記述は実用上正確であり、Certainty の分類は維持する。ただし「TCP が一切使えない」という絶対表現は避け、「HTTP/HTTPS API または Cloudflare Bindings を使う方式が現実的」という表現に注記を加えることが望ましい。

- C2: 月額コスト 0 円（無料ティア内）— ADR-0001 で確定した制約として Certainty 分類は適切。検証済み、分類維持。

- C3: シングルユーザー構成・大規模スケーリング不要 — ADR-0001 で確定した制約として Certainty 分類は適切。検証済み、分類維持。

- C4: 画面表示 3 秒以内の要件 — ADR-0001 で確定した要件として Certainty 分類は適切。検証済み、分類維持。

- S1: データ件数の上限見積もり — 検証方法が具体的（1年後に実測、D1 制限の 10% を閾値）であり、Supposition として適切に記述されている。分類維持。

- S2: キーワード検索の充足性 — 検証方法が記述されており Supposition として適切。ただし上記「暗黙の仮定」で指摘したアクセスパターン拡張とあわせて補完することを推奨。分類維持。

- S3: バックアップ頻度の想定 — 個人用途という前提と整合しており、Supposition として適切。検証方法（自身への確認）も現実的。分類維持。

- S5: Turso 無料プランの継続 — プライシング変更リスクは実在し、Supposition として適切に記述されている。分類維持。

- D1: D1 の FTS5 仮想テーブルのバックアップ制約 — 影響範囲・リスクが具体的に記述されており Doubt として適切。分類維持。

- D2: D1 Free プラン制限変更リスク — C5 との統合を提案しているが、現状の Doubt 分類自体は適切。分類維持（C5 の再分類を受けて統合検討）。

- D3: Workers-D1 リージョン間レイテンシ — 個人用途による限定的な影響という括弧書きが Doubt の深刻度を下げる文脈として機能しており、記述は適切。分類維持。ただし「個人ユーザーのロケーション = 開発者自身のロケーション」を Certainty として追加することで D3 のリスク評価がより正確になる（上記暗黙の仮定を参照）。
