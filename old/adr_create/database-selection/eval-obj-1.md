# OBJ-1: データアクセスの応答性能 — 評価結果

## 提案した代替案

- **Cloudflare D1**（Binding 接続）: Binding による接続は HTTP オーバーヘッドがなく、Workers との同一プラットフォーム統合のためラウンドトリップが最小。コネクションプールなし環境（毎リクエスト接続確立）においても Binding は事実上の接続オーバーヘッドゼロで動作し、OBJ-1 を最適化できる
- **Turso (libSQL)**（HTTP ドライバー + エッジレプリカ）: Workers に最近接のレプリカへ自動ルーティングされる分散アーキテクチャにより、外部 HTTP 接続でありながら物理的レイテンシを最小化できる。OBJ-1 の「物理的距離・レイテンシ特性」の観点で有利な代替案

## 評価

### ALT-1: Cloudflare D1

- 評価: ◎
- 利点:
  - Workers Binding による接続は HTTP API や TCP 接続と異なり、プロセス内通信に近い低オーバーヘッドで動作する。毎リクエスト接続確立の制約（C5）下では最も有利な接続方式
  - D1 は Workers と同一の Cloudflare エッジインフラ上で動作するため、ネットワークレイテンシが構造的に小さい
  - リサーチにより FTS5 を含む SQLite クエリが低レイテンシで動作することが確認されている（`research.md` 観点4）
  - 3秒以内（C4）の要件に対して、接続方式・インフラ距離の両面で最も有利な構成
- 欠点:
  - D1 データベースのリージョンは作成時に固定され、Workers の Edge ロケーションとの物理的距離が離れる場合がある
  - Workers が世界中のエッジで起動するのに対し、D1 は単一リージョンのため、ユーザーが D1 リージョンと距離のある地域からアクセスした場合はレイテンシが増加する
- 根拠:
  - C5（コネクションプールなし制約）の下では、毎リクエストの接続確立コストが低いことが応答性能に直結する。Binding は Workers ランタイムと同一プラットフォームであり、接続確立コストが実質ゼロに近い
  - C3（シングルユーザー、開発者のロケーション固定）と組み合わせると、D1 リージョンをユーザー近くに設定することでリージョン間レイテンシを制御可能（D3 参照）
- CSD依存:
  - D3（Workers Edge と D1 リージョン間レイテンシ）: D1 リージョンの選択次第でレイテンシが変動する。ただし C3 によりユーザーは開発者1名・ロケーション固定のため、初期設定でリージョンを適切に選べば影響は軽微と判断する

### ALT-2: Supabase

- 評価: △
- 利点:
  - PostgreSQL の全文検索機能（pg_tsvector / pg_search 等）は高度な検索性能を提供する
  - REST API（PostgREST）は HTTPS ベースで Workers から接続可能（C1 を満たす）
  - Supabase はリージョン選択が可能（東京等）であり、ユーザーのロケーション近くに配置できる
- 欠点:
  - Workers から Supabase への接続は外部 HTTP API 呼び出しとなる。毎リクエスト HTTP 接続確立（TCP ハンドシェイク + TLS ネゴシエーション）が発生し、C5 の制約下では接続オーバーヘッドが D1 Binding より大きい
  - Workers Edge ロケーション → Supabase リージョン間のネットワークホップが D1 より増加する
  - 3秒要件（C4）のうち、データベースアクセスに割り当てられるマージンが D1 より小さくなる
- 根拠:
  - D1 Binding との比較では、外部 HTTP API 呼び出しによる接続レイテンシ増加が評価を下げる主要因。数百ミリ秒オーダーの差が発生しうる
  - S6（CRUD + キーワード検索のみ）が成立する現状では、PostgreSQL の高度なクエリ機能は OBJ-1 の観点での優位性に直結しない
  - 3秒要件（C4）は厳しい閾値ではないため、Supabase のレイテンシが全体を超過させるリスクは低い。ただし接続方式として構造的に D1 より不利
- CSD依存:
  - S6（アクセスパターン仮定）: CRUD + キーワード検索の範囲では PostgreSQL の高度な機能は応答性能に貢献しない。将来的に複雑なクエリが必要になった場合、PostgreSQL のクエリ最適化により逆転の可能性あり
  - D3 相当のリスク: Supabase のリージョンを適切に選択すれば Workers との物理距離は制御可能だが、Workers の Edge ロケーションとの距離は完全には最適化できない

### ALT-3: TiDB Serverless

- 評価: △
- 利点:
  - MySQL 互換で幅広い SQL 機能をサポートし、複雑なクエリでも安定した性能を提供できる分散アーキテクチャを持つ
  - HTTP ドライバー（TiDB Cloud Data Service）経由で Workers から接続可能（C1 を満たす）
- 欠点:
  - 外部 HTTP API 接続であり、C5 の制約下での接続オーバーヘッドは D1 Binding より大きい（Supabase と同様の構造的不利）
  - TiDB Serverless はコールドスタート（アイドル後の初回クエリ遅延）が報告されており、低頻度アクセスのアプリケーションで応答時間が不安定になるリスクがある
  - 個人リーディングリスト管理（数千レコード、低トラフィック）は TiDB Serverless の分散アーキテクチャの恩恵を受けにくく、そのオーバーヘッドのほうが目立ちやすい
  - リージョンは限定的（アジア太平洋は東京・シンガポール等）で、Workers の Edge ロケーションとの距離が最適化しにくい場合がある
- 根拠:
  - C5（コネクションプールなし）+ 外部 HTTP 接続の組み合わせによる接続オーバーヘッドは Supabase と同等。加えてコールドスタート問題が低頻度個人利用で顕在化しやすい点が追加の懸念
  - S1（数千レコード程度）の規模では分散 NewSQL の性能優位が発揮されず、単純な SQLite ベース（D1）の方が小規模読み取りで低レイテンシになる可能性が高い
- CSD依存:
  - S1（レコード数仮定）: 将来的にレコード数が数十万〜数百万規模に拡大した場合、TiDB Serverless の分散アーキテクチャが性能面で逆転する可能性がある。ただし S1 が成立する限り現状のスコープでは不利

### ALT-4: Turso (libSQL)

- 評価: ○
- 利点:
  - エッジレプリカによる自動ルーティングにより、Workers が起動した Cloudflare エッジロケーションに最も近いレプリカへ接続される。D1 が単一リージョンに固定されるのに対し、Turso はグローバルなレプリカ配置によって物理的レイテンシを構造的に最小化できる
  - `@libsql/client/web` HTTP ドライバーは C1（Workers からの接続可能性）と C5（コネクションプールなし環境）の両制約を満たす。外部 HTTP 接続ではあるが、HTTP ドライバーは各リクエストで接続を確立するよう設計されており、C5 制約下での動作を前提としている
  - グローバルに分散したユーザー（将来的な拡張シナリオ）での応答性能では D1 より有利になる可能性がある
  - research.md の観点2で、Workers との公式統合が整備されていることが確認されており、HTTP 接続の実装が公式にサポートされている
- 欠点:
  - Workers Binding のような同一プラットフォーム接続ではなく、外部 HTTP API 呼び出しとなる。TCP ハンドシェイク + TLS ネゴシエーションのオーバーヘッドが毎リクエスト発生し、D1 Binding より接続オーバーヘッドは大きい
  - C3（ユーザーは開発者1名・ロケーション固定）の制約下では、エッジレプリカによる分散の恩恵が限定的である。ユーザーが単一拠点に固定されるなら、D1 のリージョンをユーザー近くに設定することで同等の物理距離制御が可能
  - Turso の平均レイテンシはエッジレプリカのロケーション次第であり、特定リージョンへの保証がない不確実性がある
- 根拠:
  - 接続オーバーヘッドの観点（C5 制約下）: ALT-1（D1 Binding）の方が構造的に有利。ただし外部 HTTP 接続である Supabase（ALT-2）・TiDB Serverless（ALT-3）と同等の接続方式であり、それらと比較してエッジレプリカによる物理距離の優位性がある
  - C3（シングルユーザー・ロケーション固定）の条件下では、エッジレプリカの分散効果よりも D1 Binding の接続オーバーヘッド最小化の方が OBJ-1 への寄与が大きいと判断する。ただし C4（3秒要件）は厳しい閾値ではないため、Turso の HTTP 接続オーバーヘッドが要件を超過させるリスクは低い
- CSD依存:
  - S5（Turso 無料プラン継続）: この評価は Turso サービスの継続提供を前提としている。サービス停止・プラン変更が発生した場合は前提が崩れる
  - C3（ユーザーロケーション固定）: 将来的に複数ロケーションからアクセスするシナリオが発生した場合、Turso のエッジレプリカ分散の優位性が増し、D1 との評価差が縮まる可能性がある
