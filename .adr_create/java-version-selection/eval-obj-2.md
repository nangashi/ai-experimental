# OBJ-2: エコシステムの成熟度と互換性 — 評価結果

## 提案した代替案

- **Java 17（LTS）**: エコシステムの安定性と成熟度が最も高い。Spring Boot 3.xの最小要件であり、主要フレームワーク・ライブラリ・ツールの互換性が広く検証されている。2021年リリースから5年以上経過し、ドキュメント・サンプルコード・トラブルシューティング情報が豊富。

- **Java 21（LTS）**: エコシステムの成熟度は中程度。2023年9月リリースで約2年が経過しており、主要フレームワーク（Spring Boot 3.2+）とライブラリのサポートは確立済み。Virtual Threadsなど新機能の活用事例も増加中だが、Java 17と比較すると情報量は少ない。

- **Java 25（LTS）**: エコシステムの成熟度が最も低い。2025年9月リリースで半年未満。主要フレームワークの対応状況は限定的で、サードパーティライブラリの互換性検証が不足している。ドキュメント・コミュニティサポートが未発達。

## 評価

### Java 17（LTS）

- 評価: ◎
- 利点:
  - Spring Boot 3.xが正式サポート（最小要件）。主要エンタープライズフレームワークとの互換性が広く検証されている
  - サードパーティライブラリのJava 17対応は成熟段階。互換性問題が発生しても情報が豊富
  - IDE（IntelliJ IDEA、Eclipse、VS Code）、ビルドツール（Maven、Gradle）、CI/CDツール全般で安定サポート
  - トラブルシューティング情報・Stack Overflow・コミュニティサポートが最も充実
  - 企業内での採用実績が多く、社内ナレッジの蓄積や外部エキスパートの確保が容易
- 欠点:
  - 新しい機能（Virtual Threads、新しいGC改善等）は含まれない。最新技術の恩恵を受けられない
  - 2026年時点で既にリリースから5年経過しており、最先端からは距離がある
- 根拠:
  - research.mdより、Spring Boot 3.0の最小要件がJava 17であり、数年間のセキュリティパッチと更新が保証される
  - 2021年リリースのため、主要なライブラリ・ツールの対応が完了し、エコシステム全体で安定稼働の実績がある
  - リリースから十分な期間が経過し、ドキュメント・ブログ記事・Q&Aサイトの情報が最も豊富
- CSD依存: なし

### Java 21（LTS）

- 評価: ○
- 利点:
  - Spring Boot 3.2以降で正式サポート。主要フレームワークの対応は完了している
  - Virtual Threadsなど新機能の活用事例が増加中で、ドキュメントやベストプラクティスが蓄積されつつある
  - IDE・ビルドツール・CI/CDツールのサポートは安定稼働段階。Java 17と同等の開発体験を提供
  - Java 17より新しい機能を使いながらも、LTSとしての安定性を享受できる
- 欠点:
  - Java 17と比較すると、トラブルシューティング情報・Stack Overflowの回答数・コミュニティサポートが相対的に少ない
  - サードパーティライブラリの互換性は改善中だが、一部のレガシーライブラリやニッチなツールでは検証不足の可能性がある
  - Virtual Threads等の新機能は、フレームワーク側の最適化が完全には進んでいないケースがあり得る（S2: Java 21のVirtual Threadsが並行処理で有効に機能する、に依存）
- 根拠:
  - research.mdより、Spring Boot 3.xはJava 19以降でもテスト済みであり、Java 21は十分にサポートされている
  - 2023年9月リリースで約2年が経過し、主要なエコシステム要素は対応完了済み。ただしJava 17（5年経過）と比較すると情報の蓄積に差がある
  - Virtual Threadsの性能改善は確認されているが、アプリケーション特性やフレームワークの最適化状況に依存する
- CSD依存:
  - S2（Java 21のVirtual Threadsが並行処理で有効に機能する）: Virtual ThreadsがSpring Bootなど主要フレームワークで最適に動作する保証は、フレームワークの実装に依存する。未検証の場合、新機能の恩恵が限定的になる可能性
  - D1（サードパーティライブラリのJava 21互換性）: 使用する予定のライブラリが未特定の場合、互換性問題が後から発覚するリスクがある

### Java 25（LTS）

- 評価: ×
- 利点:
  - 最新のLTSバージョンであり、長期的にはエコシステムがこのバージョンに追従する
  - 最新のJava仕様と機能を利用できる（ただし、エコシステムの成熟度とはトレードオフ）
- 欠点:
  - 2025年9月リリースで半年未満。主要フレームワーク（Spring Boot等）の正式サポートが限定的または未発表の可能性が高い
  - サードパーティライブラリの互換性検証が不足。多くのライブラリがJava 25を動作確認していない段階
  - IDE・ビルドツール・CI/CDツールのサポートは初期段階。バグや非対応機能が残っている可能性
  - ドキュメント・サンプルコード・トラブルシューティング情報が著しく不足。問題発生時に解決が困難
  - コミュニティサポート（Stack Overflow、フォーラム等）での情報が極めて少なく、社内ナレッジも蓄積されていない
- 根拠:
  - 2026年2月時点でJava 25は半年未満の新しいバージョン。エコシステムの成熟には通常2-3年を要する
  - 主要フレームワークのロードマップ上、新LTSへの正式対応には半年〜1年のタイムラグがある
  - リリース直後のバージョンは、予期しない互換性問題やツールの未対応が頻発するリスクが高い
- CSD依存:
  - D6（Java 25のエコシステム成熟度）: 2026年2月時点でJava 25のツール・ライブラリ・IDE・ドキュメントの成熟度が低いことは確実。早期採用による情報不足で開発効率が著しく低下するリスクがある
  - D1（サードパーティライブラリのJava 21/25互換性）: Java 25の場合、互換性問題が発覚する確率がJava 21よりさらに高い

### Java 11（LTS）

- 評価: △
- 利点:
  - エコシステムの成熟度は最高水準（2018年リリース、8年以上の実績）
  - ほぼ全てのライブラリ・ツール・フレームワークが完全サポート。互換性問題のリスクが最小
  - トラブルシューティング情報が極めて豊富。あらゆる問題に対する解決策が蓄積されている
- 欠点:
  - **Spring Boot 3.xが利用不可**（最小要件はJava 17）。最新のエンタープライズフレームワークに追従できない
  - 2026年9月にOracle Java 11がEOLを迎えるため、新規プロジェクトでの採用は推奨されない
  - エコシステム全体がJava 17以降に移行しつつあり、新しいライブラリやツールがJava 11非対応になる傾向がある
- 根拠:
  - research.mdより、Spring Boot 3.0の最小要件はJava 17。Java 11ではSpring Boot 2.x系を使用する必要があり、最新のエコシステムから取り残される
  - Java 11のEOL（2026年9月）が迫っており、新規プロジェクトでの採用はリスクが高い
  - エコシステムの成熟度は高いが、「将来も利用可能な成熟度」という観点では評価が低下する
- CSD依存:
  - S7（Java 11のサポート終了時期はディストリビューションに依存する）: OpenJDKディストリビューション（Eclipse Temurin等）では2026年9月以降もサポートが継続する可能性があるが、エコシステム全体のJava 17以降へのシフトは不可逆

### Java 8（LTS）

- 評価: ×
- 利点:
  - エコシステムの成熟度は最高水準（2014年リリース、12年以上の実績）
  - レガシーシステムとの互換性が最も高い
- 欠点:
  - **Spring Boot 3.xが完全に利用不可**。Spring Boot 2.x系すら最新バージョンではJava 8のサポートが縮小傾向
  - エコシステム全体がJava 8から完全に離脱しつつあり、新しいライブラリ・ツールのサポートが終了している
  - モダンな開発体験（Records、Pattern Matching、Stream API改善等）が得られない
  - セキュリティパッチの提供が有償化されている（Oracle）または限定的（OpenJDK）
- 根拠:
  - 新規開発プロジェクトでJava 8を採用することは、エコシステムの成熟度ではなく「時代遅れの技術の選択」を意味する
  - 主要フレームワークが既にJava 8サポートを終了または縮小しており、新規プロジェクトには不適切
- CSD依存: なし（選択肢として成立しない）
