# レッドチーム検証結果

## 再分類が必要な項目

### C2: Java 11のOracle JDKサポートは2026年9月にEOLを迎える

**問題点**: Certainty に分類されているが、スコープに矛盾がある

**反証**: 決定ステートメントで「JDKディストリビューション（Oracle JDK vs OpenJDK系）の選択は対象外」と明示されている。Oracle JDKの特定のサポート終了は、ディストリビューションに依存する事実である。OpenJDK系ディストリビューション（Eclipse Temurin, Amazon Corretto等）では、Java 11は異なるサポート期間を持つ

**具体例**: Eclipse Temurin 11はAdoptiumのサポート方針により、2027年10月までサポート継続の可能性がある。Amazon Corretto 11も同様に独自のサポート期間を持つ

**提案**: Suppositon に格下げ。「Oracle JDKのサポート終了は確実だが、プロジェクトで採用するディストリビューションが未定のため、実際のJava 11サポート終了時期は確定していない」として S7 に再分類

### C5: Javaの後方互換性は比較的高い

**問題点**: 「比較的高い」という相対的な表現が曖昧であり、検証可能な形式ではない

**反証**: Java 8→11では `javax.*` パッケージの一部がJava EE分離により削除された。Java 11→17では Nashorn JavaScript エンジンが削除され、セキュリティマネージャーが非推奨化された。「比較的高い」の基準が不明確で、プロジェクトの依存関係によっては互換性問題が発生する

**具体例**: 「可逆性がmedium」という評価は、後方互換性の高さではなく「移行が可能」という意味に過ぎない。段階的な移行が可能であることと、互換性が高いことは別の概念

**提案**: Supposition に格下げ。「一般的なJavaアプリケーションでは後方互換性が保たれることが多いが、使用するライブラリやAPIによっては破壊的変更の影響を受ける可能性がある」として S8 に再分類

### C6: JDKディストリビューション（Oracle JDK vs OpenJDK系）の選択は対象外

**問題点**: 「対象外」はスコープの定義であり、制約ではない

**反証**: これは技術的・ビジネス的な制約ではなく、この意思決定プロセスで扱う範囲の定義である。CSD分類は「決定に影響する制約」を対象とするものであり、スコープ定義はCSD分類の対象ではない

**提案**: CSD分類から除外。スコープ定義は decision-statement.md に既に記載されており、重複記載する必要はない

## 検出された暗黙の仮定

### 暗黙の仮定1: プロジェクトは単一のJavaバージョンを統一的に使用する

**詳細**: マイクロサービス構成など、複数のJavaバージョンを並行利用する可能性が考慮されていない。サービスごとに異なるバージョンを採用するシナリオも存在する

**影響**: 単一バージョン選択を前提とした議論が、実際のアーキテクチャと合致しない可能性がある

**提案**: Supposition として S9 を追加 — 「プロジェクトは単一のJavaバージョンで統一される。検証方法: システムアーキテクチャ（モノリス/マイクロサービス）を確認し、バージョン統一の必要性を評価する」

### 暗黙の仮定2: プロジェクトは長期運用を前提としている

**詳細**: S4で「開発期間は2年以上」と仮定されているが、運用期間の仮定が明示されていない。開発期間と運用期間は別の要素であり、運用期間がLTSサポート期間の重要性に直接影響する

**影響**: 短期プロジェクト（1-2年で終了）の場合、LTSサポート期間の重要性が低下し、最新機能へのアクセスが優先される可能性がある

**提案**: Supposition として S10 を追加 — 「プロジェクトは少なくとも3年以上の運用を前提とする。検証方法: プロジェクトのライフサイクル計画を確認し、運用終了時期の見込みを把握する」

### 暗黙の仮定3: エコシステムの成熟度は採用判断に影響する

**詳細**: D6でJava 25のエコシステム成熟度を不確実要素として挙げているが、逆に「エコシステムの成熟度が重要な判断基準である」という前提が暗黙に存在する。早期採用による競争優位性を重視する組織では、成熟度より新機能へのアクセスが優先される場合もある

**影響**: エコシステム成熟度の重要性が組織の技術戦略に依存する

**提案**: Supposition として S11 を追加 — 「プロジェクトはエコシステムの成熟度を重視し、安定性を優先する。検証方法: 組織の技術採用方針（保守的 vs 革新的）を確認する」

### 暗黙の仮定4: プロジェクトにはコンテナ化されたデプロイメントが想定されている

**詳細**: D5で「インフラ環境の詳細」が不確実とされているが、リサーチ結果でコンテナ起動時間やGCの最適化が調査されていることから、コンテナ環境が暗黙の前提になっている可能性がある

**影響**: 従来型のオンプレミス環境やVM環境では、コンテナ特有の最適化（起動時間短縮、メモリ効率等）の価値が低下する

**提案**: Doubt として D7 を追加 — 「デプロイメント環境（コンテナ/VM/物理サーバー）とオーケストレーション方式。影響範囲: コンテナ環境特有の最適化（CDS, Generational ZGC等）の効果が環境依存。リスク: 環境が確定していない場合、性能最適化の優先順位が評価できない」

### 暗黙の仮定5: プロジェクトはJava言語自体を継続利用する

**詳細**: 新規開発であるにもかかわらず、Javaを使用すること自体が自明とされている。KotlinやScalaなどのJVM言語への移行、あるいは他言語の選択は考慮されていない

**影響**: Javaバージョン選択の議論が、「そもそもJavaを使うべきか」という上位の意思決定を見落としている可能性がある

**評価**: これは決定ステートメントのスコープ外であり、「新規開発プロジェクトでのJavaバージョン選択」という議題が明示されている。ただし、決定後に「Javaではなく別言語を使うべきだった」というシナリオがPremortem分析で考慮されるべき

**提案**: CSDには追加せず、Phase 5（プレモーテム分析）で「Java採用自体の妥当性」を検証項目として含めることを推奨
