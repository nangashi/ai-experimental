# レッドチーム検証結果

## 再分類提案

- C5: Certainty → Supposition — 理由: 「Gradle 9.3.1 と Maven 3.9.12 が最新安定版として利用可能である」は2026年2月16日時点の事実だが、これ自体は決定時点では確実でも、決定の実行時には新しいバージョンがリリースされる可能性がある。また、「最新安定版」という記述は相対的であり時間とともに変化する。この項目は「リサーチ時点での事実」としては確実だが、「決定の前提として動かない制約」としては仮定に近い。ただし、この項目が制約として機能する場合（特定バージョンの採用を前提とする場合）は Certainty として維持可能。現在の記述では、バージョン情報は「現時点の事実」として記録されているため、Supposition に格下げを推奨する。検証方法: プロジェクト開始時点での最新バージョンを確認する

- C6: Certainty → Supposition — 理由: 「パフォーマンス面では Gradle が Maven より優位である」は research.md の調査結果に基づくが、これは一般論であり、プロジェクト固有の条件（規模、ビルド構成、CI/CD環境等）で結果が変わる可能性がある。また、Maven 4 のリリースにより状況が変化する可能性もある（S6参照）。プロジェクト固有の計測結果がない限り、これは仮定として扱うべき。検証方法: プロジェクト固有の条件で実測する

- C7: Certainty → Supposition — 理由: 「Maven のエコシステムは成熟している」は確かだが、これが「今回のプロジェクトで必要な全てのプラグインが利用可能」を意味するかは検証が必要。特定のユースケース（例: Java 21 固有の機能、Spring Boot 3.x との統合の詳細）で必要なプラグインが Maven に存在するかは、プロジェクト固有の検証が必要。検証方法: 必要なプラグインの存在と互換性を確認する

- C8: Certainty → Supposition — 理由: 「Gradle は Groovy/Kotlin DSL により柔軟な設定が可能である」は一般的な事実だが、この柔軟性が「今回のプロジェクトで実際に必要か」は不確実（D3参照）。また、柔軟性がチームにとってメリットか（学習コスト、保守性の観点でデメリットになる可能性もある）は D1 に依存する。この項目自体は Certainty だが、「柔軟性が必要」という前提は暗黙的に含まれており、それは仮定として分離すべき。ただし、C8 自体はツールの特性の記述として正確なため、Certainty 維持も可能。むしろ、「柔軟性が必要」を S4 の一部として明示的に扱うべき

## 暗黙の仮定の検出

- プロジェクトのビルドツール選択は自律的に行える（組織標準やガバナンス要件に制約されない） — 推奨分類: Doubt（D6として既に記載されているが、その重要性を強調）、理由: D6 は既に記録されているが、この仮定が崩れた場合、全ての技術的評価が無意味になる可能性がある。最優先で検証すべき項目として明記すべき

- チームは Groovy/Kotlin DSL（Gradle の場合）または XML（Maven の場合）の記述に抵抗がない — 推奨分類: Supposition、理由: D1 で「習熟度」は記録されているが、「記述形式への好み・抵抗感」は別の要素。特に Kotlin DSL は Java 開発者にとって学習が必要であり、チームの受容性は仮定として明示すべき。検証方法: チームメンバーに DSL サンプルを提示してフィードバックを得る

- プロジェクトは単一のビルドツールに統一される — 推奨分類: Supposition、理由: 複数のサブプロジェクトやマイクロサービスで異なるビルドツールを使用する可能性は考慮されていない。長期運用前提のプロジェクトでは、将来的な分割や統合が発生する可能性があり、その際にビルドツール混在のリスクを検証すべき。検証方法: プロジェクトアーキテクチャの将来計画を確認する

- ビルドツールの移行コストは許容範囲内である — 推奨分類: Supposition、理由: C9 で「可逆性は medium」と定義されているが、「medium の移行コストが許容できる」という前提は明示されていない。移行コストの許容度はビジネス判断であり、技術的な制約ではない。検証方法: プロジェクトの予算とスケジュールの余裕度を確認する

- Java 21 の新機能（Virtual Threads 等）は Gradle/Maven 両方で問題なく利用できる — 推奨分類: Supposition、理由: S8 で「Virtual Threads等をビルドプロセスで活用する可能性」は記録されているが、「Gradle/Maven のどちらでも Java 21 の全機能が問題なく利用できる」という前提は暗黙的。特に Maven 3.x 系列での Java 21 サポート状況は検証が必要（Maven 4 では改善される可能性がある: S6参照）。検証方法: 各ツールの Java 21 サポート状況とベストプラクティスを調査する
