# クロスデバイス対応リーディングリスト管理ツール: 要件定義書

## ステータス

Confirmed

## 日付

2026-02-19

## 1. 問題定義

### 1.1 何が問題か

RSSリーダーとXで見つけた記事を一元管理する仕組みがなく、各アプリのブックマーク機能が分散しているため、記事が行方不明になったり読み忘れが発生する。特にXのブックマークは見返す習慣がなく漏れやすい。

### 1.2 なぜ今解決が必要か

複数ソース（RSSリーダー・X・Web閲覧中の関連記事）の併用が定着し、共通のブックマーク手段がない不便さが蓄積してきた。また、AI開発の学習機会としても活用したい。

### 1.3 期待するゴール状態

有用な記事が行方不明にならず、読み忘れも防げる状態。どのデバイスからでも保存した記事にアクセスでき、未読状況を一元管理できる。

## 2. 背景

### 2.1 現状のプロセス

RSSリーダーとXそれぞれのブックマーク機能を個別に利用。RSSリーダー側は大きな不満はないが、X側はブックマークを見返す習慣がなく読み忘れが発生しやすい。2つのアプリにまたがるため未読管理がしにくい。

### 2.2 過去の試行

Notion保存を検討したが、運用の手間がネックで不採用。

### 2.3 ステークホルダー

| 役割 | 関心事 | 影響度 |
|------|--------|--------|
| 自分自身（個人開発者・利用者） | 記事の一元管理、未読消化の効率化、AI開発の学習 | 高 |

## 3. アプローチ選定

### 3.1 検討した代替案

| # | アプローチ | 概要 | 採否 |
|---|-----------|------|------|
| 1 | 自作アプリ | Claude APIを活用したリーディングリスト管理ツールを自作開発 | 採用 |
| 2 | 既存サービス活用 | CuraQ等の無料枠で運用 | 不採用 |
| 3 | やらない | 現状のまま。多少不便だが致命的ではない | 不採用 |

### 3.2 選定結果

- **採用**: 自作アプリ（Claude APIを活用したリーディングリスト管理ツール）
- **理由**: AI開発の学習を兼ねられる。無料枠の制約を受けずに自由に機能拡張できる
- **不採用案の棄却理由**: CuraQ等は無料枠に制約があり学習目的も満たせない。やらない場合は不便が継続する

## 4. 制約

| 種別 | 内容 | 根拠 |
|------|------|------|
| コスト | 無償で実現（Claude APIは契約済みを利用） | 個人利用のため |
| デバイス | 既存のAndroidスマホ・PCを使用 | 新規デバイス購入の予定なし |
| 記事ソース | 主にRSSリーダー・X・Web閲覧中の関連記事 | 現在の情報収集習慣 |
| 認証モデル | アクセスは認証済みの所有者本人に限定（シングルユーザー前提） | Phase 1 ギャップ解決: シングルユーザー前提、アクセスは所有者本人に限定 |
| 接続性 | オンライン前提（オフライン操作不要） | Phase 1 ギャップ解決: オンライン前提 |

## 5. スコープ

### 5.1 Must（必須）

- 記事の保存（スマホ・PCから素早く保存できること）
- 未読一覧の表示・既読管理（既読化・未読戻し）
- 記事の削除
- 過去の記事の検索
- 重複保存の警告
- 未読件数の表示・蓄積警告
- クロスデバイスアクセス（スマホ・PC）

### 5.2 Nice-to-have（あれば良い）

- AIによるおすすめ度評価・並替（UC-11）
- 要約・学習ポイントの自動抽出・表示（UC-12）
- 自動タグ付け・タグ検索（UC-13）
- 興味に基づく新規記事提案（UC-14）
- 記事の詳細要約（UC-15）
- フラッシュカード復習機能（UC-16）

### 5.3 スコープ外（明示的除外）

- AI追加機能全般（UC-11〜UC-16）: 基本機能実装後に優先度を決定
- リンク切れの自動検出: 気づいた時に手動削除で対応
- 別URLの同一記事検出: チェック処理の実装コストに見合わない（将来検討）

## 7. システム境界

### 7.1 システム概要

クロスデバイス対応リーディングリスト管理ツールは、記事URLの保存・未読管理・検索をクロスデバイスで提供するシステムである。

### 7.2 アクター一覧

| ID | アクター | 種別 | 概要 |
|----|---------|------|------|
| ACT-1 | 所有者（自分自身） | 人間 | 記事の保存・閲覧・管理を行う唯一のユーザー |
| ACT-2 | Androidブラウザ/アプリ | 外部システム | 共有メニュー経由で記事URLを送信する |
| ACT-3 | PCブラウザ | 外部システム | Webアプリへのアクセス手段を提供する |

### 7.3 外部インターフェース

| ID | インターフェース | 接続先 | 方向 | 概要 |
|----|-----------------|--------|------|------|
| IF-1 | Android共有メニュー経由のURL受信 | ACT-2: Androidブラウザ/アプリ | 入力 | Android共有メニューから記事URLを受け取る |
| IF-2 | Webアプリ画面 | ACT-1: 所有者, ACT-3: PCブラウザ | 双方向 | 記事の保存（URLペースト）・一覧表示・管理操作を提供する |

### 7.4 スコープ

**内部（システムが担う範囲）:**
- 記事URLの保存・既読管理・削除
- 未読一覧の表示・未読件数の表示・蓄積警告
- 記事の検索（タイトル・キーワード）
- 重複保存の検出・警告
- アクセス制御（所有者本人のみ利用可能）
- データの永続保持・クロスデバイス同期

**外部（システムが担わない範囲）:**
- 記事コンテンツのホスティング（リンク先サイトが提供）
- Android共有メニューの送信側の制御（ブラウザ/RSSリーダー/Xアプリが担う）
- AI追加機能（UC-11〜16: MVP後に優先度決定）
- リンク切れの自動検出
- 別URLの同一記事検出

## 8. 機能要件

| SR-ID | 要件 | EARS型 | トレース元 |
|-------|------|--------|-----------|
| SR-001 | ユーザーがWebアプリ画面でURLを入力して保存を実行したとき、システムは記事のタイトルおよび保存日時を取得して記事リストに追加すること | Event-Driven | UC-1 |
| SR-002 | ユーザーがAndroid共有メニューから記事URLを送信したとき、システムは当該URLを受け取り記事のタイトルおよび保存日時を取得して記事リストに追加すること | Event-Driven | UC-1 |
| SR-003 | システムは全ての保存済み記事に保存日時を記録すること | Ubiquitous | UC-1 |
| SR-004 | ユーザーが未読一覧画面を開いたとき、システムは未読状態の記事の一覧を表示すること | Event-Driven | UC-2 |
| SR-005 | 未読記事の一覧はどのデバイス（スマートフォン・PC）からアクセスした場合も同一の内容を表示すること | Ubiquitous | UC-2 |
| SR-006 | ユーザーが記事に対して既読操作を実行したとき、システムは当該記事の状態を既読に変更し、未読一覧から除外すること | Event-Driven | UC-3 |
| SR-007 | ユーザーが既読記事に対して未読に戻す操作を実行したとき、システムは当該記事の状態を未読に変更し、未読一覧に再表示すること | Event-Driven | UC-4 |
| SR-008 | ユーザーが記事に対して削除操作を実行したとき、システムは当該記事を記事リストから削除すること | Event-Driven | UC-5 |
| SR-009 | ユーザーが検索キーワードを入力して検索を実行したとき、システムはタイトルまたはURLにキーワードを含む記事の一覧を未読・既読を問わず表示すること | Event-Driven | UC-6 |
| SR-010 | ユーザーが既に保存済みのURLを再度保存しようとしたとき、システムは重複保存の警告メッセージを表示し、保存操作を中止すること | Event-Driven | UC-7 |
| SR-011 | 未読記事が1件以上存在する間、システムは未読件数を表示すること | State-Driven | UC-8 |
| SR-012 | 未読記事が0件である間、システムは未読件数を0として表示すること | State-Driven | UC-8 |
| SR-013 | 未読記事件数が20件を超えた場合、システムは未読蓄積の警告メッセージを表示すること | Unwanted | UC-9 |
| SR-014 | 入力されたURLの形式が不正である場合、システムはエラーメッセージを表示し保存操作を中止すること | Unwanted | UC-10 |
| SR-015 | 記事の保存処理中にネットワークエラーが発生した場合、システムは保存に失敗した旨のエラーメッセージを表示すること | Unwanted | UC-10 |
| SR-016 | 保存操作においてURLが空の場合、システムはエラーメッセージを表示し保存操作を中止すること | Unwanted | UC-10 |
| SR-017 | ユーザーが記事の状態を変更したとき（保存・既読化・未読化・削除）、システムは変更を他のデバイスからの次回アクセス時に反映すること | Event-Driven | UC-2, UC-3, UC-4, UC-5 |
| SR-018 | ユーザーが既読一覧画面を開いたとき、システムは既読状態の記事の一覧を表示すること | Event-Driven | UC-6 |

## 9. 非機能要件

| NFR-ID | 観点 | 要件 | 測定基準 | 測定方法 | 導出根拠 |
|--------|------|------|---------|---------|---------|
| NFR-1 | コスト | 月額運用コスト上限: 0円（Claude API利用料を除く）。Claude APIは既存契約の範囲内で利用する | 月額インフラ費用 0円 | 月次のインフラ利用料明細で確認 | 制約「個人利用のため」。Claude APIはMVP範囲外 |
| NFR-2 | 運用負荷 | リリース作業は手動操作1コマンド以内で完了すること。定常運用は月1回30分以内 | リリース操作コマンド数 ≤ 1、月次運用作業 ≤ 30分 | リリース手順書のステップ数、月次運用作業の所要時間を記録 | 個人開発者が本業の合間に運用する前提 |
| NFR-3 | パフォーマンス | 画面表示・操作応答: 3秒以内。記事保存（タイトル取得含む）: 5秒以内。検索結果表示: 3秒以内。デバイス間同期: 次回アクセス時に反映済み | 画面表示・操作応答 ≤ 3秒、記事保存 ≤ 5秒、検索結果表示 ≤ 3秒 | ブラウザ開発者ツールによるレスポンス時間計測 | Nielsenの応答時間閾値に基づく。記事保存は外部通信含むため5秒に緩和 |
| NFR-4 | セキュリティ | アクセスは認証済みの所有者本人に限定。APIキー等はソースコードに含めず環境変数または暗号化設定で管理。通信はHTTPSで暗号化 | 認証なしアクセス拒否、ソースコード内認証情報ゼロ件、HTTPS使用 | 認証なしアクセス拒否確認、ソースコード内認証情報スキャン、HTTPS確認 | シングルユーザー・個人利用。インターネット経由アクセスのため認証・暗号化は最低限必要 |
| NFR-5 | 可用性 | 稼働率目標: 設定しない。許容停止時間: 数日程度。RTO: 1日以内。RPO: 直近バックアップ時点。定期バックアップを行うこと | RTO ≤ 1日、バックアップ実施の確認 | 障害発生時の復旧所要時間記録、バックアップからのリストア確認 | 個人利用でSLA不要。データは再作成コストが高いためバックアップ必要 |
| NFR-6 | ユーザビリティ | 記事保存の操作ステップ: スマホは共有メニューから2ステップ以内、PCはURLコピー後3ステップ以内で完了すること | スマホ保存 ≤ 2ステップ、PC保存 ≤ 3ステップ | 各デバイスでの保存操作ステップ数のカウント | Phase 1で合意した「素早く保存」の定量化 |

## 11. 成功基準・リスク

### 11.1 完了条件

- システムが実際に動作し、CuraQ等の類似サービスと比較して使い勝手が良い
- 記事保存の運用を開発したシステムに移行した状態

### 11.2 リスク

| リスク | カテゴリ | 影響 | 対策 |
|--------|----------|------|------|
| 利用サービスの停止・無料枠条件変更 | 依存関係 | システム継続不可 | 特定サービスへの依存度を意識し、移行可能な設計にする |
| 使い勝手が悪いシステムになる | その他 | 利用しなくなり失敗 | CuraQ等を参考にUX重視で開発、早期に実運用で検証 |
| 開発・リリースに手間がかかりすぎる | スコープ肥大化 | 既存サービスに流れる | リリース作業の自動化、スコープを最小限に保つ |
| 学習目的で不慣れな技術を選び開発が長期化 | スコープ肥大化 | 完成が遅れ運用移行できない | 学習と実用のバランスを意識した技術選定 |

## 付録

### A. 用語集

| 用語 | 定義 |
|------|------|
| リーディングリスト | 後で読むために保存した記事のURL一覧。未読・既読の状態を持つ |
| 未読 | 保存済みであるが、まだ既読操作を行っていない記事の状態 |
| 既読 | ユーザーが明示的に既読操作を行った記事の状態 |
| クロスデバイス | 複数の異なる種類のデバイス（スマートフォン・PC等）をまたいで同一のデータにアクセスできること |
| Android共有メニュー | Androidアプリからコンテンツを別のアプリへ送信するためのOS標準機能 |
| RSSリーダー | RSSフィードを購読し、記事を一覧表示するアプリケーション |
| EARS記法 | Easy Approach to Requirements Syntax。要件を「〜のとき、〜すること」等のパターンで記述する手法 |
| MVP | Minimum Viable Product。最小限の機能を備えた実用可能なプロダクト |
| RTO | Recovery Time Objective（目標復旧時間）。障害発生からシステムが復旧するまでの目標時間 |
| RPO | Recovery Point Objective（目標復旧地点）。障害発生時に許容できるデータ損失の範囲（時点） |
| SR | System Requirement（システム要件）。本ドキュメントで定義する機能要件の識別子 |
| NFR | Non-Functional Requirement（非機能要件）。パフォーマンス・セキュリティ等の品質要件の識別子 |
| IR | Interface Requirement（インターフェース要件）。外部インターフェースに関する要件の識別子 |
| HTTPS | HyperText Transfer Protocol Secure。TLSにより通信内容を暗号化したHTTPプロトコル |
| CuraQ | 個人向けリーディングリスト管理の既存サービス（本プロジェクトの比較対象） |
