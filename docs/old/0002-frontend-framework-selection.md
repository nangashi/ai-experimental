# ADR-0002: フロントエンドフレームワーク選定

## ステータス

Proposed

## 日付

2026-02-19

## コンテキスト

Cloudflare Pages + Cloudflare Workers（Hono + TypeScript）構成のリーディングリスト管理PWAにおいて、フロントエンドフレームワークとしてどの技術を採用するかを決定する必要がある。

バックエンドはADR-0001でCloudflare Workers + Hono + TypeScriptに確定済みであり、ホスティング基盤もCloudflare Pagesに確定済みである。今回の決定はフロントエンドのUIフレームワーク/ライブラリの選定に限定される。

**スコープ:**
- 対象: フロントエンドのUIフレームワーク/ライブラリの選定。PWA対応（Web Share Target API含む）、レスポンシブ対応（Android + PC）、Cloudflare Pagesとの互換性、学習目的としての適性を評価する
- 対象外: バックエンド技術（Cloudflare Workers + Hono + TypeScript で確定済み — ADR-0001）、ホスティング基盤（Cloudflare Pages で確定済み — ADR-0001）、CSSフレームワーク/UIコンポーネントライブラリ、データベース選定、認証方式

**可逆性:** medium — フレームワーク変更はUIコード全体の書き直しが必要だが、バックエンドAPI（Hono）には影響しない。初期段階（CRUD UIが小規模）での変更コストは比較的低い。

## 関連する既存決定

- **ADR-0001: PWA+REST API構成のホスティング基盤選定** — Cloudflare Pages（フロントエンド）+ Cloudflare Workers（バックエンド）を採用する決定。フロントエンドのビルド成果物はCloudflare Pagesにデプロイ可能な静的ファイル形式（HTML/CSS/JS）であること、月額コスト0円（無料ティア内）の制約、シングルユーザー構成、画面表示3秒以内の要件が今回の決定に引き続き適用される。

## 前提条件

### 確実（Certainty）

- C1: フロントエンドのデプロイ先は Cloudflare Pages に確定済みである — 根拠: ADR-0001 で決定済み
- C2: バックエンドは Cloudflare Workers + Hono + TypeScript（REST API）で確定済みである — 根拠: ADR-0001 で決定済み
- C4: 月額コストは 0 円（Cloudflare 無料ティア内）の制約が適用される — 根拠: ADR-0001 の C3 より。ただし主要候補フレームワークは全て OSS であり、この制約で除外される候補は実質ない
- C6: 画面表示 3 秒以内の要件が引き続き適用される — 根拠: ADR-0001 の C5 より
- C7: PWA 対応（マニフェスト + Service Worker）および Web Share Target API のサポートが必要 — 根拠: 決定ステートメントのスコープに明記
- C8: レスポンシブ対応（Android + PC）が必要 — 根拠: 決定ステートメントのスコープに明記。CSS レベルの実装であり、主要候補フレームワークはいずれも妨げない
- C10: Web Share Target API の受け取りエンドポイントは Cloudflare Workers（Hono）側で実装する — 根拠: research.md の調査結果

### 仮定（Supposition）

- S1: 開発者（1名）は React の基本的な知識を持つが、Svelte / SolidJS の実務経験はない — 根拠: 「学習目的としての適性を評価する」という記述から推定 — 検証方法: 開発者に直接確認
- S2: アプリの規模は小規模（CRUD UI が中心）であり、長期的に大きく拡張する可能性は低い — 根拠: リーディングリスト管理という用途特性 — 検証方法: 今後のロードマップを確認
- S3: 主要候補フレームワーク（React / Vue / Svelte / SolidJS）はいずれも Vite ベースの構成で Cloudflare Pages にデプロイ可能である — 根拠: research.md の調査結果。SvelteKit v2 も Vite を標準採用しており、Vite ベースと見なせる — 検証方法: 選択したフレームワークの公式ドキュメントで確認
- S4: vite-plugin-pwa が選択するフレームワークで安定的に動作する — 根拠: vite-plugin-pwa は Vite ベースフレームワークを広くサポート。ただし SvelteKit は専用パッケージ（@vite-pwa/sveltekit）が必要であり、設定方法・統合難易度が異なる可能性がある — 検証方法: 実際の Cloudflare Pages へのデプロイで検証
- S5: 選択するフレームワークのメジャーバージョンは今後 1-2 年で大きく変化しない — 根拠: Svelte 5 安定版、React 19 安定版、Vue 3.x 安定。SolidStart は移行期でリスクが高い — 検証方法: 各フレームワークのロードマップを確認
- S6: 現構成では静的出力を前提とするが、Cloudflare Pages Functions を利用すれば SSR 構成も実現可能である — 根拠: レッドチーム指摘。SvelteKit 等は @sveltejs/adapter-cloudflare で SSR 対応可能 — 検証方法: SSR を検討する場合、Pages Functions の制約（CPU時間等）を確認
- S7: 現時点はシングルユーザー前提だが、D2（チーム化の可能性）により変化しうる — 根拠: レッドチーム指摘。D2 が現実化すると崩れる — 検証方法: D2 の解消に伴い再評価
- S8: バックエンドが TypeScript であり、フロントエンドも TypeScript を採用することが強く望ましい — 根拠: バックエンドとの言語統一は合理的だが、フロントエンドへの TypeScript 強制は明示されていない — 検証方法: 決定者に確認
- S9: 評価対象は React / Vue / Svelte / SolidJS に限定する — 根拠: Preact / Qwik / Lit 等は暗黙的に除外されている。主要フレームワークに絞ることで評価コストを抑える判断 — 検証方法: 除外候補が有力な場合は追加検討
- S10: Service Worker の実装は vite-plugin-pwa（または同等プラグイン）に委譲できる — 根拠: 手動実装が必要になるシナリオは未検討 — 検証方法: 候補フレームワークでの PWA 実装事例を確認

### 不確実（Doubt）

- D1: 開発者の「学習目的」の優先度が、生産性（既存スキルの活用）とどの程度トレードオフになるか明確ではない — 影響範囲: フレームワーク選定の重要な評価軸に直接影響 — リスク: 学習コストを過小/過大評価した選定
- D2: プロジェクトが個人利用を超えてチーム開発に拡張される可能性がある — 影響範囲: エコシステムの大きさ・採用市場でのスキル汎用性の評価に影響。S7 と関連 — リスク: 将来のチーム参加時にフレームワーク変更が必要になる可能性
- D3: SolidStart 2.0 の API 変更が既存の学習リソースを陳腐化させる速度が不明 — 影響範囲: SolidJS/SolidStart を採用した場合のみ影響 — リスク: ドキュメント混在による学習効率の著しい低下

## 判断基準

優先度は決定者（人間）が設定する。

- **OBJ-1: PWA要件の実現容易性** — マニフェスト・Service Worker・Web Share Target APIを含むPWA要件を、過剰なハックや手動実装なしに実現できること。評価の視点: vite-plugin-pwa（または同等プラグイン）との統合サポートの成熟度、Web Share Target APIのフロントエンド受け取り処理との相性、Service Workerのキャッシュ戦略をカスタマイズできる柔軟性、公式または信頼できるPWA実装事例・ドキュメントの充実度
- **OBJ-2: 初期立ち上げ速度と開発継続性** — 開発者が既存スキルを活かしつつアプリを完成させられること、または新しいスキルを習得しながらも開発を止めずに進められること。評価の視点: 開発者の既存React知識との連続性（学習コストの初期障壁）、スターターテンプレート・CLIツールの充実度、エラー時の問題解決情報の充実度、小規模CRUD UIを構築するまでに必要な概念数の少なさ
- **OBJ-3: 新しい概念・パターンの学習価値** — このプロジェクトを通じて、既存スキル（React）と異なるアプローチやアーキテクチャパターンを学習できること。評価の視点: Reactと異なる設計思想（リアクティビティモデル・コンパイル時最適化・シグナルベースなど）の体験可能性、Web標準に近い書き方を学べるか、習得した知識の汎用性、学習リソースの質と現時点での最新性
- **OBJ-4: Cloudflare Pagesとの統合安定性** — Cloudflare Pagesへの静的ビルド・デプロイが安定して動作し、将来的なSSR移行オプションも確保できること。評価の視点: Viteベースの静的出力ビルドの実績、Cloudflare Pages向けのアダプター・設定の存在と成熟度、SSR移行時のCloudflare Pages Functions対応状況（S6の対処）、ビルド設定の複雑さ
- **OBJ-5: エコシステムの長期的安定性** — 採用後1-3年間、フレームワーク自体のAPI破壊的変更やコミュニティの縮小によって追加コストが発生しないこと。評価の視点: メジャーバージョンの安定性、npmダウンロード数・GitHubスター数・コントリビューター数などのコミュニティ規模、大手企業またはOSS財団のサポートの有無、既存学習リソースの量と陳腐化リスク（D3との関連）

## 検討した選択肢

### ALT-1: React（Vite + React 19）

既存スキルを最大限活用できる最大エコシステムのフレームワーク。Vite ベース SPA として Cloudflare Pages に静的デプロイ。SSR 移行時は React Router v7 / OpenNext 経由が必要。OBJ-1（情報量の多さ）、OBJ-2（既存スキル活用）、OBJ-4（静的デプロイの安定性）、OBJ-5（最大エコシステム・Meta 支援）の観点から提案。

### ALT-2: Vue 3（Vite + Vue）

React 経験者に学習障壁が低く、Composition API で React hooks に近い開発体験を提供。Nuxt が Cloudflare Workers GA に含まれ、SSR 移行パスも存在。OBJ-1（React に次ぐ事例数）、OBJ-2（React との近似性による低い移行コスト）、OBJ-3（リアクティビティの違いを学べる）、OBJ-5（安定した 3.x 系）の観点から提案。

### ALT-3: Svelte 5 / SvelteKit

コンパイル時リアクティビティという根本的に異なる設計思想。Svelte 5 の Runes システムで現代的なシグナルパターンを体験可能。`@sveltejs/adapter-cloudflare` による公式 Cloudflare 対応が最も手厚い。OBJ-1（@vite-pwa/sveltekit 専用パッケージ）、OBJ-3（最高の学習価値）、OBJ-4（Cloudflare GA 対応・SSR 移行が最もシームレス）、OBJ-5（高い開発者満足度）の観点から提案。

### ALT-4: SolidJS / SolidStart

React に似た JSX 構文で仮想 DOM を使わないシグナルベースのリアクティビティ。学習価値は高いが、SolidStart 2.0 移行期によるドキュメント混在リスクとエコシステムの小ささが課題。OBJ-3（シグナルベースのリアクティビティの純粋な実装）の観点から提案。

### 除外した選択肢

なし（S9 により Preact / Qwik / Lit 等は評価対象外としたが、制約による除外ではなく評価コスト削減のための限定）

## 根拠

Reactに慣れておらず学習したいと考えており（S1の前提「React知識あり」は実態と異なる）、React自体が学習対象として機能する。加えて、OBJ-2（立ち上げ速度）・OBJ-5（長期安定性）の◎評価、OBJ-1（PWA要件）・OBJ-4（CF Pages統合）の○評価と、学習以外の判断基準でも高い評価を得ているため総合的に選択した。

## 受け入れたトレードオフ

OBJ-3（新しい概念・パターンの学習価値）の評価は×であったが、これはS1（開発者がReactの基本知識を持つ）を前提とした評価である。実態としてReactに慣れておらず学習目的であるため、React自体が新しい概念の学習対象となり、OBJ-3の×は実質的なトレードオフとならない。ただし、コンパイル時リアクティビティやシグナルベースなどReact以外の設計思想を学ぶ機会は得られない点は認識している。
