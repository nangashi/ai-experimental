# ADR-0003: データベース選定

## ステータス

Proposed

## 日付

2026-02-19

## コンテキスト

Cloudflare Workers + Hono + TypeScript バックエンドのリーディングリスト管理アプリにおいて、データベース技術を選定する必要がある。

管理するデータは記事 URL（数百〜数千レコード）であり、タイトル・URL に対するキーワード検索、およびバックアップ・リストア対応が要件として挙げられている。

バックエンドは Cloudflare Workers（V8 Isolate ランタイム）上で動作するため、TCP ソケットの直接使用に制限があり、採用するデータベースは Workers ランタイムと互換性を持つ方式に限定される。具体的には、HTTP/HTTPS ベースの API を持つデータベースサービス、または Cloudflare が公式に提供する Bindings（D1 等）に限定される。

対象外: バックエンドフレームワーク（Cloudflare Workers + Hono + TypeScript で確定済み — ADR-0001）、ホスティング基盤（Cloudflare で確定済み — ADR-0001）、フロントエンドフレームワーク（React で確定済み — ADR-0002）、テーブル設計・スキーマ設計、マイグレーション方式。

可逆性: medium。データベース変更にはスキーマ移行・データ移行・ORM/クエリ層の書き換えが必要だが、小規模データ（数千レコード）のため移行コストは限定的。初期段階での変更は比較的容易。

## 関連する既存決定

- **ADR-0001: PWA+REST API構成のホスティング基盤選定** — Cloudflare Pages（フロントエンド）+ Cloudflare Workers（バックエンド）を採用。バックエンドは V8 Isolate ランタイム上で動作するため、データベースは Workers から接続可能な方式に限定される。月額コスト 0 円（無料ティア内）、シングルユーザー構成、画面表示 3 秒以内の制約が引き続き適用される。
- **ADR-0002: フロントエンドフレームワーク選定** — React（Vite + React 19）を採用。フロントエンドとデータベースは API 層で完全に分離されており、本決定に直接の影響はない。

## 前提条件

### 確実（Certainty）

- C1: データベースは Cloudflare Workers（V8 Isolate ランタイム）から接続可能な方式でなければならない — 根拠: ADR-0001 で確定。HTTP/HTTPS ベースの API を持つデータベースサービス、または Cloudflare が公式に提供する Bindings（D1 等）を使う方式が現実的（注: `cloudflare:sockets` による一部 TCP 接続は技術的に可能だが、従来型 DB プロトコルへの直接接続は一般的に推奨されない）
- C2: 月額コストは 0 円（無料ティア内）でなければならない — 根拠: ADR-0001 の C3 で確定。有料プランへの移行は前提としない
- C3: シングルユーザー構成であり、実運用ユーザーは開発者自身の 1 名のみ。大規模スケーリングへの対応は不要 — 根拠: ADR-0001 の C4 で確定。ユーザーのロケーションは開発者の居住地に固定される
- C4: データベースアクセスのレイテンシを含む画面表示 3 秒以内の要件がある — 根拠: ADR-0001 の C5 で確定
- C5: Workers はリクエストをまたいだ持続的接続を持たない（コネクションプールなし） — 根拠: V8 Isolate モデルの特性。毎リクエスト接続確立となるため、接続オーバーヘッドの低い方式（D1 Binding や HTTP API 型）が有利

### 仮定（Supposition）

- S1: 管理するレコード数は将来的にも数千レコード（最大でも数万レコード）程度に収まる — 根拠: 個人向けリーディングリストという用途特性 — 検証方法: 1年後にデータ件数・サイズを実測し、D1 Free プラン制限（500 MB）の 10% を超えていないか確認
- S2: キーワード検索の要件は、タイトル・URL に対する部分一致検索で充足できる — 根拠: スコープに「タイトル・URLのキーワード検索」とある。LIKE 検索または FTS5 で満たせると想定 — 検証方法: 実際の検索利用パターンを観察し、性能・機能的に十分かを確認
- S3: バックアップは日次〜週次程度の手動スナップショットで十分であり、頻繁な実行は不要 — 根拠: 個人用途のため — 検証方法: バックアップ頻度・復元要件を利用開始後に確認
- S4: Cloudflare D1 は現時点で GA 済みでプロダクション利用可能であり、Free プラン（500 MB/DB・10 DB）が継続提供される — 根拠: リサーチで確認（2026年2月時点）。ただし製品状態・プライシングは変更されうる — 検証方法: Cloudflare D1 リリースノート・プライシングページを定期確認
- S5: Turso の無料プランが継続して提供される — 根拠: 現時点の無料枠（5 GB、5億ロウ読み取り/月）をリサーチで確認 — 検証方法: Turso の料金ページを定期確認
- S6: データアクセスパターンは CRUD + キーワード検索のみで十分である — 根拠: リーディングリスト管理の基本機能から推定。ただし将来的にタグフィルタリング、ステータス管理、外部サービス連携等の機能追加でアクセスパターンが変化する可能性がある — 検証方法: 機能拡張検討時にデータベースの集計クエリ・リレーション設計への影響を評価

### 不確実（Doubt）

- D1: D1 の FTS5 仮想テーブルを含むデータベースが wrangler d1 export の対象外であることの、実運用上の許容可能性 — 影響範囲: D1 で FTS5 を使用した場合のバックアップ・リストア。FTS5 仮想テーブル再作成手順の別途整備が必要 — リスク: 手順の複雑化、リストア後の FTS5 インデックス再構築忘れリスク
- D2: Cloudflare D1 の Free プラン制限が将来縮小または変更されるリスク — 影響範囲: D1 選択時のコスト構造 — リスク: 無料枠超過時のエラー返却仕様が 2025-02-10 に変更されたばかりで、今後の制限変更の予測が困難
- D3: Workers の Edge ロケーションと D1 データベースのリージョン間レイテンシが 3 秒要件に影響するか — 影響範囲: D1 選択時のレスポンスタイム。D1 は特定リージョンに固定される — リスク: C3 により実ユーザーは開発者 1 名のため、D1 リージョンをユーザーのロケーション近くに設定すれば軽減可能。深刻度は低い

## 判断基準

優先度は決定者（人間）が設定する。

- **OBJ-1: データアクセスの応答性能** — データベースへのアクセスが、画面表示 3 秒以内の要件を安定して満たせること。評価の視点: Cloudflare Workers（V8 Isolate）からの接続方式における接続オーバーヘッドの低さ（Binding vs HTTP API vs TCP）、データベースのリージョン・エッジとの物理的な距離・レイテンシ特性、コネクションプールなし環境（毎リクエスト接続確立）での現実的な応答時間。
- **OBJ-2: 認証機能の統合性** — ユーザー認証を低コストで実現できること。DBサービス自体が認証機能を提供しているか、またはWorkers側での認証実装とDBの統合がスムーズか。評価の視点: DBサービスが提供する組み込み認証機能の有無・成熟度（Supabase Auth 等）、認証を自前実装する場合のセッション/トークン管理ストレージとしての適性、認証実装の追加工数・複雑さへの影響。
- **OBJ-3: バックアップ・リストアの実行可能性** — データの損失リスクに対して、現実的な手順でバックアップを取得・リストアできること。評価の視点: エクスポート・インポート機能の有無と対応フォーマット、バックアップ手順の複雑さ（スクリプト化・自動化の容易さ）、リストア手順の確実性（FTS インデックス等の再構築要否を含む）。
- **OBJ-4: 運用負荷の最小化** — シングルユーザーの個人開発として、データベースのセットアップ・維持管理に要する労力が最小であること。評価の視点: 初期セットアップの手順数・複雑さ、Cloudflare Workers との統合の容易さ（公式サポート・ドキュメントの充実度）、スキーマ変更・データ操作の手軽さ（CLI・管理ツールの存在）、障害・トラブル発生時の対処のしやすさ。
- **OBJ-5: 長期的なコスト安定性** — 無料ティア内での継続利用が現実的に維持できること。評価の視点: 現在の無料プランの制限内容（ストレージ上限・リクエスト数・DB数）、無料プランの持続可能性（サービスの安定性・料金変更履歴）、無料枠を超過した場合のコスト増加の予測可能性。

## 検討した選択肢

### ALT-1: Cloudflare D1

Cloudflare のエッジ SQLite データベース。Workers との同一プラットフォーム統合。SQL クエリ・LIKE 検索・FTS 対応。無料枠: 500 MB/DB、読み取り 500 万回/日、書き込み 10 万回/日。Binding 接続により接続オーバーヘッドが構造的に最小。ユーザー提供案。

### ALT-2: Supabase

PostgreSQL ベースの BaaS。RESTful API・リアルタイム機能・認証機能（Supabase Auth）を統合提供。無料枠: 500 MB ストレージ、5 万 MAU。外部サービスとして Workers から HTTP 接続。ユーザー提供案。

### ALT-3: TiDB Serverless

MySQL 互換の分散 NewSQL データベース。サーバーレス構成で自動スケーリング。無料枠: 5 GiB ストレージ、5,000 万 Request Units/月。外部サービスとして Workers から HTTP/TCP 接続。ユーザー提供案。

### ALT-4: Turso (libSQL)

SQLite 互換の分散エッジデータベース。Cloudflare 公式パートナー。`@libsql/client/web` HTTP ドライバーで Workers から接続可能。エッジレプリカによる低レイテンシ。無料枠: 5 GB、5 億ロウ読み取り/月。評価エージェント提案（OBJ-1, OBJ-5 の観点）。

### 除外した選択肢

なし

### ALT-1: Cloudflare D1 の詳細

**OBJ-1（◎）**: Binding 接続は接続確立コストが実質ゼロ。Workers と同一エッジインフラで動作しネットワークレイテンシが構造的に最小。欠点: D1 リージョンは作成時に固定で Workers Edge との物理距離が離れる場合あり（D3 依存）。

**OBJ-2（△）**: 認証機能なし。JWT 発行・検証・パスワードハッシュ等を完全自前実装。セッション管理に Workers KV 等の追加実装が必要。C3（シングルユーザー）が崩れ複数ユーザー拡張時に実装工数が大幅増加。

**OBJ-3（△）**: `wrangler d1 export` で SQL ダンプ取得可能。SQLite 形式でローカル確認が容易。欠点: FTS5 仮想テーブルを含む DB はエクスポート非サポート（D1 依存）。リストア後の FTS5 インデックス再構築手順が必要。S2（LIKE 検索で充足するか）次第で評価が変わる。

**OBJ-4（◎）**: 同一プラットフォーム統合で管理対象追加ゼロ。`wrangler` CLI 一元管理。Binding 方式で環境変数不要。欠点: GUI ツールの充実度は Supabase に劣る。FTS5 使用時のバックアップ手順整備が追加負荷（S4 依存）。

**OBJ-5（○）**: Free プラン 500 MB/DB は個人用途で十分。超過時は従量課金で予測可能。Cloudflare は大企業でインフラとしての安定性が高い。欠点: 2025-02-10 に無料枠超過挙動が変更されており、制限変更の予見性がない（S4、D2 依存）。

### ALT-2: Supabase の詳細

**OBJ-1（△）**: リージョン選択可能（東京等）。PostgreSQL の高度なクエリ最適化。欠点: 外部 HTTP API 呼び出しで毎リクエスト TCP+TLS のオーバーヘッド。D1 Binding より構造的に不利。3 秒要件超過リスクは低いが、接続方式として構造的に D1 より劣る（S6 依存: 複雑クエリが必要になれば PostgreSQL の優位が発揮される可能性）。

**OBJ-2（◎）**: Supabase Auth（GoTrue ベース）がメール/パスワード・ソーシャルログイン・マジックリンクを即座に提供。JWT 管理自動化。RLS との統合。欠点: 外部サービス依存による認証リクエストのレイテンシ。`@supabase/supabase-js` の Workers 互換性確認要。

**OBJ-3（○）**: PostgreSQL ベースで FTS 特有のエクスポート制約なし。`pg_dump` 互換。REST API 経由の JSON エクスポートも可能。欠点: 無料プランで自動バックアップ・PITR なし。手動操作はローカル環境から（S3 依存）。

**OBJ-4（○）**: 充実した GUI ダッシュボード。Auth 統合で認証運用が簡略化。Supabase CLI でローカル開発可能。欠点: 環境変数 2 つ（URL, ANON_KEY）の管理。外部アカウント管理。Free プランの非アクティブプロジェクト自動停止仕様。

**OBJ-5（△）**: 500 MB ストレージ・5 万 MAU。シングルユーザーでは MAU は問題なし。欠点: 非アクティブプロジェクト自動停止仕様。VC 資金調達型スタートアップで料金変更リスクが相対的に高い。有料移行時 $25/月。

### ALT-3: TiDB Serverless の詳細

**OBJ-1（△）**: MySQL 互換で幅広い SQL 機能。HTTP ドライバー経由で Workers 接続可能。欠点: 外部 HTTP 接続のオーバーヘッド。低頻度個人利用でのコールドスタート問題。分散アーキテクチャの恩恵を受けにくい規模（S1 依存）。

**OBJ-2（△）**: MySQL 互換でユーザーテーブル作成は技術的に可能。欠点: 認証機能なし。Workers との統合エコシステムが未整備で自前実装 + 調査コスト（S6 依存）。

**OBJ-3（△）**: MySQL 互換で `mysqldump` 相当の操作が原理的に可能。欠点: Serverless 構成の接続制約により標準ツールでの直接ダンプの実行可能性が未確認。無料プランのバックアップ機能詳細が不明（Doubt 相当）。

**OBJ-4（△）**: Serverless でインフラ管理は自動化。MySQL ツールチェーンが利用可能。欠点: Workers との公式統合が薄い。接続設定が手動。分散 SQL の管理機能が個人用途にはオーバースペック（C1 依存）。

**OBJ-5（○）**: 無料枠 5 GiB は D1 の 10 倍。RU ベース課金で軽量アクセスでは超過しにくい。欠点: RU 単位の粒度が不明確で消費量の予測が難しい。日本市場での認知が少なく変更の予兆を察知しにくい。

### ALT-4: Turso (libSQL) の詳細

**OBJ-1（○）**: エッジレプリカによる自動ルーティングで Workers 起動ロケーションに最近接レプリカへ接続。物理的レイテンシを構造的に最小化。欠点: 外部 HTTP API 呼び出しで TCP+TLS オーバーヘッドが毎リクエスト発生。C3（ユーザー 1 名・ロケーション固定）ではエッジ分散の恩恵が限定的（S5 依存）。

**OBJ-2（△）**: SQLite 互換で認証関連テーブルを通常の SQL で扱える。欠点: 認証機能なし。完全自前実装が必要。D1・TiDB と同様に認証は自前実装。Supabase Auth と比較して明確に劣る（C3、S5 依存）。

**OBJ-3（△）**: SQLite 互換で `turso db shell` + `.dump` による SQL ダンプが原理的に可能。欠点: バックアップ手順の公式ドキュメントが薄い。Workers からの直接バックアップ不可。無料プランの自動バックアップ有無が不明確（S3、S5 依存）。

**OBJ-4（○）**: Cloudflare Integrations Marketplace 公式パートナー。環境変数自動設定導線あり。Drizzle ORM が libSQL サポート。欠点: wrangler + turso CLI の 2 ツール管理。認証情報 2 つの管理。初期セットアップが複数ステップ（S5 依存）。

**OBJ-5（○）**: 無料枠 5 GB・5 億ロウ読み取り/月で個人用途では長期超過しにくい。有料移行時 $4.99/月と低コスト。欠点: 独立スタートアップで事業継続リスクが相対的に高い。過去に無料枠条件の見直しあり（S5 依存）。

## 決定

ALT-1: Cloudflare D1 を採用する。

## 根拠

MVPではCloudflareのインフラ統一を重視した。ADR-0001 で採用した Cloudflare Workers + Pages と同一プラットフォーム上にデータベースも統合することで、OBJ-4（運用負荷の最小化 ◎）と OBJ-1（応答性能 ◎）を同時に達成する。個人利用のためバックアップ（OBJ-3）はそこまで重視しておらず、認証の自前実装（OBJ-2 △）は受け入れる。

## 受け入れたトレードオフ

(1) 認証機能の統合性（OBJ-2 △）について、D1 には組み込み認証機能がなく、JWT 発行・検証・セッション管理を Workers 側で自前実装する必要がある。Supabase Auth のような成熟した認証基盤を利用できないが、シングルユーザー（C3）のMVPとして認証の複雑さは限定的であり、実装コストを受け入れる。

(2) バックアップ・リストアの実行可能性（OBJ-3 △）について、FTS5 仮想テーブルを使用した場合に `wrangler d1 export` が非サポートとなる制約（D1）を認識している。個人利用のため、バックアップの優先度は低く設定する。LIKE 検索で充足する場合は制約を回避でき、FTS5 を使用する場合もリストア手順の整備で対処可能と判断する。
