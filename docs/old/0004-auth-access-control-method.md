# ADR-0004: 認証・アクセス制御方式の選定

## ステータス

Proposed

## 日付

2026-02-21

## コンテキスト

シングルユーザーWebアプリの認証・アクセス制御として、どの認証方式を採用するかを決定する必要があった。

バックエンドは Cloudflare Workers（Hono）、フロントエンドは Cloudflare Pages にデプロイされた React SPA であり（ADR-0001・ADR-0002 より確定済み）、データベースは Cloudflare D1 で組み込み認証機能を持たない（ADR-0003 より確定済み）。インターネット公開された個人用ツールであるため、第三者の不正アクセスを遮断する認証が必要であり、かつ Web Share Target API からの POST リクエストとの互換性確保が技術的な主要制約となっている。

決定のスコープは以下のとおり:
- 対象: 認証方式（Cloudflare Access / 自前JWT / Basic認証等）の選定、セッション管理方式、Web Share Target API からの POST リクエストとの互換性
- 対象外: 認可（権限管理）の詳細設計（シングルユーザーのため不要）、アプリケーション層のビジネスロジック

可逆性は medium。認証方式はミドルウェアレベルで全エンドポイントに影響するが、シングルユーザーのためユーザーデータのマイグレーションは不要。Hono のミドルウェア差し替えで認証方式を変更可能。

## 関連する既存決定

- **ADR-0001: PWA+REST API構成のホスティング基盤選定** — Cloudflare Pages + Cloudflare Workers（Hono）を採用。認証方式の選択肢がCloudflare基盤に制約される。月額コスト0円、V8 Isolateランタイム、シングルユーザー構成、3秒表示要件が制約として確定した。
- **ADR-0002: フロントエンドフレームワーク選定** — React（Cloudflare Pages上の静的SPA）を採用。認証フロー（ログイン画面、セッション/トークンのクライアント管理）はReactで実装することが確定。Cloudflare Accessを採用する場合はフロントエンド側での認証処理が不要になる。
- **ADR-0003: データベース選定** — Cloudflare D1 を採用。D1 には組み込み認証機能がないため、「認証を自前実装する」ことをトレードオフとして受け入れている。ただしCloudflare Accessを採用する場合はWorkers側での認証実装は不要になりうる。

## 前提条件

### 確実（Certainty）

- C1: 有料の認証サービスは採用しない — 根拠: ADR-0001 C3 より確定済み。Auth0有料プラン・Supabase Pro等は選択不可。
- C2: バックエンドは Cloudflare Workers（V8 Isolate ランタイム）上の Hono で動作する — 根拠: ADR-0001 の決定内容より確定済み。Node.js専用ライブラリやOS依存の認証ライブラリは利用不可。
- C3: シングルユーザー構成であり、マルチユーザー向けの複雑な認可設計は不要 — 根拠: ADR-0001 C4 より確定済み。
- C4: フロントエンドは React（Cloudflare Pages にデプロイされた静的 SPA）であり、サーバーサイドレンダリングは行わない — 根拠: ADR-0002 より確定済み。
- C5: 画面表示 3 秒以内の要件が存在する — 根拠: ADR-0001 C5 より確定済み。認証処理のレイテンシも要件の対象。
- C6: Cloudflare D1 採用済みであり、D1 には組み込み認証機能がない — 根拠: ADR-0003 より確定済み。Cloudflare Access を採用する場合は Workers 側での認証実装は不要になりうる。
- C7: Web Share Target API の POST リクエストに Authorization ヘッダーはブラウザから自動付与されない — 根拠: MDN・Chrome for Developers ドキュメントより。Bearer JWT 認証方式は Web Share Target API との互換性に根本的な課題を持つ。
- C8: Web Share Target API の利用には PWA としてのインストールが前提となる — 根拠: Web Share Target API の仕様より。
- C9: HTTPS 通信が保証されている — 根拠: Cloudflare Pages および Workers はデフォルトで HTTPS を強制する。Cookie の Secure 属性・JWT の署名検証等の前提。

### 仮定（Supposition）

- S1: アプリを利用するのは作者本人のみであり、将来もマルチユーザー化しない — 崩れる可能性: ADR-0003 に「将来のマルチユーザー化」への言及あり。検証方法: プロダクトの将来計画を確認する。
- S2: iOS Safari を含む複数のブラウザ・OS 環境からの利用が想定される — 崩れる可能性: 利用端末が特定環境に限定される場合がある。検証方法: 利用端末の環境を確認する。
- S3: パスワード等のシークレットは Cloudflare Workers の Secrets 機能で安全に管理できる — 崩れる可能性: ローカル開発環境での誤コミットリスク。検証方法: Workers Secrets の設定・ローテーション方針を確認する。
- S4: Hono JWT ミドルウェアは継続的にメンテナンスされ、セキュリティ修正が適時提供される — 崩れる可能性: 過去に脆弱性報告あり（アルゴリズム混同・aud未検証等）。検証方法: GitHub Security Advisory の継続的モニタリング。
- S5: Cloudflare Access の料金プラン・機能（無料 50 ユーザー枠）は現状維持される — 崩れる可能性: Cloudflare のサービス条件は変更されうる。検証方法: Cloudflare の料金ページを定期的に確認する。
- S6: セッション管理のための Workers KV が必要に応じて利用可能（無料枠の範囲内で十分）— 崩れる可能性: Workers KV 無料枠は 100,000 読み取り/日・1,000 書き込み/日。検証方法: セッション発行・検証頻度が無料枠内に収まるか試算する。
- S7: 認証方式の変更はミドルウェア差し替えで対応可能（中可逆性）— 崩れる可能性: Cloudflare Access と自前実装の間の移行はインフラ層の変更を伴い、ミドルウェア差し替えのみでは完結しない場合がある。検証方法: 採用する方式と他方式間の移行パスを確認する。
- S8: このアプリはインターネット公開されており、作者本人以外のアクセスは不正アクセスとして拒否する必要がある — 崩れる可能性: 脅威モデルの認識が変わる可能性。検証方法: 脅威モデル（誤操作防止レベルか攻撃者対策か）を確認する。
- S9: 認証は全 API エンドポイントに一律の方式で適用する — 崩れる可能性: C7 の制約（Web Share Target POST で Authorization ヘッダー非自動付与）を踏まえると、エンドポイント別のハイブリッド構成も検討しうる。検証方法: エンドポイント別に異なる認証方式を許容するか確認する。

### 不確実（Doubt）

- D1: iOS Safari スタンドアロン PWA モードでの Cookie 共有問題が、このアプリの実際の使用シナリオで問題になるか — 影響範囲: iOS で Cookie ベースのセッション管理を利用する場合、初回認証フローで追加実装が必要になる可能性。リスク: Web Share Target API からの投稿が認証エラーで失敗する。
- D2: 自前 JWT 実装における設定ミスのリスクを開発者が許容できるか — 影響範囲: Hono JWT ミドルウェアの設定ミスがセキュリティ脆弱性に直結。リスク: 個人用途ゆえセキュリティレビューが省かれやすく、発見が遅れる可能性。
- D3: Web Share Target API 経由の POST リクエスト時のセッション Cookie の実際の動作（Cloudflare Workers + Hono 環境での動作未検証）— 影響範囲: Cookie の Secure/SameSite 属性・ドメインマッチ等がエッジ環境で想定通りに動作するか未検証。リスク: 互換性要件を満たせない場合がある。

## 判断基準

優先度は決定者（人間）が設定する。以下は定義のみを示す。

- **OBJ-1: Web Share Target API との互換性** — 認証方式が、ブラウザから Authorization ヘッダーを自動付与できない Web Share Target API の POST リクエストと共存できること。評価の視点: Web Share Target POST で認証エラーなく機能するか / 追加の実装コストなしに互換性を確保できるか / iOS Safari スタンドアロン PWA モードでの Cookie 挙動（D1・D3）に左右されないか。
- **OBJ-2: セキュリティ強度と設定ミスリスクの低さ** — インターネット公開された個人用ツールとして、第三者の不正アクセスを確実に遮断できること。かつ、開発者単独による実装・設定ミスでセキュリティホールが生まれにくいこと。評価の視点: 既知の攻撃ベクター（セッションハイジャック・トークン窃取・CSRF等）への耐性 / 設定ミスが致命的な脆弱性に直結するリスクの大きさ（D2）/ セキュリティ修正の適時提供・メンテナンス継続性（S4）。
- **OBJ-3: Cloudflare Workers 環境での実装・運用の簡潔さ** — V8 Isolate ランタイムの制約下で、認証ロジックの実装・デプロイ・長期運用が最小の負担で行えること。評価の視点: Node.js 非互換ライブラリへの依存なく実装できるか（C2）/ 認証基盤のセットアップ・設定変更・トラブルシューティングの煩雑さ / 外部サービス依存度（S5）。
- **OBJ-4: 認証処理のレイテンシの低さ** — 画面表示 3 秒以内の要件（C5）を満たすうえで、認証処理がボトルネックにならないこと。評価の視点: 認証処理が追加するネットワークラウンドトリップ数 / Cloudflare Workers のエッジで完結するか、外部サービスへの通信が必要か / セッション検証のコスト（Workers KV 参照回数など）。
- **OBJ-5: コスト持続可能性** — 個人用ツールとして、認証基盤の運用コストが長期的に無料または最小限であること。評価の視点: 採用する認証基盤の無料枠と想定使用量のマージン（S5・S6）/ 無料枠の変更・廃止リスク（S5）/ 有料プランへの強制移行が生じた場合の代替手段の有無。

## 検討した選択肢

### ALT-1: Cloudflare Access（Zero Trust エッジ認証）

Cloudflare Zero Trust の認証レイヤーを Workers の前段に配置。IdP（GitHub・Google等）経由で認証し、cf_authorization Cookie で透過的にセッション管理。Workers 側に認証コードを書かない。OBJ-1（Web Share Target互換性）、OBJ-2（セキュリティ強度）、OBJ-3（実装簡潔さ）の観点から最有力として提案された。

### ALT-2: Cookie セッション + Workers KV（ステートフル）

Hono カスタムミドルウェアでセッション管理を自前実装。セッションIDを HttpOnly/Secure/SameSite=Strict Cookie で発行し、Workers KV にセッション情報を保存。即時ログアウト・セッション無効化が可能。OBJ-1（Cookie互換性）、OBJ-2（Cookie属性によるセキュリティ）、OBJ-3（Workers完結型実装）の観点から提案された。

### ALT-3: 署名付き Cookie / JWT in Cookie（ステートレス）

JWT を Cookie として保持し、Hono JWT ミドルウェアで Workers 内完結の署名検証を行うステートレス方式。Workers KV 不要。Workers Secrets にシークレットを保存。OBJ-4（最小レイテンシ）、OBJ-5（コスト持続可能性）の観点から最有力として提案された。

### 除外した選択肢

- JWT Bearer 認証（Authorization ヘッダー）: C7（Web Share Target POST で Authorization ヘッダー非自動付与）に直接抵触。OBJ-1 で × 評価。
- Basic 認証: C7 と同様の問題（Authorization ヘッダー非互換）。OBJ-1 で ×、OBJ-2 で × 評価。業界非推奨。
- エンドポイント別ハイブリッド構成: 実装複雑性が高く、他の方式（ALT-1〜3）で Web Share Target 互換性を確保可能。OBJ-1 で △ 評価。

## トレードオフ分析

| 判断基準 | ALT-1: Cloudflare Access | ALT-2: Cookie+KV | ALT-3: JWT in Cookie |
|----------|--------------------------|-------------------|----------------------|
| OBJ-1: Web Share Target互換性 | ◎ | ○ | ○ |
| OBJ-2: セキュリティ強度 | ◎ | ○ | △ |
| OBJ-3: 実装・運用簡潔さ | ◎ | ○ | ○ |
| OBJ-4: レイテンシ | ○ | △ | ◎ |
| OBJ-5: コスト持続可能性 | ○ | ○ | ◎ |

### 各セルの詳細

**ALT-1 × OBJ-1（◎）**
- 利点: cf_authorization Cookie がブラウザに保持され、Web Share Target POST 時にも自動付与される。Workers 側での認証処理が不要で透過的に互換性確保。
- 欠点: D1（iOS Safari PWA モードでの Cookie 共有問題）の影響を受ける可能性。cf_authorization Cookie がスタンドアロン/ブラウザ間で共有されるかは未検証。
- CSD依存: D1・D3 の影響を受ける可能性がある（未検証）。

**ALT-1 × OBJ-2（◎）**
- 利点: 認証ロジックの自前実装がゼロ。設定ミスの表面積が最小。Cloudflare による DDoS 耐性・ブルートフォース対策・レート制限が自動提供。IdP 経由の MFA 利用可能。
- 欠点: Cloudflare Access のポリシー設定ミスや、Workers 側での CF-Access-Jwt-Assertion 検証省略のリスクは残る。

**ALT-1 × OBJ-3（◎）**
- 利点: Workers 側に認証コードを書かない。ダッシュボードで認証ポリシー管理。セキュリティパッチは Cloudflare 側が担当。V8 Isolate の制約に非依存。
- 欠点: 初期セットアップ（テナント・アプリ登録・ポリシー設定）の学習コスト。サービス仕様変更時のダッシュボード設定見直し。

**ALT-1 × OBJ-4（○）**
- 利点: 認証処理はエッジで Workers 呼び出し前に完結。セッション確立後の通常リクエストのレイテンシ影響は小さい。
- 欠点: Workers 内での CF-Access-JWT-Assertion の署名検証に JWKS 公開鍵取得が必要（初回/再検証時に外部 I/O の可能性）。

**ALT-1 × OBJ-5（○）**
- 利点: 現時点ではシングルユーザーで無料枠（50 ユーザー）内。追加の KV・D1 書き込み不要。
- 欠点: S5（Cloudflare Access 無料枠の現状維持）に強く依存。51 ユーザー目から全席分 $7/席/月の急激な課金構造。Zero Trust サービスの個人向け無料枠の長期継続性に不確実性。
- CSD依存: S5 の変化が評価を直接悪化させる。

**ALT-2 × OBJ-1（○）**
- 利点: セッション Cookie が Web Share Target POST 時にブラウザから自動付与される。C7 の制約をクリア。
- 欠点: D1・D3 の影響を受ける。Cookie の SameSite/Secure 属性設定次第で POST が失敗する可能性。
- CSD依存: D1・D3 に依存。

**ALT-2 × OBJ-2（○）**
- 利点: HttpOnly/Secure/SameSite=Strict Cookie で XSS・CSRF リスクを低減。サーバーサイドセッションで即時無効化可能。
- 欠点: Cookie 属性の設定ミスが脆弱性に直結。パスワードハッシュ化アルゴリズムの V8 Isolate 互換性検証が必要。開発者のセキュリティ知識に依存（D2）。

**ALT-2 × OBJ-3（○）**
- 利点: Workers 内完結。外部サービス依存なし。コードベースで動作を把握・変更可能。
- 欠点: セッション ID 生成・保存・検証・失効のフル実装が必要。KV 読み書きコードとライフサイクル管理の保守負担。

**ALT-2 × OBJ-4（△）**
- 利点: Cookie 送受信はブラウザが自動処理。セッション無効化が即座に反映。
- 欠点: リクエストごとに Workers KV への読み取りラウンドトリップが発生。KV の読み取りレイテンシは通常 1〜数十ms だがコールドキャッシュ時に増加しうる。C5（3秒要件）への余裕が少ない。

**ALT-2 × OBJ-5（○）**
- 利点: Workers KV 無料枠（100,000 読み取り/日・1,000 書き込み/日）でシングルユーザー用途は十分。
- 欠点: S6（KV 無料枠継続）に依存。KV の読み書きを認証で消費するため、アプリの他用途への枠が減少。
- CSD依存: S6 に依存。

**ALT-3 × OBJ-1（○）**
- 利点: JWT を Cookie として保持するため、Web Share Target POST 時にブラウザから自動付与される。C7 の制約をクリア。
- 欠点: D1・D3 の影響を受ける。

**ALT-3 × OBJ-2（△）**
- 利点: Cookie（HttpOnly/Secure）で JWT を配信するため、localStorage 保存と比べ XSS による窃取リスクは低い。ステートレスで KV 依存がなく攻撃面が限定的。
- 欠点: Hono JWT ミドルウェアの過去の脆弱性（アルゴリズム混同・aud 未検証）への対応が必要（S4 依存）。アルゴリズム明示指定・aud 検証等の設定ミスリスクが高い（D2）。ステートレスのため発行済みトークンの即時無効化が困難。
- CSD依存: S4・D2 に依存。

**ALT-3 × OBJ-3（○）**
- 利点: Hono 公式 JWT ミドルウェア利用で実装量が少ない。KV 不要のステートレス設計でアーキテクチャがシンプル。
- 欠点: セキュリティ設定（アルゴリズム・aud 検証・シークレット管理）の自前設定が必要。OBJ-1 対応のため Cookie 配信にする追加実装が必要。

**ALT-3 × OBJ-4（◎）**
- 利点: JWT 署名検証が Workers 内（V8 Isolate）で完結。外部ネットワークラウンドトリップなし。HMAC-SHA256 検証は数マイクロ秒〜数ミリ秒。ステートレスで KV 参照不要。
- 欠点: トークン失効にブラックリスト管理が必要な場合、KV 参照が追加されステートレスの利点が薄れる。

**ALT-3 × OBJ-5（◎）**
- 利点: Workers Secrets（無料）のみ使用。追加サービス・ストレージ依存ゼロ。KV 無料枠を消費しない。料金変更リスクなし。
- 欠点: トークン即時無効化のために KV を使う場合は S6 依存が生じる。

## リスク分析

### ALT-1: Cloudflare Access（Zero Trust エッジ認証）

**失敗シナリオ 1: Cloudflare Access 無料枠の突然の廃止または条件変更**
- 原因: S5（Cloudflare Access の料金プラン・機能の現状維持）の崩壊。代替の出口戦略を準備しなかった場合。
- 深刻度: high / 発生確率: low
- 軽減策: Cloudflare の料金ページを四半期ごとにモニタリングする。ALT-2 または ALT-3 への移行パスをあらかじめ設計し、Workers 側に認証ロジックを移植する際の手順書を作成しておく。

**失敗シナリオ 2: iOS Safari PWA モードでの認証フロー断絶**
- 原因: D1（iOS Safari PWA モードでの Cookie 共有問題）を検証せずに進めた場合。Cloudflare Access の認証フロー（リダイレクトベース）が iOS のスタンドアロン PWA と正しく動作しない。
- 深刻度: high / 発生確率: medium
- 軽減策: 開発初期段階で iOS Safari PWA モードと Cloudflare Access の認証リダイレクトの挙動を実機検証する。

**失敗シナリオ 3: Workers 側での CF-Access-Jwt-Assertion 検証省略によるバイパス**
- 原因: 「Cloudflare Access を設定すれば安全」という誤った前提で Workers 側の JWT 検証実装を省いた場合。
- 深刻度: high / 発生確率: medium
- 軽減策: Workers 側で CF-Access-Jwt-Assertion ヘッダーの署名検証を必ず実装する。Access ポリシーの設定と Workers の検証は「二重防御」として両方必須とする。

**失敗シナリオ 4: Cloudflare Access の仕様変更・API 互換性の破壊**
- 原因: S5（Cloudflare Access の料金プラン・機能の現状維持）の部分的崩壊。外部マネージドサービスの仕様変更に対する追跡コストを過小評価した場合。
- 深刻度: medium / 発生確率: low
- 軽減策: Cloudflare の変更履歴（ブログ・Changelog・GitHub Issues）を定期的にモニタリングする。Workers 側の JWT 検証ロジックを抽象化し、公開鍵取得元・検証アルゴリズムを設定で切り替えられるようにしておく。

### ALT-3: 署名付き Cookie / JWT in Cookie（ステートレス）（参考：評価時の比較対象として分析）

**失敗シナリオ 1: JWT 設定ミスによるセキュリティ脆弱性の放置**
- 原因: D2（自前 JWT 設定ミスリスク）の顕在化。alg を明示指定せずデフォルト値のまま使用した場合、アルゴリズム混同攻撃によるバイパスのリスク。
- 深刻度: high / 発生確率: medium

**失敗シナリオ 2: ステートレス方式によるセッション無効化不能による被害拡大**
- 原因: デバイス紛失・認証情報漏洩時に発行済み JWT を即時無効化できない。
- 深刻度: high / 発生確率: low

## 将来の意思決定への影響

### LD-1: マルチユーザー化・チーム共有機能の追加（想定時期: 1〜2年後）

- ALT-1: 選択肢が広がる — Cloudflare Access はユーザー数・グループポリシーの追加がダッシュボード設定のみで完結。ただし無料枠（50ユーザー）を超えると $7/席/月の課金が発生。
- ALT-2: 中立 — ユーザーモデルの拡張が必要になるが、認証方式自体の変更は最小限。
- ALT-3: 選択肢が狭まる — ユーザー追加・権限変更が即時反映されない問題が表面化しやすく、KV によるトークン無効化リストの導入が必要になる可能性が高い。

### LD-2: マイクロサービス化・複数 Workers サービスにまたがる API 分離（想定時期: 1〜3年後）

- ALT-1: 選択肢が広がる — 複数のアプリケーション・Workers サービスに対して同一のポリシーを適用可能。
- ALT-2: 中立 — Workers 間でセッション情報を共有する場合は KV の設計変更が必要。
- ALT-3: 選択肢が広がる — ステートレス JWT は複数サービス間でトークン検証の共有鍵さえ配布すれば対応可能。ただしシークレット管理の設計が必要。

### LD-3: モバイルアプリ・ネイティブクライアントへの対応（想定時期: 1〜2年後）

- ALT-1: 選択肢が狭まる — ネイティブアプリからは Cloudflare Access の認証フロー（ブラウザリダイレクト）を利用できない。移行コストが最も高い。
- ALT-2: 中立 — API トークン（長期セッションID）の発行を追加実装することで対応可能。
- ALT-3: 選択肢が広がる — JWT はネイティブアプリとの相性が良い。Cookie から Authorization ヘッダーへの切り替えが容易。

### LD-4: CI/CD・ステージング環境の整備（想定時期: 6ヶ月〜1年後）

- ALT-1: 選択肢が狭まる — ステージング環境ごとに別の Cloudflare Access アプリケーションを登録する必要がある。自動化テストでの認証回避が困難な場合がある。
- ALT-2: 中立 — 環境ごとに KV 名前空間を分けることで独立したセッション管理が可能。自前実装のため環境ごとの設定変更がコードレベルで管理できる。
- ALT-3: 選択肢が広がる — テスト用のJWTをシークレットを使って生成・検証できるため、E2Eテストやステージング環境での認証回避が最も容易。

### LD-5: アクセスログ・監査証跡の要件発生（想定時期: 1〜3年後）

- ALT-1: 選択肢が広がる — Cloudflare Access は認証ログ・アクセスログを自動でダッシュボードに記録。Logpush で外部SIEMへの転送も可能。
- ALT-2: 中立 — セッションIDに紐づいたアクセス記録を Workers のログに出力する実装を追加することで監査証跡を実現可能。
- ALT-3: 選択肢が狭まる — ステートレスJWTはサーバーサイドに状態を持たないため、「誰がいつ何にアクセスしたか」の記録を取るには Workers 側での明示的なログ実装が必要。

### 可逆性評価

| 代替案 | 可逆性 | 理由 |
|--------|--------|------|
| ALT-1: Cloudflare Access | 部分可逆 | Workers側の認証コードはゼロのため既存コードへの影響は最小限。ただしダッシュボード設定・IdP連携・ポリシー設定は別途移行作業が必要。移行コスト: 中。 |
| ALT-2: Cookie セッション + Workers KV | 部分可逆 | Hono ミドルウェアの差し替えで変更可能。KV のセッションデータはシングルユーザーのため無効化のみで良い。移行コスト: 低〜中。 |
| ALT-3: 署名付き Cookie / JWT in Cookie | 可逆 | Hono JWT ミドルウェアの差し替えのみで移行可能。ステートレスのためサーバーサイドのデータマイグレーション不要。移行コスト: 低。 |

## 決定

ALT-1: Cloudflare Access（Zero Trust エッジ認証）を採用する。

## 根拠

自前実装が不要になる簡易さを重視した。OBJ-3（実装・運用の簡潔さ）を最優先とし、認証ロジックを Workers 側に書かないことでセキュリティリスク（OBJ-2）も同時に低減できる点を評価した。

## 受け入れたトレードオフ

S5（Cloudflare Access 無料枠の現状維持）への依存を受け入れる。無料枠に収まらなくなった場合は別の方式（ALT-2 または ALT-3）に切り替える出口戦略を想定。OBJ-4（レイテンシ）では◎ではなく○だが、3秒要件に対して十分な余裕があると判断。
