# AIワークフロー設計の知見 — 実装パターン集

[ai-workflow-knowledge.md](ai-workflow-knowledge.md) の各原則を実装する際の具体的なパターン。

---

## §1. 定義フェーズ

### パターン: 共通語彙ファイルの確立

ワークフロー冒頭で、主要概念の定義・スコープ・前提条件を構造化したファイルを生成し、後続の全ステップがこのファイルを入力として参照する。

```
Phase 0:
  入力: ユーザーの要求テキスト
  処理: 以下を定義してファイルに保存
    - 対象スコープ（何が対象で、何が対象外か）
    - 主要概念の定義と分類基準
    - 前提条件と制約
  出力: definitions.md

Phase 1+:
  入力: definitions.md + 各フェーズ固有の入力
  （definitions.md を全サブエージェントの共通入力に含める）
```

**ポイント**: 定義ファイルは「後続フェーズの入力仕様の一部」として組み込む。「参照してください」という指示ではなく、サブエージェントへの入力として渡すことで参照漏れを防止する。

## §2. 外部基準による接地

### パターン: フレームワーク準拠のフェーズ構成

ワークフローのフェーズ構成自体を確立されたフレームワークに準拠させることで、検討観点の網羅性を構造的に担保する。

```
例: 意思決定ワークフロー → PrOACT フレームワーク
  Phase 1: Problem（問題定義）
  Phase 2: Objectives（判断基準の定義）
  Phase 3: Alternatives（代替案の生成）
  Phase 4: Consequences（各案の評価）
  Phase 5: Tradeoffs（トレードオフの明示）

例: 設計レビューワークフロー → ISO 25010 品質特性
  各品質特性（セキュリティ、性能、保守性、...）を
  独立した評価エージェントの視点として割り当てる
```

**ポイント**: フレームワークは「チェックリスト」ではなく「フェーズ構造」として使う。チェックリスト方式は項目の存在確認に留まるが、フェーズ構造にすると各観点での深い分析が行われる。

## §3. ドメイン知識の事前整理

### パターン: 知見蓄積ファイルの参照

反復ワークフローでは、過去のイテレーションで判明した有効/無効な変更を知見ファイルに蓄積し、次のイテレーションの生成ステップで入力として渡す。

```
知見ファイルの構造例:
  ## 有効だった変更
  | 変更内容 | 効果(pt) | 対象カテゴリ | ラウンド |
  |---------|----------|------------|---------|
  | 具体例の追加 | +2.3 | 検出漏れ | R2 |

  ## 無効だった変更
  | 変更内容 | 効果(pt) | 対象カテゴリ | ラウンド |
  |---------|----------|------------|---------|
  | 網羅的列挙の指示 | -0.5 | 全体 | R1 |

生成ステップへの入力:
  - 現在の成果物
  - エラー分析結果
  - 知見ファイル（有効/無効な変更の履歴）
```

**ポイント**: 知見は自然言語の感想ではなく、変更内容・定量効果・対象の3つ組で記録する。これにより「何が効いたか」の判断が曖昧にならない。

## §4. 構造的分離

### パターン: 生成者と評価者のコンテキスト分離

成果物を生成するエージェントと、その品質を評価するエージェントを別サブエージェントとして起動し、評価エージェントには生成プロセスの情報を渡さない。

```
生成エージェント:
  入力: 要件 + 改善指示
  出力: 成果物（ファイルに保存）
  → 改善指示のコンテキスト（何を意図して変更したか）は評価エージェントに渡さない

評価エージェント:
  入力: 成果物 + 評価基準（ファイルから読み込み）
  出力: 評価結果（ファイルに保存）
  → 成果物が「何を改善しようとしたか」を知らずに評価する
```

**ポイント**: 「改善意図」を評価エージェントに渡すと、意図に沿った甘い評価になる。評価エージェントは「成果物が基準を満たしているか」だけを判断すべき。

### パターン: 批評の並列独立実行

1つの成果物に対して複数の批評エージェントを並列起動し、それぞれ異なる観点に集中させる。各批評エージェントは他の批評結果を見ない。

```
成果物 → 並列に4つのサブエージェントを起動:
  批評1: 有効性（この定義で目的を達成できるか）
  批評2: 完全性（欠落している観点はないか）
  批評3: 明確性（曖昧な記述はないか）
  批評4: 汎用性（特定ケースに過適合していないか）

各批評エージェントの入力: 成果物のみ（他の批評結果を含めない）
```

**ポイント**: 批評を直列に行うと、先行する批評にアンカリングされる。並列実行 + 結果の事後統合により、独立した多角的批評が得られる。

## §5. 入力記述量の偏り対策

### パターン: 視点別サブエージェント分離

入力文書全体を1つのエージェントに渡すのではなく、視点（カテゴリ）ごとに独立したサブエージェントを起動し、各エージェントは担当視点のレンズで入力を分析する。

```
入力文書（セキュリティの記述が多く、運用の記述が少ない場合）:

× 1エージェントに全視点を処理させる
  → セキュリティ関連の出力が過剰、運用関連が不足

○ 視点ごとにサブエージェントを分離
  エージェント1（セキュリティ視点）: 入力文書 + 「セキュリティの観点で分析せよ」
  エージェント2（運用視点）: 入力文書 + 「運用の観点で分析せよ」
  エージェント3（保守性視点）: 入力文書 + 「保守性の観点で分析せよ」
  → 各エージェントは担当視点に集中するため、記述量の偏りに引きずられにくい
```

**ポイント**: 視点ごとに最小出力件数を設定すると効果が高まる。「最低3件」のような制約により、記述が薄い領域でも一定の深さが確保される。

## §6. 自己参照的基準の排除

### パターン: 事前定義の正解キーによる評価

「良い出力か」を出力自身の性質で判断するのではなく、事前に定義した正解キー（期待される検出項目のリスト）との照合で評価する。

```
× 自己参照的な評価:
  「重要な問題を見落としていないか」→ 見落としている問題は評価者にも見えない

○ 外部基準による評価:
  1. 事前にテスト文書を作成し、意図的に問題を埋め込む
  2. 各問題に ID・カテゴリ・難易度を付与した正解キーを作成
  3. エージェント出力を正解キーと照合し、○（検出）/△（部分検出）/×（未検出）を判定

正解キーの構造例:
  | ID | カテゴリ | 難易度 | 問題の説明 | 埋め込み箇所 |
  |----|---------|--------|-----------|------------|
  | P1 | セキュリティ | 高 | SQLインジェクション脆弱性 | §3.2 |
  | P2 | 性能 | 中 | N+1クエリ | §4.1 |
```

**ポイント**: 「重要度が高い順に選定」「見落としやすいものを優先」のような基準は自己参照的。「外部規格の品質特性に対応する順」「入力文書への紐付きが明確な順」のように、評価対象から独立した基準を使う。

## §7. 方針と具体化の分離

### パターン: 2段階フェーズ構成

前半フェーズで「何を重視するか」（方針）を確定させ、後半フェーズで「具体的にどうするか」（手順・代替案）を詳細化する。

```
方針フェーズ（先に確定）:
  Phase 1: 問題のスコープと制約を定義
  Phase 2: 判断基準（何を重視するか）を定義
  → ユーザー確認を挟み、方針レベルで合意を形成

具体化フェーズ（方針確定後に実行）:
  Phase 3: 方針に基づいて代替案を生成
  Phase 4: 判断基準に基づいて各案を評価
  → 評価基準が後から変動しない
```

**ポイント**: 方針と具体化を同時に行うと、「具体的な案が先に出て、それに合う方針を後付けする」逆転が起きる。方針フェーズでユーザー確認を挟むことで逆転を防止する。

## §8. 対立討論

### パターン: 提案者 vs レッドチーム

一方のエージェントが分類・提案を行い、別のエージェントがその弱点を積極的に攻撃する対立構造。

```
Phase 1a — ファシリテーター:
  入力: 要件・制約
  出力: 分類結果（ファイルに保存）

Phase 1b — レッドチーム:
  入力: 分類結果のみ（ファシリテーターのコンテキストなし）
  指示: 「以下の分類の弱点を指摘せよ。特に、確実と分類されているが
         実は仮定に過ぎない項目、見落とされている観点を探せ」
  出力: 再分類提案（ファイルに保存）

Phase 1c — 統合（親エージェント）:
  入力: 分類結果 + 再分類提案
  処理: レッドチームの指摘を反映して最終分類を生成
```

**ポイント**: 反論ラウンドは対称にする。片方だけにレッドチームを当てると評価が不公平になる。複数案の比較では、各案に対して同じ構造の攻撃を行う。

## §11. シナリオ走査

### パターン: 正解キー付きテスト文書

手順の検証のために、意図的に問題を埋め込んだテスト入力を作成し、手順通りに処理した結果を正解キーと照合する。

```
テスト文書の構成:
  - 対象ドメインのリアルな文書として自然に読める
  - 複数カテゴリ・複数難易度の問題が埋め込まれている
  - 正解キーに各問題の位置・カテゴリ・期待される検出内容が定義されている

検証の実行:
  1. テスト文書をワークフローの入力として処理する
  2. 出力を正解キーと照合し、各問題の検出状況を記録
  3. 複数回実行して再現性を確認する（同じ入力で結果がばらつく箇所 = 手順の曖昧な箇所）
```

**ポイント**: テスト文書は異なるドメインで2つ以上作成する。1つだけだと特定ドメインへの過適合が検出できない。複数回実行（同一入力・同一手順）で結果がばらつく箇所は、手順が一意に定まっていない証拠。

## §12. 多観点検証の組み合わせ

### パターン: 検証フェーズの多層化

異なる検出対象を持つ検証手段を複数のフェーズとして組み合わせる。各検証は異なるエージェントで独立に実行する。

```
検証レイヤー1 — 前提の検証:
  入力の前提条件を「確実/仮定/不確実」に分類する

検証レイヤー2 — 各視点からの独立評価:
  判断基準ごとに独立したエージェントが評価する

検証レイヤー3 — 失敗シナリオ分析:
  「この案が失敗するとしたら何が原因か」を逆方向から分析する

検証レイヤー4 — 将来影響分析:
  「この決定が将来の選択肢をどう制約するか」を分析する
```

**ポイント**: 各レイヤーは異なるカテゴリの問題を検出する。レイヤー1は前提の誤り、レイヤー2は評価の偏り、レイヤー3は想定外のリスク、レイヤー4は長期的な制約。成果物の重要度に応じてレイヤー数を調整する。

## §13. 前提の検証

### パターン: 前提の確実性分類

議論の前提を「確実（事実として確認済み）」「仮定（妥当だが未確認）」「不確実（検証が必要）」に3分類し、不確実の件数に応じて追加検証の深さを調整する。

```
分類例:
  確実: 「PostgreSQL 15を使用する」（技術選定済み）
  仮定: 「月間アクセス数は10万PV程度」（推定値）
  不確実: 「外部APIの応答時間は100ms以内」（未計測）

不確実が3件以上 → 追加の失敗シナリオ分析を推奨
不確実が1-2件 → 該当項目の検証のみ
```

## §14. カバレッジ充実度

### パターン: カテゴリ別検出率の追跡と回帰検出

総スコアだけでなくカテゴリ別の検出率を追跡し、特定カテゴリの劣化（回帰）にペナルティを課す。

```
評価結果の追跡:
  | カテゴリ | ベースライン検出率 | 今回の検出率 | 差分 | 回帰判定 |
  |---------|-----------------|------------|------|---------|
  | セキュリティ | 0.80 | 0.90 | +0.10 | - |
  | 性能 | 0.70 | 0.50 | -0.20 | 回帰（閾値0.15超） |
  | 保守性 | 0.60 | 0.65 | +0.05 | - |

回帰加重スコア:
  adjusted_diff = raw_diff - 1.5 × Σ(回帰カテゴリの低下幅)
  → 総スコアが上がっても、特定カテゴリの大幅低下があればペナルティ
```

**ポイント**: 総スコアだけを最適化すると「得意カテゴリに注力して苦手カテゴリを放棄する」局所最適に陥る（注意力再配分トラップ）。カテゴリ別の回帰検出がこれを防止する。

## §15. 統合後の整合性チェック

### パターン: クロスソース一致度の検証

複数のソース（異なるドメインのテスト入力、異なるエージェントの評価結果）を統合する際に、ソース間の乖離度を計算して過適合や不整合を検出する。

```
例: 2つのテスト文書での評価スコア
  文書1（医療ドメイン）: 85点
  文書2（ECドメイン）: 60点
  cross_doc_gap = |85 - 60| = 25

gap が大きい → 特定ドメインへの過適合の疑い
gap が小さい → ドメイン非依存の安定した性能
```

**ポイント**: 統合スコア（平均）だけでは「片方が高く片方が低い」不整合が隠れる。ソース間の乖離度を明示的に計算して報告する。

## §16. 限界の明記

### パターン: 検出限界の定量指標

方式が対応できないケースを定量的に記録し、運用者が方式の能力を過信しないための指標を提供する。

```
限界サマリの例:
  識別力高=5問, 中=8問, 低（天井）=4問, 低（床）=3問

  床問題（この方式では検出不能）:
  - P7: 暗黙のビジネスルール違反（難易度: 高）
  - P12: コンテキスト依存の設計判断（難易度: 高）
  - P15: ドメイン固有の慣習からの逸脱（難易度: 高）

  → 床問題は全て難易度「高」であり、ドメイン専門家による
    補完レビューが必要な領域を示している
```

### パターン: 循環構造の識別

安全網や検証ステップが、解決対象と同じ構造の問題を持っていないかを確認する。

```
× 循環構造:
  問題: 「網羅的に洗い出す」精度が不確実
  安全網: 「漏れなく補完せよ」→ 同じ不確実性を別の場所に移動しただけ

○ 循環の識別と対処:
  反復改善ループで天井/床問題を識別する:
    天井: 全実行で検出 → 既に最大性能、改善不要
    床:   全実行で未検出 → 構造的限界、改善不可能
    変動: 実行ごとに異なる → 改善の余地あり

  床問題を限界として明示し、改善可能な問題に注力する
```

**ポイント**: 「検出できないものを検出せよ」は循環構造。床問題を明示的に除外することで、改善ループが無限に空回りすることを防止する。

## §17. プロセス原因分析

### パターン: エラー→構造原因の因果追跡

評価結果のエラーを「出力の何が問題か」ではなく「プロセスのどの設計判断がこの問題を生んだか」に変換し、構造的改善を導出する。

```
× 表面的修正:
  エラー: 「性能に関する指摘が不足」
  修正: 「性能についても必ず言及すること」を指示に追加
  → 同じ構造の問題（他カテゴリの不足）が再発する

○ 構造原因分析:
  エラー: 「性能に関する指摘が不足」
  原因追跡: 「視点リストに性能が含まれているが、
    チェック手順が『重要度の高い順に処理』となっており、
    性能の優先度が低く設定されている」
  構造改善: チェック手順を「視点ごとに最低N件」の
    均等配分に変更する
  → 同じ構造の問題（特定カテゴリの不足）が全て解消される
```

### パターン: 修正の構造的検証

修正後に、修正が表面的な症状対処ではなく構造的原因を解消しているかを確認する。§14「カテゴリ別検出率の追跡と回帰検出」が修正検証を兼ねる — 修正版の評価時に回帰を検出することで、「修正が新たな問題を導入していないか」を構造的に検証する。

**ポイント**: 因果追跡の結果は知見ファイル（§3）に有効/無効な変更として蓄積し、次のイテレーションで参照する。
