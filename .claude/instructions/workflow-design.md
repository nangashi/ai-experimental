# AI Workflow Design

マルチステップのAIワークフロー設計における構造設計・コンテキスト管理の原則集。エントリは推奨適用順に配置している。
実装パターンは [examples/workflow-structure-examples.md](examples/workflow-structure-examples.md) を参照。

## 主要概念を定義してからワークフローを設計する

- **scope**: マルチステップワークフローで主要概念の解釈が揺れうる場合
- **actions**:
  - ワークフロー設計時に、主要概念の定義・スコープ・前提条件・制約を確定し、ワークフローへの入力として与える
  - 区別が必要な概念は分類を明示する（例: 「漏れ」→ 識別漏れ / 深度不足 / 引き継ぎ漏れ）
  - 定義はファイルとして用意し、後続ステップから共通参照させる
- **rationale**: マルチエージェント構成では、エージェント間で前提・解釈が一致しないまま作業が進行し、後続エージェントが先行エージェントの出力を誤解釈する、タスクの目的から逸脱する等の問題が生じる。概念の定義を先行させることでこれを予防できる
- **condition**: 入力ごとに定義が変わる場合（未知ドメインの文書分析等）は、ワークフロー内に定義を確立するステップを設けることも有効
- **sources**: [Why Do Multi-Agent LLM Systems Fail? (Cemri et al., 2025)](https://arxiv.org/abs/2503.13657)

## 方向性の決定を詳細化の前に確定する

- **scope**: 方向性の決定（方針・アプローチ選定等）と、それに依存する詳細化（手順策定・実装設計等）を含むタスク
- **actions**:
  - 方向性の決定と詳細化を別ステップに分離し、方向性を確定してから詳細化に進む
  - 各ステップの出力に求める具体性を明示する（例: 手順策定では入出力フォーマット・データ受渡し方法・数値パラメータの根拠まで含める）
- **rationale**: 方向性の決定と詳細化を同一ステップで行うと、方向性が未確定のまま詳細化が先行し、局所的には妥当だが全体方針と整合しない成果物になる。分離することで方向性の決定に十分なアテンションが配分され、詳細化が全体方針の制約下で行われる
- **sources**: [Plan-and-Act: Improving Planning of Agents for Long-Horizon Tasks (2025)](https://arxiv.org/abs/2503.09572)

## マルチフェーズワークフローではメインエージェントをオーケストレーターの役割にする

- **scope**: 複数フェーズで構成されるワークフローの構造設計
- **actions**:
  - 各フェーズの実作業をサブエージェントに委譲し、メインエージェントはフェーズ間の制御に限定する
  - サブエージェントとのデータ受け渡しはファイル経由にする。生成結果・参照コンテンツともにファイルに置き、親コンテキストには要約・メタデータのみ持つ。参照コンテンツは親のプロンプトに埋め込まず、サブエージェントが直接読み込む
  - フェーズ間で引き継ぐのは確定した成果物・決定事項のみとし、会話履歴や中間出力を持ち越さない
- **rationale**: 200,000+のシミュレーション会話に基づく研究で、マルチターンでの平均39%の性能低下が確認されている。原因は過去の出力へのアンカリング、中間ターンの軽視、出力の冗長化の累積である。サブエージェントは毎回新規コンテキストで起動されるため、フェーズごとにサブエージェント化することでアンカリング劣化を構造的に回避できる。また親コンテキスト窓は全フェーズで共有される有限リソースであり、データ受け渡しをファイル経由にすることでアテンション精度の低下と自動圧縮による情報喪失を防ぐ
- **sources**:
  - [LLMs Get Lost In Multi-Turn Conversation (Laban et al., Microsoft Research, 2025)](https://arxiv.org/abs/2505.06120)
  - [Effective Context Engineering for AI Agents — Anthropic (2025)](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)

## 生成と評価は構造的に分離する

- **scope**: 生成した成果物に対してレビュー・検証・受け入れ判定を行う場合
- **actions**:
  - 生成と評価はワークフロー分割やサブエージェント化で分離し、生成時のコンテキスト（中間出力・試行錯誤の履歴等）を評価側に持ち込まない
- **rationale**: LLMはアンカリング効果により先行結果に引きずられる。この傾向はプロンプト改善では解消できない。構造的分離により先行結果へのアンカリングを排除できる
- **condition**: 生成と評価が同一コンテキストの場合、生成過程へのアンカリングが加わり評価の歪みが強まる。最低でもコンテキスト分離（別サブエージェント化）を確保する
- **sources**: [Anchoring Bias in Large Language Models: An Experimental Study (Zhao & Shen, 2024)](https://arxiv.org/abs/2412.06593)

## 複数の妥当なアプローチがある場合は対立討論を活用する

- **scope**: 方式選定・設計方針の比較など、複数の妥当なアプローチがあり最適解が自明でない場合
- **actions**:
  - 各方式に擁護エージェントを割り当て対立討論を行う。形式: (1) 各擁護エージェントが自案を主張し相手方式への反論を提示（並列起動） (2) 相手の主張を踏まえた再反論（並列起動） (3) 論点の統合と推奨の提示。討論ラウンドは2-3回に制限する
- **rationale**: 単一エージェントに複数方式を比較させると、提示順序に依存する位置バイアスにより評価が系統的に歪む（順序入替だけで評価が逆転するケースが大量に確認されている）。方式ごとに擁護エージェントを分離すると、各方式が独立したコンテキストで主張され、単一エージェント内での順序比較が発生しないため位置バイアスが排除される。ただしSycophancyがラウンドの進行とともに強化されるため、ラウンド数の制限が必要である
- **sources**:
  - [Large Language Models are not Fair Evaluators (Wang et al., ACL 2024)](https://arxiv.org/abs/2305.17926)
  - [Peacemaker or Troublemaker: Sycophancy in Multi-Agent Debate (2025)](https://arxiv.org/abs/2509.23055)

## 網羅性が求められるタスクでは外部基準を参照して担保する

- **scope**: 分類・列挙・カバレッジ評価など、網羅性が成果物の品質に直結するタスク
- **actions**:
  - ISO規格・業界標準・確立されたフレームワーク等の外部基準を明示的に参照させる
  - 外部基準はチェックリスト（項目の存在確認）ではなくフェーズ構造（各観点での深い分析）として活用する
  - 参照結果をカバレッジマップとして可視化し、後続ステップの検証基準に使う
  - カバレッジマップ自体の偏り（特定視点への集中）を検出するメタチェックを含める
  - 検索による情報収集を伴う場合は、検索実行前に調査すべきカテゴリ・観点をリストアップし、各カテゴリについて検索させる。LLMが自然に優先しにくい領域（規制・法律、非技術的側面等）が重要な場合は、カテゴリを明示的に指定する
- **rationale**: LLMの内部知識のみに依存した分類や列挙は、訓練データ分布に起因する系統的偏りを持つ。例えば技術インフラ層の項目は豊富に生成されるが、ドメインモデリングやビジネスロジック層は薄くなる傾向がある。この偏りはプロンプト指示では矯正できない。外部基準を参照することでカバレッジの偏りが構造的に緩和され、特定領域の系統的な欠落を防止できる。また曖昧な検索指示に対してLLMは早期に満足化し少数回で打ち切る傾向があり、カテゴリ列挙のプロセス指示により検索回数・情報源数がほぼ倍増することを実験で確認している（検索13回→23回、情報源約20→約40）
- **condition**: 外部基準自体が特定の視点に偏っている場合があるため、複数の基準を組み合わせる。検索実験はClaude Opus 4.6のサブエージェント（WebSearch使用）で実施しており、モデルやツールのバージョンで挙動が変わる可能性がある
- **sources**:
  - [Mind the Data Gap: Bridging LLMs to Enterprise Data Integration (Kayali et al., 2024)](https://arxiv.org/abs/2412.20331)
  - [Advancing Software Quality: A Standards-Focused Review of LLM-Based Assurance Techniques (2025)](https://arxiv.org/abs/2505.13766)

## タスク内の修正サイクルにはリトライ上限とコンテキストリセットを組み込む

- **scope**: サブエージェントが成果物の生成と検証を繰り返すステップ（コード生成→テスト実行→修正等）を含むワークフローを設計する場合
- **actions**:
  - 同一コンテキスト内での修正は2-3回を上限とし、改善しない場合のエスカレーション（試行内容・失敗原因・現時点の成果物の報告）をワークフローに組み込む
  - エスカレーション後は新規コンテキスト（別サブエージェント）での再試行を設計する。前回の試行結果は成果物とエラー情報のみをファイル経由で引き継ぎ、修正過程の会話履歴は持ち込まない
- **rationale**: LLMは初期ターンで立てた仮定に固着し、同一コンテキスト内では回復しない（"get lost and do not recover"）。2ターン以上のマルチターンで平均39%の性能低下が確認されており、主因は早期仮定への固着と回答の肥大化である。新規コンテキストでの再開がこの劣化を回復する最も有効な手段であるため、リトライ上限とコンテキストリセットをワークフロー構造として埋め込む
- **sources**: [LLMs Get Lost In Multi-Turn Conversation (Laban et al., Microsoft Research, 2025)](https://arxiv.org/abs/2505.06120)

## ユーザーチェックポイントを方向転換コストに基づいて配置する

- **scope**: 人間との対話を含むマルチフェーズワークフローを設計する場合
- **actions**:
  - 方向転換コストが高い地点にチェックポイントを配置する: スコープ定義後、方針確定後、不可逆な出力（ファイル書き出し・デプロイ）の前
  - 品質検証が機械的に判定可能な場合は、ユーザーチェックポイントの代わりに自動品質ゲートで代替する（整合性チェック、カバレッジ検証等）
  - 反復ループ（生成→評価→改善）の境界では、収束判定の指標を提示して継続/終了の判断を支援する
- **rationale**: チェックポイントの過不足はどちらもコストを生む。不足すると、方向の誤りが後続フェーズに伝播し手戻りコストが増大する。過剰だと、コンテキスト消費（確認ごとに質問+回答がコンテキストに蓄積）とユーザー疲労（形式的な承認の連続で注意力が低下し、問題を見落とす）を招く。配置基準がないと「念のため」の確認が増殖し、ワークフローが冗長になる。方向転換コストに基づいて配置することで、必要な方向修正を早期に捕捉しつつコンテキスト消費とユーザー負荷を最小化できる
- **condition**: 「ok」のみの承認が続く場合は、チェックポイントの粒度が細かすぎる兆候。フェーズ単位での一括確認への統合を検討する
- **sources**: [Practices for Governing Agentic AI Systems (Shavit, Agarwal et al., OpenAI, 2023)](https://openai.com/index/practices-for-governing-agentic-ai-systems/)

## 手順の検証にはシナリオ走査を使う

- **scope**: 設計したワークフローの手順記述を検証する場合
- **actions**:
  - シナリオ走査で検証する。手順の記述のみに従い暗黙の補完を禁止することで、手順が暗黙に前提としている補完を検出する
  - 手順の構造的限界（この手順では検出できないケースは何か）を明示する
- **rationale**: 議論やレビューでは手順レベルの曖昧性や欠落が見落とされやすい。具体的なシナリオを「手順に書かれていない暗黙の補完を行わない」制約で走査すると、手順が暗黙に前提としている補完を炙り出し、実装時に動作が一意に定まらない箇所を確実に検出できる。検証手段として最も信頼性が高い
- **condition**: シナリオの選定自体にバイアスがかかりうるため、手順が想定していない入力パターンを意識的に含める
- **sources**: [Detecting Ambiguities in Requirements Documents Using Inspections (Kamsties, Berry, Paech, 2001)](https://cs.uwaterloo.ca/~dberry/FTP_SITE/reprints.journals.conferences/KamstiesBerryPaech2001DetectingAmbiguity.pdf)
