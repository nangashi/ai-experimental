# AI Workflow Design — 実装パターン集

[workflow-design.md](../workflow-design.md) の各原則を実装する際の具体的なパターン。

## 主要概念の定義

### パターン: 共通語彙ファイルの確立

ワークフロー冒頭で、主要概念の定義・スコープ・前提条件を構造化したファイルを生成し、後続の全ステップがこのファイルを入力として参照する。

```
Phase 0:
  入力: ユーザーの要求テキスト
  処理: 以下を定義してファイルに保存
    - 対象スコープ（何が対象で、何が対象外か）
    - 主要概念の定義と分類基準
    - 前提条件と制約
  出力: definitions.md

Phase 1+:
  入力: definitions.md + 各フェーズ固有の入力
  （definitions.md を全サブエージェントの共通入力に含める）
```

**ポイント**: 定義ファイルは「後続フェーズの入力仕様の一部」として組み込む。「参照してください」という指示ではなく、サブエージェントへの入力として渡すことで参照漏れを防止する。

## 方向性の決定と詳細化の分離

### パターン: 2段階フェーズ構成

前半フェーズで「何を重視するか」（方針）を確定させ、後半フェーズで「具体的にどうするか」（手順・代替案）を詳細化する。

```
方針フェーズ（先に確定）:
  Phase 1: 問題のスコープと制約を定義
  Phase 2: 判断基準（何を重視するか）を定義
  → ユーザー確認を挟み、方針レベルで合意を形成

具体化フェーズ（方針確定後に実行）:
  Phase 3: 方針に基づいて代替案を生成
  Phase 4: 判断基準に基づいて各案を評価
  → 評価基準が後から変動しない
```

**ポイント**: 方針と具体化を同時に行うと、「具体的な案が先に出て、それに合う方針を後付けする」逆転が起きる。方針フェーズでユーザー確認を挟むことで逆転を防止する。

## 生成と評価の構造的分離

### パターン: 生成者と評価者のコンテキスト分離

成果物を生成するエージェントと、その品質を評価するエージェントを別サブエージェントとして起動し、評価エージェントには生成プロセスの情報を渡さない。

```
生成エージェント:
  入力: 要件 + 改善指示
  出力: 成果物（ファイルに保存）
  → 改善指示のコンテキスト（何を意図して変更したか）は評価エージェントに渡さない

評価エージェント:
  入力: 成果物 + 評価基準（ファイルから読み込み）
  出力: 評価結果（ファイルに保存）
  → 成果物が「何を改善しようとしたか」を知らずに評価する
```

**ポイント**: 「改善意図」を評価エージェントに渡すと、意図に沿った甘い評価になる。評価エージェントは「成果物が基準を満たしているか」だけを判断すべき。

### パターン: 批評の並列独立実行

1つの成果物に対して複数の批評エージェントを並列起動し、それぞれ異なる観点に集中させる。各批評エージェントは他の批評結果を見ない。

```
成果物 → 並列に4つのサブエージェントを起動:
  批評1: 有効性（この定義で目的を達成できるか）
  批評2: 完全性（欠落している観点はないか）
  批評3: 明確性（曖昧な記述はないか）
  批評4: 汎用性（特定ケースに過適合していないか）

各批評エージェントの入力: 成果物のみ（他の批評結果を含めない）
```

**ポイント**: 批評を直列に行うと、先行する批評にアンカリングされる。並列実行 + 結果の事後統合により、独立した多角的批評が得られる。

## オーケストレーターパターン

### パターン: フェーズ別サブエージェント委譲

メインエージェントはフェーズ間の制御に限定し、各フェーズの実作業をサブエージェントに委譲する。データ受け渡しはファイル経由とし、親コンテキストには要約・メタデータのみ持つ。

```
メインエージェント（オーケストレーター）:
  Phase 1: サブエージェントAを起動
    → サブエージェントA: requirements.md を読み込み、analysis.md を出力
  Phase 2: サブエージェントBを起動
    → サブエージェントB: analysis.md を読み込み、design.md を出力
  Phase 3: サブエージェントCを起動
    → サブエージェントC: design.md を読み込み、implementation.md を出力

メインエージェントが保持する情報:
  ○ 各フェーズの完了状態、成果物のファイルパス
  × 各フェーズの中間出力、会話履歴、試行錯誤の過程

サブエージェントへの参照コンテンツの渡し方:
  × 親が読み込んでプロンプトに埋め込む（親コンテキストを消費）
  ○ ファイルパスを渡し、サブエージェントが直接読み込む
```

**ポイント**: 親コンテキスト窓は全フェーズで共有される有限リソース。参照コンテンツを親が中継すると、フェーズが進むほどコンテキストが圧迫され、アテンション精度が低下する。サブエージェントに直接読み込ませることで親コンテキストの消費を最小化する。

## 対立討論

### パターン: 擁護エージェント方式

各方式に擁護エージェントを割り当て、独立したコンテキストで主張・反論を行わせる。

```
Phase 1a — 擁護エージェントA:
  入力: 要件・制約 + 方式Aの詳細
  指示: 方式Aの利点を主張し、方式Bへの反論を提示せよ
  出力: argument-a.md

Phase 1b — 擁護エージェントB（並列起動）:
  入力: 要件・制約 + 方式Bの詳細
  指示: 方式Bの利点を主張し、方式Aへの反論を提示せよ
  出力: argument-b.md

Phase 2a — 擁護エージェントA（再反論）:
  入力: argument-b.md + 方式Aの詳細
  出力: rebuttal-a.md

Phase 2b — 擁護エージェントB（再反論、並列起動）:
  入力: argument-a.md + 方式Bの詳細
  出力: rebuttal-b.md

Phase 3 — 統合（親エージェントまたは別エージェント）:
  入力: argument-a.md + argument-b.md + rebuttal-a.md + rebuttal-b.md
  処理: 各論点を整理し、トレードオフを明示して推奨を提示
```

**ポイント**: 各エージェントに異なる推論方法（評価軸・分析フレームワーク等）を指定する。ペルソナの違いだけでは思考パターンが均質化する。ラウンドは2-3回に制限する（Sycophancyがラウンド進行とともに強化されるため）。

## ユーザーチェックポイントの配置

### パターン: 方向転換コストに基づく配置

チェックポイントを「方向転換コストが高い地点」に限定し、機械的に検証可能な品質確認は自動ゲートで代替する。

```
× チェックポイント過剰（全フェーズ後に確認）:
  Phase 1: スコープ定義 → ユーザー確認
  Phase 2: 方針決定 → ユーザー確認
  Phase 3: 候補案生成 → ユーザー確認
  Phase 4: 評価結果 → ユーザー確認
  Phase 5: 最終出力 → ユーザー確認
  → 「ok」の連続で注意力が低下し、重要な確認を見逃す

○ 方向転換コストに基づく配置:
  Phase 1: スコープ定義 → ユーザー確認（方向転換コスト高: スコープ変更は以降の全フェーズに影響）
  Phase 2: 方針決定 → ユーザー確認（方向転換コスト高: 具体化フェーズの前提）
  Phase 3: 候補案生成 → 自動品質ゲート（形式・整合性の機械的検証）
  Phase 4: 評価結果
  Phase 5: 最終出力 → ユーザー確認（不可逆: ファイル書き出し前）
```

### パターン: 反復ループの収束判定

反復ループ（生成→評価→改善）の境界では、収束状況の指標を提示して継続/終了の判断を支援する。

```
ラウンドN 完了時の提示例:
  スコア推移: R1=62 → R2=71 → R3=74 → R4=75（収束傾向）
  改善幅: +9 → +3 → +1
  未解決の重要指摘: 2件（前回比 -1）
  推奨: 収束傾向のため終了を推奨。継続する場合の期待改善幅は+0-1pt
```

**ポイント**: チェックポイントの過不足はどちらもコストを生む。過剰だとコンテキスト消費とユーザー疲労、不足だと手戻りコスト。「ok」のみの承認が続く場合は粒度が細かすぎる兆候。

## 外部基準による網羅性担保

### パターン: フレームワーク準拠のフェーズ構成

ワークフローのフェーズ構成自体を確立されたフレームワークに準拠させることで、検討観点の網羅性を構造的に担保する。

```
例: 意思決定ワークフロー → PrOACT フレームワーク
  Phase 1: Problem（問題定義）
  Phase 2: Objectives（判断基準の定義）
  Phase 3: Alternatives（代替案の生成）
  Phase 4: Consequences（各案の評価）
  Phase 5: Tradeoffs（トレードオフの明示）

例: 設計レビューワークフロー → ISO 25010 品質特性
  各品質特性（セキュリティ、性能、保守性、...）を
  独立した評価エージェントの視点として割り当てる
```

**ポイント**: フレームワークは「チェックリスト」ではなく「フェーズ構造」として使う。チェックリスト方式は項目の存在確認に留まるが、フェーズ構造にすると各観点での深い分析が行われる。外部基準自体が特定の視点に偏っている場合があるため、複数の基準を組み合わせる。

### パターン: カバレッジマップによる偏り検出

参照結果をカバレッジマップとして可視化し、観点の偏り（特定視点への集中）を構造的に検出する。

```
Step 1: 外部基準の観点を軸としたマップを作成
  | 観点（外部基準由来） | 検討済み項目数 | 深度（概要/詳細/検証済み） |
  |---------------------|-------------|------------------------|
  | セキュリティ         | 8           | 詳細                    |
  | 性能                | 6           | 詳細                    |
  | 保守性              | 2           | 概要                    |  ← 偏り検出
  | 信頼性              | 0           | -                      |  ← 欠落検出

Step 2: メタチェック（偏りの自動検出）
  - 項目数が平均の50%未満の観点を警告
  - 深度が「概要」以下の観点を警告
  - 外部基準に存在するが未検討の観点を欠落として報告

Step 3: 偏り是正
  - 警告された観点について追加分析を実施
  - カバレッジマップを更新して再検証
```

**ポイント**: カバレッジマップは後続ステップの検証基準としても使う。成果物のレビュー時に「この観点は検討されているか」をマップで照合することで、網羅性の検証が機械的に行える。

## シナリオ走査

### パターン: 暗黙の補完を禁止した手順走査

手順の記述のみに従い、暗黙の補完を一切行わないことで、手順が前提としている未記述の情報を検出する。

```
走査の制約:
  - 手順に明記されていない判断基準で分岐しない
  - 「常識的に」「当然」等の暗黙知で手順を補完しない
  - 手順が曖昧で複数の解釈が可能な場合、全ての解釈を列挙する

走査記録の例:
  手順: 「重要な問題を優先的に報告する」
  走査結果:
    - 「重要」の判断基準が未定義 → 解釈A: 影響範囲の広さ / 解釈B: 修正コスト / 解釈C: 発生頻度
    - 「優先的に」の具体的動作が未定義 → 解釈A: 上位N件のみ報告 / 解釈B: 重要度順にソート / 解釈C: 重要なものから順に処理
    → 動作が一意に定まらない箇所として報告

  手順: 「入力データを検証し、不正なデータを除外する」
  走査結果:
    - 「不正」の定義が未記述 → 型エラー？値の範囲外？ビジネスルール違反？
    - 除外後の処理が未定義 → ログ記録？エラー通知？サイレント除外？
    → 暗黙に前提としている補完を検出
```

**ポイント**: この技法は手順の曖昧性検出に特化している。検出精度の測定（後続の「正解キー付きテスト文書」パターン）とは目的が異なる。手順の曖昧性を先に解消し、明確化された手順の検出精度を正解キーで測定する、という順序で組み合わせると効果的。

### パターン: 正解キー付きテスト文書

手順の検証のために、意図的に問題を埋め込んだテスト入力を作成し、手順通りに処理した結果を正解キーと照合する。

```
テスト文書の構成:
  - 対象ドメインのリアルな文書として自然に読める
  - 複数カテゴリ・複数難易度の問題が埋め込まれている
  - 正解キーに各問題の位置・カテゴリ・期待される検出内容が定義されている

検証の実行:
  1. テスト文書をワークフローの入力として処理する
  2. 出力を正解キーと照合し、各問題の検出状況を記録
  3. 複数回実行して再現性を確認する（同じ入力で結果がばらつく箇所 = 手順の曖昧な箇所）
```

**ポイント**: テスト文書は異なるドメインで2つ以上作成する。1つだけだと特定ドメインへの過適合が検出できない。複数回実行（同一入力・同一手順）で結果がばらつく箇所は、手順が一意に定まっていない証拠。

### パターン: 事前定義の正解キーによる評価

「良い出力か」を出力自身の性質で判断するのではなく、事前に定義した正解キー（期待される検出項目のリスト）との照合で評価する。

```
× 自己参照的な評価:
  「重要な問題を見落としていないか」→ 見落としている問題は評価者にも見えない

○ 外部基準による評価:
  1. 事前にテスト文書を作成し、意図的に問題を埋め込む
  2. 各問題に ID・カテゴリ・難易度を付与した正解キーを作成
  3. エージェント出力を正解キーと照合し、○（検出）/△（部分検出）/×（未検出）を判定

正解キーの構造例:
  | ID | カテゴリ | 難易度 | 問題の説明 | 埋め込み箇所 |
  |----|---------|--------|-----------|------------|
  | P1 | セキュリティ | 高 | SQLインジェクション脆弱性 | §3.2 |
  | P2 | 性能 | 中 | N+1クエリ | §4.1 |
```

**ポイント**: 「重要度が高い順に選定」「見落としやすいものを優先」のような基準は自己参照的。「外部規格の品質特性に対応する順」「入力文書への紐付きが明確な順」のように、評価対象から独立した基準を使う。

## 反復改善のプロセス原因分析

### パターン: 知見蓄積ファイルの参照

反復ワークフローでは、過去のイテレーションで判明した有効/無効な変更を知見ファイルに蓄積し、次のイテレーションの生成ステップで入力として渡す。

```
知見ファイルの構造例:
  ## 有効だった変更
  | 変更内容 | 効果(pt) | 対象カテゴリ | ラウンド |
  |---------|----------|------------|---------|
  | 具体例の追加 | +2.3 | 検出漏れ | R2 |

  ## 無効だった変更
  | 変更内容 | 効果(pt) | 対象カテゴリ | ラウンド |
  |---------|----------|------------|---------|
  | 網羅的列挙の指示 | -0.5 | 全体 | R1 |

生成ステップへの入力:
  - 現在の成果物
  - エラー分析結果
  - 知見ファイル（有効/無効な変更の履歴）
```

**ポイント**: 知見は自然言語の感想ではなく、変更内容・定量効果・対象の3つ組で記録する。これにより「何が効いたか」の判断が曖昧にならない。

### パターン: エラー→構造原因の因果追跡

評価結果のエラーを「出力の何が問題か」ではなく「プロセスのどの設計判断がこの問題を生んだか」に変換し、構造的改善を導出する。

```
× 表面的修正:
  エラー: 「性能に関する指摘が不足」
  修正: 「性能についても必ず言及すること」を指示に追加
  → 同じ構造の問題（他カテゴリの不足）が再発する

○ 構造原因分析:
  エラー: 「性能に関する指摘が不足」
  原因追跡: 「視点リストに性能が含まれているが、
    チェック手順が『重要度の高い順に処理』となっており、
    性能の優先度が低く設定されている」
  構造改善: チェック手順を「視点ごとに最低N件」の
    均等配分に変更する
  → 同じ構造の問題（特定カテゴリの不足）が全て解消される
```

### パターン: カテゴリ別検出率の追跡と回帰検出

総スコアだけでなくカテゴリ別の検出率を追跡し、特定カテゴリの劣化（回帰）にペナルティを課す。

```
評価結果の追跡:
  | カテゴリ | ベースライン検出率 | 今回の検出率 | 差分 | 回帰判定 |
  |---------|-----------------|------------|------|---------|
  | セキュリティ | 0.80 | 0.90 | +0.10 | - |
  | 性能 | 0.70 | 0.50 | -0.20 | 回帰（閾値0.15超） |
  | 保守性 | 0.60 | 0.65 | +0.05 | - |

回帰加重スコア:
  adjusted_diff = raw_diff - 1.5 × Σ(回帰カテゴリの低下幅)
  → 総スコアが上がっても、特定カテゴリの大幅低下があればペナルティ
```

**ポイント**: 総スコアだけを最適化すると「得意カテゴリに注力して苦手カテゴリを放棄する」局所最適に陥る（注意力再配分トラップ）。カテゴリ別の回帰検出がこれを防止する。

### パターン: 修正の構造的検証

修正後に、修正が表面的な症状対処ではなく構造的原因を解消しているかを確認する。「カテゴリ別検出率の追跡と回帰検出」パターンが修正検証を兼ねる — 修正版の評価時に回帰を検出することで、「修正が新たな問題を導入していないか」を構造的に検証する。

**ポイント**: 因果追跡の結果は知見ファイルに有効/無効な変更として蓄積し、次のイテレーションで参照する。

### パターン: 検出限界の定量指標

方式が対応できないケースを定量的に記録し、運用者が方式の能力を過信しないための指標を提供する。

```
限界サマリの例:
  識別力高=5問, 中=8問, 低（天井）=4問, 低（床）=3問

  床問題（この方式では検出不能）:
  - P7: 暗黙のビジネスルール違反（難易度: 高）
  - P12: コンテキスト依存の設計判断（難易度: 高）
  - P15: ドメイン固有の慣習からの逸脱（難易度: 高）

  → 床問題は全て難易度「高」であり、ドメイン専門家による
    補完レビューが必要な領域を示している
```

### パターン: 循環構造の識別

安全網や検証ステップが、解決対象と同じ構造の問題を持っていないかを確認する。

```
× 循環構造:
  問題: 「網羅的に洗い出す」精度が不確実
  安全網: 「漏れなく補完せよ」→ 同じ不確実性を別の場所に移動しただけ

○ 循環の識別と対処:
  反復改善ループで天井/床問題を識別する:
    天井: 全実行で検出 → 既に最大性能、改善不要
    床:   全実行で未検出 → 構造的限界、改善不可能
    変動: 実行ごとに異なる → 改善の余地あり

  床問題を限界として明示し、改善可能な問題に注力する
```

**ポイント**: 「検出できないものを検出せよ」は循環構造。床問題を明示的に除外することで、改善ループが無限に空回りすることを防止する。

## 修正失敗時の中断

### パターン: 2回失敗時のエスカレーション

同一タスクで2回修正を試みて改善しない場合、追加修正を中止してエスカレーションする。

```
サブエージェントの場合:
  試行1: 修正を適用 → 評価で改善なし
  試行2: 別の修正を適用 → 評価で改善なし
  → 追加修正を中止し、以下をメインエージェントに報告:
    - 試行した修正の内容と意図
    - 各試行で改善しなかった原因の推定
    - 現時点の成果物

メインエージェントの場合:
  試行1: 修正を適用 → 改善なし
  試行2: 別の修正を適用 → 改善なし
  → 以下を整理して新規セッションで再開:
    - 現時点の成果物
    - 元の要件
    - 試行した修正と結果の要約
  → 新規コンテキストで再開することで回答アンカリングをリセット
```

**ポイント**: 修正を重ねるほど過去の出力へのアンカリングで品質が低下する。新規コンテキストでの再開がアンカリング劣化を回復する唯一の手段。

## 判断記録の均等粒度

### パターン: 採用・棄却の同粒度記録

永続ドキュメントに判断履歴を蓄積する際、棄却した選択肢にも採用例と同じ粒度で記録を残す。

```
× 採用偏重の記録:
  ## 有効だったアプローチ
  | アプローチ | 効果 | 根拠 | 適用条件 |
  |-----------|------|------|---------|
  | 視点別分離 | +3.2pt | 検出均等化 | 視点間のバランスが必要な場合 |
  | 正解キー照合 | +2.8pt | 客観評価 | 検出精度の測定が必要な場合 |

  ## 棄却したアプローチ
  - チェックリスト方式（効果なし）
  - 直列批評（非推奨）

○ 均等粒度の記録:
  ## 有効だったアプローチ
  | アプローチ | 効果 | 根拠 | 適用条件 |
  |-----------|------|------|---------|
  | 視点別分離 | +3.2pt | 検出均等化 | 視点間のバランスが必要な場合 |

  ## 棄却したアプローチ
  | アプローチ | 効果 | 棄却理由 | 再検討条件 |
  |-----------|------|---------|-----------|
  | チェックリスト方式 | -0.5pt | 存在確認に留まり深い分析不足 | 項目数が少なく深度不要の場合 |
  | 直列批評 | +0.3pt | 先行批評へのアンカリング発生 | 批評間に依存関係がある場合 |
```

**ポイント**: 採用例のみが詳細だと、後続セッションが過去の採用パターンの改善に収斂し、代替アプローチの検討が抑制される。棄却理由と再検討条件を残すことで、状況が変わった際の代替手段の発見を支援する。
