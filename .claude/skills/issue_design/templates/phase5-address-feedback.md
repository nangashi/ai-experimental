以下の手順でPRフィードバックに基づいて設計ドキュメントを修正してください:

## 手順

### 1. 要件の把握

{requirements_path} を Read で読み込み、設計が満たすべき要件を把握する。

### 2. 現在の設計の把握

{design_path} を Read で読み込み、現在の設計内容を把握する。

### 3. PRフィードバックの把握

{feedback_path} を Read で読み込み、全コメントを把握する。
各コメントの `comment_id` を保持し、対応サマリにそのまま記録すること。

### 4. コメント分類・設計修正

各コメントを以下のカテゴリに分類する:
- **design_change**: 設計の修正が必要（仕様変更、設計方針の見直し等）
- **clarification**: 設計の説明追記・明確化が必要（不明瞭な記述の改善等）
- **acknowledged**: 対応不要（情報提供、賛同、質問への回答済み等）

design_change と clarification の各コメントに対して設計を修正する:

- **要件との整合性を維持する**: 修正が要件に矛盾しないことを確認する
- **設計箇所を特定して修正する**: コメントが指すセクションを特定し、該当箇所を修正する
- **修正不可能なコメント**: 要件上の制約等で対応できない場合は、理由を記録する

### 5. 修正後の設計を保存

修正後の設計全体を {design_path} に Write で上書き保存する。

修正がない場合（全コメントが acknowledged の場合）は、設計ファイルを変更しない。

### 6. 対応サマリの保存

{reply_save_path} に Write で保存する:

```markdown
# フィードバック対応サマリ

## 対応済み

### {コメント要約1}
- comment_id: {対応したコメントのcomment_id}
- 対応内容: {修正内容}
- 設計箇所: {修正したセクション}

### {コメント要約2}
- comment_id: {対応したコメントのcomment_id}
- 対応内容: {修正内容}
- 設計箇所: {修正したセクション}

## 対応不要

### {コメント要約}
- comment_id: {対応不要コメントのcomment_id}
- 理由: {対応不要の理由}
```

対応済みが 0 件の場合は「## 対応済み」セクションに「なし」と記載する。
対応不要が 0 件の場合は「## 対応不要」セクションに「なし」と記載する。

### 7. 返答

以下のフォーマットで返答する:

```
result: success
addressed: {対応したコメント数}
no_action: {対応不要のコメント数}
summary: {修正内容の1行サマリ}
```
