以下の手順でタスクリスト構造の網羅性・依存関係・規模を検証してください:

## 手順

### 1. 入力の読み込み

以下のファイルを全て Read で読み込む:
- {task_structure_path}（タスクリスト構造）
- {req_path}（要件定義書 — SR/NFR/IR テーブルの参照元）
- {arch_path}（アーキテクチャ設計書 — コンポーネント一覧・トレーサビリティの参照元）
- {dp_path}（開発プロセス設計書 — プロセス成果物一覧の参照元）
- {dd_path}（詳細設計書 — テーブル定義・Service メソッド・API 契約の参照元）

コマンドがエラーを返した場合、またはデータが空・期待する構造と異なる場合は、以降のステップを続行せず、エラー内容と影響範囲を返却に含めて即座に返却する。

### 2. チェック実行

以下の14項目をチェックする。

| # | チェック | 基準 | 出力 |
|---|---------|------|------|
| 1 | SR カバレッジ | 全 SR が少なくとも1タスクにマッピングされている | 未カバー SR リスト |
| 2 | NFR カバレッジ・マッピング妥当性 | 全 NFR が少なくとも1タスクにマッピングされている。横断的 NFR（複数タスクにまたがる性能目標等）が、その NFR を検証する E2E テストを含むタスクまたは CI/CD タスクにマッピングされていること | 未カバー NFR リスト + 不適切マッピングリスト |
| 3 | IR カバレッジ | 全 IR が少なくとも1タスクにマッピングされている | 未カバー IR リスト |
| 4 | コンポーネントカバレッジ | architecture.md のコンポーネント一覧テーブルの全コンポーネントが少なくとも1タスクに含まれる | 未カバーコンポーネントリスト |
| 5 | 依存サイクル | 依存グラフに循環がない | サイクルのパス（あれば） |
| 6 | 依存正当性 | 各 blocked_by 関係の依存理由が実装上妥当である（タスクAがタスクBの成果物を前提とする具体的根拠がある） | 根拠不明な依存リスト |
| 7 | カバレッジ重複 | 同一 SR/コンポーネントが複数タスクに分散している場合、責務分担が明確。SR 重複: 主担当が明記され他タスクでの関与が注記されている場合は正当、主担当不明の場合は問題。コンポーネント重複: 複数タスクが同一コンポーネントを参照する場合は正当（メソッドレベルの競合は QG-2 チェック8で検証） | 重複箇所と判定 |
| 8 | タスク規模 | **critical**: 受け入れ基準が10項目以上、または推定変更ファイル15以上。**warning**: 受け入れ基準が7項目以上、または「目的」に複数の独立した関心事が含まれる | 過大タスクリスト + 分割提案 |
| 9 | テーブル定義カバレッジ | detailed-design.md の全テーブルが少なくとも1タスクでカバーされている | 未カバーテーブルリスト |
| 10 | Service メソッドカバレッジ | 全 Service メソッドが少なくとも1タスクでカバーされている | 未カバーメソッドリスト |
| 11 | API 契約カバレッジ | 全 API 契約が少なくとも1タスクでカバーされている | 未カバー契約リスト |
| 12 | プロセス成果物カバレッジ | development-process.md のドキュメント戦略・開発環境設定で定義された全成果物がタスクリスト構造のプロセス成果物マッピングに含まれ、担当タスクが割り当てられている | 未カバー成果物リスト |
| 13 | PR凝集度 | 各タスクのスコープが単一の関心事に属するか。「スコープ」欄に独立した2つ以上の関心事が列挙されている場合にフラグ | 凝集度不足タスクリスト |
| 14 | 外部サービスセットアップカバレッジ | architecture.md に外部サービスセットアップ要件セクションがある場合、または standards.md の環境変数テーブルに外部サービスのクレデンシャルを要する変数がある場合、それらの取得に必要なセットアップ作業がいずれかのタスク（executor: human）の受け入れ基準に含まれていること | 未カバー環境変数リスト |


### 3. SR→タスク マッピングサマリの作成

チェック結果とは別に、全 SR について「SR-X は Task N が主担当」のマッピングサマリを作成する。これは QG-2 チェック10 で参照するため、必ず結果に含めること。

### 4. 結果の保存

チェック結果を {findings_save_path} に Write で保存する:

```markdown
# QG-1: 網羅性・依存関係・規模検証結果

## 判定

{pass / needs_revision / pass with warnings}

## 指摘一覧

{指摘がない場合は「指摘なし」}

### チェック{N}: {チェック名}

- **重要度**: {critical / warning}
- **内容**: {具体的な問題の説明}
- **修正案**: {具体的な修正提案}

## SR→タスク マッピングサマリ

| SR | 主担当タスク | 他タスクでの関与 |
|----|------------|----------------|
| SR-001 | Task {NNN} | {あれば記述、なければ「—」} |
| ... | | |

## サマリ

- チェック項目: {8〜14}件（dd_path 有無でチェック9-11、dp_path の成果物有無でチェック12 が変動、チェック13-14は常に実行）
- 指摘: {N}件（critical: {N}, warning: {N}）
```

### 5. 判定基準

- **pass**: チェック1〜7, 12, 14（+ 9〜11 該当時）が全て OK、かつチェック8にcriticalなし
- **needs_revision**: チェック1〜7, 12, 14（+ 9〜11 該当時）のいずれかで問題あり、またはチェック8でcritical
- **pass with warnings**: pass条件を満たすが、チェック8のwarningまたはチェック13（PR凝集度）のフラグあり

critical の基準:
- 未カバー SR/NFR/IR/コンポーネント（チェック1〜4）
- 依存サイクル（チェック5）
- 主担当不明の SR 重複（チェック7）
- タスク規模のcritical（チェック8: 受け入れ基準10項目以上 or 推定変更ファイル15以上）
- 未カバーテーブル/Service メソッド/API 契約（チェック9〜11、該当時）
- 未カバープロセス成果物（チェック12）
- 未カバー外部サービスセットアップ（チェック14）

warning の基準:
- 依存理由の根拠不足（チェック6）
- コンポーネント重複で責務分担が不明確（チェック7）
- タスク規模のwarning（チェック8: 受け入れ基準7項目以上 or 複数の独立した関心事）
- PR凝集度不足（チェック13）

### 6. 返答

以下のフォーマットで返答する:

```
verdict: {pass / needs_revision / pass with warnings}
issues: {指摘件数}
summary: {1行サマリ}
```
