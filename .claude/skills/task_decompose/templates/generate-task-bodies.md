以下の手順でタスクリスト構造に基づきイシュー本文を生成してください:

## 手順

### 1. 入力の読み込み

以下のファイルを全て Read で読み込む:
- {task_structure_path}（タスクリスト構造）
- {ps_path}（problem-statement.md）
- {req_path}（requirements.md）
- {arch_path}（architecture.md）
- {std_path}（standards.md）
- {dp_path}（development-process.md）
- {dd_path}（detailed-design.md）

コマンドがエラーを返した場合、またはデータが空・期待する構造と異なる場合は、以降のステップを続行せず、エラー内容と影響範囲を返却に含めて即座に返却する。

### 2. 本文生成

タスクリスト構造の各タスクについて、以下のテンプレートに従い本文を生成する。

#### イシュー本文テンプレート

```markdown
## 目的

全{N}タスク中の第{M}タスク。{このタスクが何を実現するか（1-2文）}。

## 受け入れ基準

- [ ] {SR/NFR から導出した検証可能な条件}
- [ ] {...}

## 入力

- `detailed-design.md` §{X}（{何の仕様か}）
- `architecture.md` §{X}（{何の仕様か}）
- `standards.md`（{関連する規約名}）
- `development-process.md`（{テスト戦略等}）
```

#### 人間タスクの場合（executor: human）

```markdown
## 目的

全{N}タスク中の第{M}タスク。{このタスクが何を実現するか（1-2文）}。

## 実行者

human — 以下の作業は人間が外部サービスのWebコンソール上で実施する。LLMは受け入れ基準の検証のみ行う。

## 作業手順

1. {サービス名}: {具体的なセットアップ手順}
2. ...

## 受け入れ基準

- [ ] {LLMが検証可能な条件（環境変数の存在確認、接続テスト等）}
- [ ] {...}

## 入力

- `architecture.md` §{X}（外部サービスセットアップ要件）
- `standards.md` §{X}（環境変数一覧）
```

#### 本文生成の方針

- **「目的」の記述**: タスクの全体における位置づけと、何を実現するかを1-2文で記述する。実装手段（ファイルパス・メソッド名等）は含めない
- **「受け入れ基準」の導出**: SR の EARS 記述を「ユーザーが〜したとき、〜が表示/動作する」形式の検証可能な条件に変換する。テスト実行の合格条件もここに含める（例: 「`pnpm vitest run` で Unit テストが全パスする」）
  - PR粒度に合わせ、1タスクにつき概ね3-5項目を目標とする（厳密な上限ではない）
  - 各基準はそのPR内で完結して検証可能な条件であること
  - 受け入れ基準に具体的な値（環境変数値、接続文字列、クレデンシャル、トークン等）を含める場合、その値の取得元タスクを task-structure.md の blocked_by チェーンと突合する。blocked_by チェーンに含まれないタスクで初めて取得・生成される値は受け入れ基準に含めず、その値を取得するタスクまたは後続タスクの基準に移動する
  - 基準が多すぎる場合はスコープの分割不足を示唆する
- **人間タスクの「作業手順」**: architecture.md の外部サービスセットアップ要件テーブルから、各サービスのセットアップ手順を具体的に記述する。Webコンソール上の操作を順序立てて列挙し、取得したクレデンシャルの `.env.local` への設定までを含める
- **人間タスクの「受け入れ基準」**: 環境変数の存在・接続テストの成功等、LLMがコマンド実行で検証可能な条件のみを記述する。Webコンソール画面の状態確認等、LLMが検証不可能な条件は含めない
- **「入力」の記述**: タスクの受け入れ基準を満たすために参照すべき設計書のセクションを列挙する。各参照はファイル名＋セクション番号＋概要の形式とする（例: `detailed-design.md` §2.1（Article テーブル定義・ArticleRepository インターフェース））。タスクに無関係なセクションは含めない
- **「ブロック」の記述**: 依存タスクがある場合、タスク番号・タイトルに加え、利用する成果物の概要を付記する（例: 「Task 1（認証付きデプロイ）— 認証基盤・DB 接続設定」）。依存なしの場合は「なし」

### 3. 保存

**全タスクの本文をまとめて1回の Write で {output_save_path} に書き込む**（タスクごとの個別 Write は前の内容を上書きするため不可）。

保存形式:

```markdown
# イシュー本文

## Task 1: {タイトル}

{イシュー本文}

---

## Task 2: {タイトル}

{イシュー本文}

---

...
```

### 4. 返答

以下のフォーマットで返答する:

```
result: success
tasks_generated: {生成したタスク数}
total_acceptance_criteria: {全タスクの受け入れ基準合計数}
total_references: {全タスクの入力参照合計数}
```
