以下の手順でイシュー本文の品質・整合性を検証してください:

## 手順

### 1. 入力の読み込み

以下のファイルを全て Read で読み込む:
- {task_bodies_path}（全イシュー本文をまとめたファイル）
- {task_structure_path}（タスクリスト構造 — SR マッピング・依存関係の参照）
- {qg1_result_path}（QG-1 の結果 — SR マッピングサマリの参照）

注意: 設計書（architecture.md, standards.md, development-process.md）は渡されない。これはチェック3（自己完結性）の独立性を確保するため。

コマンドがエラーを返した場合、またはデータが空・期待する構造と異なる場合は、以降のステップを続行せず、エラー内容と影響範囲を返却に含めて即座に返却する。

### 2. 個別チェック（各イシューに対して）

| # | チェック | 基準 | 判定方法 |
|---|---------|------|---------|
| 1 | 独立完結性 | 単独の AI 開発セッションで完了可能な規模か | 「やること」のステップ数・新規ファイル数が過度でないか（目安: 新規ファイル 12 以下） |
| 2 | 独立検証性 | 受け入れ基準が他タスクの成果物なしに検証可能か | 各基準項目が、このタスクの成果物のみで検証できるか |
| 3 | 自己完結性 | AI 開発エージェントがこのイシュー本文だけで作業開始できるか | 「やること」に暗黙の前提知識がないか。使用する技術・パターンへの言及が制約に含まれるか |
| 4 | 垂直スライス | エンドツーエンドの機能を提供するか | 「DB層だけ」「UI層だけ」になっていないか |
| 5 | スコープ外の明確性 | 他タスクとの境界が曖昧でないか | スコープ外に記載された項目が、対応する他タスクの「やること」に含まれているか（task-structure.md で照合） |
| 6 | タスク間インターフェース | 先行タスクの成果物で後続タスクが依存するものが明示されているか | blocked_by のタスクが作るファイル・型・インターフェースが「制約」欄で言及されているか |
| 7 | 制約の過不足 | 制約セクションが過不足なく記載されているか | 制約が空でないか、具体的な技術名・規約名への言及があるか、タスクのスコープに対して過剰または不足でないか |

### 3. 横断チェック（全イシューを通して）

| # | チェック | 基準 |
|---|---------|------|
| 8 | 実装の競合 | 複数タスクで同一ファイルを異なる方針で変更する記述がないか |
| 9 | インターフェース境界の一貫性 | Task A が export する型/関数と Task B が import する型/関数に不整合がないか |
| 10 | QG-1 マッピングとの整合 | qg1-result.md の SR マッピングサマリで「SR-X は Task N が担当」とされたものが、Task N の受け入れ基準に反映されているか |

### 4. 結果の保存

チェック結果を {findings_save_path} に Write で保存する:

```markdown
# QG-2: イシュー品質・整合性検証結果

## 判定

{pass / needs_revision}

## 個別チェック結果

{指摘がない場合は「指摘なし」}

### Task {N}: チェック{X} — {チェック名}

- **重要度**: {critical / warning}
- **内容**: {具体的な問題の説明}
- **修正案**: {具体的な修正提案}

## 横断チェック結果

{指摘がない場合は「指摘なし」}

### チェック{X}: {チェック名}

- **重要度**: {critical / warning}
- **対象タスク**: Task {N}, Task {M}
- **内容**: {具体的な問題の説明}
- **修正案**: {具体的な修正提案}

## サマリ

- 個別チェック: {N}件（対象タスク数 × 7項目）
- 横断チェック: 3件
- 指摘: {N}件（critical: {N}, warning: {N}）
```

### 5. 判定基準

- **pass**: 指摘が 0 件
- **needs_revision**: 指摘が 1 件以上

critical の基準:
- 独立完結性の超過（チェック1: 新規ファイル 12 超）
- 独立検証性の欠如（チェック2: 他タスク成果物に依存する受け入れ基準）
- 自己完結性の欠如（チェック3: 暗黙の前提知識あり）
- 垂直スライス違反（チェック4: 単一レイヤーのみ）
- 実装の競合（チェック8: 同一ファイルへの矛盾する変更）
- QG-1 マッピングとの不整合（チェック10: SR が受け入れ基準に未反映）

warning の基準:
- スコープ外の曖昧さ（チェック5: 他タスクとの照合不一致）
- タスク間インターフェースの不足（チェック6: 先行タスク成果物の言及不足）
- 制約の過不足（チェック7: 汎用的すぎる or 過剰）
- インターフェース境界の不一致（チェック9: 型/関数の不整合）

### 6. 返答

以下のフォーマットで返答する:

```
verdict: {pass / needs_revision}
issues: {指摘件数}
summary: {1行サマリ}
```
