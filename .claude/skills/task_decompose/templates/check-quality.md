以下の手順でイシュー本文の品質・整合性を検証してください:

## 手順

### 1. 入力の読み込み

以下のファイルを全て Read で読み込む:
- {task_bodies_path}（全イシュー本文をまとめたファイル）
- {task_structure_path}（タスクリスト構造 — SR マッピング・依存関係の参照）
- {qg1_result_path}（QG-1 の結果 — SR マッピングサマリの参照）

注意: 設計書（architecture.md, standards.md, development-process.md, detailed-design.md）は渡されない。入力セクションの参照先が実在するかの検証は行わず、参照の形式・具体性・網羅性のみをチェックする。

コマンドがエラーを返した場合、またはデータが空・期待する構造と異なる場合は、以降のステップを続行せず、エラー内容と影響範囲を返却に含めて即座に返却する。

### 2. 個別チェック（各イシューに対して）

| # | チェック | 基準 | 判定方法 |
|---|---------|------|---------|
| 1 | 独立完結性 | 1つのPRとしてレビュー・マージ可能な規模か | 推定変更ファイルが10以下で、スコープが1文で説明できること |
| 2 | 独立検証性 | 受け入れ基準が他タスクの成果物なしに検証可能か | 各基準項目が、このタスクの成果物のみで検証できるか。特に、受け入れ基準に含まれる具体的な値（環境変数値、接続文字列、トークン等）について、その値の取得元タスクが blocked_by チェーンに含まれることをタスク本文から推論して確認する。blocked_by チェーン外のタスクが生成・取得する値を前提とする基準は不適合とする |
| 3 | 自己完結性 | Issue 本文と入力セクションの参照先で作業開始できるか | 受け入れ基準を満たすために必要な設計情報が入力セクションの参照でカバーされているか |
| 3b | 条件分岐の解決性 | Issue 本文に条件分岐（「〜の場合は〜を選択」等）がある場合、選択基準が入力の参照先に含まれるか | 「または代替手段」「利用不可の場合」「確認後に選択」等の条件分岐表現を検出し、対応する選択基準の参照有無を確認 |
| 3c | 外部前提の充足性 | 受け入れ基準が外部サービスの存在・接続を前提とする場合（DB マイグレーション、OAuth 認証フロー等）、そのサービスのセットアップが blocked_by チェーンの先行タスク（executor: human）の受け入れ基準に含まれているか | blocked_by チェーンを辿り、外部サービスセットアップタスクの存在と、当該サービスのクレデンシャル設定が受け入れ基準に含まれていることを確認 |
| 4 | 検証可能性 | タスクの成果物がそのスコープ内で独立に検証可能か | エンドツーエンドの垂直スライスは求めない。レイヤー境界で分割されたタスクは、そのレイヤー内での検証（例: unit test, integration test, E2E test）で可とする |
| 5 | ブロック記述の十分性 | 先行タスクの成果物概要がブロック欄に記述されているか | blocked_by のタスクが提供する成果物の概要がブロック欄に含まれているか（例: 「Task 001（認証付きデプロイ）— 認証基盤・DB 接続設定」） |
| 6 | 入力参照の具体性 | 入力セクションの参照がセクション番号＋概要を含み、タスクの受け入れ基準を満たすために必要な設計書セクションが網羅されているか | 各参照にファイル名・セクション番号・概要があるか。受け入れ基準の各項目に対応する設計情報の参照が存在するか |

### 3. 横断チェック（全イシューを通して）

| # | チェック | 基準 |
|---|---------|------|
| 7 | 実装の競合 | 複数タスクの入力参照が同一セクションを指し、異なる方針で変更する意図がないか |
| 8 | インターフェース境界の一貫性 | 依存関係のあるタスク間で、ブロック欄の成果物概要と後続タスクの入力参照が整合しているか |
| 9 | QG-1 マッピングとの整合 | qg1-result.md の SR マッピングサマリで「SR-X は Task N が担当」とされたものが、Task N の受け入れ基準に反映されているか |

### 4. 結果の保存

チェック結果を {findings_save_path} に Write で保存する:

```markdown
# QG-2: イシュー品質・整合性検証結果

## 判定

{pass / needs_revision}

## 個別チェック結果

{指摘がない場合は「指摘なし」}

### Task {NNN}: チェック{X} — {チェック名}

- **重要度**: {critical / warning}
- **内容**: {具体的な問題の説明}
- **修正案**: {具体的な修正提案}

## 横断チェック結果

{指摘がない場合は「指摘なし」}

### チェック{X}: {チェック名}

- **重要度**: {critical / warning}
- **対象タスク**: Task {NNN}, Task {MMM}
- **内容**: {具体的な問題の説明}
- **修正案**: {具体的な修正提案}

## サマリ

- 個別チェック: {N}件（対象タスク数 × 6項目）
- 横断チェック: 3件
- 指摘: {N}件（critical: {N}, warning: {N}）
```

### 5. 判定基準

- **pass**: 指摘が 0 件
- **needs_revision**: 指摘が 1 件以上

critical の基準:
- 独立完結性の超過（チェック1: 推定変更ファイル10超過またはスコープが1文で説明不可）
- 独立検証性の欠如（チェック2: 他タスク成果物に依存する受け入れ基準）
- 自己完結性の欠如（チェック3: 受け入れ基準に対応する入力参照の欠落）
- 条件分岐の未解決（チェック3b: 選択基準の参照なしの条件分岐）
- 外部前提の未充足（チェック3c: 外部サービスを前提とする受け入れ基準に対応するセットアップタスクが blocked_by チェーンに存在しない）
- 検証手段の欠如（チェック4: スコープ内で検証手段がない）
- 実装の競合（チェック7: 同一セクションへの矛盾する変更意図）
- QG-1 マッピングとの不整合（チェック9: SR が受け入れ基準に未反映）

warning の基準:
- ブロック記述の不足（チェック5: 先行タスク成果物概要の欠落）
- 入力参照の曖昧さ（チェック6: セクション番号・概要の欠落、参照の網羅性不足）
- インターフェース境界の不一致（チェック8: ブロック成果物概要と入力参照の不整合）

### 6. 返答

以下のフォーマットで返答する:

```
verdict: {pass / needs_revision}
issues: {指摘件数}
summary: {1行サマリ}
```
