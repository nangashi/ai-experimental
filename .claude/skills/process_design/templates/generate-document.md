以下の手順で開発プロセス設計書を生成してください:

## 手順

### 1. テンプレートの読み込み

{template_path} を Read で読み込み、出力フォーマットと記入ガイダンスを把握する。

### 2. 入力の把握

{arch_path} を Read で読み込み、以下を把握する:
- 技術スタック
- コンポーネント構成
- 横断的関心事（P3 への引き継ぎ事項）
- デプロイ戦略
- ホスティングプラットフォーム

{std_path} を Read で読み込み、以下を把握する:
- Linter/Formatter 選定結果
- コーディング規約サマリ
- ディレクトリ構成
- 規約の自動強制サマリ
- P3 への引き継ぎ事項

{req_path} を Read で読み込み、以下を把握する:
- 非機能要件（NFR一覧）
- 制約

### 3. 設計決定事項の把握

{decisions_path} を Read で読み込み、Phase 1 で確定した全情報を把握する:
- グループ別プロセス一覧（各項目の確定内容 + ADR 参照 + 根拠）
- 品質チェック配置サマリ（P2 品質ツール + P3 テスト/CI を統合）
- N/A 項目一覧と理由
- P1/P2 引き継ぎ事項の反映状況

### 4. ADRの参照

{adr_dir} を Glob で確認し、存在するADRファイルの「決定」セクションを Read で把握する。

### 5. 開発プロセス設計書の生成

テンプレートに従い、設計決定事項・入力情報・ADRを統合して開発プロセス設計書を生成する。

生成ルール:
- 各セクションの内容は {decisions_path} の確定情報に忠実に従う
- ADRが存在する選定はADRファイルへの参照を含める（例: `ADR-0001`）
- 「該当なし」のセクションは省略せず「該当なし」と明記する
- 曖昧な記述を避け、具体的なツール名・設定方針を記述する
- 「AまたはB」のような未確定の選択肢を残さない
- 品質チェック配置サマリで commit 時/push 時/CI のチェック重複がないことを確認する
- P1/P2 トレーサビリティ: P1/P2 決定と導出された P3 プロセスの対応をセクション 7 に記載する
- AI 統合プロセスが省略された場合はセクション 5.5〜5.7 を「省略（AI 統合は MVP 後に決定）」と記載する

### 6. 保存

生成した開発プロセス設計書を {output_save_path} に Write で保存する。ステータスは `Draft` とする。

### 7. 返答

以下のフォーマットで返答する:

```
result: success
sections_completed: {完了セクション数}
quality_checks: {品質チェック配置サマリの行数}
p1p2_traceability: {トレーサビリティテーブルの行数}
```
