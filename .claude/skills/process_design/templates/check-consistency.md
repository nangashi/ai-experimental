以下の手順で開発プロセス設計書の整合性チェックを実行してください:

## 手順

### 1. 入力の読み込み

以下のファイルを全て Read で読み込む:
- {design_path}（開発プロセス設計書）
- {arch_path}（アーキテクチャ設計書）
- {std_path}（開発規約書）
- {req_path}（要件定義書）
- {adr_dir} 配下の全ADRファイル（Glob で検索）

### 2. 整合性チェック

以下の4カテゴリのチェックを実行する。

#### カテゴリA: プロセス間整合性

開発プロセス設計書内のセクション間の整合性を検証する:

| # | チェック | 内容 |
|---|---------|------|
| A-1 | Git↔CI/CD | Git ワークフロー（ブランチ戦略、マージ方式）と CI トリガー・CD デプロイ条件が整合しているか |
| A-2 | テスト↔CI | テスト戦略（レベル別責務、実行戦略）と CI パイプラインのテストステージが整合しているか |
| A-3 | 環境↔CD | 環境管理（dev/staging/prod）と CD の環境別デプロイフローが整合しているか |
| A-4 | シークレット統一 | アプリケーションのシークレット管理と CI/CD のシークレット管理に矛盾がないか |
| A-5 | 品質チェック重複 | 品質チェック配置サマリ内で同一チェックが不要に複数タイミングに配置されていないか |
| A-6 | リリース↔CD | リリースプロセス（バージョニング、リリースフロー）と CD のデプロイトリガーが整合しているか |

#### カテゴリB: architecture.md との整合性

| # | チェック | 内容 |
|---|---------|------|
| B-1 | ホスティング↔CD | CD のデプロイ先がアーキテクチャのホスティングプラットフォームと一致しているか |
| B-2 | 技術スタック↔テスト FW | テストフレームワーク選定がアーキテクチャの技術スタックと互換性があるか |
| B-3 | デプロイ戦略一致 | CD 戦略がアーキテクチャのデプロイ戦略と整合しているか |
| B-4 | 横断的関心事の反映 | アーキテクチャの横断的関心事（P3 引き継ぎ事項）がプロセスに反映されているか |
| B-5 | ADR一致 | プロセスが ADR の決定内容と一致しているか |

#### カテゴリC: standards.md との整合性

| # | チェック | 内容 |
|---|---------|------|
| C-1 | Linter/Formatter↔CI | P2 で選定した Linter/Formatter が CI パイプラインの lint ステージで使用されているか |
| C-2 | 品質チェック配置の一貫性 | P2 の自動強制サマリ（保存時/commit 時のチェック）と P3 の品質チェック配置サマリに矛盾・重複がないか |
| C-3 | ディレクトリ↔テスト配置 | P2 のディレクトリ構成とテストファイルの配置方針が整合しているか |
| C-4 | P2 引き継ぎ反映 | standards.md の「P3 への引き継ぎ事項」がプロセスに反映されているか |

#### カテゴリD: requirements.md との整合性

| # | チェック | 内容 |
|---|---------|------|
| D-1 | NFRの反映 | パフォーマンス・可用性関連の NFR がテスト戦略・監視設計に反映されているか |
| D-2 | 制約の遵守 | プロセスが要件定義書の制約に違反していないか |

### 3. 結果の保存

チェック結果を {findings_save_path} に Write で保存する:

```markdown
# 整合性チェック結果

## 判定

{pass / needs_revision}

## 指摘一覧

{指摘がない場合は「指摘なし」}

### {チェックID}: {チェック名}

- **重要度**: {critical / warning}
- **内容**: {具体的な不整合の説明}
- **修正案**: {具体的な修正提案}

## サマリ

- チェック項目: {N}件
- 指摘: {N}件（critical: {N}, warning: {N}）
```

### 4. 判定基準

- **pass**: critical が 0 件
- **needs_revision**: critical が 1 件以上

critical の基準:
- Git↔CI/CD の不整合（A-1）
- ホスティング↔CD の矛盾（B-1）
- デプロイ戦略の矛盾（B-3）
- ADRとの矛盾（B-5）
- Linter/Formatter の CI 不使用（C-1）
- 制約違反（D-2）

warning の基準:
- テスト↔CI の部分的不整合（A-2）
- 環境↔CD の部分的不整合（A-3）
- シークレット管理の不統一（A-4）
- 品質チェックの重複（A-5）
- リリース↔CD の不整合（A-6）
- 技術スタック↔テスト FW の互換性懸念（B-2）
- 横断的関心事の部分的な未反映（B-4）
- 品質チェック配置の部分的不一致（C-2）
- ディレクトリ↔テスト配置の不整合（C-3）
- P2 引き継ぎの部分的な未反映（C-4）
- NFR の部分的な未反映（D-1）

### 5. 返答

以下のフォーマットで返答する:

```
verdict: {pass / needs_revision}
issues: {指摘件数}
summary: {1行サマリ}
```
