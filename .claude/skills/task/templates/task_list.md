# task_list.md テンプレートガイド

タスクの「どう進めるか・今どこか」を管理するファイル。Phase 3 で調査結果を基に作成し、Phase 4 の作業中に継続的に更新する。

## 構造

```markdown
# タスクリスト

## 1. [ ] {タスク名}

**依存**
- なし

**入力**
- {このタスクの作業開始時に読むファイル}

**やること**
- {具体的な作業内容1}
- {作業内容2}

**完了条件**
- {完了と言える条件1}
- {条件2}

**成果物**
- `outputs/xxx.md`（予定）

**申し送り**

## 2. [ ] {タスク名}

**依存**
- タスク1

**入力**
- `outputs/xxx.md`（タスク1の成果物）

**やること**
- ...

**完了条件**
- ...

**成果物**
- `outputs/yyy.md`（予定）

**申し送り**
```

## 記入ガイド

### タスク見出し

`## {番号}. [{状態}] {タスク名}`

- 番号は連番。計画変更で途中挿入する場合は `2a.` 等ではなく全体を振り直す。振り直し時は依存欄の番号参照も全て新番号に更新すること
- 状態: `[ ]`（未完了）または `[x]`（完了）
- タスク名: 作業内容を端的に表す名前

### 依存

- このタスクの開始前に完了している必要があるタスクを列挙する
- `タスク1`, `タスク3` のようにタスク番号で記載する
- 依存がなければ「なし」と記入する（最初から開始可能）
- 依存が全て `[x]` のタスクは実行可能。実行可能なタスクが複数あればサブエージェントで並列実行する

### 入力

- このタスクの作業開始時に読むべきファイルを箇条書きで列挙する
- ファイルパスで記載する: `task_definition.md`, `research/xxx.md`, `src/services/auth.ts` 等
- 前タスクの成果物を参照する場合はそのパスを記載する
- 計画時点でパスが未確定の場合は「タスクNの成果物」のように書き、作業時に具体パスへ更新する
- タスクの背景理解が必要な場合は `task_definition.md` を含める（全タスクに必須ではない）
- 入力がないタスク（新規作成等）は「なし」と記入する

### やること

- タスクで行う具体的な作業を箇条書きで列挙する
- 「何をするか」を明確に。手順書ではなく作業の要点を記載する

### 完了条件

- このタスクが完了と言える条件を箇条書きで列挙する
- 曖昧な表現を避け、確認可能な形で記述する

良い例:
- `依存関係の一覧がファイルに記録されている`
- `全テストがパスする`

悪い例:
- `ちゃんと調べる`
- `問題がない`

### 成果物

- 計画時に予定パスを記入する: `outputs/xxx.md`（予定）
- サブエージェントがこのパスに出力する。完了後に「（予定）」を外して確定する
- 後続タスクの入力欄からこの予定パスを参照できるため、計画時に具体パスを決めておく
- 成果物がないタスク（リファクタ、設定変更等）は「なし」と記入する

### 申し送り

- タスク固有の文脈メモ。作業中に気づいたこと、次のタスクへの影響、計画変更の経緯など
- 再開時にこのタスク前後の申し送りを読むことでコンテキストを復帰する
- 申し送りがなければ空欄のままでよい

### 更新タイミング

| イベント | 更新内容 |
|---------|---------|
| タスク完了 | チェックボックスを `[x]` に、成果物・申し送りを記入 |
| 作業中の発見 | 該当タスクの申し送りに追記 |
| 計画変更 | 作業中タスクの申し送りに変更経緯を記録。未着手タスクを差し替え |
