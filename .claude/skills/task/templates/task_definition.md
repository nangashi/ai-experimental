# task_definition.md テンプレートガイド

タスクの「何を・なぜ」を定義するファイル。Phase 1 のヒアリング結果を基に作成する。作成後は基本的に変更しない（ゴールのチェックボックスのみ Phase 5 で更新）。

## 構造

```markdown
# {task_name}

## 目的

{なぜこのタスクが必要か。背景と動機を簡潔に}

## ゴール

- [ ] {ゴール1。検証可能な表現で記述する}
- [ ] {ゴール2}

## スコープ

{対象範囲を具体的に。ファイル名、モジュール名、機能名など}

{以下はヒアリングで言及があった場合、またはPhase 2後の確認で追加された場合のみ記載。なければセクションごと省略}

## スコープ外

{明示的にやらないこと。「今回はやらない」の境界を明確にする}

## 制約

{技術的制約、依存関係、使用ツールの制限など}

## 前提条件

{このタスクが前提としていること}

## アイデア

{ユーザーが持つアプローチや実現方法の構想。計画時に考慮する}
```

## 記入ガイド

### 目的

- 「何をするか」ではなく「なぜ必要か」を書く
- 1-3文で簡潔に

### ゴール

- 各項目は検証可能な形で記述する。「改善する」「整理する」ではなく、達成状態を具体的に
- Phase 5（自己チェック）でこのチェックリストを検証するため、曖昧な表現は避ける
- 成果物がある場合、Phase 5 で `→ ファイルパス` を追記する

良い例:
- `[ ] AuthServiceが外部依存なしで単体テスト可能になっている`
- `[ ] API仕様書が全エンドポイントを網羅している`

悪い例:
- `[ ] コードを改善する`（何が改善か不明）
- `[ ] テストを書く`（どこまで書けば完了か不明）

### スコープ / スコープ外

- スコープは「何が対象か」を具体的に列挙する
- スコープ外は「やりたくなるが今回はやらないこと」を明示する。作業中のスコープクリープを防ぐ

### 制約 / 前提条件

- 該当がなければセクションごと省略してよい
- 制約: 外部から与えられた条件（技術選定、互換性要件、時間制約など）
- 前提条件: このタスクが成立するために必要な前提（他タスクの完了、環境の状態など）

### アイデア

- Phase 2（調査）後にユーザーから提示された場合に追記する。Phase 1 時点では存在しない
- ユーザーが持つアプローチや実現方法の構想を記録する
- 計画エージェントはこのセクションを考慮してタスク分解を行う
- 制約（必ず従う）とは異なり、アイデアは計画時の参考情報。より良い方法がある場合は計画エージェントが代替案を提示してよい
