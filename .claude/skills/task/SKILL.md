計画駆動で作業を進め、セッションを跨いで再開可能なタスク実行フレームワーク。タスク定義・調査・計画・作業・検証・報告の各フェーズを人間のレビューを挟みながら進行する。

## 使い方

```
/task
```

引数なし。起動後にタスク一覧を表示し、新規作成または既存タスクの再開を選択する。

## パス変数

- `{skill_dir}`: このファイルが存在するディレクトリの絶対パス
- `{task_dir}`: `.tasks/{YYYYMMDD}-{task_name}/` の絶対パス
- `{task_name}`: タスク内容から生成する英数小文字+アンダースコアのスラッグ（30文字以内）

## Phase 0: 初期化

### Step 1: タスク一覧の表示

`.tasks/` 配下のディレクトリを日付降順で最新10件表示する。

```
最近のタスク:
  1. 20260225-refactor_auth_module
  2. 20260220-add_validation_layer
  3. ...

番号を入力して再開、または "new" で新規作成:
```

ユーザーの応答に応じて分岐する:
- **番号入力**: 該当タスクを `{task_dir}` に設定し、Step 3（再開）へ
- **"new"**: Step 2（新規作成）へ
- **キーワード入力**: ディレクトリ名に部分一致するタスクを絞り込み表示。完全一致が1件なら即再開。複数一致なら再度選択を求める

### Step 2: 新規作成

Phase 1 へ進む。Phase 1 のヒアリング後に `{task_name}` を命名する。

### Step 3: 再開

各 Phase の出力ファイルを順にチェックし、存在しないものがあればその Phase から開始する。既に存在する場合は「{ファイル名} が既にあります。スキップしますか？」とユーザーに確認し、応答に従う。

**チェック順序:**

1. `task_definition.md` → なければ Phase 1
2. `research/` 配下のファイル → なければ Phase 2
3. `task_list.md` → なければ Phase 3
4. 上記が全て存在する場合 → タスク状態抽出（後述）で現状を把握し、ユーザーに報告

Phase 4 以降の再開時は、サブエージェント（Task tool, subagent_type: Explore）に `{task_dir}/task_list.md` を読ませて以下を返させる:

- 実行可能タスク一覧（未完了かつ依存が全て完了）。各タスクについて、依存チェーン上（直接・間接に依存する完了タスク）の申し送りを付与する

返却後、各実行可能タスクについて成果物の予定パスにファイルが既に存在するかをチェックする。存在する場合は「前回の作業が中断した可能性があります」としてユーザーに報告し、再実行/スキップの判断を求める。

この返却結果をユーザーに報告してから作業を再開する。

## Phase 1: タスク定義

**目的:** 対話を通じてタスクの目的・ゴール・スコープを明確にし、ぶれない基盤を作る

### Step 1: ヒアリング

ユーザーとの対話でタスクの内容を明確にする。以下の項目を確認する:

**必須項目**（全て確認できたら Step 2 に進む）:
- **目的**: なぜこのタスクが必要か
- **ゴール**: 何が達成されたら完了か（検証可能な形で）
- **スコープ**: 対象範囲

**任意項目**（ユーザーが言及した場合のみ記録。明示的に確認しなくてよい）:
- **スコープ外**: 明示的にやらないこと
- **制約**: 技術的制約、依存関係など
- **前提条件**: このタスクが前提としていること

一度に全項目を質問するのではなく、ユーザーの説明を聞きながら自然に確認していく。必須項目に不足があれば追加で質問する。

### Step 2: task_name の命名とディレクトリ作成

ヒアリング内容からタスクの本質を表す `{task_name}` を生成する。

- 英数小文字+アンダースコア、30文字以内
- タスク内容を端的に表す命名にする

`{task_dir}` = `.tasks/{YYYYMMDD}-{task_name}/` を作成する（`{YYYYMMDD}` は当日の日付）。

### Step 3: task_definition.md の作成

`{skill_dir}/templates/task_definition.md` を Read で読み込み、ガイドに従って `{task_dir}/task_definition.md` を作成する。

作成後、内容をユーザーに提示して確認を求める:

「task_definition.md を作成しました。内容を確認してください。修正があればお知らせください。問題なければ次に進みます。」

- 修正指示あり: task_definition.md を更新して再度確認
- 承認: Phase 2 へ

---

## Phase 2: 調査

**目的:** タスク遂行に必要な情報を収集し、計画の基盤を作る

### Step 1: 調査の実施

タスク定義を基に調査すべき項目を洗い出し、情報を収集する。

調査の方針:
- サブエージェントを活用してメインエージェントのコンテキスト消費を抑える
- 実装や影響範囲の把握のためコードベースを深く調査する
- 方針決定のため最新の情報を Web で積極的に収集する
- 情報収集に MCP が利用可能であれば積極的に利用する

調査結果は `{task_dir}/research/` 配下にトピック名ベースで保存する: `research/{topic-slug}.md`（例: `research/auth-dependencies.md`, `research/api-endpoints.md`）。

### Step 2: 調査結果のレビュー

調査結果の要点をユーザーに提示し、レビューを求める:

「調査が完了しました。結果の要点:
- ...
- ...

詳細は research/ 配下のファイルを参照してください。修正・追加調査があればお知らせください。問題なければ計画作成に進みます。」

- 追加調査指示: Step 2 に戻る
- 承認: Step 3 へ

### Step 3: アイデアの確認

調査結果を踏まえて、ユーザーにアプローチや実現方法のアイデアがあるか確認する:

「調査結果を踏まえて、アプローチや実現方法について何かアイデアはありますか？ なければそのまま計画作成に進みます。」

- **アイデアあり**: `{task_dir}/task_definition.md` に「## アイデア」セクションを追記し、Phase 3 へ
- **なし**: そのまま Phase 3 へ

---

## Phase 3: 計画

**目的:** 調査結果を基に具体的なタスクリストを作成し、作業前にユーザーと合意する

### Step 1: task_list.md の作成

サブエージェント（Task tool, subagent_type: general-purpose）に計画作成を委譲する。`{skill_dir}/templates/plan_agent.md` を Read で読み込み、その内容をサブエージェントへの指示として使用する。

サブエージェントに渡す情報:
- `{task_dir}/task_definition.md` のファイルパス
- `{task_dir}/research/` 配下の調査ファイルパス一覧
- `{skill_dir}/templates/task_list.md` のファイルパス（フォーマットガイド）
- 出力先: `{task_dir}/task_list.md`

### Step 2: レビュー

task_list.md の内容をユーザーに提示し、レビューを求める:

「task_list.md を作成しました。内容を確認してください。修正があればお知らせください。作業に進んでよければ指示してください。」

- 修正指示あり: task_list.md を更新して再提示
- 作業指示: Phase 4 へ

**作業開始の明示的な指示があるまで作業に着手しない。**

---

## Phase 4: 作業

**目的:** タスクリストに従って作業を実行し、進捗を記録する

### 役割分担

- **メインエージェント**: タスクの実行指揮、完了条件の検証、task_list.md の状態管理
- **サブエージェント**: 個別タスクの実行、成果物の生成

### 作業の進め方

1. サブエージェント（Task tool, subagent_type: Explore）に `{task_dir}/task_list.md` を読ませて以下を返させる:
   - 実行可能タスク一覧（未完了 `[ ]` かつ依存タスクが全て完了 `[x]`）。各タスクのタスク番号とタスク名を含める
   - 各実行可能タスクについて、依存チェーン上（直接・間接に依存する完了タスク）の申し送りを結合したリスト
2. 実行可能タスクをサブエージェント（Task tool, subagent_type: general-purpose）に委譲する。実行可能なタスクが複数あれば並列で委譲する
3. サブエージェントへの指示には以下を伝える:
   - `{task_dir}/task_list.md` のファイルパスと該当タスク番号（サブエージェント側で該当セクションを読み込む）
   - 作業ディレクトリのパス（`{task_dir}`）
   - 該当タスクの依存チェーン上の申し送りリスト（Step 1 で取得したもの）
   - 成果物は成果物欄の予定パスに出力すること
   - 判断が必要な状況に遭遇した場合は作業を中断し、判断が必要な内容を返却に含めて即座に返却すること
   - コマンドがエラーを返した場合、またはデータが空・期待する構造と異なる場合は、以降のステップを続行せず、エラー内容と影響範囲を返却に含めて即座に返却すること
4. サブエージェントの返却後、メインエージェントが簡易チェックを行う（成果物が所定パスに存在し、明らかな欠落がないこと）。内容の妥当性は Phase 5 で検証する:
   - **合格**: task_list.md を更新（`[x]`、成果物の確定、申し送り記入）
   - **未達**: 未達の完了条件と具体的な問題点を添えてサブエージェントを再実行する
   - **エスカレーション**: サブエージェントが判断を求めてきた場合、ユーザーに中継して判断を得た後、再実行する
5. タスク完了後、Step 1 に戻り新たに実行可能なタスクを特定して続行する

### 計画変更

作業中に以下の状況が発生した場合、計画変更を提案する:

- タスクの前提が崩れた（調査で判明した事実と異なる等）
- より効率的なアプローチが見つかった
- スコープの拡大・縮小が必要

**計画変更の手順:**

1. 作業中のタスクの申し送りに「計画変更を提案: {理由}」と記入する
2. ユーザーに変更内容と理由を報告し、承認を求める
3. スコープまたはゴールの変更を伴う場合、task_definition.md のゴール・スコープ・スコープ外を更新し、ユーザーの承認を得る
4. 必要に応じて Phase 2（調査）に戻る
5. 承認後、サブエージェント（Task tool, subagent_type: general-purpose）に再計画を委譲する。`{skill_dir}/templates/plan_agent.md` を Read で読み込み、その内容をサブエージェントへの指示として使用する。サブエージェントに渡す情報:
   - `{task_dir}/task_definition.md` のファイルパス
   - `{task_dir}/research/` 配下の調査ファイルパス一覧
   - `{skill_dir}/templates/task_list.md` のファイルパス（フォーマットガイド）
   - 現在の `{task_dir}/task_list.md` のファイルパス
   - 変更理由
   - 出力先: `{task_dir}/task_list.md`（上書き）
6. 再計画後の task_list.md をユーザーに提示し、レビューを求める

---

## Phase 5: 独立検証

**目的:** 全タスク完了後、作業コンテキストから独立した視点でゴールの達成を検証する

サブエージェント（Task tool, subagent_type: general-purpose）に検証を委譲する。`{skill_dir}/templates/verify_agent.md` を Read で読み込み、その内容をサブエージェントへの指示として使用する。

サブエージェントに渡す情報:
- `{task_dir}/task_definition.md` のファイルパス
- `{task_dir}/task_list.md` のファイルパス

メインエージェントは成果物パスの抽出や検証結果の解釈を行わない。検証エージェントが成果物パスの抽出・ゴール検証・最終判定を一貫して行い、構造化された結果を返却する。

メインエージェントは返却された判定を機械的に反映する:
- **全ゴール達成（総合判定: はい）**: 返却された対応成果物パスを使い、`task_definition.md` のチェックボックスを `[x]` に更新し、成果物への参照を `→` で追記。Phase 6 へ
- **未達のゴールあり（総合判定: いいえ）**: 返却された未達ゴール一覧と不足内容をそのまま追加タスクとして task_list.md に追加し、Phase 4 に戻る。メインエージェントが未達内容を再解釈・変更しない

---

## Phase 6: 報告・最終承認

**目的:** 作業結果を報告し、ユーザーの最終承認を得る

### Step 1: 完了報告

以下の形式でユーザーに報告する:

```
## 完了報告

### 達成したゴール
- [x] ゴール1 → `outputs/xxx.md`
- [x] ゴール2 → `research/yyy.md`, `outputs/zzz.md`

### 注意事項
- {既知の問題、未解決事項、今後の対応が必要な点}
（なければ「特になし」）
```

ゴール一覧は task_definition.md から、注意事項は task_list.md の各タスクの申し送りから集約する。

### Step 2: 最終承認

ユーザーの承認を得て完了とする。修正指示があれば対応する。
