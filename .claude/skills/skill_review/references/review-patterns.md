# レビューパターン定義

skill_review の設計レビューで使用するパターン定義。各パターンに ID・検出条件・デフォルト重要度・改善方針・根拠を定義する。

---

## SR: 役割分離

### SR-1: 自己検証

- **デフォルト重要度**: 高
- **検出条件**: 同一サブエージェント（または同一 Phase 内の親エージェント）が成果物を**生成**し、かつその成果物の**品質判定・完了判定**を行っている
- **改善方針**: 検証を別サブエージェントに分離する。検証エージェントには先行の生成コンテキストを含めない
- **根拠**: workflow-structure-design.md「独立した分析には構造的分離を使い、指示に頼らない」

### SR-2: 指示的独立性への依存

- **デフォルト重要度**: 中
- **検出条件**: 「独立に」「客観的に」「バイアスなく」等の指示が、同一コンテキスト内で使われている。具体的には:
  - 同一サブエージェント内で先行結果を参照しつつ「独立に評価せよ」と指示
  - 親コンテキスト内で先行 Phase の結果が可視な状態で「独立に判断せよ」と指示
- **改善方針**: 別サブエージェント呼び出しに分離し、先行結果を入力に含めない設計にする
- **根拠**: workflow-structure-design.md「独立した分析には構造的分離を使い、指示に頼らない」— 独立性レベル表（指示的独立性 < コンテキスト独立性 < 知識源独立性）

### SR-3: オーケストレーターによる検証入力のフィルタリング

- **デフォルト重要度**: 中
- **検出条件**: 検証エージェントへの入力を親エージェントが抽出・加工してから渡している。具体的には:
  - 親が成果物から特定箇所を抽出してサブエージェントに渡す
  - 親がサブエージェントの出力を要約・フィルタしてから検証エージェントに渡す
- **改善方針**: 検証エージェントにファイルパスを渡し、検証対象の選定から最終判定までを一貫して委ねる
- **根拠**: workflow-structure-design.md「検証エージェントには検証対象の選定から最終判定までを一貫して委ね」

### SR-4: オーケストレーターによる検証結果の再解釈

- **デフォルト重要度**: 中
- **検出条件**: 検証/レビューエージェントの判定結果を親エージェントが条件付きで変更・上書き・再解釈する記述がある。具体的には:
  - 「検証結果が X の場合は Y として扱う」等の親側での変換ロジック
  - 「検証結果を踏まえて親が最終判定する」等の記述
- **改善方針**: 親の役割は判定結果の機械的な反映に限定する
- **根拠**: workflow-structure-design.md「指揮者の役割は判定結果の機械的な反映に限定する」

## WF: ワークフロー構造

### WF-1: 方針と手順の同時策定

- **デフォルト重要度**: 中
- **検出条件**: 単一の Phase/Step で以下の両方を行っている:
  - 方針レベルの決定（「何をするか」「どの方式を採用するか」の選択）
  - 手順レベルの詳細化（具体的な I/O フォーマット、処理手順、パラメータの決定）
- **改善方針**: 方針合意のステップと手順詳細化のステップを分離する
- **根拠**: workflow-structure-design.md「方針決定と手順策定を分離する」

### WF-2: 楽観フロー

- **デフォルト重要度**: 中
- **検出条件**: 以下の両方に該当する:
  - 全 Phase 遷移が成功前提で記述されている（失敗・差し戻しパスの定義がない）
  - 方向転換コストが高い地点（スコープ確定後、不可逆出力前）にフォールバックがない
- **改善方針**: 少なくとも方向転換コストが高い地点に失敗時の分岐を追加する
- **根拠**: workflow-structure-design.md「ユーザーチェックポイントを方向転換コストに基づいて配置する」

### WF-3: チェックポイント過剰

- **デフォルト重要度**: 低
- **検出条件**: 以下の両方に該当する:
  - Phase 間遷移の大半で AskUserQuestion によるユーザー確認がある
  - 確認内容が形式的（「問題なければ次へ」「ok で進む」等）で、方向転換コストが低い内部処理に対しても確認を求めている
- **改善方針**: 方向転換コストが低い箇所のチェックポイントを削除し、機械的に判定可能な品質検証は自動品質ゲートで代替する
- **根拠**: workflow-structure-design.md「品質検証が機械的に判定可能な場合は、ユーザーチェックポイントの代わりに自動品質ゲートで代替する」

### WF-4: 品質ゲートの欠落

- **デフォルト重要度**: 中
- **検出条件**: Phase 間の遷移で、前 Phase の成果物に対する検証（ユーザー確認・自動チェック・検証サブエージェントのいずれも）なしに次 Phase が成果物を入力として使用している。特に不可逆な出力（ファイル書き出し・デプロイ等）の前
- **改善方針**: 品質ゲート（自動検証またはユーザーチェックポイント）を追加する
- **根拠**: workflow-structure-design.md「ユーザーチェックポイントを方向転換コストに基づいて配置する」

## CT: コンテキスト管理

### CT-1: 先読みロード

- **デフォルト重要度**: 低
- **検出条件**: 後続 Phase でのみ使用されるファイルを、より早い Phase で Read している
- **改善方針**: Read を使用する Phase に移動する（遅延読み込み）
- **根拠**: workflow-context-management.md「参照ファイルは使用するフェーズでのみ読み込む」「コンテキストはJust-in-Timeで取得する」

### CT-2: 親経由のデータ中継

- **デフォルト重要度**: 中
- **検出条件**: サブエージェント A の出力（返答テキストまたはファイル内容）を親が受け取り、サブエージェント B に渡している。ファイルパスの受け渡しではなく、内容自体の中継
- **改善方針**: サブエージェント A がファイルに保存 → サブエージェント B がそのファイルパスを受け取り直接 Read する設計に変更する
- **根拠**: workflow-context-management.md「サブエージェント間のデータ受け渡しは親を中継せずファイル経由で行う」

### CT-3: サブエージェント返答の肥大化

- **デフォルト重要度**: 低
- **検出条件**: Task サブエージェントへの指示に返答サイズの制約がなく、かつサブエージェントが大量のコンテンツ（分析結果、生成物等）を生成する設計。以下の全てに該当:
  - 「〜とだけ返答する」「ファイルに保存」等の返答最小化指示がない
  - サブエージェントの処理内容が生成・分析等のコンテンツ生成を含む
- **改善方針**: 「結果はファイルに保存し、保存パスとサマリのみ返答する」指示を追加する
- **根拠**: workflow-context-management.md「親への返答は要約・メタデータのみにする」

## EV: 評価・レビュー設計

### EV-1: 生成・評価の同一モデル

- **デフォルト重要度**: 中
- **検出条件**: 生成タスクとレビュー/評価タスクが同一モデル指定で実行されている。具体的には:
  - 両方の Task に同じ model パラメータが指定されている
  - 片方または両方に model 指定がなく、デフォルトで同一モデルになる
- **改善方針**: レビュー/評価エージェントに異なるモデルを指定する。モデル分離が困難な場合は、視点転換の明示的フレーミングを追加する
- **根拠**: llm-evaluation-design.md「生成と評価のモデル分離」

### EV-2: 評価基準の生成プロンプト同居

- **デフォルト重要度**: 中
- **検出条件**: 生成用テンプレートに「良い出力の条件」「品質基準」「評価基準」が含まれ、同じサブエージェント内で生成と自己評価を行わせる構造
- **改善方針**: 評価基準を別ファイルに分離し、評価は別サブエージェントで実行する
- **根拠**: llm-evaluation-design.md「生成と評価のモデル分離」— 自己選好バイアスの回避

## PD: プロンプト設計

### PD-1: 探索タスクでの過剰なガイダンス

- **デフォルト重要度**: 低
- **検出条件**: 探索タスク（レビュー・分析・問題発見等）のテンプレート内で、探索方向を示すヒントや例示が3件以上含まれている
- **改善方針**: ヒント・例示を2件以内に削減する
- **根拠**: prompt-design-guidelines.md「探索タスクでの過剰なガイダンスの回避」— 3件以上では LLM が列挙された方向に固着し自発的な探索が狭まる

### PD-2: 概念未定義のまま作業開始

- **デフォルト重要度**: 中
- **検出条件**: ワークフロー内で解釈が揺れうる主要概念（タスク固有の用語、分類カテゴリ、評価基準の用語等）が以下の両方に該当:
  - 定義・スコープ・分類が明示されていない
  - 複数の Phase またはサブエージェントで使用されている
- **改善方針**: ワークフロー冒頭（または専用ステップ）で主要概念を定義し、後続ステップから共通参照させる
- **根拠**: workflow-structure-design.md「主要概念を定義してからワークフローを設計する」
