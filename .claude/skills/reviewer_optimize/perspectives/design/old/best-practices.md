# 観点定義: ベストプラクティス（設計レビュー）

## 概要
設計書が業界標準の設計原則とベストプラクティスに適合しているかを評価する。

## 評価スコープ
- SOLID原則の適用（単一責任、開放閉鎖、リスコフ置換、インターフェース分離、依存性逆転）
- デザインパターンの適切性（パターンの適用/不適用の妥当性）
- エラーハンドリング戦略（エラーの分類、伝播方式、リカバリー戦略）
- ロギング・オブザーバビリティ設計（ログレベル、構造化ログ、トレーシング）
- テスト設計（テスト戦略、テスト可能な設計、モック方針）
- API設計原則（RESTful設計、バージョニング、ページネーション）

## スコープ外
- セキュリティ固有の問題 → security で扱う
- パフォーマンス固有の問題 → performance で扱う
- 既存コードとの一貫性 → consistency で扱う

## ボーナス/ペナルティの判定指針
- **ボーナス対象**: SOLID原則違反やアンチパターンの検出で正解キーに含まれないもの
- **ペナルティ対象**: セキュリティ脆弱性やパフォーマンス問題の指摘（設計原則に無関係）
- **判断に迷うケース**: テスト容易性に関わる設計問題 → ベストプラクティスのスコープ内

## 問題バンク

以下は埋め込み可能な問題のカテゴリと例。実際の埋め込み時はこれらを参考にしつつ、テスト対象文書の文脈に合った具体的な問題を作成する。

| カテゴリ | 問題例 | 深刻度 |
|---------|--------|-------|
| SOLID | 1つのクラスに複数の責務を持たせる設計（SRP違反） | 重大 |
| エラー処理 | エラーハンドリング戦略の欠如 | 重大 |
| テスト | テスト戦略の記載なし | 中 |
| API設計 | RESTful原則に反するエンドポイント設計 | 中 |
