# 観点定義: 保守性（実装レビュー）

## 概要
ソースコードの長期的な保守性・変更容易性を評価する。

## 評価スコープ
- コードの理解容易性（関数の長さ、ネストの深さ、命名の意図明確性）
- 変更影響範囲の限定性（結合度、副作用の有無、インターフェースの安定性）
- テスタビリティ（テスト容易な構造、モック可能性、依存注入）
- 技術的負債の兆候（TODO/FIXME、ワークアラウンド、マジックナンバー）
- ドキュメンテーション（複雑なロジックの説明、APIの使用方法）
- YAGNI違反（使用されていないコード、過剰な抽象化）

## スコープ外
- セキュリティ脆弱性 → security-code で扱う
- パフォーマンス最適化 → performance-code で扱う
- コーディングスタイルの一貫性 → consistency-code で扱う

## ボーナス/ペナルティの判定指針
- **ボーナス対象**: 保守コストを増大させる問題で正解キーに含まれないもの（過剰な複雑性、隠れた依存関係等）
- **ペナルティ対象**: セキュリティやパフォーマンスの指摘
- **判断に迷うケース**: リファクタリング提案 → 保守性向上の文脈ならスコープ内

## 問題バンク

以下は埋め込み可能な問題のカテゴリと例。実際の埋め込み時はこれらを参考にしつつ、テスト対象文書の文脈に合った具体的な問題を作成する。

| カテゴリ | 問題例 | 深刻度 |
|---------|--------|-------|
| 複雑性 | 循環的複雑度が高い関数 | 重大 |
| YAGNI | 使用されていないインターフェース/抽象クラス | 中 |
| 依存 | ハードコードされた外部サービスのURL | 中 |
| 可読性 | 100行を超える関数 | 軽微 |
