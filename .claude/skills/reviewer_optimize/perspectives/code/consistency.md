# 観点定義: 一貫性（実装レビュー）

## 概要
ソースコードが既存のコードベースのコーディングスタイル、パターン、規約と整合しているかを評価する。

## 評価スコープ
- コーディングスタイル（インデント、括弧、空白、コメントのスタイル）
- 命名規約の準拠（変数、関数、クラス、定数の命名パターン）
- エラーハンドリングパターンの統一（例外処理、エラーレスポンスの形式）
- インポート・依存関係の整理（インポート順序、未使用インポート、循環依存）
- 既存ユーティリティの活用（車輪の再発明を避ける、共通関数の使用）

## スコープ外
- セキュリティ脆弱性 → security-code で扱う
- パフォーマンス最適化 → performance-code で扱う
- 設計レベルのアーキテクチャ → consistency-design で扱う

## ボーナス/ペナルティの判定指針
- **ボーナス対象**: 既存コードとの不整合で正解キーに含まれないパターン違反の検出
- **ペナルティ対象**: セキュリティやパフォーマンスの指摘
- **判断に迷うケース**: コーディングスタイルの「改善提案」→ 既存スタイルとの整合性の文脈ならスコープ内

## 問題バンク

以下は埋め込み可能な問題のカテゴリと例。実際の埋め込み時はこれらを参考にしつつ、テスト対象文書の文脈に合った具体的な問題を作成する。

| カテゴリ | 問題例 | 深刻度 |
|---------|--------|-------|
| 命名 | 同一概念に異なる名前を使用 | 中 |
| スタイル | インデントやブレースのスタイルが混在 | 軽微 |
| エラー処理 | 例外処理パターンの不統一 | 中 |
| ユーティリティ | 既存ユーティリティの未使用（車輪の再発明） | 中 |
