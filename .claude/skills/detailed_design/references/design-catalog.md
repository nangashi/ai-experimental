# 詳細設計項目カタログ

アーキテクチャ設計書・開発規約書・開発プロセス設計書の設計方針を、実装可能な具体的仕様に展開するための設計項目カタログ。4グループ + 2付加項目で構成する。

## P1/P2/P3 との性質の違い

| 観点 | P1/P2/P3 | detailed_design |
|------|----------|-----------------|
| 決定の性質 | 「どれを選ぶか」「どう書くか」「どう回すか」 | 「何を作るか」。上流決定を具体的仕様に展開 |
| ADR 密度 | 高〜低 | なし（技術選定は上流で完了） |
| 対話パターン | 選択肢提示 → 決定 | 導出仕様提示 → 確認/修正 |

## グループ編成の根拠

- **DB 先行**: エンティティ型がサービス層のメソッドシグネチャを規定するため、最初に確定する
- **Service → API**: Service メソッドのシグネチャが API 契約の入出力を規定する
- **バッチは条件付**: 全プロジェクトに必要ではないため、コンポーネント構成から自動判定する
- **型定義・状態遷移は付加**: Group 1-3 の成果物から自動合成する独立グループではない

## Group 1: データベース設計（6項目）

テーブル定義・リレーション・インデックスを含む物理データモデルの確定。全後続グループの前提。

| # | 項目 | 導出元 | 備考 |
|---|------|--------|------|
| 1 | テーブル定義 | architecture.md エンティティ概要 + standards.md スキーマ設計方針 | カラム名、型、制約、デフォルト値 |
| 2 | Enum/ステータス値定義 | architecture.md ステータス管理 | 値の列挙。状態遷移設計のベース |
| 3 | リレーション定義 | エンティティ間関係の導出 | FK制約、ON DELETE/UPDATE |
| 4 | インデックス設計 | architecture.md インデックス方針 + SR アクセスパターン | 名前、対象カラム、種別 |
| 5 | マイグレーション計画 | development-process.md マイグレーション戦略 | ファイル分割粒度、命名、ロールバック手順 |
| 6 | クエリパターン | SR + コンポーネントトレーサビリティ | Repository メソッドが必要とするクエリの概要 |

## Group 2: サービス層設計（6項目）

Service コンポーネントのメソッドシグネチャ・ビジネスルール・DI 境界の確定。Group 1 のエンティティ型に依存。

| # | 項目 | 導出元 | 備考 |
|---|------|--------|------|
| 1 | メソッドシグネチャ | SR + コンポーネント責務 | 名前、引数型、戻り値型、エラーケース |
| 2 | ビジネスルール仕様 | SR の記述 + エラーハンドリング方針 | 判定条件、処理分岐 |
| 3 | トランザクション境界 | standards.md トランザクション管理方針 | 対象操作、分離レベル |
| 4 | DI インターフェース定義 | architecture.md DI 設計方針 | 抽象化境界のメソッドシグネチャ |
| 5 | ファクトリ関数設計 | architecture.md Handler-Service 接続 | 依存注入ポイント |
| 6 | 状態遷移ロジック（条件付） | Group 1 の Enum 定義 | 遷移ルール、ガード条件、副作用 |

## Group 3: API インターフェース設計（4項目）

Server Action・Route Handler・Server Component のデータ契約の確定。Group 2 のメソッドシグネチャに依存。

| # | 項目 | 導出元 | 備考 |
|---|------|--------|------|
| 1 | Server Action 定義 | SR + Handler コンポーネント | 入力スキーマ（Zod）、出力型、認証、エラー |
| 2 | Route Handler 定義 | IR + Handler コンポーネント | HTTP メソッド、パス、リクエスト/レスポンス |
| 3 | Server Component データ契約 | コンポーネント責務 + SR | データ形状、ソース Service メソッド |
| 4 | バリデーションスキーマ定義 | standards.md バリデーション戦略 | Zod スキーマの具体的定義 |

## Group 4: バッチ処理・非同期処理設計（4項目、条件付）

スケジュールジョブ・非同期処理コンポーネントが存在する場合のみ対象。Group 1 のデータモデルと Group 2 の Service メソッドに依存。

| # | 項目 | 導出元 | 備考 |
|---|------|--------|------|
| 1 | ジョブ定義 | architecture.md バッチ/非同期コンポーネント | 名前、トリガー、処理内容、タイムアウト |
| 2 | データフロー | SR + コンポーネント依存関係 | 入力 → 処理 → 出力 |
| 3 | べき等性設計 | ジョブ特性の分析 | 重複実行の安全性保証方法 |
| 4 | リトライ/バックオフ | NFR + ジョブ特性 | ジョブごとの具体値 |

---

## グループ間依存

```
Group 1 (DB) → Group 2 (Service) → Group 3 (API)
Group 1 (DB) → Group 4 (Batch, 条件付)
```

Group 2 は Group 1 のエンティティ型に依存。Group 3 は Group 2 のメソッドシグネチャに依存。Group 4 は Group 1 のデータモデルと Group 2 の Service メソッドに依存。
