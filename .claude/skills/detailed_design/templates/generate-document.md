以下の手順で詳細設計書を生成してください:

## 手順

### 1. テンプレートの読み込み

{template_path} を Read で読み込み、出力フォーマットと記入ガイダンスを把握する。

### 2. 入力の把握

{req_path} を Read で読み込み、以下を把握する:
- 機能要件テーブル（SR一覧）
- 非機能要件テーブル（NFR一覧）
- インターフェース要件テーブル（IR一覧）

{arch_path} を Read で読み込み、以下を把握する:
- 技術スタック
- コンポーネント一覧（名前、責務、技術、関連SR）
- データスキーマ設計（エンティティ概要、スキーマ設計方針）
- 認証・アクセス制御設計
- エラーハンドリング方針
- DI 設計方針
- ディレクトリ構造
- 要件トレーサビリティ

{std_path} を Read で読み込み、以下を把握する:
- 命名規則
- スキーマ設計方針
- バリデーション戦略
- エラーハンドリングパターン
- トランザクション管理方針

{dp_path} を Read で読み込み、以下を把握する:
- マイグレーション戦略
- テスト戦略サマリ

### 3. 設計決定事項の把握

{decisions_path} を Read で読み込み、Phase 1 で確定した全情報を把握する:
- テーブル定義（カラム名、型、制約、デフォルト値）
- Enum/ステータス値定義
- リレーション定義
- インデックス設計
- マイグレーション計画
- Service メソッドシグネチャ
- ビジネスルール仕様
- トランザクション境界
- DI インターフェース定義
- ファクトリ関数設計
- Server Action / Route Handler 定義
- Server Component データ契約
- バリデーションスキーマ定義
- バッチ処理定義（該当する場合）
- 状態遷移ロジック（該当する場合）

### 4. 詳細設計書の生成

テンプレートに従い、設計決定事項・入力情報を統合して詳細設計書を生成する。

生成ルール:
- 各セクションは {decisions_path} の確定情報に忠実に従う
- 型定義カタログ（セクション5）は セクション1-3 から自動合成する
- 状態遷移設計（セクション6）は Group 1-3 の該当項目から合成する
- 条件付セクション（4, 6）は該当なしの場合「該当なし」と明記
- トレーサビリティ（セクション7）は全 SR/NFR/IR の網羅を保証する
- 曖昧な記述（「適切に処理する」等）を避け、具体的な仕様を記述する
- 「AまたはB」のような未確定の選択肢を残さない
- 全仕様に導出元（SR-ID / architecture.md セクション番号 / standards.md セクション番号）を付記する

### 5. 保存

生成した詳細設計書を {output_save_path} に Write で保存する。ステータスは `Draft` とする。

### 6. 返答

以下のフォーマットで返答する:

```
result: success
sections_completed: {完了セクション数}
sr_traceability: {SR トレーサビリティテーブルの行数}
tables_defined: {テーブル定義数}
methods_defined: {Service メソッド定義数}
api_contracts_defined: {API 契約定義数（Server Action + Route Handler）}
```
