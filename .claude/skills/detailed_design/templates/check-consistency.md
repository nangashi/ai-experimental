以下の手順で詳細設計書の整合性チェックを実行してください:

## 手順

### 1. 入力の読み込み

以下のファイルを全て Read で読み込む:
- {design_path}（詳細設計書）
- {arch_path}（アーキテクチャ設計書）
- {std_path}（開発規約書）
- {req_path}（要件定義書）

コマンドがエラーを返した場合、またはデータが空・期待する構造と異なる場合は、以降のステップを続行せず、エラー内容と影響範囲を返却に含めて即座に返却する。

### 2. 整合性チェック

以下の4カテゴリのチェックを実行する。

#### カテゴリA: 内部整合性（detailed-design.md 内）

| # | チェック | 内容 |
|---|---------|------|
| A-1 | DB↔Service 型整合 | エンティティカラム型と Service メソッド引数/戻り値型の互換性 |
| A-2 | Service↔API 契約整合 | Service メソッドシグネチャと Server Action/Route Handler の入出力整合 |
| A-3 | バリデーション↔エンティティ整合 | Zod スキーマフィールドがエンティティカラムと一致（幽霊フィールド/欠落チェック） |
| A-4 | エラー型カバレッジ | Service メソッドが参照するエラー型が全てエラー型カタログに定義されている |
| A-5 | DI インターフェース↔実装整合 | DI インターフェースのメソッドが仕様化された Repository/Service メソッドと一致 |
| A-6 | 状態遷移整合 | ステータスカラムの Enum 値、Service の遷移ルール、API のレスポンスが整合 |

#### カテゴリB: architecture.md との整合性

| # | チェック | 内容 |
|---|---------|------|
| B-1 | エンティティ忠実性 | テーブル定義が architecture.md のエンティティ概要と一致（欠落/幽霊なし） |
| B-2 | コンポーネント忠実性 | Service/API 仕様が architecture.md のコンポーネント一覧と一致 |
| B-3 | ディレクトリ整合 | 仕様で参照するファイルパスが architecture.md のディレクトリ構造と整合 |
| B-4 | エラーハンドリング整合 | エラー型と処理パターンが architecture.md の方針と整合 |
| B-5 | DI 設計整合 | DI インターフェースが architecture.md の DI 設計方針と整合 |

#### カテゴリC: standards.md との整合性

| # | チェック | 内容 |
|---|---------|------|
| C-1 | 命名規則準拠 | テーブル/カラム/メソッド/インターフェース名が命名規則に準拠 |
| C-2 | バリデーション戦略整合 | バリデーションスキーマがレイヤー別バリデーション戦略と整合 |
| C-3 | トランザクション方針準拠 | トランザクション境界が standards.md の方針と整合 |
| C-4 | スキーマ設計方針準拠 | カラム型、制約、デフォルト値が standards.md のスキーマ設計方針と整合 |

#### カテゴリD: requirements.md との整合性

| # | チェック | 内容 |
|---|---------|------|
| D-1 | SR カバレッジ | 全 SR がトレーサビリティセクションで仕様にマッピングされている |
| D-2 | NFR 反映 | パフォーマンス/セキュリティ NFR がインデックス設計、認証要件等に反映 |
| D-3 | IR カバレッジ | 全 IR が API 仕様にマッピングされている |

### 3. 結果の保存

チェック結果を {findings_save_path} に Write で保存する:

```markdown
# 整合性チェック結果

## 判定

{pass / needs_revision}

## 指摘一覧

{指摘がない場合は「指摘なし」}

### {チェックID}: {チェック名}

- **重要度**: {critical / warning}
- **内容**: {具体的な不整合の説明}
- **修正案**: {具体的な修正提案}

## サマリ

- チェック項目: {N}件
- 指摘: {N}件（critical: {N}, warning: {N}）
```

### 4. 判定基準

- **pass**: critical が 0 件
- **needs_revision**: critical が 1 件以上

critical の基準:
- エンティティ不一致（B-1）
- コンポーネント不一致（B-2）
- DB↔Service 層間型不整合（A-1）
- Service↔API 層間型不整合（A-2）
- 命名規則違反（C-1）
- SR 未カバー（D-1）
- IR 未カバー（D-3）

warning の基準:
- エラー型カバレッジ不足（A-4）
- 状態遷移整合の部分的不整合（A-6）
- ディレクトリパス不一致（B-3）
- バリデーション戦略の部分的逸脱（C-2）
- トランザクション方針の部分的逸脱（C-3）
- スキーマ設計方針の部分的逸脱（C-4）
- NFR の部分的未反映（D-2）

### 5. 返答

以下のフォーマットで返答する:

```
verdict: {pass / needs_revision}
issues: {指摘件数}
summary: {1行サマリ}
```
