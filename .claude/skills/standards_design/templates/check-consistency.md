以下の手順で開発規約書の整合性チェックを実行してください:

## 手順

### 1. 入力の読み込み

以下のファイルを全て Read で読み込む:
- {design_path}（開発規約書）
- {arch_path}（アーキテクチャ設計書）
- {req_path}（要件定義書）
- {adr_dir} 配下の全ADRファイル（Glob で検索）

### 2. 整合性チェック

以下の3カテゴリのチェックを実行する。

#### カテゴリA: 規約間整合性

開発規約書内のセクション間の整合性を検証する:

| # | チェック | 内容 |
|---|---------|------|
| A-1 | 命名規則↔ディレクトリ | 命名規則（ファイル名ケーススタイル等）がディレクトリ構成の命名と一致しているか |
| A-2 | 命名規則↔スキーマ | コード命名規則とスキーマ命名規約が一貫しているか（camelCase vs snake_case の使い分け明確化） |
| A-3 | Linter↔規約 | 選定 Linter のルールセットが規約（命名、インポート順序等）と矛盾しないか |
| A-4 | コンポーネント設計↔ディレクトリ | コンポーネント設計パターンとディレクトリ構成パターンが整合しているか |
| A-5 | バリデーション↔エラーハンドリング | バリデーション戦略のエラー報告方式がエラーハンドリング規約と整合しているか |
| A-6 | セキュリティヘッダー↔CORS/CSP | HTTP セキュリティヘッダーと CORS/CSP 設定方針に矛盾がないか |

#### カテゴリB: アーキテクチャ設計書との整合性

| # | チェック | 内容 |
|---|---------|------|
| B-1 | 技術スタック一致 | 規約が参照する技術（Linter、Formatter、FW 機能等）がアーキテクチャの技術スタックと一致しているか |
| B-2 | コンポーネント網羅 | アーキテクチャのコンポーネントが全てディレクトリ構成にマッピングされているか |
| B-3 | 横断的関心事の反映 | アーキテクチャの横断的関心事（P2 引き継ぎ事項）が規約に反映されているか |
| B-4 | エラーハンドリング整合 | エラーハンドリング規約がアーキテクチャの全体方針と整合しているか |
| B-5 | ADR一致 | 規約が ADR の決定内容と一致しているか |

#### カテゴリC: 要件定義書との整合性

| # | チェック | 内容 |
|---|---------|------|
| C-1 | NFRの反映 | セキュリティ・パフォーマンス関連の NFR が規約に反映されているか |
| C-2 | 制約の遵守 | 規約が要件定義書の制約に違反していないか |

### 3. 結果の保存

チェック結果を {findings_save_path} に Write で保存する:

```markdown
# 整合性チェック結果

## 判定

{pass / needs_revision}

## 指摘一覧

{指摘がない場合は「指摘なし」}

### {チェックID}: {チェック名}

- **重要度**: {critical / warning}
- **内容**: {具体的な不整合の説明}
- **修正案**: {具体的な修正提案}

## サマリ

- チェック項目: {N}件
- 指摘: {N}件（critical: {N}, warning: {N}）
```

### 4. 判定基準

- **pass**: critical が 0 件
- **needs_revision**: critical が 1 件以上

critical の基準:
- 技術スタックとの矛盾（B-1）
- エラーハンドリング方針との矛盾（B-4）
- ADRとの矛盾（B-5）
- 制約違反（C-2）

warning の基準:
- コンポーネントマッピングの不完全さ（B-2）
- 横断的関心事の部分的な未反映（B-3）
- 規約間の部分的な不整合（A-1〜A-6）
- NFR の部分的な未反映（C-1）

### 5. 返答

以下のフォーマットで返答する:

```
verdict: {pass / needs_revision}
issues: {指摘件数}
summary: {1行サマリ}
```
