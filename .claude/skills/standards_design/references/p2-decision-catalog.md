# P2 決定項目カタログ

classification.md の P2（開発規約設計）33 項目を依存関係とテーマに基づき 5 議論グループに編成する。

## P1 との性質の違い

| 観点 | P1（arch_design） | P2（standards_design） |
|------|-------------------|----------------------|
| 決定の性質 | 「どれを選ぶか」型。比較検討が中心 | 「どう書くか」型。方針策定が中心 |
| ADR 密度 | 高（29/39 項目） | 低（2/33 項目） |
| 項目ごとの対話量 | 多い（トレードオフ分析、ADR 作成） | 少ない（P1 決定からの導出が中心） |
| 対話パターン | 項目ごとに議論 → ADR → 確認 | グループ一括で規約提案 → 確認・修正 |

## グループ編成の根拠

- **Linter/Formatter 先行**: ツール選定が他の規約の自動強制手段を規定するため、最初に決定する
- **コードパターン → 構成**: コードレベルの規約（エラーハンドリング、非同期処理等）を先に確定し、その上でプロジェクト構成（ディレクトリ、モジュール境界）を決定する
- **セキュリティ・データは独立**: P1 決定（認証方式、DB/ORM）に依存するが、他の P2 グループへの依存は弱い。テーマの凝集度を優先して統合
- **FE は FW 依存**: P1 の FE FW 選定結果に強く依存し、他 P2 グループとは独立

Linter/Formatter 選定と基本コードスタイル。全後続グループの前提。

| # | 項目 | ADR | 手戻り | 備考 |
|---|------|:---:|:---:|------|
| 1 | Linter 設定 | 要 | 大 | Biome/ESLint。全コードに影響 |
| 2 | Formatter 設定 | 要 | 大 | Biome/Prettier。Linter と連動。1 と同一 ADR の可能性 |
| 3 | TypeScript 設定方針 | — | 大 | strict: true が新規では標準 |
| 4 | 命名規則 | — | 大 | 全コードの可読性・一貫性に影響 |
| 5 | コメント・ドキュメント方針 | — | 小 | JSDoc/TSDoc 使用方針 |
| 6 | インポート順序・整理ルール | — | 小 | 自動ソート設定。Linter 選定に依存 |
| 7 | Git commit メッセージ規約 | — | 小 | Conventional Commits 等 |

**ADR グルーピング**: Linter + Formatter は選択肢が連動する（Biome 採用時は 1 ADR で両方解決）。ADR 単位はスキル実行時に判断する。

## 議論グループ間の依存関係

```
P1 (architecture.md)
  ├→ Group 1（コーディングツール・スタイル）
  │    ├→ Group 2（コードパターン）
  │    └→ Group 3（プロジェクト構成）
  ├→ Group 4（セキュリティ・データ管理）
  └→ Group 5（フロントエンド UX・AI）
```

Group 2, 3 は Group 1 に依存。Group 4, 5 は P1 に依存するが Group 1-3 とは独立。議論順序は 1 → 2 → 3 → 4 → 5 とする（依存順で自然に流れる）。
