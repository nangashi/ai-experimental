# Plan テンプレートガイド

plan.md を作成する際の構造ガイド。成果物の種類（コード / プロンプト・エージェント定義）に応じて、各セクションの記載粒度を調整する。

## 必須セクション

### 1. 概要

変更の目的と背景を1-3文で。「何を」「なぜ」変更するかを明確にする。

### 2. 変更スコープ

変更対象ファイルの一覧と、各ファイルでの変更概要。

```markdown
| ファイル | 変更内容 |
|---------|---------|
| path/to/file | 変更の概要 |
```

### 3. アプローチ

実装方針の詳細。以下は成果物の種類に応じた記載ポイント。

#### コード変更の場合

- **変更内容のスニペット**: 主要な変更を擬似コードまたは実コードで示す。特にインターフェースの変更は変更前→変更後を明示する
- **既存パターンとの整合**: コードベース内の類似実装がある場合は参照先を明示し、それに揃える
- **テスト方針**: 新規テスト追加 / 既存テスト修正 / 手動検証のいずれか。テストがない領域はその旨を記載
- **互換性**: 破壊的変更がある場合、マイグレーション手順や影響範囲を明示する

例:

```markdown
`UserService.getById()` の返却型を `User | null` から `Result<User>` に変更する。

呼び出し元（3箇所）は以下のパターンで修正:

- `api/users.ts:42` — エラーハンドリングを Result 型に対応
- `api/users.ts:78` — 同上
- `middleware/auth.ts:15` — 同上

テスト: `user-service.test.ts` の既存テストを Result 型に対応させる。新規テストは不要。
```

#### プロンプト・エージェント定義変更の場合

- **変更セクションの特定**: 変更するセクション名や指示を具体的に引用する
- **振る舞いの変化**: 変更前→変更後で期待する振る舞いがどう変わるか
- **下流への影響**: このファイルを参照する他のファイル・スキル・エージェントへの影響
- **検証方法**: 変更が意図通りに機能するかの確認手段

例:

```markdown
SKILL.md の Phase 2 に入力バリデーションステップを追加する。

変更前: Phase 1 の出力をそのまま Phase 2 の入力として使用
変更後: Phase 2 開始時にフォーマット検証を行い、不正な場合は Phase 1 を再実行

影響: このスキルを呼び出す CLAUDE.md の Workflow セクションには変更不要。
検証: 不正フォーマットの入力でスキルを実行し、再実行が発動することを確認。
```

### 4. トレードオフ・検討事項

代替アプローチがあればその比較。採用したアプローチのリスクや制約。
代替案がなく、アプローチが自明な場合はこのセクションを省略してよい。

### 5. チェックリスト

実装タスクをステップごとに分解する。Phase 3 で完了時にチェックを付けていく。

```markdown
- [ ] タスク1の説明
- [ ] タスク2の説明
  - [ ] サブタスク2aの説明
  - [ ] サブタスク2bの説明
- [ ] タスク3の説明
```

粒度の目安: 1タスク = 1つの論理的変更単位。「ファイルAのX関数を修正」「SKILL.md の Phase N にステップを追加」程度。
