既存ファイルの変更を伴う作業において、調査→計画→実装の3段階で進行する。各段階でユーザーのレビューと承認を経てから次に進む。

## 使い方

```
/research_and_plan <task description>
```

CLAUDE.md の Workflow セクションの条件に合致する場合、自動的に実行される。手動での実行も可能。

## パス変数

- `{skill_dir}`: このファイルが存在するディレクトリの絶対パス
- `{work_dir}`: `.tasks/{yyyy-mm-dd}_{work_name}/` の絶対パス
- `{work_name}`: タスク記述から生成する英数字+ハイフンのスラッグ（30文字以内）

## ワークフロー

### Phase 0: 初期化

1. タスク記述から `{work_name}` を生成する（英数字+ハイフン、30文字以内。日本語はローマ字化）
2. `{work_dir}` = `.tasks/{yyyy-mm-dd}_{work_name}/` を設定する（`{yyyy-mm-dd}` は当日の日付）
3. `Bash`: `mkdir -p {work_dir}`

---

### Phase 1: Research

**目的:** 変更対象と周辺の深い理解を文書化し、誤解を実装前に検出する

#### Step 1: 調査

タスクに関連するファイル・コードを深く調査する。表面的な読解ではなく、変更に必要な詳細レベルで理解する。

特に以下を把握する:
- 変更対象ファイルの現在の構造と役割
- 関連する依存ファイル・参照元
- 既存の規約・パターン
- 変更の影響範囲

#### Step 2: 文書化

調査結果を `{work_dir}/research.md` に Write で保存する。

- 形式は自由。調査対象の理解を正確に文書化することが目的
- 誤解があればこの段階で発見できるよう、具体的な事実（ファイルパス、行番号、既存の振る舞い）を含める
- 推測や不確実な点があれば明示する

#### Step 3: レビュー依頼

ユーザーにレビューを依頼する:

「research.md を作成しました。内容を確認してください。修正・補足があればお知らせください。問題なければ『ok』と回答してください。」

- 承認（ok等）: Phase 2 へ
- 修正指示あり: research.md を更新して再度レビュー依頼

---

### Phase 2: Plan

**目的:** チェックリスト付きの実装計画を作成し、実装前にユーザーと合意する

#### Step 1: 計画作成

1. `{skill_dir}/templates/plan.md` を Read で読み込み、ガイドに従って `{work_dir}/plan.md` を作成する
2. research.md の調査結果を基に、具体的な実装計画を策定する
3. チェックリスト（タスク分解）を必ず含める

#### Step 2: レビュー依頼

ユーザーにレビューを依頼する:

「plan.md を作成しました。内容を確認してください。修正・補足があればお知らせください。実装に進んでよければ『実装して』と回答してください。」

- 修正指示あり: plan.md を更新して再度レビュー依頼
- 実装指示（「実装して」「go」「ok」等）: Phase 3 へ

**実装開始の明示的な指示があるまで実装に着手しない。**

---

### Phase 3: Implementation

**目的:** plan.md に従って実装し、進捗を記録する

1. plan.md のチェックリストに従って順に実装する
2. 各タスク完了時に plan.md のチェックボックスを `[x]` に Edit で更新する
3. 全タスク完了後、ユーザーに報告する

```
## 完了

plan.md の全タスクが完了しました。

{変更内容のサマリ}
```
