以下の手順で基本設計書の整合性チェックを実行してください:

## 手順

### 1. 入力の読み込み

以下のファイルを全て Read で読み込む:
- {design_path}（基本設計書）
- {arch_path}（アーキテクチャ設計書）
- {req_path}（要件定義書）
- {adr_dir} 配下の全ADRファイル（Glob で検索）

### 2. 整合性チェック

以下の3カテゴリのチェックを実行する。

#### カテゴリA: 項目間整合性

基本設計書内の4セクション間の整合性を検証する:

| # | チェック | 内容 |
|---|---------|------|
| A-1 | ライブラリ↔ディレクトリ | 選定したライブラリが要求するディレクトリ（例: Prisma → `prisma/`）がディレクトリ構造に存在するか |
| A-2 | 規約↔API設計 | 命名規則（APIパス・フィールド名）がAPIエンドポイント設計で実際に守られているか |
| A-3 | 規約↔ディレクトリ | ファイル命名規則がディレクトリ構造の命名と一致しているか |
| A-4 | ライブラリ↔規約 | 選定ライブラリのデフォルト規約とプロジェクト規約に矛盾がないか |

#### カテゴリB: アーキテクチャ設計書との整合性

| # | チェック | 内容 |
|---|---------|------|
| B-1 | コンポーネント網羅 | アーキテクチャ設計書のコンポーネントが全てディレクトリ構造にマッピングされているか |
| B-2 | 技術スタック一致 | 基本設計のライブラリ選定がアーキテクチャの技術スタックと矛盾しないか |
| B-3 | 横断的関心事の反映 | アーキテクチャの横断的関心事（ログ・監視、エラーハンドリング等）が規約に反映されているか |
| B-4 | ADR一致 | ライブラリ選定がADRの決定内容と一致しているか |

#### カテゴリC: 要件定義書との整合性

| # | チェック | 内容 |
|---|---------|------|
| C-1 | UC/SR網羅 | Must のユースケース・機能要件に対応するAPIエンドポイントが存在するか |
| C-2 | リソース網羅 | ユースケースで扱うデータがリソース定義でカバーされているか |

### 3. 結果の保存

チェック結果を {findings_save_path} に Write で保存する:

```markdown
# 整合性チェック結果

## 判定

{pass / needs_revision}

## 指摘一覧

{指摘がない場合は「指摘なし」}

### {チェックID}: {チェック名}

- **重要度**: {critical / warning}
- **内容**: {具体的な不整合の説明}
- **修正案**: {具体的な修正提案}

## サマリ

- チェック項目: {N}件
- 指摘: {N}件（critical: {N}, warning: {N}）
```

### 4. 判定基準

- **pass**: critical が 0 件
- **needs_revision**: critical が 1 件以上

critical の基準:
- 要件の未カバー（C-1, C-2）
- アーキテクチャとの矛盾（B-2, B-4）
- 項目間の明確な矛盾（A-1〜A-4）

warning の基準:
- コンポーネントマッピングの不完全さ（B-1）
- 横断的関心事の部分的な未反映（B-3）

### 5. 返答

以下のフォーマットで返答する:

```
verdict: {pass / needs_revision}
issues: {指摘件数}
summary: {1行サマリ}
```
