以下の手順で開発プロセス設計書の整合性チェックを実行してください:

## 手順

### 1. 入力の読み込み

以下のファイルを全て Read で読み込む:
- {design_path}（開発プロセス設計書）
- {arch_path}（アーキテクチャ設計書）
- {req_path}（要件定義書）
- {adr_dir} 配下の全ADRファイル（Glob で検索）

### 2. 整合性チェック

以下の3カテゴリのチェックを実行する。

#### カテゴリA: 項目間整合性

開発プロセス設計書内のセクション間の整合性を検証する:

| # | チェック | 内容 |
|---|---------|------|
| A-1 | ライブラリ↔ディレクトリ | 選定したライブラリが要求するディレクトリ（例: Prisma → `prisma/`）がディレクトリ構造に存在するか |
| A-2 | 規約↔ディレクトリ | ファイル命名規則がディレクトリ構造の命名と一致しているか |
| A-3 | ライブラリ↔規約 | 選定ライブラリのデフォルト規約とプロジェクト規約に矛盾がないか |
| A-4 | 品質ツール↔品質ゲート | 選定したリンター・フォーマッターが品質ゲートのチェック内容に含まれているか |
| A-5 | 品質ゲート↔CI/CD | commit時/push時のチェックとCIのチェックに意図しない重複がないか |
| A-6 | パッケージマネージャ↔CI/CD | CIのインストールステップがパッケージマネージャと一致しているか |

#### カテゴリB: アーキテクチャ設計書との整合性

| # | チェック | 内容 |
|---|---------|------|
| B-1 | コンポーネント網羅 | アーキテクチャ設計書のコンポーネントが全てディレクトリ構造にマッピングされているか |
| B-2 | 技術スタック一致 | ライブラリ選定・ランタイム管理がアーキテクチャの技術スタックと矛盾しないか |
| B-3 | 横断的関心事の反映 | アーキテクチャの横断的関心事（ログ・監視、エラーハンドリング等）が規約に反映されているか |
| B-4 | デプロイ戦略一致 | CD パイプラインがアーキテクチャのデプロイ戦略と整合しているか |
| B-5 | ADR一致 | ライブラリ・ツール選定がADRの決定内容と一致しているか |

#### カテゴリC: 要件定義書との整合性

| # | チェック | 内容 |
|---|---------|------|
| C-1 | テスト方針の充足性 | テスト方針がNFRのテスト要求を満たしているか |
| C-2 | 制約の遵守 | 選定したツール・ライブラリが要件定義書の制約に違反していないか |

### 3. 結果の保存

チェック結果を {findings_save_path} に Write で保存する:

```markdown
# 整合性チェック結果

## 判定

{pass / needs_revision}

## 指摘一覧

{指摘がない場合は「指摘なし」}

### {チェックID}: {チェック名}

- **重要度**: {critical / warning}
- **内容**: {具体的な不整合の説明}
- **修正案**: {具体的な修正提案}

## サマリ

- チェック項目: {N}件
- 指摘: {N}件（critical: {N}, warning: {N}）
```

### 4. 判定基準

- **pass**: critical が 0 件
- **needs_revision**: critical が 1 件以上

critical の基準:
- アーキテクチャとの矛盾（B-2, B-4）
- ADRとの矛盾（B-5）
- 制約違反（C-2）
- 項目間の明確な矛盾（A-1, A-3）

warning の基準:
- コンポーネントマッピングの不完全さ（B-1）
- 横断的関心事の部分的な未反映（B-3）
- チェックの重複（A-5）
- 部分的な不整合（A-2, A-4, A-6, C-1）

### 5. 返答

以下のフォーマットで返答する:

```
verdict: {pass / needs_revision}
issues: {指摘件数}
summary: {1行サマリ}
```
