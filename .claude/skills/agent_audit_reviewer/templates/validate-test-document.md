以下の手順でテスト対象文書と正解キーの品質を検証してください:

## 入力の読み込み

1. Read で以下のファイルを読み込む:
   - {test_document_path} （テスト対象文書）
   - {answer_key_path} （正解キー）
   - {perspective_path} （観点定義 — 問題バンクを含まない作業コピー）

## 検証1: テスト対象文書の自然さ

2. テスト対象文書を**正解キーを参照せずに**通読し、以下の不自然パターンに該当する箇所がないか確認する:

   **不自然パターン（1つでも該当すればFAIL）**:
   - **括弧内正当化**: 弱い設計判断の直後に括弧で防御的な理由を述べている（例: 「〜は行わない（Xで十分と判断）」）
   - **明示的アンチパターン推奨**: 設計者がセキュリティ上明らかに悪い慣行を意図的に採用している記述（例: 機密データの意図的なログ出力指示、シークレットのリポジトリ格納指示）
   - **機密データの不自然な列挙**: 問題を目立たせるためにセンシティブな要素を不自然に並べた記述

   不自然な箇所が見つかった場合、行番号・パターン名・具体的な記述を記録する

## 検証2: 正解キーの検出判定基準の明確性

3. 正解キーの各問題（P01-P15）の検出判定基準を確認する:
   - ○と△の境界が**一意に解釈可能**か（読み手によって判定が分かれないか）
   - △の基準が「関連カテゴリへの言及はあるが、{核心的なポイント}に触れていない」の形式に沿っているか
   - 曖昧な基準（「適切に指摘している」「十分に言及している」等の主観的表現）がないか
   - 問題ごとに「明確 / 要改善」を判定する

## 検証3: 分布要件の充足

4. 以下の分布要件をカウントして確認する:

   | 項目 | 要件 | 実際 | 判定 |
   |------|------|------|------|
   | 総数 | 15問 | {N} | OK/NG |
   | 深刻度: 重大 | 4-5 | {N} | OK/NG |
   | 深刻度: 中 | 6-7 | {N} | OK/NG |
   | 深刻度: 軽微 | 4-5 | {N} | OK/NG |
   | 検出難易度: easy | 3-4 | {N} | OK/NG |
   | 検出難易度: medium | 6-8 | {N} | OK/NG |
   | 検出難易度: hard | 3-5 | {N} | OK/NG |

## 検証4: perspectiveカテゴリのカバレッジ

5. perspective.md の「評価スコープ」に列挙されたカテゴリと、正解キーの各問題のカテゴリを照合する:
   - 各カテゴリに最低1問が配分されているか確認する
   - 出題のないカテゴリがあれば記録する

## 結果の返答

6. 以下のフォーマットで検証結果のみ返答する:

```
## 検証結果サマリ
- 自然さ: {OK / 問題あり（N箇所）}
- 判定基準の明確性: {OK / 要改善（N問）}
- 分布要件: {OK / NG（項目名）}
- カテゴリカバレッジ: {OK / 欠落あり（カテゴリ名）}
- 総合判定: {PASS / FAIL}

## 問題詳細（総合判定がFAILの場合のみ）
{問題の詳細リスト}
```
