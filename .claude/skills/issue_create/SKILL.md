対話形式でユーザーから要件をヒアリングし、適切な粒度に分解してGitHub Issueを作成する。
複数Issueに分解した場合は、GitHub のブロッキング依存関係を設定する。

## 使い方

```
/issue_create
```

引数なし。対話形式でIssue内容を収集する。

## 前提

- リポジトリ内で実行すること
- `gh` CLIが認証済みであること

## パス変数

- `{adr_dir}`: `docs/adr/` の絶対パス（プロジェクトルートからの相対）

## ワークフロー

Phase 0（初期化）→ 1（ヒアリング）→ 2（重複チェック）→ 3（分解）→ 4（確認プレビュー）→ 5（実行）

---

### Phase 0: 初期化

**目的:** 前提条件の検証とリポジトリ情報の取得

#### Step 1: リポジトリ情報の取得

`Bash` で以下を実行する:

```
git remote get-url origin
```

出力から `{owner}` と `{repo}` を導出する:
- SSH形式 `git@github.com:owner/repo.git` → owner, repo
- HTTPS形式 `https://github.com/owner/repo.git` → owner, repo

失敗した場合: 「Gitリポジトリ内で実行してください」と出力し、スキル終了。

#### Step 2: GitHub認証の確認

`Bash` で以下を実行する:

```
gh auth status
```

失敗時: 「`gh auth login` でGitHubに認証してください」と出力し、スキル終了。

#### Step 3: ラベル一覧の取得

`Bash` で以下を実行する:

```
gh label list --json name --limit 100
```

`{available_labels}` として保持する。取得失敗時はラベルなしで続行する。

#### Step 4: 既存ADRのスナップショット

`Bash` で以下を実行する:

```
ls docs/adr/[0-9][0-9][0-9][0-9]-*.md 2>/dev/null
```

出力を `{adr_snapshot}` として保持する（ファイル名の一覧）。ファイルが存在しない場合は空リストとする。

テキスト出力:
```
## Phase 0 完了
- リポジトリ: {owner}/{repo}
- ラベル: {N}件
- 既存ADR: {N}件

Phase 1（ヒアリング）に進みます。
```

---

### Phase 1: ヒアリング

以下をテキスト出力し、ユーザーの応答を待つ:

「何を開発したいですか？背景・やりたいこと・制約・完了条件などわかる範囲で教えてください。」

ユーザーの回答から以下を構造化する:

- `{background}`: 背景・動機
- `{tasks}`: 具体的にやること
- `{constraints}`: 制約（なければ「なし」）
- `{acceptance_criteria}`: 受け入れ基準（検証可能な完了条件）
- `{out_of_scope}`: スコープ外（なければ「なし」）

情報が不足している場合は、不足している項目について追加質問する。

Phase 2 へ進む。

---

### Phase 2: 重複チェック

#### Step 1: 類似Issue検索

Phase 1 の `{tasks}` からキーワードを抽出し、`Bash` で以下を実行する:

```
gh issue list --state open --search "{keywords}" --limit 10 --json number,title
```

- 取得失敗時: 「Issue一覧の取得に失敗しました。`gh auth status` を確認してください」と出力し、スキル終了

#### Step 2: 重複判定

類似Issueが0件の場合: Phase 3 へ進む。

類似Issueが1件以上の場合: 以下のテーブルを提示し、`AskUserQuestion` で確認する:

```
以下の類似Issueが見つかりました:

| # | タイトル |
|---|---------|
| #{number} | {title} |
```

選択肢:
- **続行** — 新しいIssueを作成する
- **中止** — 既存Issueを使用する

- 続行: Phase 3 へ
- 中止: 「既存Issueを使用してください」と出力し、スキル終了

---

### Phase 3: 分解

**目的:** 要件を分析し、適切な粒度のIssue群に分解する

#### Step 0: 技術採用検出とADR委譲

**目的:** 新しい技術採用を伴う要件を検出し、必要に応じてadr_createに委譲する

##### 0-1. 技術採用シグナルの検出

Phase 1 の `{tasks}`, `{constraints}`, `{acceptance_criteria}` を分析し、以下のシグナルに該当する項目を抽出する。

**検出シグナル（いずれか1つ該当で候補とする）:**
- 具体的な技術・フレームワーク・ライブラリの名前が挙がっており、プロジェクトで未採用である
- 「〜を導入する」「〜に移行する」「〜を採用する」等の新規技術採用を示す表現がある
- 複数の技術候補が併記されている（例:「AまたはBを使って」）
- インフラ・アーキテクチャレベルの方式変更を含む（例: REST→GraphQL、モノリス→マイクロサービス）

**検出対象外（ADR不要）:**
- 既存ADRで決定済みの技術の利用
- 標準ライブラリや言語組み込み機能の使用
- 制約により選択肢が実質1つしかない場合

##### 0-2. 既存ADRとの突合

検出した技術採用候補それぞれについて:

1. `{adr_snapshot}` に含まれるADRの「決定」セクションのみを `Read` する
2. 既存ADRで既に決定済みの技術に該当する場合は候補から除外する
3. 候補が0件の場合: Step 1（分解判定）へ進む

##### 0-3. ユーザーへのADR提案

残った候補を以下の形式で提示し、`AskUserQuestion` で確認する:

```
以下の技術採用について、トレードオフ分析（ADR作成）が必要と判断しました:

### 技術選定: {選定ポイント名}
- 対象: {何を選定するか}
- 検出理由: {なぜADRが必要と判断したか — 1行}
```

候補が複数件の場合は、各候補について上記を繰り返す。

選択肢:
- **ADR作成** — 別セッションでADRを作成する（複数件の場合は順次実行）
- **不要** — ADRなしでIssue作成を続行する

- 不要: Step 1（分解判定）へ進む
- ADR作成: Step 0-4 へ

##### 0-4. ADR委譲

各ADR候補について以下を順次実行する。

テキスト出力:

```
### 技術選定: {選定ポイント名}

この判断は選択肢のトレードオフ分析が必要です。
以下のコマンドを **別セッション** で実行してください:

`/adr_create {選定ポイント名}。スコープ: {tasks から導出した対象項目}。制約: {constraints から導出した制約}`

完了したら「完了」と回答してください。
```

ユーザーが「完了」と回答したら:

1. `{adr_dir}` の ADR ファイル一覧を取得し、`{adr_snapshot}` に含まれないファイルを新ADRとして特定する
2. 新ADRの「決定」「受け入れたトレードオフ」セクションのみ `Read` する
3. 以下を `{adr_decisions}` に追加する:
   - `{adr_path}`: ADRファイルパス
   - `{adr_title}`: ADRタイトル
   - `{adr_decision}`: 決定内容の要約（1-2文）
   - `{adr_tradeoffs}`: 受け入れたトレードオフの要約（1-2文）
4. `{adr_snapshot}` を更新する（新ADRを含める）
5. 次の候補がある場合は次の候補について繰り返す。全候補完了後、Step 1（分解判定）へ進む

テキスト出力:

```
ADR読込完了: {adr_title}
- 決定: {adr_decision}
- トレードオフ: {adr_tradeoffs}
```

---

#### Step 1: 分解判定

Phase 1 の `{tasks}` と `{acceptance_criteria}` を以下で分析する。

**分割する（いずれか1つ該当）:**
- 要件が2つ以上の独立した機能領域にまたがる
- 受け入れ基準が独立して検証可能なものが4つ以上ある

**単一Issueとする:**
- 分割しても各部分が独立して価値を持たない

#### Step 2: 分解実行

**単一Issueの場合:**

「要件は十分にスコープが絞られています。単一Issueとして作成します。」と出力し、以下を生成する:

- `{issues}`: 1件のIssue
  - `{title}`: 命令形のタイトル
  - `{body}`: Issue本文（Phase 4 のテンプレートで生成）
  - `{labels}`: 適切なラベル（`{available_labels}` から選択）
  - `{blocked_by}`: なし

Phase 4 へ進む。

**複数Issueに分解する場合:**

以下の原則に従って分解する:

1. **垂直スライス優先** — 水平分割（DB層・API層・UI層の分離）ではなく、機能ごとのエンドツーエンドのスライスにする
2. **独立完結性** — 各Issueは単独のAI開発セッションで完了可能であること
3. **独立検証性** — 各Issueに独立して検証可能な受け入れ基準があること（「後続の準備」のみのIssueは作らない）
4. **依存の明示** — Issue間にブロッキング依存がある場合、`{blocked_by}` に記録する

以下を生成する:

- `{issues}`: Issue配列（依存順序で並べる）
  - 各要素:
    - `{title}`: 命令形のタイトル
    - `{body}`: Issue本文
    - `{labels}`: 適切なラベル（`{available_labels}` から選択）
    - `{blocked_by}`: ブロッキング依存する他のIssueのインデックス配列（なければ空）

Phase 4 へ進む。

---

### Phase 4: 確認プレビュー

#### Step 1: Issue内容の生成

各Issueの本文を以下のテンプレートに従って生成する:

**Issue本文テンプレート:**

```markdown
## 背景

{background}

## やること

{tasks — 箇条書きで}

## 受け入れ基準

- [ ] {acceptance_criteria — 検証可能な条件をチェックボックス形式で}

## 制約

{constraints — なければ「なし」}

## スコープ外

{out_of_scope — なければ「なし」}
```

複数Issueの場合、各Issueの `{background}` はそのIssueのスコープに関連する部分のみを記載する。`{out_of_scope}` には他のIssueで扱う部分を含める。

**ADR参照の統合:**

`{adr_decisions}` が存在する場合（Phase 3 Step 0 でADRを作成した場合）、以下のようにIssue本文に統合する:

- `{background}` セクション: ADRの決定結果を背景情報として含める。「{adr_title} で {adr_decision} と決定した（詳細: {adr_path}）」の形式で記載する
- `{constraints}` セクション: ADRの受け入れたトレードオフを制約として追加する。「{adr_title} の決定に基づく制約: {adr_tradeoffs}」の形式で記載する

複数のADRがある場合は、各Issueのスコープに関連するADRのみを含める。

#### Step 2: 記載妥当性チェック

**目的:** メインエージェントとユーザーが共有する会話コンテキストに起因するバイアス（知識の呪い）を、コンテキストを持たないサブエージェントで検出する。

各Issueについて `Task`（subagent_type: `general-purpose`, model: `haiku`）を起動する。複数Issueの場合は全Issueを同一メッセージ内で並列起動する。

**サブエージェントへのプロンプト:**

```
あなたはAIコード開発エージェントです。多様な技術スタックでのフルスタック開発経験があり、与えられたIssueの記述のみに基づいてコードベースを調査し実装を完了します。

以下のGitHub Issueがあなたに割り当てられました。このIssue本文以外のプロジェクト背景情報はありません。

---
タイトル: {title}

{body}
---

このIssue本文だけで作業を開始できるか、以下の4観点で評価してください:

1. 情報の欠落: 実装を始めるために必要だが記載されていない情報はあるか
2. 要件の曖昧さ: 複数の異なる実装が正当化される曖昧な記述はあるか
3. 暗黙の前提: 明示されていないが前提とされている技術選定・仕様・制約はあるか
4. 検証不能な基準: 受け入れ基準の中に、客観的に合否を判定できないものはあるか

問題がなければ「指摘なし」とだけ回答してください。問題を発見した場合は遠慮なく指摘してください。

出力形式（指摘がある場合のみ）:
1. **箇所**: {対象セクション・記述}
   **問題**: {何が不足・曖昧か}
   **影響**: {このまま作業した場合に起こりうる問題}
```

サブエージェントの返却を `{review_findings_N}` として保持する（Nは各Issueのインデックス）。

#### Step 3: プレビューの提示

**単一Issueの場合:**

以下をテキスト出力し、`AskUserQuestion` で確認する:

```
### Issue

**タイトル**: {title}
**ラベル**: {labels}

**本文**:
{body}
```

`{review_findings_1}` が「指摘なし」でない場合、以下を追記する:

```
### 記載チェック指摘事項

{review_findings_1}
```

選択肢:
- **作成** — この内容でIssueを作成する
- **修正** — 修正点を指示する

以下の観点についても確認を促す:
- 独立完結性: 単独のAI開発セッションで完了可能か
- 独立検証性: 受け入れ基準だけで合否判定が可能か
- 垂直スライス: エンドツーエンドの機能を提供するか（「DBスキーマだけ」「APIだけ」になっていないか）
- 自己完結性: AI開発エージェントがこのIssue本文だけで作業開始できる十分なコンテキストがあるか

- 作成: Phase 5 へ
- 修正: ユーザーの修正指示を反映し、Step 1 から再生成する。再生成後、Step 2 のチェックは再実行しない（ユーザーが修正内容を判断済みのため）

**複数Issueの場合:**

以下をテキスト出力する:

```
### Issue一覧 ({N}件)

#### [{1}/{N}] {title_1}
**ラベル**: {labels_1}
**ブロック**: なし

**本文**:
{body_1}

**記載チェック**: {review_findings_1 — 「指摘なし」の場合はこの行自体を省略}

---

#### [{2}/{N}] {title_2}
**ラベル**: {labels_2}
**ブロック**: [{1}] {title_1} の完了後に着手可能

**本文**:
{body_2}

**記載チェック**: {review_findings_2 — 「指摘なし」の場合はこの行自体を省略}

---
...（全Issue分）

### 依存関係
{title_1} → {title_2} → {title_3}
（{title_4} は独立して着手可能）
```

`AskUserQuestion` で確認する:

選択肢:
- **作成** — この内容で全Issueを作成する
- **修正** — 修正点を指示する

各Issueについて以下の観点も確認を促す:
- 独立完結性: 単独のAI開発セッションで完了可能か
- 独立検証性: 受け入れ基準だけで合否判定が可能か
- 垂直スライス: エンドツーエンドの機能を提供するか（「DBスキーマだけ」「APIだけ」になっていないか）
- 自己完結性: AI開発エージェントがこのIssue本文だけで作業開始できる十分なコンテキストがあるか

- 作成: Phase 5 へ
- 修正: ユーザーの修正指示を反映し、Step 1 から再生成する。再生成後、Step 2 のチェックは再実行しない（ユーザーが修正内容を判断済みのため）

---

### Phase 5: 実行

**目的:** Issueの作成とブロッキング依存関係の設定

#### Step 1: Issue作成

**単一Issueの場合:**

`Bash` で以下を実行する:

```
gh issue create --title "{title}" --body "{body}" --label "{labels}"
```

- 失敗時: 「Issue作成に失敗しました。`gh auth status` を確認してください」と出力し、スキル終了

出力からIssue番号とURLを取得する。Step 3 へ進む。

**複数Issueの場合:**

`{issues}` を依存順序で1件ずつ作成する。各Issueについて:

`Bash` で以下を実行する:

```
gh issue create --title "{title}" --body "{body}" --label "{labels}"
```

出力からIssue番号（`{number}`）とURL（`{url}`）を取得する。

失敗時: 「Issue '{title}' の作成に失敗しました。ここまでに作成したIssueは以下です:」と作成済みIssue一覧を出力し、スキル終了。

#### Step 2: ブロッキング依存関係の設定

`{blocked_by}` を持つIssueが存在しない場合: Step 3 へ進む。

`{blocked_by}` を持つ各Issueについて:

1. 対象IssueのNode IDを取得する:

```
gh api graphql -f query='query { repository(owner:"{owner}", name:"{repo}") { issue(number:{blocked_number}) { id } } }'
```

2. ブロッキングIssueのNode IDを取得する:

```
gh api graphql -f query='query { repository(owner:"{owner}", name:"{repo}") { issue(number:{blocking_number}) { id } } }'
```

3. ブロッキング依存関係を設定する:

```
gh api graphql -f query='mutation { addBlockedBy(input: {issueId: "{blocked_node_id}", blockingIssueId: "{blocking_node_id}"}) { blockedItem { ... on Issue { number } } blockingItem { ... on Issue { number } } } }'
```

- 失敗時: 「ブロッキング依存の設定に失敗しました（#{blocked_number} ← #{blocking_number}）。手動で設定してください。」と警告を出力し、**続行する**（依存設定の失敗はIssue作成を止めない）

#### Step 3: 完了情報の出力

**単一Issueの場合:**

```
## Issue作成完了

- Issue: {url}
```

**複数Issueの場合:**

```
## Issue作成完了

| # | タイトル | ブロック元 | 依存設定 |
|---|---------|-----------|---------|
| #{number_1} | {title_1} | — | — |
| #{number_2} | {title_2} | #{number_1} | OK/失敗 |
| #{number_3} | {title_3} | — | — |

依存関係のないIssueから順に、AI開発セッションを開始してください。
```
